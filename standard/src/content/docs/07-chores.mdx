# 7. Chores [#chores]
> **Normative.** This chapter defines the `chore` top-level form: a thin specialization of `recipe` for command-runner-shaped bodies. The productions referenced here are given formally in App. A §{grammar-appendix.recipes} and §{grammar-appendix.steps}; where this chapter's prose disagrees with App. A, App. A is authoritative (§{notation.grammar-precedence}).

## 7.1. Chore header [#chores.header]
A chore is introduced by a **chore header** of the form `chore name (":" dependency_list)?`, where `name` is either a `BARE_IDENTIFIER` or a double-quoted `STRING` (App. A.5). An optional `:` introduces a dependency list (semantics identical to §{recipes.dep-list}).

The keyword `chore` MUST be followed by a separator — whitespace, a tab, or a `"` character — before the name. The keyword is recognised only at column 0 (§{lexical.keywords}); an indented `chore foo` line inside a recipe body or chore body is a `Content` token dispatched as a `shell_command`.

The reserved-segment rule of §{grammar-appendix.declarations} applies: a conforming implementation MUST reject a chore whose name's final dot-separated segment is one of `stem`, `name`, `ext`, `dir`, `in`, `out`, `all`.

The **duplicate-name rule** of §{grammar-appendix.declarations} also applies: chores share the callable namespace with recipes, so a chore whose name matches that of an earlier `chore` or `recipe` declaration in the same Cookfile MUST be rejected at parse time. The diagnostic identifies both the offending declaration and the source line of the prior declaration.

A chore body is terminated by the implicit-termination rule of §{toplevel.termination}: it ranges from the line after the header to the next column-0 line classified as `recipe`, `chore`, `config`, `use`, or `import`, or end-of-file (whichever comes first).

### Example 7.1.1

```cook
chore clean
    rm -rf build

chore play: build
    fzf < build/menu.txt
```

Two chores: `clean` (no deps), `play` (depends on the recipe `build`).

## 7.1.1. Parameters [#chores.params]

A chore header MAY declare zero or more **parameters** between the chore name and the optional `:`. The full parameter grammar (including variadic and Lua-expression-default forms) is given in App. A.3.1; this subsection covers the **required** and **defaulted-string** forms.

A `param` is one of:

- **Required parameter:** a `BARE_IDENTIFIER`. Must be supplied at invocation as a positional argument.
- **Defaulted parameter (string literal):** `BARE_IDENTIFIER "=" STRING`. If no argv is supplied at this position at invocation time, the literal string binds.

**Ordering rule (normative).** Within a `param_list`, every required parameter MUST precede every defaulted parameter. A conforming implementation MUST reject a chore header that violates this order.

**Reserved-name rule (normative).** Parameter names MUST NOT equal one of the reserved identifiers from §A.2 (`stem`, `name`, `ext`, `dir`, `in`, `out`, `all`, `env`). Parameter names MUST NOT contain `.`. Duplicate parameter names within one header MUST be rejected at parse time.

### Example 7.1.1.1

```cook
chore deploy target host="prod.example.com"
    > deploy.run(target, host)
```

`deploy` declares a required parameter `target` and a defaulted-string parameter `host`. Invoking `cook deploy production` binds `target="production"` and `host="prod.example.com"`.

## 7.1.2. Parameter binding inside the body [#chores.param-binding]

When a chore body is executed, every declared parameter binds as a **Lua local** in the body's scope, visible to inline-Lua (`>>`, `>>{}`), execute-Lua (`>`, `>{}`), and lua_block forms. The local's value is a Lua string for required and defaulted parameters; for variadics (§7.1.3) it is a Lua sequence (list-style table) of strings.

A conforming implementation MUST make these bindings available before any step of the chore body (including the first `>>` inline-Lua line) is executed.

Placeholder substitution (`$<NAME>`) and environment-variable export to shell children are governed by later subsections (§9 and §7.1.2.X respectively).

## 7.2. Chore body — allowed and banned step kinds [#chores.body]

A chore body MAY contain any of the following step kinds:

| § | Step kind | Phase |
|---|---|---|
| §{steps.shell}     | `shell_command`       | execute (default-interactive) |
| §{steps.shell}     | `interactive_command` | execute |
| §{steps.lua}       | `lua_line`            | execute |
| §{steps.lua}       | `lua_block`           | execute |
| §{steps.lua}       | `inline_lua_line`     | register |
| §{steps.lua}       | `inline_lua_block`    | register |

A chore body MUST NOT contain any of the following step kinds:

| § | Step kind | Diagnostic guidance |
|---|---|---|
| §{steps.ingredients} | `ingredients_step` | The diagnostic identifies the keyword and directs the author to `recipe`. |
| §{steps.cook-single} | `cook_step` | The diagnostic identifies the keyword and directs the author to `recipe`. |
| §{steps.plate} | `plate_step` | The diagnostic identifies the keyword and directs the author to `recipe`. |
| §{steps.test} | `test_step` | The diagnostic identifies the keyword and directs the author to `recipe`. |

A conforming implementation MUST reject a chore body that contains any of the four banned step kinds. The diagnostic MUST identify the offending step's line and the offending keyword by name; the recommended message form names the keyword and suggests the `recipe` form.

The recipe-body region rule (§{recipes.region-rule}) applies to chore bodies as well, restricted to the allowed step kinds: `inline_lua_line` and `inline_lua_block` are declarative-region; `shell_command`, `interactive_command`, `lua_line`, `lua_block` are imperative-region. A conforming implementation MUST reject a chore body in which a declarative-region step appears after the first imperative-region step.

**Migration note (CS-0072 / v0.9).** Prior to CS-0072 a chore body admitted a `module_call` step (register-phase, desugaring to `inline_lua_line` / `inline_lua_block`). CS-0072 removes `module_call` from `chore_declarative_item`; authors who want to invoke a module function from a chore body MUST use the explicit `>>` (`inline_lua_line`) or `>>{` (`inline_lua_block`) form, or move the call to a top-level `register_block` (§{decl.register}) or a top-level `module_call` (§{toplevel.module-call}).

## 7.3. Default-interactive shell semantics [#chores.default-interactive]

Every `shell_command` step inside a chore body MUST be treated as if it were `@`-prefixed (§{steps.shell}, "interactive_command"). The `@` marker remains valid (no-op) for symmetry with recipe bodies, but is unnecessary. A chore therefore inherits the TTY-allocation, stdio-inheritance semantics of an interactive recipe step.

The Lua-step phases (`>`/`>{`/`>>`/`>>{`) are unchanged by this rule. An execute-phase Lua step inside a chore body runs on the chore's drain thread (the controlling terminal), not on a worker VM — chore bodies share a single drain across all imperative-region steps including Lua bodies (CS-0051). The body-bundling rule (§{exec.body-bundling}) does NOT apply to chore bodies.

A chore body MUST execute as a **single drain** (CS-0051): once the first imperative-region step of the body begins, no work unit of any other recipe MAY run until the body completes — successfully or with a failure. The body's shell steps and execute-phase Lua steps run in source order on the controlling terminal under one shared drain window. The interactive-drain rule of §{exec.interactive-drain} continues to govern individual interactive units in non-chore contexts (recipe-internal `@` steps); only the chore-body case shares a single drain across multiple imperative-region steps.

### Example 7.3.1

```cook
chore deploy
    cd build
    rsync -av out/ user@host:/srv/app/
    > print("deploy complete")
```

The two shell steps are interactive: `cd build` and `rsync ...` run with the user's TTY inherited. The `>` line runs at execute phase on the drain thread (controlling terminal), not on a worker VM (CS-0051). There is no body-bundling, and the two shell steps and the `>` Lua line execute under one shared drain — no work unit of any other recipe interleaves with them (CS-0051).

## 7.4. No caching [#chores.no-caching]

A chore body MUST NOT register a cached work unit. This is satisfied trivially by the parse-time ban on `cook` and `plate` steps (§{chores.body}); a chore has no declarative cook step that would produce a cacheable output. The Cook Lua API helpers `cook.add_unit` (§{lua.add-unit}) and the unit-recording helpers, if called from an `inline_lua_line` or `inline_lua_block` inside a chore body, MUST register their units with `cache = false`. A conforming implementation MUST raise a Lua runtime error from `inline_lua` inside a chore body if the call would record a unit with `cache = true` (or any equivalent caching opt-in).

This rule preserves the chore-vs-recipe semantic distinction: chores always run when invoked; they never short-circuit on a cache hit.

### Note 7.4.1

The engine does not need a chore-specific code path. A chore compiles to a recipe-body Lua function whose registered units are all `cache = false` and whose shell steps default to `interactive = true`. The distinction is a parse-time / codegen-time layer over the existing engine.

## 7.5. Cross-form dependencies [#chores.cross-form-deps]

A chore MAY depend on a recipe (run a build, then post-process) and a recipe MAY depend on a chore (rare but legal — for example, a recipe that depends on a one-off database-seed chore). Dependency resolution and ordering follow the rules of §{xref}; the form of the depender and depended-upon does not affect the resolution algorithm.

### Example 7.5.1

```cook
recipe build
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" using { cc -c $<in> -o $<out> }
    cook "build/app" using { cc $<all> -o $<out> }

chore play: build
    ./build/app

recipe install: play
    cp build/app /usr/local/bin/
```

`play` depends on the recipe `build`; the recipe `install` depends on the chore `play`. The graph: `build` (recipe, cached) → `play` (chore, never cached) → `install` (recipe, cached). Each edge is a normal name reference; no special wording for cross-form.
