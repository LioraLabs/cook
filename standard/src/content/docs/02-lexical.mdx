# 2. Lexical structure [#lexical]
> **Normative.** This chapter defines how a Cookfile's text is broken into tokens. The productions in §{grammar} and Appendix A consume the token classes defined here.

## 2.1. Source representation [#lexical.source-representation]
A Cookfile is a sequence of Unicode characters encoded as UTF-8. A conforming implementation MUST accept any valid UTF-8 input and MUST NOT require a byte-order mark. The behaviour on input that is not valid UTF-8 is unspecified.

Lexical analysis is **line-oriented**: the source is divided into lines by the newline character `U+000A` (`\n`), and each line is classified independently as a single token (§{lexical.tokens}). A conforming implementation MUST accept `\r\n` as a line terminator by treating the carriage return as part of the preceding line's trailing whitespace, which is trimmed before classification (§{lexical.line-classification}). The final line of the source need not be terminated by `\n`.

Leading and trailing whitespace on a line are not significant to token classification, with the following exceptions already stated in App. A: column-zero positioning distinguishes an `implicit_header` from a `shell_command` (App. A.3), and leading whitespace is preserved inside brace-balanced block bodies (§{lexical.brace-blocks}) because the body is reassembled from the raw source lines. In all other cases, a line is trimmed of leading and trailing whitespace before its prefix is tested (§{lexical.line-prefixes}, §{lexical.line-classification}).

### Note 2.1.1

The reference tokenizer in `cli/crates/cook-lang/src/lexer.rs` iterates `source.lines()`, which is defined by Rust to split on `\n` and strip a trailing `\r`. The observable effect matches the rule above: `\r\n` and `\n` produce the same token stream.

## 2.2. Tokens [#lexical.tokens]
The Cookfile lexer produces **exactly one token per source line**. A token carries its 1-indexed line number; downstream phases (§{grammar}, §{recipes}) use that number for diagnostics and for locating multi-line block bodies (§{lexical.brace-blocks}). There is no concept of "multiple tokens on one line" at the Cookfile layer — the interior structure of a line (for example, the output patterns of a `cook` step) is parsed at the syntactic layer on top of the single `Content` token.

The token classes produced by the lexer are:

| Token class            | Produced for                                                                 | Defined in |
|---|---|---|
| `Comment`              | Line whose first non-whitespace character is `#`                              | §{lexical.comments}      |
| `RecipeHeader`         | Explicit `recipe` line or column-zero `BARE_IDENTIFIER ":" …`                 | §{grammar.recipe-syntax}      |
| `ConfigHeader`         | Line `config` or `config` + name                                              | §{grammar.config-blocks}      |
| `UseDecl`              | Line `use` + name                                                             | §{grammar.use-declarations}      |
| `ImportDecl`           | Line `import` + `BARE_IDENTIFIER` + path                                      | §{grammar.import-declarations}      |
| `RecipeEnd`            | Line whose only non-whitespace content is `end`                               | §{grammar.recipe-syntax}      |
| `LuaLine`              | Line whose first non-whitespace characters are `>` (but not `>{`)             | §{lexical.line-prefixes}, §{recipes.lua-steps} |
| `LuaBlockOpen`         | Line whose first non-whitespace characters are `>{`                           | §{lexical.line-prefixes}, §{lexical.brace-blocks} |
| `Blank`                | Line containing only whitespace                                               | §{lexical.line-classification}     |
| `Content`              | Any other line; its classification within a recipe body is deferred to §{grammar.step-dispatch}  | §{grammar.step-dispatch}      |

A conforming implementation MAY represent these classes with any internal data structure. The token names above are used descriptively throughout this Standard; they do not mandate a specific variant name in an implementation.

### Note 2.2.1

`Content` is deliberately broad: the lexer does not distinguish `ingredients "…"`, `cook "…" using …`, `@vim notes.txt`, or `gcc -o main main.c` — all four produce `Content` tokens whose string form is the line after trimming. The step-dispatch cascade in App. A.4 resolves which step kind applies.

## 2.3. Identifiers [#lexical.identifiers]
A `BARE_IDENTIFIER` is a sequence of ASCII characters matching the regular expression in App. A.5:

```ebnf
BARE_IDENTIFIER ::= /[A-Za-z_][A-Za-z0-9_.\-]*/
```

The first character MUST be an ASCII letter (`A`–`Z`, `a`–`z`) or underscore (`_`); each subsequent character MUST be an ASCII letter, ASCII digit, underscore, hyphen (`-`), or dot (`.`). A conforming implementation MUST reject any other character. Unicode letters and digits are NOT part of `BARE_IDENTIFIER`.

`BARE_IDENTIFIER` is used in three syntactic roles, each with its own supplementary constraints documented where that role is defined:

| Role                                   | Supplementary constraint                                                      |
|---|---|
| Import alias (`import_declaration`)    | MUST be unique across imports within the Cookfile (App. A.2).                 |
| Recipe / dependency name               | Final dot-separated segment MUST NOT be a reserved segment (§{lexical.keywords}, App. A.2). |
| First segment of a `module_call`       | MUST NOT contain `.` or `-` (App. A.4).                                       |

Where the grammar admits either a `BARE_IDENTIFIER` or a double-quoted `STRING` (for example, a recipe name or a dependency), the two forms are equivalent in meaning; the quoted form exists to admit names that would not otherwise match `BARE_IDENTIFIER`.

### Example 2.3.1

```cook
recipe backend.build: lib setup
    gcc -o lib.o lib.c
end
```

`backend.build` is a valid `BARE_IDENTIFIER` (dot is an interior character). `lib` and `setup` are valid `BARE_IDENTIFIER` dependency names. The final segment `build` is not a reserved recipe segment (§{lexical.keywords}), so the header is accepted.

## 2.4. Keywords [#lexical.keywords]
The Cookfile language reserves seven identifiers contextually, each in a single role: the final dot-separated segment of a recipe name. Outside that role the identifiers are ordinary tokens.

| Reserved recipe segment | Role where reserved             |
|---|---|
| `stem`                 | Final segment of a recipe name   |
| `name`                 | Final segment of a recipe name   |
| `ext`                  | Final segment of a recipe name   |
| `dir`                  | Final segment of a recipe name   |
| `in`                   | Final segment of a recipe name   |
| `out`                  | Final segment of a recipe name   |
| `all`                  | Final segment of a recipe name   |

A conforming implementation MUST reject a recipe whose name's final dot-separated segment is a reserved recipe segment (App. A.2). A reserved recipe segment MAY appear as a non-final segment — for example, `backend.build` is valid because `build` is the final segment, whereas `backend.stem` is rejected because `stem` is the final segment.

The keywords that introduce other top-level or recipe-body productions — `recipe`, `config`, `end`, `ingredients`, `cook`, `plate`, `using`, `use`, `import`, `test` — are recognised as keywords only when followed by a keyword separator (space, tab, or `"`). A `Content` line whose first word is one of these keywords followed directly by other identifier characters (for example, `cooking`, `testify`) is a `shell_command`. The dispatch rule is normative in App. A.4.

### Note 2.4.1

The separator test is the governing subtlety. See the test `test_recipe_prefix_is_shell_command` in `cli/crates/cook-lang/src/lexer.rs`, which confirms that `recipes_cleanup` produces a `Content` token and not a `RecipeHeader`.

## 2.5. Strings [#lexical.strings]
A `STRING` token is a double-quoted sequence of characters defined in App. A.5:

```ebnf
STRING ::= /"[^"]*"/
```

A `STRING` opens with `U+0022` (`"`), contains zero or more characters none of which is `"`, and closes with `U+0022` (`"`). A conforming implementation MUST NOT interpret backslash escapes inside a `STRING`: the character sequence is taken literally. A conforming implementation MUST reject a `STRING` that is not terminated on the same line — `STRING` tokens do not span lines.

The empty string `""` is a valid `STRING`. A double quote inside a `STRING` is not expressible in the current grammar; authors needing such content can use the inline-block forms of §{lexical.brace-blocks}, whose interior language (Lua or shell) has its own quoting rules.

`STRING` appears in recipe and dependency names, in `ingredients`, `cook`, `plate`, and `test` steps, in the optional name of a `config_block` header, and as the single-string form of `using_clause`. In each of those roles, a conforming implementation parses the `STRING` inside the single line that produced the governing token.

### Example 2.5.1

```cook
recipe "build"
    ingredients "src/*.c" !"src/legacy.c"
    cook "bin/app" using "gcc {all} -o {out}"
end
```

Each `"…"` is a `STRING`. The `!"src/legacy.c"` form is an `ingredient` exclusion, not a distinct string class (App. A.4): it is the literal `!` followed by an ordinary `STRING`.

### Note 2.5.1

The reference implementation of the single-quoted-string parser is in `cli/crates/cook-lang/src/cook_line.rs`, function `parse_single_quoted_string`. The lack of escape processing is observable: the test `test_cook_step_shell` asserts that a `STRING` value containing `{in}` and `{out}` is delivered verbatim to the step.

## 2.6. Comments [#lexical.comments]
A **comment** is a line whose first non-whitespace character is `#`. The comment extends to the end of the line. A conforming implementation MUST produce a `Comment` token for such a line and MUST NOT treat the `#` character as beginning a comment in any other lexical position.

In particular, a `#` that appears after the first non-whitespace character of a line is part of that line's content and is not stripped by the lexer. Downstream phases (notably the shell runtime) MAY assign their own meaning to in-line `#` characters; that meaning is not part of Cookfile lexical syntax.

A `Comment` token is accepted anywhere a `toplevel_item` (App. A.1) or a `recipe_item` (App. A.3) is admitted, and is discarded at the syntactic layer. Comments are permitted inside `config_body` (App. A.2) and inside brace-balanced blocks (§{lexical.brace-blocks}), where they are preserved verbatim in the collected block body because their removal is the interior language's concern, not the Cookfile lexer's.

### Example 2.6.1

```cook
# top-level comment: discarded at the syntactic layer

recipe "build"
    # indented comment: also a Comment token
    gcc -o main main.c   # not a comment — part of the shell command
end
```

The first two `#` lines produce `Comment` tokens. The `# not a comment` sequence on the third source line is part of the `Content` token for the `gcc` command; its treatment is up to the shell runtime (§{xref}).

## 2.7. Line prefixes [#lexical.line-prefixes]
A **line prefix** is a character sequence that, when it appears as the first non-whitespace characters of a Cookfile line, selects a token classification other than `Content`. The Cookfile language recognises four lexical line prefixes:

| Prefix | Token produced            | Phase    | Defined in |
|---|---|---|---|
| `>>{`  | `InlineLuaBlockOpen`      | register | §{lexical.brace-blocks}, §{recipes.lua-steps} |
| `>>`   | `InlineLuaLine`           | register | §{recipes.lua-steps} |
| `>{`   | `LuaBlockOpen`            | execute  | §{lexical.brace-blocks}, §{recipes.lua-steps} |
| `>`    | `LuaLine`                 | execute  | §{recipes.lua-steps} |

A conforming implementation MUST test these prefixes in **longest-prefix order**: `>>{` before `>>` before `>{` before `>`. Without this ordering, a `>>{` would be consumed as `>` followed by `>{`, and a `>>` would be consumed as `>` followed by `>`. A conforming implementation MUST apply these prefixes only after leading whitespace has been trimmed from the line; thus `    >>{` opens an inline Lua block identically to `>>{` at column zero.

A run of three or more `>` characters at the start of a trimmed line — `>>>`, `>>>>`, etc. — is reserved and MUST be rejected with a parse error. The reservation keeps space for future prefixes without ambiguity.

The phase column above is informative; the normative phase rule is in §{recipes.lua-steps} and §{exec.two-phase}. The single-`>` family is execute-phase; the doubled-`>` family is register-phase. See §{rationale.lua-prefix-split} for why the surface is split this way.

The `@` character is not a lexical prefix. A line whose first non-whitespace character is `@` produces a `Content` token whose string form includes the leading `@`. Its classification as `interactive_command` is made at step-dispatch time (§{grammar.step-dispatch}, §{recipes.shell-steps}) and only inside a recipe body. Outside a recipe body, such a line remains a `Content` token, which the syntactic layer rejects because `Content` tokens are not valid at the top level (§{grammar.top-level-ordering}).

### Example 2.7.1

```cook
recipe build
    >> local std = cook.env.CXX_STD or "c++20"
    >>{
        cook.env.CXXFLAGS = "-std=" .. std
    }
    > print("build complete")
    >{
        if not fs.exists("build/app") then error("missing binary") end
    }
    @vim .cook/notes.txt
    gcc -o main main.c
end
```

The `>>` line produces an `InlineLuaLine` token (register-phase). The `>>{` line produces an `InlineLuaBlockOpen` token (register-phase). The `>` line produces a `LuaLine` token (execute-phase). The `>{` line produces a `LuaBlockOpen` token (execute-phase). The `@vim …` and `gcc …` lines both produce `Content` tokens; the former dispatches to `interactive_command` in §{recipes.shell-steps}, the latter to `shell_command`.

### Note 2.7.1

A `LuaLine` body MAY be empty: the line `>` alone produces a `LuaLine` whose body is the empty string. The same applies to `InlineLuaLine`: the line `>>` alone produces an `InlineLuaLine` whose body is the empty string. See §{recipes.lua-steps}.

## 2.8. Numbers [#lexical.numbers]
A `NUMBER` token is a non-empty sequence of ASCII digits defined in App. A.5:

```ebnf
NUMBER ::= /[0-9]+/
```

`NUMBER` has a single use in the Cookfile grammar: the `timeout` operand of a `test_step` (App. A.4). The `NUMBER` is parsed as an unsigned integer denoting a whole number of seconds. A conforming implementation MUST reject a `test` step whose `timeout` operand is not a valid `NUMBER`. Signs, decimal points, fractional digits, and unit suffixes are NOT permitted.

No other step kind accepts a bare `NUMBER`; digits that appear inside a shell command, a `STRING`, or a Lua block are the interior language's concern and are not Cookfile tokens.

### Example 2.8.1

```cook
recipe "run-tests"
    test "./{out}" timeout 30 should_fail
end
```

`30` is a `NUMBER` denoting a 30-second timeout.

### Note 2.8.1

The reference implementation is `parse_test_timeout` in `cli/crates/cook-lang/src/cook_line.rs`. The test `test_test_step_with_timeout_and_should_fail` demonstrates the admitted operand ordering: a `NUMBER` directly after `timeout`, followed by the optional `should_fail` marker.

## 2.9. Brace-balanced blocks [#lexical.brace-blocks]
A **brace-balanced block** is a span of source text delimited by `{` and a matching `}`, where matching is defined by a single-pass brace-counting algorithm that ignores braces appearing inside certain syntactic contexts. Both the `LUA_BLOCK_CONTENT` (inside `>{ … }`) and the `SHELL_BLOCK_CONTENT` (inside `{ … }` after `using`) are brace-balanced blocks; they differ only in how their collected body is normalised after the block closes.

The opening brace is consumed on the line that introduced the block (a `LuaBlockOpen` token for `>{`, or the `{` immediately after `using` for `shell_block`), establishing initial depth `1`. Each subsequent source line is examined character by character; `{` increments depth and `}` decrements depth, except that braces are ignored when they appear:

- inside a Lua line comment, beginning at `--` and extending to the end of the line;
- inside a double-quoted string `"…"`, with `\`-escapes honoured;
- inside a single-quoted string `'…'`, with `\`-escapes honoured;
- inside a Lua long string `[[ … ]]`.

A conforming implementation MUST apply this exclusion consistently for both Lua blocks and shell blocks, so that a block's closing `}` is never mistaken for a literal `}` inside string or comment content. The block closes on the first line where the running depth returns to zero or below after the line's brace deltas have been applied; that closing line is consumed by the lexer but is NOT included in the block body.

The `LUA_BLOCK_CONTENT` body is the concatenation of the interior lines in source order, joined by newlines, preserving their original whitespace. The `SHELL_BLOCK_CONTENT` body is further normalised per App. A.5: each interior line is trimmed of leading and trailing whitespace, and lines that are empty after trimming are discarded; the result is an ordered sequence of command strings, not a single raw string.

A conforming implementation MUST reject a block whose depth never returns to zero by end of input.

### Example 2.9.1

```cook
recipe "build"
    >{
        if mode == "release" then
            local t = {1, 2, 3}   -- a literal `}` in a comment: }
            local s = "a { brace in a string }"
            print(s)
        end
    }
    cook "out.js" using {
        wasm-pack build
        cp out/main.js out.js
    }
end
```

The `>{ … }` block's body includes the `if`/`end` structure and the nested table `{1, 2, 3}`; the braces inside the string `"a { brace in a string }"` and after the `--` comment are ignored by the brace counter. The `using { … }` block's body, after line normalisation, is the two-element command list `["wasm-pack build", "cp out/main.js out.js"]`.

### Note 2.9.1

The governing examples are the tests `test_lua_block_nested_braces`, `test_lua_block_brace_in_string`, and `test_lua_block_brace_in_comment` in `cli/crates/cook-lang/src/tests.rs`, and the tests `collects_three_lines`, `drops_blank_lines`, and `respects_nested_braces_in_content` in `cli/crates/cook-lang/src/shell_block.rs`.

## 2.10. Line classification cascade [#lexical.line-classification]
For each source line, a conforming implementation MUST classify the line into exactly one token by applying the following tests in order, with the first matching test winning. The line's trimmed form (leading and trailing whitespace removed) is the subject of each test except where noted.

A **keyword separator** in tests 6–8 below is a space (`U+0020`), a tab (`U+0009`), or a double quote (`U+0022`). The double quote is admitted so that authors can write `recipe "foo"` with no space between the keyword and the opening quote of a quoted name; the quote is not consumed by the keyword test and remains available to the `STRING` parser (§{lexical.strings}). Test 9 (`import`) admits only space or tab: no quoted form exists for the `import` alias (App. A.2).

1. If the trimmed line is empty, the token is `Blank`.
2. If the trimmed line's first character is `#`, the token is `Comment` (§{lexical.comments}).
3. If the trimmed line begins with `>>{`, the token is `InlineLuaBlockOpen` (§{lexical.line-prefixes}, §{lexical.brace-blocks}, §{recipes.lua-steps}).
4. If the trimmed line begins with `>>` (but not `>>{`, by test 3), the token is `InlineLuaLine` (§{lexical.line-prefixes}, §{recipes.lua-steps}).
5. If the trimmed line begins with `>{`, the token is `LuaBlockOpen` (§{lexical.line-prefixes}, §{lexical.brace-blocks}, §{recipes.lua-steps}).
6. If the trimmed line begins with `>` (but not `>>` by test 4 or `>{` by test 5), the token is `LuaLine` (§{lexical.line-prefixes}, §{recipes.lua-steps}).
7. If the trimmed line is exactly `end`, the token is `RecipeEnd` (§{grammar.recipe-syntax}).
8. If the trimmed line begins with the keyword `recipe` followed by a keyword separator, the token is `RecipeHeader` (§{grammar.recipe-syntax}).
9. If the trimmed line is exactly `config`, or begins with `config` followed by a keyword separator, the token is `ConfigHeader` (§{grammar.config-blocks}).
10. If the trimmed line begins with `use` followed by a keyword separator, the token is `UseDecl` (§{grammar.use-declarations}).
11. If the trimmed line begins with `import` followed by a whitespace separator (space or tab only; not `"`), the token is `ImportDecl` (§{grammar.import-declarations}).
12. If the original (untrimmed) line has no leading whitespace — it starts at column 0 — and its trimmed form is `BARE_IDENTIFIER ":"` optionally followed by a dependency list, the token is `RecipeHeader` via the implicit-header form (App. A.3).
13. Otherwise, the token is `Content`. Its further classification as `ingredients_step`, `cook_step`, `plate_step`, `test_step`, `module_call`, `interactive_command`, or `shell_command` is the concern of the step-dispatch cascade in App. A.4 and is made only inside a recipe body.

The ordering is significant. In particular, the four Lua-prefix tests (3–6) MUST appear in longest-prefix order — `>>{` before `>>` before `>{` before `>` — so that a longer prefix is never consumed as a shorter one followed by a literal character (§{lexical.line-prefixes}). Tests 8–11 MUST require a separator so that identifiers whose first word merely starts with a keyword (for example, `recipes_cleanup`, `configure`, `useful`, `important`) fall through to test 13 and are not misclassified as declarations.

A trimmed line beginning with three or more consecutive `>` characters (`>>>`, `>>>>`, etc.) does not match any of tests 3–6 and is reserved (§{lexical.line-prefixes}). A conforming implementation MUST reject such a line with a parse error rather than fall through to test 13.

Tests 1–11 and 13 apply to lines whether or not they are indented; only test 12 (implicit recipe header) is sensitive to column-0 positioning, by the constraint in App. A.3.

### Example 2.10.1

```cook
use "cpp"                 # test 10: UseDecl
config debug              # test 9:  ConfigHeader
    env.CC = "gcc"
end

build: lib setup          # test 12: implicit-header RecipeHeader
    # test 2: Comment
    ingredients "src/*.c" # test 13: Content (step-dispatched to ingredients_step)
    >>{                   # test 3:  InlineLuaBlockOpen (register-phase)
        cook.env.CFLAGS = "-Wall"
    }
    >> local n = 4        # test 4:  InlineLuaLine (register-phase)
    >{                    # test 5:  LuaBlockOpen (execute-phase)
        print("hi")
    }
    > os.exit(0)          # test 6:  LuaLine (execute-phase)
    recipes_cleanup       # test 13: Content (falls through test 8 — no separator)
end
```
