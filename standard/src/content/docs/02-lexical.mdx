# 2. Lexical structure

> **Normative.** This chapter defines how a Cookfile's text is broken into tokens. The productions in § 3 and Appendix A consume the token classes defined here.

## 2.1. Source representation

A Cookfile is a sequence of Unicode characters encoded as UTF-8. A conforming implementation MUST accept any valid UTF-8 input and MUST NOT require a byte-order mark. The behaviour on input that is not valid UTF-8 is unspecified.

Lexical analysis is **line-oriented**: the source is divided into lines by the newline character `U+000A` (`\n`), and each line is classified independently as a single token (§ 2.2). A conforming implementation MUST accept `\r\n` as a line terminator by treating the carriage return as part of the preceding line's trailing whitespace, which is trimmed before classification (§ 2.10). The final line of the source need not be terminated by `\n`.

Leading and trailing whitespace on a line are not significant to token classification, with the following exceptions already stated in App. A: column-zero positioning distinguishes an `implicit_header` from a `shell_command` (App. A.3), and leading whitespace is preserved inside brace-balanced block bodies (§ 2.9) because the body is reassembled from the raw source lines. In all other cases, a line is trimmed of leading and trailing whitespace before its prefix is tested (§ 2.7, § 2.10).

### Note 2.1.1

The reference tokenizer in `cli/crates/cook-lang/src/lexer.rs` iterates `source.lines()`, which is defined by Rust to split on `\n` and strip a trailing `\r`. The observable effect matches the rule above: `\r\n` and `\n` produce the same token stream.

## 2.2. Tokens

The Cookfile lexer produces **exactly one token per source line**. A token carries its 1-indexed line number; downstream phases (§ 3, § 4) use that number for diagnostics and for locating multi-line block bodies (§ 2.9). There is no concept of "multiple tokens on one line" at the Cookfile layer — the interior structure of a line (for example, the output patterns of a `cook` step) is parsed at the syntactic layer on top of the single `Content` token.

The token classes produced by the lexer are:

| Token class            | Produced for                                                                 | Defined in |
|---|---|---|
| `Comment`              | Line whose first non-whitespace character is `#`                              | § 2.6      |
| `RecipeHeader`         | Explicit `recipe` line or column-zero `BARE_IDENTIFIER ":" …`                 | § 3.7      |
| `ConfigHeader`         | Line `config` or `config` + name                                              | § 3.6      |
| `VarDecl`              | Top-level `BARE_IDENTIFIER STRING`                                            | § 3.3      |
| `UseDecl`              | Line `use` + name                                                             | § 3.4      |
| `ImportDecl`           | Line `import` + `BARE_IDENTIFIER` + path                                      | § 3.5      |
| `RecipeEnd`            | Line whose only non-whitespace content is `end`                               | § 3.7      |
| `LuaLine`              | Line whose first non-whitespace characters are `>` (but not `>{`)             | § 2.7, § 4.9 |
| `LuaBlockOpen`         | Line whose first non-whitespace characters are `>{`                           | § 2.7, § 2.9 |
| `Blank`                | Line containing only whitespace                                               | § 2.10     |
| `Content`              | Any other line; its classification within a recipe body is deferred to § 3.8  | § 3.8      |

A conforming implementation MAY represent these classes with any internal data structure. The token names above are used descriptively throughout this Standard; they do not mandate a specific variant name in an implementation.

### Note 2.2.1

`Content` is deliberately broad: the lexer does not distinguish `ingredients "…"`, `cook "…" using …`, `@vim notes.txt`, or `gcc -o main main.c` — all four produce `Content` tokens whose string form is the line after trimming. The step-dispatch cascade in App. A.4 resolves which step kind applies.

## 2.3. Identifiers

A `BARE_IDENTIFIER` is a sequence of ASCII characters matching the regular expression in App. A.5:

```ebnf
BARE_IDENTIFIER ::= /[A-Za-z_][A-Za-z0-9_.\-]*/
```

The first character MUST be an ASCII letter (`A`–`Z`, `a`–`z`) or underscore (`_`); each subsequent character MUST be an ASCII letter, ASCII digit, underscore, hyphen (`-`), or dot (`.`). A conforming implementation MUST reject any other character. Unicode letters and digits are NOT part of `BARE_IDENTIFIER`.

`BARE_IDENTIFIER` is used in four syntactic roles, each with its own supplementary constraints documented where that role is defined:

| Role                                   | Supplementary constraint                                                      |
|---|---|
| Variable name (`variable_declaration`) | MUST NOT be any of the reserved keywords listed in § 2.4 (App. A.2).          |
| Import alias (`import_declaration`)    | MUST be unique across imports within the Cookfile (App. A.2).                 |
| Recipe / dependency name               | Final dot-separated segment MUST NOT be a reserved segment (§ 2.4, App. A.2). |
| First segment of a `module_call`       | MUST NOT contain `.` or `-` (App. A.4).                                       |

Where the grammar admits either a `BARE_IDENTIFIER` or a double-quoted `STRING` (for example, a recipe name or a dependency), the two forms are equivalent in meaning; the quoted form exists to admit names that would not otherwise match `BARE_IDENTIFIER`.

### Example 2.3.1

```cook
recipe backend.build: lib setup
    gcc -o lib.o lib.c
end
```

`backend.build` is a valid `BARE_IDENTIFIER` (dot is an interior character). `lib` and `setup` are valid `BARE_IDENTIFIER` dependency names. The final segment `build` is not a reserved recipe segment (§ 2.4), so the header is accepted.

## 2.4. Keywords

Two disjoint reserved-word sets exist in the Cookfile language: **blocking keywords** (which MUST NOT be used as `variable_declaration` left-hand sides) and **reserved recipe segments** (which MUST NOT appear as the final dot-separated segment of a recipe or dependency name). Reservation is **contextual**: each keyword is reserved only in the role below, not as a general identifier.

| Blocking keyword | Role where reserved           |
|---|---|
| `recipe`        | Variable name                  |
| `config`        | Variable name                  |
| `end`           | Variable name                  |
| `ingredients`   | Variable name                  |
| `cook`          | Variable name                  |
| `plate`         | Variable name                  |
| `using`         | Variable name                  |
| `use`           | Variable name                  |
| `import`        | Variable name                  |
| `test`          | Variable name                  |

| Reserved recipe segment | Role where reserved             |
|---|---|
| `stem`                 | Final segment of a recipe name   |
| `name`                 | Final segment of a recipe name   |
| `ext`                  | Final segment of a recipe name   |
| `dir`                  | Final segment of a recipe name   |
| `in`                   | Final segment of a recipe name   |
| `out`                  | Final segment of a recipe name   |
| `all`                  | Final segment of a recipe name   |

A conforming implementation MUST reject a `variable_declaration` whose left-hand side is one of the blocking keywords (App. A.2). A conforming implementation MUST reject a recipe whose name's final dot-separated segment is a reserved recipe segment (App. A.2). A reserved recipe segment MAY appear as a non-final segment — for example, `backend.build` is valid because `build` is the final segment, whereas `backend.stem` is rejected because `stem` is the final segment.

Outside these two roles, the identifiers above are ordinary tokens. In particular, a `Content` line whose first word is `cook`, `plate`, `test`, or `ingredients` is dispatched by App. A.4 to the corresponding step kind only when followed by a separator; a `Content` line whose first word is one of these keywords followed directly by other identifier characters (for example, `cooking`, `testify`) is a `shell_command`.

### Note 2.4.1

The separator test is the governing subtlety. See the test `test_recipe_prefix_is_shell_command` in `cli/crates/cook-lang/src/lexer.rs`, which confirms that `recipes_cleanup` produces a `Content` token and not a `RecipeHeader`.

## 2.5. Strings

A `STRING` token is a double-quoted sequence of characters defined in App. A.5:

```ebnf
STRING ::= /"[^"]*"/
```

A `STRING` opens with `U+0022` (`"`), contains zero or more characters none of which is `"`, and closes with `U+0022` (`"`). A conforming implementation MUST NOT interpret backslash escapes inside a `STRING`: the character sequence is taken literally. A conforming implementation MUST reject a `STRING` that is not terminated on the same line — `STRING` tokens do not span lines.

The empty string `""` is a valid `STRING`. A double quote inside a `STRING` is not expressible in the current grammar; authors needing such content can use the inline-block forms of § 2.9, whose interior language (Lua or shell) has its own quoting rules.

`STRING` appears in top-level `variable_declaration` (the value), in recipe and dependency names, in `ingredients`, `cook`, `plate`, and `test` steps, and as the single-string form of `using_clause`. In each of those roles, a conforming implementation parses the `STRING` inside the single line that produced the governing `Content` token.

### Example 2.5.1

```cook
CFLAGS "-Wall -Wextra -I/usr/local/include"

recipe "build"
    ingredients "src/*.c" !"src/legacy.c"
    cook "bin/app" using "gcc {all} -o {out}"
end
```

Each `"…"` is a `STRING`. The `!"src/legacy.c"` form is an `ingredient` exclusion, not a distinct string class (App. A.4): it is the literal `!` followed by an ordinary `STRING`.

### Note 2.5.1

The reference implementation of the single-quoted-string parser is in `cli/crates/cook-lang/src/cook_line.rs`, function `parse_single_quoted_string`. The lack of escape processing is observable: the test `test_cook_step_shell` asserts that a `STRING` value containing `{in}` and `{out}` is delivered verbatim to the step.

## 2.6. Comments

A **comment** is a line whose first non-whitespace character is `#`. The comment extends to the end of the line. A conforming implementation MUST produce a `Comment` token for such a line and MUST NOT treat the `#` character as beginning a comment in any other lexical position.

In particular, a `#` that appears after the first non-whitespace character of a line is part of that line's content and is not stripped by the lexer. Downstream phases (notably the shell runtime) MAY assign their own meaning to in-line `#` characters; that meaning is not part of Cookfile lexical syntax.

A `Comment` token is accepted anywhere a `toplevel_item` (App. A.1) or a `recipe_item` (App. A.3) is admitted, and is discarded at the syntactic layer. Comments are permitted inside `config_body` (App. A.2) and inside brace-balanced blocks (§ 2.9), where they are preserved verbatim in the collected block body because their removal is the interior language's concern, not the Cookfile lexer's.

### Example 2.6.1

```cook
# top-level comment: discarded at the syntactic layer

recipe "build"
    # indented comment: also a Comment token
    gcc -o main main.c   # not a comment — part of the shell command
end
```

The first two `#` lines produce `Comment` tokens. The `# not a comment` sequence on the third source line is part of the `Content` token for the `gcc` command; its treatment is up to the shell runtime (§ 5).

## 2.7. Line prefixes

A **line prefix** is a character sequence that, when it appears as the first non-whitespace characters of a Cookfile line, selects a token classification other than `Content`. The Cookfile language recognises two lexical line prefixes:

| Prefix | Token produced | Defined in |
|---|---|---|
| `>{`   | `LuaBlockOpen` | § 2.9 |
| `>`    | `LuaLine`      | § 4.9 |

A conforming implementation MUST test for `>{` before `>`, so that a Lua-block prefix is never consumed as a Lua-line prefix followed by a literal `{`. A conforming implementation MUST apply these prefixes only after leading whitespace has been trimmed from the line; thus `    >{` opens a Lua block identically to `>{` at column zero.

The `@` character is not a lexical prefix. A line whose first non-whitespace character is `@` produces a `Content` token whose string form includes the leading `@`. Its classification as `interactive_command` is made at step-dispatch time (§ 3.8, § 4.10) and only inside a recipe body. Outside a recipe body, such a line remains a `Content` token, which the syntactic layer rejects because `Content` tokens are not valid at the top level (§ 3.2).

### Example 2.7.1

```cook
recipe build
    > print("registering build")
    >{
        print("multi-line")
    }
    @vim .cook/notes.txt
    gcc -o main main.c
end
```

The `>` line produces a `LuaLine` token. The `>{` line produces a `LuaBlockOpen` token that begins a brace-balanced block (§ 2.9) closing at the matching `}`. The `@vim …` and `gcc …` lines both produce `Content` tokens; the former dispatches to `interactive_command` in § 4.10, the latter to `shell_command`.

### Note 2.7.1

A `LuaLine` body MAY be empty: the line `>` alone produces a `LuaLine` whose body is the empty string. See § 4.9.

## 2.8. Numbers

A `NUMBER` token is a non-empty sequence of ASCII digits defined in App. A.5:

```ebnf
NUMBER ::= /[0-9]+/
```

`NUMBER` has a single use in the Cookfile grammar: the `timeout` operand of a `test_step` (App. A.4). The `NUMBER` is parsed as an unsigned integer denoting a whole number of seconds. A conforming implementation MUST reject a `test` step whose `timeout` operand is not a valid `NUMBER`. Signs, decimal points, fractional digits, and unit suffixes are NOT permitted.

No other step kind accepts a bare `NUMBER`; digits that appear inside a shell command, a `STRING`, or a Lua block are the interior language's concern and are not Cookfile tokens.

### Example 2.8.1

```cook
recipe "run-tests"
    test "./{out}" timeout 30 should_fail
end
```

`30` is a `NUMBER` denoting a 30-second timeout.

### Note 2.8.1

The reference implementation is `parse_test_timeout` in `cli/crates/cook-lang/src/cook_line.rs`. The test `test_test_step_with_timeout_and_should_fail` demonstrates the admitted operand ordering: a `NUMBER` directly after `timeout`, followed by the optional `should_fail` marker.

## 2.9. Brace-balanced blocks

A **brace-balanced block** is a span of source text delimited by `{` and a matching `}`, where matching is defined by a single-pass brace-counting algorithm that ignores braces appearing inside certain syntactic contexts. Both the `LUA_BLOCK_CONTENT` (inside `>{ … }`) and the `SHELL_BLOCK_CONTENT` (inside `{ … }` after `using`) are brace-balanced blocks; they differ only in how their collected body is normalised after the block closes.

The opening brace is consumed on the line that introduced the block (a `LuaBlockOpen` token for `>{`, or the `{` immediately after `using` for `shell_block`), establishing initial depth `1`. Each subsequent source line is examined character by character; `{` increments depth and `}` decrements depth, except that braces are ignored when they appear:

- inside a Lua line comment, beginning at `--` and extending to the end of the line;
- inside a double-quoted string `"…"`, with `\`-escapes honoured;
- inside a single-quoted string `'…'`, with `\`-escapes honoured;
- inside a Lua long string `[[ … ]]`.

A conforming implementation MUST apply this exclusion consistently for both Lua blocks and shell blocks, so that a block's closing `}` is never mistaken for a literal `}` inside string or comment content. The block closes on the first line where the running depth returns to zero or below after the line's brace deltas have been applied; that closing line is consumed by the lexer but is NOT included in the block body.

The `LUA_BLOCK_CONTENT` body is the concatenation of the interior lines in source order, joined by newlines, preserving their original whitespace. The `SHELL_BLOCK_CONTENT` body is further normalised per App. A.5: each interior line is trimmed of leading and trailing whitespace, and lines that are empty after trimming are discarded; the result is an ordered sequence of command strings, not a single raw string.

A conforming implementation MUST reject a block whose depth never returns to zero by end of input.

### Example 2.9.1

```cook
recipe "build"
    >{
        if mode == "release" then
            local t = {1, 2, 3}   -- a literal `}` in a comment: }
            local s = "a { brace in a string }"
            print(s)
        end
    }
    cook "out.js" using {
        wasm-pack build
        cp out/main.js out.js
    }
end
```

The `>{ … }` block's body includes the `if`/`end` structure and the nested table `{1, 2, 3}`; the braces inside the string `"a { brace in a string }"` and after the `--` comment are ignored by the brace counter. The `using { … }` block's body, after line normalisation, is the two-element command list `["wasm-pack build", "cp out/main.js out.js"]`.

### Note 2.9.1

The governing examples are the tests `test_lua_block_nested_braces`, `test_lua_block_brace_in_string`, and `test_lua_block_brace_in_comment` in `cli/crates/cook-lang/src/tests.rs`, and the tests `collects_three_lines`, `drops_blank_lines`, and `respects_nested_braces_in_content` in `cli/crates/cook-lang/src/shell_block.rs`.

## 2.10. Line classification cascade

For each source line, a conforming implementation MUST classify the line into exactly one token by applying the following tests in order, with the first matching test winning. The line's trimmed form (leading and trailing whitespace removed) is the subject of each test except where noted.

A **keyword separator** in tests 6–8 below is a space (`U+0020`), a tab (`U+0009`), or a double quote (`U+0022`). The double quote is admitted so that authors can write `recipe "foo"` with no space between the keyword and the opening quote of a quoted name; the quote is not consumed by the keyword test and remains available to the `STRING` parser (§ 2.5). Test 9 (`import`) admits only space or tab: no quoted form exists for the `import` alias (App. A.2).

1. If the trimmed line is empty, the token is `Blank`.
2. If the trimmed line's first character is `#`, the token is `Comment` (§ 2.6).
3. If the trimmed line begins with `>{`, the token is `LuaBlockOpen` (§ 2.7, § 2.9).
4. If the trimmed line begins with `>` (but not `>{`, by test 3), the token is `LuaLine` (§ 2.7, § 4.9).
5. If the trimmed line is exactly `end`, the token is `RecipeEnd` (§ 3.7).
6. If the trimmed line begins with the keyword `recipe` followed by a keyword separator, the token is `RecipeHeader` (§ 3.7).
7. If the trimmed line is exactly `config`, or begins with `config` followed by a keyword separator, the token is `ConfigHeader` (§ 3.6).
8. If the trimmed line begins with `use` followed by a keyword separator, the token is `UseDecl` (§ 3.4).
9. If the trimmed line begins with `import` followed by a whitespace separator (space or tab only; not `"`), the token is `ImportDecl` (§ 3.5).
10. If the original (untrimmed) line has no leading whitespace — it starts at column 0 — and its trimmed form is `BARE_IDENTIFIER ":"` optionally followed by a dependency list, the token is `RecipeHeader` via the implicit-header form (App. A.3).
11. Otherwise, if the trimmed line is `BARE_IDENTIFIER STRING` and the `BARE_IDENTIFIER` is not one of the blocking keywords of § 2.4, the token is `VarDecl`. This test is applied both at column 0 and on indented lines; the syntactic layer rejects a `VarDecl` that appears after the first recipe (App. A.2).
12. Otherwise, the token is `Content`. Its further classification as `ingredients_step`, `cook_step`, `plate_step`, `test_step`, `module_call`, `interactive_command`, or `shell_command` is the concern of the step-dispatch cascade in App. A.4 and is made only inside a recipe body.

The ordering is significant. In particular, test 3 (`>{`) MUST precede test 4 (`>`) so that a Lua-block prefix is not consumed as a Lua-line prefix followed by a literal `{` (§ 2.7); tests 6–9 MUST require a separator so that identifiers whose first word merely starts with a keyword (for example, `recipes_cleanup`, `configure`, `useful`, `important`) fall through to test 11 or 12 and are not misclassified as declarations.

Tests 1–9 and 11–12 apply to lines whether or not they are indented; only test 10 (implicit recipe header) is sensitive to column-0 positioning, by the constraint in App. A.3.

### Example 2.10.1

```cook
CC "gcc"                  # test 11: VarDecl
use "cpp"                 # test 8:  UseDecl
config debug              # test 7:  ConfigHeader
    env.CC = CC
end

build: lib setup          # test 10: implicit-header RecipeHeader
    # test 2: Comment
    ingredients "src/*.c" # test 12: Content (step-dispatched to ingredients_step)
    >{                    # test 3:  LuaBlockOpen
        print("hi")
    }
    recipes_cleanup       # test 12: Content (falls through test 6 — no separator)
end
```

### Note 2.10.1

Test 11 admits `VarDecl` on indented lines, but the syntactic layer (App. A.2) rejects any `VarDecl` that follows a recipe header. The test `test_var_after_recipe_is_shell_command` in `cli/crates/cook-lang/src/tests.rs` demonstrates the user-visible consequence: a line of the form `NAME "value"` inside a recipe body is re-interpreted as a `shell_command`, so that a `VarDecl` token emitted at test 11 is effectively reclassified by the step layer when it appears inside a recipe.
