# Appendix C. Rationale (informative)

> **Informative.** This appendix records the reasoning behind design decisions in the Standard. It is non-binding.

## C.0. On §{intro} Introduction [#rationale.intro]

### C.0.1. Versioning posture pre-1.0 [#rationale.versioning-pre-1-0]
The §{intro.version} model — `MAJOR.MINOR` pre-1.0, batched cuts, convention-only conformance claim, git-tagged corpus — embeds four design decisions worth recording.

**Why `MAJOR.MINOR` not strict SemVer pre-1.0.** Breaking changes are normal pre-1.0; the discipline of distinguishing MAJOR from MINOR is performative when every cut may contain a breakage. PATCH buys nothing — there is no installed base whose conformance is broken by a typo fix that nobody has to back-port. Strict SemVer arrives at 1.0 along with an installed base for which the discipline starts to matter.

**Why version cuts MAY batch CSes.** Authoring (one CS per merge) and publication (one cut per N CSes) are different concerns. Batching keeps the cut cadence at "when an implementation needs to claim a new version" rather than "after every spec edit," which would inflate the version number into the dozens with little reader benefit.

**Why the conformance-claim mechanism is convention, not a normative requirement.** No consumer pre-1.0; §{intro.non-scope} excludes the CLI surface; no automated check enforces a MUST without an ecosystem to enforce against. The convention is documented in `CONTRIBUTING.md`, where it is amendable without a CS.

**Why the corpus is identified by git tag rather than versioned subdirectory.** Git already snapshots prose and corpus together at the tag. Versioned subdirectories duplicate that snapshot in a place where it must be manually maintained; they also imply a back-port story that pre-1.0 explicitly disclaims.

## C.3. On §{lexical} Lexical structure [#rationale.lexical]
Chapter 2 is deliberately small. Cookfiles are tokenized line by line, each line classifies into exactly one token, and the step-dispatch cascade in App. A.4 handles every syntactic distinction the lexer declines to make. This pushes complexity out of the lexer and into a single prose cascade (§{lexical.line-classification}) that mirrors the implementation's `if`/`else if` ladder. The rationale entries below record the places where a more sophisticated lexical design was considered and rejected; the unadorned sections are either mechanical (§{lexical.source-representation}, §{lexical.comments}, §{lexical.numbers}) or fall out of constraints explained elsewhere (§{lexical.strings} follows from App. A.5, §{lexical.brace-blocks} follows from the embedded-Lua design in §{lua}).

### C.3.2. Why one token per line [#rationale.one-token-per-line]
The Cookfile grammar is strictly line-oriented: every source line produces exactly one token carrying that line's 1-indexed number. The alternative — a conventional multi-token-per-line lexer — was rejected on three grounds.

First, error locality. Nearly every diagnostic in the Rust parser reports a 1-indexed line number, and authors of Cookfiles edit them in line-oriented tools (editors, diffs, code review). A one-token-per-line scheme makes "line N" unambiguous; a multi-token scheme would need column offsets and a column-aware diagnostic renderer for the same quality of error.

Second, block locality. Brace-balanced blocks (§{lexical.brace-blocks}) are collected by reading subsequent **source lines** and counting braces, not by reading subsequent tokens. A line-oriented lexer composes cleanly with that design: the block-collector asks the line iterator for the next N raw lines and the lexer asks for a single classification per line, and neither layer needs to hand the other a partial line.

Third, authoring model. Cookfile authors think in steps, not in expressions, and one step per line is the idiom every existing example reinforces. There is no authoring convenience the multi-token model would unlock.

The cost is that a `Content` token must be re-examined by the step-dispatch cascade (App. A.4) to distinguish `ingredients`, `cook`, `test`, module calls, and — in a chore body — shell commands (CS-0134 removed the `@` interactive-command kind and rejects loose shell in recipe bodies). That re-examination is a small, linear, keyword-prefix test — cheaper than a general tokenizer would be and easier to reason about.

### C.3.3. Why `BARE_IDENTIFIER` is ASCII-only [#rationale.ascii-identifiers]
The `BARE_IDENTIFIER` character class in App. A.5 is ASCII — no Unicode letters, no Unicode digits, no confusables. The reason is interoperability with the shell.

Most `BARE_IDENTIFIER` values end up as filesystem-visible artefacts: recipe names become command-line arguments to the `cook` CLI, and module-call first-segments correspond to on-disk module roots. ASCII-only identifiers guarantee those names survive copy-paste between a Cookfile, a terminal, a CI log, a URL, and a filesystem without encoding surprises. Unicode identifiers would force the implementation to take a stance on normalisation, case-folding, and confusable-detection, none of which is justified by the expected author population.

The cost is paid by authors working in non-Latin scripts, who can still use a `STRING`-form recipe name (App. A.5 allows `name ::= BARE_IDENTIFIER | STRING`) at the price of double quoting. The escape hatch keeps the simple case simple and charges the complexity only to the users who actually need it.

### C.3.4. Why keywords are reserved contextually, not globally [#rationale.contextual-keywords]
The Cookfile language defines step keywords (`recipe`, `config`, `cook`, `ingredients`, `using`, `use`, `import`, `end`, `test`) and reserved recipe segments (`stem`, `name`, `ext`, `dir`, `in`, `out`); neither set is reserved globally. A recipe named `backend.build` is accepted because `build` is not on the reserved-segment list; `backend.stem` is rejected because `stem` is. A `Content` line whose first word is `recipes_cleanup` or `configure` is dispatched as a `shell_command`, not as a declaration, because the step-keyword rule requires a separator after the keyword.

Global reservation would have been simpler to specify but would have broken the most common shell idioms. Cookfiles frequently call tools named `configure`, `cook-book`, `tester`, `using-*` — forbidding those as shell commands to protect a handful of language keywords would be a poor trade. The contextual rule instead places the reservation only where it is needed: the final segment of a recipe name, where the `$<stem>`-style substitution rule would otherwise collide with a literal segment of the same name.

The cost is that the step-dispatch cascade in App. A.4 must insist on a separator after each step keyword (`recipe`, `config`, `cook`, `ingredients`, `using`, `use`, `import`, `end`, `test`) so that a `Content` line whose first word is `recipes_cleanup` or `configure` is not misclassified as a declaration. That cost is borne by the Standard and the implementation, not by the author.

### C.3.7. Why the `@` interactive prefix was removed [#rationale.at-not-lexical]
A natural reading of pre-CS-0134 Cookfiles suggested that `@` was a third line prefix alongside `>` and `>{`, marking a shell command as interactive. The Standard never defined it as a lexical prefix — and CS-0134 removed it from the language altogether.

Two facts made `@` redundant. First, a recipe body is now purely declarative (§{rationale.recipe-body-regions}): it holds no shell steps at all, interactive or otherwise, so it has no use for an interactivity marker. Second, every chore-body `shell_command` is interactive by the chore default (§{chores.default-interactive}) — the `@` marker was a no-op on the only construct that still admitted shell commands. A marker that is illegal in one body kind and a no-op in the other carries no information, so CS-0134 deleted it: a line beginning with `@` is now invalid syntax everywhere.

Keeping `@` would also have forced a context-sensitive lexer: `@cmd` at the top level is a parse error while `@cmd` in a body was an interactive step, a distinction a line-oriented, context-free lexer cannot make. Removing the marker outright is simpler than either a context-aware lexer or a syntactic-layer strip, and costs the author nothing — interactivity is now implied by the chore form itself.

## C.4. On §{toplevel.overview} Syntactic grammar [#rationale.grammar]
Chapter 3 walks through a grammar that is almost entirely keyword-led at the outer layer. Most of the design decisions in §{toplevel.overview} are mechanical consequences of the lexical structure defined in §{lexical}: each production is selected by a single leading token, and no backtracking is required. The rationale entries below address the places where §{toplevel.overview} imposes a constraint that goes beyond "pick the production the leading token names" — places where the grammar is deliberately narrower than a permissive reading would allow.

### C.4.2. Why the top-level prefix is ordered [#rationale.ordered-prefix]
Cookfiles allow declarations of four kinds — variables, `use`, `import`, and `config` — to precede recipes, and the Standard requires that they form a contiguous prefix: once the first recipe has been parsed, no further declaration of any of the four kinds is accepted.

The ordering is not necessary for parsing — a context-free grammar that accepted interleaved declarations and recipes would parse unambiguously. It is a readability and configuration-layering choice. Config blocks and variable declarations are meant to be read as a header that frames the recipes that follow; scattering them between recipes defeats that reading, and permits confusing patterns where a `use` resolved by one recipe is not yet in scope for a sibling recipe that was parsed earlier in the same file. Requiring the prefix to be contiguous removes the question entirely: every declaration is in scope for every recipe.

The rule is trivially removable if it ever proves onerous; nothing downstream depends on it. But while it is in place, the diagnostic is actionable — the parser names the offending item's line — and authors get a clean separation between file-level configuration and recipe definitions.

### C.4.9. Config blocks as the sole variable surface [#rationale.config-only-variables]
A previous revision of the Standard admitted a top-level `variable_declaration` form (`NAME "value"`) that fed values into `cook.env` independently of any `config_block`. This amendment removes that form. The remaining surface is the `config_block` Lua body, which writes `cook.env` (and any other Cook API state) through ordinary Lua assignment.

Three forces motivate the consolidation. First, named overlay configs cannot override values set outside any config block without ad-hoc precedence wording; folding the surface into config blocks gives a single layered model with well-defined override semantics (§{decl.config-composition}). Second, the bare top-level form is sugar for one specific case of what config block bodies already express; removing it deletes a duplicate surface, the `Token::VarDecl` lexical class, the contextual blocking-keyword reservation, and the recipe-body reclassification rule that the form required. Third, the consolidation moves the choice of *when* a value is set (always, vs. only-when-an-overlay-is-active) into the same surface as the value itself, which keeps the source of truth localised.

Layered single-select was chosen over multi-select for this revision: at most one overlay can be active at a time. Multi-select would force the Standard to specify ordering across multiple selected overlays and to define conflict resolution for keys written by more than one of them. Single-select avoids both questions and is non-breaking to relax in a future revision if real use cases warrant it.

The selection mechanism for the active overlay is implementation-defined. Config selection is a CLI affordance, and the Standard's general posture (§{intro}) is to specify Cookfile-language behaviour rather than tool invocation surfaces. Implementations are free to expose a flag, an environment variable, a file-level default, or any combination; the validation rule for the selected name is normatively specified in §{decl.config-composition}.

### C.4.10. Implicit termination of recipe and config bodies [#rationale.implicit-end]

Each `recipe` and `config` block in v0.3 closed with an explicit `end` line. The closing keyword was ceremony — every author wrote it without thinking, and the parser had no need for it that the next top-level construct or EOF could not serve. A Cookfile with ten recipes had ten lines that existed only to satisfy the grammar. CS-0019 / v0.4 drops the keyword and replaces it with the implicit-termination rule of §{toplevel.termination}.

Three forces motivate the change. First, the brace-block-as-opaque-source-lines model already cleanly separates Lua- and shell-block boundaries from recipe boundaries: a `recipe` keyword appearing inside a Lua block's source is never seen by the recipe-body parser, because the brace-counter skips the token stream over the block (§{lexical.brace-blocks}). Implicit termination by next top-level keyword does not interfere with embedded blocks. Second, the lexer's column-0 enforcement on top-level keywords (§{lexical.keywords}) prevents an indented `recipe foo` line inside a recipe body from silently terminating the outer recipe — the column-0 requirement is the exact discriminator the parser needs. Third, the diagnostic surface improves: the prior "you forgot to close this recipe" diagnostic is replaced by a more specific diagnostic when an embedded keyword token violates region rules, since there is no missing `end` to forget.

The cost is that a deliberately empty recipe at the head of a Cookfile (no body, no immediately-following next-recipe header) is now indistinguishable from "header followed by the rest of the file as body until EOF" — but that ambiguity does not actually exist, because EOF is a valid terminator. The recipe `recipe empty\n` followed by EOF is a valid empty recipe.

A bare `end` line is a `Content` token (§{lexical.line-classification}). At top level it is rejected by §{toplevel.overview}. Inside a recipe body it dispatches as a `shell_command` running the literal command `end`, which the user's shell will interpret per its own rules. This is a deliberate non-special-case: there is no `end` keyword in the recipe-body cascade; the line gets the same treatment as any other Content line.

Migration: in this commit, every spec example (chapters 2 through 8 and App. C) drops its trailing `end` lines so the spec consistently teaches v0.4 surface form. The conformance-corpus migration (every fixture under `standard/conformance/`) and the project-Cookfile migration (root, `cli/`, `standard/`, `tree-sitter-cook/`, `examples/`) follow in subsequent commits of the same v0.4 cut. The Standard cuts at v0.4 with `cs-standard/v0.4` tagged at the merge commit per the parse-format-altering rule in CONTRIBUTING.md.

## C.6. On §{recipes} Recipes and step kinds [#rationale.recipes]
Chapter 4 defines the largest single surface area of the Cookfile language: the step kinds available inside a recipe body and the dispatch rules that separate them. The design is pragmatic rather than minimalist. The Cook authors had to balance three competing forces: (a) make the common cases (a shell command, a one-to-one build, an interactive prompt) trivially expressible; (b) keep the lexer and parser context-free and line-oriented so that editor tooling remains simple; (c) allow Lua to extend the language without requiring it to do so for basic use. The sections below record the non-obvious choices that follow from those forces.

### C.6.6. Why `cook "a" "b" using "cmd"` is a parse error [#rationale.multi-output-using-error]
The single-string `using` form is convenient in the one-output case because the command template can reference `$<out>` and the runtime knows exactly what to substitute. In the multi-output case that determinism evaporates: a single `"cmd"` template has no way to describe which of several outputs is produced by which sub-command, nor which sub-command produces any at all.

The Standard could have papered over this by, for example, running the same `"cmd"` multiple times with `$<out>` bound in turn to each output. That semantics is surprising — the user typically wants a single invocation of a generator that happens to produce several artifacts, not N invocations. It would also hide common bugs, because a `"cmd"` that produces only the first output would silently "succeed" on the others.

Requiring a block body — either `{ ... }` or `>{ ... }` — forces the author to state exactly how the outputs are produced: either as a literal sequence of shell commands (whose joint effect produces all outputs) or as a Lua block (which has full access to the `outputs` array and can write each entry deliberately). The parser's rejection of the ambiguous single-string form is therefore normative (§{grammar-appendix.steps}, "Multi-output cook-step rule") and produces a diagnostic that names the block-body remedy directly.

### C.6.8. Why `>` is execute-phase and `>>` was removed [#rationale.lua-prefix-split]
The Cookfile language once carried two Lua-prefix surfaces: the single-`>` family (`>`, `>{ … }`) for execute-phase Lua, and a doubled-`>` family (`>>`, `>>{ … }`) for register-phase Lua. CS-0134 **removes the doubled-`>` family from the language entirely** and, with it, the standalone single-`>` step from recipe bodies. The single-`>` syntax survives only as a **block-body opener** — the `>{ … }` body of a `cook` / `test` / `probe` step, and standalone `>` / `>{ … }` steps inside a **chore** body. This entry records why the split existed and why it was retired.

The teachable meaning of the single-`>` form is unchanged: it means "execute-phase Lua." A reader who encounters `> print("hi")` in a chore body, or a `cook "out" >{ … }` body, expects the code to run when the work runs, not when Cook plans it. That intuition aligns with the rest of execute — a chore `shell_command` runs at execute, and so does a `>` line.

The doubled prefix existed to reverse the timing for register-phase orchestration, where the Lua body's job is to call Cook Lua API helpers (`cook.add_unit`, `cook.export`, `cook.step_group`) that record work for execute. That role has not disappeared, but its surface has: CS-0134 routes register-phase Lua through a **bare module-call line** (§{rationale.module-call-register}) inside a recipe body, and through a top-level `register` block (§{decl.register}) or top-level `module_call` (§{toplevel.module-call}) everywhere else. A dedicated `>>` sigil for the same effect was redundant visual weight once module calls carry the register-phase default; retiring it collapses two ways of writing register-phase Lua into one and removes the "how many arrows?" question from the language.

The cook-body Lua block (`cook STR >{ … }`) keeps the single-`>` prefix. The `>{ … }` syntax means "execute-phase Lua chunk" wherever it appears; the cook-step body position adds bindings (`input`, `output`, `inputs`, `outputs`, `input_N` per §{lua.using-block-globals}) but does not change the phase. A `>>{ … }` body is not merely rejected in that position — the `>>{` prefix no longer exists (§{lexical.line-prefixes}, Note 3.7.1).

### C.6.9. Why a recipe-body `module_call` is register-phase [#rationale.module-call-register]
A bare `ns.fn(...)` line in a recipe body is auto-classified as a **register-phase** `module_call` (§{steps.dispatch}, rule 7). CS-0134 **reinstates** this classification, reversing the CS-0072 removal that this entry originally justified: for two revisions authors had to spell register-phase module calls with an explicit `>>` prefix, and CS-0134's removal of `>>` from the language made a bare, register-phase module-call line the natural home again. The alternative — treating a recipe-body module call as execute-phase — was considered and rejected.

Module functions in the wild record work: they call `cook.add_unit`, `cook.step_group`, `cook.export`, and similar register-only helpers. A function like `cpp.static_library("widget", { … })` exists to enumerate sources, register one compile unit per source, and record a final archive unit. None of that is meaningful at execute time — the DAG is already built, and `cook.add_unit` raises by §{rationale.execute-phase-api}. Defaulting the bare call to register-phase fits the overwhelmingly common case without ceremony.

A user who wants to do execute-phase work via a module function does so where execute-phase Lua now lives: a `>{ … }` block body of a `cook` / `test` step, or a `>` / `>{ … }` step inside a chore body. There the module function's body runs at execute with the execute-phase API surface (§{rationale.execute-phase-api}). The split is now carried by **body position** rather than by an arrow count: register-phase at the recipe-body top level, execute-phase inside a block body or chore.

### C.6.10. Why the recipe body is purely declarative [#rationale.recipe-body-regions]
A `recipe_body` is **purely declarative** (§{recipes.region-rule}): it carries only register-phase items — `ingredients` / `cook` / `test` steps and bare `module_call` lines — and its execute-phase work lives exclusively inside the block body of a `cook` or `test` step (`>{ … }` / `{ … }`). CS-0134 arrived at this shape after an intermediate design that partitioned the body into a declarative region followed by an imperative region, enforced by a parse-time out-of-order error. This entry records why the whole intra-body ordering apparatus was dropped in favour of body position.

The two earlier alternatives both failed. **Free interleaving** of register and execute steps was tried first, and produced two footguns. Body-unit diagnostics became fragile: a `> print(x)` after a `cook` step coalesced into one body unit, a later `cook` recorded another, a third `> print(x)` yet another, scattering a recipe's units so failures cited line numbers without context. And scope-sharing across the register/execute boundary became impossible to reason about — a user who wrote `>> local x = …` then `> print(x)` expected the local to print, but `x` was a register-phase Lua local invisible to the execute worker VM. The **region-ordering rule** fixed interleaving but kept both surfaces in the same body, so the reader still had to track a mid-body phase transition and the scope discontinuity that rode along with it.

CS-0134 removes the footgun at its root: the register/execute split now maps **1:1 to body position**, not to intra-body ordering. Register-phase work is the recipe-body top level (declarative); execute-phase work is a `cook` / `test` block body; and imperative command-runner work is a **chore** (§{chores}). There is no in-body ordering rule to violate because there is no imperative region in a recipe body, and there is no cross-boundary scope surprise because the two phases no longer share one body. A reader sees a recipe body that is uniformly a plan, and a block body that is uniformly the doing.

The cost is that a recipe author who wants ad-hoc execute-phase glue reaches for a `cook`/`test` block body or a chore rather than dropping a loose `>`/shell line into the recipe. The tradeoff is deliberate: a recipe body that mixed register and execute work in alternating slabs was, almost always, a plan and a chore pretending to be one recipe.

### C.6.11. Why recipe execute-work is now one unit per block body [#rationale.body-bundling-per-recipe]
Because a recipe body is purely declarative (§{rationale.recipe-body-regions}), a recipe's execute-phase work comes exclusively from the block body of a `cook` or `test` step, and **each block body is its own work unit** (§{exec.body-bundling} is vacuous under CS-0134). This entry records how that shape settled a question the intermediate design fought over — how loose recipe-body imperative steps should have been grouped into units.

The intermediate design admitted standalone `>` / `>{ … }` / shell / `@` steps in a recipe body's imperative region and had to decide how a run of them bundled. **Per-line units** — one unit per step — were rejected because they made "Make-`.ONESHELL`" semantics impossible: a `cd bin` followed by `./app` ran in two shell processes, the `cd` did not persist, and the most basic intuition for a sequence of body lines broke. **One unit per region** was rejected because an interactive step has draining semantics (§{exec.interactive-drain}) that cannot live inside a shared Lua chunk. The compromise — bundle consecutive steps until an interactive step interrupts — worked, but it was a subtle rule with a source-line-range progress label papering over the fact that one surface region produced several hidden units.

CS-0134 dissolves the question. There is no loose imperative region to bundle: the unit boundary is now **syntactically explicit** — it is the `{ … }` / `>{ … }` block the author wrote. A multi-command sequence that wants one-shell persistence goes inside one `{ … }` shell body (whose lines join into a single `set -e` shell process, §{steps.cook-single}); Lua glue that wants shared locals goes inside one `>{ … }` block. Interactive command-runner work moves to a **chore** (§{chores}), where every shell step is interactive by default and the single-drain rule (§{chores.default-interactive}) — not body bundling — governs execution. The user sees exactly as many units as blocks they wrote, and the progress UI needs no line-range heuristic.

### C.6.12. Why register-only Cook Lua API helpers raise from execute-phase Lua [#rationale.execute-phase-api]
`cook.add_unit`, `cook.exec`, `cook.interactive`, `cook.step_group`, and `cook.recipe` are register-only by §{lua.cook-sh}–§{lua.cook-recipe} and §{lua.add-unit}. A conforming implementation MUST raise a Lua runtime error when any of them is called from execute-phase Lua (a `lua_line`, a `lua_block`, or a `>{ … }` payload). The alternatives — silently doing nothing, returning a stub value, or re-recording a unit that runs after the DAG closed — were rejected.

The register/execute split makes a sharp claim: the DAG is fully known before execution begins. Allowing register-only helpers to "soft fail" inside an execute-phase chunk would erode that claim. A user who calls `cook.add_unit` inside a `>{ … }` body unit is, by the structure of the language, doing something incoherent: the unit being added would have no scheduling context. An error caught at the offending call is the right diagnostic — it identifies the function name and the calling step kind, and points the user at the corrective form (move the call to a bare `module.call()` line in the recipe body, a top-level `register` block, or a module function).

`cook.sh` is the one helper exposed in both phases (§{lua.cook-sh}). The user-visible behaviour — "run this command, give me its stdout, error on non-zero" — is the same in both contexts; only the surrounding scheduling differs. Promoting it to both phases keeps execute-phase Lua — a `cook`/`test` block body or a chore body — usable for shell-out work without inventing a parallel `cook.run`-style helper.

### C.6.13. Why module-call dispatch is a first-two-tokens heuristic [#rationale.module-call-heuristic]
The module-call pattern of §{grammar-appendix.steps} inspects only the first identifier, a literal `.`, and the first character after the dot. It does not attempt to parse the rest of the line as Lua. This is deliberate.

A full Lua parse at dispatch time would couple the Cook parser to the Lua grammar, making the Cook parser's correctness depend on a Lua front-end and making its diagnostics worse: a Lua syntax error inside a module-call line would have to be reported as a Cook error at a Cook source position. It would also require the Cook parser to make an early commit between "this is a shell command" and "this is Lua", and a wrong early commit in either direction produces terrible error messages.

The two-token heuristic is cheap, lexer-local, and has no false positives in practice. A line that looks like `identifier.identifier` and is not a valid Lua expression is virtually never a valid shell command either (hyphens in the first segment are excluded precisely to protect `cargo-build ...` and similar shell forms). When the heuristic matches, the Lua runtime is the right place to report a Lua syntax error: it has the full Lua grammar and a dedicated error reporter. When the heuristic does not match, the fall-through to `shell_command` is lossless — the line is passed verbatim to the shell, which will report its own error if the line is malformed there.

The multi-line extension (continuing across source lines while braces are unbalanced) is the same brace-balance algorithm used for `lua_block` (§{lexical.brace-blocks}). Reusing it keeps the lexer consistent: the Cookfile parser has exactly one way to recognise a brace-balanced region, and that region's body is always handed to Lua untouched.

### C.6.14. Why chore is a separate form rather than a flag on recipe [#rationale.chore-form-not-flag]

The chore-vs-recipe distinction could have been spelled `recipe foo (chore)`, `recipe foo @chore`, `recipe! foo`, or any number of recipe-with-modifier shapes. CS-0020 instead introduces `chore` as a peer top-level keyword. Three forces drive the choice.

First, the keyword is the carrier of intent. A reader who scans a Cookfile reads `recipe build` and `chore deploy` differently — the keyword tells the eye whether the body is a build target or a command runner. A modifier on `recipe` is read after the keyword, by which time the reader has already committed to "build target". The split keyword is a pure UX win for navigation.

Second, the parse-time step-kind ban is naturally expressed at the production level. A chore production has a smaller `chore_body` alternation than a recipe production has for `recipe_body`, and the ban falls out of the grammar rather than being a separate rule the parser checks after the fact. The implementation is less code than a `recipe (chore)` modifier whose validation runs after step-kind dispatch.

Third, the syntactic peerage models the semantic peerage. Chores and recipes are equally first-class — a recipe may depend on a chore, a chore on a recipe — and giving them peer keywords is an accurate model of that. A modifier shape would suggest one is a special case of the other.

The cost is a new keyword to remember. Cook's keyword surface remains small (`recipe`, `chore`, `config`, `use`, `import`, `ingredients`, `cook`, `test`, `using`); even with `chore` added, the set is well within "memorable in one sitting" range.

### C.6.15. Why chore bans `ingredients`, `cook`, `test` [#rationale.chore-banned-steps]

The three banned step kinds — `ingredients`, `cook`, `test` — share a common property: each declares part of a build-graph node (inputs, outputs, validation). A chore is by definition a body that does NOT participate in the build graph: it always runs, it produces no cacheable output, and its outputs are not consumed by other recipes via the cross-recipe-reference mechanism (§{xref}).

Allowing a `cook` step inside a chore body would be incoherent: either (a) the engine ignores the cook step's output declaration, in which case it serves no purpose, or (b) the engine treats it as a build target despite the chore's no-cache rule, in which case the rule is violated. The same argument applies to `ingredients` (declares inputs, but a chore has no cache to invalidate against) and `test` (validates a cook step's output, but no cook step).

Banning module calls was considered and rejected. A module call typically expands into a sequence of register-phase declarative steps, some of which might be `cook` steps and would be caught by the recipe-body region rule's existing diagnostics. A flat ban would be over-strict — a module that exposes a `cleanup()` function returning no `cook.add_unit` calls is a perfectly valid chore body call. The conservative choice is therefore to allow `module_call` in chore bodies; module authors who emit `cook` units inside a chore-context call see a diagnostic at the `cook.add_unit` site (per §{chores.no-caching}) rather than at parse time.

### C.6.16. Why default-interactive [#rationale.chore-default-interactive]

A chore is by intent a thing the user runs at a terminal. The `@` marker (§{steps.shell}) on every shell line of a recipe-shaped command-runner is ceremony — the author already chose `chore` over `recipe` because the body is interactive. The default-interactive rule deletes the ceremony.

The `@` marker is preserved as a no-op inside chore bodies for symmetry: a recipe body author migrating to `chore` does not need to strip every `@`, and a Cookfile that is moved between forms during refactoring continues to parse. The non-symmetric direction (chore → recipe) requires the author to add `@` markers where they want interactivity preserved; this is an authoring choice and is the expected path for "this used to be a chore but I want to add cook steps now".

### C.6.17. Why `test` is a cook step with no declared outputs [#rationale.plate-test-no-output-decl]
Once cook gained block bodies and three iteration modes (CS-0022), the only structural difference between a cook step and a test step was the presence of an output declaration. Modeling `test` as "cook with no outputs" makes the body grammar uniform, the placeholder vocabulary uniform, the cross-recipe substitution rules uniform, and the substitution timing uniform. The author's mental model stays a single body surface across `cook` and `test`. (CS-0135 removed the former `plate` step, which was the other "cook with no outputs" kind; its distinguishing traits — no declared output, un-sandboxed body — are exactly what a `chore` provides.)

The `test` keyword survives as its own step kind rather than folding into a generic "cook with no outputs" form because it signals author intent a reader relies on: scanning a recipe and seeing `test` says "this is a validation gate," and the `cook test` runner reports its verdict — a DAG-positioned pass/fail node that a bare output-less cook step would not name.

The `using` keyword does not appear on a `test` line. Cook once earned it because a cook line carried two things — output pattern list and body — that needed a separator (the keyword was itself later removed, CS-0099). A `test` line carries only a body, so no such separator was ever warranted.

### C.6.18. Why iteration mode is deduced from the body for test [#rationale.plate-test-body-deduction]
Cook owns iteration mode via the output pattern (CS-0022) because the output pattern is *declarative*: it states the unit's filenames and falls out the iteration shape. A `test` has no outputs to declare, so the body's placeholder content is the only available signal. The CS-0022 footgun — an output-pattern signal contradicting a body signal — cannot occur in a step that has no output pattern. Deducing mode from body content is therefore sound for `test`, even though it would be incoherent for cook.

The deduction is purely syntactic: it scans the body's text (shell) or its free-identifier references (Lua) at register time. It does not require Lua evaluation. The alternative — letting the runtime decide which binding to populate based on which one was named — would require dynamic mode selection, breaking the static-shape DAG `cook.add_unit` produces.

### C.6.19. Why `$<out>` is rejected in test bodies [#rationale.plate-test-out-rejected]
CS-0022 fixed `$<out>` to mean the unit's declared output. A `test` declares no outputs, so `$<out>` has no referent. An earlier (pre-CS-0024) surface used `$<out>` to mean the iteration item — but that is what `$<in>` means everywhere else in the language. A per-step-kind name for "the iteration item" would re-introduce the position-dependent rule that CS-0022 paid to remove.

### C.6.20. Why `$<lib.ACCESSOR>` is rejected in test bodies [#rationale.plate-test-lib-accessor-rejected]
A `test` has no output pattern and therefore cannot declare a `lib`-driven iteration. §{xref.dep-driven}'s firewall — `$<lib.ACCESSOR>` rejected in any cook-step body — applies trivially to a test body: there is never a position in such a body where `$<lib.ACCESSOR>` could be valid. A bare `$<lib>` reference is admitted and substitutes per §{xref.string-substitution}.

### C.6.21. Why a static Lua identifier scan, not a runtime decision [#rationale.plate-test-lua-static-scan]
A Lua body that references both `input` and `inputs` is making contradictory claims about iteration shape. Detecting that at register time produces a sharp diagnostic with a line number; deferring it to "the runtime decides which binding is bound based on which one was named" would either require dynamic mode selection (incompatible with the static-shape DAG) or two distinct mode-resolution passes (one syntactic for shell, one runtime for Lua). One pass, applied uniformly, is the better trade.

The scan is whole-word, identifier-position. A property-name suffix (`cook.input`, `obj:input(...)`) is not a reference to the binding because the Lua grammar resolves the name as a field access. The scan respects the existing string/comment/long-string ignore rules of §{lexical.brace-blocks}.

### C.6.22. Why `as STRING` and not `name STRING` [#rationale.test-as-not-name]
`as` reads as natural English ("this test, as 'parses-config'") and aligns with SQL's `AS` aliasing posture. `name` would collide with the table-field `name` in `cook.add_test` (§{lua.cook-add-test}), creating two ways to spell the same concept with subtly different timing — the modifier substitutes at register time, whereas the `name` field is constructed when the table is assembled at execute time. Choosing distinct surface tokens for the two surfaces makes the timing difference legible.

### C.6.23. Why the modifier order is fixed [#rationale.test-modifier-order-fixed]
LL(1) parsability is a hard constraint for the Cook parser (`cook-lang`'s recursive-descent shape). Free-order modifiers either require backtracking or a peek/look-ahead that the parser does not currently support. The fixed order also produces canonical reading order in source: identity (`as`) → constraint (`timeout`) → interpretation (`should_fail`). Authors who reach for a different order are flagged immediately rather than silently parsed differently.

### C.6.24. Why test caching is universal in the Standard [#rationale.test-caching-universal]
The alternative was caching as an implementation choice with no Standard contract. That posture works for v0.x but creates a portability hazard at v1.0: a Cookfile that passes on cloud-cache machine A and fails on machine B — because A cached a test and B did not — is a contract violation only if the Standard says so. Specifying the cache key inputs and the recordable-outcomes rule (§{exec.cache.test-unit}) means cross-machine cache replay is correct or it is a Standard violation. Without a contract, the cloud-cache feature degrades to "works on machines configured the same way," which is the wrong promise for a build-system test runner.

### C.6.25. Why no per-step `volatile` modifier [#rationale.test-no-volatile]
A `volatile` modifier teaches authors that hidden inputs are normal. The right answer for tests with filesystem inputs Cook cannot see (vitest reading `.test.ts`, pytest reading `.py`) is the existing `discovered_inputs` mechanism (§{exec.cache.discovered-inputs}) plus depfile emission. The right answer for tests with non-filesystem hidden inputs (network, clock, `$RANDOM`) is the operator-side `cook --test --rerun [PATTERN]` flag, applied per-invocation. A persistent per-step opt-out would convert "I know this is non-deterministic" into "I'm too lazy to declare my inputs," and the cache becomes a sieve.

The CLI escape hatch costs nothing in language surface and pushes authors toward correct input declaration; the language escape hatch costs spec breadth and grants permission for sloppiness. The trade is in favour of the CLI form.

### C.6.26. Why timed-out + `should_fail` is "timed out", not "passed" [#rationale.test-timeout-not-passed]
A timeout is not an exit status; it is a runtime intervention. Treating it as failure-equivalent for the purpose of `should_fail` would mean any sufficiently slow test passes any `should_fail` claim, which destroys the modifier's diagnostic value. The dedicated "timed out" outcome (§{exec.cache.test-unit}) preserves the runner's ability to flag timeouts distinctly in the report.

### C.6.27. Why the member index is spelled `[in]`, and why bare `$<NAME>` was not narrowed [#rationale.member-index-in]
The v0.14 spelling `$<NAME[]>` (CS-0098) was shorthand for a keyed-join design — `$<NAME[k]>`, "the producer's output with the same key" — whose key surface was deferred with cross-probe joins. With keys gone from v1.0, the empty brackets indexed by nothing: line noise that suggests an array where there is none. CS-0137 respells the accessor `$<NAME[in]>`: the index names what actually fills the slot — the current member binding, the same `in` the fan-out body already reads via `$<in>`. It reads as real indexing ("render at *this member*"), and it re-admits the key idea additively in 1.x (`$<NAME[in.FIELD]>`, `$<NAME[key]>`) — the cheap direction under the hard-cut philosophy, which is why every bracket content other than the literal `in` is reserved and rejected today.

Two alternatives for the bare form were considered and declined. *Inference* — making bare `$<NAME>` mean the per-member output whenever the consumer iterates the same probe as `NAME` — makes the reference's arity depend on the remote recipe's header: the meaning of `$<render>` stops being decidable at the reference site, and a refactor that changes `render`'s driver silently flips every consumer between per-member and all-outputs. *Rejection* — a hard error on bare `$<NAME>` inside a fan-out body that could have used `[in]` — was declined in favour of one uniform bare meaning: bare `$<NAME>` is all of `NAME`'s terminal outputs, space-joined, in every position (§{steps.cross-recipe-member}). Per-member is always explicit; all-outputs is always bare.

## C.13. On §{exec.phases} Execution model [#rationale.exec]
The execution model in §{exec.phases} is organised around a two-phase design: every recipe is evaluated once to discover its work (the register phase) and then arranged for execution (the execute phase). This split is not cosmetic. It is the mechanism by which the Standard turns a scripting-flavoured surface syntax — arbitrary Lua embedded in a recipe body — into a dependency graph that can be reasoned about, parallelised, and cached. The rationale subsections below record why several load-bearing decisions in §{exec.phases} are the way they are.

### C.13.1. Why the register phase is pure and non-blocking [#rationale.register-phase-pure]
The register phase evaluates each recipe's Lua body once, with capture-mode bindings for the Cook Lua API (§{exec.capture-mode}). The requirement that register-phase calls be pure with respect to the build state — no subprocesses against declared outputs, no output-file mutations, no blocking on external I/O — exists for three concrete reasons.

First, **determinism of the graph.** If the register phase could block on a subprocess whose exit depended on a file produced by another recipe's execute phase, the topology of the dependency graph would depend on the order in which the implementation chose to register recipes. Two conforming implementations would then produce different graphs for the same Cookfile. By confining register-phase effects to appending entries to a record, the graph becomes a deterministic function of the Cookfile source and a bounded set of pre-existing inputs.

Second, **bounded time.** The register phase runs on a single thread per recipe. If it could block, a long-running subprocess in one recipe's body would stall the discovery of every other recipe's work. Since the execute phase cannot begin until registration completes, this would reduce the whole build to serial execution even when the Cookfile would otherwise permit heavy parallelism.

Third, **cache reasoning.** The cache (§{exec.cache}) keys work units on their register-phase record. If that record could reflect the transient state of a subprocess, cache hits would become order-dependent: the same command, on the same inputs, could produce different cache keys depending on whether an unrelated sibling step had already run. Purity eliminates that class of confusion.

The narrowly-scoped exception allowed by the closing paragraph of §{exec.capture-mode} — implementations MAY permit register-phase effects whose only purpose is to feed the register-phase Lua control flow itself — is load-bearing for the `cook.sh` utility in §{lua}. That call's return value drives whether a `cook` step fires at all, and suppressing it would force authors to split dynamic build logic across two passes.

### C.13.2. Why chore bodies drain rather than serialise by default [#rationale.interactive-drain]
A chore body is implicitly interactive (§{chores.default-interactive}): §{exec.interactive-drain} requires it to drain all other running work before its commands begin. The alternative — allowing an interactive command to run concurrently with background work and only inherit the TTY — was rejected because it produces user-hostile behaviour in the common case. (Before CS-0134 this draining was requested per-line with an `@` prefix; CS-0134 removed the `@` marker and made every chore-body command drain by default, since chores exist precisely for terminal-sharing work.)

Chores are used for dev-loop commands like `./build/app`, `vim notes.md`, and `./serve`. These share the terminal with the human operator. If background work continues while such a command is in progress, its stdout and stderr interleave with the interactive session, corrupting the user's view and often stealing focus from whatever TUI the command is presenting. Users who saw this behaviour during prototyping reported it as a defect even when the concurrent output was mechanically correct.

Draining is not the same as serial execution of the whole build. Work units of other recipes before the chore body ran MAY have run in parallel; work units after it MAY again. The drain is local to the chore body's single-drain window (§{chores.default-interactive}). This preserves parallelism for build work while giving interactive work the quiet terminal users expect.

Authors who want concurrent background work alongside an interactive session can obtain it by running the background work in a separate recipe and not depending on the chore — the dependency graph continues to permit parallel execution of independent recipes under the usual rules.

### C.13.3. Why cache semantics are abstract rather than specified by hash algorithm [#rationale.abstract-cache]
Section 8.6 fixes the **set** of observable inputs a cache MAY consider — command text, input paths, output paths, environment, and optionally input-file content and output-file mtime — but it does not fix the hash function, the on-disk layout, or the policy for when cache entries are evicted. The boundary is deliberate.

**Cross-machine cache correctness.** §8.6's enumeration of permissible cache observables now includes machine identity and tool-binary content. The motivation is the shared/team cache: an artifact produced on one machine and served to another must remain correct against differences in compiler version, libc, locale, and the resolved binary that the command text names. A Cook implementation that hashes only "command text + input files" is correct on a single machine but unsafe across machines; the additional observables let an implementation be cross-machine-safe without taking on those observables when running locally. The denylist clause acknowledges that not every difference in the resolved environment affects build output (ambient session state, secrets unrelated to the artifact), and lets implementations filter without violating the minimum-observable enumeration of ¶2.

The set of observable inputs is a language-level property: a Cookfile author reasoning about why a step did or did not re-run needs to know what the implementation is allowed to consider. The hash function and storage format are engineering concerns that implementations should be free to evolve — a switch from one hash family to another, or from a flat file to an embedded database, changes nothing the author sees at the Cookfile level.

Specifying the hash function in the Standard would also lock every implementation into bit-compatibility with the reference, making caches transferable between implementations. That has never been a design goal. Caches are local artefacts; two implementations that both respect the observable-input rule produce equivalent cache decisions even if their on-disk caches are mutually unreadable.

The closing paragraph of §{exec.cache} — that caching is optional altogether — follows from the same principle. An implementation that treats every run as a clean build satisfies the dependency rules of §{exec.two-phase}–§{exec.interactive-drain} and therefore conforms. The Standard does not require caching; it regulates it when it is present.

### C.13.4. Why diagnostic ordering is normative but diagnostic wording is not [#rationale.diagnostic-ordering]
The ordering rule in §{exec.diagnostic-ordering} — syntactic diagnostics before semantic, first syntax error halts parsing — is normative because it determines what a Cookfile author sees on a single run. An implementation that reported a dependency cycle before reporting that a `cook` line was unterminated would mislead authors into debugging the graph when the real problem was a quote.

The wording is not normative because diagnostic quality is a product concern where implementations should be free to differentiate. A terse parser-level message suitable for a CI log and a richly-formatted message suitable for an editor overlay are both conforming so long as the ordering holds and the class of error is identifiable. This matches the convention established in §{intro.non-scope} and §{conf.criteria}: exact wording is out of scope; the diagnostic class is in scope.

### C.13.5. Why the work-unit DAG is unified across recipes [#rationale.dag-unification]

Earlier reference implementations of the register phase maintained two graphs: a coarse recipe-level dependency graph and a per-wave work-unit DAG. The recipe-level graph drove a wave-grouping pass that grouped recipes into synchronisation barriers; the per-wave DAG handled fine-grained unit ordering within each wave. Cross-recipe edges were plumbed through the recipe-level structure rather than living on the work-unit graph directly. This was an implementation accident, not a design property, and §{exec.dag-unification} retires it.

Two concrete costs motivated the unification.

First, **same-wave merging logic was hidden in scheduler code.** When `{dep}`-style implicit cross-recipe edges (§{exec.cross-recipe-ordering}) appeared in a Cookfile, the wave grouper had to merge the producing and consuming recipes into one wave so the per-wave DAG could see them as intra-wave edges. The merging rule was correct, but bespoke; nothing in the Cookfile language motivated its existence — it was scaffolding to bridge the recipe-level and unit-level graphs.

Second, **recipe boundaries leaked into scheduling.** Lifecycle events (`RecipeStarted`, `RecipeCompleted`) fired at wave boundaries rather than at the unit-state transitions that actually marked the start or end of a recipe's work. The reported timing did not match observable execution.

The unified DAG eliminates both. Cross-recipe edges become ordinary edges between work units; the executor's existing parallelism rules (§{exec.step-groups}) handle them directly. Recipe boundaries survive only as labels on regions of the DAG — useful for lifecycle reporting, not load-bearing for scheduling. Lifecycle events fire at first/last unit transitions per recipe, giving event timing that mirrors observable work.

The trade-off is a single ordering constraint at register time: `cook.dep_output(other_recipe)` is evaluated inside a recipe body and must observe the terminal outputs of `other_recipe`, which means `other_recipe`'s body must have been invoked first. A conforming implementation satisfies this by topologically sorting recipes by their declared `requires` before invoking bodies. That sort is a register-time sequencing helper, not a normative graph structure; §{exec.dag-unification} explicitly leaves the ordering implementation-defined.

### C.13.N. Name references as the sole implicit-dep source

CS-0009 removed the implicit cross-recipe dependency rule based on byte-equality between an ingredient path of one recipe and a cook-output path of another (old §{xref.string-substitution}). Under the new rule (§{exec.cross-recipe-ordering}), only name references (§{xref}) establish cross-recipe edges.

The motivation is conservatism about hidden coupling. The string-match rule coupled scheduling to filesystem path coincidence: a refactor that renamed `build/lib/libmath.a` to `build/artifact/libmath.a` would silently rewire the DAG, with no diagnostic pointing at the recipe boundary that shifted. Authors routinely write output paths that happen to match an unrelated recipe's ingredient, particularly in large workspaces where paths are constructed algorithmically. The edge can surprise.

The rule is also redundant once name references exist. Any dep the author meant to express by path equality can be expressed by `$<lib>` or `$<lib.ACCESSOR>` in the consuming recipe, or by `: lib` in the header. The former creates a fine-grained edge at the placeholder site (§{xref.dep-implications}) — strictly more precise than the path-match rule, which always produced a whole-recipe edge.

The small cost is a one-line migration for any existing Cookfile that relied on the rule: add `$<lib>` to the shell body, or add `: lib` to the header.

### C.13.O. Dep-driven iteration via output-pattern accessors

CS-0009 specifies that `$<lib.ACCESSOR>` appearing in a `cook` step's output pattern list is both a template substitution AND an iteration-source declaration: the step's work units are generated per item in `lib`'s output list rather than per item in this recipe's own input list (§{xref.dep-driven}).

The alternative considered was an explicit declaration such as `cook "build/$<stem>.o" from lib using "…"`, where `from lib` names the iteration driver and placeholder substitutions are orthogonal. The explicit form is easier to spec and eliminates the context-sensitivity of `$<lib>` vs `$<lib.ACCESSOR>`. It was rejected on surface-syntax grounds: the accessor placeholder already mirrors the own-iteration placeholder set (`$<stem>`, `$<name>`, `$<ext>`, `$<dir>`, §{phl.cook-step}), no new keyword is introduced, and Cookfiles already shipped with the terse form.

The cost — that a step's iteration source is determined by placeholder inspection — is contained by the "one distinct recipe per step" rule (§{xref.dep-driven}): the inspection is unambiguous, and the diagnostic for mixing two distinct drivers is straightforward.

### C.13.P. Closed accessor set

The set of path accessors (§{xref.path-accessors}) is `{stem, name, ext, dir}` and is closed. Adding an accessor is a change to this Standard.

The alternative — opening the set to every zero-arg `path.*` function — was rejected because the resolver in §{xref.resolution} splits on a dot and checks the SUFFIX against a small fixed set. Opening the set would require the resolver to consult the `path` module at parse time, introducing semantic lookup into what is currently a syntactic pass and exposing resolver output to extensions of the standard library. Growth of surface syntax should be a Standard change; growth of Lua helpers should not require one.

A zero-arg `path.*` addition not in the accessor set remains reachable from step bodies via a `{ … }` Lua block (§{steps.cook-single}, §{steps.cook-multi}). The closed-set rule does not restrict what authors can compute — only what syntactic shortcuts exist for the common cases.

## C.16. On §{exec.ord} Cross-recipe ordering [#rationale.ordering]

§{exec.cross-recipe-ordering} admits exactly one source of ordering — a name reference — and §{xref.dep-implications} says what does not count as one. The subsections below record why those two rules survived a mechanism (§{lua.cook-require-recipe}) that at first reading looks like an exception to them, and why the two diagnostics that sit alongside them (§{exec.output-uniqueness}, §{exec.read-after-write}) are deliberately scoped differently from each other and from their §{lua.add-unit-output-globs} cousin. The subsections above (§{rationale.exec}, in particular the "Name references as the sole implicit-dep source" note recording CS-0009's removal of the ingredient/serves byte-equality rule — the note this Standard's older text cited as "App. B.5.N" before the v0.10 reorg renumbered App. B into App. C) give the historical background these sections build on.

### C.16.1. Why a name reference is evidence and a path match is not [#rationale.name-reference-evidence]

§{xref.dep-implications} forbids inferring a cross-recipe edge from path-string equality. §{lua.cook-require-recipe} lets a module, running inside a recipe body it does not own, declare a cross-recipe edge outright. These read as contradictory only if the prohibition is taken to be about *mechanisms*. It is not: it is about **evidence**.

An edge asserts that the author meant one recipe to run before another. The question a conforming implementation must answer is what licenses that assertion. A path string that happens to equal another recipe's declared output licenses nothing, because the author never named the referent — the match is a coincidence of spelling, and an implementation that promoted it to an edge would be *guessing* at an intent nobody expressed. Two authors constructing output paths algorithmically in a large workspace collide by accident; a refactor renaming `build/lib/libmath.a` to `build/artifact/libmath.a` silently rewires the graph with no diagnostic pointing at the boundary that moved.

`cook.require_recipe(name)` supplies exactly the evidence §{xref.dep-implications} demands. Its argument is a **recipe name** — never a path — written by the author directly, or by a module acting on a name the author wrote (`links = {"idLib"}`). The referent is named, so the edge is **authored, not inferred**. §{lua.cook-require-recipe} is therefore an *instance* of the "Name reference ⇒ cross-recipe edge" rule and not an exception to it, which is why §{xref.dep-implications}'s prohibition needs no amendment to accommodate it: the API cannot launder a path match into an edge, because it never sees a path. Only §{exec.cross-recipe-ordering}'s *enumeration* of where a name reference may be written had to widen; the category it enumerates did not move.

The alternative design — inferring the ordering from an output↔input path match, so that a consumer declaring `build/gen.a` as an input would be ordered after whichever recipe declares it as an output — was rejected on precisely this ground, and the cost of taking it would have been structural rather than stylistic. It would have reversed §{xref.dep-implications} and §{exec.cross-recipe-ordering}, reinstating by the back door the byte-equality rule CS-0009 removed. It would also have forced a re-think of §{lua.add-unit-output-globs}'s terminality rule, which exists **because** globs cannot be lexically matched: an implementation that derives ordering from path matching owes an answer for the outputs it cannot match, and the only answers available are to leave glob-output recipes silently unordered (the footgun §{rationale.output-globs-terminal} closes) or to resolve globs against the filesystem at plan time (which makes the graph a function of disk state rather than of the source). Declining to infer keeps both problems from arising: an unmatched glob and an unmatched literal are treated identically, because neither was ever evidence of anything.

What the author loses by this choice is nothing they cannot spell. The dependency they meant is one `: producer`, one `$<producer>`, or one `cook.require_recipe("producer")` away — and where they plainly meant it and did not say it, §{exec.read-after-write} tells them so rather than leaving them to find out under `--jobs > 1`.

### C.16.2. Why §{exec.read-after-write} is closure-scoped where §{lua.add-unit-output-globs} is workspace-wide [#rationale.read-after-write-scope]

§{lua.add-unit-output-globs}'s cross-recipe-edge check fires on any lexical match anywhere in the workspace, without regard to what is being built. §{exec.read-after-write} fires only when producer and consumer are in the same build closure. The two look like inconsistent scopings of the same idea. They are not the same idea.

§{lua.add-unit-output-globs} states an **ownership** claim. A glob or directory output declares that the build owns a pattern or a subtree whose membership is not knowable until execute time (§{rationale.output-globs-terminal}); another recipe naming a file inside it as a literal input has made a claim on bytes it cannot be told about, because there is no register-time file-content edge to be had. That defect is a property of the two declarations alone. It is wrong on Tuesday and wrong under `--jobs 1`, and no scheduling makes it right, so scoping it to a closure would only make it detectable in some invocations and not others.

§{exec.read-after-write} states a **scheduling** hazard. A literal output is an ordinary artifact, build-owned by nothing, and legitimately referenceable: an author may declare `build/gen.a` as an output of one recipe and an input of another and be entirely correct about it, so long as the two never run without an ordering between them. `cook producer && cook consumer` is exactly that arrangement, and it is a workflow the Standard has no business rejecting — the consumer's inputs are on disk, put there by an invocation that has already finished. The only defect is missing ordering, and missing ordering is a hazard only where the scheduler is free to pick the wrong one, which is only where both recipes are in one closure. A workspace-wide scoping would therefore reject a legitimate build for a race it cannot have, and the tripwire CS-0142 left behind for exactly this decision (`raw_path_input_does_not_order_producer_before_consumer`, which builds its consumer alone) is what a workspace-wide check reddens first.

The two rules divide cleanly along that line, and the division is why they never double-report: terminal outputs stay §{lua.add-unit-output-globs}'s business, literal outputs stay §{exec.read-after-write}'s.

### C.16.3. Why §{exec.read-after-write} is directed where §{exec.output-uniqueness} is undirected [#rationale.read-after-write-directed]

§{exec.output-uniqueness} is satisfied by an ordering path between the two recipes *in either direction*. §{exec.read-after-write} is satisfied only by a path from the consumer to the producer. The asymmetry follows from what each rule is protecting against.

Write-write is symmetric. Two recipes writing one path race, and the harm is that the winner is undefined — the artifact's contents become non-deterministic and every cache key derived from them becomes unstable. Either ordering fixes that, because either ordering makes the outcome defined. The Standard has no opinion about *which* of the two writers should win; that is the author's call, and by adding an edge in either direction they have made it. Any path suffices, so the predicate is undirected.

Read-after-write is asymmetric. Only producer-first is correct. The reverse path — the consumer ordered *before* the producer — is not a partial fix or a lesser evil: it pins the stale read in place on every run, turning what would otherwise be a scheduling-dependent race into a **deterministic** wrong answer. It is worse than no ordering at all, since an unordered build at least fails loudly some fraction of the time. An undirected predicate would wave it through as "these two are dep-related, nothing to see here" — silently blessing the one arrangement that is guaranteed to be wrong, and doing so precisely because the author took the trouble to declare an edge. §{exec.read-after-write} therefore passes only when the consumer transitively requires the producer, and the reference implementation's directed reachability walk is deliberately not shared with §{exec.output-uniqueness}'s undirected `connected()`: the two predicates answer different questions and unifying them would erase this distinction.

## C.21. On §{lua} Cook Lua API [#rationale.lua]
The Cook Lua API is deliberately small and sharply phase-split. Most build-system DSLs expose a single large API whose functions do different things depending on when they are called — declaration during a "load" pass, execution during a "run" pass, with the split encoded by convention rather than by contract. The Cook Standard commits to the split in its normative text: every function is classified as register-phase, execute-phase, or both, and a conforming implementation MUST install each function in the phases it is documented for.

The reasoning is pragmatic:

- **The register phase is the cache boundary.** Everything that happens during registration is observed by the cache (§{exec.cache}). Writing a file during registration makes that file a register-time input in a way that cache hashes do not see. The small `cook.sh` / `cook.exec` surface is the entire register-time effect mechanism; the larger `fs.*` surface is available in both phases because authors legitimately want it in both.
- **The execute phase is the parallelism boundary.** Execute-phase work runs on a pool of worker VMs, each with its own Lua state. A function that mutates shared capture state cannot sensibly run on a worker. `cook.add_unit` is register-phase only for exactly this reason.
- **Keeping the surface small means the phase split is memorable.** A surface this size can be held in an author's head: `cook.add_unit`, the `cook.sh`/`cook.exec`/`cook.interactive` helpers, seven `fs.*` functions, six `path.*` functions, and the execute-phase using-block globals. That is intentional.

### C.21.1. Why placeholder substitutions exist alongside the using-block globals [#rationale.placeholder-vs-globals]
§{phl.cook-step} defines `$<in>` (unit-centric: the loop member in one-to-one, the space-joined collected input set in many-to-one — App. E CS-0130), `$<out>`, `$<out_N>`, and the dotted-accessor `$<NAME.ACCESSOR>` family for shell blocks (and, by §{xref.string-substitution}, `test` / bare `shell_command`). §{lua.using-block-globals} defines `input`, `output`, `inputs`, `outputs`, and `input_N` for `>{ ... }` Lua-code blocks. Both express the same fundamental idea — "the current input and output of a step" — and an earlier draft of the Standard considered collapsing them to one.

They remain separate because they inhabit different typographic contexts. A shell using-string is a shell line; shell has no binding mechanism for "the current file", so a placeholder syntax is the only option that keeps the surface readable. A Lua using-block is Lua code; Lua has direct variable binding, so exposing `input` and `outputs` as globals is more Lua-idiomatic than expanding `$<in>` inside Lua text.

The substitution also differs in timing. Shell using-string placeholders are substituted at register-time code generation — by the time the unit is recorded, the shell string is already concrete text. Lua using-block globals are bound at execute time on the worker VM. Collapsing the two would require either executing Lua fragments at register time (a re-entrancy hazard) or deferring shell substitution to execute time (a loss of register-phase observability). Keeping them separate is the less-bad outcome.

### C.21.2. Why `fs.*` and `path.*` are separate namespaces [#rationale.fs-path-namespaces]
`fs.*` performs I/O against the Cookfile's working directory. `path.*` performs pure string manipulation on path components and is working-directory-independent. An earlier sketch merged them under a single `path` namespace, distinguishing "pure" from "I/O" functions by naming convention. This was rejected.

Keeping the namespaces apart matches a common Lua convention (see the standard library's `io` vs. `string`), and — more importantly — it mirrors the phase-split reasoning in the chapter intro. `path.*` can never observe anything phase-sensitive, so merging it with `fs.*` would invite readers to assume every `fs.*` call is similarly pure. The namespace split is a small, cheap signal that I/O is happening.

### C.21.3. Why `cook.add_unit` is register-phase only [#rationale.add-unit-register-only]
A naive design might allow `cook.add_unit` inside `>{ ... }` Lua blocks — that is, inside an execute-phase step — as a way to dynamically produce further units based on build outputs. The Standard forbids this: `cook.add_unit` is register-phase only.

The reason is the cache. The set of work units is the input to the scheduler (§{exec.phases}), and the scheduler's correctness depends on knowing all units before any of them runs. A unit added during execution would have to either (a) force re-planning mid-execute — which invalidates any parallelism already in flight — or (b) sneak past the scheduler and run outside its ordering guarantees. Both options break the contract. Authors who need data-driven unit counts call `cook.sh` at register time (§{lua.cook-sh}) to compute the data, then call `cook.add_unit` for each computed unit, all during registration. The examples in `cpp.lua` (e.g., `cpp.static_library`) are the canonical shape.

### C.21.4. Why the cook-step using-string recognises recipe names [#rationale.recipe-name-priority]
A `$<TOKEN>` in a using-string that is neither a known placeholder nor a recipe name falls through to `cook.env[TOKEN]` (§{phl.cook-step}). A `$<TOKEN>` that matches a recipe name in the same Cookfile is instead rewritten to `cook.dep_output("TOKEN")`. This is a deliberate asymmetry: in practice, cross-recipe references to "the outputs of another recipe" are far more common than environment variables that happen to share a name with a recipe. Giving recipe names priority in the fallback order means authors rarely need to choose between the two at the token level — the parser picks the right one.

### C.21.5. Why output-pattern is the sole iteration source [#rationale.output-pattern-iteration]
Pre-CS-0022, iteration mode was decided by two non-adjacent regions: §6.7 said "`$<in>` in the using-string drives one-to-one"; §5.4 said "`$<lib.ACCESSOR>` in the output pattern drives dep-driven one-to-one." Both rules co-existed peacefully when authors wrote consistent code, but they failed silently on inconsistent code: `cook "out" using "do $<in>"` (literal output, `$<in>` in using-string) iterated per input under the §6.7 rule, with every iteration writing to the same literal output, clobbering successive results.

CS-0022 makes the output pattern the sole iteration source. The output pattern is the *declarative* part of a cook step — it describes what files the step produces. If those filenames are parameterized (via `$<in.ACCESSOR>` or `$<lib.ACCESSOR>`), iteration follows. If they are literal, the step does not iterate. Authors who wrote the silent-clobber pattern get a load-time error pointing at the literal-output / `$<in>`-in-body mismatch.

### C.21.6. Why `using "cmd"` was removed [#rationale.using-cmd-removed]
The string and block forms expressed the same thing — a shell command — through different surfaces. With shell blocks first-class (CS-0022 also extends §{xref.string-substitution} and §{phl.cook-step} to cover `shell_block` content) the string form bought one character of brevity per call site against the cost of a separate normative branch in §6.7 and a separate dispatch arm in the body production. The cost was paid every time an author moved a one-line command onto two lines. The Standard now has one shell surface — `{ cmd }` for one-liners; `{ cmd1\n cmd2 }` for multi-line — and one Lua surface, `>{ ... }`.

### C.21.7. Why bare path-accessors were removed [#rationale.bare-accessors-removed]
Two spellings of the same concept (`$<stem>` and `$<in.stem>`) with position-dependent admissibility (`$<stem>` valid in output patterns, `$<in.stem>` valid in using-clause bodies under earlier proposals) is the kind of memorise-by-rote rule a Standard pays for forever. CS-0022 collapses the surface to one spelling — `$<NAME.ACCESSOR>` — valid in both positions, with `NAME` ∈ `{ in, out, out_N, libname }`. Bare `$<stem>` reads, post-CS-0022, as a `$<TOKEN>` resolving through §{xref.resolution} step 4 to `cook.env["stem"]`. That fallthrough is the wrong failure mode for code being migrated, so the implementation MUST emit a specific diagnostic for the four bare accessor names that points at the new spelling.

### C.21.8. Why `$<lib.ACCESSOR>` is rejected in using-clause bodies [#rationale.lib-accessor-firewall]
The §{xref.dep-driven} firewall rejecting `$<lib.ACCESSOR>` in a step whose output pattern does not declare `lib` as the driver is preserved verbatim by CS-0022. Inside a step where `lib` *is* the driver, the spelling `$<in.ACCESSOR>` is used — the local name `in` for "the current iteration item" is universal across drivers, so the using-clause body never names the driver explicitly. Allowing `$<lib.ACCESSOR>` in the body would let an author smuggle a second iteration source past the output-pattern declaration, undoing §{steps.iteration-mode}'s single-driver invariant.

### C.21.9. Why recipe names within a Cookfile MUST be unique [#rationale.recipe-name-uniqueness]

Earlier reference implementations of `cook.recipe` accepted duplicate registrations silently: whichever registration appeared first in the codegen-emitted Lua source won, and subsequent registrations of the same name became no-ops. The §22.3 uniqueness rule retires this behaviour and requires a hard error naming both sites.

The motivation is two common authoring footguns.

First, a surface `recipe build` block combined with a `cook_cc.bin("build", …)` call inside a `register` block. Authors writing these together expected one form to override the other; neither did so deterministically without documentation; the actual behaviour was source-order dependent. Source-order dependence is invisible at the Cookfile level — the author cannot see codegen output and would not think to consider it.

Second, two wrapper calls registering the same name (for example, two `cook_cc.bin("game", …)` calls in different `register` blocks of the same Cookfile). The first call's registration won; the second silently no-op'd. An author writing the second call had no signal that it was being ignored short of running the build and noticing the second call's side effects (probes, output paths) were absent.

Both patterns are authoring errors. The §22.3 rule makes them diagnosable. The diagnostic names both registration sites and identifies the kind of each so the author knows precisely what to rename.

The choice not to designate a winner (i.e. to error rather than to define a precedence) reflects a broader principle: silent source-order precedence is a recurring source of long-lived bugs in build systems. When two declarations conflict, naming both sites pushes the resolution to the author at the moment of conflict, instead of compounding into mis-debugged runtime behaviour later.

### C.21.10. Why a module-minted recipe is annotated rather than refused, and why `origin` is explicit [#rationale.recipe-origin-annotation]

Recipe identity belongs to the user: `cook list` should read as a list of the things *this project* can do, and every name in it should be one the Cookfile author put there. Modules are therefore step contributors (§{mods.authoring.minting}), and the two shapes that still require a module to mint a recipe — data-driven fan-out, where the names derive from parsed data the author never typed, and a support recipe, whose work has nowhere else to live — are carve-outs from that rule rather than an alternative to it. `origin` (§{lua.cook-recipe}) is what keeps those carve-outs honest: a name the user did not write is listed together with the module surface that wrote it, so a reader can tell the two apart at a glance instead of grepping module sources.

**Why annotate rather than refuse.** The support-recipe carve-out was described during design as "non-invocable by intent", which left open whether an implementation should *refuse* a dispatch naming such a recipe rather than merely annotate it. It should not, for five reasons.

- Paragraph 2 of §{lua.reg.discovery} is **already a normative MUST**: a dispatch naming a recipe registered only through register-phase Lua executes identically to one naming a surface declaration. Refusing would mean amending an existing MUST, not clarifying one — a materially larger change than an annotation warrants, and one that would have to be argued on its own terms.
- Refusing invents an **invocability tier the language does not have**. Nothing else in Cook distinguishes "registered" from "registered but not runnable". Minting that distinction to serve a presentation annotation is disproportionate to what the annotation is for.
- It would **make the listing lie**. A refusal would have `cook list` advertise a name that errors when typed — inverting the point of surfacing minted recipes loudly in the first place.
- A support recipe is **a real DAG node**. Other recipes depend on its outputs; running it directly is harmless, and is exactly what an author wants when debugging why a generated file came out wrong.
- "By intent" is the **module author's** intent, and the annotation is how that intent reaches the user. It says: this name is not yours, here is who made it and why. That is the whole of what the phrase was ever promising. The engine need not police it, and an engine that did would be enforcing one author's preference as a language rule.

**Why `origin` is explicit rather than inferred.** An implementation could plausibly derive an origin without the author's help — by walking the Lua call stack at the `cook.recipe` call site, or by tagging every registration that did not come from the code generator. Both were rejected on one ground: **misattribution**. A call-stack walk names whatever frame happens to be on the stack, which for a module dispatching through a shared internal helper is the helper, not the surface the user called. A registration-site tag records *that a module registered the recipe*, never *which module surface a reader should go look up*. Both produce a plausible-looking string that is wrong in precisely the cases where the annotation matters most — deep module internals — and a wrong attribution is worse than none, because the reader has no way to know it is wrong. `origin` is author-supplied because the module author is the only party that knows the answer.

This does not reverse the CS-0077 stance. That entry deliberately pushed the **surface-vs-Lua registration distinction** out of the Standard: how an implementation tells a codegen-emitted registration from a hand-written `cook.recipe` call is implementation-defined (the reference implementation tags surface registrations with a codegen-private helper), and the Standard does not require that distinction to be observable anywhere. `origin` is not that distinction re-admitted through a side door, because it is not a property of *where* a registration came from at all. It is module-author-supplied intent — a value a human wrote into a metadata table for another human to read. An implementation remains free to tag registration sites however it likes, and still may not surface such a tag as attribution. That `origin` rides only the public `cook.recipe` path, and cannot appear on a surface declaration, follows from who supplies it: a surface `recipe`/`chore` declaration has no author-facing metadata table to put one in, and its author is the user, who needs no attribution to themselves.

## C.11. On §{comp.overview} Cross-Cookfile composition and §{mods.use} `use` [#rationale.modules]
Cross-Cookfile composition (§{comp.overview}) and Lua module loading (§{mods.use}) are deliberately split into two declarations that look similar but answer different questions. `use` answers "give me additional Lua for my recipes to call"; `import` answers "tie another Cookfile's recipes into my dependency graph." Conflating them — for example, by making `import` also bring a Lua value into scope — would force authors to pick one semantics at every call site and would blur the two namespaces the dependency resolver walks. The Standard keeps them distinct so that each declaration has one job, and lives in a different chapter: `use` belongs with the Cook Lua API because it loads Lua code, whereas `import` belongs with cross-Cookfile composition because it binds an alias for cross-recipe references.

Built-in modules are resolved before `cook_modules/` (§{mods.use}) so that adding a new built-in in a future version of the implementation cannot silently break a project that happens to have a `cook_modules/<name>.lua` of the same name: the built-in always wins, so a project author who wants to override it renames their local module or picks a different name. The converse ordering — consulting `cook_modules/` first — would make the set of built-in names a latent hazard: every new built-in would be a silent semantic change for every project that had colonised the name first. v0.1 of the Standard defines no built-ins, but fixes the ordering now so that the first built-in to ship can do so without a migration.

The choice of `cook_modules/` — an adjacent directory, resolved only relative to the enclosing Cookfile — rather than a global registry or a walk up through ancestor directories is a deliberate choice for locality. A Cookfile's modules are the ones it ships with; they travel with the project, they are the ones code review sees, and they are the ones `git clone` reproduces. A global registry would require side-channel installation; an ancestor walk would make module resolution sensitive to where on disk the project happened to live. Both are costs the Standard declines to pay.

The §7 search-path order adopts the LuaRocks tree layout (`share/lua/<ver>/`, `lib/lua/<ver>/`) as the on-disk shape so cook can interoperate with luarocks.org rockspecs without forking layout policy. Hand-vendored top-level files retain highest priority so authors can override a rock-installed version by dropping a file at `cook_modules/<name>.lua`; the LuaRocks tree is searched second.

### C.11.1. Why duplicate-`import` is a parse-time error rather than last-wins [#rationale.duplicate-import-parse-time]
A "last-wins" override is tempting because it is what most language include-systems do for simple `#include` and because it composes cleanly with machine-generated fragments. The Standard rejects it for two reasons.

First, the duplicate-import rule is the only defence against a silent bug where a user edits one `import backend ./services/backend` into `import backend ./services/backend_v2` in a diff that also intends to keep the first line around for some other recipe to consume. Last-wins would silently redirect every `backend.*` dependency in the file to the new target, and the first import would become dead code with no diagnostic. A parse-time error surfaces the mistake at the moment of the edit.

Second, a duplicate-import is diagnosable locally — it is a property of a single Cookfile's token stream, which the parser already holds in full. Pushing the check to a later stage (workspace load, DAG construction) would mean the diagnostic arrives after some non-trivial work has succeeded, and the error message would have to reconstruct which line declared which alias. Parse time is where the information is cheapest.

### C.11.2. Why import cycles are detected at workspace-load time rather than at parse time [#rationale.cycle-load-time]
Symmetrically, cycles are NOT caught at parse time because they cannot be: a single Cookfile's tokens cannot know whether `import b ../b` eventually loops back. Cycle detection requires the parser to have run on every Cookfile the edge set transitively touches, and that only happens at workspace-load time. Pushing cycle detection earlier would either require a separate pre-pass — effectively re-parsing every Cookfile before validating any — or would require `import` to be resolved synchronously during parsing, which would re-introduce the coupling that v0.1 deliberately separates (the parser produces an AST; the workspace loader consumes it).

Canonicalisation at the edge boundary (§{comp.duplicates-and-cycles}) is load-bearing. Without it, `import b ./b` and `import b ./services/../b` would be different nodes in the cycle graph, and the diamond dedup rule of §{comp.duplicates-and-cycles} would fire on one but not the other. Canonicalising to absolute, symlink-resolved paths collapses these spellings into one node.

### C.11.3. Why local module names do not collide with reserved recipe segments [#rationale.module-name-no-collision]
The reserved recipe segments of §{lexical.keywords} (`stem`, `name`, `ext`, `dir`, `in`, `out`) are reserved for the final segment of a recipe or dependency name, not for identifiers in general. A local module may therefore be named, for example, `out.lua` in `cook_modules/`, and bound by `use out`; the binding would collide with nothing in the grammar, because `use` resolves against `BARE_IDENTIFIER` and `STRING` (App. A.2) and is not subject to the recipe-segment reservation.

The Standard declines to extend the reservation to module names because the two namespaces are independently diagnosable: a recipe body that writes `out.foo()` is a `module_call` only if `out` has been bound by a prior `use` declaration in the same Cookfile (§{toplevel.module-call}). Without the `use`, the same line is a `shell_command`, and the dispatch cascade of App. A.4 will classify it so. Reserving the name globally would not prevent any bug the dispatch already handles, and would remove an axis of expressiveness from module authors.

## C.10.1. `out_0` rejection at resolution rather than fall-through [#rationale.out-zero]

The closed-set resolver introduced with the sigil placeholder syntax rejects `$<out_0>` as a `MalformedOutIndex` error rather than letting it fall through to the recipe-then-env-lookup steps. Lexically, `out_0` matches the `out_indexed` production (`"out_" DIGIT+` admits `0`); the rejection happens at resolution, not lexing.

The choice between "reject at resolution" and "fall through to env" was decided in favor of the former because the diagnostic surface is meaningfully better:

- **Reject path:** "placeholder `$<out_0>`: malformed out_N (N must be ≥ 1)" — names the exact problem.
- **Fall-through path:** "placeholder `$<out_0>`: env var 'out_0' was not declared" — misleading; suggests the author meant an env var when they actually mistyped a builtin.

Authors who type `$<out_0>` are virtually always misremembering that `out_N` is 1-indexed; the diagnostic should help them notice that, not redirect them to the env-var declaration system.

The `BuiltinMatch::Malformed` variant in the resolver makes the hard-stop explicit in code: `match_builtin` returns three outcomes — `Yes` (valid builtin), `Malformed` (builtin shape, structurally invalid), and `No` (not a builtin shape at all). Only `No` falls through; `Malformed` short-circuits to an error without consulting the recipe or env namespaces.

## C.21.1. The `cook.require_env` runtime mechanism [#rationale.require-env]

§{xref.resolution} step 3 (declared env var) defers the "declared" check to register time, after config-block evaluation has captured the env table's keys. The reference implementation realizes this contract through a Lua function `cook.require_env(name)` injected into the Cook environment alongside `cook.env`, `cook.dep_output`, and the other `cook.*` helpers.

The function checks `name` against a frozen keyset captured immediately after the config-block phase completes. A missing key produces a `RuntimeError` with a diagnostic that names the consulted token, the declared keys in scope, and the recommended `env.NAME = ...` declaration form. The keyset is frozen — additional `cook.env.NAME = "..."` writes during recipe registration are visible to `cook.env[name]` but do not retroactively make the key "declared" for `cook.require_env` purposes; the contract is fixed at the end of config-block evaluation.

The "frozen" property is enforced against env-table writes — keys added to `cook.env` after the freeze point are visible to `cook.env[name]` but are not in the declared set. The implementation's `EnvKeyset::freeze` is itself additive (multiple calls union their captured keys); the reference engine invokes `freeze` exactly once, immediately after config-block dispatch returns, so the additive property is unobservable in production code.

Two design alternatives were considered and rejected:

- **Codegen-time check.** The codegen could parse config-block source for `env.NAME = ...` patterns and reject undeclared references at register-time emission. This requires the codegen to understand Lua syntax, including conditional assignments and dynamic key construction (`env[some_var] = ...`). The runtime check is simpler, more accurate, and produces the same author-facing diagnostic at virtually the same time (register time vs. codegen time).
- **Silent fallthrough to `cook.env[name]`.** This was the pre-sigil-placeholder-syntax behaviour and it caused real bugs: a typo such as `$<HOEM>` silently expanded to an empty string, propagated through cache, and triggered spurious reruns. The closed-set resolution rule explicitly rejects this option to make every placeholder either resolve to a known thing or fail loudly.

## Why `cc` rather than `cpp` (§{cat.cc}) [#rationale.cc-not-cpp]

The blessed C-family build module is named `cc`, not `cpp`. The module handles `.c` and `.cpp` (plus `.cc`, `.cxx`) sources uniformly: extension dispatch chooses the appropriate compiler from a single toolchain triple. Naming it `cpp` would privilege C++ over C even though every public function (`cc.bin`, `cc.lib`, `cc.find`, etc.) is equally idiomatic for either language. The name `cc` aligns with the historical Unix toolchain convention (`cc(1)`, `$CC` make-variable) and with Rust's `cc` crate, both of which describe "C-family compilation" without language privilege.

## Why §9 Standard Modules is normative [#rationale.stdmods-normative]

Blessed modules are part of the Cook contract that anyone — not only Liora Labs — may reimplement. The official rock is one implementation; an alternative implementation (e.g., a Rust-side native module exposing the same Lua surface, or a strictly-pinned older snapshot) must produce semantically equivalent behaviour against the §9 sections it claims to implement. Specifying the surface normatively in numbered Standard chapters — rather than informatively in an appendix — makes conformance checkable: every public function lists its semantics with RFC-2119 MUST / SHOULD keywords; every option table field is enumerated; every error condition is documented.

## Why `cc.find` v0.1 specified pkg-config-only [#rationale.find-v01-history]

The v0.1 wording (released in `cook_cc 0.1.x`) specified pkg-config as the sole `cc.find` strategy. This was deliberate: M0 (the chapter introduction) optimised for a tight normative surface that real Cookfiles could verify against, while leaving room to extend. v0.2 (M1, this revision) generalises to a multi-strategy chain; the v0.1 wording is preserved here so anyone bisecting against `cook_cc 0.1.x` has a referenceable spec. There is no permanent break — every v0.1 call site (`cc.find("zlib")` etc.) remains a valid v0.2 call site, and pkg-config remains a chain stage.

## Why cmake-compat ships on legacy `--find-package` [#rationale.cmake-compat-legacy]

cmake 4.x marks the `--find-package` legacy mode "Do not use" in `cmake --help`. cmake-compat ships on this mode anyway because:

1. The annotation is years-old guidance, not a removal timeline. The cmake project carries it for nudging users into proper cmake projects, not as a deprecation deadline.
2. Reimplementing the equivalent Config-resolution logic natively is the post-M6 "full CMake interpretation" north star, explicitly deferred as not-committed.
3. The migration path if cmake ever removes legacy mode is to swap `cmake_compat.lua` internals to a temp-project `find_package(X CONFIG REQUIRED)` + `message(STATUS …)` extraction approach. That migration is an internal-only change with no spec impact — `cc.find`'s surface stays identical.

The trust boundary is documented: anything cmake-compat finds is found by cmake's own resolver, so accuracy is delegated; the only Cook-implemented part is the output parser specified in §9.2.3.8.2/3.

## Why cmake-compat sits after pkg-config in the default chain [#rationale.cmake-compat-position]

The original module roadmap proposed "after curated, before pkg-config" for cmake-compat. v0.3 places cmake-compat after pkg-config for three reasons:

1. **Speed.** pkg-config returns in ~10ms; the cmake subprocess takes ~100–500ms per probe. Projects whose dependencies all ship `.pc` files never pay the cmake cost in the default chain.
2. **Substrate stability.** pkg-config's upstream has not deprecated the discovery mode; cmake's `--find-package` legacy annotation gives pkg-config the longer life expectancy as a build-system contract.
3. **Curated finders own the per-library overrides.** The handful of libraries where cmake-compat is preferred over pkg-config (e.g. SDL3 has no `.pc`; old `libsdl2.pc` may lie around) get the explicit `opts.cmake = true` opt-in, which lifts cmake-compat into step 3 *and* skips pkg-config for that call. The opt-in is the right surface for "I know this is Config-only" — callers express intent at the call site rather than via global resolver order.

## Why `cc.find` v0.2 documented a four-stage chain [#rationale.find-v02-history]

The v0.2 wording (released in `cook_cc 0.2.x`) specified a four-stage chain: project → curated → pkg-config → bare-probe. v0.3 (M2) inserts cmake-compat between pkg-config and bare-probe and adds the `cmake` opt. Every v0.2 call site remains a valid v0.3 call site; pkg-config remains a chain stage. Bisects against `cook_cc 0.2.x` reference this paragraph.

## C.17. On §{exec.cache} Cache semantics [#rationale.cache]

### C.17.1. Glob output terminality [#rationale.output-globs-terminal]

Cook propagates cache invalidation between steps via the content fingerprints of their shared files: a downstream step that declares an upstream's output as one of its `inputs[]` re-fingerprints when that file's bytes change. This linkage requires the path to be statically declared at register time on both sides.

A glob pattern entry in `outputs[]` (§{lua.add-unit-output-globs}) cannot participate in this linkage because the resolved file set is not known until execute time. A downstream that declared a specific path expecting it to be produced by an upstream glob would have no way to express a file-content edge at register time, and Cook's cache check has no `did-my-upstream-rebuilt` signal. The downstream would silently consume stale bytes after the upstream re-ran.

The terminal-output rule (§22.1.2 "Cross-recipe edges") eliminates this footgun by construction. Authors who want file-content invalidation propagation use literal `outputs[]` entries; authors who only need ordering use `requires = { ... }`; the engine catches the misuse at register time via a purely lexical match of downstream input strings against upstream pattern strings. The rule aligns Cook with the Turborepo model (tasks depend on tasks, not files) that the `cook_pnpm` module adapts.

Directory outputs (CS-0119) reuse this same glob terminality machinery — a trailing-slash entry is treated as the pattern `dir/**/*` for the purposes of the cross-recipe edge rule (§22.1.2) — because the subtree's file set is unknown until execute time, for precisely the same reason a glob's match set is. The additional *build-owned reconciliation* (§17.6) closes a hit≠miss gap that surfaced when a producer (e.g. `wasm-pack pkg/`, Vite `dist/`) emits files the author did not enumerate: a cache restore that replays only the recorded file set would leave stray files from a prior run on disk, producing a subtree that differs from a fresh build. Reconciliation removes every file in the declared subtree that is not a member of the resolved or restored set, and prunes empty directories, ensuring the on-disk state is byte-identical to what a fresh run would produce. This subsumes the orphan-on-shrink stale-output sweep for the directory form: files that a previous run produced but the current run no longer declares are removed exactly as orphans are swept by §17.7. Narrow and `**` glob entries (`dir/**`, `dir/*.js`) are deliberately unchanged — they remain additive — to avoid altering the behavior of shipped patterns such as `cook_pnpm`'s `.next/**`.

## C.22. On the native `probe` declaration [#rationale.probe-native-dsl]

### C.22.1. Why probes graduated from the Lua-string `cook.probe()` API to native surface syntax [#rationale.probe-native-syntax]

The `cook.probe()` Lua API predates the native `probe` declaration by several Standard revisions. Until CS-0091 (data-driven fan-out), probes were primarily a module-author tool: `cook_cc` hoists compiler-detection probes out of recipe bodies so that all recipes share a single cached probe value rather than each shelling out independently. In that context, the string-`produce` form was acceptable — module authors write Lua fluently, the `[[ ]]` long-string quoting is idiomatic, and the call site is small relative to the surrounding module machinery.

CS-0091 changed the economics. Data-driven fan-out via `ingredients <probe>` makes probes the primary mechanism for data-driven builds: any Cookfile author who wants one output per JSON record, one deploy per host, or one test per case writes a probe. The population is no longer "module authors" — it is every author writing a data-driven build. At that scale, the ergonomic friction of the string API became a first-class concern:

- **No lexer or editor support.** A `produce = [[ … ]]` long-string is opaque to syntax highlighters, language servers, and linters. Indentation, keyword completion, and error underlining stop at the string boundary.
- **`[[ ]]` escaping hazards.** A `produce` body that embeds a Lua string containing `]]` must use a level-qualified long string (`[=[ … ]=]`), which is non-obvious and easy to get wrong silently.
- **Table-of-tables ceremony.** Expressing the `produce`, `inputs.files`, and `inputs.requires` fields in a nested table literal has more syntactic noise than the information content warrants. The resulting call is longer than the logic it encodes.
- **Discoverability.** A new Cook user reading a Cookfile that uses `ingredients <probe>` finds the probe declaration buried inside a `register` block, co-located with arbitrary Lua, rather than at top level where the Cookfile's data sources belong.

The native `probe` declaration (§{cat.probes.decl}) is pure sugar over the retained `cook.probe()` API. Every native probe lowers to an equivalent `cook.probe()` call at codegen time; the API is not deprecated and remains the correct surface for probes that declare `env` or `tools` fingerprint inputs (not yet expressible natively — deferred to a future native-surface extension). The mapping is direct:

| Native form | API equivalent |
|---|---|
| `ingredients "glob" !"glob"` | `inputs.files = {"glob", "!glob"}` |
| Header dep list `probe X: dep1 dep2` | `inputs.requires = {"dep1", "dep2"}` |
| `json { sh }` | `produce = [[ return cook.json_decode(cook.sh("sh")) ]]` |
| `lines { sh }` | `produce = [[ … cook.sh lines split … ]]` |
| `>{ lua }` | `produce = [[ lua ]]` |

The decision to ship native syntax as sugar rather than as a replacement reflects the Standard's general posture: the Lua API is the extension point; surface syntax is a readability layer over it. Authors who need capabilities the native form does not yet cover fall back to `cook.probe()` without ceremony.

## C.23. On §{lua.cook-on-register-complete} `cook.on_register_complete` [#rationale.on-register-complete]

### C.23.1. Why a run-last callback rather than an `all_recipe_names()` query [#rationale.on-register-complete-vs-query]

An implementation could, in principle, expose the finished registration pass's recipe *name* set as a queryable value rather than as a callback hook: something like `cook.all_recipe_names()`, read after the fact from a module's own bookkeeping. That would not serve the motivating consumer (§22.9, Note 22.9.1). The recipe name set is knowable early — every recipe's header is parsed, and its name registered via `cook.recipe` (§{lua.cook-recipe}), before any recipe body runs at all (§{lua.reg.discovery}). A query over names is therefore no earlier, and no more informative, than what a module can already collect itself by watching `cook.recipe` calls.

What a consumer like a `compile_commands()` aggregator actually needs is not the name set; it is **module state accumulated by recipe-body target-maker calls** — the per-target compiler-flag tables, source lists, and export records a `cc.bin` / `cc.lib` call builds up as each target's body runs. That state exists only after every recipe body has been evaluated, because a target maker populates it from inside the body it is called from, and there is no way to know from a name alone what a body will do when it runs. A name-only query answers "which recipes exist"; the consumer needs "what did every recipe's body actually register", which is a strictly later and strictly richer question.

Nor would a query let the module *order* its own end-of-register work against anything else. Writing `compile_commands.json` is itself an action with a place in the pass — it MUST happen after the last body and it MAY itself need register-phase capabilities (`fs.write`, reading other modules' exports). A query is a passive read with no hook for sequencing side effects; `cook.on_register_complete` is a callback precisely because the consumer's need is not data but a **place to run code**, at a point in the pass no query can express.

### C.23.2. Why minting recipes from a callback is banned [#rationale.on-register-complete-no-minting]

`cook.on_register_complete` fires after the registration pass's body-invocation loop has completed (§22.9). A recipe registered at that point via `cook.recipe` (§{lua.cook-recipe}) would be added to the discovered set with no further opportunity for its body to be evaluated in the current pass — the loop that evaluates bodies has already finished its single pass over the set it knew about. Such a recipe would appear dispatchable (§{lua.reg.discovery}), yet dispatching it would find no work units, no tests, and no exports: a Cookfile with a plausible-looking recipe name that silently does nothing.

This is the same failure shape §{lua.cook-require-recipe}'s "forcing MUST NOT degrade to a silent skip" rule (§22.8) exists to close on the *forcing* side: register and execute disagreeing about what is being built, with no diagnostic at any point. Banning the mint outright — rejecting the registration attempt with a named error, rather than accepting it and leaving the resulting recipe inert — is preferred to the alternative of running a second body-invocation pass over newly-minted recipes, which would reopen the same question `cook.on_register_complete` exists to close (when is the pass actually done?) and could recurse without a defined termination point if a later-minted recipe's own callback minted a further recipe. A callback that needs the effect of a recipe — a file written, a target's outputs finalized — has the tools to produce that effect directly (`cook.add_unit`, `fs.write`, `cook.export`) without minting a name for it; §22.9's restriction costs a module nothing it could otherwise safely have.
