# Appendix E. Pre-1.0 checklist (informative)

> **Informative.** This appendix tracks language-level frictions and design tensions known to need resolution before the Cook Standard is declared stable at 1.0. Each entry names a concrete symptom, the workaround in current use, and the resolution path.

## E.1. Conformance corpus is parser-impl-coupled [#pre-v1.parse-txt-coupling]

**Symptom.** The `parse.txt` files under `standard/conformance/positive/` are textual dumps of one specific implementation's AST shape. When that implementation changes its dump format â€” for example, CS-0011 dropped the `vars: []` line in v0.2 â€” a previously-tagged corpus does not match the current parser, breaking the backwards-conformance recipe (`cook standard.against-tag`, which routes through `standard/cook_modules/checks.lua`) against any prior tag whose dump predates the change.

**Workaround.** When a parser change alters the dump format, the `cs-standard/vX.Y` tag for the in-progress version is force-moved to the merge commit where the parser and the corpus agree (operating rule in `CONTRIBUTING.md`'s cut procedure). Earlier tags are accepted as broken; their corpora remain frozen historical records.

**Resolution path.** Either move `parse.txt` out of the normative corpus into impl-specific fixtures (each implementation owns its own dump checks), or define an impl-agnostic structured AST format (e.g., JSON of a normative shape) so historical tags stay verifiable.

## E.2. `${...}` syntax in `using` strings [#pre-v1.template-vs-bash-expansion]

**Status.** Fully resolved in v0.5 by [CS-0022](D-changes#changes-cs-0022) (for the `using` surface) and [CS-0024](D-changes#changes-cs-0024) (for the `plate` and `test` surfaces).

**Original symptom.** Cook's templater inside `using "..."` strings interpreted `${VAR}` as its own env-lookup form, breaking Bash parameter-expansion forms like `${VAR:-default}`.

**Resolution.** CS-0022 removed the single-string `using "..."` form in favour of `using { â€¦ }` shell blocks. Inside a shell block, `{` and `}` are brace-balanced by the Cookfile lexer (Â§{lexical.brace-blocks}) and the block content is passed verbatim to the shell â€” Cook's templater never sees the interior. `${VAR:-default}`, `${VAR%suffix}`, and all other Bash parameter-expansion forms work inside a `using { â€¦ }` block without any workaround.

**Residual scope.** ~~The `plate "..."` string form is still a Cook-templated string and retains the original symptom.~~ Fully resolved in CS-0024: both `plate` and `test` now exclusively use the brace-block body form (`plate { â€¦ }`, `test { â€¦ }`); the Cook-templated string form has been removed from the language surface.

## E.3. Cook strings have no `"` escape [#pre-v1.no-string-escape]

**Status.** Substantially resolved in v0.5 by [CS-0022](D-changes#changes-cs-0022) (for the `using` surface) and [CS-0024](D-changes#changes-cs-0024) (for the `plate` and `test` surfaces); open for the remaining STRING positions (recipe names, ingredient paths, `cook` output patterns).

**Original symptom.** The string literal in `cook "..."`, `using "..."`, `plate "..."`, `recipe "..."`, etc. is parsed by a routine that finds the first `"` and stops â€” `\"` is not recognized as an escape. Embedding a literal `"` inside any cook string truncates the string at the first inner `"`, producing surprising and silent behaviour (the truncated command runs, the rest of the line is leftover content).

**Resolution (partial).** CS-0022 removed `using "..."`. Shell content that previously had to live in a Cook-string is now written inside a `using { â€¦ }` block, where the shell's own quoting rules apply and `"` is legal in any shell-valid position.

**Residual scope.** Recipe names, ingredient paths, and `cook` output patterns still use unescaped `STRING` (Â§{lexical.strings}); a `"` in any of those positions remains inexpressible. The resolution path is unchanged. ~~`plate "..."` and `test "..."` strings still use unescaped `STRING`.~~ Fully resolved in CS-0024: `plate` and `test` now use the brace-block body form exclusively, removing them from the set of STRING surfaces with this limitation.

**Workaround (unchanged).** Avoid embedding double quotes in cook strings. When shell-quoting is genuinely needed in a command, use the `using >{ â€¦ }` Lua-block form: Lua handles its own quoting and `cook.sh` accepts arbitrary strings.

## E.4. tree-sitter-cook is stale [#pre-v1.tree-sitter-stale]

**Symptom.** `tree-sitter-cook/grammar.js` does not implement Cook Standard v0.2. Notably, it still produces `variable_declaration` for top-level `NAME "value"` (rejected by the Standard since CS-0011); other gaps are enumerated in [CS-0002](D-changes#cs-0002-planned-tree-sitter-cook-conformance-audit).

**Workaround.** The grammar's `tree-sitter test` corpus passes against its own (stale) view of the language. The Standard does not require tree-sitter conformance pre-1.0; the cook-lang Rust parser is the reference implementation.

**Resolution path.** The follow-up plan referenced by CS-0002 will: enumerate divergences, update `grammar.js` and externals, wire a tree-sitter conformance harness against `standard/conformance/`, and verify both implementations agree on the corpus.

## E.5. Implicit termination of recipe and config bodies [#pre-v1.implicit-end]

**Status.** Resolved in v0.4 by [CS-0019](D-changes#changes-cs-0019).

**Tension.** Each `recipe` and `config` block must close with an explicit `end` line. The closing keyword is ceremony â€” every author writes it without thinking, and the parser has no need for it that couldn't be served by "next top-level construct or EOF." A Cookfile with ten recipes has ten lines that exist only to satisfy the grammar.

**Status quo.** Â§{recipes} (recipes) requires `end` at the close of every `recipe` body; Â§{grammar.config-blocks} requires it for `config` blocks. App. A.3 enumerates `end` in the recipe production. The lexer produces `Token::RecipeEnd` for column-0 `end` lines; the parser consumes one per body.

**Constraints.** The brace-block-as-opaque-source-lines model already cleanly separates Lua- and shell-block boundaries from recipe boundaries â€” a `recipe` keyword appearing inside a Lua block's source is never seen by the recipe-body parser, because the brace-counter skips the token stream over the block. Implicit termination by next top-level keyword does not interfere with embedded blocks. The lexer currently classifies `recipe`/`config`/`use`/`import` keywords on the trimmed line content (any indentation), which would let an indented `recipe foo` silently close an outer recipe under the new rule. Tightening the lexer to require column 0 for those keywords is part of the resolution.

**Possible resolution.** Drop `end` from the recipe and config grammars. Body extends until the next column-0 line classified as `recipe`, `config`, `use`, or `import`, or EOF. Lexer changes: require column 0 for top-level keywords; remove `Token::RecipeEnd` (a bare `end` line then becomes `Token::Content` and falls into the existing "unexpected content outside a recipe" path or, inside a recipe, into a shell command). Conformance corpus migrates by stripping `end` lines and updating `parse.txt` line counts. Diagnostics for "you forgot to close this recipe" largely disappear because there is nothing to forget.

## E.6. Implicit recipe header form [#pre-v1.implicit-recipe-header]

**Status.** Resolved in v0.4 by [CS-0018](D-changes#changes-cs-0018).

**Tension.** Â§{grammar.recipe-syntax} normatively defines two recipe header shapes: explicit (`recipe NAME [: deps]`) and implicit (`NAME : deps`). The implicit form has zero usage in `examples/` or `standard/conformance/`; every Cookfile in the project uses the explicit form. App. B.3.7 rationalises only the form's column-0 anchor â€” the only published rationale is defensive ("here is how we keep it from causing trouble"). An accidental `name: value` at column 0 silently parses as a recipe named `name` with dep `value`, with no diagnostic.

**Status quo.** Both forms accepted, six parser tests pin the implicit form (three in `cli/crates/cook-lang/src/tests.rs`, three in `cli/crates/cook-lang/src/lexer.rs`), and App. A.3 lists both productions.

**Possible resolution.** Drop the implicit form. Â§{grammar.recipe-syntax} collapses to one shape; App. A.3's `recipe_header` becomes `"recipe" name (":" dependency_list)?`; App. B.3.7 is deleted; six implicit-form parser tests are removed; one new negative conformance case under `standard/conformance/negative/` asserts that `name: deps` at column 0 is rejected (becomes `Token::Content` and falls into "unexpected content outside of a recipe"). No corpus migration is required because the form is unused.

**Composability.** Removes the typo vector and lets a future E.5 ("drop `end`") change rely on top-level keywords as the unambiguous block-boundary signal. The two are independent but compose well in either order; bundling them in a single cut is reasonable.

## E.7. `chore` as a sibling of `recipe` [#pre-v1.chore]

**Status.** Resolved in v0.4 by [CS-0020](D-changes#changes-cs-0020).

**Tension.** Cook's keyword surface conflates two operationally distinct things: build targets (cached, ingredient-driven, output-producing) and command runners (uncached, sequence-of-shell-commands, no artifacts). Both are written as `recipe NAME ... end`, so a reader can't tell from the header what a thing actually does. In this project's own top-level `Cookfile`, only three of fifteen recipes use cook-step machinery (`ts-generate`, `ts-wasm`, `nvim-install`); the other twelve are pure command runners. The example projects show the same split: `recipe play` in `examples/fzf-picker/Cookfile` is all-interactive task work; `recipe clean` in every example is `rm -rf` with no cook step. The conflation also hurts diagnostics â€” `cook "out" using "..."` inside a recipe-the-author-meant-as-a-task silently allocates a phantom build artifact and caches it on first run, with no signal that the author had something else in mind.

**Status quo.** Authors hand-mark interactivity per shell line via the `@`-prefix (Â§{recipes.shell-steps}) and rely on convention to distinguish task-shaped recipes from build-shaped ones. The engine treats both identically; caching is implicit on the presence of a `cook` step.

**Possible resolution.** Introduce a new top-level construct, `chore NAME [: deps] ... end` (or, post-E.5, `chore NAME [: deps] ... <until-next-top-level>`), as a thin specialization of `recipe`:

- **Step kinds allowed:** `shell_command`, `interactive_command`, `lua_line`, `lua_block`, `module_call`. (Module call is debatable â€” modules typically emit cook steps; ban-from-chore is the conservative choice.)
- **Step kinds banned at parse time:** `ingredients`, `cook_step`, `plate_step`, `test_step`. The diagnostic names the keyword and points the author at `recipe`.
- **Default-interactive:** every `shell_command` inside a `chore` body is treated as if `@`-prefixed. The `@` marker remains valid (no-op) for symmetry with recipe bodies, but is unnecessary.
- **No caching.** Chores always run when invoked. Engine-identical to a recipe with no outputs; the distinction is purely an authoring/static-checking layer over the existing engine.
- **Deps:** same `: dep1 dep2` syntax. Chores may depend on recipes (run a build, post-process) and recipes may depend on chores (rare but legal).

The keyword choice is a deliberate departure from Cook's culinary metaphor (recipe / ingredients / cook / plate / menu / serve). Plain English wins here â€” `chore` reads as "thing you just run" without the cognitive hop required for a strained metaphor (`prep`, `serve`).

**Spec impact.** New normative chapter on chores parallel to Â§{recipes}, plus a `chore_def` production in App. A. App. B picks up a rationale section (why `chore` is a separate form rather than a flag on `recipe`, why the specific banned step kinds, why default-interactive). Conformance corpus gains ~3 positive cases (basic chore, chore with deps, chore with Lua block) and ~3 negative cases (chore with `ingredients`, with `cook`, with `plate`).

**Migration.** Purely additive â€” no existing Cookfile breaks. This project's top-level `Cookfile` would convert ~12 recipes to chores in a single pass; example Cookfiles would convert their `clean` and similar recipes opportunistically; conformance fixtures would stay as recipes (they exercise recipe-specific semantics).

**Composability.** Stacks cleanly with E.5 (chores adopt the same implicit termination rule) and is orthogonal to E.6. Bundling all three into a single "recipe-form simplification" cut is reasonable; staging them across cuts is also fine.

## E.8. Chore-body shell commands skip `{TOKEN}` substitution [#pre-v1.chore-shell-substitution]

**Symptom.** Â§{lua.shell-placeholders} normatively asserts that "every bare `shell_command` is a **shell text**" subject to the chapter's placeholder substitution. Inside a `recipe` body, a `plate` shell-block body, or a `cook using { â€¦ }` body the substitution fires correctly. Inside a `chore` body it does not â€” a `{TOKEN}` placeholder is emitted into the recorded unit's `command` payload verbatim, and the unit fails at execute time when the shell tries to resolve a literal `{TOKEN}` as a command word.

**Repro.** With cook-lang 0.1.0 (claiming Cook Standard v0.5):

```cook
config
    env.ADB = "adb"

chore devices
    {ADB} devices -l
```

`cook --emit-lua` emits:

```lua
cook.recipe("devices", {}, function()
    cook._enter_chore()
    cook.add_unit({command = [[{ADB} devices -l]], interactive = true, line = 4, cache = false})
    cook._exit_chore()
end)
```

`cook devices` fails with `/bin/sh: line 1: {ADB}: command not found`. The same `{ADB}` token in a `plate` shell-block body of an otherwise-identical recipe is substituted correctly to `adb`.

**Status quo.** Codegen for `chore`-body `shell_command` steps appears to bypass the Â§{lua.shell-placeholders} substitution path that fires for `recipe`-body shell steps and for `plate`-body shell blocks. Spec and implementation disagree.

**Workaround.** Route chore-body env access through Lua. Replace the bare shell line with a `> lua_line` calling either `cook.sh()` directly or a module helper:

```cook
chore devices
    > cook.sh(cook.env.ADB .. " devices -l")
```

This works because Lua sees `cook.env` directly and the substitution layer is not on the path.

**Resolution path.** Two options, listed in order of preference:

1. **Fix the implementation.** Extend the chore-body codegen to apply the same Â§{lua.shell-placeholders} substitution pass that fires for recipe-body and plate-body shell strings. The Â§4a.3 default-interactive transform already touches each `shell_command` in a chore body; adding the placeholder pass to the same site is a localized change. Add a positive conformance fixture under `standard/conformance/positive/` of the form `chore + config-env + {TOKEN} bare shell command + expected substituted invocation` so the harness pins the behaviour.
2. **Narrow the spec.** If chore shell commands are deliberately exempt from substitution (e.g., to keep them "raw shell" for predictability), update Â§{lua.shell-placeholders}'s "every bare `shell_command`" wording to explicitly exclude chore-body bare shell commands, and add a negative conformance fixture asserting non-substitution. This option is weaker â€” it removes a reasonable expectation rather than meeting it â€” but it is consistent with the existing chore-as-command-runner framing.

Either resolution is a single CS entry; option 1 needs a parser/codegen change and a positive fixture, option 2 needs only a spec wording change and a negative fixture.

**Found in.** Real-world Cookfile for the `glovemap` Android key-mapper project (`~/dotfiles/android/glovemap/Cookfile`), which exercises plate's three iteration modes for adb-deploy and tripped on this when it tried to use `{ADB}` and `{PACKAGE}` env placeholders in chore bodies for `devices` / `logs` / `uninstall`. The project routes through `> android.adb_devices(cook.env.DEVICE)` etc. as the workaround.

## E.9. No multi-line continuation for `ingredients` lists [#pre-v1.multi-line-ingredients]

**Tension.** Â§{recipes.ingredients} requires every ingredient on a single `ingredients` line: a recipe body MAY contain at most one `ingredients_step`, and the line itself accepts a sequence of space-separated quoted items but no escape for line continuation. There is no whitespace-tolerant alternative shape.

**Symptom.** Recipes whose input set spans many globs grow horizontally without bound. A real Cookfile that builds an Android APK depends on Kotlin sources, the manifest, the resource tree, and three build-script files â€” seven globs producing a 200+ character `ingredients` line:

```cook
recipe debug-apk
    ingredients "app/src/**/*.kt" "app/src/main/AndroidManifest.xml" "app/src/main/res/**/*" "app/build.gradle.kts" "build.gradle.kts" "settings.gradle.kts" "gradle.properties"
```

The natural reflex is `\`-continuation, which the lexer rejects (the trailing `\` is consumed as part of the preceding STRING or as a `Token::Content` byte). The natural fallback is two `ingredients` lines, which Â§{recipes.ingredients}'s at-most-one rule rejects with a diagnostic that points at the second line.

**Status quo.** Authors either accept long lines or split the recipe (which is wrong, since the multiple recipes share an output). The line-length friction makes diffs noisier than they need to be â€” a one-glob-added change re-wraps the entire line in editor displays and shows up as a whole-line replacement in `git diff`.

**Possible resolutions.** Three options, increasing in scope:

1. **Accept `\` line continuation on `ingredients` lines.** Smallest change. Lexer rule: when an `ingredients_step` line ends with `\` and a newline, fold the next line into the current line for purposes of step parsing. Diagnostic for `\`-then-non-newline is unchanged. Conformance impact: one positive fixture exercising a multi-line continuation. App. A grammar gains a footnote on the `ingredients_step` production. This option matches widespread shell convention and is unsurprising.
2. **Accept a bracketed list form.** `ingredients [ "a" "b" "c" ]` with whitespace (including newlines) freely admitted between items inside the brackets. Cleaner than `\`-continuation and consistent with "every block-shaped surface in cook uses delimiters" (`{}` for shell/Lua bodies). Spec impact is larger: a new ingredient-list shape in App. A.5, two new conformance fixtures (positive: bracketed list; negative: bracketed list mixed with the bare-string form on the same line), and Â§{recipes.ingredients} grows a paragraph.
3. **Lift the at-most-one-ingredients rule.** Allow a recipe body to contain multiple `ingredients` lines whose effects union (with the existing exclude semantics applying across all of them). Smallest *spec* change in word count, but contradicts the existing rule's intent (the at-most-one rule was presumably chosen for diagnostic clarity â€” "the input set lives on this one line"). This option is the least preferred of the three.

Option 1 is the least disruptive to existing Cookfiles and matches the muscle memory of every author who has used Make, shell, or Python. Option 2 is the most consistent with cook's overall "blocks have delimiters" aesthetic. Option 3 should be skipped unless options 1 and 2 are both rejected for other reasons.

**Found in.** Same `glovemap` project as E.8.
