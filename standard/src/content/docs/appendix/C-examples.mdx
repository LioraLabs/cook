# Appendix C. Worked examples (informative)

> **Informative.** This appendix presents worked Cookfile examples, each annotated with the sections of the Standard it exercises. Examples are non-binding; they illustrate the normative rules defined in chapters 2–7 and Appendix A.

## C.1. Multi-output cook step (from `examples/multi-output/Cookfile`) [#examples.multi-output-cook]
```cook
recipe generate
    ingredients "src/*.rs"
    cook "staging/out.js" "staging/out.wasm" using {
        ./fake-generator.sh
        mkdir -p staging
        cp pkg/out.js staging/out.js
        cp pkg/out.wasm staging/out.wasm
    }
```

Exercises: §{recipes.header-forms} (explicit header), §{recipes.ingredients} (ingredients), §{recipes.cook-multi-output} (multi-output cook with `using { ... }` shell block), App. A.3, App. A.4.

## C.2. Recipe with cross-recipe dependency (from `examples/cross-recipe-deps/Cookfile`) [#examples.cross-recipe-dep]
```cook
recipe tools: colorize wordcount reverse

recipe play: tools
    @mkdir -p .cook/scratch
    @printf 'colorize\nwordcount\nreverse\n' | fzf --prompt='tool> ' > .cook/scratch/tool
    @find data -type f | fzf --prompt='file> ' --preview='cat {}' > .cook/scratch/file
    @./build/bin/$(cat .cook/scratch/tool) < "$(cat .cook/scratch/file)"
```

Exercises: §{recipes.header-forms} (explicit header with deps), §{recipes.dep-list} (dependency list), §{recipes.shell-steps} (interactive `@` shell commands), App. A.3, App. A.4.

## C.3. Module use and call (from `examples/fzf-picker/Cookfile`) [#examples.module-use-call]

## C.4. Using-block Lua with multiple outputs (from `examples/multi-output/Cookfile`) [#examples.lua-multi-output]
```cook
recipe build_lua
    ingredients "src/*.rs"
    cook "staging-lua/out.js" "staging-lua/out.wasm" using >{
        os.execute("mkdir -p staging-lua")
        local sources = table.concat(inputs, " ")
        local js = io.open(outputs[1], "w")
        js:write("// generated from " .. sources .. "\n")
        js:close()
        local wasm = io.open(outputs[2], "w")
        wasm:write("BINARY\n")
        wasm:close()
    }
```

Exercises: §{recipes.cook-multi-output} (multi-output cook with `using >{ ... }` Lua block), §{lua.using-block-globals} (using-block globals `inputs` / `outputs`), App. A.4.
## C.5. Cross-recipe references [#examples.cross-recipe-refs]
This example exercises the four surfaces of cross-recipe references defined in §{xref}.

### Bare string substitution

```cook
recipe libmath
    ingredients "src/math/*.c"
    cook "build/obj/math/{stem}.o" using "gcc -c {in} -o {out}"
    cook "build/lib/libmath.a" using "ar rcs {out} {all}"

recipe app
    cook "build/bin/app" using "gcc -o {out} main.c {libmath}"
```

In `app`, the `{libmath}` placeholder is a bare name reference (§{xref.string-substitution}). It expands at register time to the single-element output list of `libmath`, i.e., `"build/lib/libmath.a"`. The resulting `app` step's command is `gcc -o build/bin/app main.c build/lib/libmath.a`.

### Dep-driven iteration

```cook
recipe protos
    ingredients "proto/*.proto"

recipe compile_protos
    cook "build/obj/{protos.stem}.pb.cc" using "protoc --cpp_out=build/obj {in}"
```

In `compile_protos`, the `{protos.stem}` placeholder in the output pattern (§{xref.dep-driven}) is an accessor placeholder. It names `protos` as the iteration driver. The step registers one work unit per item in `protos`'s output list — which, because `protos` has no cook step, is the resolved glob `proto/*.proto` passed through (§{xref.dep-recipe-output}). For each resolved path `x`, `{in}` = `x`, `{stem}` = `path.stem(x)`, and `{protos.stem}` = `path.stem(x)`.

### Ingredients-only recipe referenced by another

```cook
recipe headers
    ingredients "include/*.h"

recipe check: headers
    test "clang-tidy {headers}"
```

`headers` is a passthrough recipe with no cook step; its output is the resolved glob (§{xref.dep-recipe-output}). `{headers}` in `check`'s test command expands to the space-joined list. The `: headers` dep-list reference (§{xref.name-references} dep-list surface) creates a whole-recipe edge (§{xref.dep-implications}); `{headers}` in the test command is redundant as an edge but fine for string substitution.

### Mixed: one driver plus string substitution

```cook
recipe core
    cook "build/lib/libcore.a" using "…"

recipe protos
    ingredients "proto/*.proto"

recipe server
    cook "build/obj/{protos.stem}.o" using "gcc -c {in} -I{core}/include -o {out}"
```

The `server` step is dep-driven by `protos` (output pattern contains `{protos.stem}`). The `{core}` in the using-string is a bare string-substitution reference (§{xref.string-substitution}) to `core`'s single output. The step thus depends on both `protos` (per-unit, driving iteration) and `core` (whole-recipe or refined per-unit, per §{xref.dep-implications}) — two distinct cross-recipe edges from a single step.
