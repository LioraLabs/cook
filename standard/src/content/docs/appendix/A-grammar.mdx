# Appendix A. Grammar (normative)

> **Normative.** This appendix contains the full formal grammar of the Cookfile language in the W3C-style EBNF variant defined in §{notation.grammar}. Where this appendix disagrees with the prose of any chapter of Part I, this appendix is authoritative (§{notation.grammar-precedence}).

## A.1. Top level [#grammar-appendix.top-level]
```ebnf
cookfile              ::= toplevel_item*
toplevel_item         ::= recipe
                       | chore
                       | probe_decl
                       | config_block
                       | use_declaration
                       | import_declaration
                       | register_block
                       | top_level_module_call
                       | comment
                       | NEWLINE

register_block         ::= "register" NEWLINE register_body
register_body          ::= LUA_SOURCE       (* see §{lexical.source-representation}, §{decl.register} *)
top_level_module_call  ::= module_call  (* column-0; multi-line via brace-balance per §{lexical.brace-blocks.lua-spans} *)
```

## A.2. Top-level declarations [#grammar-appendix.declarations]
```ebnf
use_declaration       ::= "use" use_name NEWLINE
use_name              ::= LUA_IDENT | ( '"' LUA_IDENT '"' )
import_declaration    ::= "import" BARE_IDENTIFIER import_path NEWLINE
config_block          ::= "config" name? NEWLINE config_body
config_body           ::= LUA_SOURCE
```

**`use_name` constraint (normative; CS-0035).** The `use_name` MUST match the `LUA_IDENT` production (§{grammar-appendix.primitives}), regardless of whether it is written bare or inside double quotes. A conforming implementation MUST reject any `use_declaration` whose name contains characters outside `[A-Za-z0-9_]`, does not start with `[A-Za-z_]`, or is empty — including names containing whitespace, hyphens (`-`), or dots (`.`). The diagnostic MUST identify the offending name and source line. The restriction follows from the `use` binding (§{mods.use}): the name is bound at load time as a Lua local under the same spelling, and any non-identifier spelling would yield malformed generated Lua. Hyphens are rejected outright (rather than rewritten to underscores) to eliminate silent collisions between names like `foo-bar` and `foo_bar`.

**Ordering (normative).** `use_declaration`, `import_declaration`, and `config_block` MUST appear before the first `recipe` or `chore`. A conforming implementation MUST reject a Cookfile that places any of these after a `recipe` or `chore`.

**Body termination (normative).** A `config_body`'s extent is determined by the implicit-termination rule of §{toplevel.termination}: it ranges from the line after the header to the next column-0 line classified as `recipe`, `chore`, `config`, `use`, `import`, or `register`, or to a column-0 line matching the top-level module_call shape, or to end-of-file. A bare `end` line is a `Content` token (§{lexical.line-classification}) and does NOT terminate the body. A conforming implementation MUST NOT require an explicit `end` line.

**Duplicate `config_block` rule.** At most one unnamed `config_block` is permitted per Cookfile. Named `config_block` names MUST be unique. A conforming implementation MUST reject a Cookfile that violates either constraint.

**Duplicate `import_declaration` rule.** The `BARE_IDENTIFIER` (local alias) of each `import_declaration` MUST be unique within a Cookfile. A conforming implementation MUST reject duplicate import names.

**Recipe / chore declaration names — no dots.** The `name` of a `recipe_header` or `chore_header` MUST NOT contain `.` (dot). A conforming implementation MUST reject a top-level declaration `recipe NAME` or `chore NAME` whose `NAME` (whether bare or quoted) contains a dot, at parse time. Diagnostic: `recipe name 'NAME': dotted recipe names are not permitted at the declaration site; use 'import alias path' for cross-Cookfile namespacing`. Namespacing across Cookfiles is achieved exclusively via `import_declaration` (§{comp.import}); a dotted reference appearing in a `dependency_list` or in a `$<...>` placeholder resolves through an `import` alias per §{comp.qualified-refs}.

**Duplicate recipe / chore declaration name rule.** Recipes and chores share a single callable namespace within a Cookfile. The `name` of every `recipe_header` and `chore_header` MUST be unique across the union of recipe and chore declarations. A conforming implementation MUST reject, at parse time, a Cookfile that contains:

1. two `recipe_header` declarations with the same `name` (recipe-vs-recipe collision); or
2. two `chore_header` declarations with the same `name` (chore-vs-chore collision); or
3. one `recipe_header` and one `chore_header` with the same `name` (recipe-vs-chore collision).

Names compare verbatim — bare and quoted forms of the same name (e.g. `recipe build` and `recipe "build"`) refer to the same callable and therefore collide. The diagnostic MUST identify the offending kind, the colliding name, and the source line of the prior declaration. The recommended diagnostic form is `KIND 'NAME': duplicate declaration (already declared as PRIOR_KIND at line N)`, where `KIND` is the kind of the second declaration and `PRIOR_KIND` is the kind of the first. Cross-Cookfile namespacing for callables that would otherwise collide is achieved via `import_declaration` aliases per §{comp.import}.

**Reserved recipe name identifiers.** A recipe or chore declaration name MUST NOT equal one of the reserved identifiers `stem`, `name`, `ext`, `dir`, `in`, `out`, `all`, `env`. (Combined with the no-dots rule above, this is the entirety of the reserved-name constraint at the declaration site.) The same eight identifiers MUST NOT appear as the *final* dot-separated segment of a `dependency_list` entry or a `$<...>` placeholder, and `env` MUST NOT appear as the *first* dot-separated segment of either. This reserves the `env.` prefix for the explicit env-var-lookup form (§{xref.env-namespace}).

## A.3. Recipes [#grammar-appendix.recipes]
```ebnf
recipe                ::= recipe_header NEWLINE recipe_body
recipe_header         ::= "recipe" name (":" dependency_list)?
dependency_list       ::= name+
recipe_body           ::= declarative_region imperative_region?
declarative_region    ::= declarative_item*
declarative_item      ::= ingredients_step
                       | seal_step
                       | cook_step
                       | plate_step
                       | test_step
                       | inline_lua_line
                       | inline_lua_block
                       | comment
                       | NEWLINE
imperative_region     ::= imperative_item+
imperative_item       ::= lua_line
                       | lua_block
                       | shell_command
                       | interactive_command
                       | comment
                       | NEWLINE
```

**At-most-one-ingredients rule.** A `recipe_body` MUST contain at most one `ingredients_step`. A conforming implementation MUST reject a recipe with two or more `ingredients_step` items.

**Region ordering rule.** A conforming implementation MUST reject a `recipe_body` in which a `declarative_item` (other than `comment` or `NEWLINE`) appears after the first `imperative_item` (other than `comment` or `NEWLINE`) of the same recipe. The diagnostic MUST identify the offending step's line and the line on which the imperative region began. See §{steps.overview}, Note 4.4.2 for the prose treatment.

**Body termination (normative).** A `recipe_body`'s extent is determined by the implicit-termination rule of §{toplevel.termination}: it ranges from the line after the header to the next column-0 line classified as `recipe`, `chore`, `config`, `use`, `import`, or `register`, or to a column-0 line matching the top-level module_call shape, or to end-of-file. A bare `end` line is a `Content` token (§{lexical.line-classification}) and does NOT terminate the body. A conforming implementation MUST NOT require an explicit `end` line.

### A.3.1. Chore [#grammar-appendix.chore]

```ebnf
chore                    ::= chore_header NEWLINE chore_body
chore_header             ::= "chore" name param_list (":" dependency_list)?
param_list               ::= positional_params variadic_param?
positional_params        ::= required_param* defaulted_param*
required_param           ::= BARE_IDENTIFIER
defaulted_param          ::= BARE_IDENTIFIER "=" default_value
default_value            ::= STRING | "(" LUA_EXPR ")"
variadic_param           ::= ("+" | "*") BARE_IDENTIFIER
chore_body               ::= chore_declarative_region chore_imperative_region?
chore_declarative_region ::= chore_declarative_item*
chore_declarative_item   ::= inline_lua_line
                           | inline_lua_block
                           | comment
                           | NEWLINE
chore_imperative_region  ::= chore_imperative_item+
chore_imperative_item    ::= lua_line
                           | lua_block
                           | shell_command
                           | interactive_command
                           | comment
                           | NEWLINE
```

**Chore parameter ordering rule (normative).** Within a `param_list`, every `required_param` MUST precede every `defaulted_param`, and the optional `variadic_param` MUST be the final parameter. A conforming implementation MUST reject a header that violates this order.

**Chore parameter reserved-name rule (normative).** Parameter names MUST NOT equal a reserved identifier from §A.2 (`stem`, `name`, `ext`, `dir`, `in`, `out`, `all`, `env`). Parameter names MUST NOT contain `.`. Duplicate parameter names within one header MUST be rejected at parse time.

**Variadic rules (normative).** A `variadic_param` with sigil `+` (one-or-more) MUST be the final parameter; at invocation time, all remaining argv collect into a Lua sequence. If zero remain, a diagnostic MUST be raised (§7.1.3). A `variadic_param` with sigil `*` (zero-or-more) behaves identically except that zero remaining argv is valid and binds `{}`. At most one `variadic_param` is permitted per `param_list`; a `variadic_param` MUST NOT carry an `=default`.

**Chore step-kind ban (normative).** A `chore_body` MUST NOT contain `ingredients_step`, `cook_step`, `plate_step`, or `test_step`. A conforming implementation MUST reject a chore body that contains any of these. The diagnostic MUST identify the offending step's keyword.

**Chore default-interactive (normative).** Every `shell_command` inside a `chore_body` MUST be treated as if it had been an `interactive_command` (§{grammar-appendix.steps}, `interactive_command`). The `@` prefix MAY be present (no-op) but is unnecessary.

**Chore no-caching (normative).** A unit registered from inside a chore body via `cook.add_unit` or the Cook Lua API's unit-recording helpers MUST be registered with `cache = false`. A conforming implementation MUST reject a register-phase call that would set `cache = true` (or equivalent) from within a chore body.

## A.3.2. Probe [#grammar-appendix.probe]

```ebnf
probe_decl     ::= "probe" probe_name (":" probe_dep_list)? NEWLINE INDENT probe_body DEDENT
probe_name     ::= BARE_PROBE_KEY | STRING
BARE_PROBE_KEY ::= PROBE_SEG (":" PROBE_SEG)?
probe_dep_list ::= probe_ref (WS probe_ref)*
probe_ref      ::= BARE_PROBE_KEY | STRING
IDENT          ::= /[A-Za-z_][A-Za-z0-9_]*/
PROBE_SEG      ::= /[A-Za-z_][A-Za-z0-9_.\-]*/   /* CS-0131: matches BARE_IDENTIFIER's inner charset */
probe_body       ::= ingredients_step? producer NEWLINE
producer         ::= ("json" | "lines")? shell_block
                   | ("tools" | "envs") name_list
                   | exec_lua_block
name_list        ::= "{" ( IDENT ((WS | ",") WS* IDENT)* )? "}"
```

`ingredients_step` is defined in §{grammar-appendix.steps}; `shell_block` and `exec_lua_block` are also defined there (the body-dispatch tests `>{` before `{`).

**Probe body (normative).** A `probe_body` comprises an optional `ingredients_step` followed by exactly one `producer`; the semantics, region rule, and rejection diagnostics are specified in §{cat.probes.decl}. The producer KIND leads — there is no `produce` introducer keyword. The `json` and `lines` kinds are valid ONLY when the producer is a `shell_block`; applied to an `exec_lua_block` (`>{ … }`) they MUST be rejected, since a Lua block already returns a structured value (§{cat.probes.decl}). `json`, `lines`, `tools`, and `envs` are contextual keywords recognised only in this probe-producer position.

**Source-list kinds (normative).** The `tools` and `envs` forms take a
`name_list` — a brace-delimited, comma/whitespace-separated list of bare `IDENT`
names — NOT a `shell_block` or `exec_lua_block`. A conforming implementation MUST
reject an empty `name_list`, a name that is not an `IDENT`, and a body (in
particular a `>{ … }` `exec_lua_block`) supplied where a `name_list` is required
(§{cat.probes.decl}). The `name_list` is written on a single physical line
(unlike a multi-line `shell_block`).

**Module-prefix colon disambiguation (normative).** The EBNF above admits an apparent ambiguity: the `:` in `BARE_PROBE_KEY` and the `:` that introduces `probe_dep_list` are spelled identically. The rule is purely positional:

- A `:` that is immediately preceded by an `IDENT` segment **and** immediately followed (with no intervening whitespace) by an `IDENT`-start character is part of `BARE_PROBE_KEY` — the *module-prefix colon*. Example: `cc:zlib` is a single probe name.
- A `:` that is preceded by a complete `probe_name` **and** followed by whitespace or end-of-line introduces `probe_dep_list`. Example: `probe cards: dep1 dep2` has name `cards` and deps `[dep1, dep2]`.
- A contiguous third `:IDENT` segment (e.g. `a:b:c`) MUST be rejected. A conforming implementation MUST produce a diagnostic identifying the malformed name and source line. The diagnostic form is: `probe name 'NAME': a probe key is at most \`IDENT:IDENT\`; 'NAME' has too many ':' segments`.
- The quoted `STRING` form (e.g. `probe "cc:zlib"`) is always unambiguous and bypasses the positional rule entirely.

**Probe-key segment charset (normative; CS-0131).** Each `PROBE_SEG` of a `BARE_PROBE_KEY` uses the recipe-name character set (`BARE_IDENTIFIER`'s inner class — internal `-` and `.` admitted), NOT the narrow `IDENT` used by the `tools` / `envs` `name_list`. A conforming implementation MUST tokenise a hyphenated bare probe key (`demo:cc-version`) as a single `probe_name`, and MUST accept a hyphenated bare `probe_ref` in the dependency list (`probe x: demo:cc-path`). The module-prefix colon is still the sole segment separator — `-` and `.` never terminate a segment. The `STRING` form remains an exact-equivalent escape hatch for the same key (see the `probe_dep_list` element-shape note below); `"demo:cc-version"` and bare `demo:cc-version` denote the identical probe key.

**Missing probe name (normative).** If the `probe` keyword is followed by content that does not parse as a valid `probe_name` (e.g. `probe :deps`), the implementation MUST reject the input with a diagnostic of the form: `line N: expected a probe name after \`probe\``.

**Unexpected token after probe name (normative).** If the `probe` keyword is followed by a valid `probe_name` and then by content that is neither `:` (introducing `probe_dep_list`) nor end-of-line (e.g. `probe foo bar`), the implementation MUST reject the input with a diagnostic of the form: `line N: probe 'NAME': unexpected token after the probe name; a dependency list is introduced with ':' (e.g. \`probe NAME: dep1 dep2\`)`. The diagnostic MUST NOT identify the error as a recipe error.

**`probe_dep_list` element shape (normative).** Each element of `probe_dep_list` is a `probe_ref`, which follows the same `BARE_PROBE_KEY | STRING` shape as `probe_name`. Module-prefixed dep references (e.g. `cc:compiler`) are therefore syntactically valid in the dependency list. A conforming implementation MUST accept the `STRING` form in dependency position (`probe x: "cc:zlib"`), not only bare `BARE_PROBE_KEY` — the quoted arm is the escape hatch for keys a bare segment cannot spell (CS-0131).

**Column-zero constraint (normative).** The `probe` keyword is recognised as a header only when it appears at column zero (the line does not begin with whitespace). An indented `probe` line is classified as `Content` per §{lexical.line-classification}.

## A.4. Steps [#grammar-appendix.steps]
```ebnf
ingredients_step      ::= "ingredients" ( ingredient+ | probe_ref ) NEWLINE
                          /* probe_ref form is a probe member source (mutually exclusive
                             with glob items); see §8 Note 8.2.1 */
ingredient            ::= STRING            /* include */
                       | "!" STRING          /* exclude */

seal_step             ::= ("seal" | "unseal") probe_ref+ NEWLINE
                          /* recipe-body step; a determinant input stream,
                             sibling of ingredients_step. Recipe-level `unseal`
                             is rejected (trailing-only) — see §8.4.3. */
cook_step             ::= "cook" cook_output body? cook_mods NEWLINE
cook_output           ::= STRING+ | "(" LUA_EXPR ")"
exec_lua_block        ::= ">{" LUA_BLOCK_CONTENT "}"
shell_block           ::= "{" SHELL_BLOCK_CONTENT "}"
cook_mods             ::= ("seal" probe_ref+ | "unseal" probe_ref+)* share_mod?
share_mod             ::= "local" | "pinned" | "nondet"
                          /* a probe_ref in seal_step / cook_mods is a bare
                             BARE_PROBE_KEY (NOT the STRING form), exactly as in
                             §{grammar-appendix.probe}; the quoted form and a
                             third `:IDENT` segment are rejected — see §8.4.3 */

plate_step            ::= "plate" body NEWLINE
test_step             ::= "test"  body test_modifiers NEWLINE
body                  ::= exec_lua_block | shell_block
test_modifiers        ::= ("as" STRING)? ("timeout" NUMBER)? "should_fail"?

lua_line              ::= ">" LUA_LINE_CONTENT? NEWLINE
lua_block             ::= ">{" LUA_BLOCK_CONTENT "}" NEWLINE

inline_lua_line       ::= ">>" LUA_LINE_CONTENT? NEWLINE
inline_lua_block      ::= ">>{" LUA_BLOCK_CONTENT "}" NEWLINE

interactive_command   ::= "@" SHELL_LINE_CONTENT NEWLINE

module_call           ::= MODULE_CALL_TEXT NEWLINE?
                          /* MODULE_CALL_TEXT is any Content line whose first
                             token matches /[A-Za-z_][A-Za-z0-9_]*/ (alphanumeric
                             and underscore only; hyphens and dots are NOT
                             permitted in the first segment), followed by a
                             literal ".", followed by at least one character
                             matching /[A-Za-z_]/ (ident-start). The remainder
                             of the second segment is not validated at dispatch
                             time; the full Lua-expression-hood of the line is
                             the Lua runtime's concern. Braces may span
                             subsequent lines (see § 2.9). */

shell_command         ::= SHELL_LINE_CONTENT NEWLINE
                          /* Any Content line not matching a preceding
                             alternative. See § 3.8. */
```

**Multi-line patterns (CS-0078).** In both `ingredients_step` and `cook_step`, whitespace between successive `STRING` (or `ingredient`) tokens MAY include line breaks. A conforming implementation MUST accept continuation lines that begin with `"` (or `!"` in the case of `ingredients_step`) as part of the same declaration. Pattern collection terminates at the first non-quote token on the same physical line as a quoted pattern, which then resolves per the surrounding production (the body opener `{` / `>{`, or — when no body follows — a trailing `cook_mods` token, for `cook_step`; end-of-declaration for `ingredients_step`); a non-quote token on the same line as a quoted pattern that is neither a legal terminator nor a valid `cook_mods` token MUST produce a diagnostic naming the offending text. A continuation line whose first token after leading whitespace is not `"` (or `!"` for `ingredients_step`) terminates the declaration silently — the declaration is treated as complete, and the line is dispatched per §A.4's step-dispatch-priority order. The latter case is genuinely ambiguous between an intended-but-misspelled continuation and the start of a new step; the Standard chooses non-diagnosis to preserve the unambiguous declaration-only form of `cook_step` (App. A.4, "Declaration-only cook step") and the analogous "ingredients without trailing patterns" idiom.

### Example A.4.1

Both forms are equivalent:

```
cook "a.o" "b.o" "c.o" { ... }
```

```
cook "a.o"
     "b.o"
     "c.o" { ... }
```

Similarly for `ingredients`:

```
ingredients "src/*.c" !"src/legacy.c"
```

```
ingredients "src/*.c"
            !"src/legacy.c"
```

**Multi-output cook-step rule.** When `cook_step` has two or more `STRING` output patterns before the body, the body resolves per the unrestricted production above — the bare-`STRING` arm having been removed in CS-0022, a trailing `STRING` simply joins the output-pattern list (declaration-only form) rather than being read as a body.

**Iteration coherence (CS-0022).** All `STRING` output patterns of a single `cook_step` MUST share an iteration source: every output is literal (no accessor placeholder), or every output bears `$<in.ACCESSOR>` (own-input driver), or every output bears `$<lib.ACCESSOR>` for the same recipe `lib` (dep-driven driver). A conforming implementation MUST reject a `cook_step` whose output patterns mix sources, with a diagnostic naming the offending outputs. See §{steps.iteration-mode}.

**Plate/test mode coherence (CS-0024).** A `plate_step` or `test_step` body MUST NOT contain both an iteration-item placeholder/binding and a batched-source placeholder/binding. For shell-block bodies, this means a body MUST NOT contain both `$<in>` (or `$<in.ACCESSOR>`) and `$<all>`. For Lua-block bodies, the body MUST NOT reference both `input` and `inputs` (free-identifier scan, ignoring strings/comments/long-strings per §{lexical.brace-blocks}). A conforming implementation MUST reject a body that violates this rule with a diagnostic naming both tokens. See §{steps.iteration-mode-plate-test}.

**Plate/test source presence (CS-0024).** A `plate_step` or `test_step` in one-to-one or many-to-one mode MUST have a non-empty iteration source — that is, the recipe MUST contain a preceding `cook_step` whose output list is non-empty, or an `ingredients_step` whose resolved path list is non-empty. A conforming implementation MUST reject a `plate_step` or `test_step` whose mode requires source consultation but whose recipe provides none. A `plate_step` or `test_step` in one-shot mode does not consult the source and MAY appear in a recipe with neither.

**`as` modifier is test_step-only (CS-0061).** The `as STRING` modifier is syntactically valid only on `test_step`. A conforming implementation MUST reject `as` appearing after the body of a `cook_step` or `plate_step`, with a diagnostic naming the modifier and identifying it as `test_step`-only.

**`>>{` is rejected as a body.** A `cook_step` body MUST NOT use the `>>{` (register-phase Lua block) prefix. The body produces work for execute, not register-phase orchestration; a register-phase Lua block in this position is incoherent. A conforming implementation MUST reject `cook STR >>{ … }`.

**Body dispatch order (normative).** After the output pattern(s), a conforming implementation checks in this order: `>>{` (rejected as above), `>{` (Lua block), `{` (shell block). The `>>{` test is present so the diagnostic for the rejected form is sharp; the first matching admitted alternative wins for the others. The bare-`STRING` form is no longer admitted (CS-0022).

**`using` is rejected (CS-0099).** The former body introducer `using` (`cook STR { … }` / `cook STR >{ … }`, removed in CS-0099) MUST be rejected at the body position with a migration diagnostic that names CS-0099 and states that the keyword was removed. The body opener follows the output pattern(s) directly. Conformance fixture `cs0099-using-keyword-rejected` pins the diagnostic.

**Declaration-only cook step.** A `cook_step` whose body is absent is a declaration — it announces the output without providing a build command inline. Its build is assumed to be provided by preceding Lua registrations or a later amendment.

**Cook-step disposition (normative).** A unit's cache participation is declared by two crisp surfaces (COOK-171): `seal`/`unseal` **steps** (a recipe-body determinant input stream) and the trailing **`cook_mods`** on a `cook_step`. The full surface and its parse-time rules are normative in §{steps.cook-disposition}; in summary, a conforming implementation MUST enforce: a recipe-level `seal` step requires at least one `probe_ref` (bare `seal` rejected), is additive across the recipe, and contributes to a recipe-wide baseline applied to every cook regardless of position (order-independent); a recipe-level `unseal` is rejected (`unseal` is a trailing modifier only); a `cook_step`'s effective seal set is `(baseline ∪ trailing seals) − trailing unseals`, sorted and de-duplicated; the trailing `share_mod` is a single optional slot (`local`, `pinned`, or `nondet`) which MUST be last (at most one — `local`/`pinned`/`nondet` are mutually exclusive by construction); a `probe_ref` is a bare `BARE_PROBE_KEY` (the `STRING` form and a third `:IDENT` segment are rejected, as in §{grammar-appendix.probe}); and the removed `record` keyword (renamed `nondet`) is rejected with a migration diagnostic. `seal`, `unseal`, `local`, `pinned`, and `nondet` are NOT reserved words; `seal`/`unseal` are recognised as steps only at a recipe-body line position, and `local`/`pinned`/`nondet` only in the trailing `cook_mods` position.

**Step-dispatch priority (normative).** A single `Content` line inside a recipe body is dispatched in this order, with the first match winning:

1. Prefix `ingredients` + separator (space, tab, or end of token) → `ingredients_step`.
2. Prefix `seal` + separator → recipe-level `seal_step` (its `probe_ref`s contribute to the recipe seal baseline).
3. Prefix `unseal` + separator → rejected as a recipe-level step (`unseal` is a trailing `cook_mods` modifier only; §{steps.cook-disposition}).
4. Prefix `cook` + separator → `cook_step` (its trailing `cook_mods` are parsed after the body, or after the output patterns for a declaration-only cook).
5. Prefix `plate` + separator → `plate_step`.
6. Prefix `test` + separator → `test_step`.
7. First character is `@` and the remainder is non-empty → `interactive_command`.
8. Otherwise → `shell_command`. (A line whose first token is `local`, `pinned`, or `nondet` is NOT a disposition at this position — those keywords are recognised only in the trailing `cook_mods` of a `cook_step` — so such a line falls through to `shell_command`.)

Note: CS-0072 removed the former rule 5 that classified a `Content` line matching `BARE_IDENTIFIER "." BARE_IDENTIFIER(…)` as `module_call` (register-phase Lua) from the recipe-body cascade. Top-level module_call dispatch is now governed by §{toplevel.module-call}; authors who want register-phase Lua inside a recipe body MUST use the `>>` or `>>{` prefix.

## A.5. Primitives [#grammar-appendix.primitives]
```ebnf
name                  ::= BARE_IDENTIFIER | STRING
BARE_IDENTIFIER       ::= /[A-Za-z_][A-Za-z0-9_.\\-]*/
LUA_IDENT             ::= /[A-Za-z_][A-Za-z0-9_]*/
STRING                ::= /"[^"]*"/
NUMBER                ::= /[0-9]+/
import_path           ::= sigil_path | tree_path
sigil_path            ::= "//" path_segments
tree_path             ::= ( "./" | path_segment ) ( "/" path_segment )*
path_segments         ::= path_segment ( "/" path_segment )*
path_segment          ::= /[^\s\n\/]+/
path                  ::= /[^\s\n]+/   /* legacy form; import_path is the canonical path */

**No '..' segment (normative).** A `path_segment` MUST NOT consist solely of two dot characters (`..`). The grammar's regex permits `..` syntactically; this rule eliminates it normatively. See §{comp.import}.
comment               ::= /#[^\n]*/ NEWLINE
NEWLINE               ::= /\n/
LUA_LINE_CONTENT      ::= /[^\n]*/   /* zero or more characters; may be empty */
LUA_BLOCK_CONTENT     ::= BRACE_BALANCED   /* see § 2.9 */
SHELL_LINE_CONTENT    ::= /[^\n]+/
SHELL_BLOCK_CONTENT   ::= BRACE_BALANCED   /* see § 2.9 */
LUA_SOURCE            ::= /* any sequence of lines up to the next column-0
```

**Placeholder shape.** Placeholders embedded in `cook_step` output patterns and `SHELL_BLOCK_CONTENT` bodies are not separately tokenised by the Cookfile grammar — they are raw character sequences inside a `STRING` or `SHELL_BLOCK_CONTENT` production. Their well-formedness is validated at codegen time per §{phl.cook-step}. The single recognised form is the strict sigil `$<IDENT>`, normatively defined in §{phl.token} (CS-0033). Its production is reproduced here for reference in the context of surrounding grammar productions:

```ebnf
sigil_placeholder       ::= "$<" placeholder_ident ">"
placeholder_ident       ::= bare_ident | out_indexed | out_indexed_acc | file_ref
bare_ident              ::= ALPHA (ALPHA | DIGIT | "_" | ".")*
out_indexed             ::= "out_" DIGIT+
out_indexed_acc         ::= "out_" DIGIT+ "." ACCESSOR
file_ref                ::= "file:" file_path_char+
file_path_char          ::= ALPHA | DIGIT | "." | "-" | "/" | "*"
ALPHA                   ::= "a"…"z" | "A"…"Z" | "_"
DIGIT                   ::= "0"…"9"
```

The `ACCESSOR` production is defined above. The `file_ref` alternative (CS-0101) is the file-reference form `$<file:PATH>` of §{phl.cook-step}: after the literal prefix `file:`, the `file_path_char` set extends the `bare_ident` character set with the path separator (`/`) and the glob star (`*`). The `sigil_placeholder` production can appear inside `output_pattern`, `SHELL_BLOCK_CONTENT`, `SHELL_LINE_CONTENT`, and `plate_test_shell_block_body` text (the `file_ref` alternative is rejected in `output_pattern` positions per §{phl.cook-step}, CS-0101). Resolution is defined by §{xref.resolution} and §{phl.cook-step}; an implementation MAY record byte ranges of placeholder positions in its parse tree to support migration tooling such as `cook --migrate-sigil`.

**`BARE_IDENTIFIER` vs `name`.** Recipe and dependency names can be either a `BARE_IDENTIFIER` (allowing dots and hyphens as internal characters) or a double-quoted `STRING`. The `import` declaration's local alias is always a `BARE_IDENTIFIER` (no quoting).

**`LUA_IDENT` vs `BARE_IDENTIFIER` (CS-0035).** A `LUA_IDENT` is the strict Lua identifier shape `/[A-Za-z_][A-Za-z0-9_]*/` — narrower than `BARE_IDENTIFIER`, which additionally admits internal dots and hyphens. The `LUA_IDENT` production is used wherever the matched text is bound as a Lua local at load time and therefore needs to round-trip through the Lua identifier grammar; the `use_declaration`'s `use_name` (§{grammar-appendix.declarations}) is currently the sole site.

**`lua_line` MAY be empty.** The `>` prefix alone (with no following content) is a valid `lua_line` producing an empty Lua expression.

**Brace-balance algorithm.** Both `LUA_BLOCK_CONTENT` and `SHELL_BLOCK_CONTENT` use a stateful brace-balance algorithm whose interior-language opaque-span rules differ. For Lua-block content, braces are ignored inside Lua line comments (`--` to end of line), double-quoted strings, single-quoted strings, multi-line long strings (`[==[...]==]` of any `=`-level), and multi-line block comments (`--[==[...]==]` of any `=`-level). For shell-block content, braces are ignored inside single- and double-quoted strings, after a backslash-escape (`\{`, `\}`), and inside POSIX heredocs (`<<TAG`, `<<-TAG`, `<<'TAG'`, `<<"TAG"`). The brace counter carries multi-line opaque-span state (long strings, block comments, heredocs) across lines per CS-0035. The block closes on the line where depth first returns to zero or below; that closing line is consumed but not included in the block body.

**Shell block line normalisation.** Each line of `SHELL_BLOCK_CONTENT` is trimmed of leading and trailing whitespace; blank lines after trimming are discarded. The result is a sequence of command strings, not a single raw string.
