# Appendix A. Grammar (normative)

> **Normative.** This appendix contains the full formal grammar of the Cookfile language in the W3C-style EBNF variant defined in §{notation.grammar}. Where this appendix disagrees with the prose of §{grammar} or §{recipes}, this appendix is authoritative (§{notation.grammar-precedence}).

## A.1. Top level [#grammar-appendix.top-level]
```ebnf
cookfile              ::= toplevel_item*
toplevel_item         ::= recipe
                       | chore
                       | config_block
                       | use_declaration
                       | import_declaration
                       | comment
                       | NEWLINE
```

## A.2. Top-level declarations [#grammar-appendix.declarations]
```ebnf
use_declaration       ::= "use" name NEWLINE
import_declaration    ::= "import" BARE_IDENTIFIER path NEWLINE
config_block          ::= "config" name? NEWLINE config_body
config_body           ::= LUA_SOURCE
```

**Ordering (normative).** `use_declaration`, `import_declaration`, and `config_block` MUST appear before the first `recipe` or `chore`. A conforming implementation MUST reject a Cookfile that places any of these after a `recipe` or `chore`.

**Body termination (normative).** A `config_body`'s extent is determined by the implicit-termination rule of §{recipes.termination}: it ranges from the line after the header to the next column-0 line classified as `recipe`, `chore`, `config`, `use`, or `import`, or to end-of-file. A bare `end` line is a `Content` token (§{lexical.line-classification}) and does NOT terminate the body. A conforming implementation MUST NOT require an explicit `end` line.

**Duplicate `config_block` rule.** At most one unnamed `config_block` is permitted per Cookfile. Named `config_block` names MUST be unique. A conforming implementation MUST reject a Cookfile that violates either constraint.

**Duplicate `import_declaration` rule.** The `BARE_IDENTIFIER` (local alias) of each `import_declaration` MUST be unique within a Cookfile. A conforming implementation MUST reject duplicate import names.

**Reserved recipe name segments.** The identifiers `stem`, `name`, `ext`, `dir`, `in`, `out`, `all` are reserved and MUST NOT appear as the final dot-separated segment of any recipe name. A conforming implementation MUST reject a recipe whose name ends with one of these segments (e.g., `backend.stem` is rejected; `backend.build` is accepted).

## A.3. Recipes [#grammar-appendix.recipes]
```ebnf
recipe                ::= recipe_header NEWLINE recipe_body
recipe_header         ::= "recipe" name (":" dependency_list)?
dependency_list       ::= name+
recipe_body           ::= declarative_region imperative_region?
declarative_region    ::= declarative_item*
declarative_item      ::= ingredients_step
                       | cook_step
                       | plate_step
                       | test_step
                       | inline_lua_line
                       | inline_lua_block
                       | module_call
                       | comment
                       | NEWLINE
imperative_region     ::= imperative_item+
imperative_item       ::= lua_line
                       | lua_block
                       | shell_command
                       | interactive_command
                       | comment
                       | NEWLINE
```

**At-most-one-ingredients rule.** A `recipe_body` MUST contain at most one `ingredients_step`. A conforming implementation MUST reject a recipe with two or more `ingredients_step` items.

**Region ordering rule.** A conforming implementation MUST reject a `recipe_body` in which a `declarative_item` (other than `comment` or `NEWLINE`) appears after the first `imperative_item` (other than `comment` or `NEWLINE`) of the same recipe. The diagnostic MUST identify the offending step's line and the line on which the imperative region began. See §{recipes.step-kinds}, Note 4.4.2 for the prose treatment.

**Body termination (normative).** A `recipe_body`'s extent is determined by the implicit-termination rule of §{recipes.termination}: it ranges from the line after the header to the next column-0 line classified as `recipe`, `chore`, `config`, `use`, or `import`, or to end-of-file. A bare `end` line is a `Content` token (§{lexical.line-classification}) and does NOT terminate the body. A conforming implementation MUST NOT require an explicit `end` line.

### A.3.1. Chore [#grammar-appendix.chore]

```ebnf
chore                    ::= chore_header NEWLINE chore_body
chore_header             ::= "chore" name (":" dependency_list)?
chore_body               ::= chore_declarative_region chore_imperative_region?
chore_declarative_region ::= chore_declarative_item*
chore_declarative_item   ::= inline_lua_line
                           | inline_lua_block
                           | module_call
                           | comment
                           | NEWLINE
chore_imperative_region  ::= chore_imperative_item+
chore_imperative_item    ::= lua_line
                           | lua_block
                           | shell_command
                           | interactive_command
                           | comment
                           | NEWLINE
```

**Chore step-kind ban (normative).** A `chore_body` MUST NOT contain `ingredients_step`, `cook_step`, `plate_step`, or `test_step`. A conforming implementation MUST reject a chore body that contains any of these. The diagnostic MUST identify the offending step's keyword.

**Chore default-interactive (normative).** Every `shell_command` inside a `chore_body` MUST be treated as if it had been an `interactive_command` (§{grammar-appendix.steps}, `interactive_command`). The `@` prefix MAY be present (no-op) but is unnecessary.

**Chore no-caching (normative).** A unit registered from inside a chore body via `cook.add_unit` or the Cook Lua API's unit-recording helpers MUST be registered with `cache = false`. A conforming implementation MUST reject a register-phase call that would set `cache = true` (or equivalent) from within a chore body.

## A.4. Steps [#grammar-appendix.steps]
```ebnf
ingredients_step      ::= "ingredients" ingredient+ NEWLINE
ingredient            ::= STRING            /* include */
                       | "!" STRING          /* exclude */

cook_step             ::= "cook" STRING+ using_clause? NEWLINE
using_clause          ::= "using" ( using_lua_block | shell_block | STRING )
using_lua_block       ::= ">{" LUA_BLOCK_CONTENT "}"
shell_block           ::= "{" SHELL_BLOCK_CONTENT "}"

plate_step            ::= "plate" STRING NEWLINE

test_step             ::= "test" STRING ("timeout" NUMBER)? "should_fail"? NEWLINE

lua_line              ::= ">" LUA_LINE_CONTENT? NEWLINE
lua_block             ::= ">{" LUA_BLOCK_CONTENT "}" NEWLINE

inline_lua_line       ::= ">>" LUA_LINE_CONTENT? NEWLINE
inline_lua_block      ::= ">>{" LUA_BLOCK_CONTENT "}" NEWLINE

interactive_command   ::= "@" SHELL_LINE_CONTENT NEWLINE

module_call           ::= MODULE_CALL_TEXT NEWLINE?
                          /* MODULE_CALL_TEXT is any Content line whose first
                             token matches /[A-Za-z_][A-Za-z0-9_]*/ (alphanumeric
                             and underscore only; hyphens and dots are NOT
                             permitted in the first segment), followed by a
                             literal ".", followed by at least one character
                             matching /[A-Za-z_]/ (ident-start). The remainder
                             of the second segment is not validated at dispatch
                             time; the full Lua-expression-hood of the line is
                             the Lua runtime's concern. Braces may span
                             subsequent lines (see § 2.9). */

shell_command         ::= SHELL_LINE_CONTENT NEWLINE
                          /* Any Content line not matching a preceding
                             alternative. See § 3.8. */
```

**Multi-output cook-step rule.** When `cook_step` has two or more `STRING` output patterns before `using`, the `using_clause` MUST use `using_lua_block` or `shell_block`. A conforming implementation MUST reject the form `cook "a" "b" using STRING`.

**`using >>{` is rejected.** A `using_clause` MUST NOT use the `>>{` (register-phase Lua block) prefix. The using clause produces work for execute, not register-phase orchestration; a register-phase Lua block in this position is incoherent. A conforming implementation MUST reject `cook STR using >>{ … }`.

**`using` dispatch order (normative).** After the keyword `using`, a conforming implementation checks in this order: `>>{` (rejected as above), `>{` (Lua block), `{` (shell block), `"` (single quoted string). The `>>{` test is present so the diagnostic for the rejected form is sharp; the first matching admitted alternative wins for the others.

**Declaration-only cook step.** A `cook_step` whose `using_clause` is absent is a declaration — it announces the output without providing a build command inline. Its build is assumed to be provided by preceding Lua registrations or a later amendment.

**Step-dispatch priority (normative).** A single `Content` line inside a recipe body is dispatched in this order, with the first match winning:

1. Prefix `ingredients` + separator (space, tab, or end of token) → `ingredients_step`.
2. Prefix `cook` + separator → `cook_step`.
3. Prefix `plate` + separator → `plate_step`.
4. Prefix `test` + separator → `test_step`.
5. Matches the module-call pattern (first token is `BARE_IDENTIFIER "." BARE_IDENTIFIER`) → `module_call`.
6. First character is `@` and the remainder is non-empty → `interactive_command`.
7. Otherwise → `shell_command`.

## A.5. Primitives [#grammar-appendix.primitives]
```ebnf
name                  ::= BARE_IDENTIFIER | STRING
BARE_IDENTIFIER       ::= /[A-Za-z_][A-Za-z0-9_.\\-]*/
STRING                ::= /"[^"]*"/
NUMBER                ::= /[0-9]+/
path                  ::= /[^\s\n]+/
comment               ::= /#[^\n]*/ NEWLINE
NEWLINE               ::= /\n/
LUA_LINE_CONTENT      ::= /[^\n]*/   /* zero or more characters; may be empty */
LUA_BLOCK_CONTENT     ::= BRACE_BALANCED   /* see § 2.9 */
SHELL_LINE_CONTENT    ::= /[^\n]+/
SHELL_BLOCK_CONTENT   ::= BRACE_BALANCED   /* see § 2.9 */
LUA_SOURCE            ::= /* any sequence of lines up to the next column-0
                             top-level keyword or end-of-file; see §{recipes.termination} */
BRACE_BALANCED        ::= /* text in which `{` and `}` counted outside of
                             Lua comments (--), double-quoted strings, single-
                             quoted strings, and long strings ([[...]]) balance
                             to zero; see § 2.9 */
MODULE_CALL_TEXT      ::= /* see § 4.11 */
```

**`BARE_IDENTIFIER` vs `name`.** Recipe and dependency names can be either a `BARE_IDENTIFIER` (allowing dots and hyphens as internal characters) or a double-quoted `STRING`. The `import` declaration's local alias is always a `BARE_IDENTIFIER` (no quoting).

**`lua_line` MAY be empty.** The `>` prefix alone (with no following content) is a valid `lua_line` producing an empty Lua expression.

**Brace-balance algorithm.** Both `LUA_BLOCK_CONTENT` and `SHELL_BLOCK_CONTENT` use the same brace-balance algorithm: `{` increments depth and `}` decrements depth, but braces inside Lua line comments (`--` to end of line), double-quoted strings, single-quoted strings, and long strings (`[[...]]`) are ignored. The block closes on the line where depth first returns to zero or below; that closing line is consumed but not included in the block body.

**Shell block line normalisation.** Each line of `SHELL_BLOCK_CONTENT` is trimmed of leading and trailing whitespace; blank lines after trimming are discarded. The result is a sequence of command strings, not a single raw string.
