## D.10. CS-0010 — Slug-based cross-references. [#changes.cs-0010]

**Date:** 2026-04-24
**Version:** v0.1
**Sections affected:** §{notation.stable-anchors} (new); all prose source files (bulk rewrite of heading markers and inline refs); `scripts/slug-mapping.ts` (new registry file).
**Summary:** Cross-references in this Standard are now author-assigned stable slugs of the form `chapter.leaf`. Prose refs use `§{chapter.leaf}` in source and render as the current section number. Heading anchors use the slug directly (URLs such as `/02-lexical/#lexical.identifiers`). The prior positional `sec-N-M-K` anchor scheme is removed.

**Motivation.** Section renumbering previously drifted every ref pointing at the affected sections. The fix commits after the CS-0009 restructure (commits `35d46d9` and `e83c490`) repaired the visible drift; the underlying positional anchor scheme was the root cause. Slug-based refs decouple identity from ordering. See §{notation.stable-anchors} for the full chapter-prefix registry and invariants.

**Impact.** Prose sources changed in bulk; rendered output unchanged. External deep links using the old `#sec-N-M-K` anchors break — the Standard is pre-release, no external consumers were affected. A new build-time lint rejects any bare `§ N.M` in prose to prevent regressions.

**Conformance.** No impact. The change is authoring-surface only.

## D.11. CS-0011 — Remove top-level variable declarations. [#changes.cs-0011]

**Date:** 2026-04-26
**Version:** v0.2
**Sections affected:** §{lexical.tokens}, §{lexical.identifiers}, §{lexical.keywords}, §{lexical.strings}, §{lexical.line-classification} (test-cascade renumber); §{grammar.overview}, §{grammar.top-level-ordering}, §{grammar.config-blocks}, §{grammar.step-dispatch}; grammar.var-declarations (deleted); §{grammar.config-composition} (new); §{recipes.step-kinds} (Note 4.4.2 deleted); §{lua.env-alias} (new); App. A.1, A.2, A.4, A.5; App. B.2.4 (rewritten), B.3.8 (deleted), B.3.9 (new).

**Summary:** Removes the top-level `variable_declaration` form (`NAME "value"`) and its supporting machinery: the `VarDecl` lexer token (§{lexical.tokens}), the blocking-keyword half of the contextual reservation (§{lexical.keywords}), the line-classification cascade test for the form (§{lexical.line-classification} test 11; the former test 12 renumbers to 11), and the recipe-body reclassification rule (§{grammar.step-dispatch}). The §3.3 prose-grammar section, the App. A.2 production, the App. A.2 "Identifiers blocked" rule, and the App. A.4 `shell_command` grammar comment's parenthetical reference are all deleted; App. B.2.4 is rewritten to motivate the contextual-keyword rule purely on the reserved-recipe-segment set, and App. B.3.8 is deleted as moot.

In place of the removed surface, the Standard adds §{grammar.config-composition} (new §3.6.1) specifying that a Cookfile MAY contain at most one unnamed `config_block` (the *base config*) and zero or more named `config_block`s (the *overlay configs*); a conforming implementation MUST select at most one overlay at load time by an implementation-defined mechanism, MUST reject a selection that does not match any declared overlay, and MUST execute base before overlay against shared Cook Lua API state with last-write-wins semantics on overlay writes. Both bodies execute during the load phase, after `use` resolution and before recipe registration. §{lua.env-alias} (new §6.1.2) binds the bare global `env` to `cook.env` inside `config_block` bodies only, bringing the `env.X = "..."` usage already shown in examples into normative conformance. App. B.3.9 (new) records the design rationale for consolidating variables into config blocks, the single-overlay choice over multi-select, and the implementation-defined selection mechanism.

**Implementation status.** This change is spec-only. The Rust parser (`cli/crates/cook-lang`) and `tree-sitter-cook` continue to accept the old form; bringing them into conformance with the Standard at this version is the subject of a follow-up CS that will also add the §3.6.1 conformance fixtures (base-only, base + overlay last-write-wins, overlay-only, missing-overlay-is-error, base-runs-before-overlay).

**Reference:** this commit.

## D.15. CS-0015 — Recipe-body phase split (`>` execute / `>>` register / module-call register), region rule, body-unit bundling. [#changes.cs-0015]

**Date:** 2026-04-28
**Version:** v0.3
**Sections affected:** §{lexical.line-prefixes} (rewritten); §{lexical.line-classification} (cascade renumbered, `>>{` and `>>` tests added, triple-arrow reservation); §{grammar.step-dispatch} (token classes + region cross-link); §{recipes.step-kinds} (new step kinds, Note 4.4.2 region rule); §{recipes.lua-steps} (rewritten — `>` /`>{`/`>>`/`>>{` family, body bundling §{recipes.body-bundling}); §{recipes.module-call-steps} (desugaring target updated); §{lua.cook-sh} (promoted to both phases); §{lua.cook-exec} (execute-phase ban added); §{exec.two-phase} (Example 8.2.1 rewritten to `>>`); §{exec.phase-classification} (new normative table); §{exec.step-groups} (surface-construct table rewritten); App. A.3 (recipe_body split into `declarative_region` and `imperative_region`, "Region ordering rule" added); App. A.4 (`inline_lua_line` and `inline_lua_block` productions added; `inline_lua_block` of using-clause renamed to `using_lua_block`; `using >>{` rejected); App. A.5 line prefixes; App. B.4.8–B.4.12 (new rationale subsections); App. B.4.13 (renumbered from B.4.11).

**Summary:** Splits the Cookfile's Lua surface into two phase-explicit families:
- **Single-`>` family** (`>` *body*, `>{ … }`) is **execute-phase**. Lines and blocks compile into a body unit (§{recipes.body-bundling}) whose `lua_code` payload runs on a worker VM at execute, after the recipe's declarative work units complete. This is the "runs when the recipe runs" form.
- **Doubled-`>` family** (`>>` *body*, `>>{ … }`) is **register-phase**. Lines and blocks are inlined into the recipe-body Lua function and evaluated during recipe registration. This is the surface form a Cookfile uses to call register-only Cook Lua API helpers (`cook.add_unit`, `cook.export`, `cook.step_group`) and to declare register-phase locals consumed by subsequent declarative steps.
- **`module_call`** (e.g., `m.fn(…)`) desugars to `inline_lua_line` (single-line) or `inline_lua_block` (multi-line) per §4.11 — register-phase, matching the doubled-prefix family.

A **recipe-body region rule** (Note 4.4.2 / App. A.3) partitions every `recipe_body` into a declarative region followed by an imperative region. Declarative-region steps (`ingredients`, `cook`, `plate`, `test`, `inline_lua_line`, `inline_lua_block`, `module_call`) MUST appear before imperative-region steps (`lua_line`, `lua_block`, `shell_command`, `interactive_command`). A conforming implementation MUST reject a recipe in which a declarative-region step appears after the first imperative-region step. The diagnostic identifies the offending step's line and the line on which the imperative region began.

A **body-bundling rule** (§{recipes.body-bundling}) coalesces consecutive imperative-region steps into one execute-phase work unit per `@interactive`-bounded run. Inside the body unit, adjacent `shell_command` lines coalesce into a single `cook.sh` call (newline-joined, `set -e` prefixed) so that a `cd` followed by a `./app` runs in one shell process. The Lua VM persists across the whole bundle, so a `local` declared in one `>` line is visible to the next. `@interactive` steps remain their own draining units (§{exec.interactive-drain}) and break the bundle.

`cook.sh` is promoted to both phases (§{lua.cook-sh}). `cook.exec`, `cook.interactive`, `cook.add_unit`, `cook.step_group`, and `cook.recipe` remain register-only and MUST raise a Lua runtime error when called from execute-phase Lua. A new normative phase-classification table at §{exec.phase-classification} records every place a Lua expression can appear and the phase that evaluates it.

App. A.4 introduces `inline_lua_line ::= ">>" LUA_LINE_CONTENT? NEWLINE` and `inline_lua_block ::= ">>{" LUA_BLOCK_CONTENT "}" NEWLINE`. The grammar production formerly named `inline_lua_block` (the `>{ … }` form inside a `using` clause) is renamed to `using_lua_block` to disambiguate from the new step kind. App. A.4 also adds the rule "`using >>{` is rejected": a register-phase Lua block in a using-clause position is incoherent and a conforming implementation MUST reject it. §{lexical.line-prefixes} adds two new prefixes (`>>{`, `>>`) and the §{lexical.line-classification} cascade gains corresponding tests in longest-prefix order, plus a parse-time rejection for runs of three or more `>` characters at line start.

## D.17. CS-0017 — `use` aliases visible to execute-phase Lua; `cook.load_module` two-phase. [#changes.cs-0017]

**Date:** 2026-04-28
**Version:** v0.3
**Sections affected:** §{lua.cook-load-module} (new §6.3.4); §{modules.use-scope} (new §7.4 with the existing §7.4 renumbered to §7.5); §{lua.use-env} (clarification on dual-phase binding); App. B (rationale subsection on lexical scoping); positive conformance fixture 017-use-from-execute (new).

**Summary:** Extends the `use` declaration so the alias it introduces is in scope for **both** phases of every recipe in the declaring Cookfile, not register-only. The codegen (`cli/crates/cook-luagen/src/recipe.rs`) prepends a `local <alias> = cook.load_module("<name>")` line per `use` to every execute-phase body unit's `lua_code` payload, so `> alias.fn(...)` calls the same module function as `alias.fn(...)` in declarative position — the only difference is the phase it runs in.

`cook.load_module` is promoted from register-phase to **both phases** (§6.3.4 new). The execute-phase variant lives on the worker VM (`cli/crates/cook-luaotp/src/pool.rs`) and uses the same `cook_modules/<name>.lua` / `cook_modules/<name>/init.lua` lookup as its register counterpart, resolved against the calling work unit's `current_working_dir`. The `init()` hook (§7.5) runs at first load on each (worker VM, working_dir, name) tuple.

The worker VM also refreshes `package.path` per work unit to `<working_dir>/cook_modules/?.lua;<working_dir>/cook_modules/?/init.lua;<defaults>`, so `require("foo")` is a working synonym for `cook.load_module("foo")` on the worker side.

§7.4 (new) records the **lexical-per-Cookfile** scoping rule: a `use` declaration's alias is scoped to its declaring Cookfile and is not visible across `import` boundaries; cross-Cookfile composition stays at the **recipe** level via qualified name references (§{modules.qualified-refs}), not at the **module** level. Module authors document per-function which phase a function targets; both-phase functions restrict themselves to `cook.sh`, `cook.env`, `cook.cache`, `cook.platform`, `fs.*`, and `path.*`.

**Why:** Before CS-0017, the only way to invoke a `use`-d module function from imperative-region Lua was to (a) re-resolve it manually via `cook.load_module(...)` on the worker VM (which wasn't even installed there) or (b) wrap it in a register-phase call that records a unit. Both were verbose and asymmetric. CS-0017 collapses the asymmetry: `use` means "bring this alias into scope," period. Whether a given function call lands in the declarative or imperative region is decided by the recipe-body region rule (§{recipes.step-kinds}, Note 4.4.2), and the alias is bound either way.

**Implementation status.** Lockstep — codegen, worker VM, and spec land together. New positive conformance fixture (`017-use-from-execute`) pins the AST shape; the codegen-side and runtime-side checks are in `cook-luagen/tests/conformance.rs` and the runnable demo in `examples/v03-phase-split/Cookfile` (recipes `module-call-execute` and `module-via-require`).

The single-Cookfile `{NAME}` body-ref-extraction loop previously inlined in `cmd_run` is hoisted to a `compute_single_inferred_deps` helper; the workspace counterpart (`compute_workspace_inferred_deps`) was already in place from the cross-Cookfile-caching work. Both helpers are now invoked at every CLI entry that hands recipes off to `cook_engine::run::run`: `cmd_run`, `cmd_test`, and `cmd_dag`. The `cmd_dag` workspace branch — previously absent — mirrors `cmd_run`'s workspace setup (workspace recipe-info, prefix-keyed registry map, qualified-consumer/qualified-dep edge map) and routes each ready recipe through its prefix's registry by reusing the `(prefix, local_name)` split that `cook_engine::run::run` already performs internally. Conflict-warning emission (`recipe X has both explicit ': dep' and inferred '\{dep\}' dependency`) is also factored into `warn_single_dep_conflicts` and `warn_workspace_dep_conflicts` and called from each CLI path.

**Why:** The asymmetry was discoverable only by writing a Cookfile whose `test` body referenced a `cook`-step recipe via `{NAME}` and invoking `cook --test` rather than `cook NAME`. Authors who had been driving builds through `cmd_run` would conform to §\{xref.dep-implications\} by accident; the moment they switched to `cook --test` for CI, the same Cookfile would emit a register-time error from a CLI command that has no language-level reason to behave differently. Pre-1.0 stability requires that the contract hold uniformly across the CLI surface, since the language spec speaks only of recipes and dependencies, not of which `cook` subcommand is in use.

**Implementation notes.** Each of the six CLI call sites — `cmd_run`, `cmd_test`, and `cmd_dag` in both single-Cookfile and workspace branches — now reads from the same two helpers in `cli/crates/cook-cli/src/pipeline.rs`. `cmd_dag`'s register-only loop is factored into `collect_dag_units`, mirroring the per-wave dispatch in `cook_engine::run::run` (`run.rs:170+`) but stopping short of work-unit DAG construction; the dag visualizer only needs the registered units, the explicit edge map, and the inferred-deps map. Two unit tests in `pipeline.rs` (`single_inferred_deps_body_ref_produces_edge`, `single_inferred_deps_empty_when_no_body_refs`) join the existing `workspace_inferred_deps_*` cases to pin both helpers under regression. The `examples/cmd_test_inferred_deps_e10/` fixture is the end-to-end pin: a two-recipe Cookfile (`prepare` cook + `verify_prepare` test referencing `{prepare}`) plus a `verify.sh` that asserts `cook --test --jobs 1` does not surface the "no terminal output" error and that `verify_prepare` reaches `done`.

**Residual scope (out of E.10).** The `add_test` API does not yet propagate `step_group_dep_refs` into unit-DAG `dep_edges`, so a test recipe whose body refs a sibling recipe will race that sibling under `--jobs > 1`. **Conformance impact.** None at the parser/luagen layer (no AST or dump-format changes). ## D.29. CS-0029 — Diamond imports: `cook.dep_output` resolves cross-Cookfile refs via per-alias canonical importee prefix. [#changes.cs-0029]

**Date:** 2026-05-02

**Version:** v0.5 (implementation-only catch-up; the §7.5 diamond-dedup contract was already in the Standard).

**Sections affected:** none in the Standard surface. Closes a residual gap in CS-0028's §7.3 reference-implementation catch-up that was exposed once the `examples/monorepo_codegen/` workspace was built from the root.

**Summary.** Pre-CS-0029, `cook.dep_output("alias.recipe")` resolved cross-Cookfile body refs by prepending the **calling registry's** `qualified_prefix` to the local name (CS-0028's qualify-by-prefix scheme). This works for linear-transitive cases — `apps/server → //libs/queue → //libs/proto` where each Cookfile sits on a single import path — but breaks for **diamond imports**, where the same Cookfile is reachable via two chains. In `monorepo_codegen/`, both `apps/cli` and `apps/server → libs/queue` reach `libs/proto`. `find_full_prefix` deterministically picks a single canonical workspace prefix for proto (e.g., `"server.queue.proto"` if that path won the BTreeMap insertion race), and proto's recipes are stored in `terminal_outputs` under that one prefix. CLI's local alias `"proto"` then has to resolve to `"server.queue.proto"` — *not* `"cli.proto"`, which CS-0028's prefix-prepend would yield.

CS-0029 threads a per-Registry `alias_qualified_prefixes: BTreeMap<String, String>` map into `register_dep_output_api`. The map associates each direct-import alias with its target Cookfile's canonical workspace qualified prefix (the same value that `find_full_prefix(target_canon)` returns). At lookup, `cook.dep_output(name)` splits `name` on the first dot. If the first component is a key in `alias_qualified_prefixes`, the lookup uses that alias's canonical importee prefix to compose the global key (`<importee_prefix>.<recipe>`, or just `<recipe>` when the importee prefix is empty). Otherwise, the call is treated as a same-Cookfile reference and falls back to the calling registry's `qualified_prefix` (CS-0028's behavior). Path rewriting via `alias_dirs` continues to use the local alias unchanged.

The map is computed once per Cookfile by `Workspace::alias_qualified_prefixes_for(importer_dir)` (parallel to the existing `alias_dirs_for`), and supplied through the new `Registry::with_alias_qualified_prefixes` builder. `pipeline.rs` calls both methods at registry construction time for the root and every imported Cookfile.

**Why:** Diamond imports are the textbook case for cross-Cookfile caching to earn its keep — a shared lib referenced by multiple cousins, with §7.5 dedup folding the two paths. CS-0028's prefix-prepend scheme silently relied on each Cookfile having a single ancestry chain; the diamond breaks that assumption. The right resolution rule is "go to the alias's canonical target prefix, regardless of who's asking" — exactly the semantics `Workspace::alias_qualified_prefixes_for` makes explicit.

**Implementation notes.** Touches `cli/crates/cook-register/src/{dep_output_api.rs, engine.rs, unit_api.rs}` plus `cli/crates/cook-cli/src/{workspace.rs, pipeline.rs}`. Two new in-crate regression tests pin the diamond and same-Cookfile fallback paths: `test_dep_output_diamond_resolves_to_canonical_importee_prefix` and `test_dep_output_same_cookfile_uses_self_prefix`. The CS-0028 transitive-sigil test is renamed to `test_dep_output_resolves_via_alias_qualified_prefix` and updated to use the new alias-map signature. The runtime fixture lives in `examples/monorepo_codegen/`: `cook top` from the workspace root exercises the diamond on `libs/proto` (cli direct + server transitive via queue), and `cook server_bin` from `apps/server/` continues to exercise the linear-transitive case.

**Conformance impact.** None at the parser/luagen layer (no AST or dump-format changes; no language surface change). The Standard's §7.3 (cross-Cookfile body refs) and §7.5 (diamond dedup) wording was already correct; this is a reference-implementation fix that closes the gap CS-0028 missed.

**Reference:** the CS-0029 commit on the `feature/cross-cookfile-caching` branch.

## D.32. CS-0032 — `//`-anchored sigil imports: §7.2 grammar, §7.5 diamond-dedup scoping, §7.6 workspace-root determination, §7.7 cache portability. [#changes.cs-0032]

**Date:** 2026-05-03

**Version:** v0.6

**Sections affected:** §{modules.import-declaration} (§7.2 — adds the `//path` sigil grammar production and the normative rejections for `..`-bearing tree-relative paths, absolute paths, and `..`-bearing sigil paths); §{modules.qualified-refs} (§7.3 — qualified transitive-imports definition); §{modules.duplicates-and-cycles} (§7.5 — scopes diamond dedup to sigil-anchored imports); §{modules.workspace-root} (§7.6 — workspace-root determination via `--root` / `.cookroot` walk-up / tree-import inference / self-root / sigil-anchor rejection, Rules 1–5); §{modules.cache-invariants} (§7.7 — content-addressed cache invariants under workspace path moves); App. A.7 (`import_path` grammar production for tree vs sigil shapes; `path_segment` normalization).

**Summary.** Establishes the **sigil import** as a first-class language construct distinct from tree-relative imports. An `import alias //path` declaration whose path begins with `//` resolves the importee against the **workspace root**, regardless of where the importing Cookfile sits in the directory tree. Tree-relative imports (`./path`, `../path` — the latter rejected per the §7.2 normative diagnostics) continue to resolve relative to the importer's directory. The two forms compose: a workspace can mix tree-relative and sigil edges in the same Cookfile.

The workspace root is determined by §7.6 Rules 1–5, in priority order: (1) `--root <path>` flag wins; (2) `.cookroot` marker found by walk-up from the entry-point Cookfile; (3) tree-import inference walk picking the highest ancestor under which all sigil paths resolve; (4) self-root for sigil-free Cookfiles; (5) rejection with a normative diagnostic when sigils require an anchor that cannot be inferred. §7.5 is scoped so that diamond dedup applies *only* to nodes reachable through sigil-anchored paths, preserving tree-relative locality (a Cookfile imported twice via `./foo` from two different parents resolves to two distinct namespaces; the same Cookfile imported via `//libs/foo` resolves to one canonical instance with a single qualified prefix). §7.7 codifies the cache portability invariant: content-addressed cache keys MUST NOT include absolute paths, so a built workspace can be `cp -r`'d to a new path or shared across machines and continue to hit cache.

App. A.7 adds the `import_path = tree_path | sigil_path` production with the normative `path_segment` rule rejecting `..` segments. App. A is the authoritative grammar reference; the parser's enforcement (CS-0026) cites these productions verbatim.

**Why a v0.6 bump.** The sigil import form is net-new language surface. Before v0.6, every `import` declaration resolved relative to the importer's directory; this is the first time Cook supports a workspace-relative path form. The §7.6 workspace-root contract is also new — there was no notion of a "workspace" at the spec level prior to this. The §7.7 cache portability invariant promotes prior implementation behavior to a normative contract, making it a portable promise rather than a coincidence.

**Conformance impact.** Four new conformance cases land alongside CS-0032's normative prose: `standard/conformance/positive/033-import-sigil-accepted/` and three negatives (`034-import-dotdot-rejected/`, `035-import-absolute-rejected/`, `036-import-sigil-dotdot-rejected/`). Each negative pins one of the §7.2 normative diagnostics. The Rust parser (CS-0026) and the Lua codegen / register layer (CS-0027..CS-0030) realize the contract end-to-end; the conformance harness is green at the v0.6 cut.

**Implementation status.** Lockstep — the spec, parser AST (`ImportPath::Tree | ::Sigil`), parser path-shape rejection, codegen for cross-Cookfile body refs, register-side `qualified_prefix` machinery, workspace-root walk-up, and the runtime fixtures (`examples/cache_benchmarks_sigil/`, `examples/monorepo_codegen/`, the four bug-hunt fixtures) all landed across the `feature/cross-cookfile-caching` branch and merged together. ## D.33. CS-0033 — `$<IDENT>` sigil-disambiguated placeholder syntax with closed-set resolution. [#changes.cs-0033]

**Date:** 2026-05-03

**Version:** v0.7

**Sections affected:** §{lexical.placeholders} (new §2.11), §{xref.resolution} (rewrite), §{xref.env-namespace} (new §5.7), §{lua.shell-placeholders} (§6.7 + §6.7.1 rewrite), App. A `placeholder` / `sigil_placeholder` productions, App. B.8 (`out_0` rejection rationale), App. B.9 (`cook.require_env` runtime mechanism rationale), App. E.8 (fully resolved). Reference-implementation: `cli/crates/cook-{lang,luagen,register,contracts}` (lexer reservation, scanner, resolver, substitution, runtime helper, shared accessor constant), 5 new positive conformance fixtures (`034-shell-brace-expansion` through `038-shell-json-literal`), 3 new negative conformance fixtures (`037-reserved-env-recipe-rejected`, `038-out-zero-rejected-sigil`, `039-in-in-many-to-one-sigil`). 24 in-tree Cookfiles (examples + tree-sitter-cook) migrated from `{IDENT}` to `$<IDENT>`.

**Summary.** Replaces Cook's placeholder syntax `{IDENT}` with the strict sigil shape `$<IDENT>` and tightens the substitution layer's resolution rule from open (silent fallthrough to `cook.env[TOKEN]` for any unrecognized token) to closed (every placeholder MUST resolve to a builtin, an in-scope recipe, or a declared env var; otherwise a load-time diagnostic). The motivation is the structural collision between the legacy `{...}` shape and shell idioms that use `{` and `}` — Bash brace expansion, parameter expansion, awk inline scripts, `find -exec rm {} \;`, JSON literals; under the new shape these idioms pass through to the shell verbatim because the strict-bail rule (§{lexical.placeholders}) treats any byte sequence not matching `$<IDENT>` exactly as literal.

The substitution layer is wired uniformly across all four shell-text contexts: `cook` step output patterns, `cook ... using { ... }` shell-block bodies, `plate` and `test` shell-block bodies, and bare `shell_command` steps in both recipe and chore bodies. The chore-body wiring closes App. E.8 — pre-CS-0033, the chore codegen path skipped substitution entirely, surfacing as a shell error for any chore using a placeholder. App. E.2 (the original `${VAR:-default}` collision symptom) is structurally eliminated: `${VAR:-default}` does not begin with `$<` so it is unambiguously literal under the new lexer rule.

The closed-set resolver lives in `cook-luagen/src/resolver.rs` and produces one of three outcomes per IDENT: builtin (inlined), recipe in scope (lowered to `cook.dep_output("name")` or accessor variant), or env-runtime (lowered to `cook.require_env("name")`). The runtime side, `cook.require_env`, captures the `cook.env` keyset at the end of config-block evaluation and rejects any subsequent lookup whose key is not in the captured set. The "declared" check is enforced at register time, after config-block evaluation completes; subsequent recipe-time writes to `cook.env` do not retroactively satisfy the contract.

**Why a flag-day cut.** Pre-1.0, the alternative — dual-grammar transition admitting both `{IDENT}` and `$<IDENT>` for one version, then dropping `{IDENT}` — was considered and rejected. Two valid spellings invite reading-friction, the lexer disambiguation surface area grows when it should be shrinking, and every doc and example would have to choose a spelling then update again at deprecation. The migration tooling that would have justified dual-grammar (a CLI subcommand for downstream Cookfile rewriting) was not built — there is no installed base to migrate, and a one-off `re.sub(r'\{([a-zA-Z_][a-zA-Z0-9_.]*)\}', r'$<\1>', src)` pass handles the in-tree corpus.

**Implementation notes.** The scanner (`cook-luagen/src/sigil.rs`) is byte-level and bails on the first non-IDENT byte after `$<` rather than searching forward for a `>` — this is the property that makes shell idioms like `${HOME}` or `$<HOME bar>` (with embedded space) literal rather than producing a misleading "unclosed placeholder" diagnostic. The resolver distinguishes "not a builtin shape" (`BuiltinMatch::No` → fall through to recipe/env) from "malformed builtin shape" (`BuiltinMatch::Malformed` → hard error, e.g., `$<out_0>` per App. B.8). The reserved-recipe-segment rule was tightened in this change to reject `env` as either the first or last dotted segment, ensuring `$<env.X>` is unambiguously an env-var lookup and cannot collide with a recipe whose segment is named `env`.

**Conformance impact.** The corpus migrated wholesale to `$<IDENT>` syntax. Five new positive fixtures (`034-shell-brace-expansion` through `038-shell-json-literal`) pin the strict-bail rule against shell `{}`-idioms: Bash brace expansion, parameter expansion, awk script, `find -exec`, and JSON literal. Three new negative fixtures (`037-reserved-env-recipe-rejected`, `038-out-zero-rejected-sigil`, `039-in-in-many-to-one-sigil`) pin the closed-set rejection cases. The conformance harness was disabled during the parser-rewrite phase and re-enabled at corpus-migration time; both `cook-lang` and `cook-luagen` conformance test bundles pass against the cut commit.

**Reference:** the CS-0033 commit series on `feature/sigil-placeholder-syntax`.

**Date:** 2026-05-03

**Version:** v0.7

**Sections affected:** none normative (this entry is informative; no chapter prose changes). ## D.36. CS-0036 — Tool-binary fingerprinting recognises every newline-separated statement, not just the first token. [#changes.cs-0036]

**Date:** 2026-05-03

**Version:** v0.7 (post-cut amendment; reference-implementation fix that aligns the §{exec.cache} observable-inputs prose with what the cache actually fingerprints).

**Sections affected:** §{exec.cache} (Note 8.6.2 added — informative coverage statement for tool-binary fingerprinting); App. D (this entry).

**Summary.** Pre-CS-0036, `cook-cache::context::step_context_hash` fingerprinted only the first whitespace-delimited token of the unit's command text. For any multi-line shell script — the dominant Cookfile pattern — only the first line's first token contributed to the per-step context hash; every tool invoked on subsequent lines (`gcc`, `clang`, the linker, `python`, `awk`, …) was silently ignored. A second symptom: a leading POSIX env-prefix (`LC_ALL=C make foo`) was passed to `which("LC_ALL=C")`, resolved to `None`, and collapsed every such command to a single empty-tool fingerprint. The result was that toolchain swaps did not invalidate the cache for any but the most trivial single-line recipes — exactly the case the cache_benchmarks fixture's link recipe relies on.

CS-0036 reworks `step_context_hash` to (a) split the command text on newlines, (b) per line, skip leading `VAR=value` env-prefix tokens (a token whose name part is a valid shell-style identifier), (c) take the next remaining token as the executable, (d) resolve via `which` + `canonicalize`, (e) deduplicate via `BTreeSet<PathBuf>` so the fold order is deterministic, and (f) fold each unique resolved binary's content hash into the per-step fingerprint. The recognised-subset policy is documented in the new informative Note 8.6.2 in §{exec.cache}: only the first executable on each newline-separated statement is recognised; tools chained via `;`, `&&`, `||`, pipes, command substitution, subshells, or invoked indirectly through scripts are NOT recognised by this pass.

**Why §{exec.cache} gains an informative note rather than tightened normative prose.** §{exec.cache}'s normative wording is correct as written: it permits ("MAY consider") the resolved content of invoked binaries without prescribing a recognition algorithm. Recognition is undecidable in general (Rice's theorem on shell text), so the Standard cannot reasonably normatively require any particular subset. The new Note 8.6.2 is informative: it tells authors what the reference implementation fingerprints today, and it advertises the expected escape hatches (one-statement-per-line, planned `cache.tools = { ... }` declaration). Other implementations are free to recognise more or less.

**Implementation notes.** Touches `cli/crates/cook-cache/src/context.rs` only. Five new in-crate tests pin the behavior: `step_context_hash_fingerprints_tools_on_later_lines` (the headline regression), `step_context_hash_skips_env_prefix` (the `VAR=value` collapse fix), `step_context_hash_deterministic_across_line_order` (BTreeSet fold ordering), `step_context_hash_ignores_blank_and_comment_lines`, and `is_env_prefix_token_recognises_assignments` (env-prefix predicate unit-tests). The `examples/cache_audit_first_token/` fixture pins the contract end-to-end through the cook binary: a multi-line recipe whose tool changes between runs MUST invalidate the cache.

**Conformance impact.** None at the parser or codegen layer. No language surface change. The conformance corpus is unchanged. The change is observable only via cache hit/miss behavior, which is implementation-defined per §{intro.non-scope} and §{exec.cache}.

**Reference:** the CS-0036 commit on `worktree-agent-a012ebda236e76f35` (cache-fingerprint multi-line fix).

## D.38. CS-0038 — `use_name` constrained to `LUA_IDENT` at parse time. [#changes.cs-0038]

**Date:** 2026-05-03

**Version:** v0.7

**Sections affected:** §{grammar.use-declarations} (rewritten — adds the `use_name` `LUA_IDENT` constraint, new Example 3.4.2, augmented Note 3.4.1); App. A.2 (`use_declaration` production rewritten with `use_name` nonterminal; new normative paragraph "`use_name` constraint"); App. A.5 (new `LUA_IDENT` primitive production; new "`LUA_IDENT` vs `BARE_IDENTIFIER`" paragraph). Reference-implementation: `cli/crates/cook-lang/src/lexer.rs` (new `LexError::InvalidUseName` variant, new `check_use_name` helper, six new unit tests). Conformance: three new negative fixtures (`040-use-name-with-spaces-rejected`, `041-use-name-with-dash-rejected`, `042-use-name-with-dot-rejected`).

**Summary.** Tightens the `use NAME` declaration to require `NAME` to match `/^[A-Za-z_][A-Za-z0-9_]*$/` — the Lua identifier shape — independent of whether it is written bare or inside double quotes. Pre-CS-0038, the lexer accepted any `BARE_IDENTIFIER` or `STRING` after `use`, including pathological forms such as `use "has spaces"`, `use cpp.bad`, and `use foo-bar`. The codegen layer (`cook-luagen/src/recipe.rs`) then dropped the verbatim text into a `local NAME = cook.load_module(...)` binding, producing malformed Lua that surfaced as a runtime parse error far from the source line. Post-CS-0038 the lexer rejects these at parse time with `InvalidUseName { name, line }`, citing the offending name and source line directly.

**Why hyphens are rejected outright.** A subtle pre-CS-0038 misfeature: `cook-luagen/src/recipe.rs` rewrote `-` to `_` inside the generated Lua local binding (`module_name.replace('-', "_")`) so that hyphenated module names would tokenise as a Lua identifier. The rewrite was a workaround for the lax parser admitting hyphens, but it silently collapsed `foo-bar` and `foo_bar` to the same Lua local — an unobservable shadowing whose only symptom was the wrong module being bound under one of the two names. Rejecting hyphens at parse time eliminates the rewrite's reason to exist; the codegen layer needs no change because the parser now guarantees the name is already a Lua identifier.

**Why `LUA_IDENT` is its own production.** App. A's existing `BARE_IDENTIFIER` production admits internal dots and hyphens (it is the same production used for recipe names, dependency names, and import aliases, all of which legitimately want dotted forms). The `use_name` constraint is narrower: the matched text is bound at load time as a Lua local under the same spelling and therefore must round-trip through Lua's identifier grammar. The new `LUA_IDENT` primitive captures the narrower shape, making the constraint expressible as a grammar production rather than as out-of-band prose. The `use_name` is currently the sole site of `LUA_IDENT` in App. A; future productions that bind text as Lua locals (none planned at v0.7) would reuse it.

**Implementation notes.** The lexer change is a four-line addition in `cli/crates/cook-lang/src/lexer.rs`: a new `LexError::InvalidUseName { name, line }` variant, a `check_use_name` helper that performs the regex validation by hand (no regex dependency), and a single call site in the `Token::UseDecl` branch of `tokenize`. Six new unit tests pin the rejection cases (spaces, hyphen, dot, leading digit) and the accepted cases (underscore-internal, leading underscore). The codegen layer (`cook-luagen/src/recipe.rs:488-495`) is left untouched; the obsolete `module_name.replace('-', "_")` rewrite is now provably dead code (the parser rejects any name containing `-` before it ever reaches codegen), but removing it is a separate cleanup tracked outside this CS.

**Conformance impact.** Three new negative fixtures land alongside this entry: `040-use-name-with-spaces-rejected/` pins the quoted-with-spaces case, `041-use-name-with-dash-rejected/` pins the bare-with-dash case, and `042-use-name-with-dot-rejected/` pins the bare-with-dot case. Each fixture's `error.txt` asserts a substring of the new diagnostic ("`'use' name 'X' is not a valid Lua identifier`"). All existing positive fixtures that exercise `use` (`010-use-and-module-call/`, `017-use-from-execute/`) already satisfy the `LUA_IDENT` rule and continue to parse cleanly. The conformance harness passes against the CS-0038 commit.

**Reference:** the CS-0038 commit on `worktree-agent-a2d8a370be4c0889a` (the lang-audit branch).

## D.39. CS-0039 — Stateful brace-balance: multi-line Lua long strings, block comments, and shell heredocs are opaque to the brace counter. [#changes.cs-0039]

**Date:** 2026-05-03

**Version:** v0.7

**Sections affected:** §{lexical.brace-blocks} (rewritten to require stateful opaque-span tracking; new sub-sections §{lexical.brace-blocks.lua-spans} and §{lexical.brace-blocks.shell-spans}; new Examples 2.9.2 and 2.9.3); App. A `BRACE_BALANCED` production comment; App. A "Brace-balance algorithm" prose. Reference implementation: new `cli/crates/cook-lang/src/brace_scan.rs` module exposing `LuaScanner` and `ShellScanner`; `lua_block.rs`, `shell_block.rs`, and `recipe.rs::collect_module_call` rewritten to drive the stateful scanners. Conformance corpus: positive fixtures `039-lua-multiline-long-string`, `040-lua-multiline-block-comment`, `041-lua-long-string-leveled`, `042-shell-heredoc-brace-in-body`, `043-shell-heredoc-quoted-delim`.

**Summary.** The pre-CS-0039 `count_brace_delta` helper in `cli/crates/cook-lang/src/lua_block.rs` was line-local: it correctly ignored braces inside single-line Lua line comments, single- and double-quoted strings, and single-line long strings (`[[…]]`), but it did not carry opaque-span state across source lines. Three classes of multi-line Lua / shell construct were therefore broken: (i) multi-line Lua long strings `[[ … ]]` and `[==[ … ]==]`, (ii) multi-line Lua block comments `--[[ … ]]` and `--[==[ … ]==]`, and (iii) POSIX shell heredocs (`<<TAG`, `<<-TAG`, `<<'TAG'`, `<<"TAG"`). A `}` byte appearing on any line interior to one of these spans would decrement the running brace depth and prematurely close the surrounding `>{ … }` Lua block, `using { … }` shell block, or multi-line module-call body. CS-0039 makes the brace counter stateful: it now carries long-string level, block-comment level, and pending heredoc delimiters across lines, treating every brace byte interior to such a span as opaque data.

**Lua side.** A new `LuaScanner` (`brace_scan.rs`) tracks an `Option<u32>` long-bracket level. A `[==[` opener at level *n* is opaque until a `]==]` closer at the same level is seen — possibly many lines later. The same level-aware open/close logic governs `--[==[ … ]==]` block comments. Single-line constructs (line comments, quoted strings) remain single-line. `LuaScanner` drives `collect_lua_block` (for `>{ … }`) and `collect_module_call` (for multi-line `mod.fn(…)` invocations).

**Shell side.** A new `ShellScanner` (`brace_scan.rs`) maintains a FIFO queue of pending heredoc closers, each tagged with its delimiter text and `<<-` (tab-strip) flag. While the queue is non-empty, body lines are scanned only for the front-of-queue delimiter; brace bytes are ignored. Multiple heredocs queued on a single opener line (`cmd <<A <<B`) are consumed in opener order. The closer is matched against the **trimmed** body line, consistent with `SHELL_BLOCK_CONTENT` line normalisation: this lets authors write a heredoc inside an indented `{ … }` body without dedenting the closing delimiter to column 0. `ShellScanner` drives `collect_shell_block`.

**Shell-side decision: full support.** The instructions for this CS offered a tractable alternative — REJECT heredocs inside `using { … }` with a clear diagnostic — but full support was chosen because (i) heredocs are a load-bearing POSIX construct that authors expect to "just work" in any shell-like context, (ii) the scanner state is small (a `Vec<(String, bool)>`) and the implementation is comparable in size to a rejection diagnostic, and (iii) trim-first matching collapses the runtime indentation question that would otherwise force authors to dedent or use `<<-` with strict tab discipline.

**Why a non-cut entry on v0.7.** CS-0039 lands as a v0.7 erratum: it is a normative correction to the §{lexical.brace-blocks} algorithm that was already part of v0.7 but was under-specified for multi-line opaque spans. No version bump is taken — implementations that already happened to track multi-line state are conformant; implementations that were line-local (the reference parser pre-CS-0039) need to update.

**Conformance impact.** Five new positive fixtures land alongside this entry and pin the multi-line carry behaviour:

- `039-lua-multiline-long-string` — `}` byte inside multi-line `[[ … ]]` is data.
- `040-lua-multiline-block-comment` — `}` byte inside multi-line `--[[ … ]]` is data.
- `041-lua-long-string-leveled` — `[==[ … ]==]` level matching; `]]` does not close `[==[`.
- `042-shell-heredoc-brace-in-body` — `}` byte inside `<<EOF` body is data.
- `043-shell-heredoc-quoted-delim` — `<<'END'` quoted delimiter form is recognised.

Companion unit tests: the `cs_0035_*` family in `tests.rs`, `shell_block.rs`, and the per-scanner tests in `brace_scan.rs`. The conformance harness is green at the CS-0039 commit.

**Implementation status.** Lockstep — the spec rewrite, the new `brace_scan` module, the three call-site rewrites (`lua_block.rs`, `shell_block.rs`, `recipe.rs`), the unit tests, and the five conformance fixtures land in the same commit. Walkthrough fixtures live under `examples/lang_audit_lua_multiline_string/`, `examples/lang_audit_lua_block_comment/`, and `examples/lang_audit_shell_heredoc_brace/`.

**Reference:** the CS-0039 commit on `main` (this commit).

The root cause was upstream: `WorkResult::output_lines` in `cook-luaotp` was `Vec<String>`, discarding fd-of-origin at the worker boundary.

The fix plumbs stream provenance through the entire pipeline. A new `OutputStream { Stdout, Stderr }` enum lives in `cook-contracts` (the shared kernel — `cook-luaotp` already depends on it). `WorkResult::output_lines` becomes `Vec<(OutputStream, String)>`; `execute_shell` tags stderr lines `Stderr` and stdout lines `Stdout` at the point they're decoded from `output.stderr` / `output.stdout`; `execute_test` does the same. `EngineEvent::OutputLine` carries `stream: OutputStream` instead of `is_stderr: bool` and re-exports the contracts enum as `cook_engine::Stream` for callers that don't already depend on `cook-contracts`. The `cook-cli` bridge maps the contracts enum to the wire-format `cook_progress::Stream` enum (the two are isomorphic; the second exists because `cook-progress` is `serde::Serialize` for the JSON renderer and `cook-contracts` is intentionally serde-free).

**Why a patch-level fix.** This is a correctness fix for the implementation, not a language change. The `events.jsonl` wire-format `stream` field was already documented in the JSON renderer's behavior (a `"stdout"` / `"stderr"` lowercase string) and in the `node-output` schema test in `cli/crates/cook-progress/src/render/json.rs`; the field's *valid values* are unchanged, but `"stderr"` was unreachable end-to-end pre-fix. No Cookfile, conformance fixture, or downstream consumer is broken by this change — consumers that branched on `"stream"` were already prepared for both values.

**Why CS-0043 in v0.7.** The `events.jsonl` schema is a v1.0 stability surface (App. E.11 — to be filed). Cutting v0.7 with a known wire-format lie would compromise that stability promise: a future v1.0 audit might (correctly) flag the all-`"stdout"` behavior as the documented contract, freezing the bug. Recording this fix as a changelog-only entry against v0.7 (no version bump, no Standard prose change) closes the gap before any downstream tooling pins on the broken behavior.

**Pre-1.0 follow-up.** App. E should track the absence of a normative `events.jsonl` schema section. Per the Cook Standard versioning policy (CS-0012), public wire formats produced by the reference implementation and consumed by external tooling deserve normative section status before v1.0. The current `events.jsonl` shape is documented only in `cli/crates/cook-progress/src/render/json.rs`; promoting it to a normative section (covering `type` discriminator, field schemas per event kind, the `ts` / `v` envelope, and the `stream` field's `"stdout"` / `"stderr"` enumeration) is a prerequisite for v1.0.

**Reference:** the CS-0043 commit on the active branch.

## D.44. CS-0044 — Shared `fs.*`/`path.*`/`cook.platform.*` implementation across register and execute VMs. [#changes.cs-0044]

**Date:** 2026-05-03

**Version:** v0.7

**Sections affected:** §{lua.fs-helpers} (new Note 6.5.2 pinning the cross-phase identity contract).

**Summary.** Pre-CS-0044, the reference implementation shipped two parallel implementations of `fs.*`, `path.*`, and `cook.platform.*` — one in `cli/crates/cook-register/src/{fs,path,platform}_api.rs` for the register-phase VM, the other in `cli/crates/cook-luaotp/src/{fs,path,platform}_api.rs` for the execute-phase worker VMs. `path_api.rs` was byte-identical between the two crates; `platform_api.rs` was structurally identical with only the `cook`-table-fetch shape differing; `fs_api.rs` had diverged because the execute-phase VMs needed per-call working-directory resolution (CS-0017 multi-Cookfile imports) while the register-phase VM captured the cwd once at registration. The drift was a v1.0 risk: bug fixes to one copy would silently miss the other, and the §{lua.fs-helpers} "Phase: Both" promise was implementation-fragile.

CS-0044 extracts the three tables into a new `cli/crates/cook-lua-stdlib` crate and parameterizes the `fs.*` cwd source over a `WorkingDirSource` enum with two variants — `Static(PathBuf)` for register-phase use and `Live(Arc<Mutex<PathBuf>>)` for execute-phase use. Both `cook-register` and `cook-luaotp` now depend on `cook-lua-stdlib` and call its `register_fs_api` / `register_path_api` / `register_platform_api` factories instead of carrying local copies. Note 6.5.2 makes the identity contract normative: every observable property of these tables MUST be identical between the two phases, modulo the working-directory difference inherent to the multi-Cookfile-import case.

**Implementation note.** The `fs_api` unification adopts the execute-phase (`cook-luaotp`) cwd-resolution pattern as canonical: `WorkingDirSource::resolve()` is called inside every `fs.*` closure rather than the cwd being captured at registration. For the `Static` variant this is operationally indistinguishable from the previous register-phase behavior (the path is cloned from a `PathBuf` instead of read through a `Mutex`); for the `Live` variant it preserves the CS-0017 contract that lets one worker VM serve items from multiple Cookfiles. As a side effect, any latent CS-0017 cwd-resolution bug in the pre-CS-0044 register-phase `fs_api` is fixed for free — the unified implementation has only one cwd-resolution code path and it is correct under both call patterns.

**Hard non-changes.** CS-0044 does not introduce a filesystem sandbox; it replicates today's behavior exactly. The user-visible Lua API surface is unchanged — every existing recipe continues to work without modification. Sandboxing is a follow-up that this consolidation enables but does not deliver.

## D.45. CS-0045 — Project-root sandbox for `cook` / `test` / `chore` step Lua bodies. [#changes.cs-0045]

**Date:** 2026-05-03

**Version:** v0.8

**Sections affected:** §{lua.fs-helpers} (new §6.5.8 normative sandbox section, new §6.5.9 normative shell-escape-hatch section), §{recipes.plate-step} (new Note 4.7.2 pinning the plate-step exemption).

**Summary.** Pre-CS-0045, the Cook Lua API surface confined `fs.*` paths to the recipe's *working directory* (a path-resolution anchor) but applied no upper bound: a `cook` step body could read `/etc/passwd`, write to `/tmp/leaked`, `..`-traverse out of the project, or shell out via `os.execute("rm -rf /")`, and the implementation would happily comply. The cache fingerprint records the captured `lua_code` string but cannot capture out-of-project side effects, so a hostile or accidentally-broken cook step could silently corrupt the user's environment while still appearing cache-clean.

CS-0045 closes that gap by adding a normative project-root sandbox to the Cook Lua API. Every path-taking entry in `fs.*` (§6.5.1–6.5.7) MUST further confine its argument to the **project root** (the workspace root, §{modules.workspace-root}) when invoked from a sandboxed context. The sandboxed contexts are `cook` / `test` / `chore` step bodies and all register-phase Lua execution (config blocks, inline-Lua, cook-step register-time orchestration); the explicit non-sandboxed context is the `plate` step body, because shipping outside the project root is what `plate` exists for (§{recipes.plate-step-not-sandboxed}). The sandbox check is **lexical**: paths are normalized via component-by-component `..` / `.` resolution without consulting the filesystem, so `fs.write` and `fs.mkdir_p` continue to succeed against not-yet-existent paths. An out-of-project path raises a Lua runtime error tagged with `CS-0045`; no silent truncation or fallback is permitted.

The companion §6.5.9 deals with the Lua-side shell escape hatches `os.execute` and `io.popen`: in a sandboxed context, both MUST raise rather than spawn a child process, because arbitrary shell text bypasses the path-confinement check and the unit's `command_hash`. `cook.sh` (§6.3.1) is unaffected — it already runs with the recipe's working-directory rooting and is recorded in the unit's command hash. Plate-step bodies retain unrestricted access to all three.

**Implementation note.** The reference implementation lives in `cli/crates/cook-lua-stdlib/src/sandbox.rs` and `cli/crates/cook-lua-stdlib/src/shell_guard.rs`. `SandboxPolicy` is an enum with two variants (`Off` and `Confined { project_root }`) and `SandboxSource` mirrors the `WorkingDirSource` split from CS-0044 so the execute-phase worker pool can flip the policy per work item. `WorkPayload::LuaChunk` gains a `step_kind: StepKind` field that the worker reads on each item: `Cook` / `Test` / `Chore` install `Confined`, `Plate` installs `Off`. The register-phase VM is always installed with `Confined`, since every register-phase Lua execution participates in the captured cache fingerprint.

**Why `path.*` is not sandboxed.** `path.*` (§6.6) is pure string manipulation. It performs no I/O. Constraining `path.join` to in-project results would force authors to reach for `string.format` to build out-of-project path strings for plate steps, which is a worse outcome than leaving `path.*` as a transparent string utility. The sandbox is on the I/O-performing surface only; that is sufficient to enforce hermeticity.

**Why tests are sandboxed identically to cook steps.** `test` step bodies are non-cacheable but **hermetic-by-intent**: the user runs them under `cook --test` precisely to verify the project's correctness without depending on the host environment. A test body that reads `/etc/passwd` to fabricate an outcome is a worse outcome than the test failing to compile under the sandbox. Authors who want to test a plate-step's behavior should invoke that plate from the test's `cook.sh` (which runs in the recipe's working directory and is recorded in the test's `command_hash`) rather than re-creating the plate's escape-hatch surface inside the test body.

**Hard non-changes.** CS-0045 is purely additive on the rejection side: a Cookfile that previously did not depend on out-of-project I/O continues to work without modification. The user-visible Lua API signatures of `fs.*` and `path.*` are unchanged; only their behavior on out-of-project arguments tightens. `os.execute` and `io.popen` semantics in plate steps and outside Cook entirely (e.g. running a Cookfile-generated Lua script directly) are unchanged.

**Audit fixtures.** `examples/register_audit_sandbox_escape/walkthrough.sh` exercises the cook-step / config-block / `os.execute` rejection paths and the plate-step exemption. `examples/luaotp_audit_path_traversal/walkthrough.sh` exercises the relative `..` traversal, absolute `fs.write`, `fs.glob` outside-pattern, and `io.popen` rejection paths and pins the no-regression case (in-project paths MUST still succeed).

**Reference:** the CS-0045 commit on the `cs-0045-sandbox-cook-steps` branch.

## D.46. CS-0046 — Removal of `cook --test` runner. [#changes.cs-0046]

**Date:** 2026-05-03

**Version:** v0.8 (unreleased)

**Sections affected:** none in the normative chapters — the Cookfile language, `test_step` syntax (§{recipes.test-step}), and `cook.add_test` (§{lua.api-overview}) are unchanged. App. E.10 and App. E.11 have informative cross-references to the runner that are updated to reflect its removal. Reference-implementation: `cli/crates/cook-cli/src/cli.rs` (drops the `--test`, `--filter`, `--verbose`, `--timeout-multiplier`, `--wrapper`, `--list` flags), `cli/crates/cook-cli/src/main.rs` (drops the dispatch arm), `cli/crates/cook-cli/src/pipeline.rs` (drops `cmd_test`), `cli/crates/cook-cli/src/test_output.rs` (deleted — JUnit XML / JSON / terminal-summary scaffolding that only `cmd_test` consumed), `examples/cli_audit_test_exit_code/` and `examples/cmd_test_inferred_deps_e10/` and `examples/cross_cookfile_test/` (deleted — fixtures that exercised the removed runner end-to-end).

**Summary.** The pre-1.0 reference implementation shipped a `cook --test` subcommand whose contract was always provisional: it discovered every `recipe` containing a `test_step` and shelled the bodies through the same DAG executor as `cook NAME`. The bundled sub-flags (`--filter`, `--verbose`, `--timeout-multiplier`, `--wrapper`, `--list`) were stubs that printed `not yet implemented` warnings and then ran the full set anyway; the structured-output scaffolding in `test_output.rs` (JUnit XML, JSON, terminal summary) was never wired to engine results. Cutting v1.0 with this surface in place would freeze a half-shape — author-visible flags that don't do what their names imply, plus an exit-code split (CS-0041, App. E.11) that depends on structured test results that don't exist — into the stability promise. CS-0046 removes the runner outright; v1.0 will define a test execution surface whose semantics are designed up front rather than retrofitted onto today's stub.

**What stays.** The recipe-level `test` step (§{recipes.test-step}, App. A.4) is unchanged: parsing, register-phase recording via `cook.add_test`, and execute-phase invocation through the same worker pool that runs `cook` and `plate` steps all behave exactly as before. A test step continues to participate in the DAG of its enclosing recipe, and `cook RECIPE_NAME` still runs every step in that recipe — including its `test_step`s — exactly as it did before CS-0046. The removal is scoped to the CLI dispatch flag and its sub-flags; no language-surface, register-phase, or execute-phase behavior changes.

**What goes.** The `--test`, `--filter`, `--verbose`, `--timeout-multiplier`, `--wrapper`, and `--list` CLI flags; the `cmd_test` dispatcher in `pipeline.rs` and its workspace/single-Cookfile branches; the unused `test_output.rs` result-rendering module (`TestStatus`, `TestCaseResult`, `TestSuiteResult`, `TestResults`, JUnit XML emitter, JSON emitter, terminal summary). The audit fixtures under `examples/cli_audit_test_exit_code/`, `examples/cmd_test_inferred_deps_e10/`, and `examples/cross_cookfile_test/` are deleted; each was scoped to behavior of the removed runner.

**Why now.** v0.8 is the last opportunity before the v1.0 stability cut to drop a half-finished surface without a deprecation cycle. The `--test` runner's flag set was actively misleading (`--filter` ignored, no per-test timing, no machine-readable output) and its CI failure mode had needed a CS-0041 patch to surface non-zero exits at all. Shipping a v1.0 `cook --test` would then commit to backwards compatibility for behavior nobody wants long-term. Removal first, redesign later, is the conservative path.

**Forward-looking note.** Test execution is **implementation-defined** in v0.8. The Cook Standard's normative chapters describe what a `test_step` *is* (§{recipes.test-step}) and how a conforming implementation registers and runs one (§{lua.api-overview}); whether and how a CLI surface aggregates multiple recipes' test steps into a single batch run is not specified. The v0.7-track `cook --test` runner was the reference implementation's first attempt at such a surface; it has been withdrawn pending redesign.

## D.47. CS-0047 — DAG viewer split out + feature-gated. [#changes.cs-0047]

**Date:** 2026-05-03

**Version:** v0.8

**Sections affected:** none in the normative chapters (the `cook --dag` viewer is a tooling affordance of the reference implementation, not a Cookfile-language surface). Reference-implementation: new `cli/crates/cook-dag-viewer/` crate (carries `dag_data.rs`, `dag_server.rs`, the embedded `dag_viewer.html`, and the `cmd_dag` data-shaping + serve entry point); `cli/crates/cook-cli/Cargo.toml` (drops `tiny_http` and `open` direct deps, adds `cook-dag-viewer` as an optional dep behind a new `viewer` cargo feature); `cli/crates/cook-cli/src/{main.rs,pipeline.rs}` (the `mod dag_data` / `mod dag_server` modules are gone, and `cmd_dag` becomes feature-gated — viewer-less builds short-circuit with a clear error pointing at `--features viewer`); `cli/Cargo.toml` (registers the new workspace member). App. D Versions index gains a v0.8 row alongside CS-0045 and CS-0046.

**Summary.** Splits the browser-based DAG viewer (`cook --dag`) out of `cook-cli` into its own `cook-dag-viewer` crate, and gates the wiring on `cook-cli` behind a new `viewer` cargo feature. Pre-CS-0047, the viewer's `tiny_http` HTTP server, `open`-based browser-launch, and ~400 lines of embedded HTML/JS were unconditional dependencies of every `cook` build. Post-CS-0047, vanilla `cargo install cook-cli` (and `cargo build` in the workspace) produces a slim binary with no HTTP-server dependency; users who want the viewer rebuild with `cargo build --features viewer` (or `cargo install --features viewer`). Invoking `cook --dag` against a viewer-less binary produces a clear diagnostic naming the rebuild flag, rather than a `command not found` surprise.

**`cook --dag` is implementation-optional.** The Standard does not require conforming implementations to ship a graphical DAG viewer. The reference implementation provides one as a tooling affordance, and its presence is observable through the `cook --dag` flag; whether that flag is wired in any given build of the reference binary is now a build-time decision. Other implementations MAY omit `--dag` entirely, MAY ship their own viewer under a different flag, or MAY emit a JSON snapshot for an external tool. The pre-1.0 references to `cmd_dag` in App. D and App. E are descriptive of the reference implementation's behavior, not normative requirements.

**Why off by default.** Two reasons. First, `tiny_http` is a network-listening dependency: even bound to `127.0.0.1`, an HTTP server in the default build expands the supply-chain surface area of a tool whose primary job is parsing Cookfiles and orchestrating subprocesses. Second, the viewer is a developer-affordance flag — the median `cook` invocation in CI, in containers, and in scripted callers never reaches `--dag`. Pulling `tiny_http`, `open`, and the embedded HTML asset into every binary to support a flag most users never run is the wrong default. Distros and packagers who want the viewer enable it explicitly; everybody else gets a smaller binary and a smaller dependency footprint. The opposite default ("on, but opt-out via `--no-default-features`") would have been defensible for symmetry with cargo idioms, but the supply-chain argument and the "most users never run `--dag`" observation tip the choice toward `default = []`. This is the conservative pre-1.0 default; if v1.0 ships with the viewer reaching feature parity with mainstream visualization tools and gaining broad enough adoption that turning it off becomes the surprising case, the default can be revisited then.

**Reference-implementation contract.** When the `viewer` feature is enabled, `cook --dag <RECIPE>` MUST behave exactly as it did pre-CS-0047: prepare the wave-grouped DAG, start a local HTTP server bound to a random `127.0.0.1` port, open the user's default browser at that port, and serve the embedded viewer page until interrupted. When the feature is disabled, `cook --dag` MUST exit with a non-zero status and a diagnostic that names the rebuild flag (`cargo build --features viewer`). The split is a refactor — no behavior visible to a viewer-feature-enabled build is changed. The data-shaping helper (`build_wave_dag_data` from §App. E.10's CS-0027 work) and the serve loop (`serve_dag`) move bytes-for-bytes into the new crate; the only Rust-level change is replacing the local `CookError` use in `dag_server.rs` with a crate-local `ViewerError` variant that the cook-cli wrapper maps back to `CookError::Other` at the boundary.

**Conformance impact.** None. The Standard does not pin `cmd_dag` behavior, the conformance harness does not exercise it, and no Cookfile authoring surface is touched.

**Reference:** the CS-0047 commit on the `cs-0047-dag-viewer-extract` branch.

## D.48. CS-0048 — Wire-format schema versioning for events.jsonl, recipe cache, DAG viewer JSON. [#changes.cs-0048]

**Date:** 2026-05-03

**Version:** v0.8 (unreleased)

**Sections affected:** none in the normative chapters (Cookfile language and execution semantics are unchanged). App. E.11 (`events.jsonl` wire format) and App. E.12 (events.jsonl normative-section gap) gain informative cross-references to this entry. Reference-implementation: `cli/crates/cook-progress/src/event.rs` (new `PROGRESS_SCHEMA_VERSION` constant), `cli/crates/cook-progress/src/render/json.rs` (writer reads the constant; new `check_schema_version` reader helper with `SchemaCheckError`), `cli/crates/cook-progress/src/log_store.rs` (the `events.jsonl` line writer reads the same constant), `cli/crates/cook-progress/src/lib.rs` (re-exports), `cli/crates/cook-cache/src/store.rs` (`RecipeCache.version` renamed to `schema_version` with serde `alias = "version"` for bincode-positional-back-compat; load-time policy doc-comment), `cli/crates/cook-cache/src/manager.rs` (test field rename), `cli/crates/cook-dag-viewer/src/lib.rs` (new `VIEWER_SCHEMA_VERSION` constant), `cli/crates/cook-dag-viewer/src/dag_data.rs` (`schema_version: u32` added to `WaveDagData`), `cli/crates/cook-dag-viewer/src/dag_viewer.html` (entry-point JS guard refuses payloads with `schema_version > 1`).

**Summary.** Cook ships three serialized wire formats that v1.0 commits to as long-lived integration surfaces: the `events.jsonl` event stream (App. E.11), the on-disk recipe cache (`.cook/cache/<recipe>.bin`, referenced by App. D.42), and the DAG-viewer JSON payload served by `cook --dag` to the embedded viewer. Pre-CS-0048, only the recipe cache carried an explicit version number on the wire (an integer field tagged `version` and equality-checked against `CACHE_VERSION` on load); the events.jsonl format already emitted a top-level `v` integer per line but it was a hardcoded literal with no exported constant or reader-side policy, and the DAG-viewer JSON had no version field at all. CS-0048 pins all three to `schema_version = 1` at the v1.0 cut and codifies the read/write/evolution policy that future v1.x revisions will follow.

**The three constants.** A single named constant per crate sources the writer-side value, so a future bump is a single-line edit:

- **`cook_progress::PROGRESS_SCHEMA_VERSION: u32 = 1`** — the value emitted as the top-level `v` field on every `events.jsonl` line and on every `--output=json` line. Both the `JsonWriter` renderer and the `LogStore` event-recorder read from this constant.
- **`cook_cache::store::CACHE_VERSION: u32 = 3`** (re-exported from `cook_fingerprint::record`) — the value of the `schema_version` field on every persisted `RecipeCache` and a fingerprint input. The dual role is intentional: a fingerprint-side bump (any change to `StepEntry` / `FileRecord` / per-step keys) is by definition an incompatible on-disk-format change, so the two move together.
- **`cook_dag_viewer::VIEWER_SCHEMA_VERSION: u32 = 1`** — the value of the `schema_version` field on the `WaveDagData` JSON payload that `cook --dag` splices into the embedded viewer. The matching JS-side constant in `dag_viewer.html` enforces the read policy in the viewer.

**Read policy.** A reader MUST refuse a payload whose `schema_version` exceeds the highest version it recognises ("the file was written by a future cook"). A reader MUST accept any `schema_version` at or below its highest recognised version (additive-only evolution within a major version means older writers' missing fields are filled by `#[serde(default)]`). Concretely:

- `events.jsonl`: `cook_progress::check_schema_version` returns `Ok(v)` for `v <= PROGRESS_SCHEMA_VERSION` and `Err(SchemaCheckError::Unsupported { found, max_known })` otherwise. Lines whose `v` field is missing or non-integer are also refused (`MissingVersion`, `InvalidJson`).
- Recipe cache: `RecipeCache::load` returns `None` (cache-miss; the file is regeneratable) if the persisted `schema_version` does not match `CACHE_VERSION`. The check is exact equality today, not the forward-compatible `<= CACHE_VERSION` form, because pre-v1.0 caches used a positional bincode layout where field evolution was non-additive — older files cannot be safely deserialized into the current struct shape regardless of intent. The `<=` form takes effect at v1.0 once the additive-only contract starts.
- DAG-viewer JSON: the embedded JS at the entry point of `dag_viewer.html` refuses payloads whose `schema_version` exceeds `VIEWER_SCHEMA_VERSION`, replacing the canvas with a diagnostic banner and a rebuild instruction. A missing field is treated as legacy v1 (defensive default — no v0.x payload omitted the field, but the guard prefers a render to a hard error if the field is somehow stripped in transit).

**Evolution policy.** Future revisions MUST be additive-only within a major version. New fields are introduced via `#[serde(default = "...")]` (or, on the viewer side, a JS `typeof === 'undefined'` guard with a sensible default). Existing fields are NEVER renamed or removed; enum variants are NEVER renamed or removed. A `schema_version` bump signals an incompatible structural change — at v1.0 all three formats lock in `schema_version = 1` (events.jsonl, DAG viewer) / `schema_version = CACHE_VERSION` (recipe cache); v1.x revisions that need an incompatible change bump to 2 and document the migration in App. D.

**Why the events.jsonl envelope stays flat.** The plan considered three shapes for the events.jsonl envelope: (a) wrap each line in a `LineEnvelope { schema_version, event }` outer struct; (b) flatten a shared `schema_version` field onto every variant via `#[serde(flatten)]`; (c) build the envelope manually next to the existing internally-tagged enum. The reference implementation already emits a flat envelope with `ts`, `v`, and the event payload's keys interleaved in lex order ([CS-0035](#changes-cs-0035)) — i.e. shape (c). CS-0048 does NOT switch to (a): the lex-order invariant is a documented v1.0 stability contract, and a nested `event` object would shift every key one level deeper and reorder the payload around it. Instead, the existing flat shape is recognised as the envelope, the existing `v` field is recognised as the schema-version field (this matches the App. E.12 description of the events.jsonl envelope already), and the only behavioural change is that `v` now sources its value from `PROGRESS_SCHEMA_VERSION` and reader-side validation is provided by `check_schema_version`. The wire bytes are unchanged from the v0.7 cut.

**Why `RecipeCache.version` was renamed to `schema_version`.** The v0.7 field name `version` was a vestige of the cook-cache crate's pre-cook-fingerprint era when the same constant tagged both a schema-version and a "cook ABI" generation. Post-CS-0048, the field is exclusively a schema-version, and the rename aligns its name with the same semantics as the events.jsonl `v` field and the DAG-viewer payload's `schema_version` field. `bincode` is positional, so the rename is invisible to existing on-disk caches — the field's serialized position is unchanged. A `#[serde(alias = "version")]` is added defensively for any future JSON / non-positional encoding of the same struct.

**Why the DAG-viewer JSON gains a field.** The `--dag` viewer is the only one of the three formats with a JS-side reader living outside the Rust workspace. Pre-CS-0048, an experimental viewer on top of an older payload (or a future cook on top of an older viewer) had no way to detect the mismatch — the JS would just render whatever shape it found, with silent omissions on missing fields and silent ignores on unrecognised ones. The new `schema_version` field plus the JS guard at the entry point of `dag_viewer.html` makes the mismatch loud and named.

**Conformance impact.** The conformance harness exercises Cookfile parsing, register-phase semantics, and execute-phase semantics; none of these surfaces touch the three wire formats covered by CS-0048. The new behaviour is pinned by unit tests in `cook-progress` (`writer_emits_schema_version_constant`, `check_schema_version_accepts_current_version`, `check_schema_version_accepts_lower_versions`, `check_schema_version_rejects_higher_versions`, `check_schema_version_rejects_missing_v_field`, `check_schema_version_rejects_invalid_json`) and `cook-cache` (`load_v2_returns_none_via_version_check` retained, new `load_future_schema_version_returns_none` and `schema_version_field_round_trips_at_cache_version`).

**Cross-references.** App. E.11 (informative) describes the `events.jsonl` envelope keys (`ts`, `v`, `type`) and the per-event field schemas; CS-0048 is the entry that pins `v` as `PROGRESS_SCHEMA_VERSION`. App. E.12 (informative) flagged the lack of a normative section for `events.jsonl`; CS-0048 is the v0.x reference-implementation half of that work — pinning the version constant and the read policy ahead of the v1.0 normative-section pass that App. E.12 calls for.

**Reference:** the CS-0048 commit on the `cs-0048-schema-versioning` branch.

## D.49. CS-0049 — Rust public-API hygiene: `#[non_exhaustive]` on language-surface enums + cook-luagen dead-code removal. [#changes.cs-0049]

**Date:** 2026-05-03

**Version:** v0.8

**Sections affected:** none in the Standard prose. ## D.50. CS-0050 — Engine MUST create parent directories of declared `cook`-step outputs before the step runs. [#changes.cs-0050]

**Date:** 2026-05-03

**Version:** v0.8

**Sections affected:** §{exec.output-materialisation} (new §8.7 normative subsection in the Execution model chapter; §8.7 "Diagnostic ordering" renumbered to §8.8). App. D Versions index v0.8 row extended to include CS-0050. `standard/scripts/slug-mapping.ts` registers the new clause and renumbers the diagnostic-ordering slug. Reference-implementation: `cli/crates/cook-engine/src/executor.rs` (new `ensure_output_parent_dirs` helper, threaded through `process_ready` and the interactive main-thread path; new `failures` parameter on `process_ready`; three new unit tests pinning the create / parent-is-file diagnostic / idempotent paths plus a unit-helper test for the no-parent edge case). New audit fixture `examples/engine_audit_auto_mkdir_outputs/` exercises the end-to-end behaviour against the built `cook` binary.

**Summary.** Pre-CS-0050, the engine dispatched a `cook` step's shell text into a worker subprocess without first ensuring that the directory of the step's declared output existed. A recipe such as `cook "build/out/foo.txt" using { ... }` would fail with `bash: build/out/foo.txt: No such file or directory` unless the author manually wrote `mkdir -p build/out` (or `mkdir -p $<out.dir>`) at the head of every cook-step shell body. An audit pass found this boilerplate in nearly every example fixture; it is unconditional, mechanical, and exactly the kind of deterministic prelude a build engine can synthesize on the author's behalf.

CS-0050 closes that gap by adding a normative §{exec.output-materialisation} subsection that requires a conforming implementation to ensure the parent directory of every output path declared by a `cook` step exists before invoking the step's shell text. The creation MUST be idempotent — `mkdir -p`-style — and MUST NOT proceed if the parent path resolves to a non-directory (e.g., a regular file already exists at that path); the implementation MUST in that case surface a clear diagnostic naming the output and the offending parent, and MUST NOT execute the step's shell text. Output paths are resolved against the unit's working directory using the same resolution that the cache fingerprint applies (per §{exec.cache}), so the rule composes cleanly with cross-Cookfile imports (CS-0017, CS-0044) and the `//`-anchored sigil import path form (CS-0032).

**Scope.** The rule applies to `cook` steps only. `plate` (§{recipes.plate-step}) and `test` (§{recipes.test-step}) steps declare no cacheable output paths and are therefore not covered; authors of `plate` and `test` step bodies that need to create directories MUST do so explicitly, exactly as before. Inputs are not affected — a missing input remains a hard error (the user's bug). An output path with no parent component (a path that resolves to the working directory itself) is a no-op.

**Why this is the engine's job.** Every `cook` step declares its output paths up-front so the cache can fingerprint them, the DAG can route dependencies, and the §{exec.cross-recipe-ordering} collision detection (CS-0042) can reject duplicate writers at plan time. The parent directory of a declared output is a mechanical consequence of that declaration; making the author repeat it in shell text adds boilerplate without improving determinism, cache fidelity, or sandbox guarantees. The precedent in adjacent build systems is consistent (Make's `$(@D)` idiom, Ninja's autogenerated mkdir rules, Bazel's sandbox-setup parent-dir creation); Cook now matches.

**Idempotence and concurrency.** `create_dir_all`-style creation is concurrency-safe under POSIX: multiple processes (or step-group siblings) targeting the same parent directory all succeed without contention. CS-0042 already prevents two recipes from declaring the same output path without a dep edge, but two recipes legitimately sharing a parent directory (e.g., both writing into `dist/`) is the common case and continues to work without synchronization. A symlinked parent that points to an existing directory is idempotently accepted; a symlinked parent that points at a regular file is rejected with the same diagnostic shape as a direct regular-file collision.

**Sandbox interaction.** The pre-step directory creation is the engine acting on a path that the register phase has already declared as an output of the unit. It is not a Cook Lua API call and is therefore independent of the `fs.*` sandbox in §{lua.fs-helpers} (CS-0045). Out-of-project output paths are governed by the cache contract in §{exec.cache} and the path-resolution rules in §{modules.workspace-root}; CS-0050 does not relax them.

**Existing recipes.** The rule is purely additive on the success side: a recipe that already contains `mkdir -p $<out.dir>` (or any equivalent) in its `cook`-step shell text continues to work. The author's explicit `mkdir -p` becomes idempotently harmless after the engine's pre-step `mkdir -p` runs. Sweeping the redundant boilerplate out of existing fixtures is left to a follow-up commit; CS-0050 does not require it.

**Implementation note.** The reference implementation lives in `cli/crates/cook-engine/src/executor.rs`. A new private helper `ensure_output_parent_dirs(work_node) -> Result<(), String>` walks `work_node.cache_meta.output_paths` (the gate is non-empty `cache_meta.output_paths`, which is set only for cook steps — `plate` and `test` register-time helpers leave `cache_meta = None`), resolves each output against `work_node.working_dir`, takes the parent component, checks for a non-directory at that parent, and falls through to `std::fs::create_dir_all`. The helper is invoked from `process_ready` immediately after `NodeStarted` and before `pool.submit`, and from the interactive main-thread loop immediately before `run_interactive_on_main`. On `Err`, the call site surfaces a `NodeFailed` event with a `CS-0050:`-tagged diagnostic, records the failure, and cancels the transitive dependent subtree — the same control flow the executor uses for a worker-pool failure.

**Audit fixture.** `examples/engine_audit_auto_mkdir_outputs/walkthrough.sh` cleans any pre-existing `build/` directory, runs `cook build` against the just-built debug binary, and asserts that the declared output `build/out/foo.txt` exists with the expected content even though the recipe does not contain `mkdir -p`. The fixture is a single positive case; the negative parent-is-file path is pinned by the `test_executor_cs_0050_parent_is_file_diagnostic` unit test in `cook-engine`.

**Conformance impact.** None at the harness level — the conformance corpus exercises Cookfile parsing, register-phase semantics, and cross-recipe scheduling, and a recipe whose declared output sits in a missing subdirectory was already accepted by parser and register paths. The runtime behavioural change (pre-step parent-dir creation) is pinned by the new `cli/crates/cook-engine/src/executor.rs` unit tests and the `examples/engine_audit_auto_mkdir_outputs/` audit fixture. `cargo test --workspace` passes (773 → 773 + 4 new tests).

**Reference:** the CS-0050 commit on `cs-0050-auto-mkdir-output-parents`.

## CS-0051 — Chore body executes as a single drain

**Date:** 2026-05-04

**Version:** v0.8

**Sections affected:** §{chores.default-interactive} (4a.3 prose tightened; the per-step draining sentence is replaced with a normative single-drain rule, and Example 4a.3.1 commentary is aligned). App. D Versions index v0.8 row extended to include CS-0051. Reference-implementation across `cli/crates/cook-contracts`, `cli/crates/cook-register`, `cli/crates/cook-luagen`, `cli/crates/cook-engine`, and `cli/crates/cook-progress`; the layered Rust changes land on `cook-progress-cargo-style` over the CS-0051 commit series. Design spec in `docs/superpowers/specs/2026-05-04-cook-progress-chore-handoff-design.md`.

**Summary.** Pre-CS-0051, a chore body's shell steps each compiled to a separate interactive work unit, and the engine treated each as its own drain (§{exec.interactive-drain}). §4a.3 carried the explicit normative claim "every shell step is its own draining unit." The reference progress renderer therefore emitted one `Running <chore>` line per shell step and leaked the parser's internal `@<line>` fallback labels into the verb stream as `Cooked <chore>/@N` between body steps. The chore-vs-recipe semantic — that a chore takes the controlling terminal as one continuous handoff session — was implicit; the normative wording contradicted user expectation by permitting other recipes' work to interleave between body steps.

CS-0051 makes the single-drain semantic normative for chore bodies. A conforming implementation MUST execute every imperative-region step of a chore body within a single drain: from the moment the first imperative-region step begins, no work unit of any other recipe MAY run until the body completes. The body's shell steps and execute-phase Lua steps run in source order on the controlling terminal under one shared drain window. The interactive-drain rule of §{exec.interactive-drain} continues to govern individual interactive units in non-chore contexts (recipe-internal `@` steps); only the chore-body case is collapsed.

**Scope.** The single-drain rule covers chore bodies only. Recipe-internal `@`-prefixed shell steps and the `interactive = true` flag on `cook.add_unit` (§{lua.add-unit}) continue to drain per-unit per the existing §{exec.interactive-drain} rule. A chore body's failure semantics are unchanged from the per-unit case: the first failing step ends the drain and propagates through the cancellation cascade as before. The prior "every shell step is its own draining unit" wording's implication that a chore body could partially complete with mixed-recipe interleaving is removed.

**Implementation note.** The reference implementation flags chore-body units at register time. `cook-contracts::WorkPayload::Interactive` gains an `is_chore: bool` field; the runtime's `cook.add_unit` registrar (`cook-register/src/unit_api.rs`) reads `CaptureState.current_chore_active` (set by `cook._enter_chore` / `cook._exit_chore`) and stamps the flag accordingly. `cook-luagen::compile_chore` ensures every chore-body unit (shell or Lua-bundle) is emitted with `interactive = true` so they all reach the executor's interactive-drain queue. The executor (`cli/crates/cook-engine/src/executor.rs`) groups consecutive same-recipe interactive entries marked `is_chore = true` into a single drain window, emits one `InteractiveStart` / `InteractiveEnd` event pair for the whole window, and on mid-drain failure attributes the failure to the body via a `step k/n: …` diagnostic. The progress renderer (`cli/crates/cook-progress/src/render/event_writer.rs`) renders the window as a single `Running <chore>` line and suppresses per-step verb lines for the body's anonymous shell units.

**Conformance impact.** None at the parser level; CS-0051 is an execute-time rule. The conformance corpus does not currently exercise chore-body scheduling shape — the reference-implementation behaviour change is pinned by new unit tests in `cook-engine` (chore-window grouping; mid-drain failure attribution; legacy non-chore interactive regression) and end-to-end rendering snapshots in `cook-progress`.

**Test note.** The `cook-register` runtime test suite pins the chore-mode flag plumbing via `add_unit_inside_chore_marks_payload_is_chore_true` and `add_unit_outside_chore_marks_payload_is_chore_false` — the two canonical cases of `_enter_chore` / `_exit_chore` round-tripping into `WorkPayload::Interactive::is_chore`.

**Lua-bundle integration.** The reference implementation enforces the single-drain rule for both shell-step and Lua-bundle chore bodies. `WorkPayload::LuaChunk` gains an `is_chore: bool` field set by the runtime registrar from `CaptureState.current_chore_active`, mirroring the `Interactive` plumbing. The executor's chore-window pre-walk admits either payload variant when the flag is set; the per-step execution branches on payload kind — `Interactive` runs via `run_interactive_on_main` on the drain thread, `LuaChunk` submits to the worker pool and synchronously awaits the single result (chore windows enter only when `pending == 0`, so the next `pool_rx.recv()` returns the in-flight Lua item). The user-observable property of CS-0051 — that a chore body executes as one drain on the controlling terminal — holds for any mix of shell and Lua imperative-region steps in the body.

**Reference:** the CS-0051 commit series on `cook-progress-cargo-style`.

## CS-0052 — Declared toolchain pinning for cross-machine cache correctness [#changes.cs-0052]

**Date:** 2026-05-04

**Version:** v0.8

**Sections affected:** §{exec.cache} Note 8.6.2 amended (final paragraph added — declared-toolchain mechanism). Reference-implementation: `cli/crates/cook-cache/src/cloud_config.rs` (`CacheSection.tools: Vec<String>` field plus `cache_tools()` accessor), `cli/crates/cook-fingerprint/src/context.rs` (`DeclaredToolError`, `ExecutionContext::probe_with_declared_tools`, `ExecutionContext::declared_tools_hash`, `ExecutionContext::declared_tools`, `compute_declared_tools_hash`; `step_context_hash` folds the declared-tools hash before the per-step tool fold), `cli/crates/cook-engine/src/run.rs` (build-time wiring at the cache bootstrap site; misdeclaration becomes `EngineError::CacheError` before any unit registers). Design spec in `standard/specs/2026-05-04-cache-declared-tools-design.md`.

**Summary.** Cook's per-step `context_hash` already folds the SHA-256 content of every binary the engine can identify by walking the rendered command body — but that walk has a documented coverage limit (first executable token of each newline-separated statement; tools invoked downstream of `&&`/`||`/pipes/`bash -c` etc. are not captured). On a single-developer machine the gap is harmless because the elided tools are constant; on a shared cache it is a soundness gap. CS-0052 introduces an explicit, opt-in **declared toolchain** — a list of executable names the project promises will be relevant to every unit's identity regardless of whether the auto-detection sees them in the literal command text. Each declared name is resolved via `which` at build start, the resulting binary is SHA-256'd, and a single combined hash folds into every step's `context_hash`. An empty or absent declaration produces a deterministic sentinel-zero contribution (so projects without the configuration get a stable, distinguishable identity contribution); a misdeclared name causes the build to fail before any unit executes (loud failure surfaces typos, which is the entire point of explicit declaration). The reference implementation exposes the mechanism as `cache.tools = ["gcc", "ld", …]` in `.cook/cloud.toml`. The intended audience is environments where toolchain identity is governed at the environment layer — dev containers, `nix-shell`, hermetic Docker images — where atomic environment swaps match global-fold semantics and the auto-detection cannot see binaries hidden behind shell indirections.

**Why now.** The cross-machine cache is the load-bearing piece of Cook Cloud (SHI-24). The 2026-05-01 cache-cloud-readiness work locked the trait surface and the per-step context hash; cross-machine correctness — Bob serving Alice's cache-hit safely — requires that toolchain identity participates in the key. Auto-detection covers the typical case (`gcc -c …`); the declared-toolchain mechanism is the explicit escape hatch for the cases auto-detection cannot reach. Without it, a consumer who upgrades only their `strip` (while leaving `gcc` pinned) silently serves stripped objects produced by the *old* `strip` to other machines.

**Conformance impact.** No grammar change. No Lua API change. No on-disk schema change. `StepEntry.context_hash`'s storage and shape are unchanged; its *value* differs once the eight-byte declared-tools-hash field folds into the hash input — this is a one-shot rebuild for every project on the upgrade boundary regardless of whether the project opts into `cache.tools`. Eviction reaps the orphaned pre-CS-0052 entries on its own schedule. The `[cache] tools` field has `#[serde(default)]`, so pre-amendment `.cook/cloud.toml` files parse identically.

**Test plan.** Seven unit tests in `cook-fingerprint::context::tests` (CS-0052 markers) pin: empty-input sentinel zero, determinism, content sensitivity, membership sensitivity, order independence, missing-tool errors, and step-context-hash fold contribution. Four integration tests in `cli/crates/cook-cache/tests/integration_declared_tools.rs` exercise the full `cloud_key`-construction path: empty-tools sentinel, declared-tool changes the cloud_key, misdeclared tool errors at probe with a diagnostic that names devcontainer/nix-shell, and content-drift on the same declared name yields different cloud_keys. End-to-end coverage in `examples/cache_benchmarks/verify.sh` scenario 18: adding `tools = ["gcc"]` to `.cook/cloud.toml` causes a one-shot full rebuild, the next run fully hits, and a misdeclared tool prints the configured error.

**Reference:** the CS-0052 commit series on `main`.

## CS-0054 — SHA-256 content integrity for cloud-grade cache backends [#changes.cs-0054]

**Date:** 2026-05-04

**Version:** v0.8

**Sections affected:** §{exec.cache.integrity} amended (one normative SHOULD-sentence appended to the "MAY use any deterministic fingerprint" paragraph — implementations exposing the artifact store to writers other than the local build SHOULD use a cryptographically secure fingerprint). Reference-implementation: `cli/crates/cook-fingerprint/src/backend.rs` (`ArtifactMeta.content_hash: [u8; 32]` field with `#[serde(default)]` plus `ArtifactMeta::zero_content_hash()` placeholder helper; `CacheBackend::get` / `CacheBackend::put` trait-doc contract tightened — implementations MUST self-verify content integrity before returning bytes, and MUST stamp `content_hash` with SHA-256 of the bytes before persisting the sidecar), `cli/crates/cook-cache/src/backend.rs` (`LocalBackend::put` computes SHA-256 over the bytes and stamps the sidecar; `LocalBackend::get` recomputes and fails closed on mismatch, missing sidecar, or malformed sidecar; sidecar write becomes atomic via `meta.json.tmp` + rename), `cli/crates/cook-engine/src/executor.rs` (four call-site struct constructors initialise `content_hash` with the zero sentinel — `put` overwrites authoritatively). Design spec in `standard/specs/2026-05-04-cache-cloud-grade-integrity-design.md`.

**Summary.** Cook's pre-CS-0054 cache integrity check used `xxh3_64` (`cli/crates/cook-fingerprint/src/check.rs:362-365`) — fast, deterministic, sound against accidental drift, but non-cryptographic. A 64-bit collision is `~2^32` work under birthday bounds, well within reach for an adversary with write access to a shared artifact bucket. CS-0054 closes the soundness gap by adding a backend-self-verifying SHA-256 layer at the `CacheBackend::get` boundary: `put` computes SHA-256 of the bytes and stamps it into `ArtifactMeta.content_hash`; `get` recomputes the SHA-256, compares to the recorded value, and returns `Ok(None)` on mismatch (treat as miss, fail closed). The bytes returned by a conforming `get` are byte-identical to the bytes most recently `put` under that key — the contract holds equally for `LocalBackend` and any future cloud backend, with no change to the trait signature. The existing `xxh3_64` check in `check.rs` is retained: it guards the engine-to-backend value contract (bytes match the recorded `output.hash`), while CS-0054 guards the backend's internal storage contract (what came out is what was put in).

**Why now.** The single-machine cache survives non-cryptographic integrity because the only writer is the local build — nobody else has byte-level access to the CAS. A multi-tenant SaaS backend (SHI-24) does not have that property: bytes can be tampered with by parties other than the local build, by network corruption, or by a misconfigured replication tier. The existing trait surface was locked in 2026-05-01 specifically so the cloud backend could ship without re-litigating the engine-side seam; the integrity primitive is the last piece of that seam that needed cloud-grade hardening before the cloud backend lands.

**Threat model.** Defended: bit-flip / disk corruption (local), network corruption (shared backend), and bytes-only tampering on a shared backend (an adversary with write access to the byte object but not the meta sidecar). The collision-resistance bound is `~2^128` for SHA-256, vs. the prior `~2^32` for `xxh3_64`. Out of scope (deferred to signed-artifact / SLSA work): an adversary that consistently rewrites both bytes and `meta.content_hash` to a self-consistent pair. CS-0054 is the bytes-only-tampering layer; the bytes-and-meta-consistent-tampering layer is a separate design.

**Conformance impact.** No grammar change. No Lua API change. No `CACHE_VERSION` bump. The new `content_hash` field on `ArtifactMeta` carries `#[serde(default = "ArtifactMeta::zero_content_hash")]`, so pre-CS-0054 sidecars deserialize cleanly; their `content_hash` defaults to `[0; 32]`, which fails the integrity check on first read and surfaces as a miss — a one-shot rebuild for any project that upgrades across the boundary, identical to the orphan-on-upgrade behaviour CS-0052 §4.4 documented. Eviction reaps pre-amendment entries on its own schedule. The `CacheBackend::get` and `CacheBackend::put` signatures are unchanged; the *contract* tightens (implementations MUST self-verify, MUST stamp `content_hash`). This is a documented breaking change for any out-of-tree implementer; the in-tree `LocalBackend` is the only implementer today.

**Test plan.** Five new unit tests in `cook-cache::backend::tests` pin: `put_computes_sha256_in_meta` (round-trip, sidecar inspection, known-answer SHA-256 match, sentinel-overwrite assertion), `get_succeeds_when_bytes_match_meta` (happy round-trip), `get_fails_closed_on_byte_tamper` (mutate the on-disk bytes file post-`put`, sidecar untouched), `get_fails_closed_on_meta_tamper` (rewrite the sidecar's `content_hash` to a different value), `get_fails_closed_on_missing_meta` (`rm` the sidecar — no integrity proof, must fail closed). Existing call sites and integration tests across `cli/crates/cook-engine/src/executor.rs` and `cli/crates/cook-cache/tests/` were updated to pass the zero sentinel; their continued green status is the regression bar. End-to-end coverage in `examples/cache_benchmarks/verify.sh` (31 scenarios) and `examples/cache_dep_drift/verify.sh` (3 scenarios) exercises the full `put`/`get` round-trip through real builds.

**Reference:** the CS-0054 commit on `saas-prep-phase-1-2`.

## CS-0055 — Backend trait idempotency contract for the artifact store [#changes.cs-0055]

**Date:** 2026-05-04

**Version:** v0.8

**Sections affected:** §{exec.cache.integrity} amended (one normative MUST-paragraph appended — the write-side analogue of the read-side fingerprint check; conflicting `put`s MUST error, identical re-puts MUST succeed). Reference-implementation: `cli/crates/cook-fingerprint/src/backend.rs` (`CacheBackend::put` trait-doc tightened with an explicit "Idempotency contract (CS-0055)" paragraph: a `put` to a key that already holds an artifact MUST distinguish identical bytes — `Ok(())` no-op — from conflicting bytes — `BackendError::Other` with a diagnostic naming the key and the differing hashes; a missing or unparseable existing sidecar is treated as no prior artifact and writes through), `cli/crates/cook-cache/src/backend.rs` (`LocalBackend::put` pre-flights the existing sidecar's `content_hash` against `SHA-256(new_bytes)`; idempotent on match, errors on conflict, falls through on missing/malformed/zero-sentinel sidecar). Design spec in `standard/specs/2026-05-04-cache-backend-idempotency-design.md`.

**Summary.** Pre-CS-0055 `LocalBackend::put` overwrote silently: `put(k, bytes_a)` followed by `put(k, bytes_b)` with `bytes_a != bytes_b` succeeded and the next `get(k)` returned `bytes_b`. On a single-developer machine this is harmless — the only writer is the local build, and a determinism bug that produces different bytes for the same `cloud_key` is a rebuild problem the user fixes locally. On a multi-tenant SaaS backend (SHI-24) it is a soundness gap: two clients can race on the same `cloud_key`, and one client running with a poisoned toolchain (a binary not declared in `cache.tools` whose bytes drifted, an `LD_PRELOAD` shim, a compromised build environment) silently replaces the other's legitimate artifact. The CS-0054 read-side check does not help — the conflicting `put`'s sidecar is self-consistent against its own bytes. CS-0055 closes the gap by requiring the write path to detect the conflict using the SHA-256 primitive CS-0054 already established: `put` reads the existing sidecar's `content_hash`, compares against `SHA-256(new_bytes)`, and on mismatch refuses with `BackendError::Other` and a diagnostic naming the key in hex. The write-side invariant the CS-0054 read-side check assumes — "one key, one byte sequence over the artifact's lifetime" — becomes a normative property of the trait rather than an emergent property of the local-only deployment.

**Why now.** CS-0054 is the read-side soundness primitive for the cloud backend. CS-0055 is the write-side analogue. Both must land before the cloud backend ships, because the cloud backend's HTTP-level CAS (e.g., R2's `If-Match: <etag>`) needs a trait contract to honour — and the contract must be locked in before out-of-tree implementers exist. Pairing CS-0055 with CS-0054 keeps the integrity primitives in lockstep with the trait surface that 2026-05-01 locked.

**Threat model.** Defended: cross-tenant key collision with bytes drift (tenant A's legitimate `put(k, bytes_a)` cannot be silently replaced by tenant B's `put(k, bytes_b)`); local determinism-bug detection (a build that produces different bytes than a previous build for the same `cloud_key` surfaces as a `put` error at the moment the bug occurs, not as silent stale-cache symptoms downstream). Out of scope (deferred to signed-artifact / SLSA work): adversary with `delete` access (delete-then-put defeats the conflict check); adversary with sidecar-only write access (rewriting the sidecar's `content_hash` to match the new bytes' hash defeats the conflict check, though the CS-0054 read-side check still catches the inconsistent sidecar-vs-bytes pair on the next `get`).

**Conformance impact.** No grammar change. No Lua API change. No `CACHE_VERSION` bump. `CacheBackend::put`'s signature is unchanged; the *contract* tightens (implementations MUST reject conflicting bytes, MUST treat identical re-puts as idempotent). This is a documented breaking change for any out-of-tree implementer that previously overwrote silently. The in-tree `LocalBackend` is the only implementer today. Pre-CS-0054 sidecars (whose `content_hash` deserializes to the zero sentinel) are treated as "no prior artifact" by the write-through path, so the CS-0054 read-side fail-closed plus the CS-0055 sentinel-write-through give a clean orphan-on-upgrade behaviour.

**Test plan.** Four new unit tests in `cook-cache::backend::tests` pin the contract: `put_idempotent_on_same_bytes` (re-put with identical bytes succeeds and `get` round-trips), `put_rejects_conflict` (re-put with different bytes returns `BackendError::Other` whose message contains "conflict" and the key in hex; prior bytes preserved on disk), `put_recovers_from_missing_meta` (sidecar deleted between puts — second put writes through and restores a healthy entry), `put_recovers_from_corrupt_meta` (sidecar overwritten with non-JSON garbage — second put writes through). Existing tests (including `local_backend_put_idempotent`, which already passed identical bytes twice) continue to pass; the engine's call sites never `put` conflicting bytes at the same key in steady state because `cloud_key` collapses identical inputs to identical keys. End-to-end coverage in `examples/cache_benchmarks/verify.sh` (31 scenarios) and `examples/cache_dep_drift/verify.sh` (3 scenarios) exercises the full `put`/`get` round-trip through real builds; every steady-state `put` is either to a fresh key or an idempotent re-put with identical bytes — both succeed under the new contract.

**Reference:** the CS-0055 commit on `saas-prep-phase-1-2`.

## CS-0056 — Streaming `CacheBackend` trait via `Box<dyn Read>` [#changes.cs-0056]

**Date:** 2026-05-04

**Version:** v0.8

**Sections affected:** §{exec.cache.integrity} amended (one permissive sentence appended to the "MAY use any deterministic fingerprint" paragraph — fingerprint verification MAY be performed as bytes flow through a streaming reader, surfacing as an end-of-stream error; an implementation is not required to materialise the full artifact in memory before verifying). Reference-implementation: `cli/crates/cook-fingerprint/src/backend.rs` (`CacheBackend::get` returns `Option<Box<dyn Read + Send>>` instead of `Option<Vec<u8>>`; `CacheBackend::put` takes `&mut dyn Read` and `&mut ArtifactMeta` instead of `&[u8]` and `&ArtifactMeta`; the `&mut` on meta lets `put` stamp `content_hash` in-place so the caller observes the canonical hash without re-reading the sidecar; trait docs add a new contract — `put` MUST stream into a temporary, hash on the fly, finalize at EOF, and reject with `BackendError::Other("caller-claimed content_hash differs from streamed bytes")` if the caller passed a non-zero hash that disagrees with the streamed bytes), `cli/crates/cook-cache/src/backend.rs` (`VerifyingReader<R>` newtype that tees bytes through a SHA-256 hasher and surfaces mismatch as `io::Error(InvalidData)` at EOF; `get_bytes` / `put_bytes` free-function convenience helpers wrapping the streaming surface for callers that already have bytes in hand; `LocalBackend::get` returns a `VerifyingReader`-wrapped `File`; `LocalBackend::put` streams to a 64 KiB-buffered tmp file, computes SHA-256 on the fly, performs the CS-0055 conflict check after EOF, then commits via tmp-rename), `cli/crates/cook-fingerprint/src/check.rs` (`try_restore` drains the streaming reader via `read_to_end` and treats any read failure as a miss), `cli/crates/cook-engine/src/executor.rs` (four `put` call sites switched to the `put_bytes` helper; `ArtifactMeta` bound as `mut` so `put_bytes` can stamp `content_hash`). Design spec in `standard/specs/2026-05-04-cache-backend-streaming-design.md`.

**Summary.** Pre-CS-0056, the `CacheBackend` trait pinned its I/O shape at `Vec<u8>` / `&[u8]`, forcing every artifact into memory in full on both `get` and `put`. For the typical 10–100 KB object file this is fine; for multi-GB artifacts (linked binaries, debug-info-rich shared objects, container layers in CI workflows) it OOMs the build. The shape also silently encoded "all bytes resident before verification" into the contract — there was no way for an implementer to hash-as-they-stream from disk on the local backend, or hash-as-they-stream from an HTTP body on the future `CloudBackend`, without violating the trait. CS-0056 reshapes the trait to `Box<dyn Read + Send>` / `&mut dyn Read`, preserving the CS-0054 integrity primitive by surfacing verification *as bytes flow* (a `VerifyingReader<R>` wrapper tees bytes through a SHA-256 hasher and raises `io::Error(InvalidData)` at EOF on mismatch) and preserving the CS-0055 conflict-detection contract by streaming the new bytes into a temporary file, computing the hash on the fly, comparing against any existing sidecar's `content_hash`, and only renaming-into-place if the conflict check passes. Two free-function helpers (`get_bytes`, `put_bytes`) wrap the streaming surface with the old `Vec<u8>` / `&[u8]` ergonomics for engine call sites that already have bytes in hand and don't benefit from streaming.

**Why now.** CS-0054 and CS-0055 hardened the trait's *integrity* and *write-side soundness*; CS-0056 hardens its *liveness*. The future `CloudBackend` consumes streaming bodies natively (every modern content-addressable backend — R2, S3, GCS, KV — hands you a streaming reader), and a `Vec<u8>` shape would force the cloud backend to buffer multi-GB HTTP responses in memory before the engine could see a byte. The trait surface was locked in 2026-05-01 specifically so the cloud backend could ship without re-litigating the engine-side seam; the I/O shape is the last piece of that seam that needed cloud-grade hardening before the cloud backend lands.

**Threat model.** No new threat-model category. CS-0056 preserves CS-0054's bytes-only-tampering defence and CS-0055's cross-tenant-collision defence under a streaming I/O shape. The verification still surfaces at the backend boundary; the only delta is the timing — "as bytes flow + finalize at EOF" instead of "buffer then compare". A caller that reads partial bytes from the verifier and acts on them before EOF would see attacker-controlled bytes; the engine's restore path (and the `get_bytes` helper) drains the full reader before installing anything, so this is a discipline a caller MUST honour but is not a new vulnerability — pre-CS-0056, the same caller would have received the bytes in a `Vec<u8>` and acted on them before any verification. The streaming shape *adds* the option to verify-as-you-go; it does not remove the existing buffer-then-verify path (`get_bytes` is exactly that path, layered on top).

**Conformance impact.** No grammar change. No Lua API change. No `CACHE_VERSION` bump. The `.bin` and `.meta.json` on-disk formats are byte-identical to CS-0055; only the in-memory I/O shape changes. The `CacheBackend::get` and `CacheBackend::put` signatures CHANGED — out-of-tree implementers must rewrite, and the in-memory `ArtifactMeta` binding shifts from `let meta = …` to `let mut meta = …` at every call site. The in-tree `LocalBackend` is the only implementer today; the future `CloudBackend` consumes the new shape natively. The §{exec.cache.integrity} amendment is permissive (no new MUST) — the trait shape is implementation-internal and not language-surface; the new sentence makes explicit that incremental / streaming verification is conformant.

**Test plan.** Ten new unit tests in `cook-cache::backend::tests` pin the streaming contract: `verifying_reader_passes_through_on_match`, `verifying_reader_errors_on_mismatch`, `verifying_reader_passes_through_on_empty_match`, `local_backend_get_returns_streaming_reader`, `local_backend_get_streaming_errors_on_byte_tamper`, `local_backend_put_streams_with_zero_sentinel`, `local_backend_put_rejects_caller_hash_mismatch`, `local_backend_put_accepts_caller_hash_match`, `put_bytes_helper_round_trip`, `get_bytes_helper_round_trip`. Existing CS-0054 / CS-0055 tests preserved with semantic coverage intact, just rewritten against the streaming surface (most via the `put_bytes` / `get_bytes` helpers): `put_computes_sha256_in_meta`, `get_succeeds_when_bytes_match_meta`, `get_fails_closed_on_byte_tamper`, `get_fails_closed_on_meta_tamper`, `get_fails_closed_on_missing_meta`, `put_idempotent_on_same_bytes`, `put_rejects_conflict`, `put_recovers_from_missing_meta`, `put_recovers_from_corrupt_meta`. Five integration tests rewritten in lockstep: `integration_first_build_depfile.rs`, `integration_config_toggle.rs`, `integration_multi_output_restore.rs`, `integration_discovered_inputs_restore.rs`, `integration_restore_on_hit.rs`. End-to-end coverage in `examples/cache_benchmarks/verify.sh` (31 scenarios) and `examples/cache_dep_drift/verify.sh` (3 scenarios) exercises the full `put`/`get` round-trip through real builds.

**Reference:** the CS-0056 commit on `saas-prep-phase-1-2`.

## CS-0057 — `BackendConfig` — timeout, retry, max-artifact knobs [#changes.cs-0057]

**Date:** 2026-05-04

**Version:** v0.8

**Sections affected:** None normative. The Standard already permits implementation-defined cache tunables under §{exec.cache} ("the hash function, on-disk layout, eviction policy, and storage medium are implementation-defined"); CS-0057 is a reference-implementation amendment that exposes user-overrideable backend knobs without amending normative text. Reference-implementation: `cli/crates/cook-fingerprint/src/backend.rs` (new `BackendConfig` struct alongside the trait — `timeout: Duration`, `max_retries: u32`, `backoff_initial: Duration`, `backoff_max: Duration`, `max_artifact_bytes: u64`; `Default` impl pins the spec defaults at 30s / 3 / 100ms / 5s / 1 GiB), `cli/crates/cook-cache/src/backend.rs` (`LocalBackend` gains a `config: BackendConfig` field and a `with_config(root, config)` constructor; the existing `new(root)` delegates to `with_config(root, BackendConfig::default())` for back-compat; `LocalBackend::put` enforces `max_artifact_bytes` mid-stream — the existing 64 KiB-buffered streaming loop accumulates total bytes via `saturating_add`, and on overflow discards the temp file and returns `BackendError::Other("artifact exceeds max_artifact_bytes ({total}); cap {limit}")`; the `timeout` / `max_retries` / `backoff_initial` / `backoff_max` fields are no-ops on disk I/O but ride along for the future `CloudBackend` constructor; new public `config()` accessor for diagnostic / test use), `cli/crates/cook-cache/src/cloud_config.rs` (`CloudSection` gains five optional `Option<u64>` / `Option<u32>` fields — `timeout_secs`, `max_retries`, `backoff_initial_ms`, `backoff_max_ms`, `max_artifact_mib`; new `CloudConfig::backend_config()` method overlays the present values onto `BackendConfig::default()`), `cli/crates/cook-engine/src/run.rs` (cache-bootstrap site switched from `LocalBackend::new(cache_dir)` to `LocalBackend::with_config(cache_dir, cloud_config.backend_config())` so the per-build `BackendConfig` is active for the build's lifetime). Design spec in `standard/specs/2026-05-04-cache-backend-config-design.md`.

**Summary.** CS-0054 / CS-0055 / CS-0056 hardened the `CacheBackend` trait's *integrity*, *write-side soundness*, and *liveness*; the surface that remained soft was *operational* — the four tunables every production cloud-cache client needs (per-call timeout, retry-with-exponential-backoff, backoff cap, max-artifact-size cap). CS-0057 lays the wiring without touching the trait shape: a new `BackendConfig` struct with sensible defaults lives alongside the trait in `cook-fingerprint::backend`; `LocalBackend::with_config(root, BackendConfig)` is the new constructor, with `LocalBackend::new(root)` preserved as a thin wrapper that delegates to `with_config(_, default())`; the engine's cache bootstrap reads `[cloud]` overrides from `.cook/cloud.toml` (five new optional keys: `timeout_secs`, `max_retries`, `backoff_initial_ms`, `backoff_max_ms`, `max_artifact_mib`) and threads the resulting `BackendConfig` into the backend constructor. `LocalBackend` honours `max_artifact_bytes` today (a streamed-byte counter inside the 64 KiB-buffered `put` loop aborts oversize puts with `BackendError::Other` naming both the streamed total and the configured cap); the HTTP-shaped knobs (`timeout`, `max_retries`, `backoff_initial`, `backoff_max`) are no-ops on disk I/O but documented and ready for the upcoming `CloudBackend` whose constructor will accept the same `BackendConfig` shape with no plumbing churn.

**Why now.** `CloudBackend` does not exist yet — it lands in the next ticket. CS-0057's job is to land the constructor-side seam *before* the cloud backend ships, so the cloud backend can be reviewed in isolation against a stable `BackendConfig` contract rather than introducing both the network code path and the configuration surface in a single PR. Pinning `max_artifact_bytes` enforcement at the streaming-put boundary today (on `LocalBackend`) also gets the test coverage in place against a deterministic, easily-pinned implementation; the future `CloudBackend` re-uses the same enforcement logic on its tmp-buffer-or-stream path.

**Threat model.** No new threat-model category. The `max_artifact_bytes` cap defends against *resource exhaustion* — a runaway producer (a build that mistakenly emits a multi-GB log file as an output, or a poisoned step that produces an enormous payload) can no longer fill local disk or burn cloud-storage budget unbounded. Defended: oversize puts (refused at streaming, with diagnostic naming the limit and total). Out of scope: server-side enforcement (the cloud backend will independently enforce server-side quotas; CS-0057 is the client-side fail-fast that saves the round-trip on oversized payloads), per-recipe overrides (a future `[cache.recipes."foo"]` table could lift the cap for specific recipes), and adaptive timeouts (a separate "auto-tune timeout based on observed latency" refinement).

**Conformance impact.** No grammar change. No Lua API change. No `CACHE_VERSION` bump. The on-disk byte and sidecar formats are byte-identical to CS-0056. The `CacheBackend` trait surface is unchanged — `BackendConfig` is constructor-side; nothing in the trait observes it. `LocalBackend::new(root)` is preserved as a thin wrapper over `with_config(root, default())`, so existing call sites (engine bootstrap, all unit tests, all integration fixtures) continue to compile and behave identically. Pre-CS-0057 `.cook/cloud.toml` files parse identically — the five new keys are `#[serde(default)]` and produce a `BackendConfig::default()` when absent.

**Test plan.** Four new unit tests in `cook-cache::backend::tests` pin the constructor and oversize-rejection contract: `backend_config_default_values` (sanity-check that `BackendConfig::default()` produces 30s / 3 / 100ms / 5s / 1 GiB), `local_backend_with_config_honored` (construct with a custom config, observe each field via the `config()` accessor), `local_backend_put_rejects_oversize_artifact` (set `max_artifact_bytes = 100`, put 200 bytes; assert `Err`, message contains "exceeds" and "100", `BackendError::Other` variant, no artifact persisted at the key), `local_backend_put_accepts_artifact_at_limit` (put exactly `max_artifact_bytes` bytes; assert success and round-trip equality). Two new unit tests in `cook-cache::cloud_config::tests` pin the TOML overlay: `backend_config_uses_defaults_when_unset` (empty `[cloud]` produces `BackendConfig::default()` field-for-field), `backend_config_overrides_from_toml` (TOML with all five fields set produces the right `Duration` / `u64` values, including the `_secs` / `_ms` / `_mib` unit conversions). End-to-end coverage in `examples/cache_benchmarks/verify.sh` (31 scenarios) and `examples/cache_dep_drift/verify.sh` (3 scenarios) exercises the `LocalBackend::new` → `LocalBackend::with_config(_, default)` rewrite path; their continued green status is the regression bar.

**Reference:** the CS-0057 commit on `saas-prep-phase-1-2`.

## CS-0058 — `CloudBackend` skeleton — sync HTTP client over a v1 wire protocol [#changes.cs-0058]

**Date:** 2026-05-04

**Version:** v0.8

**Sections affected:** None normative. The wire protocol is implementation-defined per §{exec.cache} ("the on-disk layout, eviction policy, and storage medium are implementation-defined" — the network protocol is the cloud backend's "storage medium"). Reference-implementation: new `cli/crates/cook-cache/src/cloud_backend.rs` (`pub struct CloudBackend` implementing `CacheBackend` over HTTP/1.1 against `/v1/artifacts/...` — `batch_query` POSTs JSON, `get` returns a `VerifyingReader`-wrapped response body keyed on `X-Cook-Content-Hash`, `put` streams the caller's reader through a `CappedReader` that enforces `max_artifact_bytes` mid-stream, `delete` folds 404 into idempotent success, `health` GETs `/v1/health`; internal helpers for retry-with-jittered-exponential-backoff and HTTP-status-to-`BackendError` mapping); `cli/crates/cook-cache/src/cloud_config.rs` (new optional `api_key: Option<String>` field on `CloudSection`; new `CloudConfigError::MissingApiKey` and `MissingEndpoint` error variants; new `resolved_api_key()` accessor that prefers `COOK_CLOUD_API_KEY` env var over the TOML field, with empty env var treated as unset; `load_or_default` validation extended to require both `endpoint` and a resolved API key when `cloud.enabled = true`); `cli/crates/cook-cache/src/lib.rs` (one new `pub mod cloud_backend;` plus a `pub use` re-export of `CloudBackend`); `cli/crates/cook-cache/Cargo.toml` (one new runtime dependency — `ureq = { version = "2", features = ["tls", "json"] }` for sync HTTP + rustls TLS + `send_json` / `into_json` ergonomics; one new dev-dependency — `mockito = "1"` for HTTP mocking in unit tests); `cli/crates/cook-engine/src/run.rs` (cache-bootstrap site branches on `cloud_config.cloud.enabled` — when enabled, constructs `CloudBackend::new(endpoint, resolved_api_key, backend_config)` from the validated config; otherwise falls through to the existing `LocalBackend::with_config` path; the existing health-check pattern at the bottom of the bootstrap applies in either branch). Design spec in `standard/specs/2026-05-04-cache-cloud-backend-skeleton-design.md`.

**Summary.** CS-0054 / CS-0055 / CS-0056 / CS-0057 hardened the `CacheBackend` trait — its integrity, write-side soundness, liveness, and operational tunables. The trait was ready to be implemented over a network. CS-0058 is that implementation: the *client side* of the wire protocol that the future Cook Cloud SaaS artifact server will speak. The protocol is versioned under `/v1/`: `POST /v1/artifacts/batch_query` (JSON body `{keys: [hex...]}`, response `{present: [hex...]}`), `GET /v1/artifacts/{key_hex}` (bytes streamed in body, meta carried in `X-Cook-*` headers — `X-Cook-Content-Hash` is load-bearing for the `VerifyingReader` wrap, the rest are meta), `PUT /v1/artifacts/{key_hex}` (bytes streamed from caller's reader, same `X-Cook-*` headers, `X-Cook-Content-Hash` MAY be the zero sentinel and the server SHOULD stamp the computed hash on receipt), `DELETE /v1/artifacts/{key_hex}` (204 success, 404 idempotent success), `GET /v1/health` (200 healthy, anything else `BackendError::Transient` so the existing engine bootstrap pattern logs "backend unavailable" and disables for the build). Auth is bearer-token via `Authorization: Bearer <api_key>` on every request; the API key is resolved from `COOK_CLOUD_API_KEY` env var (priority 1) or `[cloud] api_key` in `.cook/cloud.toml` (priority 2); a missing key when `cloud.enabled = true` is a build-start error (`MissingApiKey`), identical in shape to the existing `MissingProject`. The status-code map is: `200/201/204` → success, `400` → `BackendError::Other("bad request: ...")`, `401/403` → `BackendError::Unauthorized`, `404` → caller-handled (get → `Ok(None)`, delete → `Ok(())`), `409` → `BackendError::Other("conflict at <key>: ...")` (CS-0055 over the wire), `413` → `BackendError::Other("server rejected oversize artifact: ...")` (CS-0057 server-side), `429` → `BackendError::QuotaExceeded`, `5xx` → `BackendError::Transient` (retried). The retry shell honours CS-0057's `BackendConfig`: only `Transient` retries, up to `max_retries`, jittered exponential backoff (`backoff_initial` doubling each step, capped at `backoff_max`, ±25% uniform jitter applied to each sleep). The PUT path enforces `max_artifact_bytes` mid-stream via a `CappedReader` wrapper around the caller's reader; on overflow the in-flight HTTP body write is aborted and the client returns `BackendError::Other("artifact exceeds max_artifact_bytes ({total}); cap {limit}")` — bytes never reach the wire on overflow, saving the round-trip on oversize payloads. The GET path wraps the response body in the same `VerifyingReader` `LocalBackend::get` uses on disk — the bytes either stream through to the caller's `read_to_end` byte-identical to what the server sent, or the EOF read raises `io::Error(InvalidData)`, identical fail-closed semantics on either backend.

**Why now.** CS-0054–CS-0057 locked the trait shape. CS-0058 is the first non-`LocalBackend` implementer; the wire protocol it pins is what the Cook Cloud server (a separate ticket) will be built against. Landing the client first lets the protocol shape be reviewed in isolation against a deterministic mock (mockito) before the server-side design absorbs both the protocol and the storage layer. The shape is also load-bearing for any future out-of-tree backend (an enterprise self-hosted artifact server, a vendor-specific S3-compatible facade); having the protocol pinned in a referenced spec makes that surface stable.

**Threat model.** No new threat-model category. The same CS-0054 byte-only-tampering defence applies (`X-Cook-Content-Hash` carries the SHA-256 from sender to receiver; the client's `VerifyingReader` finalizes at EOF and refuses bytes that don't match the header), the same CS-0055 cross-tenant-collision defence applies (server returns `409 Conflict` on differing-bytes-at-same-key, mapped to `BackendError::Other("conflict at <key>: ...")`), the same CS-0057 resource-exhaustion defence applies (client-side `max_artifact_bytes` mid-stream cap; server-side `413` enforcement in addition). The `Authorization: Bearer <api_key>` header is NEVER logged and NEVER surfaced in `BackendError` diagnostics — the helper that maps HTTP errors to `BackendError` includes the response body's diagnostic (server-supplied) but never the request's auth header value. The `tracing::debug!` lines at the request boundary likewise redact the API key. A `[cloud] api_key = "..."` form checked into a shared / CI repository is documented as not recommended (env-var form is preferred) — a build-time tracing warning when the TOML form is detected is a follow-up enhancement. Out of scope: server-side authentication beyond bearer-token, OAuth, mTLS, signed-URL auth, multipart / chunked-resumable uploads, compression of bodies, HTTP/2, server-stamped-hash response header (the cloud client cannot stamp `meta.content_hash` in-place without buffering or hashing client-side; this is a documented asymmetry with `LocalBackend` — a future server-side enhancement could return `X-Cook-Server-Stamped-Content-Hash` on `200 OK`).

**Conformance impact.** No grammar change. No Lua API change. No `CACHE_VERSION` bump. The `CacheBackend` trait surface is unchanged — `CloudBackend` is a new in-tree implementer of an existing trait. The `.cook/cloud.toml` schema gains one new optional field — `api_key` under `[cloud]` — with `#[serde(default)]`. Pre-CS-0058 `.cook/cloud.toml` files parse identically; validation tightens *only* when `cloud.enabled = true` (the disabled path — the default — sees no behavioural change). When enabled, the new validation requires `endpoint` and a resolved API key in addition to the existing `project` requirement; this is a tightening, but the disabled path is unaffected and the enabled path was non-functional pre-CS-0058 (the cloud backend it pre-emptively hinted at didn't exist). The `LocalBackend::with_config` call path in `run.rs` is byte-for-byte preserved on the `cloud.enabled = false` branch.

**Server side is future work.** This change ships only the *client* of the v1 wire protocol. The server side — the Cook Cloud artifact server, its R2 / D1 / KV storage layer, its multi-tenant auth, its server-side `max_artifact_bytes` and rate-limit policy — is tracked in a separate ticket and will land independently. The wire protocol is pinned in CS-0058's design spec as the contract both sides honour.

**Test plan.** Fourteen new unit tests in `cook-cache::cloud_backend::tests` against `mockito`-based mock servers pin the protocol-and-error-mapping contract: `cloud_backend_get_round_trips` (200 with `X-Cook-Content-Hash`; bytes round-trip through the verifier), `cloud_backend_get_returns_none_on_404`, `cloud_backend_get_fails_closed_on_byte_tamper` (200 with bytes whose SHA-256 disagrees with the header; `read_to_end` raises `InvalidData`), `cloud_backend_put_round_trips` (200; mockito body matcher confirms the bytes), `cloud_backend_put_rejects_oversize` (`max_artifact_bytes = 100`, 500 bytes through PUT; client surfaces `BackendError::Other` with "exceeds" and "100" in the diagnostic; mockito `expect_at_most(1)` because the abort timing is implementation detail), `cloud_backend_put_handles_409_conflict`, `cloud_backend_batch_query_round_trips`, `cloud_backend_retries_on_5xx` (first call 503, second call 404; both mocks `expect(1)`; backend issues both calls), `cloud_backend_does_not_retry_on_401` (mock `expect(1)`; auth failure surfaces immediately), `cloud_backend_does_not_retry_on_429` (same shape, `BackendError::QuotaExceeded`), `cloud_backend_health_returns_ok_on_200`, `cloud_backend_health_returns_transient_on_5xx` (4 calls — 1 initial + 3 retries — all 503; then `Transient`), `cloud_backend_delete_204_succeeds`, `cloud_backend_delete_404_idempotent`, `cloud_backend_get_errors_on_missing_content_hash_header` (200 OK without `X-Cook-Content-Hash` is server misbehaviour, surfaces as `BackendError::Other` mentioning the missing header), `jitter_factor_in_range` (50 samples all in [0.75, 1.25]). **Date:** 2026-05-06

**Version:** v0.8

**Sections affected:** None normative. The `cook --dag` viewer is a tooling affordance of the reference implementation, not a Cookfile-language surface (CS-0047 already pinned this — `cmd_dag` is implementation-optional and the Standard does not require a graphical viewer at all). Reference-implementation: `cli/crates/cook-dag-viewer/` (the `tiny_http` HTTP server, the `open`-based browser-launch, the embedded `dag_viewer.html` template, and the `serve_dag` entry point that CS-0047 split out are removed; in their place a `ratatui`-based three-pane TUI ships behind the same `viewer` cargo feature, driven by a new internal `ViewFrame` trait whose v1 `SnapshotFrame` impl wraps the existing register-only walk; `build_wave_dag_data` and the `WaveDagData` schema CS-0048 pinned at `schema_version = 1` are preserved bytes-for-bytes — the TUI consumes the same snapshot the HTML viewer consumed, only the rendering surface changes); `cli/crates/cook-cli/Cargo.toml` (the `viewer` feature's transitive deps swap from `tiny_http` + `open` to `ratatui` + `crossterm`); `cli/crates/cook-cli/src/{main.rs,pipeline.rs}` (the feature-gated `cmd_dag` branch wires into the TUI entry point instead of `serve_dag`; the `--theme` option and the non-TTY fallback are added at the same dispatch site); `cli/crates/cook-dag-viewer/src/{tui/*.rs,view_frame.rs,snapshot_frame.rs}` (new TUI scaffold — index pane, graph pane, detail pane, fuzzy-search overlay, help overlay, theme module). Design spec in `standard/specs/2026-05-05-dag-tui-viewer-design.md`.

**Replaces.** The browser-based DAG viewer originally introduced in [CS-0047](#changes-cs-0047) and versioned by [CS-0048](#changes-cs-0048). The historical record stands; CS-0060 is the change that swaps the rendering surface. The CS-0048 wire-format pin (`WaveDagData.schema_version = 1`) is preserved — the TUI is a new consumer of the same schema, not a new schema.

**Summary.** `cook --dag` now opens a terminal UI in the user's current shell instead of spawning a local HTTP server and launching a browser tab. The data shape (`WaveDagData`) and the `viewer` cargo feature are unchanged; only the presentation layer is replaced. The TUI uses a three-pane layout — the index pane (left) lists every node in the wave-grouped DAG, the graph pane (center) renders the wave layout with the current node highlighted, and the detail pane (right) shows the selected node's metadata (recipe, unit, declared inputs/outputs, declared dependencies). A fuzzy-search overlay (`/`) jumps to a node by name, and a help overlay (`?`) lists the keymap. A non-TTY caller (`cook --dag | less`, CI redirected output) gets a printed wave/recipe/unit tree fallback rather than the interactive UI.

**Rationale.** The browser viewer required port allocation, a window switch into the user's default browser, and a running HTTP server tied to the lifetime of the `cook` process. A TUI delivers the same DAG inspection experience inside the terminal that the build was launched from, eliminates the port-and-browser dependency, and composes naturally with terminal multiplexers (tmux, zellij) and remote shells (SSH). The supply-chain argument from CS-0047 — that an HTTP server in the default-on path is a poor fit for a build-orchestration tool — applies equally to a viewer-feature-enabled binary; the TUI keeps the viewer feature opt-in (CS-0047's gate) while removing the network-listening dependency from the feature-enabled path entirely. The ground laid by CS-0048 (additive `WaveDagData` schema with a load-bearing `schema_version` field) is fully preserved — the TUI consumes the same `WaveDagData` snapshot the HTML viewer consumed, so the data-shaping helper (`build_wave_dag_data` from CS-0027) and the schema pin (CS-0048) remain authoritative.

**User-visible changes.** `cook --dag` opens a TUI (three-pane layout: index / graph / detail). `cook --theme {auto,mono}` picks the TUI color theme — `auto` honours the terminal's color capability and respects `NO_COLOR`, `mono` forces a monochrome palette suitable for terminals where ANSI color escapes interfere with multiplexer rendering or the user prefers a flat presentation. The browser-based viewer and its embedded HTML template are removed; the `tiny_http` and `open` runtime deps the `viewer` feature pulled in are gone. The `--dag` keymap: `j` / `k` navigate the index, `[` / `]` jump along incoming / outgoing edges from the selected node, `H` / `J` / `K` / `L` pan the graph-pane camera, `/` opens fuzzy search, `?` opens the help overlay, `q` quits. The full keymap is rendered by `?` and is also documented in the new design spec.

**Architecture note.** The `cook-dag-viewer` crate now exposes an internal `ViewFrame` trait that abstracts "what the TUI is looking at right now". v1 ships a `SnapshotFrame` impl backed by the existing register-only walk — it is computed once at `cook --dag` entry and is immutable for the lifetime of the TUI session. A future release MAY add a `LiveFrame` impl that mutates in response to executor events from the `cook-progress` `events.jsonl` stream (App. E.11), enabling a "live build mode" where node states light up during a concurrent `cook build`. The trait is internal to the viewer crate and not part of any published Rust-API surface; no external consumers depend on it yet. The `ViewFrame` shape is therefore free to evolve in a follow-up change without an App. D entry until it is ever exposed across a crate boundary.

**Conformance impact.** None. The Standard does not pin `cmd_dag` behaviour (CS-0047 explicitly carves the viewer out as implementation-optional), the conformance harness does not exercise it, and no Cookfile authoring surface is touched. The CS-0048 `WaveDagData.schema_version = 1` pin is preserved — the TUI is a new in-tree consumer of the same wire format, not a schema change. The `viewer` cargo feature gate from CS-0047 is preserved — vanilla `cargo install cook-cli` continues to produce a slim binary with no viewer dependencies, and the diagnostic naming the rebuild flag (`cargo build --features viewer`) is preserved word-for-word for viewer-less builds invoked with `cook --dag`.

**Migration.** This is a breaking user-visible change for anyone scripting around the old browser behaviour — for example, a wrapper that read the `127.0.0.1` port number from `cook --dag`'s stderr and curl'd the viewer JSON, or a CI step that asserted on the browser-launch side effect. Such callers SHOULD migrate to consuming the `WaveDagData` JSON directly via the existing in-tree codegen path, or to the non-TTY fallback (`cook --dag | less`, which prints the wave/recipe/unit tree and exits). There is no deprecation period for the HTML viewer — it is removed in this release; the rationale for skipping a deprecation cycle is that the viewer feature was opt-in (CS-0047), the affected user population is therefore narrow, and the new TUI is feature-equivalent for the inspection use cases the HTML viewer served.

## CS-0062 — Search-path order within `cook_modules/` [#changes.cs-0062]

**Summary.** §7 grows a new normative clause defining the search-path order an implementation MUST configure for `package.path` and `package.cpath` per execute-phase work unit. ## CS-0063 — `cook.add_unit` rejects directory inputs and outputs at register time [#changes.cs-0063]

**Summary.** §6.2 grows a new normative **Path-shape restriction** clause: every entry in `inputs`, `output`, and `outputs` is a *file* path, and a conforming implementation MUST raise a Lua runtime error at register time when any path in `inputs` resolves (relative to the recipe's working directory) to an existing directory, and when any path in `output` / `outputs` resolves to an existing directory. Non-existent output paths MUST still be accepted (declared outputs are typically not yet built at register time). The diagnostic MUST contain the substring `is a directory` and the offending path.

**Rationale.** Cook's cache hashes file content, not directory trees. Silently accepting a directory input produced an empty cache record (only `_source_hash`) and the unit re-ran on every invocation — a silent correctness bug surfaced in Book ch9 §9.7's `copy.tree` example. Failing fast at register time gives the author an actionable diagnostic ("use a glob like 'dir/*' or list specific files") instead of mysterious cache misses on the second run. Listing the input files individually keeps the cache key tied to file content, so a successful run is genuinely reusable.

**Conformance impact.** A conforming implementation MUST stat each path the Cookfile (or a module called from it) passes in `inputs` / `output` / `outputs` to `cook.add_unit`, and MUST reject the call with a runtime error containing `is a directory` when the stat resolves to a directory. Symlinks to directories MUST also be rejected. New negative fixture `052-directory-input-rejected/` (with `register_error.txt`) pins the rule; a companion register-phase conformance harness in `cook-register/tests/conformance.rs` consumes fixtures with `register_error.txt` (mirroring the codegen-phase harness's treatment of `codegen_error.txt`).

**Reference:** this commit.

## CS-0064 — `fs.glob` drops directory matches [#changes.cs-0064]

**Summary.** §6.5.6 grows a normative clause: `fs.glob` MUST drop matches whose final (symlink-resolved) type is a directory. Pre-CS-0064 the result included every path the underlying glob library matched, files and directories alike. Post-CS-0064 the result is files only; sub-directories that happen to match the pattern are silently filtered. A pattern that matches *only* directories MUST return an empty sequence (it is not an error).

**Rationale.** CS-0063 made directory paths a register-time error in `cook.add_unit` inputs/outputs. The natural composition `cook.add_unit({ inputs = fs.glob("dir/*"), ... })` would then fail whenever `dir/` contained any sub-directory — for a path the author never wrote by hand. Filtering at the glob layer makes the common "all files matching a pattern" idiom well-defined without weakening CS-0063: directory entries are silently dropped at enumeration time, and any directory path written by hand still trips CS-0063's diagnostic. Symlink-to-directory follows `std::fs::metadata` semantics (the link is followed) so a symlink whose terminal type is a directory is also dropped, matching CS-0063's symlink rule. The same filter is applied to the register-side helpers backing `recipe.ingredients` and `cook.resolve_ingredients` so the rule holds uniformly across every glob expansion the language exposes.

**Conformance impact.** A conforming `fs.glob` implementation MUST stat each candidate match and exclude entries whose terminal type is a directory. A broken symlink (or any other stat error) is treated as "not a directory" — the entry is kept, and any downstream consumer that requires the path to actually exist will diagnose the missing file with its own, more specific message. Implementations MAY perform the filtering at any point during enumeration; the observable behavior is what conformance pins. No new fixture is required: directory filtering is an execute-phase observation and the conformance harness does not yet execute Lua bodies, but unit tests in `cli/crates/cook-lua-stdlib/src/fs_api.rs` (`static_glob_filters_out_directories`, `static_glob_filters_symlink_to_directory`, `static_glob_keeps_symlink_to_file`) pin the three cases, and a register-side test in `cli/crates/cook-register/src/context.rs` (`resolve_glob_filters_directories`) pins the parallel filter on `recipe.ingredients` / `cook.resolve_ingredients`.

**Reference:** this commit.

## CS-0065 — Register-phase `cook.load_module` MUST honour the `cook_modules/` resolution order specified in §7 [#changes.cs-0065]

**Summary.** §7.4 grows a normative clause extending the four-path `cook_modules/` search order (originally CS-0062, execute-phase only) to register-phase `cook.load_module(name)` calls. **Reference:** this commit.

## CS-0066 — Register-phase module loading MUST configure `package.path`/`package.cpath` and MUST permit C extensions [#changes.cs-0066]

**Summary.** §7.4 gains a normative paragraph requiring a conforming implementation to configure `package.path` and `package.cpath` on the register-phase Lua VM before executing any module body loaded via `cook.load_module`. The search order mirrors the execute-phase order established by CS-0062. Additionally, a conforming implementation MUST NOT restrict native C extension loading in the register-phase VM; C extensions (`.so`/`.dll`) installed under `cook_modules/` MUST be loadable at register time.

**Rationale.** Multi-file rocks (such as `cook_cc`) use Lua's native `require()` internally to load their sub-modules (e.g., `require("cook_cc.toolchain")` inside `cook_cc/init.lua`). Without `package.path` and `package.cpath` configured for the register-phase VM, these internal requires fail because they search only the system Lua paths, not the project's `cook_modules/`. This class of failure was latent before real multi-file rocks existed; the migration of `examples/lua-build` from a vendored single-file `cpp.lua` to the multi-file `cook_cc` rock triggered it. The companion C-extension rule closes the parallel gap: `cook_cc/finder.lua` depends on `lpeg` (a C extension), so a register-phase VM that blocks native modules cannot load `cook_cc` at all. The register phase already runs fully user-trusted Lua (Cookfile bodies, module code); permitting C extensions introduces no new trust boundary.

**Conformance impact.** A conforming implementation MUST configure `package.path` and `package.cpath` (same order as CS-0062) before executing each module body in the register phase. The reference implementation stashes the original `package.path`/`package.cpath` on first load per VM using the `_cook_original_path`/`_cook_original_cpath` sentinel approach, mirroring the execute-phase idiom. A conforming implementation MUST create the register-phase Lua VM with all standard libraries enabled (equivalent to `mlua::Lua::unsafe_new()`) so that native module loading (`package.loadlib`) is available. The project-root `fs.*` sandbox (CS-0045) MUST still be installed on the register-phase VM immediately after creation; the expanded VM capabilities do not relax the filesystem confinement rules.

## CS-0068 — cc cmake-compat strategy (M2) [#cs-0068]

**Summary.** §9.2.3.8 grows from v0.2's four-stage `cc.find` chain (project → curated → pkg-config → bare-probe) to v0.3's five-stage chain inserting `cmake-compat` between pkg-config and bare-probe. `FindOpts` gains a `cmake` boolean that lifts cmake-compat to position 3 (replacing pkg-config's slot for that call). §9.2.3.8.1 normatively specifies the cmake-compat strategy steps (driver detection, EXIST/COMPILE/LINK probes, imported-target rejection). §9.2.3.8.2 and §9.2.3.8.3 specify the COMPILE-mode and LINK-mode output grammars. §9.2.5 grows three catalogue rows naming the cmake-compat-specific failure modes. `Attempt.strategy` enumeration grows `"cmake-compat"`. App. B gains three rationale sections (legacy-mode trust boundary, chain-order trade-off, v0.2 preservation).

**Rationale.** A long tail of modern C/C++ libraries ship `XxxConfig.cmake` but no `.pc` (SDL3, glfw3, Vulkan SDK, fmt, nlohmann_json, most vendor SDKs). M1's curated short list and pkg-config-only chain leave Cook unable to discover them. Hand-writing curated finders per library does not scale. cmake-compat reuses cmake's own Config-resolution implementation (via `cmake --find-package` legacy mode) as a discovery shim — cmake is never invoked to build anything; Cook's own compiler invocations drive the build. The trust boundary is explicit: cmake-compat's accuracy is delegated to cmake; the only Cook-implemented part is output parsing.

**Conformance impact.** A conforming v0.3 implementation MUST treat `opts.cmake = false` (default) calls identically to v0.2 calls for stages 1–3 (project → curated → pkg-config); the chain extension is additive. New behaviour: stages 4 (cmake-compat) and 5 (bare-probe) are consulted after pkg-config misses. `opts.cmake = true` skips pkg-config entirely.

**Reference:** the CS-0068 commits on `shi-135-m2-cmake-compat`.

## CS-0069 — Execute-phase `cook.load_module` MUST honour the §7 resolution order [#cs-0069]

**Summary.** §7 grows a clarifying paragraph: a conforming implementation MAY expose `cook.load_module(name)` on the execute-phase Lua VM as an extension over the register-phase requirement, and when exposed it MUST honour the same four-path resolution order specified by CS-0065 (`cook_modules/<name>.lua`, `cook_modules/<name>/init.lua`, `cook_modules/share/lua/5.4/<name>.lua`, `cook_modules/share/lua/5.4/<name>/init.lua`).

**Rationale.** The reference implementation has exposed an execute-phase `cook.load_module(name)` since M0 to support recipe-body Lua scripts that load modules ad-hoc. Pre-CS-0069 the execute-phase implementation only searched two paths (`cook_modules/<name>.lua`, `cook_modules/<name>/init.lua`), causing recipe-body load_module calls to fail for any LuaRocks-installed module — most notably breaking the M1 raylib-game integration example at runtime (the Cookfile's `use cook_cc` translates to an execute-phase `cook.load_module("cook_cc")` whose only resolvable copy lives under `cook_modules/share/lua/5.4/cook_cc/init.lua`). CS-0069 closes this register-vs-execute parity gap.

**Conformance impact.** Implementations MUST search the four paths in §7's order and the diagnostic on miss MUST list all four attempted paths. Implementations that previously searched only the two top-level paths MUST extend their search; the change is additive and cannot break any Cookfile that worked under the pre-CS-0069 behavior.

**Reference:** the CS-0069 commits on `main` after SHI-135 M2.

## CS-0070 — Execute-phase Lua VM MUST expose `cook.cache` [#cs-0070]

**Summary.** §6.3.4 grows a normative clause requiring `cook.cache.get`, `cook.cache.set`, and `cook.cache.scope` to be available on BOTH the register-phase and execute-phase Lua VMs. The execute-phase implementation MAY use in-memory-only storage scoped to the worker's lifetime; the register-phase implementation MUST persist values across invocations per the cache contract.

**Rationale.** Standard §6.3.4 lists `cook.cache` among "both-phase APIs" but the reference implementation only registered it on the register-phase VM. The omission was silent until CS-0069 fixed execute-phase `cook.load_module` resolution — raylib-game's `use cook_cc` now correctly resolves and runs `cook_cc.init()`, which calls `cook.cache.get("compiler")` to rehydrate the toolchain probe from the register-phase cache. The execute-phase VM had no `cook.cache` table at all, raising `attempt to index a nil value (field 'cache')`. CS-0070 closes the both-phase parity gap explicitly: implementations MUST expose the cache API on both VMs.

**Conformance impact.** Implementations MUST expose `cook.cache.{get,set,scope}` on both VMs. The execute-phase storage MAY be ephemeral (in-memory, per worker); persistence semantics for the execute-phase API are explicitly NOT specified here and remain available for refinement (see e.g., the disk-persistence design proposal under [SHI-214](https://linear.app/shiny-guru/issue/SHI-214) for cross-run persistence). The minimum bar is that calls do not error.

## CS-0074 — Probe units: `cook.probe`, `probes` field on `cook.add_unit`, demand-driven scheduling [#cs-0074]

**Date:** 2026-05-15
**Version:** v0.11

**Summary.** CS-0074 introduces probe units. New register-phase API `cook.probe(key, opts)` registers a DAG unit whose output is a Lua value rather than a file (§22.5). Consumer units declare probe-edge dependencies via a `probes` field on `cook.add_unit`; the `$<key:field>` sigil in command templates auto-populates this field (§22.5.5, §22.5.6). Probe values are msgpack-encoded and stored via the existing cache backend; the `ArtifactMeta.kind` field disambiguates probe-value artifacts from file artifacts (§22.5.4, §22.5.7). Probes execute on demand: a probe MUST NOT run unless at least one scheduled non-probe consumer transitively references its key (§22.5.7). `cook.cache.set` is deprecated and raises a runtime error on the execute-phase VM (§24). `cook.cache.get` on the execute-phase VM reads from a per-run probe-value store populated by the scheduler (§24).

**Motivation.** Modules such as `cook_cc` have used `cook.cache.set`/`cook.cache.get` as a hand-rolled memoisation layer for configure-time probes (compiler detection, package finder results, toolchain version strings). ## CS-0075 — cc module probe-based migration

**Affects:** `cook_cc` rock (§{cat.cc}), surface v0.4 → v0.5.

The cc module's register-phase ad-hoc caches migrate to first-class
`cook.probe` units (§{cat.probes}). The following probe keys are now
canonical:

- `cc:compiler:<override>` — toolchain detection
- `cc:linker-search-dirs` — bare-probe linker search paths
- `cc:cmake-driver` — cmake-compat driver discovery
- `cc:find:<name>` — per-library find result

Target makers (`cc.bin`, `cc.lib`, `cc.shared`, `cc.headers`) accept a
`needs = {…}` field for declarative system-library discovery. The
imperative `cc.find` and `cc.find_or_error` functions return a
*sigil record* — a table of `$<cc:find:<name>.<field>>` placeholders
resolved at execute time by the existing template-expansion pipeline
(§{cat.probes.templates}).

`cc.find_or_error` failure becomes demand-driven: missing libraries
fail the build at execute time rather than raising during register.
This is a deliberate tradeoff — recipes that do not consume the
missing library no longer fail with register-time noise.

The `cc:*` probe namespace is non-normative pattern documentation;
cook_cc itself is not part of the conformance surface, but the design
informs other blessed modules.

## CS-0078 — Multi-line `cook` outputs and `ingredients` patterns [#changes.cs-0078]

**Summary.** App. A.4 grows a normative paragraph specifying that whitespace between successive `STRING` outputs in `cook_step` and successive `ingredient`s in `ingredients_step` MAY include line breaks. Pattern collection across continuation lines is required when each continuation line begins with `"` (or `!"` for `ingredients_step`). A non-quote token on the same physical line as a quoted pattern is a diagnostic when it is not the legal terminator (`using` for cook). A continuation line whose first token is not a quote terminates the declaration without a diagnostic — the standalone-line case is ambiguous between misspelled continuation and a new step, and the Standard preserves the declaration-only form by choosing silent termination.

**Rationale.** Multi-output `cook` steps with three or more outputs (and ingredients declarations with several patterns) become unreadable on a single line. The Doom 3 demo Cookfile uses three-output AAS `cook` steps as the forcing example. The change is backward-compatible: single-line forms continue to parse unchanged; only the previously-rejected multi-line form now parses. The decision to NOT diagnose a standalone non-quote line preserves the declaration-only `cook_step` shape (the user might have intended to terminate the declaration and start a new step).

**Conformance impact.** A conforming parser MUST accept multi-line forms. Positive fixtures `057-cook-multi-line-outputs` and `058-multi-line-ingredients` pin the new shape. Negative fixtures `multi-line-cook-non-quote-continuation` and `multi-line-ingredients-non-quote-continuation` pin the diagnostic behavior when a non-quote token appears trailing a quoted pattern on the same physical line. The silent-termination case for standalone non-quote continuation lines is intentionally not pinned by a negative fixture (the behavior is "no error, declaration treated as complete," which is observed via the absence of a diagnostic, not via a specific diagnostic match).

## CS-0079 — `fs.glob` accepts an array of patterns [#changes.cs-0079]

**Summary.** §25.7 widens the signature of `fs.glob` from `(pattern: string)` to `(pattern: string | array of string)`. When given an array, a conforming implementation MUST glob each pattern in call order and concatenate the per-pattern results in that order, preserving per-pattern internal order. The CS-0064 directory-filter rule applies independently to each pattern. Invalid argument types raise a Lua runtime error tagged `fs.glob`.

**Rationale.** The Doom 3 demo Cookfile (see `docs/superpowers/specs/2026-05-19-doom3-demo-roadmap-design.md` §1.2a and §4) selects platform sources by globbing several patterns and concatenating the results. The single-string form forced a `for` + `table.insert` loop per joined pattern, which dominated the Cookfile's platform-source block visually. The array form expresses the intent in one call.

**Backward compatibility.** Single-string callers continue to work bit-for-bit identically; the array form is purely additive.

**Conformance impact.** A conforming implementation MUST accept both forms. Positive fixtures `fs-glob-array-of-patterns` and `fs-glob-array-preserves-order` pin the Cookfile surface; the runtime contract (call-order concatenation, per-pattern CS-0064 filter, sandbox check applied per pattern, error on bad argument types) is verified by `cook-lua-stdlib::fs_api` unit tests.

## CS-0080 — `cook_cc` PRIVATE/PUBLIC propagation, surface v0.7 [#changes.cs-0080]

**Summary.** §28.4 introduces a PRIVATE/PUBLIC distinction for `cook_cc` target options. The bare fields `defines`, `system_libs`, `frameworks`, `extra_ldflags` are PRIVATE-by-default: a conforming implementation MUST use them only for the target's own compile/link and MUST NOT export them through `cook.export(name, info)`. The new fields `export_defines`, `export_system_libs`, `export_frameworks`, `export_extra_ldflags` are the explicit-public counterparts and propagate to consumers via the transitive `links` walk. `export_includes` retains its existing fall-back to bare `includes` (the sole backcompat carve-out). Toolchain defaults (`cc.toolchain({ ... })`) are merged into the target's bare fields under this rule and therefore stay target-local. The conformance surface bumps from v0.6 to v0.7; the reference implementation is `cook_cc-0.8.0-1`.

**Rationale.** dhewm3 has 8+ static libs that consume each other via `links = { ... }`. Under v0.6 implicit-public propagation, every define a lib carried leaked to consumers. `idLib`'s `ID_LIB_INTERNAL` would pollute every consumer's compile; `idLib`'s `USE_OPENAL` (which DOES want to be public) was indistinguishable from `ID_LIB_INTERNAL` in the surface. M4-narrow draws the line. The doom 3 demo Cookfile (`docs/superpowers/specs/2026-05-19-doom3-demo-roadmap-design.md` §4) is the forcing example.

**Backward compatibility.** Single-target Cookfiles that use only `cc.bin` (no downstream consumer) observe no behavior change. Multi-target Cookfiles that relied on bare-field propagation through `links = { ... }` must rename the leaking field to `export_*` to preserve the v0.6 behavior. The `export_includes` fall-back means `includes`-only projects (the majority of single-lib examples) continue to work unchanged. The Standard's example corpus required one migration (`examples/lua-build/Cookfile`, `liblua`'s `system_libs` → `export_system_libs`).

**Conformance impact.** Surface v0.7 implementations MUST:
- Set `info.defines = opts.export_defines or {}` (NO fall-back to bare `defines`).
- Set `info.system_libs = opts.export_system_libs or {}`.
- Set `info.frameworks = opts.export_frameworks or {}`.
- Set `info.extra_ldflags = opts.export_extra_ldflags or ""`.
- Set `info.includes = opts.export_includes or opts.includes` (fall-back kept).

Three positive conformance fixtures pin the new surface: `cc-private-define-does-not-propagate` (negative-shape positive fixture: parses a Cookfile that relies on the PRIVATE rule), `cc-export-define-propagates` (the PUBLIC path), and `cc-export-includes-backcompat-fallback` (locks the fall-back carve-out). Runtime semantics are verified by the `cook_cc` busted suite (`cook_cc/spec/targets_spec.lua` "cc PRIVATE/PUBLIC propagation" describe-block).

## CS-0081 — Per-platform source selection canonical pattern in §28.7 [#changes.cs-0081]

**Summary.** Adds §28.7 "Per-platform source selection" to the `cook_cc` chapter, naming the `cook.platform.os` + `fs.glob` Lua idiom as the canonical pattern for selecting per-platform sources for cc targets. The new section is structured in three subsections: §28.7.1 specifies the canonical pattern with two interchangeable Lua forms; §28.7.2 calls out and excludes the structured `sources_per_platform = { linux = ..., darwin = ... }` non-pattern, with the design rationale enumerated; §28.7.3 documents the interaction with the CS-0080 PRIVATE/PUBLIC propagation rules. **Rationale.** The doom 3 demo Cookfile (`docs/superpowers/specs/2026-05-19-doom3-demo-roadmap-design.md` §4) is the forcing example: it gathers per-platform engine sources via the `cook.platform.os` + `fs.glob` join shown verbatim in §28.7.1. The pattern works today with zero implementation changes (`cook.platform` is §24.6, `fs.glob` array form is §25.7/CS-0079, plain Lua `if` and `table.insert` are the host language), but it is not the only shape an author might invent and the Standard had no anchor a demo viewer could cite. **Conformance impact.** A conforming implementation gains no new normative obligation it did not already have under v0.7. §28.7 is documentary in posture: it names a pattern that already conforms (canonical) and a pattern that already does not (structured per-platform table — fails the "unknown option key" rule of §28.5). One positive fixture is added: `cc-platform-source-selection` (a Cookfile that exercises the canonical pattern with `cook.platform.os == "linux"` gating an `fs.glob` extension). The fixture is parse-only at this stage; runtime behaviour is verified by the implementation's existing `cook.platform` + `fs.glob` + `cc.bin` test coverage and does not require a new harness entry point.

## CS-0084 — `cc.shared` accepts an explicit `output` override [#changes.cs-0084]

**Summary.** Adds an optional `output` field to `SharedOpts` (§{cat.cc.shared}). When present, the implementation MUST use it verbatim as the link unit's output path and MUST record it as the recipe's `lib_path` export. When absent, the existing default `build/lib/lib<name>.<ext>` is unchanged. The `cook_cc` reference implementation surface ticks v0.7 → v0.8 (additive only; no existing call shape changes); rock version bumps to `0.10.0-1`.

**Rationale.** The dhewm3 bring-up (Plan 5 wall W22, COOK-2) is the forcing example. dhewm3's engine `dlopen`s game plugins by basename: `sys_local.cpp:DLL_GetFileName` builds `<basename><BUILD_LIBRARY_SUFFIX>` (i.e. `base.so`, `d3xp.so` — no `lib` prefix) and `framework/Common.cpp:LoadGameDLLbyName` searches `PATH_EXE` first (the binary's directory, `build/bin/` in the cook layout). The hard-coded `build/lib/lib<name>.so` shape of v0.7 `cc.shared` produces neither the right filename nor the right directory for any plugin-style runtime. Without an override, every engine-style consumer must paper over the gap with a post-link `cp` chore or symlink — a band-aid that fails reproducibility (the artifact ledger no longer reflects the loadable file) and forces an out-of-band convention. Promoting `output` to a documented surface field codifies the only knob authors actually need and keeps the artifact ledger honest.

**Backward compatibility.** Purely additive. No existing Cookfile becomes non-conforming. Implementations that already accept arbitrary unknown option keys (rather than erroring) are out of compliance with §28.5 independently of CS-0084; this entry does not change that. Default behaviour (no `output`) is byte-identical to v0.7.

**Conformance impact.** A conforming implementation MUST honour `opts.output` verbatim when present and MUST flow the override through `cook.export`'s `lib_path` so consumers' transitive link steps see the right artifact location. No new positive conformance fixture is required at the parse layer (the field is a Lua table key); the reference implementation's busted suite locks the behaviour with a `targets.shared("plug", { output = "build/bin/plug.so" })` case asserting `units[#units].output == "build/bin/plug.so"`. Windows and macOS authors are responsible for authoring the platform-correct suffix in `output`; the implementation MUST NOT replace it.
