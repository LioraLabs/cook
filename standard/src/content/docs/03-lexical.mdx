# 3. Lexical structure [#lexical]
> **Normative.** This chapter defines how a Cookfile's text is broken into tokens. The productions in §{toplevel}, §{decl}, §{recipes}, §{steps}, and Appendix A consume the token classes defined here.

## 3.1. Source representation [#lexical.source-representation]
A Cookfile is a sequence of Unicode characters encoded as UTF-8. A conforming implementation MUST accept any valid UTF-8 input and MUST NOT require a byte-order mark. The behaviour on input that is not valid UTF-8 is unspecified.

Lexical analysis is **line-oriented**: the source is divided into lines by the newline character `U+000A` (`\n`), and each line is classified independently as a single token (§{lexical.tokens}). A conforming implementation MUST accept `\r\n` as a line terminator by treating the carriage return as part of the preceding line's trailing whitespace, which is trimmed before classification (§{lexical.line-classification}). The final line of the source need not be terminated by `\n`.

Leading and trailing whitespace on a line are not significant to token classification, with the following exception already stated in §{lexical.brace-blocks}: leading whitespace is preserved inside brace-balanced block bodies (§{lexical.brace-blocks}) because the body is reassembled from the raw source lines. In all other cases, a line is trimmed of leading and trailing whitespace before its prefix is tested (§{lexical.line-prefixes}, §{lexical.line-classification}).

### Note 3.1.1

The reference tokenizer in `cli/crates/cook-lang/src/lexer.rs` iterates `source.lines()`, which is defined by Rust to split on `\n` and strip a trailing `\r`. The observable effect matches the rule above: `\r\n` and `\n` produce the same token stream.

## 3.2. Tokens [#lexical.tokens]
The Cookfile lexer produces **exactly one token per source line**. A token carries its 1-indexed line number; downstream phases (§{toplevel}, §{recipes}) use that number for diagnostics and for locating multi-line block bodies (§{lexical.brace-blocks}). There is no concept of "multiple tokens on one line" at the Cookfile layer — the interior structure of a line (for example, the output patterns of a `cook` step) is parsed at the syntactic layer on top of the single `Content` token.

The token classes produced by the lexer are:

| Token class            | Produced for                                                                 | Defined in |
|---|---|---|
| `Comment`              | Line whose first non-whitespace character is `#`                              | §{lexical.comments}      |
| `RecipeHeader`         | Line beginning with keyword `recipe` followed by a separator                  | §{recipes.header-forms}      |
| `ConfigHeader`         | Line `config` or `config` + name                                              | §{decl.config}      |
| `UseDecl`              | Line `use` + name                                                             | §{decl.use}      |
| `ImportDecl`           | Line `import` + `BARE_IDENTIFIER` + path                                      | §{decl.import}      |
| `LuaLine`              | Line whose first non-whitespace characters are `>` (but not `>{`)             | §{lexical.line-prefixes}, §{steps.lua} |
| `LuaBlockOpen`         | Line whose first non-whitespace characters are `>{`                           | §{lexical.line-prefixes}, §{lexical.brace-blocks} |
| `Blank`                | Line containing only whitespace                                               | §{lexical.line-classification}     |
| `Content`              | Any other line; its classification within a recipe body is deferred to §{steps.dispatch}  | §{steps.dispatch}      |

A conforming implementation MAY represent these classes with any internal data structure. The token names above are used descriptively throughout this Standard; they do not mandate a specific variant name in an implementation.

### Note 3.2.1

`Content` is deliberately broad: the lexer does not distinguish `ingredients "…"`, `cook "…" …`, `@vim notes.txt`, or `gcc -o main main.c` — all four produce `Content` tokens whose string form is the line after trimming. The step-dispatch cascade in App. A.4 resolves which step kind applies.

## 3.3. Identifiers [#lexical.identifiers]
A `BARE_IDENTIFIER` is a sequence of ASCII characters matching the regular expression in App. A.5:

```ebnf
BARE_IDENTIFIER ::= /[A-Za-z_][A-Za-z0-9_.\-]*/
```

The first character MUST be an ASCII letter (`A`–`Z`, `a`–`z`) or underscore (`_`); each subsequent character MUST be an ASCII letter, ASCII digit, underscore, hyphen (`-`), or dot (`.`). A conforming implementation MUST reject any other character. Unicode letters and digits are NOT part of `BARE_IDENTIFIER`.

`BARE_IDENTIFIER` is used in four syntactic roles, each with its own supplementary constraints documented where that role is defined:

| Role                                   | Supplementary constraint                                                      |
|---|---|
| Import alias (`import_declaration`)    | MUST be unique across imports within the Cookfile (App. A.2).                 |
| Recipe / chore name (declaration site) | MUST NOT contain `.` (App. A.2). The name is a single undotted segment; namespacing is achieved via `import` (§{comp.import}), not via dotted recipe names. |
| Dependency name / cross-recipe reference | MAY contain `.`. A dotted dependency name resolves through an `import` alias per §{comp.qualified-refs}; the segment before the leftmost `.` selects the import, the remainder is the recipe name in the imported Cookfile. |
| First segment of a `module_call`       | MUST NOT contain `.` or `-` (App. A.4).                                       |

Where the grammar admits either a `BARE_IDENTIFIER` or a double-quoted `STRING` (for example, a recipe name or a dependency), the two forms are equivalent in meaning; the quoted form exists to admit names that would not otherwise match `BARE_IDENTIFIER`. The "no dots" restriction on declared recipe and chore names applies to both forms — a quoted recipe name MUST also be undotted.

### Example 3.3.1

```cook
import backend ./services/backend

recipe ship: backend.build lib setup
    cook "ship.out" { echo deploying > $<out> }
```

`ship`, `lib`, and `setup` are valid undotted recipe / dependency names. `backend.build` is a valid dotted *dependency* name: it is not a recipe defined here, but a qualified reference to the `build` recipe of the Cookfile imported under the alias `backend`. By contrast, a top-level declaration `recipe backend.build` would be rejected at parse time — namespaced recipes live behind `import`, not in dotted top-level definitions.

## 3.4. Keywords [#lexical.keywords]
The Cookfile language reserves seven identifiers contextually, each in a single role: the final dot-separated segment of a recipe name. Outside that role the identifiers are ordinary tokens.

| Reserved recipe segment | Role where reserved             |
|---|---|
| `stem`                 | Final segment of a recipe name   |
| `name`                 | Final segment of a recipe name   |
| `ext`                  | Final segment of a recipe name   |
| `dir`                  | Final segment of a recipe name   |
| `in`                   | Final segment of a recipe name   |
| `out`                  | Final segment of a recipe name   |
| `env`                  | Final **or first** segment of a recipe name |

`all` is NOT a reserved recipe segment (CS-0130): `recipe all` and a dependency segment `all` are both well-formed. The `all`-spelled sigil was removed as a placeholder form by the same change; it now resolves as an ordinary name reference — a recipe if `all` is in scope, otherwise an env var lookup (§{phl.token}, App. E CS-0130).

A conforming implementation MUST reject a *placeholder identifier* (§{phl.token}) or a *dependency name* whose final dot-separated segment is a reserved recipe segment (App. A.2). The reserved-segment rule is contextual to the resolution surface — it constrains what dotted reference forms are well-formed, not the lexical class of `BARE_IDENTIFIER` itself. (At a top-level recipe or chore declaration site the name is undotted by §{lexical.identifiers}, and the reserved-segment rule does NOT constrain it: `recipe out`, `chore name`, and `recipe env` are all well-formed declaration names (CS-0132). The reservation binds only the *reference* surfaces below — a `dependency_list` final segment, a `$<...>` placeholder identifier, and the `env`-first prefix — never a declaration name or a chore parameter name.) The identifier `env` is further restricted: it also MUST NOT appear as the first segment of a dotted reference, so `$<env.foo>` is the explicit env-var-lookup form and a *dependency* `env.foo` is rejected at parse time. This restriction reserves the `env.` prefix for the explicit env-var-lookup form and eliminates the resolution ambiguity that would otherwise arise between `$<env.X>` (env-var lookup) and an accessor on a recipe named `env`.

The keywords that introduce other top-level or recipe-body productions — `recipe`, `chore`, `config`, `ingredients`, `cook`, `plate`, `use`, `import`, `test` — are recognised as keywords only when followed by a keyword separator (space, tab, or `"`). (The former body introducer `using` was removed in CS-0099 and is no longer a keyword; at its old position it is rejected with a migration diagnostic, §{grammar-appendix.steps}.) Among these, the **top-level keywords** `recipe`, `chore`, `config`, `use`, `import`, and `register` are recognised as keywords only when their line begins at column 0 (no leading whitespace); an indented line whose first word is a top-level keyword is a `Content` token, dispatched per §{steps.dispatch}. The recipe-body keywords (`ingredients`, `cook`, `plate`, `test`) are not subject to the column-0 rule. The `register` keyword is recognised when followed by space, tab, or NEWLINE — `register` alone on a line is admitted as the canonical header form because the block takes no name.

### Note 3.4.1

The separator test is the governing subtlety. See the test `test_recipe_prefix_is_shell_command` in `cli/crates/cook-lang/src/lexer.rs`, which confirms that `recipes_cleanup` produces a `Content` token and not a `RecipeHeader`.

## 3.5. Strings [#lexical.strings]
A `STRING` token is a double-quoted sequence of characters defined in App. A.5:

```ebnf
STRING ::= /"[^"]*"/
```

A `STRING` opens with `U+0022` (`"`), contains zero or more characters none of which is `"`, and closes with `U+0022` (`"`). A conforming implementation MUST NOT interpret backslash escapes inside a `STRING`: the character sequence is taken literally. A conforming implementation MUST reject a `STRING` that is not terminated on the same line — `STRING` tokens do not span lines.

The empty string `""` is a valid `STRING`. A double quote inside a `STRING` is not expressible in the current grammar; authors needing such content can use the inline-block forms of §{lexical.brace-blocks}, whose interior language (Lua or shell) has its own quoting rules.

`STRING` appears in recipe and dependency names, in `ingredients`, `cook`, `plate`, and `test` steps, and in the optional name of a `config_block` header. In each of those roles, a conforming implementation parses the `STRING` inside the single line that produced the governing token. The `cook_step` body (§{steps.cook-single}) takes a brace-block form, not a `STRING`; the former single-string body was removed in CS-0022, and the `using` introducer itself in CS-0099.

### Example 3.5.1

```cook
recipe "build"
    ingredients "src/*.c" !"src/legacy.c"
    cook "bin/app" { gcc $<in> -o $<out> }
```

Each `"…"` is a `STRING`. The `!"src/legacy.c"` form is an `ingredient` exclusion, not a distinct string class (App. A.4): it is the literal `!` followed by an ordinary `STRING`. The `{ … }` body is a shell block (§{lexical.brace-blocks}), not a `STRING`.

### Note 3.5.1

The reference implementation of the string parser is in `cli/crates/cook-lang/src/cook_line.rs`, function `parse_single_quoted_string`. The lack of escape processing is observable: the test `test_cook_step_shell` asserts that a `STRING` value is delivered verbatim to the step.

## 3.6. Comments [#lexical.comments]
A **comment** is a line whose first non-whitespace character is `#`. The comment extends to the end of the line. A conforming implementation MUST produce a `Comment` token for such a line and MUST NOT treat the `#` character as beginning a comment in any other lexical position.

In particular, a `#` that appears after the first non-whitespace character of a line is part of that line's content and is not stripped by the lexer. Downstream phases (notably the shell runtime) MAY assign their own meaning to in-line `#` characters; that meaning is not part of Cookfile lexical syntax.

A `Comment` token is accepted anywhere a `toplevel_item` (App. A.1) or a `recipe_item` (App. A.3) is admitted, and is discarded at the syntactic layer. Comments are permitted inside `config_body` (App. A.2) and inside brace-balanced blocks (§{lexical.brace-blocks}), where they are preserved verbatim in the collected block body because their removal is the interior language's concern, not the Cookfile lexer's.

### Example 3.6.1

```cook
# top-level comment: discarded at the syntactic layer

chore "build"
    # indented comment: also a Comment token
    gcc -o main main.c   # not a comment — part of the shell command
```

The first two `#` lines produce `Comment` tokens. The `# not a comment` sequence on the third source line is part of the `Content` token for the `gcc` command (a chore-body `shell_command`); its treatment is up to the shell runtime (§{xref}). The lexer classifies `Content` tokens the same way regardless of the enclosing body; a chore body is used here because a recipe body admits no loose shell line (§{steps.dispatch}).

## 3.7. Line prefixes [#lexical.line-prefixes]
A **line prefix** is a character sequence that, when it appears as the first non-whitespace characters of a Cookfile line, selects a token classification other than `Content`. The Cookfile language recognises two lexical line prefixes:

| Prefix | Token produced            | Phase    | Defined in |
|---|---|---|---|
| `>{`   | `LuaBlockOpen`            | execute  | §{lexical.brace-blocks}, §{steps.lua} |
| `>`    | `LuaLine`                 | execute  | §{steps.lua} |

A conforming implementation MUST test these prefixes in **longest-prefix order**: `>{` before `>`. Without this ordering, a `>{` would be consumed as `>` followed by a literal `{`. A conforming implementation MUST apply these prefixes only after leading whitespace has been trimmed from the line; thus `    >{` opens an inline Lua block identically to `>{` at column zero.

A run of three or more `>` characters at the start of a trimmed line — `>>>`, `>>>>`, etc. — is reserved and MUST be rejected with a parse error. The reservation keeps space for future prefixes without ambiguity.

The phase column above is informative; the normative phase rule is in §{steps.lua} and §{exec.two-phase}. Both `>{` and `>` are execute-phase prefixes; as of CS-0134 a conforming implementation MUST admit them only in **body position** — inside a `cook`, `test`, `plate`, or `probe` body, or inside a chore body — and MUST NOT admit them as a standalone recipe-body step. The position constraint is normative in §{steps.lua}; this chapter defines only the prefix-to-token mapping. See §{rationale.lua-prefix-split} for background on the surface split.

The `@` character is not a lexical prefix, and — as of CS-0134 — carries no meaning at any layer. A line whose first non-whitespace character is `@` produces a `Content` token whose string form includes the leading `@`. CS-0134 removed the `@` interactive prefix and the `interactive_command` step kind it once introduced (Note 3.7.1); an `@` line is therefore never a distinct step kind. Inside a **chore** body it is an ordinary `shell_command` whose body text begins with a literal `@` and is passed verbatim to the shell (which will typically report `@…: command not found`); inside a **recipe** body it is a `Content` line matching no declarative step kind and is rejected (§{steps.dispatch}); outside any body it is rejected because `Content` tokens are not valid at the top level (§{toplevel.ordering}).

### Note 3.7.1 — Removal of the doubled-`>` prefixes (CS-0134)

Prior to CS-0134, the language additionally recognised a register-phase pair of doubled-`>` prefixes: `>>{` (`InlineLuaBlockOpen`) and `>>` (`InlineLuaLine`), alongside the execute-phase `>{` / `>` above. CS-0134 REMOVED both doubled-`>` prefixes entirely: the recipe body is now purely declarative, and a bare `ident.ident(...)` module-call line auto-classifies as register-phase (§{steps.dispatch}) in the position that `>>` / `>>{` used to occupy.

A trimmed line whose first non-whitespace token is `>>` — and that is not one of the reserved `>>>+` forms above — is still *recognised* by the lexer (§{lexical.line-classification}), but a conforming implementation MUST reject it with a migration diagnostic directing the author to a top-level `register` block (§{decl.register}) or to a bare `module.call()` line. The same applies to `>>{`.

A `LuaLine` body MAY be empty: the line `>` alone produces a `LuaLine` whose body is the empty string. See §{steps.lua}.

## 3.8. Numbers [#lexical.numbers]
A `NUMBER` token is a non-empty sequence of ASCII digits defined in App. A.5:

```ebnf
NUMBER ::= /[0-9]+/
```

`NUMBER` has a single use in the Cookfile grammar: the `timeout` operand of a `test_step` (App. A.4). The `NUMBER` is parsed as an unsigned integer denoting a whole number of seconds. A conforming implementation MUST reject a `test` step whose `timeout` operand is not a valid `NUMBER`. Signs, decimal points, fractional digits, and unit suffixes are NOT permitted.

No other step kind accepts a bare `NUMBER`; digits that appear inside a shell command, a `STRING`, or a Lua block are the interior language's concern and are not Cookfile tokens.

### Example 3.8.1

```cook
recipe "run-tests"
    test { ./$<in> } timeout 30 should_fail
```

`30` is a `NUMBER` denoting a 30-second timeout.

### Note 3.8.1

The reference implementation is `parse_test_modifier_tail` in `cli/crates/cook-lang/src/recipe.rs`. The test `test_test_step_with_timeout_and_should_fail` demonstrates the admitted operand ordering: a `NUMBER` directly after `timeout`, followed by the optional `should_fail` marker.

## 3.9. Brace-balanced blocks [#lexical.brace-blocks]
A **brace-balanced block** is a span of source text delimited by `{` and a matching `}`, where matching is defined by a stateful brace-counting algorithm that ignores braces appearing inside certain syntactic contexts. Both the `LUA_BLOCK_CONTENT` (inside `>{ … }`) and the `SHELL_BLOCK_CONTENT` (inside `{ … }`) are brace-balanced blocks; they differ in their interior language (Lua vs. shell) and in how their collected body is normalised after the block closes.

The opening brace is consumed on the line that introduced the block (a `LuaBlockOpen` token for `>{`, or the `{` that opens a `shell_block` body), establishing initial depth `1`. The block's body lines are then scanned in source order. Each line is examined character by character; `{` increments depth and `}` decrements depth, except that braces are ignored when they appear inside one of the **opaque spans** enumerated in §{lexical.brace-blocks.lua-spans} (for Lua blocks) or §{lexical.brace-blocks.shell-spans} (for shell blocks). The brace counter MUST carry **opaque-span state across lines** so that a span opened on one line and closed on a later line keeps every brace byte between the open and the close opaque.

A conforming implementation MUST apply this exclusion consistently for both Lua blocks and shell blocks, so that a block's closing `}` is never mistaken for a literal `}` inside string, comment, or heredoc content. The block closes on the first line where the running depth returns to zero or below after the line's brace deltas have been applied; that closing line is consumed by the lexer but is NOT included in the block body.

The `LUA_BLOCK_CONTENT` body is the concatenation of the interior lines in source order, joined by newlines, preserving their original whitespace. The `SHELL_BLOCK_CONTENT` body is further normalised per App. A.5: each interior line is trimmed of leading and trailing whitespace, and lines that are empty after trimming are discarded; the result is an ordered sequence of command strings, not a single raw string.

A conforming implementation MUST reject a block whose depth never returns to zero by end of input.

### 3.9.1. Opaque spans inside Lua blocks [#lexical.brace-blocks.lua-spans]

For a `LUA_BLOCK_CONTENT` body and for the body of a multi-line module call (App. A.4 `module_call`), braces are ignored when they appear:

- inside a Lua line comment, beginning at `--` and extending to the end of the line (single-line span);
- inside a double-quoted string `"…"`, with `\`-escapes honoured (single-line span);
- inside a single-quoted string `'…'`, with `\`-escapes honoured (single-line span);
- inside a Lua **long string** `[==[ … ]==]` of any `=`-level — including the level-0 form `[[ … ]]` (single-line **or** multi-line span);
- inside a Lua **block comment** `--[==[ … ]==]` of any `=`-level — including the level-0 form `--[[ … ]]` (single-line **or** multi-line span).

For long strings and block comments, the closing bracket MUST have the same `=`-level as the opener: a `]]` token does not close a `[==[` opener, and a `]==]` token does not close a `[[` opener. This is the long-bracket rule of the Lua 5.4 reference manual; a conforming implementation MUST honour it for any number of `=` characters.

When a long string or block comment is opened on one source line and closed on a later one, every brace byte in the intervening lines is opaque to the brace counter. CS-0035 establishes this multi-line carry as a normative requirement; see App. D.

### 3.9.2. Opaque spans inside shell blocks [#lexical.brace-blocks.shell-spans]

For a `SHELL_BLOCK_CONTENT` body, braces are ignored when they appear:

- inside a single-quoted string `'…'` (no escapes; single-line span);
- inside a double-quoted string `"…"`, with `\`-escapes honoured (single-line span);
- after a backslash-escape `\{` or `\}` (the next byte is consumed as data; single-line span);
- inside a POSIX **heredoc body** opened by `<<TAG`, `<<-TAG`, `<<'TAG'`, or `<<"TAG"` and closed by a line whose trimmed text equals `TAG` (multi-line span).

The heredoc closer is matched against the **trimmed** body line: this is consistent with the `SHELL_BLOCK_CONTENT` line-normalisation rule above, which trims each interior line before assembling the shell script. Authors writing a heredoc inside an indented `{ … }` body therefore do not need to dedent the closing delimiter to column 0 in the source; the trim-first match closes the heredoc as soon as the trimmed line matches the delimiter.

When a shell heredoc is opened on one source line and closed on a later one, every brace byte in the intervening lines is opaque to the brace counter. Multiple heredocs queued on a single opener line (`cmd <<A <<B`) MUST be consumed in opener order: the first delimiter line closes the first heredoc, the second closes the second, and so on. CS-0035 establishes this multi-line carry as a normative requirement; see App. D.

### Example 3.9.1

```cook
recipe "build"
    cook "out.wasm" >{
        if mode == "release" then
            local t = {1, 2, 3}   -- a literal `}` in a comment: }
            local s = "a { brace in a string }"
            print(s)
        end
    }
    cook "out.js" {
        wasm-pack build
        cp out/main.js out.js
    }
```

Both blocks here are `cook`-step bodies (CS-0134: a `>{ … }` or `{ … }` block appears only as a step body, never standalone in a recipe body). The first `cook` step's `exec_lua_block` (`>{ … }`) body includes the `if`/`end` structure and the nested table `{1, 2, 3}`; the braces inside the string `"a { brace in a string }"` and after the `--` comment are ignored by the brace counter. The second `cook` step's `{ … }` shell body, after line normalisation, is the two-element command list `["wasm-pack build", "cp out/main.js out.js"]`.

### Example 3.9.2

```cook
recipe "build"
    cook "doc.txt" >{
        local doc = [==[
            multi-line long string with a } byte
            and a stray ]] that does NOT close it
        ]==]
        print(doc)
    }
```

The `>{ … }` here is the `cook` step's `exec_lua_block` body (CS-0134: a `>{ … }` block appears only as a step body). The `[==[ … ]==]` long string spans four source lines. The `}` byte on the second body line and the `]]` token on the third are interior to the long string and MUST be opaque to the brace counter. The `>{ … }` block closes only at the trailing `}` on the line after `print(doc)`. The corresponding pin is fixture `041-lua-long-string-leveled` in the conformance corpus.

### Example 3.9.3

```cook
recipe emit
    cook "out.txt" {
        cat <<EOF > out.txt
        a heredoc with a } brace
        and a } here too
        EOF
        echo done
    }
```

The `cat <<EOF` opens a shell heredoc; its body extends through the next two lines, in which `}` bytes are heredoc data and MUST be opaque to the brace counter. The heredoc closes on the trimmed `EOF` line; brace counting resumes for `echo done` and the trailing `}` closes the `{ … }` block. The corresponding pin is fixture `042-shell-heredoc-brace-in-body`.

### Note 3.9.1

The governing examples are the tests `test_lua_block_nested_braces`, `test_lua_block_brace_in_string`, `test_lua_block_brace_in_comment`, and the `cs_0035_*` family in `cli/crates/cook-lang/src/tests.rs`; the tests `collects_three_lines`, `drops_blank_lines`, `respects_nested_braces_in_content`, and the `cs_0035_*` family in `cli/crates/cook-lang/src/shell_block.rs`; and the unit tests in `cli/crates/cook-lang/src/brace_scan.rs`.

## 3.10. Line classification cascade [#lexical.line-classification]
For each source line, a conforming implementation MUST classify the line into exactly one token by applying the following tests in order, with the first matching test winning. The line's trimmed form (leading and trailing whitespace removed) is the subject of each test except where noted.

A **keyword separator** in tests 7–11 below is a space (`U+0020`), a tab (`U+0009`), or a double quote (`U+0022`). The double quote is admitted so that authors can write `recipe "foo"` with no space between the keyword and the opening quote of a quoted name; the quote is not consumed by the keyword test and remains available to the `STRING` parser (§{lexical.strings}). Test 11 (`import`) admits only space or tab: no quoted form exists for the `import` alias (App. A.2).

1. If the trimmed line is empty, the token is `Blank`.
2. If the trimmed line's first character is `#`, the token is `Comment` (§{lexical.comments}).
3. If the trimmed line begins with `>>{`, the line is recognised as a doubled-`>` prefix; per CS-0134 this prefix was REMOVED from the language, and a conforming implementation MUST reject the line with a migration diagnostic rather than emit a usable token (§{lexical.line-prefixes}).
4. If the trimmed line begins with `>>` (but not `>>{`, by test 3), the line is recognised as a doubled-`>` prefix; per CS-0134 this prefix was REMOVED from the language, and a conforming implementation MUST reject the line with the same migration diagnostic as test 3 (§{lexical.line-prefixes}).
5. If the trimmed line begins with `>{`, the token is `LuaBlockOpen` (§{lexical.line-prefixes}, §{lexical.brace-blocks}, §{steps.lua}).
6. If the trimmed line begins with `>` (but not `>>` by test 4 or `>{` by test 5), the token is `LuaLine` (§{lexical.line-prefixes}, §{steps.lua}).
7. If the line begins at column 0 and the trimmed content begins with the keyword `recipe` followed by a keyword separator, the token is `RecipeHeader` (§{recipes.header-forms}). An indented line whose trimmed content begins with `recipe` falls through to test 12.
8. If the line begins at column 0 and starts with `chore` followed by a separator (space, tab, or `"`), the token is `ChoreHeader`. The keyword separator rule of §{lexical.keywords} applies: `chores_cleanup` (no separator after `chore`) is not a `ChoreHeader`. (CS-0020 / v0.4.)
9. If the line begins at column 0 and the trimmed content is exactly `config`, or begins with `config` followed by a keyword separator, the token is `ConfigHeader` (§{decl.config}). An indented line falls through to test 12.
10. If the line begins at column 0 and the trimmed content begins with `use` followed by a keyword separator, the token is `UseDecl` (§{decl.use}). An indented line falls through to test 12.
11. If the line begins at column 0 and the trimmed content begins with `import` followed by a whitespace separator (space or tab only; not `"`), the token is `ImportDecl` (§{decl.import}). An indented line falls through to test 12.
11.5. If the line begins at column 0 and the trimmed content is exactly `register`, or begins with `register` followed by a keyword separator (space or tab), the token is `RegisterHeader` (§{decl.register}). An indented line falls through to test 12. A `register` line with non-whitespace content after the keyword on the same line is rejected at the syntactic layer per §{decl.register}; the lexer admits the `RegisterHeader` token regardless and lets the parser issue the diagnostic.
12. Otherwise, the token is `Content`. The lexer emits `Content` uniformly; its further classification is the concern of the step-dispatch cascade (App. A.4, §{steps.dispatch}) and is made by the **syntactic layer per enclosing container**. The admissible dispatch results differ by container (CS-0134): inside a **recipe body** a `Content` token classifies to `ingredients_step`, `seal_step`, `cook_step`, `plate_step`, `test_step`, or a bare `module_call` (register-phase, §{steps.dispatch} rule 7) — a loose `shell_command` is **rejected**, because a recipe body is purely declarative (§{recipes.region-rule}). Inside a **chore body** a `Content` token classifies to a `shell_command`, which remains admitted (§{chores.body}) and is interactive by default (§{chores.default-interactive}). CS-0134 removed the `@` interactive prefix and its `interactive_command` step kind from the language (Note 3.7.1), so an `@` line is no longer a distinct classification in either container. (Top-level `module_call` is dispatched separately per §{toplevel.module-call}.)

The ordering is significant. In particular, the four Lua-prefix tests (3–6) MUST appear in longest-prefix order — `>>{` before `>>` before `>{` before `>` — so that a longer prefix is never consumed as a shorter one followed by a literal character, and so that a doubled-`>` line receives the correct migration diagnostic rather than falling through to a `>{` / `>` classification (§{lexical.line-prefixes}). Tests 3 and 4 never yield a usable token: they exist only to recognise the removed `>>{` / `>>` prefixes precisely enough to reject them with the migration diagnostic of CS-0134 (Note 3.7.1) rather than let such a line fall through to test 12 as unclassified `Content`. Tests 7–11 apply only to lines beginning at column 0, and MUST require a separator so that identifiers whose first word merely starts with a keyword (for example, `recipes_cleanup`, `chores_cleanup`, `configure`, `useful`, `important`) fall through to test 12 and are not misclassified as declarations.

A trimmed line beginning with three or more consecutive `>` characters (`>>>`, `>>>>`, etc.) does not match any of tests 3–6 and is reserved (§{lexical.line-prefixes}). A conforming implementation MUST reject such a line with a parse error rather than fall through to test 12.

Tests 1–6 and test 12 apply to lines whether or not they are indented. Tests 7–11 are sensitive to column-0 positioning: an indented line whose first non-whitespace content would otherwise match a top-level keyword test falls through to test 12 and is classified as `Content`.

### Example 3.10.1

```cook
use "cpp"                 # test 10: UseDecl (column 0)
config debug              # test 9:  ConfigHeader (column 0)
    env.CC = "gcc"

recipe build: lib setup   # test 7:  RecipeHeader (column 0)
    # test 2: Comment
    ingredients "src/*.c" # test 12: Content (step-dispatched to ingredients_step)
    >>{                   # test 3:  recognised, then REJECTED (CS-0134 — doubled-`>` removed)
        cook.env.CFLAGS = "-Wall"
    }
    >> local n = 4        # test 4:  recognised, then REJECTED (CS-0134 — doubled-`>` removed)
    >{                    # test 5:  LuaBlockOpen (lexes; recipe parser REJECTS a standalone >{ per CS-0134)
        print("hi")
    }
    > os.exit(0)          # test 6:  LuaLine (lexes; recipe parser REJECTS a standalone > per CS-0134)
    recipes_cleanup       # test 12: Content (falls through test 7 — no separator)
```

### Example 3.10.2

```cook
register                  # test 11.5: RegisterHeader (bare form)
register foo              # test 11.5: RegisterHeader (parser rejects the trailing arg)
register	foo           # test 11.5: RegisterHeader (tab separator)
    register              # test 12:   Content (indented; falls through 11.5)
register_cleanup          # test 12:   Content (no separator after `register`)
registers_cleanup         # test 12:   Content (no separator; not a keyword match)
```

The bare `register` and `register foo` lines both produce a `RegisterHeader` token; per §{decl.register}, the parser rejects the second because a `register_block` admits no name. The tab-separator form is equivalent to the space-separator form at the lexical layer. The indented `register` line is a `Content` token because tests 7–11.5 are sensitive to column-0 positioning. `register_cleanup` and `registers_cleanup` fall through to test 12 because the byte after `register` is `_` and `s`, not a keyword separator.
