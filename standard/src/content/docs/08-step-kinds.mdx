# 8. Step kinds [#steps]
> **Normative.** This chapter defines every step kind that appears in a recipe body or a chore body, the dispatch cascade that classifies each `Content` token, and the iteration-mode rules for `cook`, `plate`, and `test` steps. The execute-phase behaviour of a body's compiled units — including body bundling — is specified in §{exec.groups}.

## 8.1. Step-dispatch cascade [#steps.dispatch]
Inside a `recipe_body`, each source line is first classified by the lexer (§{lexical.line-classification}) into one of `Comment`, `Blank`, `LuaLine`, `LuaBlockOpen`, `InlineLuaLine`, `InlineLuaBlockOpen`, or `Content`. Comments and blank lines are skipped; `LuaLine` becomes a `lua_line` step; `LuaBlockOpen` begins a `lua_block` step collected by the brace-balance algorithm (§{lexical.brace-blocks}); `InlineLuaLine` becomes an `inline_lua_line` step; `InlineLuaBlockOpen` begins an `inline_lua_block` step (also brace-balanced).

Prior to CS-0072 (Cook Standard v0.9), this cascade included a fifth rule classifying `BARE_IDENTIFIER "." BARE_IDENTIFIER(…)` as `module_call` (register-phase Lua). CS-0072 removes that rule from the recipe-body cascade and adds an equivalent top-level dispatch in §{toplevel.module-call}. Authors who want register-phase Lua inside a recipe body MUST use the `>>` (`inline_lua_line`) or `>>{` (`inline_lua_block`) prefix.

A `Content` token inside a `recipe_body` is dispatched to exactly one `declarative_item` or `imperative_item` (App. A.3) by the priority cascade below (App. A.4, "Step-dispatch priority"). The first match wins; a conforming implementation MUST test the alternatives in this order:

1. Prefix `ingredients` + separator (space, tab, or end of token) → `ingredients_step`.
2. Prefix `cook` + separator → `cook_step`.
3. Prefix `plate` + separator → `plate_step`.
4. Prefix `test` + separator → `test_step`.
5. First character is `@` and the remainder of the line is non-empty → `interactive_command`.
6. Otherwise → `shell_command`.

In step 6, an `@` with no following content MUST be rejected rather than silently dispatched as any of the preceding alternatives. The diagnostic MUST identify the offending line.

A `Content` token inside a `recipe_body` whose trimmed text begins with the keyword `register` followed by a separator (space, tab, or end-of-line) MUST be rejected with the diagnostic "`register` blocks are top-level only; move this outside the recipe body". The rejection applies regardless of indentation depth: an indented `register` cannot be a `RegisterHeader` (which is column-0 only by §{lexical.line-classification}) and is also not a permitted shell command in this position. A `Content` token whose first token is the bare identifier `register` *without* a separator (e.g. `register_foo`) remains a `shell_command` per the (post-CS-0072) rule 7. The reference parser fires this rejection in `parse_recipe` and `parse_chore`'s step-dispatch loops (informative).

A `recipe_body` MUST contain at most one `ingredients_step`. A conforming implementation MUST reject a recipe with two or more `ingredients_step` items (App. A.3).

A `recipe_body` MUST satisfy the **region ordering rule** (App. A.3, "Region ordering rule"; §{recipes.region-rule}): every `declarative_item` (other than `comment` or `NEWLINE`) MUST appear before the first `imperative_item` of the recipe body. A conforming implementation MUST reject a recipe in which a declarative-region step (cook, plate, test, ingredients, `>>`, or `>>{`) appears after the first imperative-region step (`>`, `>{`, shell, or `@`). The diagnostic MUST identify the offending step's line and the line on which the imperative region began.

### Example 8.1.1

```cook
recipe build
    ingredients "src/*.c"
    cook "bin/$<in.stem>" { cc $<in> -o $<out> }
    test { ./bin/app }
    @vim notes.txt
    echo done
```

The five body lines dispatch, in order, to: `ingredients_step`, `cook_step`, `test_step`, `interactive_command` (rule 6), and `shell_command` (rule 7). A bare `cpp.link()` line in a recipe body — which before CS-0072 dispatched to `module_call` — now falls through to rule 7 and becomes a `shell_command`.

### Example 8.1.2

```cook
recipe build
    CC "gcc"
```

The body line `CC "gcc"` is a `Content` token; the step-dispatch cascade matches no preceding alternative and dispatches it as a `shell_command` whose text is the trimmed source line. The emitted command preserves the original surface form — the identifier and its quoted argument.

### Note 8.1.1

The `cook-lang` parser test `test_indented_quoted_pair_is_shell_command` pins the recipe-body treatment of `NAME "value"`. The test `test_empty_interactive_step_errors` pins rejection of a bare `@`.

The reference parser at `cli/crates/cook-lang` enforces the CS-0072 amendment by removing the `is_module_call` branch from the recipe-body step-dispatch in `parse_recipe` / `parse_chore` (informative).

## 8.2. Ingredients [#steps.ingredients]
A recipe body MAY contain at most one `ingredients_step` (§{grammar-appendix.recipes}, "At-most-one-ingredients rule"). A second `ingredients` line within the same recipe body MUST be rejected.

The `ingredients_step` is introduced by the keyword `ingredients` followed by one or more `ingredient` items (§{grammar-appendix.steps}, `ingredients_step`). Each item is either:

| Form | Meaning |
|---|---|
| `STRING`        | **include**: the quoted text is a glob pattern added to the ingredient set. |
| `!` `STRING`    | **exclude**: the quoted text is a glob pattern subtracted from the ingredient set. |

A conforming implementation MUST accept an `ingredients` line containing only excludes, only includes, or a mix of the two in any order. The semantic effect is the set difference `(union of includes) \ (union of excludes)`; ordering of items on the line does not affect the result.

The glob language itself — which characters are wildcards, whether `**` matches across directory separators, and so on — is not defined by §{recipes} and is resolved by the execution model (§{exec.phases}).

### Example 8.2.1

```cook
recipe lib
    ingredients "src/*.c" "include/*.h" !"src/lua.c" !"src/luac.c"
    cook "build/libmath.a" { ar rcs $<out> $<all> }
```

The ingredient set is every `.c` file in `src/` and every `.h` file in `include/`, minus `src/lua.c` and `src/luac.c`.

### Note 8.2.1 — overload rule

An `ingredients` argument is any iterable source. A quoted token (`"glob"` / `!"exclude"`) is a filesystem glob yielding **path** members; a bare identifier is a **probe key** yielding the probe's array members (§{cat.probes}, §22.5.9). The two forms are mutually exclusive within a recipe. Members of both forms bind as `$<in>` — path accessors (`$<in.stem>`, …) for path members, record-field access (`$<in.FIELD>`) for record members.

The `!` exclusion marker (glob form only) MUST immediately precede the opening `"` with no intervening whitespace: `! "foo"` is rejected; `!"foo"` is accepted. See parser tests `test_ingredients_with_excludes` and `test_ingredients_excludes_only`.

## 8.3. Step kinds — overview [#steps.overview]
A `recipe_body` is a sequence of `recipe_item`s (§{grammar-appendix.recipes}). Excluding the `ingredients_step` covered in §{steps.ingredients} and the structural `comment` / `NEWLINE` items, a conforming implementation MUST recognise exactly the following nine step kinds:

| § | Step kind | Introduced by | Phase |
|---|---|---|---|
| §{steps.cook-single}, §{steps.cook-multi} | `cook_step`           | keyword `cook` | register |
| §{steps.plate}      | `plate_step`          | keyword `plate` | register |
| §{steps.test}      | `test_step`           | keyword `test` | register |
| §{steps.lua}      | `inline_lua_line`     | `>>` line prefix (§{lexical.line-prefixes}) | register |
| §{steps.lua}      | `inline_lua_block`    | `>>{` line prefix (§{lexical.line-prefixes}) | register |
| §{steps.lua}      | `lua_line`            | `>` line prefix (§{lexical.line-prefixes}) | execute |
| §{steps.lua}      | `lua_block`           | `>{` line prefix (§{lexical.line-prefixes}) | execute |
| §{steps.shell}     | `shell_command`       | default Content line | execute |
| §{steps.shell}     | `interactive_command` | `@` as the first non-whitespace character of a Content line | execute |

The phase column classifies each step kind by **when its body is evaluated**: register-phase steps run during recipe registration (§{exec.two-phase}), execute-phase steps run when scheduled work units fire. A normative cross-section table appears in §{exec.phases.classification}.

A `Content` token inside a recipe body is classified into exactly one of the keyword-introduced kinds using the step-dispatch priority of §{grammar-appendix.steps} ("Step-dispatch priority"), reproduced here for convenience:

1. `ingredients` + separator → `ingredients_step` (§{steps.ingredients}).
2. `cook` + separator → `cook_step` (§{steps.cook-single}, §{steps.cook-multi}).
3. `plate` + separator → `plate_step` (§{steps.plate}).
4. `test` + separator → `test_step` (§{steps.test}).
5. First character is `@` and the remainder is non-empty → `interactive_command` (§{steps.shell}).
6. Otherwise → `shell_command` (§{steps.shell}).

The four Lua-prefix forms (`>>`, `>>{`, `>`, `>{`) are not subject to this cascade: they are selected at the lexical layer (§{lexical.line-prefixes}) and always produce `inline_lua_line`, `inline_lua_block`, `lua_line`, or `lua_block` steps respectively, regardless of what follows the prefix.

### Note 8.3.1

The "separator" after a step keyword is space, tab, or end of token. A Content line whose first token is `cooking` is a `shell_command`, not a malformed `cook_step`. See parser test `test_strip_keyword`.

## 8.4. `cook` — single-output form [#steps.cook-single]
A `cook_step` (§{grammar-appendix.steps}, `cook_step`) declares a build output. The single-output form has exactly one `STRING` output pattern before the optional body (CS-0099 removed the `using` introducer; the body opener follows the pattern directly):

```ebnf
cook "out" body?
```

The output is a `STRING` whose interior MAY contain substitution placeholders (e.g. `$<in.stem>`, `$<in>`, `$<out>`). The placeholder language is defined by §{phl} for substitution shapes and by §{xref} for cross-recipe references; §{recipes} treats placeholder text as opaque.

A conforming implementation MUST accept all three of the following forms:

| Form | `body` | Semantic |
|---|---|---|
| `cook "out"`                    | absent                  | declaration-only: announces the output without a build command (§{grammar-appendix.steps}, "Declaration-only cook step"). The build command is expected to be registered by preceding Lua code or supplied by a later amendment. |
| `cook "out" >{ ... }`           | `exec_lua_block`        | execute-phase Lua block; produces the build effect, with `inputs` / `outputs` / `input` / `output` bindings per §{lua.using-block-globals}. |
| `cook "out" { ... }`            | `shell_block`           | sequence of shell commands, one per non-blank line of the block (§{lexical.brace-blocks}, "Shell block line normalisation"). The block's textual placeholders (§{phl.cook-step}) are substituted at register time. |

After the output pattern(s), the body-dispatch order of §{grammar-appendix.steps} ("body dispatch order") applies: `>{` is tested before `{`. The first match wins; the bare-string form is no longer admitted (CS-0022), and the former `using` introducer is rejected with a migration diagnostic (CS-0099).

### Example 8.4.1

```cook
recipe lib
    ingredients "lib/*.c"
    cook "build/obj/$<in.stem>.o" { gcc -c $<in> -o $<out> }
    cook "build/libmath.a" { ar rcs $<out> $<all> }
```

The first `cook` is a one-to-one mapping from each ingredient to one object file. The second is a many-to-one mapping producing a single archive from the collected set. Both use the inline (one-line) `shell_block` body form.

### Example 8.4.2

```cook
recipe build
    ingredients "src/*.c"
    cook "bin/app"
    gcc src/main.c -o bin/app
```

The `cook "bin/app"` step is declaration-only. The following `shell_command` supplies the actual build command; the execution model (§{exec.phases}) is responsible for associating it with the declared output.

### Note 8.4.1

The `@` character inside a `shell_block` body is preserved verbatim — it is part of the block's shell text, not an interactive-step marker. The interactive-step dispatch of §{steps.shell} applies only to a Content line's first non-whitespace character, not to text nested inside a `cook` step's body.

### 8.4.1. Iteration mode [#steps.iteration-mode]
A `cook_step`'s **iteration mode** is determined by its output pattern list, before the body is consulted. Three modes are defined:

| Output pattern shape | Mode | Driver | Units produced |
|---|---|---|---|
| At least one output contains `$<in.ACCESSOR>` (own-input accessor) | **One-to-one over own inputs** | The step's resolved `ingredients` list (§{steps.ingredients}) | One per input |
| At least one output contains `$<lib.ACCESSOR>` (dep-driven accessor, §{xref.dep-driven}) | **One-to-one over dep outputs** | `lib`'s output list (§{xref.dep-recipe-output}) | One per `lib` output |
| All outputs are literal (no accessor placeholders) | **Many-to-one** (the "one" refers to *one unit*, not one output) | None — one unit runs with the full input list visible | Exactly one |
| The recipe carries an `ingredients <probe>` source (§{steps.ingredients}) | **One-to-one over data members** | The probe's resolved member list | One per member |

A conforming implementation MUST reject, with a load-time diagnostic, a `cook_step` whose output patterns mix iteration sources — that is, where two outputs select different drivers (one bears `$<in.ACCESSOR>` while another bears `$<libmath.ACCESSOR>`, or one bears `$<in.ACCESSOR>` while another is literal). All output patterns of a single `cook_step` MUST share one driver.

The iteration mode determines the number of work units produced; it is orthogonal to the `cook_step`'s declared output count. Each mode admits any number of declared outputs (one or more), as detailed in §{steps.cook-multi}.

When the recipe carries an `ingredients <probe>` source (§{steps.ingredients}), the `cook_step` is in **one-to-one over data members** mode regardless of its output-pattern accessors: the per-member binding is `item` (Lua) / `$<in>` / `$<in.ACCESSOR>` (where `ACCESSOR` selects a record field rather than a path component), and the path-accessor interpretation of `$<in.ACCESSOR>` does not apply (a data member has no ingredient path). A conforming implementation MUST NOT allow `$<in.ACCESSOR>` accessors to be interpreted as path accessors when the recipe is driven by an `ingredients <probe>` source.

### Example 8.4.1.1

```cook
recipe build
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" { gcc -c $<in> -o $<out> }    # one-to-one over own inputs
    cook "build/app" { gcc $<all> -o $<out> }              # many-to-one (literal output)
```

### Note 8.4.1.1

The CS-0022 unification removes `$<in>` from the body's contents as a separate iteration trigger. Iteration mode is decided exclusively by the output pattern list. A body that contains `$<in>` while the output pattern list is fully literal is rejected at load time — see §{phl.cook-step} for the diagnostic shape.

### 8.4.2. Lua-expression output [#steps.cook-lua-expr-output]

A `cook_step` MAY substitute a parenthesised Lua expression for the quoted output-path `STRING`:

```ebnf
cook_step ::= "cook" ( STRING+ | "(" LUA_EXPR ")" ) body? NEWLINE
```

A conforming implementation MUST accept the Lua-expression form under the following rules.

1. **Evaluation.** The expression is evaluated **once per ingredient**, at register time, against the Cookfile-scope Lua VM (the same VM used for chore-parameter Lua-expression defaults, §{chores.params}).
2. **Bindings.** The expression sees the name `input` in scope: the current ingredient's path, a Lua `string`. When the recipe is driven by an `ingredients <probe>` source (§{steps.ingredients}), the expression instead sees `item` bound to the current data member (a table for record members, a string/number/boolean for scalar members) rather than `input`. The expression MAY also reference declarative-region locals (top-level Lua bindings, `config` block writes, register-phase `inline_lua_line` / `inline_lua_block` locals visible to the recipe per §{steps.lua-execution-phase}). The standard library is available subject to the §{lua} sandbox in effect at register time.
3. **Return value.** The expression's return value MUST coerce to a non-empty string under the same coercion rules the Standard applies to chore Lua-expression defaults (§{chores.params}): `string` verbatim, `number` via Lua `tostring`, `boolean` to `"true"` or `"false"`; any other type — including `nil`, `table`, `function`, `thread`, `userdata` — is a register-phase diagnostic. The empty string is likewise a register-phase diagnostic.
4. **One-to-one only.** The Lua-expression form is admitted exclusively when the resulting `cook_step` is in **one-to-one over own inputs** mode or **one-to-one over data members** mode (§{steps.iteration-mode}); in the data-member case the expression sees `item` rather than `input` (rule 2). The expression is evaluated per iteration element, so dep-driven and many-to-one modes — neither of which iterates element-by-element — have no well-defined evaluation contract. A conforming implementation MUST reject a Lua-expression-output `cook_step` whose recipe has no `ingredients_step`, or whose iteration source resolves to the empty set, with a register-phase diagnostic.
5. **Mutual exclusion with quoted-string outputs.** Within a single `cook_step`, the output declaration is either (a) one or more `STRING` patterns with the `$<...>` sigil vocabulary of §{phl.cook-step}, or (b) exactly one parenthesised Lua expression. A conforming implementation MUST reject a `cook_step` that mixes the two forms (e.g. `cook "a.out" (input:gsub(...)) { ... }`) with a register-phase diagnostic naming the offending tokens.
6. **Body.** The body of a Lua-expression-output `cook_step` follows the same grammar as the quoted-string form (§{steps.cook-single}): the declaration-only form (no body), `exec_lua_block` (`>{ ... }`), and `shell_block` (`{ ... }`) are all admitted. The placeholder vocabulary inside the body (§{phl.cook-step}) is the one-to-one vocabulary: `$<in>`, `$<in.ACCESSOR>`, `$<out>`. Note that `$<out>` resolves to the Lua-expression's evaluated string for the current iteration; placeholders that index the output position list (`$<out_1>`, `$<out_2>`, …) have no use in this form because the form declares exactly one output per iteration.

### Example 8.4.2.1

```cook
recipe translate-fr
    ingredients "docs/en/**/*.md"
    cook (input:gsub("/en/", "/fr/")) >{
        fs.write_file(output, transform(fs.read_file(input)))
    }
```

The expression `input:gsub("/en/", "/fr/")` runs once per matched ingredient with `input` bound to that ingredient's path; the resulting string becomes the unit's `outputs[1]`. The `exec_lua_block` sees the `input` and `output` bindings per §{lua.using-block-globals}; `output` is the same string the Lua-expression produced.

### Example 8.4.2.2

```cook
recipe thumbnail
    ingredients "assets/photos/*.jpg"
    cook (input:gsub("assets/photos", "build/thumb")) { convert $<in> -resize 256x256 $<out> }
```

Mixes the Lua-expression output form with a `shell_block` body. The shell block's `$<in>` and `$<out>` placeholders resolve to the iteration's ingredient path and the Lua-expression's evaluated string, respectively.

### Note 8.4.2.1 — Cache key impact [#steps.cook-lua-expr-output.cache]

The expression's resolved string is recorded as the unit's `outputs[1]`, and so participates in the cache key via the output-path observable of §{exec.cache.abstract} #3. The expression source text itself is part of the recipe's parsed body and is therefore already folded into the unit's command-text / Lua-chunk fingerprint per §{exec.cache.abstract} #1 — no separate fingerprinting pass for the expression is required.

### Note 8.4.2.2 — Why one-to-one only [#steps.cook-lua-expr-output.rationale]

The Lua-expression form's defining feature is per-ingredient evaluation: each ingredient produces one unit whose output path is the expression evaluated with `input` bound to that ingredient. Dep-driven iteration (§{xref.dep-driven}) iterates over a *dependency's* output list, not over the step's own ingredient list, so `input` would have no canonical meaning. Many-to-one iteration runs the unit exactly once over the full input list, so per-ingredient evaluation has no addressable iteration. The restriction is therefore not a guard against unsafe semantics but a consequence of the bind-`input`-per-ingredient contract.

### Note 8.4.2.3 — Register-phase diagnostic shape [#steps.cook-lua-expr-output.diagnostic]

A conforming implementation rejecting the resolved output value MUST surface a register-phase diagnostic whose message identifies (a) the offending ingredient path and (b) the rule violated (non-string return, empty string). The reference implementation emits `cook (LUA_EXPR) returned non-string or empty value for input <PATH>` from the per-ingredient register-phase guard; an alternative implementation MAY phrase the message differently but MUST carry equivalent information.

### 8.4.3. Cook-step disposition [#steps.cook-disposition]

A `cook_step` MAY carry a *disposition* that declares how its artifact participates in the cache. This section defines the disposition surface and its parse-time rules; the cache-key *effect* of each disposition is normative in §{exec.cache} (Cache-trust v3, COOK-160). A disposition is expressed as one or more **decorator lines** immediately above a `cook` step, or as a **disposition block** wrapping several `cook` steps. There is NO project-default (column-0) seal set and NO `seal only` narrowing form.

There are four dispositions:

| disposition | shape | meaning (cache effect deferred to §{exec.cache}) |
|---|---|---|
| `seal` | `seal` `seal_ref`* | Asserts purity: the step's output is a pure function of its declared inputs, command, consulted env, and the named probe determinants. **Additive and stackable.** |
| `local` | `local` | Opt out of sharing — local cache only. |
| `pinned` | `pinned` | Designated-producer / fetch-only. |
| `record` | `record` | Marks an intrinsically non-reproducible artifact; byte-equivalence is waived at the cache decision (§{exec.cache.record}). |

A `seal_ref` is a bare `BARE_PROBE_KEY` — `IDENT (":" IDENT)?`, module-prefixable (e.g. `cc:toolchain`). The quoted `STRING` form of §{grammar-appendix.probe} is NOT admitted in a seal list, and a third contiguous `:IDENT` segment (`a:b:c`) MUST be rejected, exactly as for `probe_ref` in §{grammar-appendix.probe}.

#### Decorator lines [#steps.cook-disposition.lines]

```ebnf
disposition_line ::= disposition NEWLINE
disposition      ::= "seal" seal_ref* | "local" | "pinned" | "record"
```

A conforming implementation MUST enforce, at parse time:

1. **Decorator must precede a `cook`.** A run of one or more `disposition_line`s MUST be immediately followed by a `cook_step`. Comments and blank lines between the decorators and the `cook` are permitted and ignored. A decorator run terminated by any other step kind, or by the end of the recipe, MUST be rejected with a diagnostic naming the dangling decorator.
2. **`seal` is additive and stackable.** Multiple `seal` decorator lines union their `seal_ref`s onto the following `cook` step.
3. **Sorted and de-duplicated.** The effective `seal_ref` set MUST be sorted and de-duplicated before it is recorded; the set is order-insensitive.
4. **`local`/`pinned` mutual exclusion.** `local` and `pinned` MUST NOT both apply to one `cook` step (in either order, whether by decorator line or by block); the combination MUST be rejected.
5. **`local`/`pinned`/`record` take no refs.** A `seal_ref` after `local`, `pinned`, or `record` MUST be rejected. (A bare `local`/`pinned`/`record` keyword followed by a non-`{` token is therefore NOT a disposition line — see rule 7.)
6. **Empty `seal` is legal.** Bare `seal` with zero `seal_ref`s asserts a pure step with no ambient determinants; it is distinct from an unannotated `cook_step`.
7. **Non-reserved words.** `seal`, `local`, `pinned`, and `record` are NOT reserved words; they are recognised as dispositions only in this position (a recipe-body line whose first token is one of them, with a separator, immediately above a `cook` or as a block opener). A line whose first token is `local`/`pinned`/`record` followed by other non-`{` content is an ordinary `shell_command` (§{steps.shell}).

#### Disposition blocks [#steps.cook-disposition.block]

```ebnf
disposition_block      ::= disposition "{" NEWLINE disposition_block_item+ "}" NEWLINE
disposition_block_item ::= disposition_line* cook_step
                        | disposition_block
                        | comment
                        | NEWLINE
```

A `disposition` followed by `{` opens a block whose enclosed `cook` steps each inherit the block's disposition **additively**. A conforming implementation MUST enforce:

8. **Additive inheritance.** Each enclosed `cook` step's effective disposition is the additive union of every enclosing block's disposition and the step's own decorator lines: `seal` refs union, `record` propagates.
9. **Blocks nest.** A `disposition_block` MAY contain further `disposition_block`s; inheritance composes through the nesting.
10. **`local`/`pinned` override.** When a `local` or `pinned` disposition (a decorator line or an inner block) applies to a `cook` step nested under a `seal` block, the step is `local`/`pinned` and its inherited `seal` refs are dropped — it is no longer sealed/shared. `local` and `pinned` remain mutually exclusive (rule 4).
11. **`cook`-only body.** A `disposition_block` body consists only of `cook` steps, nested `disposition_block`s, decorator lines (each immediately preceding a `cook`), comments, and blank lines. It MUST contain at least one `cook_step` (directly or via a nested block); an empty block, a block whose decorators dangle with no following `cook`, or a block containing any other step kind (`ingredients`, `plate`, `test`, a shell or Lua step) MUST be rejected. The other step kinds carry no disposition and belong in the recipe body, not inside a disposition block.
12. **Termination.** A `disposition_block` is terminated by a recipe-body line that is exactly `}`. An unterminated block (end of recipe before the matching `}`) MUST be rejected with a diagnostic naming the opening line.
13. **Opener is line-final.** The opening `{` MUST end its physical line (the block body begins on the following line, per `disposition "{" NEWLINE`). A fully-inline single-line form such as `local { cook "s.o" { … } }` is NOT a `disposition_block`. Where a `disposition_block` is grammatically expected — inside another `disposition_block`'s body (rule 11) — such a line MUST be rejected. Elsewhere (at recipe-body top level) the line is not a disposition at all: its first token `local`/`pinned`/`record`/`seal` is non-reserved (rule 7), so the line falls through to `shell_command` per the step-dispatch priority.

### Example 8.4.3.1

Decorator lines stack additively above a `cook`:

```cook
recipe build
    seal host
    seal gpu driver
    record
    cook "x.o" { cc -c x.c -o x.o }
```

The effective seal set of `x.o` is `{driver, gpu, host}` (sorted, de-duplicated) and its `record` flag is set.

### Example 8.4.3.2

A `seal` block wraps several cooks; an inner `local` block overrides:

```cook
recipe build
    seal host {
        cook "a.o" { cc -c a.c -o a.o }
        cook "b.o" { cc -c b.c -o b.o }

        local { cook "scratch.o" { cc -c scratch.c -o scratch.o } }
    }
```

`a.o` and `b.o` are sealed on `host`; `scratch.o` is `local` and carries no seal refs.

### Note 8.4.3.1 — Conformance fixtures [#steps.cook-disposition.fixtures]

Positive: `standard/conformance/positive/cook-disposition-seal-line`, `cook-disposition-seal-stacked`, `cook-disposition-local-line`, `cook-disposition-pinned-line`, `cook-disposition-seal-block`, `cook-disposition-local-overrides-seal-block`, `cook-disposition-nested-block`.

Negative: `standard/conformance/negative/cook-disposition-dangling`, `cook-disposition-local-and-pinned`, `cook-disposition-seal-quoted-ref`, `cook-disposition-seal-triple-colon`, `cook-disposition-unterminated-block`.

## 8.5. `cook` — multi-output form [#steps.cook-multi]
A `cook_step` MAY declare two or more outputs by listing additional `STRING` patterns before the body:

```ebnf
cook "a" "b" ... body
```

When two or more outputs are present and a body is given, the body is one of the two block forms (§{steps.cook-single}): `exec_lua_block` (`>{ ... }`) or `shell_block` (`{ ... }`). The bare-string form was removed in CS-0022, and `cook "a" "b" "cmd"` parses as a three-output declaration-only step rather than a body-bearing one (§{grammar-appendix.steps}).

The declaration-only form of §{steps.cook-single} also applies to the multi-output case: a `cook_step` listing two or more outputs with no body is accepted and announces the outputs without providing an inline build command. Its build is assumed to be provided by preceding Lua registrations or a later amendment (App. A.4, "Declaration-only cook step").

### Example 8.5.1

```cook
recipe generate
    ingredients "src/*.rs"
    cook "staging/out.js" "staging/out.wasm" {
        ./fake-generator.sh
        mkdir -p staging
        cp pkg/out.js $<out_1>
        cp pkg/out.wasm $<out_2>
    }
```

One `cook` step declares two outputs produced together by a sequence of shell commands. The commands are the four non-blank lines of the shell block, in order (§{lexical.brace-blocks}). The placeholders `$<out_1>` and `$<out_2>` resolve to the unit's first and second declared outputs, in declaration order, per §{phl.cook-step}.

### Example 8.5.2

```cook
recipe build_lua
    ingredients "src/*.rs"
    cook "staging-lua/out.js" "staging-lua/out.wasm" >{
        os.execute("mkdir -p staging-lua")
        local sources = table.concat(inputs, " ")
        local js = io.open(outputs[1], "w")
        js:write("// generated from " .. sources .. "\n")
        js:close()
        local wasm = io.open(outputs[2], "w")
        wasm:write("BINARY\n")
        wasm:close()
    }
```

The same two outputs are produced by a Lua block. The `inputs` and `outputs` names visible to the block are established by the execution model (§{exec.phases}) and the Cook Lua API (§{lua.using-block-globals}); §{recipes} requires only that a block-form body be accepted.

### Note 8.5.1

The shape `cook "a" "b" using "cmd"` was rejected pre-CS-0022 with a "block body" diagnostic, and under CS-0022 with a bare-string-removal diagnostic. Since CS-0099 the `using` token itself is rejected at this position with the migration diagnostic of §{grammar-appendix.steps} ("`using` is rejected"); conformance fixture `cs0099-using-keyword-rejected` pins it.

## 8.6. `plate` step [#steps.plate]
A `plate_step` (§{grammar-appendix.steps}, `plate_step`) is **a `cook` step with no declared outputs**. It has a body — either a `shell_block` (§{grammar-appendix.steps}) or an `exec_lua_block` (§{grammar-appendix.steps}) — and no output pattern. It registers work units that perform side-effecting actions whose results do not contribute to the recipe's output list (which remains passthrough per §{xref.dep-recipe-output}).

```ebnf
plate_step ::= "plate" body NEWLINE
body       ::= shell_block | exec_lua_block
```

The body grammar is identical to a `cook_step`'s body grammar (§{steps.cook-single}). The placeholder vocabulary inside a `plate_step` body is specified in §{phl.plate-test} (shell-block bodies) and §{lua.using-block-globals} (Lua-block bodies), with the per-step-kind admission rules of §{steps.iteration-mode-plate-test}.

A `plate_step` admits no leading `STRING` and no trailing modifiers. The body follows the keyword directly.

### Example 8.6.1

```cook
recipe install
    ingredients "src/*.c"
    cook "build/bin/$<in.stem>" { cc $<in> -o $<out> }
    plate {
        install -d $PREFIX/bin
        install -m755 $<in> $PREFIX/bin/$<in.name>
    }
```

The `plate` step iterates over the preceding `cook` step's output list (one unit per binary), installing each binary into `$PREFIX/bin`.

### Example 8.6.2

```cook
recipe bundle
    ingredients "src/*.c"
    cook "build/bin/$<in.stem>" { cc $<in> -o $<out> }
    plate {
        tar -czf build/bundle.tgz $<all>
    }
```

The body uses `$<all>` and no `$<in>`; the step is many-to-one and registers exactly one unit, with `$<all>` substituted as the space-joined preceding-cook output list.

### Note 8.6.1

A `plate_step`'s body grammar matches a `cook_step`'s body. Since CS-0099 the two forms are fully parallel: in both, the body opener follows the step's header material directly. See rationale §{rationale.plate-test-no-output-decl}.

### Note 8.6.2 — Plate step bodies are NOT sandboxed [#steps.plate-not-sandboxed]

A `plate_step` body is the explicit "ship outside the project" surface and runs without the project-root sandbox imposed on `cook` / `test` / `chore` step bodies (§{lua.fs-sandbox}). A `plate` body MAY pass an absolute path outside the project root to `fs.*`, MAY call `os.execute` and `io.popen`, and MAY traverse upward via `..`. This is by construction: deploys, uploads, and other release-time side effects often touch paths the project does not own (e.g. `$HOME/.config/...`, `/var/...`, an S3 bucket).

A conforming implementation MUST run a `plate_step`'s body without applying the §{lua.fs-sandbox} confinement. Authors who want the confinement on a non-cacheable side-effect step SHOULD use a `chore` instead, which IS sandboxed (chores are non-cacheable but hermetic-by-intent — the sandbox catches accidental escapes without blocking the deliberate ones plate covers).

### 8.6.1. Plate/test iteration mode [#steps.iteration-mode-plate-test]

A `plate_step` (and a `test_step`, §{steps.test}) is in one of three iteration modes, determined entirely by its body's content:

| Shell-block body contains | Lua-block body references | Mode |
|---|---|---|
| `$<in>` or `$<in.ACCESSOR>` (no `$<all>`) | `input` (no `inputs`) | **one-to-one** over source |
| `$<all>` (no `$<in>` / `$<in.ACCESSOR>`) | `inputs` (no `input`) | **many-to-one** |
| neither | neither | **one-shot** |
| both | both | **error** (mixed iteration signal) |

The **iteration source** for a `plate_step` (and `test_step`) is the preceding `cook_step`'s output list (flattened across all units it produces), falling back to the recipe's resolved ingredients (§{steps.ingredients}) if no `cook_step` precedes it. When the recipe is driven by an `ingredients <probe>` source (§{steps.ingredients}), the iteration source is instead the resolved data-member list, and the per-member placeholders remain `$<in>` / `$<in.ACCESSOR>` (Lua: `item`), where `ACCESSOR` selects a record field rather than a path component; such a step is in **one-to-one over data members** mode. A step in **one-to-one** or **many-to-one** mode MUST have a non-empty source; a step in **one-shot** mode does not consult the source.

A conforming implementation MUST reject:
- A body that contains both an iteration-item placeholder/binding (`$<in>` / `$<in.ACCESSOR>` / `input`) and a batched-source placeholder/binding (`$<all>` / `inputs`).
- A `plate_step` or `test_step` in one-to-one or many-to-one mode whose source is empty.

The Lua identifier scan operates on the same lexical text the brace-balance lexer (§{lexical.brace-blocks}) operates on — strings, comments, and long strings are excluded; only free-identifier-position references count.

## 8.7. `test` step [#steps.test]
A `test_step` (§{grammar-appendix.steps}, `test_step`) has the same body grammar as a `plate_step` (§{steps.plate}) and additionally admits three trailing modifiers:

```ebnf
test_step      ::= "test" body test_modifiers NEWLINE
test_modifiers ::= ("as" STRING)? ("timeout" NUMBER)? "should_fail"?
```

The body and iteration-mode rules of §{steps.plate} apply verbatim. A `test_step` registers work units that report pass/fail rather than producing artifacts; iteration source, mode determination, placeholder vocabulary, and §{xref.string-substitution} cross-recipe substitution are identical to `plate_step`.

The optional modifiers MUST appear in the fixed canonical order shown above: `as` → `timeout` → `should_fail`. A conforming implementation MUST reject any out-of-order modifier sequence with a diagnostic that names the offending modifier and its canonical position.

| Form | Behaviour |
|---|---|
| `test BODY` | Run the body unit; expect success (no Lua error, no non-zero `cook.sh`/`cook.exec`); no time bound. |
| `test BODY as 'NAME'` | Same, with display name `NAME` in test reports. |
| `test BODY timeout N` | Run the body unit; expect success within `N` seconds. |
| `test BODY as 'NAME' timeout N` | Same, with display name and time bound. |
| `test BODY should_fail` | Run the body unit; expect failure (a Lua error or a non-zero `cook.sh`/`cook.exec` return); no time bound. |
| `test BODY timeout N should_fail` | Run the body unit; expect failure within `N` seconds. |
| `test BODY as 'NAME' timeout N should_fail` | Same, with display name. |

`NUMBER` is the `NUMBER` token class of §{lexical.numbers}. The upper bound on representable timeouts is implementation-defined.

The `as STRING` modifier sets the test's **display name** for reporting. Without it, the implementation assigns an auto-generated identity (typically `test#N` with an iteration discriminator suffix in non-one-shot modes). With it, the auto-generated identity is replaced by the supplied string; the iteration-discriminator suffix is still appended in iteration modes unless the substituted name already varies per iteration.

The `STRING` argument admits the same placeholder substitution (`$<IDENT>`) as the body (CS-0033). Substitution occurs at register-time codegen. The display name is metadata — it is **not** part of the test's cache fingerprint; renaming a test does not bust its cache.

**Codegen mapping.** When `as STRING` is present, the conforming code generator MUST emit a `name = <substituted-expr>` field in the `cook.add_test` call that registers the unit. The `<substituted-expr>` is the result of applying placeholder substitution to `STRING` under the same binding context as the step body (§{phl.plate-test}): `_test_in` for one-to-one mode, the collected-inputs table for many-to-one mode, and the empty string for one-shot mode. When `as STRING` is absent, the code generator MUST NOT emit a `name` field; the runner assigns an auto-generated identity.

The `as` modifier is **only valid on `test_step`**. A conforming implementation MUST reject `as` on `cook_step` or `plate_step` with a diagnostic identifying the modifier and the step kind.

### Example 8.7.1

```cook
recipe smoke
    ingredients "tests/*.c"
    cook "build/$<in.stem>" { cc $<in> -o $<out> }
    test { ./$<in> } timeout 60
```

Each test binary is run with a 60-second time bound, and the step expects each to exit zero (no `should_fail`).

### Example 8.7.2

```cook
recipe coverage
    ingredients "src/*.c"
    cook "build/bin/$<in.stem>.test" { cc $<in> -o $<out> }
    test >{
        for _, bin in ipairs(inputs) do
            cook.sh("./" .. bin .. " --check")
        end
    } timeout 300
```

The Lua body references `inputs` (no `input`); the step is many-to-one and runs once with all test binaries visible as `inputs`.

### Example 8.7.3

```cook
recipe roundtrip
    ingredients "src/*.txt"
    cook "build/$<in.stem>.out" { echo > $<out> }
    test { diff $<in> $<out> } as '$<in.stem>-roundtrip' timeout 10
```

The `as` modifier includes a placeholder; each iteration gets a unique display name (e.g., `foo-roundtrip`, `bar-roundtrip`).

### Note 8.7.1

`should_fail` is a bare word, not a quoted string. It is recognised only as the trailing modifier on a `test` line; elsewhere it has no syntactic meaning and is not a reserved word (§{lexical.keywords}). The same posture applies to `timeout` and `as`.

A `test_step` in one-to-one mode that runs a Lua body where `cook.sh`/`cook.exec` returns non-zero fails by the same rule as a Lua error; `should_fail` inverts the pass/fail decision.

## 8.8. Lua steps — line and block forms [#steps.lua]

The Cookfile language has two families of Lua step. They are distinguished by line prefix and by **execution phase** (§{exec.two-phase}):

| Surface form | Token (§{lexical.line-classification}) | Phase    | Region (§{recipes.region-rule}) |
|---|---|---|---|
| `>` *body* `\n`           | `LuaLine`              | execute  | imperative |
| `>{` *body* `}` `\n`      | `LuaBlockOpen`         | execute  | imperative |
| `>>` *body* `\n`          | `InlineLuaLine`        | register | declarative |
| `>>{` *body* `}` `\n`     | `InlineLuaBlockOpen`   | register | declarative |

### 8.8.1. Delimitation [#steps.lua-delimitation]

A `lua_line` step's body is the remainder of the line after `>` and an optional single space; the body MAY be empty (App. A.5, "`lua_line` MAY be empty"). An `inline_lua_line` step's body is the remainder of the line after `>>` and an optional single space; it too MAY be empty.

A `lua_block` step's body is the source text between the opening `>{` and the matching `}`, collected by the brace-balance algorithm of §{lexical.brace-blocks}. An `inline_lua_block` step's body is the source text between the opening `>>{` and the matching `}`, by the same algorithm. The closing `}` line is consumed but is not part of either block's body.

A conforming implementation MUST NOT reinterpret the Lua body text. Comments, strings, and other Lua constructs inside the body are the Lua runtime's concern, with the single exception that the brace-balance algorithm (§{lexical.brace-blocks}) MUST ignore braces appearing inside Lua line comments, double-quoted strings, single-quoted strings, and long strings (`[[...]]`) when locating the closing `}` of a Lua block (either form).

### 8.8.2. Execution phase [#steps.lua-execution-phase]

`inline_lua_line` and `inline_lua_block` are **register-phase** steps. Their body is evaluated during recipe registration in the recipe-body Lua function. Multiple inline-Lua steps in one recipe share register-phase scope: a `local` declared in one is visible to the next, including across the surface forms of `cook` / `plate` / `test` / `ingredients` (which compile to register-phase Lua API calls). This is the surface form a Cookfile uses to call Cook Lua API functions that record work — `cook.add_unit`, `cook.export`, `cook.step_group`, and so on — and to declare register-phase locals consumed by subsequent declarative steps.

`lua_line` and `lua_block` are **execute-phase** steps. Their bodies are not evaluated during registration. They are recorded as part of a **body unit** (§{exec.body-bundling}) whose `lua_code` payload runs on a worker VM during the execute phase, after the recipe's declarative work units have completed.

Prior to CS-0072, `module_call` was also register-phase: it desugared to `inline_lua_line` (single-line) or `inline_lua_block` (multi-line). As of CS-0072 the module_call dispatch is removed from the recipe-body cascade; module calls that previously appeared in recipe bodies MUST be moved to a top-level `register` block (§{decl.register}) or written as a top-level module_call (§{toplevel.module-call}).

### 8.8.3. Examples [#steps.lua-examples]

```cook
recipe build
    >> local std = cook.env.CXX_STD or "c++20"
    >>{
        cook.env.CXXFLAGS = "-std=" .. std
    }
    cook "build/app" >{
        cook.sh(string.format("%s %s -o %s",
            cook.env.CXX or "g++", table.concat(inputs, " "), output))
    }
    > print("build complete")
```

The `>>` line and `>>{ }` block run at register, sharing scope with the `cook` step's `>{ }` template-string formation. The `> print(...)` line runs at execute, after the cook step's unit completes. (The `>{ }` body is execute-phase Lua-with-bindings; see §{lua.using-block-globals}.)

```cook
recipe smoke
    cd build
    ./app --selftest
    > if not fs.exists("build/.smoke.ok") then error("smoke failed") end
```

The recipe has only an imperative region: two shell lines and one `>` line. The two shell lines coalesce into one `cook.sh("set -e\ncd build\n./app --selftest")` call, so `cd build` persists into `./app --selftest`. The `>` line then runs in the same Lua VM as the coalesced shell call. (Body bundling is specified in §{exec.body-bundling}.)

```cook
recipe legacy
    > print("start")
    >{
        if true then
            local t = {1, 2, 3}
            print(#t)
        end
    }
    >
```

The first step is a single-line execute-phase Lua statement. The second is an execute-phase Lua block whose body contains a literal `{` that does not prematurely close the block. The third is an empty `lua_line`. All three coalesce into one body unit (no `@` interruption); the empty third line contributes nothing to the chunk.

### Note 8.8.1

A Lua block (either form) opened on an indented line closes on the line where the running brace-depth first returns to zero, not on the first bare `}`. A line like `    local s = "}"` does not close the enclosing block because the `}` sits inside a double-quoted string (§{lexical.brace-blocks}). See parser tests `test_lua_block_brace_in_string` and `test_lua_block_brace_in_comment`.

### Note 8.8.2

The execute-phase API surface available inside a `lua_line` or `lua_block` is a strict subset of the register-phase API: register-only helpers (`cook.add_unit`, `cook.exec`, `cook.interactive`, `cook.step_group`, `cook.recipe`) MUST raise a Lua runtime error when called from execute-phase Lua. `cook.sh` is available in both phases (§{lua.cook-sh}). See §{rationale.execute-phase-api} for the rationale.

## 8.9. Shell steps — plain and interactive [#steps.shell]
A `shell_command` is any `Content` line inside a recipe body that is not classified as one of the step kinds earlier in the §{steps.overview} dispatch cascade. Its body is the trimmed source line; a conforming implementation MUST preserve the body text verbatim, including any internal quoting, variable references, and shell metacharacters.

An `interactive_command` is a `Content` line whose first non-whitespace character is `@` and whose remainder, after removing that `@`, is non-empty (§{grammar-appendix.steps}, `interactive_command`). A conforming implementation MUST:

1. Strip the leading `@` when recording the command body.
2. Mark the step as requiring interactive stdio (the precise semantics — TTY allocation, stdio inheritance — are defined by §{exec.interactive-drain}).
3. Reject an `@`-only line with no following command text.

The `@` marker is not a lexical prefix (see §{lexical.line-prefixes} and §{rationale.at-not-lexical}); it is interpreted only at step-dispatch time, and only inside a recipe body. A bare `@` outside a recipe body is a top-level parse error under §{toplevel.ordering}.

### Example 8.9.1

```cook
recipe play
    @mkdir -p .cook/scratch
    @printf 'a\nb\nc\n' | fzf > .cook/scratch/pick
    echo "picked: $(cat .cook/scratch/pick)"
```

The first two steps are interactive commands: the runtime inherits the user's terminal so that `fzf` can draw a full-screen prompt. The third is a plain `shell_command` whose output is captured as the execution model dictates.

### Note 8.9.1

`@` embedded inside a `cook` step's `{ ... }` shell-block body, or elsewhere inside a quoted string, is a literal `@` and does not trigger interactive-step dispatch. The `@` test runs only on the first non-whitespace character of a raw `Content` line. See parser test `test_at_in_cook_body_is_not_interactive`.

### Note 8.9.2

An empty interactive command — a line consisting only of `@` — is rejected with a diagnostic stating that `@` requires a command. See parser test `test_empty_interactive_step_errors`.

## 8.10. Cross-recipe per-member output accessor [#steps.cross-recipe-member]

Inside an `ingredients <probe>` fan-out body (§{steps.ingredients}), a placeholder of the form `$<NAME[]>` — where `NAME` is a recipe declared in the same Cookfile — resolves to that recipe's terminal output **for the current fan-out member**. It is the per-member counterpart of bare `$<NAME>` (which substitutes all terminal outputs of `NAME`, space-joined, and is unchanged by this feature).

`$<NAME[]>` performs three jobs simultaneously:

1. **Path substitution.** At register time the placeholder is replaced by the member-specific output path: `cook.dep_output_member("NAME", cook.member_to_string(item))`, where `item` is the fan-out loop variable.
2. **Recipe-level DAG ordering edge.** The compiler extracts `$<NAME[]>` as an ordering constraint identical to the edge produced by a bare `$<NAME>` reference: `NAME` must build before the referencing recipe.
3. **Per-member fingerprint fold.** The referenced output is folded into the current member's cache fingerprint, not the recipe's aggregate fingerprint. Editing a single upstream member (e.g. one `.blend` file changing `render`'s output for `"outro"`) rebuilds only the `mux` member that depends on it.

**Same-probe precondition (v1).** Consumer and producer must iterate the same probe. The `[]` accessor indexes into the producer's member map using the consumer's current `item` key; if the two recipes are driven by different probes, the key may not exist. Cross-probe joins via declared alignment keys are deferred to COOK-99 and MUST NOT be attempted in this version. A conforming implementation that detects a same-probe violation at register time SHOULD emit a diagnostic; otherwise the missing-key lookup produces a register-phase error at `cook.dep_output_member`.

**Fan-out body only.** `$<NAME[]>` MUST NOT appear outside an `ingredients <probe>` fan-out body. A conforming implementation MUST reject it with a load-time error if no `ingredients <probe>` driver is present in the recipe.

**Relation to `$<NAME>`.** Both forms record a recipe-level dependency on `NAME`. `$<NAME>` substitutes the full output list (useful in a many-to-one step that needs all of `NAME`'s artifacts); `$<NAME[]>` substitutes only the current-member output (useful in a fan-out step that must stay per-member). The two forms MAY coexist in the same Cookfile but MUST NOT coexist in the same step unless `NAME` differs between the two placeholders.

### Example 8.10.1

```cook
probe scenes
    produce as json { echo '[{"id":"intro"},{"id":"outro"}]' }

recipe render
    ingredients scenes
    cook "build/$<in.id>.silent.mp4" { ffmpeg -i $<in.id>.blend $<out> }

recipe tts
    ingredients scenes
    cook "build/$<in.id>.wav" { tts $<in.id> $<out> }

recipe mux
    ingredients scenes
    cook "build/$<in.id>.mp4" { ffmpeg -i $<render[]> -i $<tts[]> $<out> }
```

For member `{"id":"intro"}`, `mux` builds `build/intro.mp4` by combining `build/intro.silent.mp4` (from `render`) and `build/intro.wav` (from `tts`). For member `{"id":"outro"}` the same pattern repeats independently. Only the `outro` member's units are dirty when `outro.blend` changes — `intro.mp4` is unaffected.

### Note 8.10.1 — Conformance fixture

`standard/conformance/positive/recipe-member-accessor` pins this surface. The `parse.txt` golden confirms that `$<render[]>` and `$<tts[]>` are preserved verbatim in the `ShellBlock` strings at the parser layer; the codegen layer (CS-0098) lowers them to `cook.dep_output_member` calls.
