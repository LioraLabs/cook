# 8. Step kinds [#steps]
> **Normative.** This chapter defines every step kind that appears in a recipe body or a chore body, the dispatch cascade that classifies each `Content` token, and the iteration-mode rules for `cook` and `test` steps. The execute-phase behaviour of a body's compiled units is specified in §{exec.groups}. As of CS-0134 a recipe body is purely declarative: it carries no imperative region, so body bundling (§{exec.body-bundling}) — which coalesced a recipe's imperative-region steps into a body unit — has nothing left to bundle and is vacuous. A recipe's execute-phase work comes exclusively from the block body of a `cook` or `test` step (§{steps.cook-single}, §{steps.test}, §{steps.lua}), each of which is its own work unit. Chore bodies keep their imperative steps; their execution model is defined in §{chores} and is unaffected by §{exec.body-bundling}.

## 8.1. Step-dispatch cascade [#steps.dispatch]
Inside a `recipe_body`, each source line is first classified by the lexer (§{lexical.line-classification}) into one of `Comment`, `Blank`, `LuaBlockOpen`, `LuaLine`, or `Content`. Comments and blank lines are skipped. (The doubled-`>` prefixes that a pre-CS-0134 lexer would have surfaced as `InlineLuaLine` / `InlineLuaBlockOpen` are no longer usable tokens anywhere in the language: §{lexical.line-prefixes} rejects `>>` and `>>{` unconditionally at the lexical layer, CS-0134, Note 3.7.1.)

CS-0134 makes the recipe body **purely declarative**. A `LuaBlockOpen` (`>{`) or `LuaLine` (`>`) token MUST NOT appear as a standalone top-level line of a `recipe_body`; a conforming implementation MUST reject such a line (see the rejection rule below). These two tokens remain admitted only in **body position** — as the opener of a `cook`, `test`, or `probe` step's body, or inside a `chore` body, per §{lexical.line-prefixes} and §{steps.lua}.

A `Content` token inside a `recipe_body` is dispatched to exactly one `declarative_item` (App. A.3) by the priority cascade below (App. A.4, "Step-dispatch priority"). The first match wins; a conforming implementation MUST test the alternatives in this order:

1. Prefix `ingredients` + separator (space, tab, or end of token) → `ingredients_step`.
2. Prefix `seal` + separator → recipe-level `seal_step` (its `probe_ref`s contribute to the recipe seal baseline; §{steps.cook-disposition}).
3. Prefix `unseal` + separator → rejected as a recipe-level step (`unseal` is a trailing `cook_mods` modifier only; §{steps.cook-disposition}).
4. Prefix `cook` + separator → `cook_step`.
5. Prefix `test` + separator → `test_step`.
6. `BARE_IDENTIFIER "." BARE_IDENTIFIER` immediately followed by `(` — a **module call** — → `module_call`, register-phase Lua (§{toplevel.module-call}).
7. Otherwise → REJECTED.

Prior to CS-0072 (Cook Standard v0.9), this cascade already included the rule-6 module-call classification. CS-0072 removed it from the recipe-body cascade, forcing authors to reach for the `>>` (`inline_lua_line`) or `>>{` (`inline_lua_block`) prefix to get register-phase Lua inside a recipe body. **CS-0134 REINSTATES rule 6 above, reversing the CS-0072 recipe-body removal**, while simultaneously retiring `>>` / `>>{` from the language entirely (§{lexical.line-prefixes}). The equivalent top-level dispatch of §{toplevel.module-call} is unaffected by this reinstatement.

A `Content` token that matches none of rules 1–6 MUST be rejected. In particular, a conforming implementation MUST reject each of the following recipe-body constructs, with a diagnostic that identifies the offending line and directs the author to the migration path — a `cook "out" >{ ... }` / `cook "out" { ... }` (or `test`) BLOCK body, where `>{` / `>` survive as body openers (§{steps.lua}), or a `chore`, whose body retains the loose-shell and execute-Lua imperative surface (§{chores}):

- A loose/anonymous shell command (any `Content` line matching none of rules 1–6).
- A line whose first non-whitespace character is `@` (empty or not). The `@` interactive prefix and its `interactive_command` step kind are removed from the language entirely by CS-0134 (§{lexical.line-prefixes}); an `@` line no longer parses as any step kind anywhere.
- A standalone `>` or `>{` line outside body position.
- A standalone `>>` or `>>{` line — already rejected at the lexical layer (§{lexical.line-prefixes}) because CS-0134 removes both prefixes from the language outright; a recipe body is not a special case of that removal, it is simply one more place the prefixes no longer parse.

A `Content` token inside a `recipe_body` whose trimmed text begins with the keyword `register` followed by a separator (space, tab, or end-of-line) MUST be rejected with the diagnostic "`register` blocks are top-level only; move this outside the recipe body". The rejection applies regardless of indentation depth: an indented `register` cannot be a `RegisterHeader` (which is column-0 only by §{lexical.line-classification}) and is also not a permitted step in this position. A `Content` token whose first token is the bare identifier `register` *without* a separator and without the module-call shape of rule 6 (e.g. bare `register_foo`) falls through to rule 7 and is rejected under the general rule above. The reference parser fires the keyword-specific rejection in `parse_recipe`'s step-dispatch loop (informative).

A `recipe_body` MUST contain at most one `ingredients_step`. A conforming implementation MUST reject a recipe with two or more `ingredients_step` items (App. A.3).

Every remaining `recipe_body` item — `ingredients_step`, `cook_step`, `test_step`, `module_call`, `comment`, `Blank` — is register-phase (App. A.3); CS-0134 leaves no `imperative_item` in a recipe body's top-level line sequence. The pre-CS-0134 **region ordering rule** (App. A.3, "Region ordering rule"; §{recipes.region-rule}) required every `declarative_item` to precede the first `imperative_item` of the recipe body; under CS-0134 that constraint is satisfied vacuously for recipe bodies (there being no `imperative_item` left to order against) and imposes no further requirement here. It is likewise vacuous for `chore` bodies: CS-0134 removes the register-phase `inline_lua_line` / `inline_lua_block` (`>>` / `>>{`) steps from the language, leaving a chore body with only execute-phase (imperative) step kinds — `shell_command`, `lua_line`, `lua_block` — and therefore no declarative region to order against (§{chores.body}).

### Example 8.1.1

```cook
recipe build
    ingredients "src/*.c"
    cc.configure()
    cook "bin/$<in.stem>" { cc $<in> -o $<out> }
    test { ./bin/app }
```

The four body lines dispatch, in order, to: `ingredients_step`, `module_call` (rule 6 — register-phase; before CS-0134 this line would have needed to be written `>> cc.configure()`), `cook_step`, and `test_step`.

### Example 8.1.2

```cook
recipe build
    vim notes.txt
```

The body line is a loose shell command (a `Content` token matching none of rules 1–6) and MUST be rejected; the diagnostic directs the author to a `chore`, which retains the loose-shell imperative surface a recipe body no longer admits.

### Note 8.1.1

The `cook-lang` parser test `test_empty_interactive_step_errors` pinned rejection of a bare `@` prior to CS-0134. CS-0134 removes the `@` marker and the `interactive_command` step kind from the language entirely (§{lexical.line-prefixes}): every `@` line — bare or carrying a command — is now invalid syntax wherever it appears, and inside a recipe body it is rejected as a `Content` line matching none of rules 1–6.

The reference parser at `cli/crates/cook-lang` enforces the CS-0134 cascade in `parse_recipe`'s step-dispatch loop (informative); `parse_chore`'s step-dispatch loop keeps the `>`/`>{` steps and loose (implicitly-interactive) shell commands a chore body admits, but — like every other context — no longer accepts an `@` line (CS-0134).

## 8.2. Ingredients [#steps.ingredients]
A recipe body MAY contain at most one `ingredients_step` (§{grammar-appendix.recipes}, "At-most-one-ingredients rule"). A second `ingredients` line within the same recipe body MUST be rejected.

The `ingredients_step` is introduced by the keyword `ingredients` followed by one or more `ingredient` items (§{grammar-appendix.steps}, `ingredients_step`). Each item is either:

| Form | Meaning |
|---|---|
| `STRING`        | **include**: the quoted text is a glob pattern added to the ingredient set. |
| `!` `STRING`    | **exclude**: the quoted text is a glob pattern subtracted from the ingredient set. |

A conforming implementation MUST accept an `ingredients` line containing only excludes, only includes, or a mix of the two in any order. The semantic effect is the set difference `(union of includes) \ (union of excludes)`; ordering of items on the line does not affect the result.

The glob language itself — which characters are wildcards, whether `**` matches across directory separators, and so on — is not defined by §{recipes} and is resolved by the execution model (§{exec.phases}).

### Example 8.2.1

```cook
recipe lib
    ingredients "src/*.c" "include/*.h" !"src/lua.c" !"src/luac.c"
    cook "build/libmath.a" { ar rcs $<out> $<in> }
```

The ingredient set is every `.c` file in `src/` and every `.h` file in `include/`, minus `src/lua.c` and `src/luac.c`.

### Note 8.2.1 — overload rule

An `ingredients` argument is any iterable source. A quoted token (`"glob"` / `!"exclude"`) is a filesystem glob yielding **path** members; a bare identifier is a **probe key** yielding the probe's array members (§{cat.probes}, §22.5.9). The two forms are mutually exclusive within a recipe. Members of both forms bind as `$<in>` — path accessors (`$<in.stem>`, …) for path members, record-field access (`$<in.FIELD>`) for record members.

The `!` exclusion marker (glob form only) MUST immediately precede the opening `"` with no intervening whitespace: `! "foo"` is rejected; `!"foo"` is accepted. See parser tests `test_ingredients_with_excludes` and `test_ingredients_excludes_only`.

## 8.3. Step kinds — overview [#steps.overview]
A `recipe_body` is a sequence of `recipe_item`s (§{grammar-appendix.recipes}). Excluding the structural `comment` / `Blank` items, CS-0134 requires a conforming implementation to recognise exactly the following five step kinds — all of them register-phase:

| § | Step kind | Introduced by | Phase |
|---|---|---|---|
| §{steps.ingredients} | `ingredients_step` | keyword `ingredients` | register |
| §{steps.cook-disposition.seal} | `seal_step` | keyword `seal` | register |
| §{steps.cook-single}, §{steps.cook-multi} | `cook_step`           | keyword `cook` | register |
| §{steps.test}      | `test_step`           | keyword `test` | register |
| §{steps.dispatch}   | `module_call`         | `BARE_IDENTIFIER "." BARE_IDENTIFIER(…)` shape (§{toplevel.module-call}) | register |

The phase column classifies each step kind by **when its body is evaluated**: every recipe-body step kind runs during recipe registration (§{exec.two-phase}). This is a consequence of CS-0134's purely declarative recipe body — the execute-phase `lua_line` / `lua_block` kinds of §{steps.lua} exist only in **body position** (inside a `cook` / `test` / `probe` step's body, or inside a `chore` body), never as a standalone recipe-body item. A normative cross-section table appears in §{exec.phases.classification}.

A `Content` token inside a recipe body is classified into exactly one of the kinds above using the step-dispatch priority of §{grammar-appendix.steps} ("Step-dispatch priority"), reproduced here for convenience:

1. `ingredients` + separator → `ingredients_step` (§{steps.ingredients}).
2. `seal` + separator → recipe-level `seal_step` (§{steps.cook-disposition}).
3. `unseal` + separator → rejected as a recipe-level step (`unseal` is trailing-only; §{steps.cook-disposition}).
4. `cook` + separator → `cook_step` (§{steps.cook-single}, §{steps.cook-multi}).
5. `test` + separator → `test_step` (§{steps.test}).
6. `BARE_IDENTIFIER "." BARE_IDENTIFIER(…)` → `module_call`, register-phase (§{steps.dispatch}, §{toplevel.module-call}).
7. Otherwise → REJECTED (§{steps.dispatch}).

The `>{` / `>` prefix forms are not subject to this cascade at all: outside body position they are simply not admitted as a recipe-body line — a standalone `>{`/`>` line is rejected under rule 7 above (§{steps.dispatch}). Inside body position they are selected at the lexical layer (§{lexical.line-prefixes}) per §{steps.lua}, regardless of what follows the prefix. The doubled-`>` forms `>>` and `>>{` no longer exist in the language (CS-0134): they are rejected unconditionally at the lexical layer (§{lexical.line-prefixes}), not by this cascade.

### Note 8.3.1

The "separator" after a step keyword is space, tab, or end of token. A Content line whose first token is `cooking` matches none of rules 1–6 above — it is not `cook` followed by a separator, and it has no `IDENT.IDENT(` module-call shape — so it is REJECTED under rule 7, not silently accepted as a malformed `cook_step`. See parser test `test_strip_keyword`.

## 8.4. `cook` — single-output form [#steps.cook-single]
A `cook_step` (§{grammar-appendix.steps}, `cook_step`) declares a build output. The single-output form has exactly one `STRING` output pattern before the body (CS-0099 removed the `using` introducer; the body opener follows the pattern directly). The body is mandatory (CS-0133):

```ebnf
cook "out" body
```

The output is a `STRING` whose interior MAY contain substitution placeholders (e.g. `$<in.stem>`, `$<in>`, `$<out>`). The placeholder language is defined by §{phl} for substitution shapes and by §{xref} for cross-recipe references; §{recipes} treats placeholder text as opaque.

A conforming implementation MUST accept both of the following forms:

| Form | `body` | Semantic |
|---|---|---|
| `cook "out" >{ ... }`           | `exec_lua_block`        | execute-phase Lua block; produces the build effect, with `inputs` / `outputs` / `input` / `output` bindings per §{lua.using-block-globals}. |
| `cook "out" { ... }`            | `shell_block`           | sequence of shell commands, one per non-blank line of the block (§{lexical.brace-blocks}, "Shell block line normalisation"). The block's textual placeholders (§{phl.cook-step}) are substituted at register time. |

After the output pattern(s), the body-dispatch order of §{grammar-appendix.steps} ("body dispatch order") applies: `>{` is tested before `{`. The first match wins; the bare-string form is no longer admitted (CS-0022), and the former `using` introducer is rejected with a migration diagnostic (CS-0099).

### Example 8.4.1

```cook
recipe lib
    ingredients "lib/*.c"
    cook "build/obj/$<in.stem>.o" { gcc -c $<in> -o $<out> }
    cook "build/libmath.a" { ar rcs $<out> $<in> }
```

The first `cook` is a one-to-one mapping from each ingredient to one object file, where `$<in>` is the loop member. The second is a many-to-one mapping producing a single archive from the collected set, where `$<in>` is the unit's full input set, space-joined. Both use the inline (one-line) `shell_block` body form.

### Note 8.4.1

The `@` character inside a `shell_block` body is preserved verbatim — it is part of the block's shell text, not an interactive-step marker. The interactive-step dispatch of §{steps.shell} applies only to a Content line's first non-whitespace character, not to text nested inside a `cook` step's body.

### 8.4.1. Iteration mode [#steps.iteration-mode]
A `cook_step`'s **iteration mode** is determined by its output pattern list, before the body is consulted. Three modes are defined:

| Output pattern shape | Mode | Driver | Units produced |
|---|---|---|---|
| At least one output contains `$<in.ACCESSOR>` (own-input accessor) | **One-to-one over own inputs** | The step's resolved `ingredients` list (§{steps.ingredients}) | One per input |
| At least one output contains `$<lib.ACCESSOR>` (dep-driven accessor, §{xref.dep-driven}) | **One-to-one over dep outputs** | `lib`'s output list (§{xref.dep-recipe-output}) | One per `lib` output |
| All outputs are literal (no accessor placeholders) | **Many-to-one** (the "one" refers to *one unit*, not one output) | None — one unit runs with the full input list visible | Exactly one |
| The recipe carries an `ingredients <probe>` source (§{steps.ingredients}) | **One-to-one over data members** | The probe's resolved member list | One per member |

A conforming implementation MUST reject, with a load-time diagnostic, a `cook_step` whose output patterns mix iteration sources — that is, where two outputs select different drivers (one bears `$<in.ACCESSOR>` while another bears `$<libmath.ACCESSOR>`, or one bears `$<in.ACCESSOR>` while another is literal). All output patterns of a single `cook_step` MUST share one driver.

The iteration mode determines the number of work units produced; it is orthogonal to the `cook_step`'s declared output count. Each mode admits any number of declared outputs (one or more), as detailed in §{steps.cook-multi}.

When the recipe carries an `ingredients <probe>` source (§{steps.ingredients}), the `cook_step` is in **one-to-one over data members** mode regardless of its output-pattern accessors: the per-member binding is `item` (Lua) / `$<in>` / `$<in.ACCESSOR>` (where `ACCESSOR` selects a record field rather than a path component), and the path-accessor interpretation of `$<in.ACCESSOR>` does not apply (a data member has no ingredient path). A conforming implementation MUST NOT allow `$<in.ACCESSOR>` accessors to be interpreted as path accessors when the recipe is driven by an `ingredients <probe>` source.

### Example 8.4.1.1

```cook
recipe build
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" { gcc -c $<in> -o $<out> }    # one-to-one: $<in> is the loop member
    cook "build/app" { gcc $<in> -o $<out> }               # many-to-one: $<in> is the collected input set
```

### Note 8.4.1.1

The CS-0022 unification removes `$<in>` from the body's contents as a separate iteration trigger. Iteration mode is decided exclusively by the output pattern list, not by which placeholders appear in the body. `$<in>` itself is admitted in both one-to-one and many-to-one bodies (CS-0130, §{phl.cook-step}): only its *value* differs by mode — the loop member in one-to-one, the space-joined collected set in many-to-one.

### 8.4.2. Lua-expression output [#steps.cook-lua-expr-output]

A `cook_step` MAY substitute a parenthesised Lua expression for the quoted output-path `STRING`:

```ebnf
cook_step ::= "cook" ( STRING+ | "(" LUA_EXPR ")" ) body NEWLINE
```

A conforming implementation MUST accept the Lua-expression form under the following rules.

1. **Evaluation.** The expression is evaluated **once per ingredient**, at register time, against the Cookfile-scope Lua VM (the same VM used for chore-parameter Lua-expression defaults, §{chores.params}).
2. **Bindings.** The expression sees the name `input` in scope: the current ingredient's path, a Lua `string`. When the recipe is driven by an `ingredients <probe>` source (§{steps.ingredients}), the expression instead sees `item` bound to the current data member (a table for record members, a string/number/boolean for scalar members) rather than `input`. The expression MAY also reference register-phase locals visible to the recipe — top-level Lua bindings, `config` block writes, and locals established by a `register` block (§{decl.register}) or a recipe-body `module_call` (§{steps.dispatch}, rule 6). The standard library is available subject to the §{lua} sandbox in effect at register time.
3. **Return value.** The expression's return value MUST coerce to a non-empty string under the same coercion rules the Standard applies to chore Lua-expression defaults (§{chores.params}): `string` verbatim, `number` via Lua `tostring`, `boolean` to `"true"` or `"false"`; any other type — including `nil`, `table`, `function`, `thread`, `userdata` — is a register-phase diagnostic. The empty string is likewise a register-phase diagnostic.
4. **One-to-one only.** The Lua-expression form is admitted exclusively when the resulting `cook_step` is in **one-to-one over own inputs** mode or **one-to-one over data members** mode (§{steps.iteration-mode}); in the data-member case the expression sees `item` rather than `input` (rule 2). The expression is evaluated per iteration element, so dep-driven and many-to-one modes — neither of which iterates element-by-element — have no well-defined evaluation contract. A conforming implementation MUST reject a Lua-expression-output `cook_step` whose recipe has no `ingredients_step`, or whose iteration source resolves to the empty set, with a register-phase diagnostic.
5. **Mutual exclusion with quoted-string outputs.** Within a single `cook_step`, the output declaration is either (a) one or more `STRING` patterns with the `$<...>` sigil vocabulary of §{phl.cook-step}, or (b) exactly one parenthesised Lua expression. A conforming implementation MUST reject a `cook_step` that mixes the two forms (e.g. `cook "a.out" (input:gsub(...)) { ... }`) with a register-phase diagnostic naming the offending tokens.
6. **Body.** The body of a Lua-expression-output `cook_step` follows the same grammar as the quoted-string form (§{steps.cook-single}): `exec_lua_block` (`>{ ... }`) and `shell_block` (`{ ... }`) are both admitted; the body is mandatory (CS-0133). The placeholder vocabulary inside the body (§{phl.cook-step}) is the one-to-one vocabulary: `$<in>`, `$<in.ACCESSOR>`, `$<out>`. Note that `$<out>` resolves to the Lua-expression's evaluated string for the current iteration; placeholders that index the output position list (`$<out_1>`, `$<out_2>`, …) have no use in this form because the form declares exactly one output per iteration.

### Example 8.4.2.1

```cook
recipe translate-fr
    ingredients "docs/en/**/*.md"
    cook (input:gsub("/en/", "/fr/")) >{
        fs.write(output, transform(fs.read(input)))
    }
```

The expression `input:gsub("/en/", "/fr/")` runs once per matched ingredient with `input` bound to that ingredient's path; the resulting string becomes the unit's `outputs[1]`. The `exec_lua_block` sees the `input` and `output` bindings per §{lua.using-block-globals}; `output` is the same string the Lua-expression produced.

### Example 8.4.2.2

```cook
recipe thumbnail
    ingredients "assets/photos/*.jpg"
    cook (input:gsub("assets/photos", "build/thumb")) { convert $<in> -resize 256x256 $<out> }
```

Mixes the Lua-expression output form with a `shell_block` body. The shell block's `$<in>` and `$<out>` placeholders resolve to the iteration's ingredient path and the Lua-expression's evaluated string, respectively.

### Note 8.4.2.1 — Cache key impact [#steps.cook-lua-expr-output.cache]

The expression's resolved string is recorded as the unit's `outputs[1]`, and so participates in the cache key via the output-path observable of §{exec.cache.abstract} #3. The expression source text itself is part of the recipe's parsed body and is therefore already folded into the unit's command-text / Lua-chunk fingerprint per §{exec.cache.abstract} #1 — no separate fingerprinting pass for the expression is required.

### Note 8.4.2.2 — Why one-to-one only [#steps.cook-lua-expr-output.rationale]

The Lua-expression form's defining feature is per-ingredient evaluation: each ingredient produces one unit whose output path is the expression evaluated with `input` bound to that ingredient. Dep-driven iteration (§{xref.dep-driven}) iterates over a *dependency's* output list, not over the step's own ingredient list, so `input` would have no canonical meaning. Many-to-one iteration runs the unit exactly once over the full input list, so per-ingredient evaluation has no addressable iteration. The restriction is therefore not a guard against unsafe semantics but a consequence of the bind-`input`-per-ingredient contract.

### Note 8.4.2.3 — Register-phase diagnostic shape [#steps.cook-lua-expr-output.diagnostic]

A conforming implementation rejecting the resolved output value MUST surface a register-phase diagnostic whose message identifies (a) the offending ingredient path and (b) the rule violated (non-string return, empty string). The reference implementation emits `cook (LUA_EXPR) returned non-string or empty value for input <PATH>` from the per-ingredient register-phase guard; an alternative implementation MAY phrase the message differently but MUST carry equivalent information.

### 8.4.3. Cook-step disposition [#steps.cook-disposition]

**The `cook` step IS the cacheable unit.** One step → one content-addressed key → one disposition, however many output files it emits. The whole surface teaches a single mental model:

> **Body = what it's made of. `cook` = one unit (≥1 output). Tail = what to do with that unit.**

A unit's cache participation is declared by two surfaces — *inputs* and *output-facts* — which split along a seam that was already there: `seal`/`unseal` name a determinant **input**, and `local`/`pinned`/`nondet` state a **fact about the unit's output**. They are therefore the different kinds they are.

| Concept | Written as | Where | Scope |
|---|---|---|---|
| File inputs | `ingredients "…"` | body line | recipe-wide |
| Probe inputs | `seal <probes>` | body line | recipe-wide |
| The unit | `cook "out1" "out2" { … }` | body line | one key, ≥1 output |
| Per-unit identity tweak | `… seal/unseal <probes>` | tail of the `cook` | that unit |
| Disposition | `… local / pinned / nondet` | tail of the `cook` | that unit |

**One rule:** a body line names a recipe-wide input; a tail tunes one `cook` unit. The cache-key *effect* of each is normative in §{exec.cache} (Cache-trust v3); this section defines the surface and its parse-time rules.

A `probe_ref` (used by `seal`/`unseal` everywhere) is a bare `BARE_PROBE_KEY` — `IDENT (":" IDENT)?`, module-prefixable (e.g. `cc:toolchain`). The quoted `STRING` form of §{grammar-appendix.probe} is NOT admitted, and a third contiguous `:IDENT` segment (`a:b:c`) MUST be rejected, exactly as for `probe_ref` in §{grammar-appendix.probe}. A `seal`/`unseal` ref is an *invalidate-only* determinant — it is never interpolated; there is no `$<seal:…>`. To *consume* a probe's members as data, use `ingredients <probe>` / `for_each` (§{steps.ingredients}), not `seal`.

**A seal/unseal ref is ALWAYS a probe name.** There is no `env.X` / `tool.X` ambient-determinant ref form — neither a way to seal an environment variable inline nor a way to seal a tool by file path. `probe_ref` is therefore complete as written above; it needs no extension. The referenced probe MAY be any probe — a shell or Lua probe, a `tools` probe, or an `envs` probe (§{cat.probes.decl}) — but the surface is uniform: every seal ref is just a probe name. The two ambient-determinant paths fold through that one surface:

- **Unconsumed environment.** An environment variable that no unit consumes (so §{exec.cache.lua-env-reads} does not auto-fold it) is folded by declaring an `envs` probe over the variable(s) and sealing that probe by name — `seal <envsprobe>`. The `envs` probe's value is a cached, shareable fingerprint over the named variables (§{cat.probes.decl}). Environment a unit *does* consume continues to auto-fold via §{exec.cache.lua-env-reads}, unchanged.
- **Tool determinants.** A toolchain is folded by sealing a `tools` probe (§{cat.probes.decl}) by name. The asymmetry with environment is deliberate: tools want cached, shared fingerprinting — exactly what a probe provides — whereas an `env.X` inline ref would have provided neither grouping, naming, nor shareability. Both ambient kinds therefore resolve to the same uniform "seal a named probe" surface, with no `env.X` / `tool.X` special forms to parse.

#### `seal` / `unseal` steps [#steps.cook-disposition.seal]

`seal` and `unseal` are recipe-body **steps** — a second input stream, sibling of `ingredients`, declaring *determinant* probe inputs (probe values folded into the unit cache key) alongside `ingredients`' *file* inputs. A conforming implementation MUST enforce, at parse time:

1. **Recipe-level `seal` is the baseline.** A `seal <probes>` step adds its `probe_ref`s to the recipe's determinant baseline, which applies to **every** `cook` unit in the recipe. Bare `seal` (zero refs) MUST be rejected — seal/unseal are always explicitly named.
2. **Additive and order-independent.** Multiple recipe-level `seal` steps union their refs. The baseline applies to every cook regardless of textual position (it is *declarative*, not positional): `seal` after a `cook` still applies to it. The effective set is sorted and de-duplicated.
3. **Recipe-level `unseal` is rejected.** `unseal` is a trailing modifier on a `cook` step only (rule 5). The recipe is the outermost seal scope; there is nothing inherited to release, so a recipe-level `unseal` would read as a no-op and MUST be rejected.

**Recipe boundaries break seals implicitly.** The recipe is the outermost seal scope; nothing propagates across recipe dependency edges. A downstream recipe folds an upstream's determinants in transitively via the upstream's OUTPUT CONTENT hashes (consumed as `ingredients`), so re-sealing downstream would double-count. There are exactly two seal tiers — recipe baseline and per-unit tail — and no config-default inheritance tier.

> **Note 8.4.3.1 — boundary of a future blanket seal (informative).** The seal surface is *named-only*: every determinant a unit keys on is an explicitly named probe ref. No absolute / blanket form (a hypothetical "seal everything visible") exists, and a bare `seal` is rejected (rule 1). This note fixes the boundary such a form would have to draw, so that introducing one later cannot silently change today's keys. A blanket seal would have to enumerate a *universe* comprising (a) the unit's *visible-but-unconsumed* probes — probes declared in the Cookfile that the unit neither consumes as `ingredients`/`for_each` data nor names in its effective seal set — plus (b) ambient determinants not modelled as probes at all (consulted environment beyond §{exec.cache.lua-env-reads}, host/machine identity, tool identity). The current surface declares no such universe: a unit keys on exactly the probes named in its effective seal set and nothing more (§{exec.cache.single-key}). Any future blanket form is a NEW determinant set, not a re-reading of the named-only one, and MUST be introduced under its own change ID with its own key-version bump.

#### Trailing modifiers (`cook_mods`) [#steps.cook-disposition.tail]

```ebnf
cook_mods ::= ("seal" probe_ref+ | "unseal" probe_ref+)* share_mod?
share_mod ::= "local" | "pinned" | "nondet"
```

The tail of a `cook` step tunes that one unit. The `cook_mods` are line-final: they follow the body's closing `}` on the same physical line (a cook step always has a body, CS-0133). Tokens on a subsequent line are not part of the `cook_mods` (a `}` on its own line followed by a `seal …` line dispatches the latter as a recipe-level `seal` step). A conforming implementation MUST enforce:

4. **Per-unit seal/unseal adjust the baseline.** A unit's *effective seal set* is `(baseline ∪ trailing seals) − trailing unseals`, sorted and de-duplicated. Bare trailing `seal`/`unseal` (no refs) MUST be rejected.
5. **`unseal` is trailing-only.** `unseal <probes>` removes refs from this unit's effective set; it is valid only here (not as a recipe-level step, rule 3).
6. **`share_mod` is a single optional slot.** At most one of `local` / `pinned` / `nondet` MAY appear, and it MUST be the last token — nothing may follow it. The three are mutually exclusive by construction (a single slot). `local` opts out of sharing (§{exec.cache.sharing}); `pinned` is designated-producer / fetch-only (§{exec.cache.sharing}); `nondet` declares the output intrinsically non-reproducible, waiving byte-equivalence (§{exec.cache.record}).
7. **`record` is renamed `nondet`.** The v3 `record` disposition is renamed `nondet` (it states a *fact* — the output is nondeterministic — rather than an imperative policy). The `record` keyword in `cook_mods` MUST be rejected with a migration diagnostic.
8. **Non-reserved words.** `seal`, `unseal`, `local`, `pinned`, and `nondet` are NOT reserved words. `seal`/`unseal` are recognised as steps only at a recipe-body line position; `local`/`pinned`/`nondet` only in the trailing `cook_mods` position. A recipe-body line whose first token is `local`/`pinned`/`nondet` (followed by other content or alone) is an ordinary `shell_command` (§{steps.shell}).

### Example 8.4.3.1 — the whole surface once

The probes `toolchain`, `flags`, and `rev` are declared elsewhere (§{cat.probes}):

```cook
recipe build
    ingredients "src/*.c"          # ┐ recipe-wide inputs
    seal toolchain flags           # ┘ → identity
    cook "bin/app" "bin/app.map" { cc … -Wl,-Map=bin/app.map }  # one unit, two files, one key
    cook "bin/app.dbg" { cc -g … } local                       # separate unit, kept local
    cook "bin/plugin"  { cc … } seal rev                       # also keyed on the git rev
    cook "gen/ver.h"   { … } nondet                            # non-reproducible → reuse stored copy
```

`bin/app`/`bin/app.map` form one unit (one key, two outputs) sealed on `toolchain` and `flags`; `bin/app.dbg` is the same baseline but kept `local`; `bin/plugin` additionally seals on `rev`; `gen/ver.h` is `nondet`. A multi-output *unit* carries one tail; several *separate* `cook` steps wanting the same disposition repeat the word (an accepted rough edge — split the recipe, or add a recipe-level default later without breaking the grammar).

### Example 8.4.3.2 — per-unit unseal

```cook
recipe build
    seal a b
    cook "x.o" { cc -c x.c -o x.o } unseal a seal c
```

The effective seal set of `x.o` is `{b, c}`: baseline `{a, b}`, plus trailing `seal c`, minus trailing `unseal a`.

### Note 8.4.3.2 — Conformance fixtures [#steps.cook-disposition.fixtures]

Positive: `standard/conformance/positive/cook-disposition-seal-line`, `cook-disposition-seal-stacked`, `cook-disposition-local-line`, `cook-disposition-pinned-line`, `cook-disposition-nondet`, `cook-disposition-trailing-seal-unseal`, `cook-disposition-seal-order-independent`, `cook-disposition-seal-multi-cook-baseline`, `cook-disposition-unseal-one-unit-only`, `cook-disposition-seal-envs-probe`, `cook-disposition-seal-tools-probe`.

Negative: `standard/conformance/negative/cook-disposition-local-and-pinned`, `cook-disposition-double-share-mod`, `cook-disposition-bare-seal`, `cook-disposition-recipe-unseal`, `cook-disposition-record-renamed`, `cook-disposition-seal-quoted-ref`, `cook-disposition-seal-triple-colon`.

## 8.5. `cook` — multi-output form [#steps.cook-multi]
A `cook_step` MAY declare two or more outputs by listing additional `STRING` patterns before the body:

```ebnf
cook "a" "b" ... body
```

When two or more outputs are present and a body is given, the body is one of the two block forms (§{steps.cook-single}): `exec_lua_block` (`>{ ... }`) or `shell_block` (`{ ... }`). The bare-string form was removed in CS-0022, and `cook "a" "b" "cmd"` parses as a three-output cook with no body — a trailing `STRING` joins the output-pattern list rather than being read as a body — which is then rejected because a body is mandatory (CS-0133).

Like the single-output form, a multi-output `cook_step` MUST have a body (CS-0133); `cook "a" "b"` with no body is a parse error.

### Example 8.5.1

```cook
recipe generate
    ingredients "src/*.rs"
    cook "staging/out.js" "staging/out.wasm" {
        ./fake-generator.sh
        mkdir -p staging
        cp pkg/out.js $<out_1>
        cp pkg/out.wasm $<out_2>
    }
```

One `cook` step declares two outputs produced together by a sequence of shell commands. The commands are the four non-blank lines of the shell block, in order (§{lexical.brace-blocks}). The placeholders `$<out_1>` and `$<out_2>` resolve to the unit's first and second declared outputs, in declaration order, per §{phl.cook-step}.

### Example 8.5.2

```cook
recipe build_lua
    ingredients "src/*.rs"
    cook "staging-lua/out.js" "staging-lua/out.wasm" >{
        os.execute("mkdir -p staging-lua")
        local sources = table.concat(inputs, " ")
        local js = io.open(outputs[1], "w")
        js:write("// generated from " .. sources .. "\n")
        js:close()
        local wasm = io.open(outputs[2], "w")
        wasm:write("BINARY\n")
        wasm:close()
    }
```

The same two outputs are produced by a Lua block. The `inputs` and `outputs` names visible to the block are established by the execution model (§{exec.phases}) and the Cook Lua API (§{lua.using-block-globals}); §{recipes} requires only that a block-form body be accepted.

### Note 8.5.1

The shape `cook "a" "b" using "cmd"` was rejected pre-CS-0022 with a "block body" diagnostic, and under CS-0022 with a bare-string-removal diagnostic. Since CS-0099 the `using` token itself is rejected at this position with the migration diagnostic of §{grammar-appendix.steps} ("`using` is rejected"); conformance fixture `cs0099-using-keyword-rejected` pins it.

## 8.6. `test` step [#steps.test]
A `test_step` (§{grammar-appendix.steps}, `test_step`) registers a work unit that reports pass/fail rather than producing artifacts. It has a body — either a `shell_block` or an `exec_lua_block` (§{grammar-appendix.steps}) — and admits no leading `STRING` and no trailing modifiers. The body follows the keyword directly.

```ebnf
test_step ::= "test" body NEWLINE
body      ::= shell_block | exec_lua_block
```

A conforming implementation MUST accept both `test { ... }` (a `shell_block` body) and `test >{ ... }` (an `exec_lua_block` body), and MUST reject any leading `STRING` or trailing token after the body.

Iteration is decided by a single signal — whether the body references the iteration item (§{steps.iteration-mode-plate-test}). A shell body containing `$<in>` / `$<in.ACCESSOR>`, or a Lua body binding `input`, puts the step in **one-to-one** mode: a fan-out over the iteration source with one cached pass/fail unit per item. A body that references no iteration item is a **single unit** — one node — whose caching is derived from the recipe's declared file source: when the recipe supplies a source (`ingredients`, or an upstream `cook`'s outputs) the unit is keyed on the whole source set (the engine's `ManyToOne`); when the recipe supplies no source at all the unit runs uncached on every invocation (the engine's `OneShot`). Caching is **derived**, not declared — it follows from whatever file source the recipe provides, exactly as for a `cook_step`; there is no test-specific cache surface to configure.

This is the deliberate parallel with `cook` (§{steps.iteration-mode}): the iteration accessor lives wherever the step's defining payload is — the output pattern for `cook`, the body for `test`. CS-0022's hazard (a body `$<in>` conflicting with a literal output pattern) cannot arise for `test`, which has no output pattern for the body signal to conflict with.

**Trailing content after the body is rejected.** The body follows the `test` keyword directly and takes no trailing modifiers; a conforming implementation MUST reject any token after the body's closing `}`. When the rejected trailing token is `should_fail`, `timeout`, or `as`, a conforming implementation SHOULD emit a did-you-mean diagnostic naming the removed modifier and the v1.0 migration path:

- `should_fail`: "`should_fail` was removed in v1.0; invert the check in the body, e.g. `test { ! grep -q foo $<in> }`".
- `timeout`: "`timeout` was removed in v1.0".
- `as`: "`as` was removed in v1.0; tests auto-name".

None of `should_fail`, `timeout`, or `as` were ever reserved words (§{lexical.keywords}); they simply stop being recognised in this trailing position.

### 8.6.1. Test iteration mode [#steps.iteration-mode-plate-test]

A `test_step` has two author-facing modes, selected by whether its body references the iteration item. The single-unit mode maps onto one of the engine's two batch dispositions according to whether the recipe supplies a source:

| Body references the iteration item? | Recipe source | Mode |
|---|---|---|
| `$<in>` / `$<in.ACCESSOR>` (shell) or `input` (Lua) present | non-empty (required) | **one-to-one** — one cached pass/fail unit per item |
| absent | a source is declared (non-empty) | **single unit**, keyed on the whole source set (engine `ManyToOne`) |
| absent | no source declared | **single unit**, uncached, runs every invocation (engine `OneShot`) |

A shell body has no batched-set placeholder (App. E CS-0130 removed the former all-inputs shell placeholder), so a shell single-unit body runs its command once over the unit. A Lua single-unit body MAY bind `inputs` to observe the collected source set. A Lua body MUST NOT reference both `input` and `inputs`.

The **iteration source** for a `test_step` is the preceding `cook_step`'s output list (flattened across all units it produces), falling back to the recipe's resolved ingredients (§{steps.ingredients}) if no `cook_step` precedes it. When the recipe is driven by an `ingredients <probe>` source (§{steps.ingredients}), the iteration source is instead the resolved data-member list, and the per-member placeholders remain `$<in>` / `$<in.ACCESSOR>` (Lua: `item`), where `ACCESSOR` selects a record field rather than a path component; such a step is in **one-to-one over data members** mode.

A conforming implementation MUST reject:
- A Lua-block body that references both an iteration-item binding (`input`) and a batched-source binding (`inputs`).
- A `test_step` whose mode consults a declared source — one-to-one, or single-unit with a source — but whose source resolves empty.

A `test_step` with no declared source at all — no preceding `cook`, no `ingredients` — is admitted and runs uncached, always (the engine's one-shot disposition); it simply does not consult the source.

The Lua identifier scan operates on the same lexical text the brace-balance lexer (§{lexical.brace-blocks}) operates on — strings, comments, and long strings are excluded; only free-identifier-position references count.

### Example 8.6.1

```cook
recipe smoke
    ingredients "tests/*.c"
    cook "build/$<in.stem>" { cc $<in> -o $<out> }
    test { ./$<in> }
```

Each test binary from the `cook` step's output is run in turn; the step is one-to-one and expects each invocation to exit zero.

### Example 8.6.2

```cook
recipe check
    test { cargo test }
```

No preceding `cook` and no `ingredients`: the step has no source at all, so it is one-shot and runs uncached on every invocation of `check`.

### Example 8.6.3

```cook
recipe coverage
    ingredients "src/*.c"
    cook "build/bin/$<in.stem>.test" { cc $<in> -o $<out> }
    test >{
        for _, bin in ipairs(inputs) do
            cook.sh("./" .. bin .. " --check")
        end
    }
```

The Lua body references `inputs` (no `input`); the recipe supplies a source (the `cook` step's outputs), so the step is a single unit keyed on the whole source set (engine `ManyToOne`), run once with all test binaries visible as `inputs`.

### Note 8.6.1

A `test_step` running a Lua body where `cook.sh`/`cook.exec` returns non-zero fails the test by the same rule as an uncaught Lua error.

## 8.7. Lua steps — line and block forms [#steps.lua]

The Cookfile language has one family of Lua step, the execute-phase `>` / `>{` forms, distinguished from each other only by whether they open a nested block. As of CS-0134, both forms are admitted **only in body position** — as the opener of a `cook`, `test`, or `probe` step's body, or as a standalone step inside a `chore` body — and MUST NOT appear as a standalone top-level line of a `recipe_body` (§{steps.dispatch}).

| Surface form | Token (§{lexical.line-classification}) | Phase    | Admitted positions |
|---|---|---|---|
| `>` *body* `\n`           | `LuaLine`              | execute  | `cook`/`test`/`probe` body opener; standalone inside a chore body |
| `>{` *body* `}` `\n`      | `LuaBlockOpen`         | execute  | `cook`/`test`/`probe` body opener; standalone inside a chore body |

### Note 8.7.0 — Removal of the doubled-`>` forms (CS-0134)

Prior to CS-0134, a second family existed: the register-phase doubled-`>` prefixes `>>` (`InlineLuaLine`) and `>>{` (`InlineLuaBlockOpen`), admitted as standalone declarative-region steps inside a recipe body. CS-0134 REMOVED both prefixes from the language entirely — they are rejected unconditionally at the lexical layer (§{lexical.line-prefixes}, Note 3.7.1), not merely repositioned. A Cookfile that needs register-phase Lua inside a recipe body now writes a bare module call instead (§{steps.dispatch}, rule 6): `ns.fn(...)` where a pre-CS-0134 Cookfile would have written `>> ns.fn(...)`.

### 8.7.1. Delimitation [#steps.lua-delimitation]

A `lua_line` step's body is the remainder of the line after `>` and an optional single space; the body MAY be empty (App. A.5, "`lua_line` MAY be empty").

A `lua_block` step's body is the source text between the opening `>{` and the matching `}`, collected by the brace-balance algorithm of §{lexical.brace-blocks}. The closing `}` line is consumed but is not part of the block's body.

A conforming implementation MUST NOT reinterpret the Lua body text. Comments, strings, and other Lua constructs inside the body are the Lua runtime's concern, with the single exception that the brace-balance algorithm (§{lexical.brace-blocks}) MUST ignore braces appearing inside Lua line comments, double-quoted strings, single-quoted strings, and long strings (`[[...]]`) when locating the closing `}` of a Lua block.

### 8.7.2. Execution phase [#steps.lua-execution-phase]

`lua_line` and `lua_block` are **execute-phase** steps in both of their admitted positions. Their bodies are never evaluated during registration.

When a `lua_line` or `lua_block` opens a `cook` / `test` / `probe` step's body, its body IS that step's registered work unit (§{steps.cook-single}, §{steps.test}), evaluated execute-phase with the `inputs` / `outputs` / `input` / `output` bindings of §{lua.using-block-globals}.

When a `lua_line` or `lua_block` occurs as a standalone step inside a `chore` body, its `lua_code` payload runs during the execute phase on the chore's single drain thread; the execution model for chore-body imperative steps — which is the single-drain rule of §{chores.default-interactive}, NOT the (CS-0134-vacuous) recipe body-bundling of §{exec.body-bundling} — is specified in §{chores}.

Register-phase Lua inside a recipe body — the role the removed `>>` / `>>{` prefixes used to fill — is now written as a bare **module call** (§{steps.dispatch}, rule 6): `ns.fn(...)`, dispatched and evaluated in the recipe-registration Lua scope alongside `ingredients` / `cook` / `test` (§{steps.dispatch}, §{toplevel.module-call}).

### 8.7.3. Examples [#steps.lua-examples]

```cook
recipe build
    cc.configure()
    cook "build/app" >{
        cook.sh(string.format("%s %s -o %s",
            cook.env.CXX or "g++", table.concat(inputs, " "), output))
    }
```

The bare `cc.configure()` line is a `module_call` (§{steps.dispatch}, rule 6), register-phase — the role a pre-CS-0134 Cookfile would have filled with `>> cc.configure()`. The `cook` step's `>{ }` body is a `lua_block` in body position: execute-phase Lua-with-bindings, run after the recipe's other register-phase steps have been recorded; see §{lua.using-block-globals}.

```cook
chore smoke
    cd build
    ./app --selftest
    > if not fs.exists("build/.smoke.ok") then error("smoke failed") end
```

Standalone `lua_line` / `lua_block` steps survive inside a `chore` body, which is unaffected by CS-0134's recipe-body restriction (§{chores}). The `>` line runs after the two preceding shell steps, in the chore's execute-phase model.

```cook
chore legacy
    > print("start")
    >{
        if true then
            local t = {1, 2, 3}
            print(#t)
        end
    }
    >
```

The first step is a single-line execute-phase Lua statement. The second is an execute-phase Lua block whose body contains a literal `{` that does not prematurely close the block. The third is an empty `lua_line`. All three are standalone steps of the chore body (§{chores}); the empty third line contributes nothing.

### Note 8.7.1

A Lua block (either form) opened on an indented line closes on the line where the running brace-depth first returns to zero, not on the first bare `}`. A line like `    local s = "}"` does not close the enclosing block because the `}` sits inside a double-quoted string (§{lexical.brace-blocks}). See parser tests `test_lua_block_brace_in_string` and `test_lua_block_brace_in_comment`.

### Note 8.7.2

The execute-phase API surface available inside a `lua_line` or `lua_block` is a strict subset of the register-phase API: register-only helpers (`cook.add_unit`, `cook.exec`, `cook.interactive`, `cook.step_group`, `cook.recipe`) MUST raise a Lua runtime error when called from execute-phase Lua. `cook.sh` is available in both phases (§{lua.cook-sh}). See §{rationale.execute-phase-api} for the rationale.

## 8.8. Shell steps [#steps.shell]
A `shell_command` is a **chore-body** step kind (§{chores.body}). CS-0134 removed it from the recipe-body dispatch cascade: a recipe body is purely declarative (§{steps.dispatch}), and a recipe-body `Content` line that matches none of the declarative step kinds is REJECTED (§{steps.dispatch}, rule 7; Note 8.1.1), not classified as a shell step. A `shell_command` is defined here and is admitted only inside a `chore` body (§{chores}).

A `shell_command` is any `Content` line inside a chore body that is not classified as one of the step kinds earlier in the §{chores.body} dispatch cascade. Its body is the trimmed source line; a conforming implementation MUST preserve the body text verbatim, including any internal quoting, variable references, and shell metacharacters.

Every chore-body `shell_command` is **implicitly interactive**: it runs on the chore's single drain thread with the controlling terminal inherited (§{chores.default-interactive}), so a child such as `fzf` sees a real TTY. There is no separate marker or step kind for interactivity. CS-0134 removed the `@` interactive prefix — and the `interactive_command` step kind it once introduced — from the language entirely (§{lexical.line-prefixes}): a line beginning with `@` is now invalid syntax everywhere. The interactivity a pre-CS-0134 author spelled with `@` is simply the chore-body default.

### Example 8.8.1

```cook
chore play
    mkdir -p .cook/scratch
    printf 'a\nb\nc\n' | fzf > .cook/scratch/pick
    echo "picked: $(cat .cook/scratch/pick)"
```

All three lines are `shell_command` steps of the chore body. Because they sit in a chore body they each run under the chore's default-interactive single drain (§{chores.default-interactive}), so `fzf` inherits the user's terminal and can draw a full-screen prompt. No marker is needed; the chore form supplies the interactivity.

### Note 8.8.1

A `@` embedded inside a `cook` step's `{ ... }` shell-block body, or elsewhere inside a quoted string, is a literal `@` — ordinary shell text, not a language marker, since CS-0134 removed `@` from the Cookfile grammar entirely (§{lexical.line-prefixes}). See parser test `test_at_in_cook_body_is_not_interactive`.

## 8.9. Cross-recipe per-member output accessor [#steps.cross-recipe-member]

Inside an `ingredients <probe>` fan-out body (§{steps.ingredients}), a placeholder of the form `$<NAME[]>` — where `NAME` is a recipe declared in the same Cookfile — resolves to that recipe's terminal output **for the current fan-out member**. It is the per-member counterpart of bare `$<NAME>` (which substitutes all terminal outputs of `NAME`, space-joined, and is unchanged by this feature).

`$<NAME[]>` performs three jobs simultaneously:

1. **Path substitution.** At register time the placeholder is replaced by the member-specific output path: `cook.dep_output_member("NAME", cook.member_to_string(item))`, where `item` is the fan-out loop variable.
2. **Recipe-level DAG ordering edge.** The compiler extracts `$<NAME[]>` as an ordering constraint identical to the edge produced by a bare `$<NAME>` reference: `NAME` must build before the referencing recipe.
3. **Per-member fingerprint fold.** The referenced output is folded into the current member's cache fingerprint, not the recipe's aggregate fingerprint. Editing a single upstream member (e.g. one `.blend` file changing `render`'s output for `"outro"`) rebuilds only the `mux` member that depends on it.

**Same-probe precondition (v1).** Consumer and producer must iterate the same probe. The `[]` accessor indexes into the producer's member map using the consumer's current `item` key; if the two recipes are driven by different probes, the key may not exist. Cross-probe joins via declared alignment keys are deferred to COOK-99 and MUST NOT be attempted in this version. A conforming implementation that detects a same-probe violation at register time SHOULD emit a diagnostic; otherwise the missing-key lookup produces a register-phase error at `cook.dep_output_member`.

**Fan-out body only.** `$<NAME[]>` MUST NOT appear outside an `ingredients <probe>` fan-out body. A conforming implementation MUST reject it with a load-time error if no `ingredients <probe>` driver is present in the recipe.

**Relation to `$<NAME>`.** Both forms record a recipe-level dependency on `NAME`. `$<NAME>` substitutes the full output list (useful in a many-to-one step that needs all of `NAME`'s artifacts); `$<NAME[]>` substitutes only the current-member output (useful in a fan-out step that must stay per-member). The two forms MAY coexist in the same Cookfile but MUST NOT coexist in the same step unless `NAME` differs between the two placeholders.

### Example 8.9.1

```cook
probe scenes
    json { echo '[{"id":"intro"},{"id":"outro"}]' }

recipe render
    ingredients scenes
    cook "build/$<in.id>.silent.mp4" { ffmpeg -i $<in.id>.blend $<out> }

recipe tts
    ingredients scenes
    cook "build/$<in.id>.wav" { tts $<in.id> $<out> }

recipe mux
    ingredients scenes
    cook "build/$<in.id>.mp4" { ffmpeg -i $<render[]> -i $<tts[]> $<out> }
```

For member `{"id":"intro"}`, `mux` builds `build/intro.mp4` by combining `build/intro.silent.mp4` (from `render`) and `build/intro.wav` (from `tts`). For member `{"id":"outro"}` the same pattern repeats independently. Only the `outro` member's units are dirty when `outro.blend` changes — `intro.mp4` is unaffected.

### Note 8.9.1 — Conformance fixture

`standard/conformance/positive/recipe-member-accessor` pins this surface. The `parse.txt` golden confirms that `$<render[]>` and `$<tts[]>` are preserved verbatim in the `ShellBlock` strings at the parser layer; the codegen layer (CS-0098) lowers them to `cook.dep_output_member` calls.
