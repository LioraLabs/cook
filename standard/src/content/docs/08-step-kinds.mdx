# 8. Step kinds [#steps]
> **Normative.** This chapter defines every step kind that appears in a recipe body or a chore body, the dispatch cascade that classifies each `Content` token, and the iteration-mode rules for `cook`, `plate`, and `test` steps. The execute-phase behaviour of a body's compiled units ‚Äî including body bundling ‚Äî is specified in ¬ß{exec.groups}.

## 8.1. Step-dispatch cascade [#steps.dispatch]
Inside a `recipe_body`, each source line is first classified by the lexer (¬ß{lexical.line-classification}) into one of `Comment`, `Blank`, `LuaLine`, `LuaBlockOpen`, `InlineLuaLine`, `InlineLuaBlockOpen`, or `Content`. Comments and blank lines are skipped; `LuaLine` becomes a `lua_line` step; `LuaBlockOpen` begins a `lua_block` step collected by the brace-balance algorithm (¬ß{lexical.brace-blocks}); `InlineLuaLine` becomes an `inline_lua_line` step; `InlineLuaBlockOpen` begins an `inline_lua_block` step (also brace-balanced).

Prior to CS-0072 (Cook Standard v0.9), this cascade included a fifth rule classifying `BARE_IDENTIFIER "." BARE_IDENTIFIER(‚Ä¶)` as `module_call` (register-phase Lua). CS-0072 removes that rule from the recipe-body cascade and adds an equivalent top-level dispatch in ¬ß{toplevel.module-call}. Authors who want register-phase Lua inside a recipe body MUST use the `>>` (`inline_lua_line`) or `>>{` (`inline_lua_block`) prefix.

A `Content` token inside a `recipe_body` is dispatched to exactly one `declarative_item` or `imperative_item` (App. A.3) by the priority cascade below (App. A.4, "Step-dispatch priority"). The first match wins; a conforming implementation MUST test the alternatives in this order:

1. Prefix `ingredients` + separator (space, tab, or end of token) ‚Üí `ingredients_step`.
2. Prefix `cook` + separator ‚Üí `cook_step`.
3. Prefix `plate` + separator ‚Üí `plate_step`.
4. Prefix `test` + separator ‚Üí `test_step`.
5. First character is `@` and the remainder of the line is non-empty ‚Üí `interactive_command`.
6. Otherwise ‚Üí `shell_command`.

In step 6, an `@` with no following content MUST be rejected rather than silently dispatched as any of the preceding alternatives. The diagnostic MUST identify the offending line.

A `Content` token inside a `recipe_body` whose trimmed text begins with the keyword `register` followed by a separator (space, tab, or end-of-line) MUST be rejected with the diagnostic "`register` blocks are top-level only; move this outside the recipe body". The rejection applies regardless of indentation depth: an indented `register` cannot be a `RegisterHeader` (which is column-0 only by ¬ß{lexical.line-classification}) and is also not a permitted shell command in this position. A `Content` token whose first token is the bare identifier `register` *without* a separator (e.g. `register_foo`) remains a `shell_command` per the (post-CS-0072) rule 7. The reference parser fires this rejection in `parse_recipe` and `parse_chore`'s step-dispatch loops (informative).

A `recipe_body` MUST contain at most one `ingredients_step`. A conforming implementation MUST reject a recipe with two or more `ingredients_step` items (App. A.3).

A `recipe_body` MUST satisfy the **region ordering rule** (App. A.3, "Region ordering rule"; ¬ß{recipes.region-rule}): every `declarative_item` (other than `comment` or `NEWLINE`) MUST appear before the first `imperative_item` of the recipe body. A conforming implementation MUST reject a recipe in which a declarative-region step (cook, plate, test, ingredients, `>>`, or `>>{`) appears after the first imperative-region step (`>`, `>{`, shell, or `@`). The diagnostic MUST identify the offending step's line and the line on which the imperative region began.

### Example 8.1.1

```cook
recipe build
    ingredients "src/*.c"
    cook "bin/$<in.stem>" { cc $<in> -o $<out> }
    test { ./bin/app }
    @vim notes.txt
    echo done
```

The five body lines dispatch, in order, to: `ingredients_step`, `cook_step`, `test_step`, `interactive_command` (rule 6), and `shell_command` (rule 7). A bare `cpp.link()` line in a recipe body ‚Äî which before CS-0072 dispatched to `module_call` ‚Äî now falls through to rule 7 and becomes a `shell_command`.

### Example 8.1.2

```cook
recipe build
    CC "gcc"
```

The body line `CC "gcc"` is a `Content` token; the step-dispatch cascade matches no preceding alternative and dispatches it as a `shell_command` whose text is the trimmed source line. The emitted command preserves the original surface form ‚Äî the identifier and its quoted argument.

### Note 8.1.1

The `cook-lang` parser test `test_indented_quoted_pair_is_shell_command` pins the recipe-body treatment of `NAME "value"`. The test `test_empty_interactive_step_errors` pins rejection of a bare `@`.

The reference parser at `cli/crates/cook-lang` enforces the CS-0072 amendment by removing the `is_module_call` branch from the recipe-body step-dispatch in `parse_recipe` / `parse_chore` (informative).

## 8.2. Ingredients [#steps.ingredients]
A recipe body MAY contain at most one `ingredients_step` (¬ß{grammar-appendix.recipes}, "At-most-one-ingredients rule"). A second `ingredients` line within the same recipe body MUST be rejected.

The `ingredients_step` is introduced by the keyword `ingredients` followed by one or more `ingredient` items (¬ß{grammar-appendix.steps}, `ingredients_step`). Each item is either:

| Form | Meaning |
|---|---|
| `STRING`        | **include**: the quoted text is a glob pattern added to the ingredient set. |
| `!` `STRING`    | **exclude**: the quoted text is a glob pattern subtracted from the ingredient set. |

A conforming implementation MUST accept an `ingredients` line containing only excludes, only includes, or a mix of the two in any order. The semantic effect is the set difference `(union of includes) \ (union of excludes)`; ordering of items on the line does not affect the result.

The glob language itself ‚Äî which characters are wildcards, whether `**` matches across directory separators, and so on ‚Äî is not defined by ¬ß{recipes} and is resolved by the execution model (¬ß{exec.phases}).

### Example 8.2.1

```cook
recipe lib
    ingredients "src/*.c" "include/*.h" !"src/lua.c" !"src/luac.c"
    cook "build/libmath.a" { ar rcs $<out> $<all> }
```

The ingredient set is every `.c` file in `src/` and every `.h` file in `include/`, minus `src/lua.c` and `src/luac.c`.

### Note 8.2.1 ‚Äî overload rule

An `ingredients` argument is any iterable source. A quoted token (`"glob"` / `!"exclude"`) is a filesystem glob yielding **path** members; a bare identifier is a **probe key** yielding the probe's array members (¬ß{cat.probes}, ¬ß22.5.9). The two forms are mutually exclusive within a recipe. Members of both forms bind as `$<in>` ‚Äî path accessors (`$<in.stem>`, ‚Ä¶) for path members, record-field access (`$<in.FIELD>`) for record members.

The `!` exclusion marker (glob form only) MUST immediately precede the opening `"` with no intervening whitespace: `! "foo"` is rejected; `!"foo"` is accepted. See parser tests `test_ingredients_with_excludes` and `test_ingredients_excludes_only`.

## 8.3. Step kinds ‚Äî overview [#steps.overview]
A `recipe_body` is a sequence of `recipe_item`s (¬ß{grammar-appendix.recipes}). Excluding the `ingredients_step` covered in ¬ß{steps.ingredients} and the structural `comment` / `NEWLINE` items, a conforming implementation MUST recognise exactly the following nine step kinds:

| ¬ß | Step kind | Introduced by | Phase |
|---|---|---|---|
| ¬ß{steps.cook-single}, ¬ß{steps.cook-multi} | `cook_step`           | keyword `cook` | register |
| ¬ß{steps.plate}      | `plate_step`          | keyword `plate` | register |
| ¬ß{steps.test}      | `test_step`           | keyword `test` | register |
| ¬ß{steps.lua}      | `inline_lua_line`     | `>>` line prefix (¬ß{lexical.line-prefixes}) | register |
| ¬ß{steps.lua}      | `inline_lua_block`    | `>>{` line prefix (¬ß{lexical.line-prefixes}) | register |
| ¬ß{steps.lua}      | `lua_line`            | `>` line prefix (¬ß{lexical.line-prefixes}) | execute |
| ¬ß{steps.lua}      | `lua_block`           | `>{` line prefix (¬ß{lexical.line-prefixes}) | execute |
| ¬ß{steps.shell}     | `shell_command`       | default Content line | execute |
| ¬ß{steps.shell}     | `interactive_command` | `@` as the first non-whitespace character of a Content line | execute |

The phase column classifies each step kind by **when its body is evaluated**: register-phase steps run during recipe registration (¬ß{exec.two-phase}), execute-phase steps run when scheduled work units fire. A normative cross-section table appears in ¬ß{exec.phases.classification}.

A `Content` token inside a recipe body is classified into exactly one of the keyword-introduced kinds using the step-dispatch priority of ¬ß{grammar-appendix.steps} ("Step-dispatch priority"), reproduced here for convenience:

1. `ingredients` + separator ‚Üí `ingredients_step` (¬ß{steps.ingredients}).
2. `cook` + separator ‚Üí `cook_step` (¬ß{steps.cook-single}, ¬ß{steps.cook-multi}).
3. `plate` + separator ‚Üí `plate_step` (¬ß{steps.plate}).
4. `test` + separator ‚Üí `test_step` (¬ß{steps.test}).
5. First character is `@` and the remainder is non-empty ‚Üí `interactive_command` (¬ß{steps.shell}).
6. Otherwise ‚Üí `shell_command` (¬ß{steps.shell}).

The four Lua-prefix forms (`>>`, `>>{`, `>`, `>{`) are not subject to this cascade: they are selected at the lexical layer (¬ß{lexical.line-prefixes}) and always produce `inline_lua_line`, `inline_lua_block`, `lua_line`, or `lua_block` steps respectively, regardless of what follows the prefix.

### Note 8.3.1

The "separator" after a step keyword is space, tab, or end of token. A Content line whose first token is `cooking` is a `shell_command`, not a malformed `cook_step`. See parser test `test_strip_keyword`.

## 8.4. `cook` ‚Äî single-output form [#steps.cook-single]
A `cook_step` (¬ß{grammar-appendix.steps}, `cook_step`) declares a build output. The single-output form has exactly one `STRING` output pattern before the body (CS-0099 removed the `using` introducer; the body opener follows the pattern directly). The body is mandatory (CS-0133):

```ebnf
cook "out" body
```

The output is a `STRING` whose interior MAY contain substitution placeholders (e.g. `$<in.stem>`, `$<in>`, `$<out>`). The placeholder language is defined by ¬ß{phl} for substitution shapes and by ¬ß{xref} for cross-recipe references; ¬ß{recipes} treats placeholder text as opaque.

A conforming implementation MUST accept both of the following forms:

| Form | `body` | Semantic |
|---|---|---|
| `cook "out" >{ ... }`           | `exec_lua_block`        | execute-phase Lua block; produces the build effect, with `inputs` / `outputs` / `input` / `output` bindings per ¬ß{lua.using-block-globals}. |
| `cook "out" { ... }`            | `shell_block`           | sequence of shell commands, one per non-blank line of the block (¬ß{lexical.brace-blocks}, "Shell block line normalisation"). The block's textual placeholders (¬ß{phl.cook-step}) are substituted at register time. |

After the output pattern(s), the body-dispatch order of ¬ß{grammar-appendix.steps} ("body dispatch order") applies: `>{` is tested before `{`. The first match wins; the bare-string form is no longer admitted (CS-0022), and the former `using` introducer is rejected with a migration diagnostic (CS-0099).

### Example 8.4.1

```cook
recipe lib
    ingredients "lib/*.c"
    cook "build/obj/$<in.stem>.o" { gcc -c $<in> -o $<out> }
    cook "build/libmath.a" { ar rcs $<out> $<all> }
```

The first `cook` is a one-to-one mapping from each ingredient to one object file. The second is a many-to-one mapping producing a single archive from the collected set. Both use the inline (one-line) `shell_block` body form.

### Note 8.4.1

The `@` character inside a `shell_block` body is preserved verbatim ‚Äî it is part of the block's shell text, not an interactive-step marker. The interactive-step dispatch of ¬ß{steps.shell} applies only to a Content line's first non-whitespace character, not to text nested inside a `cook` step's body.

### 8.4.1. Iteration mode [#steps.iteration-mode]
A `cook_step`'s **iteration mode** is determined by its output pattern list, before the body is consulted. Three modes are defined:

| Output pattern shape | Mode | Driver | Units produced |
|---|---|---|---|
| At least one output contains `$<in.ACCESSOR>` (own-input accessor) | **One-to-one over own inputs** | The step's resolved `ingredients` list (¬ß{steps.ingredients}) | One per input |
| At least one output contains `$<lib.ACCESSOR>` (dep-driven accessor, ¬ß{xref.dep-driven}) | **One-to-one over dep outputs** | `lib`'s output list (¬ß{xref.dep-recipe-output}) | One per `lib` output |
| All outputs are literal (no accessor placeholders) | **Many-to-one** (the "one" refers to *one unit*, not one output) | None ‚Äî one unit runs with the full input list visible | Exactly one |
| The recipe carries an `ingredients <probe>` source (¬ß{steps.ingredients}) | **One-to-one over data members** | The probe's resolved member list | One per member |

A conforming implementation MUST reject, with a load-time diagnostic, a `cook_step` whose output patterns mix iteration sources ‚Äî that is, where two outputs select different drivers (one bears `$<in.ACCESSOR>` while another bears `$<libmath.ACCESSOR>`, or one bears `$<in.ACCESSOR>` while another is literal). All output patterns of a single `cook_step` MUST share one driver.

The iteration mode determines the number of work units produced; it is orthogonal to the `cook_step`'s declared output count. Each mode admits any number of declared outputs (one or more), as detailed in ¬ß{steps.cook-multi}.

When the recipe carries an `ingredients <probe>` source (¬ß{steps.ingredients}), the `cook_step` is in **one-to-one over data members** mode regardless of its output-pattern accessors: the per-member binding is `item` (Lua) / `$<in>` / `$<in.ACCESSOR>` (where `ACCESSOR` selects a record field rather than a path component), and the path-accessor interpretation of `$<in.ACCESSOR>` does not apply (a data member has no ingredient path). A conforming implementation MUST NOT allow `$<in.ACCESSOR>` accessors to be interpreted as path accessors when the recipe is driven by an `ingredients <probe>` source.

### Example 8.4.1.1

```cook
recipe build
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" { gcc -c $<in> -o $<out> }    # one-to-one over own inputs
    cook "build/app" { gcc $<all> -o $<out> }              # many-to-one (literal output)
```

### Note 8.4.1.1

The CS-0022 unification removes `$<in>` from the body's contents as a separate iteration trigger. Iteration mode is decided exclusively by the output pattern list. A body that contains `$<in>` while the output pattern list is fully literal is rejected at load time ‚Äî see ¬ß{phl.cook-step} for the diagnostic shape.

### 8.4.2. Lua-expression output [#steps.cook-lua-expr-output]

A `cook_step` MAY substitute a parenthesised Lua expression for the quoted output-path `STRING`:

```ebnf
cook_step ::= "cook" ( STRING+ | "(" LUA_EXPR ")" ) body NEWLINE
```

A conforming implementation MUST accept the Lua-expression form under the following rules.

1. **Evaluation.** The expression is evaluated **once per ingredient**, at register time, against the Cookfile-scope Lua VM (the same VM used for chore-parameter Lua-expression defaults, ¬ß{chores.params}).
2. **Bindings.** The expression sees the name `input` in scope: the current ingredient's path, a Lua `string`. When the recipe is driven by an `ingredients <probe>` source (¬ß{steps.ingredients}), the expression instead sees `item` bound to the current data member (a table for record members, a string/number/boolean for scalar members) rather than `input`. The expression MAY also reference declarative-region locals (top-level Lua bindings, `config` block writes, register-phase `inline_lua_line` / `inline_lua_block` locals visible to the recipe per ¬ß{steps.lua-execution-phase}). The standard library is available subject to the ¬ß{lua} sandbox in effect at register time.
3. **Return value.** The expression's return value MUST coerce to a non-empty string under the same coercion rules the Standard applies to chore Lua-expression defaults (¬ß{chores.params}): `string` verbatim, `number` via Lua `tostring`, `boolean` to `"true"` or `"false"`; any other type ‚Äî including `nil`, `table`, `function`, `thread`, `userdata` ‚Äî is a register-phase diagnostic. The empty string is likewise a register-phase diagnostic.
4. **One-to-one only.** The Lua-expression form is admitted exclusively when the resulting `cook_step` is in **one-to-one over own inputs** mode or **one-to-one over data members** mode (¬ß{steps.iteration-mode}); in the data-member case the expression sees `item` rather than `input` (rule 2). The expression is evaluated per iteration element, so dep-driven and many-to-one modes ‚Äî neither of which iterates element-by-element ‚Äî have no well-defined evaluation contract. A conforming implementation MUST reject a Lua-expression-output `cook_step` whose recipe has no `ingredients_step`, or whose iteration source resolves to the empty set, with a register-phase diagnostic.
5. **Mutual exclusion with quoted-string outputs.** Within a single `cook_step`, the output declaration is either (a) one or more `STRING` patterns with the `$<...>` sigil vocabulary of ¬ß{phl.cook-step}, or (b) exactly one parenthesised Lua expression. A conforming implementation MUST reject a `cook_step` that mixes the two forms (e.g. `cook "a.out" (input:gsub(...)) { ... }`) with a register-phase diagnostic naming the offending tokens.
6. **Body.** The body of a Lua-expression-output `cook_step` follows the same grammar as the quoted-string form (¬ß{steps.cook-single}): `exec_lua_block` (`>{ ... }`) and `shell_block` (`{ ... }`) are both admitted; the body is mandatory (CS-0133). The placeholder vocabulary inside the body (¬ß{phl.cook-step}) is the one-to-one vocabulary: `$<in>`, `$<in.ACCESSOR>`, `$<out>`. Note that `$<out>` resolves to the Lua-expression's evaluated string for the current iteration; placeholders that index the output position list (`$<out_1>`, `$<out_2>`, ‚Ä¶) have no use in this form because the form declares exactly one output per iteration.

### Example 8.4.2.1

```cook
recipe translate-fr
    ingredients "docs/en/**/*.md"
    cook (input:gsub("/en/", "/fr/")) >{
        fs.write(output, transform(fs.read(input)))
    }
```

The expression `input:gsub("/en/", "/fr/")` runs once per matched ingredient with `input` bound to that ingredient's path; the resulting string becomes the unit's `outputs[1]`. The `exec_lua_block` sees the `input` and `output` bindings per ¬ß{lua.using-block-globals}; `output` is the same string the Lua-expression produced.

### Example 8.4.2.2

```cook
recipe thumbnail
    ingredients "assets/photos/*.jpg"
    cook (input:gsub("assets/photos", "build/thumb")) { convert $<in> -resize 256x256 $<out> }
```

Mixes the Lua-expression output form with a `shell_block` body. The shell block's `$<in>` and `$<out>` placeholders resolve to the iteration's ingredient path and the Lua-expression's evaluated string, respectively.

### Note 8.4.2.1 ‚Äî Cache key impact [#steps.cook-lua-expr-output.cache]

The expression's resolved string is recorded as the unit's `outputs[1]`, and so participates in the cache key via the output-path observable of ¬ß{exec.cache.abstract} #3. The expression source text itself is part of the recipe's parsed body and is therefore already folded into the unit's command-text / Lua-chunk fingerprint per ¬ß{exec.cache.abstract} #1 ‚Äî no separate fingerprinting pass for the expression is required.

### Note 8.4.2.2 ‚Äî Why one-to-one only [#steps.cook-lua-expr-output.rationale]

The Lua-expression form's defining feature is per-ingredient evaluation: each ingredient produces one unit whose output path is the expression evaluated with `input` bound to that ingredient. Dep-driven iteration (¬ß{xref.dep-driven}) iterates over a *dependency's* output list, not over the step's own ingredient list, so `input` would have no canonical meaning. Many-to-one iteration runs the unit exactly once over the full input list, so per-ingredient evaluation has no addressable iteration. The restriction is therefore not a guard against unsafe semantics but a consequence of the bind-`input`-per-ingredient contract.

### Note 8.4.2.3 ‚Äî Register-phase diagnostic shape [#steps.cook-lua-expr-output.diagnostic]

A conforming implementation rejecting the resolved output value MUST surface a register-phase diagnostic whose message identifies (a) the offending ingredient path and (b) the rule violated (non-string return, empty string). The reference implementation emits `cook (LUA_EXPR) returned non-string or empty value for input <PATH>` from the per-ingredient register-phase guard; an alternative implementation MAY phrase the message differently but MUST carry equivalent information.

### 8.4.3. Cook-step disposition [#steps.cook-disposition]

**The `cook` step IS the cacheable unit.** One step ‚Üí one content-addressed key ‚Üí one disposition, however many output files it emits. The whole surface teaches a single mental model:

> **Body = what it's made of. `cook` = one unit (‚â•1 output). Tail = what to do with that unit.**

A unit's cache participation is declared by two surfaces ‚Äî *inputs* and *output-facts* ‚Äî which split along a seam that was already there: `seal`/`unseal` name a determinant **input**, and `local`/`pinned`/`nondet` state a **fact about the unit's output**. They are therefore the different kinds they are.

| Concept | Written as | Where | Scope |
|---|---|---|---|
| File inputs | `ingredients "‚Ä¶"` | body line | recipe-wide |
| Probe inputs | `seal <probes>` | body line | recipe-wide |
| The unit | `cook "out1" "out2" { ‚Ä¶ }` | body line | one key, ‚â•1 output |
| Per-unit identity tweak | `‚Ä¶ seal/unseal <probes>` | tail of the `cook` | that unit |
| Disposition | `‚Ä¶ local / pinned / nondet` | tail of the `cook` | that unit |

**One rule:** a body line names a recipe-wide input; a tail tunes one `cook` unit. The cache-key *effect* of each is normative in ¬ß{exec.cache} (Cache-trust v3); this section defines the surface and its parse-time rules.

A `probe_ref` (used by `seal`/`unseal` everywhere) is a bare `BARE_PROBE_KEY` ‚Äî `IDENT (":" IDENT)?`, module-prefixable (e.g. `cc:toolchain`). The quoted `STRING` form of ¬ß{grammar-appendix.probe} is NOT admitted, and a third contiguous `:IDENT` segment (`a:b:c`) MUST be rejected, exactly as for `probe_ref` in ¬ß{grammar-appendix.probe}. A `seal`/`unseal` ref is an *invalidate-only* determinant ‚Äî it is never interpolated; there is no `$<seal:‚Ä¶>`. To *consume* a probe's members as data, use `ingredients <probe>` / `for_each` (¬ß{steps.ingredients}), not `seal`.

**A seal/unseal ref is ALWAYS a probe name.** There is no `env.X` / `tool.X` ambient-determinant ref form ‚Äî neither a way to seal an environment variable inline nor a way to seal a tool by file path. `probe_ref` is therefore complete as written above; it needs no extension. The referenced probe MAY be any probe ‚Äî a shell or Lua probe, a `tools` probe, or an `envs` probe (¬ß{cat.probes.decl}) ‚Äî but the surface is uniform: every seal ref is just a probe name. The two ambient-determinant paths fold through that one surface:

- **Unconsumed environment.** An environment variable that no unit consumes (so ¬ß{exec.cache.lua-env-reads} does not auto-fold it) is folded by declaring an `envs` probe over the variable(s) and sealing that probe by name ‚Äî `seal <envsprobe>`. The `envs` probe's value is a cached, shareable fingerprint over the named variables (¬ß{cat.probes.decl}). Environment a unit *does* consume continues to auto-fold via ¬ß{exec.cache.lua-env-reads}, unchanged.
- **Tool determinants.** A toolchain is folded by sealing a `tools` probe (¬ß{cat.probes.decl}) by name. The asymmetry with environment is deliberate: tools want cached, shared fingerprinting ‚Äî exactly what a probe provides ‚Äî whereas an `env.X` inline ref would have provided neither grouping, naming, nor shareability. Both ambient kinds therefore resolve to the same uniform "seal a named probe" surface, with no `env.X` / `tool.X` special forms to parse.

#### `seal` / `unseal` steps [#steps.cook-disposition.seal]

`seal` and `unseal` are recipe-body **steps** ‚Äî a second input stream, sibling of `ingredients`, declaring *determinant* probe inputs (probe values folded into the unit cache key) alongside `ingredients`' *file* inputs. A conforming implementation MUST enforce, at parse time:

1. **Recipe-level `seal` is the baseline.** A `seal <probes>` step adds its `probe_ref`s to the recipe's determinant baseline, which applies to **every** `cook` unit in the recipe. Bare `seal` (zero refs) MUST be rejected ‚Äî seal/unseal are always explicitly named.
2. **Additive and order-independent.** Multiple recipe-level `seal` steps union their refs. The baseline applies to every cook regardless of textual position (it is *declarative*, not positional): `seal` after a `cook` still applies to it. The effective set is sorted and de-duplicated.
3. **Recipe-level `unseal` is rejected.** `unseal` is a trailing modifier on a `cook` step only (rule 5). The recipe is the outermost seal scope; there is nothing inherited to release, so a recipe-level `unseal` would read as a no-op and MUST be rejected.

**Recipe boundaries break seals implicitly.** The recipe is the outermost seal scope; nothing propagates across recipe dependency edges. A downstream recipe folds an upstream's determinants in transitively via the upstream's OUTPUT CONTENT hashes (consumed as `ingredients`), so re-sealing downstream would double-count. There are exactly two seal tiers ‚Äî recipe baseline and per-unit tail ‚Äî and no config-default inheritance tier.

> **Note 8.4.3.1 ‚Äî boundary of a future blanket seal (informative).** The seal surface is *named-only*: every determinant a unit keys on is an explicitly named probe ref. No absolute / blanket form (a hypothetical "seal everything visible") exists, and a bare `seal` is rejected (rule 1). This note fixes the boundary such a form would have to draw, so that introducing one later cannot silently change today's keys. A blanket seal would have to enumerate a *universe* comprising (a) the unit's *visible-but-unconsumed* probes ‚Äî probes declared in the Cookfile that the unit neither consumes as `ingredients`/`for_each` data nor names in its effective seal set ‚Äî plus (b) ambient determinants not modelled as probes at all (consulted environment beyond ¬ß{exec.cache.lua-env-reads}, host/machine identity, tool identity). The current surface declares no such universe: a unit keys on exactly the probes named in its effective seal set and nothing more (¬ß{exec.cache.single-key}). Any future blanket form is a NEW determinant set, not a re-reading of the named-only one, and MUST be introduced under its own change ID with its own key-version bump.

#### Trailing modifiers (`cook_mods`) [#steps.cook-disposition.tail]

```ebnf
cook_mods ::= ("seal" probe_ref+ | "unseal" probe_ref+)* share_mod?
share_mod ::= "local" | "pinned" | "nondet"
```

The tail of a `cook` step tunes that one unit. The `cook_mods` are line-final: they follow the body's closing `}` on the same physical line (a cook step always has a body, CS-0133). Tokens on a subsequent line are not part of the `cook_mods` (a `}` on its own line followed by a `seal ‚Ä¶` line dispatches the latter as a recipe-level `seal` step). A conforming implementation MUST enforce:

4. **Per-unit seal/unseal adjust the baseline.** A unit's *effective seal set* is `(baseline ‚à™ trailing seals) ‚àí trailing unseals`, sorted and de-duplicated. Bare trailing `seal`/`unseal` (no refs) MUST be rejected.
5. **`unseal` is trailing-only.** `unseal <probes>` removes refs from this unit's effective set; it is valid only here (not as a recipe-level step, rule 3).
6. **`share_mod` is a single optional slot.** At most one of `local` / `pinned` / `nondet` MAY appear, and it MUST be the last token ‚Äî nothing may follow it. The three are mutually exclusive by construction (a single slot). `local` opts out of sharing (¬ß{exec.cache.sharing}); `pinned` is designated-producer / fetch-only (¬ß{exec.cache.sharing}); `nondet` declares the output intrinsically non-reproducible, waiving byte-equivalence (¬ß{exec.cache.record}).
7. **`record` is renamed `nondet`.** The v3 `record` disposition is renamed `nondet` (it states a *fact* ‚Äî the output is nondeterministic ‚Äî rather than an imperative policy). The `record` keyword in `cook_mods` MUST be rejected with a migration diagnostic.
8. **Non-reserved words.** `seal`, `unseal`, `local`, `pinned`, and `nondet` are NOT reserved words. `seal`/`unseal` are recognised as steps only at a recipe-body line position; `local`/`pinned`/`nondet` only in the trailing `cook_mods` position. A recipe-body line whose first token is `local`/`pinned`/`nondet` (followed by other content or alone) is an ordinary `shell_command` (¬ß{steps.shell}).

### Example 8.4.3.1 ‚Äî the whole surface once

The probes `toolchain`, `flags`, and `rev` are declared elsewhere (¬ß{cat.probes}):

```cook
recipe build
    ingredients "src/*.c"          # ‚îê recipe-wide inputs
    seal toolchain flags           # ‚îò ‚Üí identity
    cook "bin/app" "bin/app.map" { cc ‚Ä¶ -Wl,-Map=bin/app.map }  # one unit, two files, one key
    cook "bin/app.dbg" { cc -g ‚Ä¶ } local                       # separate unit, kept local
    cook "bin/plugin"  { cc ‚Ä¶ } seal rev                       # also keyed on the git rev
    cook "gen/ver.h"   { ‚Ä¶ } nondet                            # non-reproducible ‚Üí reuse stored copy
```

`bin/app`/`bin/app.map` form one unit (one key, two outputs) sealed on `toolchain` and `flags`; `bin/app.dbg` is the same baseline but kept `local`; `bin/plugin` additionally seals on `rev`; `gen/ver.h` is `nondet`. A multi-output *unit* carries one tail; several *separate* `cook` steps wanting the same disposition repeat the word (an accepted rough edge ‚Äî split the recipe, or add a recipe-level default later without breaking the grammar).

### Example 8.4.3.2 ‚Äî per-unit unseal

```cook
recipe build
    seal a b
    cook "x.o" { cc -c x.c -o x.o } unseal a seal c
```

The effective seal set of `x.o` is `{b, c}`: baseline `{a, b}`, plus trailing `seal c`, minus trailing `unseal a`.

### Note 8.4.3.2 ‚Äî Conformance fixtures [#steps.cook-disposition.fixtures]

Positive: `standard/conformance/positive/cook-disposition-seal-line`, `cook-disposition-seal-stacked`, `cook-disposition-local-line`, `cook-disposition-pinned-line`, `cook-disposition-nondet`, `cook-disposition-trailing-seal-unseal`, `cook-disposition-seal-order-independent`, `cook-disposition-seal-multi-cook-baseline`, `cook-disposition-unseal-one-unit-only`, `cook-disposition-seal-envs-probe`, `cook-disposition-seal-tools-probe`.

Negative: `standard/conformance/negative/cook-disposition-local-and-pinned`, `cook-disposition-double-share-mod`, `cook-disposition-bare-seal`, `cook-disposition-recipe-unseal`, `cook-disposition-record-renamed`, `cook-disposition-seal-quoted-ref`, `cook-disposition-seal-triple-colon`.

## 8.5. `cook` ‚Äî multi-output form [#steps.cook-multi]
A `cook_step` MAY declare two or more outputs by listing additional `STRING` patterns before the body:

```ebnf
cook "a" "b" ... body
```

When two or more outputs are present and a body is given, the body is one of the two block forms (¬ß{steps.cook-single}): `exec_lua_block` (`>{ ... }`) or `shell_block` (`{ ... }`). The bare-string form was removed in CS-0022, and `cook "a" "b" "cmd"` parses as a three-output cook with no body ‚Äî a trailing `STRING` joins the output-pattern list rather than being read as a body ‚Äî which is then rejected because a body is mandatory (CS-0133).

Like the single-output form, a multi-output `cook_step` MUST have a body (CS-0133); `cook "a" "b"` with no body is a parse error.

### Example 8.5.1

```cook
recipe generate
    ingredients "src/*.rs"
    cook "staging/out.js" "staging/out.wasm" {
        ./fake-generator.sh
        mkdir -p staging
        cp pkg/out.js $<out_1>
        cp pkg/out.wasm $<out_2>
    }
```

One `cook` step declares two outputs produced together by a sequence of shell commands. The commands are the four non-blank lines of the shell block, in order (¬ß{lexical.brace-blocks}). The placeholders `$<out_1>` and `$<out_2>` resolve to the unit's first and second declared outputs, in declaration order, per ¬ß{phl.cook-step}.

### Example 8.5.2

```cook
recipe build_lua
    ingredients "src/*.rs"
    cook "staging-lua/out.js" "staging-lua/out.wasm" >{
        os.execute("mkdir -p staging-lua")
        local sources = table.concat(inputs, " ")
        local js = io.open(outputs[1], "w")
        js:write("// generated from " .. sources .. "\n")
        js:close()
        local wasm = io.open(outputs[2], "w")
        wasm:write("BINARY\n")
        wasm:close()
    }
```

The same two outputs are produced by a Lua block. The `inputs` and `outputs` names visible to the block are established by the execution model (¬ß{exec.phases}) and the Cook Lua API (¬ß{lua.using-block-globals}); ¬ß{recipes} requires only that a block-form body be accepted.

### Note 8.5.1

The shape `cook "a" "b" using "cmd"` was rejected pre-CS-0022 with a "block body" diagnostic, and under CS-0022 with a bare-string-removal diagnostic. Since CS-0099 the `using` token itself is rejected at this position with the migration diagnostic of ¬ß{grammar-appendix.steps} ("`using` is rejected"); conformance fixture `cs0099-using-keyword-rejected` pins it.

## 8.6. `plate` step [#steps.plate]
A `plate_step` (¬ß{grammar-appendix.steps}, `plate_step`) is **a `cook` step with no declared outputs**. It has a body ‚Äî either a `shell_block` (¬ß{grammar-appendix.steps}) or an `exec_lua_block` (¬ß{grammar-appendix.steps}) ‚Äî and no output pattern. It registers work units that perform side-effecting actions whose results do not contribute to the recipe's output list (which remains passthrough per ¬ß{xref.dep-recipe-output}).

```ebnf
plate_step ::= "plate" body NEWLINE
body       ::= shell_block | exec_lua_block
```

The body grammar is identical to a `cook_step`'s body grammar (¬ß{steps.cook-single}). The placeholder vocabulary inside a `plate_step` body is specified in ¬ß{phl.plate-test} (shell-block bodies) and ¬ß{lua.using-block-globals} (Lua-block bodies), with the per-step-kind admission rules of ¬ß{steps.iteration-mode-plate-test}.

A `plate_step` admits no leading `STRING` and no trailing modifiers. The body follows the keyword directly.

### Example 8.6.1

```cook
recipe install
    ingredients "src/*.c"
    cook "build/bin/$<in.stem>" { cc $<in> -o $<out> }
    plate {
        install -d $PREFIX/bin
        install -m755 $<in> $PREFIX/bin/$<in.name>
    }
```

The `plate` step iterates over the preceding `cook` step's output list (one unit per binary), installing each binary into `$PREFIX/bin`.

### Example 8.6.2

```cook
recipe bundle
    ingredients "src/*.c"
    cook "build/bin/$<in.stem>" { cc $<in> -o $<out> }
    plate {
        tar -czf build/bundle.tgz $<all>
    }
```

The body uses `$<all>` and no `$<in>`; the step is many-to-one and registers exactly one unit, with `$<all>` substituted as the space-joined preceding-cook output list.

### Note 8.6.1

A `plate_step`'s body grammar matches a `cook_step`'s body. Since CS-0099 the two forms are fully parallel: in both, the body opener follows the step's header material directly. See rationale ¬ß{rationale.plate-test-no-output-decl}.

### Note 8.6.2 ‚Äî Plate step bodies are NOT sandboxed [#steps.plate-not-sandboxed]

A `plate_step` body is the explicit "ship outside the project" surface and runs without the project-root sandbox imposed on `cook` / `test` / `chore` step bodies (¬ß{lua.fs-sandbox}). A `plate` body MAY pass an absolute path outside the project root to `fs.*`, MAY call `os.execute` and `io.popen`, and MAY traverse upward via `..`. This is by construction: deploys, uploads, and other release-time side effects often touch paths the project does not own (e.g. `$HOME/.config/...`, `/var/...`, an S3 bucket).

A conforming implementation MUST run a `plate_step`'s body without applying the ¬ß{lua.fs-sandbox} confinement. Authors who want the confinement on a non-cacheable side-effect step SHOULD use a `chore` instead, which IS sandboxed (chores are non-cacheable but hermetic-by-intent ‚Äî the sandbox catches accidental escapes without blocking the deliberate ones plate covers).

### 8.6.1. Plate/test iteration mode [#steps.iteration-mode-plate-test]

A `plate_step` (and a `test_step`, ¬ß{steps.test}) is in one of three iteration modes, determined entirely by its body's content:

| Shell-block body contains | Lua-block body references | Mode |
|---|---|---|
| `$<in>` or `$<in.ACCESSOR>` (no `$<all>`) | `input` (no `inputs`) | **one-to-one** over source |
| `$<all>` (no `$<in>` / `$<in.ACCESSOR>`) | `inputs` (no `input`) | **many-to-one** |
| neither | neither | **one-shot** |
| both | both | **error** (mixed iteration signal) |

The **iteration source** for a `plate_step` (and `test_step`) is the preceding `cook_step`'s output list (flattened across all units it produces), falling back to the recipe's resolved ingredients (¬ß{steps.ingredients}) if no `cook_step` precedes it. When the recipe is driven by an `ingredients <probe>` source (¬ß{steps.ingredients}), the iteration source is instead the resolved data-member list, and the per-member placeholders remain `$<in>` / `$<in.ACCESSOR>` (Lua: `item`), where `ACCESSOR` selects a record field rather than a path component; such a step is in **one-to-one over data members** mode. A step in **one-to-one** or **many-to-one** mode MUST have a non-empty source; a step in **one-shot** mode does not consult the source.

A conforming implementation MUST reject:
- A body that contains both an iteration-item placeholder/binding (`$<in>` / `$<in.ACCESSOR>` / `input`) and a batched-source placeholder/binding (`$<all>` / `inputs`).
- A `plate_step` or `test_step` in one-to-one or many-to-one mode whose source is empty.

The Lua identifier scan operates on the same lexical text the brace-balance lexer (¬ß{lexical.brace-blocks}) operates on ‚Äî strings, comments, and long strings are excluded; only free-identifier-position references count.

## 8.7. `test` step [#steps.test]
A `test_step` (¬ß{grammar-appendix.steps}, `test_step`) has the same body grammar as a `plate_step` (¬ß{steps.plate}) and additionally admits three trailing modifiers:

```ebnf
test_step      ::= "test" body test_modifiers NEWLINE
test_modifiers ::= ("as" STRING)? ("timeout" NUMBER)? "should_fail"?
```

The body and iteration-mode rules of ¬ß{steps.plate} apply verbatim. A `test_step` registers work units that report pass/fail rather than producing artifacts; iteration source, mode determination, placeholder vocabulary, and ¬ß{xref.string-substitution} cross-recipe substitution are identical to `plate_step`.

The optional modifiers MUST appear in the fixed canonical order shown above: `as` ‚Üí `timeout` ‚Üí `should_fail`. A conforming implementation MUST reject any out-of-order modifier sequence with a diagnostic that names the offending modifier and its canonical position.

| Form | Behaviour |
|---|---|
| `test BODY` | Run the body unit; expect success (no Lua error, no non-zero `cook.sh`/`cook.exec`); no time bound. |
| `test BODY as 'NAME'` | Same, with display name `NAME` in test reports. |
| `test BODY timeout N` | Run the body unit; expect success within `N` seconds. |
| `test BODY as 'NAME' timeout N` | Same, with display name and time bound. |
| `test BODY should_fail` | Run the body unit; expect failure (a Lua error or a non-zero `cook.sh`/`cook.exec` return); no time bound. |
| `test BODY timeout N should_fail` | Run the body unit; expect failure within `N` seconds. |
| `test BODY as 'NAME' timeout N should_fail` | Same, with display name. |

`NUMBER` is the `NUMBER` token class of ¬ß{lexical.numbers}. The upper bound on representable timeouts is implementation-defined.

The `as STRING` modifier sets the test's **display name** for reporting. Without it, the implementation assigns an auto-generated identity (typically `test#N` with an iteration discriminator suffix in non-one-shot modes). With it, the auto-generated identity is replaced by the supplied string; the iteration-discriminator suffix is still appended in iteration modes unless the substituted name already varies per iteration.

The `STRING` argument admits the same placeholder substitution (`$<IDENT>`) as the body (CS-0033). Substitution occurs at register-time codegen. The display name is metadata ‚Äî it is **not** part of the test's cache fingerprint; renaming a test does not bust its cache.

**Codegen mapping.** When `as STRING` is present, the conforming code generator MUST emit a `name = <substituted-expr>` field in the `cook.add_test` call that registers the unit. The `<substituted-expr>` is the result of applying placeholder substitution to `STRING` under the same binding context as the step body (¬ß{phl.plate-test}): `_test_in` for one-to-one mode, the collected-inputs table for many-to-one mode, and the empty string for one-shot mode. When `as STRING` is absent, the code generator MUST NOT emit a `name` field; the runner assigns an auto-generated identity.

The `as` modifier is **only valid on `test_step`**. A conforming implementation MUST reject `as` on `cook_step` or `plate_step` with a diagnostic identifying the modifier and the step kind.

### Example 8.7.1

```cook
recipe smoke
    ingredients "tests/*.c"
    cook "build/$<in.stem>" { cc $<in> -o $<out> }
    test { ./$<in> } timeout 60
```

Each test binary is run with a 60-second time bound, and the step expects each to exit zero (no `should_fail`).

### Example 8.7.2

```cook
recipe coverage
    ingredients "src/*.c"
    cook "build/bin/$<in.stem>.test" { cc $<in> -o $<out> }
    test >{
        for _, bin in ipairs(inputs) do
            cook.sh("./" .. bin .. " --check")
        end
    } timeout 300
```

The Lua body references `inputs` (no `input`); the step is many-to-one and runs once with all test binaries visible as `inputs`.

### Example 8.7.3

```cook
recipe roundtrip
    ingredients "src/*.txt"
    cook "build/$<in.stem>.out" { echo > $<out> }
    test { diff $<in> $<out> } as '$<in.stem>-roundtrip' timeout 10
```

The `as` modifier includes a placeholder; each iteration gets a unique display name (e.g., `foo-roundtrip`, `bar-roundtrip`).

### Note 8.7.1

`should_fail` is a bare word, not a quoted string. It is recognised only as the trailing modifier on a `test` line; elsewhere it has no syntactic meaning and is not a reserved word (¬ß{lexical.keywords}). The same posture applies to `timeout` and `as`.

A `test_step` in one-to-one mode that runs a Lua body where `cook.sh`/`cook.exec` returns non-zero fails by the same rule as a Lua error; `should_fail` inverts the pass/fail decision.

## 8.8. Lua steps ‚Äî line and block forms [#steps.lua]

The Cookfile language has two families of Lua step. They are distinguished by line prefix and by **execution phase** (¬ß{exec.two-phase}):

| Surface form | Token (¬ß{lexical.line-classification}) | Phase    | Region (¬ß{recipes.region-rule}) |
|---|---|---|---|
| `>` *body* `\n`           | `LuaLine`              | execute  | imperative |
| `>{` *body* `}` `\n`      | `LuaBlockOpen`         | execute  | imperative |
| `>>` *body* `\n`          | `InlineLuaLine`        | register | declarative |
| `>>{` *body* `}` `\n`     | `InlineLuaBlockOpen`   | register | declarative |

### 8.8.1. Delimitation [#steps.lua-delimitation]

A `lua_line` step's body is the remainder of the line after `>` and an optional single space; the body MAY be empty (App. A.5, "`lua_line` MAY be empty"). An `inline_lua_line` step's body is the remainder of the line after `>>` and an optional single space; it too MAY be empty.

A `lua_block` step's body is the source text between the opening `>{` and the matching `}`, collected by the brace-balance algorithm of ¬ß{lexical.brace-blocks}. An `inline_lua_block` step's body is the source text between the opening `>>{` and the matching `}`, by the same algorithm. The closing `}` line is consumed but is not part of either block's body.

A conforming implementation MUST NOT reinterpret the Lua body text. Comments, strings, and other Lua constructs inside the body are the Lua runtime's concern, with the single exception that the brace-balance algorithm (¬ß{lexical.brace-blocks}) MUST ignore braces appearing inside Lua line comments, double-quoted strings, single-quoted strings, and long strings (`[[...]]`) when locating the closing `}` of a Lua block (either form).

### 8.8.2. Execution phase [#steps.lua-execution-phase]

`inline_lua_line` and `inline_lua_block` are **register-phase** steps. Their body is evaluated during recipe registration in the recipe-body Lua function. Multiple inline-Lua steps in one recipe share register-phase scope: a `local` declared in one is visible to the next, including across the surface forms of `cook` / `plate` / `test` / `ingredients` (which compile to register-phase Lua API calls). This is the surface form a Cookfile uses to call Cook Lua API functions that record work ‚Äî `cook.add_unit`, `cook.export`, `cook.step_group`, and so on ‚Äî and to declare register-phase locals consumed by subsequent declarative steps.

`lua_line` and `lua_block` are **execute-phase** steps. Their bodies are not evaluated during registration. They are recorded as part of a **body unit** (¬ß{exec.body-bundling}) whose `lua_code` payload runs on a worker VM during the execute phase, after the recipe's declarative work units have completed.

Prior to CS-0072, `module_call` was also register-phase: it desugared to `inline_lua_line` (single-line) or `inline_lua_block` (multi-line). As of CS-0072 the module_call dispatch is removed from the recipe-body cascade; module calls that previously appeared in recipe bodies MUST be moved to a top-level `register` block (¬ß{decl.register}) or written as a top-level module_call (¬ß{toplevel.module-call}).

### 8.8.3. Examples [#steps.lua-examples]

```cook
recipe build
    >> local std = cook.env.CXX_STD or "c++20"
    >>{
        cook.env.CXXFLAGS = "-std=" .. std
    }
    cook "build/app" >{
        cook.sh(string.format("%s %s -o %s",
            cook.env.CXX or "g++", table.concat(inputs, " "), output))
    }
    > print("build complete")
```

The `>>` line and `>>{ }` block run at register, sharing scope with the `cook` step's `>{ }` template-string formation. The `> print(...)` line runs at execute, after the cook step's unit completes. (The `>{ }` body is execute-phase Lua-with-bindings; see ¬ß{lua.using-block-globals}.)

```cook
recipe smoke
    cd build
    ./app --selftest
    > if not fs.exists("build/.smoke.ok") then error("smoke failed") end
```

The recipe has only an imperative region: two shell lines and one `>` line. The two shell lines coalesce into one `cook.sh("set -e\ncd build\n./app --selftest")` call, so `cd build` persists into `./app --selftest`. The `>` line then runs in the same Lua VM as the coalesced shell call. (Body bundling is specified in ¬ß{exec.body-bundling}.)

```cook
recipe legacy
    > print("start")
    >{
        if true then
            local t = {1, 2, 3}
            print(#t)
        end
    }
    >
```

The first step is a single-line execute-phase Lua statement. The second is an execute-phase Lua block whose body contains a literal `{` that does not prematurely close the block. The third is an empty `lua_line`. All three coalesce into one body unit (no `@` interruption); the empty third line contributes nothing to the chunk.

### Note 8.8.1

A Lua block (either form) opened on an indented line closes on the line where the running brace-depth first returns to zero, not on the first bare `}`. A line like `    local s = "}"` does not close the enclosing block because the `}` sits inside a double-quoted string (¬ß{lexical.brace-blocks}). See parser tests `test_lua_block_brace_in_string` and `test_lua_block_brace_in_comment`.

### Note 8.8.2

The execute-phase API surface available inside a `lua_line` or `lua_block` is a strict subset of the register-phase API: register-only helpers (`cook.add_unit`, `cook.exec`, `cook.interactive`, `cook.step_group`, `cook.recipe`) MUST raise a Lua runtime error when called from execute-phase Lua. `cook.sh` is available in both phases (¬ß{lua.cook-sh}). See ¬ß{rationale.execute-phase-api} for the rationale.

## 8.9. Shell steps ‚Äî plain and interactive [#steps.shell]
A `shell_command` is any `Content` line inside a recipe body that is not classified as one of the step kinds earlier in the ¬ß{steps.overview} dispatch cascade. Its body is the trimmed source line; a conforming implementation MUST preserve the body text verbatim, including any internal quoting, variable references, and shell metacharacters.

An `interactive_command` is a `Content` line whose first non-whitespace character is `@` and whose remainder, after removing that `@`, is non-empty (¬ß{grammar-appendix.steps}, `interactive_command`). A conforming implementation MUST:

1. Strip the leading `@` when recording the command body.
2. Mark the step as requiring interactive stdio (the precise semantics ‚Äî TTY allocation, stdio inheritance ‚Äî are defined by ¬ß{exec.interactive-drain}).
3. Reject an `@`-only line with no following command text.

The `@` marker is not a lexical prefix (see ¬ß{lexical.line-prefixes} and ¬ß{rationale.at-not-lexical}); it is interpreted only at step-dispatch time, and only inside a recipe body. A bare `@` outside a recipe body is a top-level parse error under ¬ß{toplevel.ordering}.

### Example 8.9.1

```cook
recipe play
    @mkdir -p .cook/scratch
    @printf 'a\nb\nc\n' | fzf > .cook/scratch/pick
    echo "picked: $(cat .cook/scratch/pick)"
```

The first two steps are interactive commands: the runtime inherits the user's terminal so that `fzf` can draw a full-screen prompt. The third is a plain `shell_command` whose output is captured as the execution model dictates.

### Note 8.9.1

`@` embedded inside a `cook` step's `{ ... }` shell-block body, or elsewhere inside a quoted string, is a literal `@` and does not trigger interactive-step dispatch. The `@` test runs only on the first non-whitespace character of a raw `Content` line. See parser test `test_at_in_cook_body_is_not_interactive`.

### Note 8.9.2

An empty interactive command ‚Äî a line consisting only of `@` ‚Äî is rejected with a diagnostic stating that `@` requires a command. See parser test `test_empty_interactive_step_errors`.

## 8.10. Cross-recipe per-member output accessor [#steps.cross-recipe-member]

Inside an `ingredients <probe>` fan-out body (¬ß{steps.ingredients}), a placeholder of the form `$<NAME[]>` ‚Äî where `NAME` is a recipe declared in the same Cookfile ‚Äî resolves to that recipe's terminal output **for the current fan-out member**. It is the per-member counterpart of bare `$<NAME>` (which substitutes all terminal outputs of `NAME`, space-joined, and is unchanged by this feature).

`$<NAME[]>` performs three jobs simultaneously:

1. **Path substitution.** At register time the placeholder is replaced by the member-specific output path: `cook.dep_output_member("NAME", cook.member_to_string(item))`, where `item` is the fan-out loop variable.
2. **Recipe-level DAG ordering edge.** The compiler extracts `$<NAME[]>` as an ordering constraint identical to the edge produced by a bare `$<NAME>` reference: `NAME` must build before the referencing recipe.
3. **Per-member fingerprint fold.** The referenced output is folded into the current member's cache fingerprint, not the recipe's aggregate fingerprint. Editing a single upstream member (e.g. one `.blend` file changing `render`'s output for `"outro"`) rebuilds only the `mux` member that depends on it.

**Same-probe precondition (v1).** Consumer and producer must iterate the same probe. The `[]` accessor indexes into the producer's member map using the consumer's current `item` key; if the two recipes are driven by different probes, the key may not exist. Cross-probe joins via declared alignment keys are deferred to COOK-99 and MUST NOT be attempted in this version. A conforming implementation that detects a same-probe violation at register time SHOULD emit a diagnostic; otherwise the missing-key lookup produces a register-phase error at `cook.dep_output_member`.

**Fan-out body only.** `$<NAME[]>` MUST NOT appear outside an `ingredients <probe>` fan-out body. A conforming implementation MUST reject it with a load-time error if no `ingredients <probe>` driver is present in the recipe.

**Relation to `$<NAME>`.** Both forms record a recipe-level dependency on `NAME`. `$<NAME>` substitutes the full output list (useful in a many-to-one step that needs all of `NAME`'s artifacts); `$<NAME[]>` substitutes only the current-member output (useful in a fan-out step that must stay per-member). The two forms MAY coexist in the same Cookfile but MUST NOT coexist in the same step unless `NAME` differs between the two placeholders.

### Example 8.10.1

```cook
probe scenes
    json { echo '[{"id":"intro"},{"id":"outro"}]' }

recipe render
    ingredients scenes
    cook "build/$<in.id>.silent.mp4" { ffmpeg -i $<in.id>.blend $<out> }

recipe tts
    ingredients scenes
    cook "build/$<in.id>.wav" { tts $<in.id> $<out> }

recipe mux
    ingredients scenes
    cook "build/$<in.id>.mp4" { ffmpeg -i $<render[]> -i $<tts[]> $<out> }
```

For member `{"id":"intro"}`, `mux` builds `build/intro.mp4` by combining `build/intro.silent.mp4` (from `render`) and `build/intro.wav` (from `tts`). For member `{"id":"outro"}` the same pattern repeats independently. Only the `outro` member's units are dirty when `outro.blend` changes ‚Äî `intro.mp4` is unaffected.

### Note 8.10.1 ‚Äî Conformance fixture

`standard/conformance/positive/recipe-member-accessor` pins this surface. The `parse.txt` golden confirms that `$<render[]>` and `$<tts[]>` are preserved verbatim in the `ShellBlock` strings at the parser layer; the codegen layer (CS-0098) lowers them to `cook.dep_output_member` calls.
