# 4. Recipes and step kinds [#recipes]
> **Normative.** This chapter defines the structure of a recipe body and the meaning of each step kind. The productions referenced here are given formally in Appendix A §{grammar-appendix.recipes} and §{grammar-appendix.steps}; where this chapter's prose disagrees with Appendix A, Appendix A is authoritative (§{notation.grammar-precedence}).

## 4.1. Recipe header [#recipes.header-forms]
A recipe is introduced by a **recipe header** of the form `recipe name (":" dependency_list)?`, where `name` is either a `BARE_IDENTIFIER` or a double-quoted `STRING` (App. A.5, `name`). An optional `:` introduces a dependency list (§{recipes.dep-list}).

The keyword `recipe` MUST be followed by a separator — whitespace, a tab, or a `"` character — before the name. An identifier whose first word merely begins with `recipe` (for example, `recipes_cleanup`) is a `Content` token, not a header (§{lexical.line-classification}).

The reserved-segment rule of §{grammar-appendix.declarations} applies: a conforming implementation MUST reject a recipe whose name's final dot-separated segment is one of `stem`, `name`, `ext`, `dir`, `in`, `out`, `all`.

### Example 4.1.1

```cook
recipe "build": "setup"
    gcc -o main main.c

recipe build: setup
    gcc -o main main.c
```

Both recipes above are semantically identical; the first uses a quoted name, the second a bare name.

### Note 4.1.1

The token `recipe` is recognised as a top-level keyword only when followed by a separator (space, tab, or `"`). An identifier whose first word merely begins with `recipe` (for example, `recipes_cleanup`) is a `Content` token, not a header. See lexer test `test_recipe_prefix_is_shell_command`.

## 4.1.1. Body termination [#recipes.termination]
A `recipe_body` (§{grammar-appendix.recipes}) and a `config_body` (§{grammar-appendix.declarations}) both extend from the line following the introducing header up to, but not including, the next **terminating line** of the surrounding Cookfile. A line is a terminating line when it begins at column 0 (no leading whitespace) and its first whitespace-separated token is one of the top-level keywords `recipe`, `chore`, `config`, `use`, `import` (§{lexical.keywords}). End-of-file is also a terminating boundary.

A conforming implementation MUST consume only the source lines strictly between the header and the next terminating line (or EOF) as the body. The terminating line itself is parsed independently as the next `toplevel_item`. There is no closing keyword.

Comments and blank lines inside a body are part of the body (per the existing recipe-body and config-body conventions) and do not terminate it. An indented `recipe foo` or `chore foo` line inside a recipe body is not a terminating line — the column-0 requirement applies — and falls through to the body's step-dispatch cascade as a `shell_command` (§{recipes.shell-steps}).

A bare `end` line is not a terminating line. It is a `Content` token: at top level it is rejected (§{grammar.overview}); inside a recipe body it dispatches as a `shell_command` whose body text is `end` (§{recipes.shell-steps}).

### Example 4.1.1.1

```cook
recipe build
    gcc -o main main.c

recipe test
    ./main --selftest
```

The `recipe build` body extends from the `gcc ...` line up to (but not including) the column-0 `recipe test` line. The `recipe test` body extends from `./main ...` to EOF. No `end` lines appear.

### Example 4.1.1.2

```cook
recipe legacy
    end
    echo done
```

The bare `end` on line 2 is a `Content` token (§{lexical.line-classification}). The step-dispatch cascade (§{recipes.step-kinds}) classifies it as a `shell_command` whose body text is the literal command `end`. The recipe body has two shell steps; the recipe terminates at EOF.

### Note 4.1.1.1

The terminator-set is the column-0 top-level keyword set fixed by §{lexical.keywords}, plus end-of-file. The set is closed within a single Standard version; a future revision that adds a new top-level keyword extends this set in lockstep.

## 4.2. Dependency list [#recipes.dep-list]
A **dependency list** is a sequence of one or more `name`s following the `:` in a recipe header (§{grammar-appendix.recipes}, `dependency_list`). Each name in a dependency list is a name reference; its resolution rules are given in §{xref.resolution} and its effect on execution ordering in §{xref.dep-implications} and §{exec.cross-recipe-ordering}.

A conforming implementation MUST accept a mix of `BARE_IDENTIFIER` and `STRING` forms within a single dependency list. Dependencies MAY use the dotted form (e.g. `backend.build`) to refer to a recipe imported under an alias (§{modules}); the dotted form is a single `BARE_IDENTIFIER` under the token rules of §{lexical.identifiers}, not two tokens separated by `.`.

The dependency list is purely syntactic at parse time. The Standard requires only that:

1. Each referenced recipe exists in the resolved Cookfile (itself, its `use` modules, or its `import` aliases). A conforming implementation MUST reject a Cookfile that references a non-existent recipe — this diagnostic MAY be deferred to the first resolution pass (§{exec}).
2. The dependency graph is acyclic. A conforming implementation MUST reject a Cookfile whose dependency graph contains a cycle — this diagnostic MAY be deferred to DAG construction (§{exec}).

### Example 4.2.1

```cook
recipe tools: colorize wordcount reverse

recipe bundle: "backend.build" "frontend.build"
```

The first recipe lists three dependencies as bare identifiers. The second uses quoted strings for the same effect; it is semantically identical to `recipe bundle: backend.build frontend.build`.

## 4.3. Ingredients [#recipes.ingredients]
A recipe body MAY contain at most one `ingredients_step` (§{grammar-appendix.recipes}, "At-most-one-ingredients rule"). A second `ingredients` line within the same recipe body MUST be rejected.

The `ingredients_step` is introduced by the keyword `ingredients` followed by one or more `ingredient` items (§{grammar-appendix.steps}, `ingredients_step`). Each item is either:

| Form | Meaning |
|---|---|
| `STRING`        | **include**: the quoted text is a glob pattern added to the ingredient set. |
| `!` `STRING`    | **exclude**: the quoted text is a glob pattern subtracted from the ingredient set. |

A conforming implementation MUST accept an `ingredients` line containing only excludes, only includes, or a mix of the two in any order. The semantic effect is the set difference `(union of includes) \ (union of excludes)`; ordering of items on the line does not affect the result.

The glob language itself — which characters are wildcards, whether `**` matches across directory separators, and so on — is not defined by §{recipes} and is resolved by the execution model (§{exec}).

### Example 4.3.1

```cook
recipe lib
    ingredients "src/*.c" "include/*.h" !"src/lua.c" !"src/luac.c"
    cook "build/libmath.a" using "ar rcs {out} {all}"
```

The ingredient set is every `.c` file in `src/` and every `.h` file in `include/`, minus `src/lua.c` and `src/luac.c`.

### Note 4.3.1

An `ingredient` item is always a `STRING` (quoted). A bare word after `ingredients` is a syntax error. The `!` exclusion marker MUST immediately precede the opening `"` with no intervening whitespace: `! "foo"` is rejected; `!"foo"` is accepted. See parser tests `test_ingredients_with_excludes` and `test_ingredients_excludes_only`.

## 4.4. Step kinds — overview [#recipes.step-kinds]
A `recipe_body` is a sequence of `recipe_item`s (§{grammar-appendix.recipes}). Excluding the `ingredients_step` covered in §{recipes.ingredients} and the structural `comment` / `NEWLINE` items, a conforming implementation MUST recognise exactly the following nine step kinds:

| § | Step kind | Introduced by | Phase |
|---|---|---|---|
| 4.5, 4.6 | `cook_step`           | keyword `cook` | register |
| 4.7      | `plate_step`          | keyword `plate` | register |
| 4.8      | `test_step`           | keyword `test` | register |
| 4.9      | `inline_lua_line`     | `>>` line prefix (§{lexical.line-prefixes}) | register |
| 4.9      | `inline_lua_block`    | `>>{` line prefix (§{lexical.line-prefixes}) | register |
| 4.9      | `lua_line`            | `>` line prefix (§{lexical.line-prefixes}) | execute |
| 4.9      | `lua_block`           | `>{` line prefix (§{lexical.line-prefixes}) | execute |
| 4.10     | `shell_command`       | default Content line | execute |
| 4.10     | `interactive_command` | `@` as the first non-whitespace character of a Content line | execute |
| 4.11     | `module_call`         | Content line matching the pattern of §{grammar-appendix.steps}, `module_call` | register |

The phase column classifies each step kind by **when its body is evaluated**: register-phase steps run during recipe registration (§{exec.two-phase}), execute-phase steps run when scheduled work units fire. A normative cross-section table appears in §{exec.phase-classification}.

A `Content` token inside a recipe body is classified into exactly one of the keyword-introduced kinds using the step-dispatch priority of §{grammar-appendix.steps} ("Step-dispatch priority"), reproduced here for convenience:

1. `ingredients` + separator → `ingredients_step` (§{recipes.ingredients}).
2. `cook` + separator → `cook_step` (§{recipes.cook-single-output}, §{recipes.cook-multi-output}).
3. `plate` + separator → `plate_step` (§{recipes.plate-step}).
4. `test` + separator → `test_step` (§{recipes.test-step}).
5. Matches the module-call pattern of §{grammar-appendix.steps} → `module_call` (§{recipes.module-call-steps}).
6. First character is `@` and the remainder is non-empty → `interactive_command` (§{recipes.shell-steps}).
7. Otherwise → `shell_command` (§{recipes.shell-steps}).

The four Lua-prefix forms (`>>`, `>>{`, `>`, `>{`) are not subject to this cascade: they are selected at the lexical layer (§{lexical.line-prefixes}) and always produce `inline_lua_line`, `inline_lua_block`, `lua_line`, or `lua_block` steps respectively, regardless of what follows the prefix.

### Note 4.4.1

The "separator" after a step keyword is space, tab, or end of token. A Content line whose first token is `cooking` is a `shell_command`, not a malformed `cook_step`. See parser test `test_strip_keyword`.

### Note 4.4.2 — Recipe-body region rule

A `recipe_body` MUST be ordered into two consecutive regions, each of which MAY be empty:

1. **Declarative region** — register-phase steps: `ingredients_step`, `cook_step`, `plate_step`, `test_step`, `inline_lua_line`, `inline_lua_block`, `module_call`, plus structural `comment` and blank lines.
2. **Imperative region** — execute-phase steps: `lua_line`, `lua_block`, `shell_command`, `interactive_command`, plus structural `comment` and blank lines.

Once any imperative-region step appears in the body, no declarative-region step is permitted afterward in the same recipe. A conforming implementation MUST reject a recipe in which a declarative-region step (cook, plate, test, ingredients, `>>`, `>>{`, or a module call) appears after the first imperative-region step (`>`, `>{`, shell, or `@`). The diagnostic MUST identify the offending step's line and the line on which the imperative region began.

Comments and blank lines do NOT trigger the region transition. The transition is governed by the most recent step kind, not the most recent line. See §{rationale.recipe-body-regions} for the rationale.

## 4.5. `cook` step — single-output form [#recipes.cook-single-output]
A `cook_step` (§{grammar-appendix.steps}, `cook_step`) declares a build output. The single-output form has exactly one `STRING` output pattern before the optional `using` clause:

```ebnf
cook "out" using_clause?
```

The output is a `STRING` whose interior MAY contain substitution placeholders (e.g. `{stem}`, `{in}`, `{out}`). The placeholder language is defined by the Cook Lua API chapter (§{lua.shell-placeholders}) and by §{xref} for cross-recipe references; §{recipes} treats placeholder text as opaque.

A conforming implementation MUST accept all four of the following forms:

| Form | `using_clause` | Semantic |
|---|---|---|
| `cook "out"`                    | absent                  | declaration-only: announces the output without a build command (§{grammar-appendix.steps}, "Declaration-only cook step"). The build command is expected to be registered by preceding Lua code or supplied by a later amendment. |
| `cook "out" using "cmd"`        | `STRING`                | inline shell command. |
| `cook "out" using >{ ... }`     | `using_lua_block`       | execute-phase Lua block; produces the build effect, with `inputs` / `outputs` / `input` / `output` bindings per §{lua.using-block-globals}. |
| `cook "out" using { ... }`      | `shell_block`           | sequence of shell commands, one per non-blank line of the block (§{lexical.brace-blocks}, "Shell block line normalisation"). |

After the `using` keyword, the `using`-dispatch order of §{grammar-appendix.steps} ("`using` dispatch order") applies: `>{` is tested before `{`, and `{` before `"`. The first match wins.

### Example 4.5.1

```cook
recipe lib
    ingredients "lib/*.c"
    cook "build/obj/{stem}.o" using "gcc -c {in} -o {out}"
    cook "build/libmath.a" using "ar rcs {out} {all}"
```

The first `cook` is a one-to-one mapping from each ingredient to one object file. The second is a many-to-one mapping producing a single archive from the collected set. Both use the inline `STRING` form of `using`.

### Example 4.5.2

```cook
recipe build
    ingredients "src/*.c"
    cook "bin/app"
    gcc src/main.c -o bin/app
```

The `cook "bin/app"` step is declaration-only. The following `shell_command` supplies the actual build command; the execution model (§{exec}) is responsible for associating it with the declared output.

### Note 4.5.1

The `@` character inside a `using "cmd"` string is preserved verbatim — it is part of the quoted shell template, not an interactive-step marker. The interactive-step dispatch of §{recipes.shell-steps} applies only to a Content line's first non-whitespace character, not to text nested inside a `cook` step's `using` clause. See parser test `test_at_in_cook_using_is_not_interactive`.

## 4.6. `cook` step — multi-output form [#recipes.cook-multi-output]
A `cook_step` MAY declare two or more outputs by listing additional `STRING` patterns before the `using` clause:

```ebnf
cook "a" "b" ... using_clause
```

When two or more outputs are present and a `using_clause` is given, the `using_clause` MUST use the block form — either `using_lua_block` (`>{ ... }`) or `shell_block` (`{ ... }`). A conforming implementation MUST reject the multi-output form with a single-string `using`, i.e. the shape `cook "a" "b" using "cmd"` (§{grammar-appendix.steps}, "Multi-output cook-step rule").

The declaration-only form of §{recipes.cook-single-output} also applies to the multi-output case: a `cook_step` listing two or more outputs with no `using_clause` is accepted and announces the outputs without providing an inline build command. Its build is assumed to be provided by preceding Lua registrations or a later amendment (App. A.4, "Declaration-only cook step").

### Example 4.6.1

```cook
recipe generate
    ingredients "src/*.rs"
    cook "staging/out.js" "staging/out.wasm" using {
        ./fake-generator.sh
        mkdir -p staging
        cp pkg/out.js staging/out.js
        cp pkg/out.wasm staging/out.wasm
    }
```

One `cook` step declares two outputs produced together by a sequence of shell commands. The commands are the four non-blank lines of the shell block, in order (§{lexical.brace-blocks}).

### Example 4.6.2

```cook
recipe build_lua
    ingredients "src/*.rs"
    cook "staging-lua/out.js" "staging-lua/out.wasm" using >{
        os.execute("mkdir -p staging-lua")
        local sources = table.concat(inputs, " ")
        local js = io.open(outputs[1], "w")
        js:write("// generated from " .. sources .. "\n")
        js:close()
        local wasm = io.open(outputs[2], "w")
        wasm:write("BINARY\n")
        wasm:close()
    }
```

The same two outputs are produced by a Lua block. The `inputs` and `outputs` names visible to the block are established by the execution model (§{exec}) and the Cook Lua API (§{lua.using-block-globals}); §{recipes} requires only that a block-form `using` clause be accepted as the body.

### Note 4.6.1

The shape `cook "a" "b" using "cmd"` is rejected with a diagnostic that mentions "block body". The rule exists so that the execution model can always identify, for multi-output steps, which command lines apply to which outputs; a single inline `STRING` cannot express that mapping unambiguously. See parser test `test_multi_output_string_form_rejected`.

## 4.7. `plate` step [#recipes.plate-step]
A `plate_step` (§{grammar-appendix.steps}, `plate_step`) has the shape `plate STRING`. It holds a single shell command template that the execution model applies once per output of a preceding `cook` step in the same recipe body. The mapping from "preceding `cook`" to "which outputs" is defined by §{exec}; §{recipes} requires only that the syntactic shape `plate STRING` be accepted.

The command template is a `STRING` and MAY contain the same placeholder set permitted in `cook` templates (e.g. `{out}`). Placeholder text is opaque at the syntactic layer.

### Example 4.7.1

```cook
recipe run-tests
    ingredients "tests/*.c"
    cook "build/{stem}" using "cc {in} -o {out}"
    plate "./{out}"
```

The `plate` step runs `./<built-test>` once for each output produced by the preceding `cook` step.

### Note 4.7.1

A `plate_step` takes exactly one `STRING`. A `plate` line with zero strings, two strings, or a `using` clause is a syntax error. The deliberate simplicity — one command template, no `using` — is rationalised in §{rationale.recipes}.

## 4.8. `test` step [#recipes.test-step]
A `test_step` (§{grammar-appendix.steps}, `test_step`) has the shape:

```ebnf
test STRING ("timeout" NUMBER)? "should_fail"?
```

The required `STRING` is a shell command template. The optional `timeout NUMBER` and `should_fail` modifiers MUST appear in that order when both are present. A conforming implementation MUST accept all four permutations:

| Form | Behaviour |
|---|---|
| `test "cmd"`                              | Run `cmd`; expect exit status 0, no time bound. |
| `test "cmd" timeout N`                    | Run `cmd`; expect exit status 0 within `N` seconds. |
| `test "cmd" should_fail`                  | Run `cmd`; expect a non-zero exit status, no time bound. |
| `test "cmd" timeout N should_fail`        | Run `cmd`; expect a non-zero exit status within `N` seconds. |

`NUMBER` is the `NUMBER` token class of §{lexical.numbers}: a sequence of one or more decimal digits, interpreted as an unsigned integer number of seconds. The upper bound on representable timeouts is implementation-defined.

### Example 4.8.1

```cook
recipe run-tests
    ingredients "tests/*.c"
    cook "build/{stem}" using "cc {in} -o {out}"
    test "./{out}" timeout 60 should_fail
```

Each test binary is run with a 60-second time bound, and the step expects each to exit with non-zero status.

### Note 4.8.1

`should_fail` is a bare word, not a quoted string. It is recognised only as the trailing modifier on a `test` line; elsewhere it has no syntactic meaning and is not a reserved word (§{lexical.keywords}). See parser tests `test_test_step_with_should_fail` and `test_test_step_with_timeout_and_should_fail`.

## 4.9. Lua steps — line and block forms [#recipes.lua-steps]

The Cookfile language has two families of Lua step. They are distinguished by line prefix and by **execution phase** (§{exec.two-phase}):

| Surface form | Token (§{lexical.line-classification}) | Phase    | Region (§{recipes.step-kinds}, Note 4.4.2) |
|---|---|---|---|
| `>` *body* `\n`           | `LuaLine`              | execute  | imperative |
| `>{` *body* `}` `\n`      | `LuaBlockOpen`         | execute  | imperative |
| `>>` *body* `\n`          | `InlineLuaLine`        | register | declarative |
| `>>{` *body* `}` `\n`     | `InlineLuaBlockOpen`   | register | declarative |

### 4.9.1. Delimitation [#recipes.lua-steps-delimitation]

A `lua_line` step's body is the remainder of the line after `>` and an optional single space; the body MAY be empty (App. A.5, "`lua_line` MAY be empty"). An `inline_lua_line` step's body is the remainder of the line after `>>` and an optional single space; it too MAY be empty.

A `lua_block` step's body is the source text between the opening `>{` and the matching `}`, collected by the brace-balance algorithm of §{lexical.brace-blocks}. An `inline_lua_block` step's body is the source text between the opening `>>{` and the matching `}`, by the same algorithm. The closing `}` line is consumed but is not part of either block's body.

A conforming implementation MUST NOT reinterpret the Lua body text. Comments, strings, and other Lua constructs inside the body are the Lua runtime's concern, with the single exception that the brace-balance algorithm (§{lexical.brace-blocks}) MUST ignore braces appearing inside Lua line comments, double-quoted strings, single-quoted strings, and long strings (`[[...]]`) when locating the closing `}` of a Lua block (either form).

### 4.9.2. Execution phase [#recipes.lua-steps-execution-phase]

`inline_lua_line` and `inline_lua_block` are **register-phase** steps. Their body is evaluated during recipe registration in the recipe-body Lua function. Multiple inline-Lua steps in one recipe share register-phase scope: a `local` declared in one is visible to the next, including across the surface forms of `cook` / `plate` / `test` / `ingredients` (which compile to register-phase Lua API calls). This is the surface form a Cookfile uses to call Cook Lua API functions that record work — `cook.add_unit`, `cook.export`, `cook.step_group`, and so on — and to declare register-phase locals consumed by subsequent declarative steps.

`lua_line` and `lua_block` are **execute-phase** steps. Their bodies are not evaluated during registration. They are recorded as part of a **body unit** (§{recipes.body-bundling}) whose `lua_code` payload runs on a worker VM during the execute phase, after the recipe's declarative work units have completed.

A `module_call` (§{recipes.module-call-steps}) is also register-phase: it desugars to `inline_lua_line` (single-line) or `inline_lua_block` (multi-line). Module-call orchestration uses the Cook Lua API to record work, just as `>>` does directly.

### 4.9.3. Body bundling [#recipes.body-bundling]

The imperative region of a recipe (Note 4.4.2) compiles to a sequence of execute-phase work units, bounded by `interactive_command` steps:

- A run of consecutive imperative-region steps containing zero `interactive_command` steps coalesces into **one body unit**. The implementation MUST record this unit via the Cook Lua API (§{lua.add-unit}) with `lua_code` set to a Lua chunk that is the source-order concatenation of:
  - each `lua_line` body, terminated by `\n`;
  - each `lua_block` body, terminated by `\n`;
  - each `shell_command` body, expressed as a `cook.sh("…")` call (§{lua.cook-sh}). **Adjacent `shell_command` steps coalesce** into a single `cook.sh` call whose argument is the joined commands prefixed by `set -e\n`. A `lua_line` or `lua_block` between two shell lines breaks the coalescence — the second `cook.sh` call is recorded as a fresh process.
- The body unit's `inputs` and `outputs` are empty. Its `cache` flag MUST be `false`.
- Each `interactive_command` step is its own draining work unit (§{exec.interactive-drain}) and breaks the bundle.

The Lua VM persists across the whole body unit. A `local` declared in one `lua_line` is visible to the next `lua_line` of the same bundle, and to the same bundle's coalesced shell calls (which see Lua locals as the surrounding chunk's bindings). The Lua VM does NOT persist across `interactive_command` boundaries: each post-`@` body unit is a fresh VM. Likewise, the coalesced shell process does NOT persist across a `lua_line` / `lua_block` interruption inside the same bundle: a `cd` inside one shell coalescence does not survive to the next shell coalescence.

Failure semantics inside the coalesced shell call match per-line halt-on-failure (the `set -e` prefix). A non-zero exit from any joined shell line aborts the unit. A Lua error from `cook.sh` or any inlined Lua statement raises through the unit and fails the recipe by standard Lua semantics.

### 4.9.4. Examples [#recipes.lua-steps-examples]

```cook
recipe build
    >> local std = cook.env.CXX_STD or "c++20"
    >>{
        cook.env.CXXFLAGS = "-std=" .. std
    }
    cook "build/app" using >{
        cook.sh(string.format("%s %s -o %s",
            cook.env.CXX or "g++", table.concat(inputs, " "), output))
    }
    > print("build complete")
```

The `>>` line and `>>{ }` block run at register, sharing scope with the `cook` step's `using >{ }` template-string formation. The `> print(...)` line runs at execute, after the cook step's unit completes. (The `using >{ }` block is execute-phase Lua-with-bindings; see §6.4.)

```cook
recipe smoke
    cd build
    ./app --selftest
    > if not fs.exists("build/.smoke.ok") then error("smoke failed") end
```

The recipe has only an imperative region: two shell lines and one `>` line. The two shell lines coalesce into one `cook.sh("set -e\ncd build\n./app --selftest")` call, so `cd build` persists into `./app --selftest`. The `>` line then runs in the same Lua VM as the coalesced shell call.

```cook
recipe legacy
    > print("start")
    >{
        if true then
            local t = {1, 2, 3}
            print(#t)
        end
    }
    >
```

The first step is a single-line execute-phase Lua statement. The second is an execute-phase Lua block whose body contains a literal `{` that does not prematurely close the block. The third is an empty `lua_line`. All three coalesce into one body unit (no `@` interruption); the empty third line contributes nothing to the chunk.

### Note 4.9.1

A Lua block (either form) opened on an indented line closes on the line where the running brace-depth first returns to zero, not on the first bare `}`. A line like `    local s = "}"` does not close the enclosing block because the `}` sits inside a double-quoted string (§{lexical.brace-blocks}). See parser tests `test_lua_block_brace_in_string` and `test_lua_block_brace_in_comment`.

### Note 4.9.2

The execute-phase API surface available inside a `lua_line` or `lua_block` is a strict subset of the register-phase API: register-only helpers (`cook.add_unit`, `cook.exec`, `cook.interactive`, `cook.step_group`, `cook.recipe`) MUST raise a Lua runtime error when called from execute-phase Lua. `cook.sh` is available in both phases (§{lua.cook-sh}). See §{rationale.execute-phase-api} for the rationale.

## 4.10. Shell steps — plain and interactive [#recipes.shell-steps]
A `shell_command` is any `Content` line inside a recipe body that is not classified as one of the step kinds earlier in the §{recipes.step-kinds} dispatch cascade. Its body is the trimmed source line; a conforming implementation MUST preserve the body text verbatim, including any internal quoting, variable references, and shell metacharacters.

An `interactive_command` is a `Content` line whose first non-whitespace character is `@` and whose remainder, after removing that `@`, is non-empty (§{grammar-appendix.steps}, `interactive_command`). A conforming implementation MUST:

1. Strip the leading `@` when recording the command body.
2. Mark the step as requiring interactive stdio (the precise semantics — TTY allocation, stdio inheritance — are defined by §{exec}).
3. Reject an `@`-only line with no following command text.

The `@` marker is not a lexical prefix (see §{lexical.line-prefixes} and §{rationale.at-not-lexical}); it is interpreted only at step-dispatch time, and only inside a recipe body. A bare `@` outside a recipe body is a top-level parse error under §{grammar.top-level-ordering}.

### Example 4.10.1

```cook
recipe play
    @mkdir -p .cook/scratch
    @printf 'a\nb\nc\n' | fzf > .cook/scratch/pick
    echo "picked: $(cat .cook/scratch/pick)"
```

The first two steps are interactive commands: the runtime inherits the user's terminal so that `fzf` can draw a full-screen prompt. The third is a plain `shell_command` whose output is captured as the execution model dictates.

### Note 4.10.1

`@` embedded inside a `cook` step's `using "..."` template, or elsewhere inside a quoted string, is a literal `@` and does not trigger interactive-step dispatch. The `@` test runs only on the first non-whitespace character of a raw `Content` line. See parser test `test_at_in_cook_using_is_not_interactive`.

### Note 4.10.2

An empty interactive command — a line consisting only of `@` — is rejected with a diagnostic stating that `@` requires a command. See parser test `test_empty_interactive_step_errors`.

## 4.11. Module-call steps [#recipes.module-call-steps]
A `module_call` is a `Content` line inside a recipe body whose shape matches the pattern given in §{grammar-appendix.steps}, `module_call`:

1. The first character is a `BARE_IDENTIFIER` start character — `[A-Za-z_]`.
2. A run of `[A-Za-z0-9_]` follows (alphanumerics and underscore; dots and hyphens are **not** permitted in the first segment).
3. A literal `.` follows.
4. A character matching `[A-Za-z_]` follows the dot.

The pattern is a **heuristic on the first two tokens only**. The remainder of the line is not validated at dispatch time; its well-formedness as a Lua expression is the Lua runtime's concern. When the first-line text contains an open `{` whose depth does not return to zero on that line, the step text MUST be extended to subsequent source lines until brace depth balances, using the same brace-balance algorithm as §{lexical.brace-blocks}. An unterminated brace balance MUST produce a parse error.

A conforming implementation MUST emit a single-line `module_call` as an `inline_lua_line` step (§{recipes.lua-steps}) and a multi-line `module_call` as an `inline_lua_block` step. The resulting step body is the original source text, with newlines preserved across lines in the multi-line case. Both targets are register-phase: a `module_call` runs during recipe registration, where its body typically calls Cook Lua API functions to record work units. (See §{rationale.module-call-register} for why module calls default to register-phase rather than execute-phase.)

### Example 4.11.1

```cook
recipe build
    cpp.compile("src/*.cpp")
    cpp.compile {
        sources = "src/*.cpp",
        output_dir = "build/obj/",
    }
```

Both steps are module calls. The first is single-line and becomes a `lua_line`. The second spans four source lines because the opening `{` on the first line is not balanced until the closing `}` on the fourth; it becomes a `lua_block` whose body is those four lines joined by newlines.

### Note 4.11.1

A leading `.` (as in `./run.sh`) fails rule 1 of the pattern and falls through to `shell_command`. A hyphenated first token (as in `cargo-build.release`) fails rule 2 — hyphens are permitted inside a `BARE_IDENTIFIER` generally, but the module-call heuristic restricts the first segment to `[A-Za-z0-9_]` to avoid swallowing hyphenated shell commands. See parser tests `test_non_module_dot_is_shell` and `test_module_call_multiline`.

### Note 4.11.2

The rationale for dispatching module calls by heuristic rather than by parsing the full Lua expression is discussed in §{rationale.recipes}.
