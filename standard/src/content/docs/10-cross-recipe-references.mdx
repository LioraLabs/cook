# 10. Cross-recipe references [#xref]
> **Normative.** This chapter defines *name references*: the surface by which one recipe refers to another. It covers three syntactic forms — dep-list references (`recipe A: B`), bare placeholders (`$<B>`), and accessor placeholders (`$<B.stem>`) — and specifies their resolution, expansion, and effect on the dependency graph. The formal grammar of the placeholders is given in App. A; this chapter specifies their meaning.

## 10.1. Name references [#xref.name-references]
A **name reference** is the sole mechanism by which a recipe refers to another recipe. A name reference MAY appear in one of three surfaces:

| Surface | Location | Form |
|---|---|---|
| Dep-list | Recipe header `: …` (§{recipes.dep-list}) | `NAME` |
| Bare placeholder | Any step template string | `$<NAME>` |
| Accessor placeholder | `cook` step output pattern (§{steps.cook-single}, §{steps.cook-multi}) | `$<NAME.ACCESSOR>` |

All three surfaces share the resolution rules of §{xref.resolution} and the dependency implications of §{xref.dep-implications}. Expansion behaviour differs by surface: dep-list references have no template expansion; bare placeholders expand per §{xref.string-substitution}; accessor placeholders drive iteration per §{xref.dep-driven}.

## 10.2. Resolution [#xref.resolution]
A conforming implementation MUST resolve a placeholder `$<TOKEN>` (defined in §{phl.token}) by applying these steps in order; first match wins:

1. **Builtin.** If `TOKEN` matches one of the builtin shapes from §{phl.cook-step}, the placeholder substitutes per the builtin's row. Builtin validity is also checked against the step's iteration mode and declared output count; mismatches MUST produce a load-time diagnostic. The number `N` in `out_N` MUST be in the range 1..=K where K is the step's declared output count; `out_0` and `out_N` for N > K MUST produce a load-time diagnostic.

2. **Recipe in scope.** If `TOKEN` is the name of a recipe reachable from the current Cookfile (own recipes, recipes imported via `use`, or recipes reachable through sigil-imported Cookfiles per §7), the placeholder substitutes to that recipe's terminal output(s) per §{xref.string-substitution}. If `TOKEN` has the shape `RECIPE.ACCESSOR` where `ACCESSOR ∈ {stem, name, ext, dir}`, the accessor applies to the recipe's single output via §{xref.path-accessors}. Multi-output recipes accessed via accessor MUST produce a load-time diagnostic. The set of recipes in scope is the full register-phase recipe set: it includes recipes registered during the register phase by a top-level `module_call` (a target maker such as `cook_cc.bin("x")`, §{toplevel.module-call}), not only recipes declared with a native `recipe` block. Because target makers register their recipes during the register phase, a conforming implementation MAY determine recipe-in-scope membership for this step at register time — after register-phase recipe registration completes — on the same footing as the register-time allowance in step 3; an implementation MUST NOT restrict this step to the statically parsed `recipe` blocks, which would mis-resolve `$<x>` to step 3 (or step 4) for a module-registered recipe `x`.

3. **Declared env var.** If `TOKEN` (or `env.TOKEN` per the explicit form defined in §{xref.env-namespace}) appears as `env.TOKEN = ...` in a config block reachable from this Cookfile's config-block scope, the placeholder substitutes to `cook.env["TOKEN"]`. The implementation MUST capture the consulted key in the unit's `consulted_env_keys` for cache invalidation. The "declared" check MAY be implemented at register time (i.e., after config-block evaluation captures the keyset).

4. **Hard error.** A conforming implementation MUST emit a load-time diagnostic for any placeholder that fails to resolve under steps 1–3. The diagnostic MUST name the unresolved TOKEN and SHOULD enumerate declared env vars and recipes in scope. Lexically valid IDENT shapes that no resolution rule handles (e.g., `$<foo.>` with trailing dot, `$<A.B.C>` with multiple unrecognized dotted segments) fall under this step.

This is a closed function: every well-lexed placeholder either resolves to a known thing or fails loudly. There is no silent fallthrough — `$<UNKNOWN>` against an undeclared env var MUST NOT produce an empty-string substitution.

A name reference appearing in a dep-list is resolved against the recipe names in scope with no accessor split: the whole string is the recipe name. Qualified cross-Cookfile dep-list references (`alias.recipe`) follow the resolution rule of §{comp}.

### 10.2.1. Dotted name references [#xref.dotted-names]
Recipe and chore names declared in a Cookfile MUST NOT contain `.` (§{lexical.identifiers}, App. A.2). A dotted name reference therefore never names a recipe defined in the current Cookfile; it is always either:

- a **qualified cross-Cookfile reference** of the form `alias.recipe`, resolved against the importing Cookfile's `import` aliases per §{comp.qualified-refs}; or
- a **path-accessor placeholder** of the form `RECIPE.ACCESSOR` (where `ACCESSOR ∈ {stem, name, ext, dir}`), per step (2) of §{xref.resolution}.

A conforming implementation resolves a dotted `$<TOKEN>` by first attempting the path-accessor split (the rightmost dot separates `RECIPE` from `ACCESSOR`); if `ACCESSOR` is one of the four path-accessor names and `RECIPE` resolves to a recipe in scope, the placeholder is an accessor reference. Otherwise the leftmost dot is taken as an import-alias separator and the reference is resolved per §{comp.qualified-refs}. A token that satisfies neither shape MUST produce a load-time diagnostic per step (4) of §{xref.resolution}.

Because no dotted name can resolve to a recipe defined in the current Cookfile, there is no whole-token-vs-split ambiguity to adjudicate.

### 10.2.2. Reserved final segment [#xref.reserved-segment]
A recipe's final dot-separated segment MUST NOT equal a path accessor (§{xref.path-accessors}) or `in` or `out`. A conforming implementation MUST reject a Cookfile that declares such a name, at parse time. The rule is enforced by the `check_reserved_recipe_name` production in App. A. `all` is NOT among the reserved segments (CS-0130).

### 10.2.3. Environment-variable shadowing [#xref.env-shadowing]
When step (2) or (3) succeeds, step (4) is NOT consulted. If `cook.env[TOKEN]` is also defined for the same `TOKEN`, the recipe name wins. A conforming implementation MUST emit a warning naming both.

## 10.3. Path accessors [#xref.path-accessors]
A **path accessor** is one of the four names `stem`, `name`, `ext`, `dir`. For a path `p`, the accessors are defined by the `path.*` helpers of §{lua.path-helpers}:

| Accessor | Equivalent |
|---|---|
| `stem` | `path.stem(p)` (§{lua.path-stem}) |
| `name` | `path.name(p)` (§{lua.path-name}) |
| `ext`  | `path.ext(p)` (§{lua.path-ext}) |
| `dir`  | `path.dir(p)` (§{lua.path-dir}) |

The set of path accessors is closed: an accessor not in this list is not a path accessor for the purposes of §{xref.resolution} or §{phl.cook-step}. Extending the set is a change to this Standard, not a library change. See rationale B.5.P.

## 10.4. Dep-driven iteration [#xref.dep-driven]
Rule. When a `cook` step's output pattern list contains a `$<NAME.ACCESSOR>` accessor placeholder naming recipe `lib`, the step's iteration driver is `lib`'s output list, not this recipe's own input list.

- **Iteration shape.** The step is a **one-to-one step (§{steps.cook-single})**. For each item `x` in `lib`'s output list (§{xref.dep-recipe-output}), one work unit is registered. Within that unit, the iteration item is `x`, and the dotted-accessor placeholders `$<in.ACCESSOR>` refer to `x`. A placeholder `$<lib.ACCESSOR>` in the same step refers to the same accessor of `x`.
- **One driver per step.** A step's output pattern list MAY contain accessor placeholders referring to AT MOST ONE distinct recipe. Multiple occurrences referring to the same `lib` (possibly with different accessors) are permitted and refer to the same driver. Accessor placeholders naming two or more distinct recipes in the same step MUST be rejected with a diagnostic.
- **Bare `$<lib>` in an output pattern.** A conforming implementation MUST reject a `cook` step whose output pattern list contains a bare `$<lib>` (no accessor) naming a recipe. `$<lib>` has no iteration semantics.
- **Accessor placeholder in a cook-step body or test body.** A `$<lib.ACCESSOR>` appearing in a `cook` `shell_block` body, a `test` body (shell or Lua), or a bare `shell_command` MUST be rejected with a diagnostic. Test steps have no output pattern and therefore can never declare a `lib`-driven driver; the rejection is universal in those bodies.
- **String-substitution references coexist.** A step MAY contain any number of `{libX}` bare references (§{xref.string-substitution}) to other recipes, orthogonally to whichever recipe (if any) drives iteration. Those references do not participate in the "one driver" rule.

### 10.4.1. A recipe's output [#xref.dep-recipe-output]
A recipe's **output** is the output list of its last step, under the following rule applied per step kind:

| Step kind | Output list |
|---|---|
| `ingredients` | Passthrough: the resolved ingredient paths (§{steps.ingredients}) |
| `cook` (one-to-one) | The per-input outputs recorded at register time |
| `cook` (many-to-one) | The single output path |
| `test` | Passthrough: the step's input list |
| Bare shell (`@` or `@!`) | Passthrough: the step's input list |

"Passthrough" means the step's output equals its input. A recipe with no steps has the empty list as its output.

A recipe assembled from register-phase API work units (a recipe body that registers units via `cook.add_unit` rather than native `cook` steps — the shape a module target maker such as `cook_cc.bin` produces) has, as its output, the output list of its **last registered unit**, by the same "last step" rule. A conforming implementation MUST record that terminal output for cross-recipe resolution (§{xref.resolution} step 2, §{xref.string-substitution}); failing to do so makes `$<recipe>` substitute the empty string for module-registered recipes.

### 10.4.2. Example [#xref.dep-driven-example]
```cook
recipe protos
    ingredients "proto/*.proto"

recipe compile_protos
    cook "build/obj/$<protos.stem>.pb.cc" { protoc --cpp_out=build/obj $<in> }
    cook "build/lib/libprotos.a" { ar rcs $<out> $<in> }
```

The `compile_protos` recipe's first step is dep-driven by `protos`; for each `x` in `protos`'s output list (the resolved globs), one work unit is registered with `$<in>` = `x` and `$<out>` = `build/obj/STEM(x).pb.cc`. The second step chains on the first (§{steps.cook-single}), not on `protos`.

## 10.5. String-substitution references [#xref.string-substitution]
Rule. A `$<NAME>` bare reference in a `cook` shell block, **`test` shell block,** or bare `shell_command` MUST be substituted by the space-joined concatenation of the named recipe's output list (§{xref.dep-recipe-output}). A `$<NAME>` reference inside an `exec_lua_block` is parsed as Lua syntax (a single-element table containing the local variable `NAME`); cross-recipe references in Lua bodies use `cook.dep_output(NAME)` and `cook.dep_output_list(NAME)` per §{comp}.

- **Empty output.** If the named recipe's output list is empty, the substitution is the empty string. A conforming implementation MUST emit a warning at register time, naming the referring recipe and the referent. The reference itself is not an error.
- **Lua lowering.** A conforming implementation MAY lower a `$<NAME>` reference to `cook.dep_output("NAME")` and a `$<NAME.ACCESSOR>` iteration to a `cook.dep_output_list("NAME")`-driven form. The Standard specifies the observable behaviour; the Lua forms are one conforming lowering.

### 10.5.1. Example [#xref.string-substitution-example]
```cook
recipe libmath
    ingredients "src/math/*.c"
    cook "build/obj/math/$<in.stem>.o" { gcc -c $<in> -o $<out> }
    cook "build/lib/libmath.a" { ar rcs $<out> $<in> }

recipe app
    cook "build/bin/app" { gcc -o $<out> main.c $<libmath> }
```

The `$<libmath>` reference in `app`'s shell body expands at register time to `"build/lib/libmath.a"` (the single-element output of `libmath`).

## 10.6. Dependency implications [#xref.dep-implications]
- **Name reference ⇒ cross-recipe edge.** A name reference to recipe `B` appearing in any step or dep-list of recipe **or chore** `A` establishes a cross-recipe dependency from `A` to `B`, observable through execution ordering (§{exec.phases}). In particular, `B` MUST join the build closure of any invocation that reaches `A`: a chore step referencing `$<B>` runs only after `B` has built, with no explicit `: B` dep-list entry required (CS-0121).
- **Granularity.** A dep-list reference (`recipe A: B`) MUST be interpreted at whole-recipe granularity: every unit of `A` waits for every unit of `B`. A placeholder reference (`$<B>` or `$<B.ACCESSOR>`) in a specific step of `A` MAY be refined by the implementation to a per-unit edge from that step's unit(s) to the unit(s) of `B` that produced the referenced output items. Coarse edges are always conformant; fine-grained refinement is an optimisation available to placeholder-sourced edges only.
- **Validation and cycles.** An implementation MUST validate that the named recipe is defined in scope before the execute phase begins, and MUST reject the Cookfile with a cycle diagnostic if the transitive closure of all cross-recipe edges contains a cycle, before executing any work unit.
- **Only name references create edges.** A string equal to another recipe's output path does NOT establish a cross-recipe edge, wherever that literal appears: in an `ingredients` line, in a `cook` step's output pattern, or in the `inputs[]` / `outputs[]` entries of `cook.add_unit` (§{lua.add-unit}). All are literal path surfaces and none of them are name references. An implementation MUST NOT infer an edge from path-string equality on any of them. Authors who want "run this after that" without a name reference MUST use an explicit `: dep`. Note that not inferring an edge is not the same as staying silent: where such a match would otherwise become a silent read-after-write race, and the implementation can detect it from the source alone, the implementation rejects the build with a diagnostic rather than synthesising the missing ordering — see §{exec.read-after-write}, which states that rule and enumerates the surfaces it covers. That enumeration is narrower than this prohibition: this bullet binds every literal surface listed above, whether or not a match on it is detectable. See rationale B.5.N.

## 10.7. The `env.` placeholder namespace [#xref.env-namespace]

> **Normative.**

A placeholder `$<TOKEN>` whose `TOKEN` begins with `env.` is reserved as the explicit env-var-lookup form. For such a placeholder, the implementation MUST resolve `TOKEN` by stripping the `env.` prefix and looking up the remaining identifier as a declared env var per §{xref.resolution} step 3, regardless of whether a recipe of the same name exists in scope.

The reserved namespace prevents resolution ambiguity. A Cookfile that declares both a recipe named `foo` and an env var `env.foo` will see bare `$<foo>` resolve to the recipe (per the rule order) and `$<env.foo>` resolve to the env var. The recipe-naming rules of §{lexical.keywords} forbid `env` as either the first or last dotted segment of a recipe name, so a recipe cannot be reached via `$<env.X>` form by construction.

## 10.8. The `file:` placeholder namespace [#xref.file-namespace]

> **Normative.**

A placeholder `$<TOKEN>` whose `TOKEN` begins with `file:` is reserved as the file-reference form (CS-0101, §{phl.cook-step}). A conforming implementation MUST dispatch on this prefix before every step of §{xref.resolution} and before the probe-value colon dispatch of §{cat.probes} — a `file:`-prefixed token never resolves as a builtin, recipe, env var, or probe value. Consequently a probe registered under a key beginning with `file:` is not addressable from the sigil surface; implementations SHOULD warn when `cook.probe` is called with such a key. `PATH` MUST be relative: a leading `/` or any `..` segment MUST produce a load-time diagnostic.
