# 6. Cook Lua API [#lua]
> **Normative.** This chapter defines the Cook API surface available to Lua code inside a Cookfile. Code encountered via `> expr` (§{recipes.lua-steps}), `>{ ... }` (§{lexical.brace-blocks}, §{recipes.lua-steps}), `using >{ ... }` (§{recipes.cook-single-output}, §{recipes.cook-multi-output}), module bodies (§{lua.use-env}), and `config` blocks (§{grammar.top-level-ordering}) runs with this API in scope.

The Cook runtime executes Lua code in two disjoint phases:

- The **register phase** runs the Lua image produced from the Cookfile exactly once on a single-threaded Lua VM. Its purpose is to record a set of work units without performing external effects. API functions in this phase typically return synchronously and mutate capture state.
- The **execute phase** runs the recorded work units on a pool of worker Lua VMs (one per worker thread). Its purpose is to perform the external effects previously recorded.

Every function defined in this chapter is classified as **register-phase only**, **execute-phase only**, or **both phases**. A conforming implementation MUST install a function in each of the phases indicated here, and MUST NOT silently install a register-phase-only function in the execute phase (or vice versa) — a Cookfile that calls a function in the wrong phase MUST observe either the documented behaviour for that phase or a Lua runtime error.

## 6.1. API surface overview [#lua.api-overview]
A conforming implementation MUST install the following top-level Lua globals before any Cookfile-derived Lua code runs:

| Global    | Type  | Phases        | Defined in |
|-----------|-------|---------------|------------|
| `cook`    | table | both          | §{lua.add-unit}, §{lua.step-helpers} |
| `fs`      | table | both          | §{lua.fs-helpers} |
| `path`    | table | both          | §{lua.path-helpers} |
| `recipe`  | table | register      | §{lua.recipe-global} |
| `inputs`  | table | execute       | §{lua.using-block-globals} |
| `outputs` | table | execute       | §{lua.using-block-globals} |
| `input`   | string | execute      | §{lua.using-block-globals} |
| `output`  | string | execute      | §{lua.using-block-globals} |
| `input_N` | table | execute       | §{lua.using-block-globals} |

The `cook` table carries recipe registration, work-unit registration, shell helpers, dependency-output accessors, module loading, platform information, export/import, and environment access. The full set of keys on the `cook` table is implementation-defined except for the members that are normatively specified in this chapter.

### 6.1.1. The `recipe` global [#lua.recipe-global]
During the register phase, and only within the body of the Lua function registered by `cook.recipe(...)` (§{modules}), the `recipe` global is bound to a table with the following fields:

- `recipe.name` — string; the declared recipe name (§{recipes.header-forms}).
- `recipe.ingredients` — a sequence (1-indexed Lua table) of ingredient groups. `recipe.ingredients[i]` is itself a sequence of relative file paths resolved from the `i`-th `ingredients` glob pattern, minus any entries matching an exclude pattern (§{recipes.ingredients}).

A conforming implementation MUST NOT expose the `recipe` global during the execute phase; it is a register-phase artefact.

### 6.1.2. The `env` alias inside `config_block` bodies [#lua.env-alias]

Within the body of a `config_block` (§{grammar.config-blocks}), the bare global `env` MUST be bound such that `env` and `cook.env` refer to the same table. Writes through either name MUST be observable through both. The `env` alias is in scope only within `config_block` bodies; it MUST NOT be bound in recipe-body Lua, in `using` blocks, in module bodies, or at any other Lua entry point.

## 6.2. `cook.add_unit` [#lua.add-unit]
**Signature.** `cook.add_unit(spec: table) -> nil`
**Phase.** Register-phase only.

`cook.add_unit` records a single work unit to be executed later. The table argument `spec` accepts the following fields:

| Field               | Type           | Default | Meaning |
|---------------------|----------------|---------|---------|
| `command`           | string         | `""`    | Shell command string, run via `/bin/sh -c` in the execute phase. |
| `lua_code`          | string         | absent  | Lua source executed in the execute phase. If present, `command` is ignored. |
| `interactive`       | boolean        | `false` | If `true`, the unit runs on the main thread and drains concurrent work (§{exec.interactive-drain}). |
| `line`              | integer        | `0`     | Source line number used in diagnostics. |
| `cache`             | boolean        | `true`  | If `false`, the unit is never cached and is re-run on every invocation. |
| `inputs`            | array of string | `{}`   | Files whose content contributes to the unit's cache key. |
| `output`            | string         | absent  | Single declared output path (§{recipes.cook-single-output}). |
| `outputs`           | array of string | absent | Multiple declared output paths (§{recipes.cook-multi-output}). |
| `ingredient_groups` | array of arrays of string | `{}` | For Lua-code units, the sequence of ingredient groups exposed as `input_N` (§{lua.using-block-globals}). |

**Output-field exclusivity.** A conforming implementation MUST reject `spec` if both `output` and `outputs` are set, raising a Lua runtime error.

**Payload selection.** A conforming implementation MUST classify the recorded unit as follows:

1. If `lua_code` is present, the unit is a **Lua-code unit**; `inputs`, `outputs`, and `ingredient_groups` are carried on the payload and used by §{lua.using-block-globals} bindings at execute time.
2. Otherwise, if `interactive` is `true`, the unit is an **interactive shell unit**.
3. Otherwise, the unit is a **non-interactive shell unit**.

**Caching.** When `cache` is `true`, a conforming implementation MUST record cache metadata containing the recipe name, an input-path list, an output-path list, and a command hash. The hash inputs, cache-key derivation, and storage format are implementation-defined (§{exec.cache}).

**Return value.** `cook.add_unit` returns nothing. A conforming implementation MUST NOT cause the function to block on external work — registration is synchronous by definition (§{exec.capture-mode}).

### Example 6.2.1

```lua
cook.add_unit({
    inputs  = { "main.c" },
    output  = "main.o",
    command = "gcc -c main.c -o main.o",
})
```

Records one shell unit with one input and one output. The same registration is produced by the surface form `cook "main.o" using { gcc -c {in} -o {out} }` in a single-output `cook` step (§{recipes.cook-single-output}, §{lua.shell-placeholders}).

### Example 6.2.2

```lua
cook.add_unit({
    inputs     = { "src/wasm.rs" },
    outputs    = { "out.js", "out.wasm" },
    lua_code   = "os.execute('wasm-pack build')",
    ingredient_groups = { { "src/wasm.rs" } },
})
```

Records one Lua-code unit with multiple outputs. At execute time, the `lua_code` runs with `inputs`, `outputs`, `input`, `output`, and `input_1` bound as described in §{lua.using-block-globals}.

**Chore-body restriction (§{chores.no-caching}).** When `cook.add_unit` is called from within the register-phase body of a chore, a conforming implementation MUST raise a Lua runtime error if `cache = true` is passed (or if `cache` is absent, since the default is `true`). The codegen layer enforces this at the source level by always emitting `cache = false` for every unit in a chore body; the runtime check is a defence-in-depth guard against inline-Lua (`>>` / `>>{`) that bypasses the codegen path.

A conforming implementation MAY use implementation-defined internal markers to track whether a chore body is executing. In the reference implementation, the generated Lua calls `cook._enter_chore()` before the body and `cook._exit_chore()` after; `cook.add_unit` checks an internal flag. These helpers are an implementation detail and MUST NOT be relied upon by Cookfile authors.

### Note 6.2.1

Adjacent `cook.add_unit` calls made inside the same register-phase `cook.step_group(fn)` invocation become siblings in the same scheduler step group (§{exec.step-groups}) and MAY run in parallel. Outside a step group they are serial. The surface `cook` step (§{recipes.cook-single-output}, §{recipes.cook-multi-output}) routes through `cook.step_group` implicitly.

## 6.3. Shell and Lua step helpers [#lua.step-helpers]
The `cook` table exposes the following helpers. Unless stated otherwise, they behave identically in both phases — the register-phase behaviour is defined in terms of capture (§{exec.capture-mode}), and the execute-phase behaviour is defined in terms of actual invocation.

### 6.3.1. `cook.sh` [#lua.cook-sh]
**Signature.** `cook.sh(cmd: string) -> string`
**Phase.** Both.

`cook.sh` executes `cmd` via `/bin/sh -c` in the Cookfile's working directory with the current `cook.env` values merged onto the inherited environment, and returns the captured standard output as a string. A non-zero exit status MUST be reported as a Lua runtime error whose message begins with the token `COOK_CMD_FAILED:`. The function name is intentionally shared between phases because the user-visible behaviour — "run this command and give me its stdout, error on non-zero" — is the same in both; only the surrounding scheduling differs.

In the **register phase**, `cook.sh` is a utility for control-flow decisions during registration (feature detection, tool discovery, reading version files). It is NOT the mechanism by which work units are recorded; use `cook.add_unit` for that. The capture-mode contract of §{exec.capture-mode} applies: a register-phase `cook.sh` call MUST be deterministic with respect to the Cookfile source and the register-phase record accumulated so far, and MUST NOT modify files under the working directory in a way that affects declared outputs.

In the **execute phase**, `cook.sh` runs on the worker VM that is evaluating an execute-phase Lua chunk — for example, the body unit of an imperative-region run (§{recipes.body-bundling}) or a `using >{ … }` payload (§{lua.using-block-globals}). It is the mechanism by which a `lua_line` or `lua_block` shells out, and it is the call into which a coalesced run of adjacent `shell_command` steps desugars (§{recipes.body-bundling}). Stdout is captured and returned to the calling Lua code; stderr is streamed to the worker's stderr. The `COOK_CMD_FAILED:` Lua error is identical in shape to the register-phase variant.

### 6.3.2. `cook.exec` and `cook.interactive` [#lua.cook-exec]
**Signatures.**
  - `cook.exec(cmd: string, line: integer) -> string`
  - `cook.interactive(cmd: string, line: integer) -> string`

**Phase.** Register-phase only.

These helpers are the Lua-side counterparts of the surface `shell_command` (§{recipes.shell-steps}) and `interactive_command` (§{recipes.shell-steps}) step forms. Each call records one unit (non-interactive or interactive, respectively) with caching disabled. The return value is the empty string; it is returned only so that callers composing results do not fail on `nil` concatenation.

A conforming implementation MUST raise a Lua runtime error when `cook.exec` or `cook.interactive` is called from execute-phase Lua (a `lua_line`, a `lua_block`, or a `using >{ … }` payload). The diagnostic MUST identify the function name and the calling step kind. The same rule applies to other register-only Cook Lua API helpers: `cook.add_unit`, `cook.step_group`, `cook.recipe`. See §{rationale.execute-phase-api}.

### 6.3.3. `cook.recipe` [#lua.cook-recipe]
**Signature.** `cook.recipe(name: string, metadata: table, body: function) -> nil`
**Phase.** Register-phase only.

Registers a recipe with the given name, metadata (`ingredients`, `excludes`, `requires`), and body function. Surface recipes (§{recipes.header-forms}) are compiled to exactly one `cook.recipe` call. A conforming implementation is not required to expose `cook.recipe` as a user-facing extension point; its contract here is what the code generator (§{grammar}, Appendix A) produces.

### 6.3.4. `cook.load_module` [#lua.cook-load-module]
**Signature.** `cook.load_module(name: string) -> table`
**Phase.** Both.

`cook.load_module` resolves `name` against the `cook_modules/` directory of the **calling work unit's source Cookfile**, loads the module file (`<name>.lua` or `<name>/init.lua`), evaluates it on the calling VM, runs the returned table's `init()` function if present (§{modules.overview}), and returns the module table. Repeated calls within one VM MUST be cached: a second `cook.load_module(name)` with the same `(working_dir, name)` MUST return the same table without re-reading or re-evaluating the file.

The lookup is **lexical per Cookfile** (§{modules.use-scope}): an imported Cookfile's body unit, whose `working_dir` is its own import directory, resolves `cook.load_module("foo")` against `<import_dir>/cook_modules/foo.lua`. The importing Cookfile cannot reach into the imported Cookfile's `cook_modules/` and vice versa.

`cook.load_module` is the resolver behind the `use` declaration (§{lua.use-env}). The codegen for `use foo` emits `local foo = cook.load_module("foo")` at the top of the Cookfile's register-phase Lua, and prepends the same binding to every execute-phase body unit's `lua_code` payload, so the alias `foo` is in scope for both phases of every recipe in the declaring Cookfile.

A conforming implementation MUST report a runtime error when `name` does not resolve to a file under the calling Cookfile's `cook_modules/`. The diagnostic MUST identify `name` and the absolute path that was searched.

The body of a module loaded into the execute-phase VM MUST NOT call register-only API (`cook.add_unit`, `cook.exec`, `cook.interactive`, `cook.step_group`, `cook.recipe`). A conforming implementation MUST surface such a call as the §{rationale.execute-phase-api} runtime error. Module functions whose bodies are restricted to both-phase API (`cook.sh`, `cook.env`, `cook.cache`, `cook.platform`, `fs.*`, `path.*`) work in either phase; module authors document per-function which phase a function targets.

### Note 6.3.1

Additional `cook.*` helpers — notably `cook.env`, `cook.platform`, `cook.cache`, `cook.export`, `cook.import`, `cook.dep_output`, and `cook.dep_output_list` — are part of the runtime but are specified in §{modules} (modules and configuration), not here. §{lua} defines only the work-unit surface and the two phase-split helper families.

## 6.4. Using-block globals [#lua.using-block-globals]
**Phase.** Execute-phase only.

When the execute phase runs a Lua-code unit recorded by `cook.add_unit` (§{lua.add-unit}) with a `lua_code` payload — for example, one recorded from `using >{ ... }` on a `cook` step (§{recipes.cook-single-output}, §{recipes.cook-multi-output}) — a conforming implementation MUST, immediately before evaluating `lua_code`, bind the following globals on the worker's Lua VM:

| Global    | Type   | Value |
|-----------|--------|-------|
| `inputs`  | table  | A 1-indexed sequence of the unit's input paths, in the order recorded. |
| `outputs` | table  | A 1-indexed sequence of the unit's output paths, in the order recorded. |
| `input`   | string | `inputs[1]`, or the empty string if `inputs` is empty. |
| `output`  | string | `outputs[1]`, or the empty string if `outputs` is empty. |
| `input_N` | table  | For each ingredient group `g` at 1-based index `N` in the unit's `ingredient_groups`, `input_N` is a 1-indexed sequence of the paths in `g`. |

A conforming implementation MUST bind `input_N` for every ingredient group recorded, in increasing `N`. Groups not recorded MUST NOT be implicitly bound — a reference to `input_3` when only two groups exist MUST read as `nil` rather than as an empty table.

### Example 6.4.1

For the multi-output Lua-block `cook` step

```cook
recipe build_lua
    ingredients "src/*.rs"
    cook "staging-lua/out.js" "staging-lua/out.wasm" using >{
        os.execute("mkdir -p staging-lua")
        local sources = table.concat(inputs, " ")
        local js = io.open(outputs[1], "w")
        js:write("// generated from " .. sources .. "\n")
        js:close()
        local wasm = io.open(outputs[2], "w")
        wasm:write("BINARY\n")
        wasm:close()
    }
```

the execute-phase worker binds `inputs` to the resolved ingredient paths, `outputs` to `{ "staging-lua/out.js", "staging-lua/out.wasm" }`, `input` to `inputs[1]`, `output` to `"staging-lua/out.js"`, and `input_1` to the first (and only) ingredient group.

### Note 6.4.1

These globals are bindings on the worker VM, not on the register-phase VM. A register-phase Lua step (`> expr` or `>{ ... }` outside a `using` clause) MUST NOT observe them.

### 6.4.1. Plate and test Lua-block bindings [#lua.using-block-globals-plate-test]

A Lua block in a `plate_step` or `test_step` body (§{recipes.plate-step}, §{recipes.test-step}) is bound according to the step's mode (§{recipes.iteration-mode-plate-test}):

| Binding | Mode | Type | Meaning |
|---|---|---|---|
| `input` | one-to-one | string | The current iteration item — one of the source items |
| `inputs` | many-to-one | table (1-indexed) | The full source list |

A conforming implementation MUST bind exactly one of `input` / `inputs` per unit, per the mode the body's content selected. The other is unbound (Lua `nil` on reference).

A conforming implementation MUST NOT bind `output` / `outputs` for plate or test units; these have no meaning in a step that declares no outputs. A conforming implementation MUST NOT bind `input_N` (indexed input access) for plate or test units; the source list is exposed wholesale via `inputs` in many-to-one mode.

Cross-recipe references inside a plate or test Lua body use `cook.dep_output()` / `cook.dep_output_list()`, identical to a `using >{ … }` Lua body (§{lua.using-block-globals}).

## 6.5. Filesystem helpers (`fs.*`) [#lua.fs-helpers]
**Phase.** Both. The same surface is installed in register-phase and execute-phase VMs; semantics are identical.

All `fs.*` functions interpret relative path arguments against the Cookfile's working directory. A conforming implementation MUST NOT silently succeed on I/O errors: failures other than those made explicit below are reported as Lua runtime errors.

### 6.5.1. `fs.exists` [#lua.fs-exists]
**Signature.** `fs.exists(path: string) -> boolean`

Returns `true` if a filesystem entry exists at `path`, `false` otherwise. Broken symlinks and permission errors are implementation-defined with respect to the boolean result, but this function MUST NOT raise for a missing path.

### 6.5.2. `fs.size` [#lua.fs-size]
**Signature.** `fs.size(path: string) -> integer`

Returns the size of the file at `path` in bytes. A missing or unreadable entry is a Lua runtime error.

### 6.5.3. `fs.read` [#lua.fs-read]
**Signature.** `fs.read(path: string) -> string`

Returns the contents of `path` as a Lua string. A missing or unreadable entry is a Lua runtime error. Encoding is not transformed: the string carries raw bytes.

### 6.5.4. `fs.write` [#lua.fs-write]
**Signature.** `fs.write(path: string, content: string) -> nil`

Writes `content` to `path`, creating parent directories as needed. Any existing file at `path` is overwritten. Write failure is a Lua runtime error.

### 6.5.5. `fs.mkdir_p` [#lua.fs-mkdir-p]
**Signature.** `fs.mkdir_p(path: string) -> nil`

Creates the directory at `path`, and any missing parent components, if they do not already exist. This function MUST succeed when `path` already exists and is a directory.

### 6.5.6. `fs.glob` [#lua.fs-glob]
**Signature.** `fs.glob(pattern: string) -> array of string`

Returns a 1-indexed sequence of absolute paths matching the glob `pattern`, interpreted relative to the working directory. The order of returned paths is implementation-defined. An invalid glob pattern is a Lua runtime error; a valid pattern with no matches MUST return an empty sequence.

### 6.5.7. `fs.mtime` [#lua.fs-mtime]
**Signature.** `fs.mtime(path: string) -> number`

Returns the modification time of `path` as seconds since the Unix epoch, as a `number`. A missing or unreadable entry is a Lua runtime error.

### Note 6.5.1

The `fs.*` surface is intentionally minimal. Authors needing operations not listed here (symlink manipulation, recursive deletion, permission changes) invoke those operations through the shell — `fs.*` is a host-portable subset for the common cases.

## 6.6. Path helpers (`path.*`) [#lua.path-helpers]
**Phase.** Both. The `path.*` surface is pure string manipulation on path components; it performs no I/O and does not depend on the working directory.

### 6.6.1. `path.stem` [#lua.path-stem]
**Signature.** `path.stem(p: string) -> string`

Returns the final path component of `p` with its extension removed. For `"src/foo.c"` the result is `"foo"`. For a path with no file name (e.g., `""` or `"/"`) the result is the empty string.

### 6.6.2. `path.name` [#lua.path-name]
**Signature.** `path.name(p: string) -> string`

Returns the final path component of `p` including its extension. For `"src/foo.c"` the result is `"foo.c"`.

### 6.6.3. `path.ext` [#lua.path-ext]
**Signature.** `path.ext(p: string) -> string`

Returns the extension of `p` as a string beginning with a literal `.`; for a file with no extension, returns the empty string. For `"src/foo.c"` the result is `".c"`.

### 6.6.4. `path.dir` [#lua.path-dir]
**Signature.** `path.dir(p: string) -> string`

Returns `p` with its final component removed. For `"src/foo.c"` the result is `"src"`.

### 6.6.5. `path.replace_ext` [#lua.path-replace-ext]
**Signature.** `path.replace_ext(p: string, new_ext: string) -> string`

Returns `p` with its extension replaced by `new_ext`. A leading `.` on `new_ext` is optional: `path.replace_ext("a.c", "o")` and `path.replace_ext("a.c", ".o")` both yield `"a.o"`.

### 6.6.6. `path.join` [#lua.path-join]
**Signature.** `path.join(a: string, b: string) -> string`

Returns the path formed by joining `b` onto `a` with the host separator. If `b` is absolute, the result is `b`.

## 6.7. Placeholder substitutions in shell text [#lua.shell-placeholders]
**Phase.** Substitution is performed by the code generator at register time; the resulting Lua expression is evaluated at register time when the unit is recorded.

A `cook` step's `using { ... }` shell-block body, every `plate` command, every `test` command, and every bare `shell_command` is a **shell text** for the purposes of this section. Before a conforming implementation records the resulting unit, it MUST substitute the placeholder set defined in this section. The set is closed: a `{TOKEN}` whose shape does not match any row of the placeholder table below resolves through §{xref.resolution}.

| Placeholder | Valid in iteration mode | Meaning |
|---|---|---|
| `{in}` | one-to-one (own or dep-driven) | The current iteration item (path) |
| `{in.ACCESSOR}` | one-to-one | `path.ACCESSOR(in)` per §{lua.path-helpers} |
| `{out}` | any mode, single declared output | The unit's single output path |
| `{out.ACCESSOR}` | any mode, single declared output | `path.ACCESSOR(out)` |
| `{out_N}` (`N` ∈ 1..) | any mode, multi-output | The unit's `N`-th declared output, in declaration order |
| `{out_N.ACCESSOR}` | any mode, multi-output | `path.ACCESSOR(out_N)` |
| `{all}` | many-to-one only | The unit's input list, joined by single space characters |

`ACCESSOR` is one of `stem`, `name`, `ext`, `dir`, with semantics defined by §{xref.path-accessors}.

A conforming implementation MUST reject the following shapes with a load-time diagnostic:

- `{in}` or `{in.ACCESSOR}` in a many-to-one step (no current iteration item exists).
- `{all}` in a one-to-one step.
- `{out}` in a multi-output step (ambiguous — use `{out_N}`).
- `{out_N}` in a single-output step (use plain `{out}`).
- `{out_N}` for `N` greater than the step's declared output count.
- `{stem}`, `{name}`, `{ext}`, or `{dir}` as a bare token (i.e. with no `NAME.` prefix). These were the pre-CS-0022 spelling for the current-iteration accessors and are no longer admitted at the placeholder layer; the current spelling is `{in.stem}` etc. The diagnostic MUST name the new form. (A bare `{stem}` would otherwise resolve through §{xref.resolution} step 4 to `cook.env["stem"]`, which is a confusing failure mode for code being migrated forward from a v0.4 surface.)

### Example 6.7.1

```cook
cook "build/{in.stem}.o" using { gcc -c {in} -o {out} }
```

with ingredient `main.c` records one shell unit per ingredient. For ingredient `main.c`, the unit's `command` field is `set -e\ngcc -c main.c -o build/main.o`. The output pattern's `{in.stem}` resolves to `main` before the body's `{out}` is substituted; the resulting `{out}` is `build/main.o`.

### Example 6.7.2

```cook
cook "out.js" "out.wasm" using {
    wasm-pack build
    cp pkg/main.js {out_1}
    cp pkg/main.wasm {out_2}
}
```

with any input list records one shell unit (many-to-one mode — outputs are literal). The body's `{out_1}` and `{out_2}` resolve to `out.js` and `out.wasm` respectively.

### Note 6.7.1

The block forms (`using { ... }` and `using >{ ... }`) and the placeholder layer compose differently. A `shell_block` is shell text and gets the substitution defined in this section. A `using_lua_block` is Lua text and does not get textual substitution; the same iteration data is exposed as Lua bindings per §{lua.using-block-globals}. The asymmetry is principled — see App. B.6.1 — and reflects that shell has no binding mechanism for "the current file" while Lua does.

### Note 6.7.2

Cross-recipe references inside a shell block follow §{xref.string-substitution}: a bare `{NAME}` substitutes to the named recipe's full output list, joined by space. The `{lib.ACCESSOR}` form is rejected inside any using-clause body, including a shell block, because it would smuggle a second iteration source past the output pattern's declaration. Authors who want a recipe's per-element accessors inside a step driven by something else use Lua (`path.stem(cook.dep_output_list("lib")[1])`).

### 6.7.1. Plate and test shell-block placeholders [#lua.shell-placeholders-plate-test]

A `plate_step` or `test_step` body that is a `shell_block` admits the following placeholders:

| Placeholder | Valid in mode | Meaning |
|---|---|---|
| `{in}` | one-to-one | The current iteration item (path) |
| `{in.ACCESSOR}` | one-to-one | `path.ACCESSOR(in)` per §{lua.path-helpers} |
| `{all}` | many-to-one | The source list, space-joined |
| `{NAME}` (NAME is a recipe in scope) | any mode | `cook.dep_output(NAME)` per §{xref.string-substitution} |
| `{TOKEN}` (none of the above) | any mode | `cook.env[TOKEN]` per §{xref.resolution} |

A conforming implementation MUST reject:
- `{in}` or `{in.ACCESSOR}` in a many-to-one or one-shot body (no current iteration);
- `{all}` in a one-to-one or one-shot body (no batched input list);
- Any `{out}` / `{out.ACCESSOR}` / `{out_N}` / `{out_N.ACCESSOR}` token in a plate or test body — the step declares no outputs. The diagnostic MUST point to `{in}` as the iteration item.
- `{NAME.ACCESSOR}` for any recipe `NAME` in scope (preserves the §{xref.dep-driven} firewall).

Bare path-accessors (`{stem}`, `{name}`, `{ext}`, `{dir}`) in a plate or test body fall through to `cook.env[TOKEN]` per §{xref.resolution} step 4. A conforming implementation MUST emit a specific diagnostic for these four names that points to `{in.ACCESSOR}` (same diagnostic CS-0022 introduced for cook bodies).

## 6.8. The `use` declaration [#lua.use-env]
> **Normative.** This section defines the `use` declaration, which brings a Lua module into the Cookfile's Lua environment. The grammar is given in App. A.2.

A `use` declaration has the form `use <name>`, where `<name>` is either a `BARE_IDENTIFIER` or a `STRING` (App. A.2). A conforming implementation MUST resolve a `use` declaration by applying the following steps in order, with the first matching step winning:

1. If `<name>` designates a built-in module (§{lua.builtin-modules}), the built-in module is selected.
2. Otherwise, the implementation MUST attempt to resolve `<name>` as a local module under `cook_modules/` by the lookup algorithm of §{lua.local-modules}.

A conforming implementation MUST reject a Cookfile for which no step succeeds, with a diagnostic that names `<name>` and indicates which locations were searched. The resolution step MUST occur before any recipe body executes; failure to resolve is a load-time error, not a run-time error.

The result of resolution is a Lua table bound in the Cookfile's Lua environment under the identifier derived from `<name>` by replacing each ASCII hyphen (`-`) with an underscore (`_`). All other characters admitted by `BARE_IDENTIFIER` (App. A.5) pass through unchanged. A conforming implementation MUST make this binding visible to every subsequent `lua_line`, `lua_block`, `module_call` (§{recipes.module-call-steps}), and `cook_step` `using` clause in the same Cookfile.

A module MAY expose a function named `init` on its returned table. If present, `init` MUST be called exactly once immediately after the module's Lua source has been executed, before any recipe body in the enclosing Cookfile is registered.

### 6.8.1. Built-in modules [#lua.builtin-modules]
A **built-in module** is supplied by the implementation and available without any project-level file. No built-in modules are defined by v0.1 of this Standard; the built-in kind is reserved to allow future extension (§{rationale.modules}).

### 6.8.2. Local modules [#lua.local-modules]
A **local module** is a file in the project's `cook_modules/` directory. The lookup algorithm resolves `<name>` against `cook_modules/<name>.lua`, then `cook_modules/<name>/init.lua`. The first existing file wins.

### Example 6.8.1

```cook
use cpp

recipe "colorize"
    cpp.bin("colorize", { sources = { "src/colorize.c" } })
```

The declaration `use cpp` binds the identifier `cpp` in the Cookfile's Lua environment to the table returned by `cook_modules/cpp.lua` (or `cook_modules/cpp/init.lua`). The `cpp.bin(...)` line in the recipe body is then dispatched as a `module_call` step (App. A.4, dispatch step 5) rather than a `shell_command`, because its first dot-separated segment `cpp` matches a bound identifier pattern (§{recipes.module-call-steps}).

### Note 6.8.1

The hyphen-to-underscore rewrite exists because `BARE_IDENTIFIER` permits internal hyphens (App. A.5) whereas Lua identifiers do not. Thus `use my-mod` binds the Lua global `my_mod`, but the on-disk lookup searches for `cook_modules/my-mod.lua` verbatim: the disk name is not rewritten.
