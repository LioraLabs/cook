# 28. cc — C-family build module [#cat.cc]
> **Normative.** This chapter specifies the public surface of the `cc` blessed module — the C-family native build module distributed through the official LuaRocks-backed registry (`rocks.usecook.com`). It is the first entry in the §{cat.index} catalogue and a conformance-checkable contract that any reimplementation MUST honour.

## 28.1. Synopsis and conformance scope [#cat.cc.synopsis]

The `cc` module specifies a portable surface for declarative C and C++ native builds. It dispatches on source-file extension (`.c`, `.C` → `cc`; `.cpp`, `.cc`, `.cxx`, `.C++` → `cxx`) so a single target can mix C and C++ sources without per-target language opt-in.

A conforming implementation MUST expose the fifteen public functions listed in §{cat.cc.surface} (the thirteen from v0.5 plus the `cc.checks` namespace and `cc.config_header`) and MUST honour their option tables, return values, and propagation semantics. A conforming implementation SHOULD provide informative error diagnostics for the negative cases described per-function.

The official `cook_cc` rock at `rocks.usecook.com` is the reference implementation. This chapter normatively specifies surface v0.8 (cook_cc 0.10.x): the v0.7 surface plus the additive `SharedOpts.output` field introduced by CS-0084 (§{cat.cc.shared}). The v0.7 surface (cook_cc 0.8.x–0.9.x) is the v0.6 surface plus the PRIVATE/PUBLIC propagation distinction introduced by CS-0080 (§{cat.cc.transitive.priv-pub}). Conformance is independent of the rock's internal file decomposition; only the surface returned from `require("cook_cc")` is normative.

## 28.2. Identity, resolution, init hook [#cat.cc.identity]

The blessed rock is published as `cook_cc`. A Cookfile imports it by:

```
use cook_cc
```

Per §{mods.use} (`use` declaration) and §{comp.overview} (cross-Cookfile composition), `use cook_cc` MUST bind the loaded module table to the local name `cook_cc` in the Cookfile's scope. The grammar of `§{mods.use}` does not currently provide an `as`-alias form; call sites refer to the module by its rock name (`cook_cc.bin(...)`, `cook_cc.find(...)`, etc.). A future Standard revision can introduce alias syntax; this clause will be updated in lockstep.

The implementation's module table MUST include an `init` function honoured by the loader contract at §{lua.cook-load-module}: the loader MUST call `init()` exactly once per Lua VM that loads the module. The `init` function MAY rehydrate state from `cook.probes` (e.g., a previously-detected compiler triple) and MUST NOT be exposed as part of the user-callable surface (callers reading §{cat.cc.surface} do not invoke `init()` directly).

## 28.3. Public surface [#cat.cc.surface]

### 28.3.1. cc.bin [#cat.cc.bin]

```
cc.bin(name: string, opts: BinOpts) -> string
```

Registers an executable target named `name`. A conforming implementation MUST compile every source listed in `opts.sources` (or globbed from `opts.dir`) and link the resulting objects to `build/bin/<name>`. The function MUST register the target in the implementation's known-targets state so `cc.compile_commands()` (§{cat.cc.compile-commands}) can recover its inputs.

`BinOpts` fields (all optional unless noted):

| Field | Type | Semantics |
|---|---|---|
| `sources` | list[string] | Source paths or globs. REQUIRED unless `dir` is provided. |
| `dir` | string | Root for recursive source globbing (`*.c`, `*.cpp`, `*.cc`, `*.cxx`). |
| `includes` | list[string] | Private include dirs. |
| `export_includes` | list[string] | Public include dirs propagated to consumers via `cook.export`. Absent → falls back to `includes` (the sole backcompat carve-out; see §{cat.cc.transitive.priv-pub}). |
| `export_defines` | list[string] | Public macros propagated to consumers via `cook.export`. Bare `defines` is PRIVATE-by-default; see §{cat.cc.transitive.priv-pub}. |
| `export_system_libs` | list[string] | Public `-l<name>` libraries propagated to consumers. Bare `system_libs` is PRIVATE-by-default. |
| `export_frameworks` | list[string] | Public macOS frameworks propagated to consumers. Bare `frameworks` is PRIVATE-by-default. No-op on non-macOS hosts. |
| `export_extra_ldflags` | string | Raw flags propagated to consumers' link step. Bare `extra_ldflags` is PRIVATE-by-default. |
| `defines` | list[string] | Macro names or `NAME=value`. PRIVATE-by-default (target-local). |
| `system_libs` | list[string] | Passed to the linker as `-l<name>`. PRIVATE-by-default. |
| `frameworks` | list[string] | macOS frameworks passed through to `cc.link` (§{cat.cc.link}). No-op on non-macOS hosts. PRIVATE-by-default. |
| `extra_cflags` | string | Raw flags appended at compile. PRIVATE (compile-only; no propagation surface). |
| `extra_ldflags` | string | Raw flags appended at link. PRIVATE-by-default. |
| `links` | list[string] | Other cc-registered targets to link against (transitive propagation per §{cat.cc.transitive}). |
| `needs` | list[string] | System-library names. Each name MUST be resolved via a `cc:find:<name>` probe registered by the implementation (§{cat.probes}). Compile units MUST carry `probes = {…, "cc:find:<name>", …}` and link units MUST weave the find probe's `libs` field via the `$<cc:find:<name>.libs>` sigil. See §{cat.cc.needs-semantics}. |
| `standard` | string | e.g., `"c++17"`, `"c11"`. |
| `warnings` | string | `"default"` (`-Wall`), `"strict"` (`-Wall -Wextra -Wpedantic`), `"none"`, or raw flags. |

Return: the target name (echo for chaining).

Error: if the source count is zero after globbing, the implementation MUST raise `[cc.bin] no sources found for target '<name>'` at the caller's line.

### 28.3.2. cc.lib [#cat.cc.lib]

```
cc.lib(name: string, opts: LibOpts) -> string
```

Registers a static-library target. A conforming implementation MUST compile sources as for `cc.bin`, then archive the objects to `build/lib/lib<name>.a`. `LibOpts` is `BinOpts` minus link-time fields (`extra_ldflags`, `system_libs`, and `frameworks` apply at consumer link time, not at archive time, but MAY be recorded for propagation).

`LibOpts.needs` behaves as on `BinOpts` (§{cat.cc.bin}): each name MUST be resolved via a `cc:find:<name>` probe and compile units MUST carry the corresponding `probes = {…, "cc:find:<name>", …}` entries. Because `cc.lib` produces an archive rather than a link unit, the find probe's `libs` field is not woven here; it is propagated to downstream consumers through `cook.export` (§{cat.cc.transitive}) so the eventual link site receives the `$<cc:find:<name>.libs>` sigil. See §{cat.cc.needs-semantics}.

The PRIVATE/PUBLIC distinction introduced by CS-0080 (§{cat.cc.transitive.priv-pub}) applies to `LibOpts` as it does to `BinOpts`. Because `cc.lib` produces an archive rather than a link unit, the bare fields `system_libs`, `frameworks`, `extra_ldflags` on `LibOpts` have no in-target effect: a conforming implementation MAY accept them silently or warn; their explicit-public counterparts (`export_system_libs`, etc.) are the load-bearing surface on `cc.lib`.

Error model identical to §{cat.cc.bin}.

### 28.3.3. cc.shared [#cat.cc.shared]

```
cc.shared(name: string, opts: SharedOpts) -> string
```

Registers a shared-library target. A conforming implementation MUST compile sources with `-fPIC` and link the objects to `build/lib/lib<name>.<ext>`, where `<ext>` is `so` on Linux and macOS and `dll` on Windows — unless `opts.output` is provided (see below). The linker MUST be invoked with `-shared` on Linux, `-dynamiclib` on macOS, and `/DLL` on Windows.

`SharedOpts` fields are identical to `BinOpts`, including `needs`: compile units MUST carry `probes = {…, "cc:find:<name>", …}` and the shared-library link unit MUST weave `$<cc:find:<name>.libs>` for each entry. See §{cat.cc.needs-semantics}.

`SharedOpts.output` (string, optional, CS-0084): explicit override of the link output path. When present, the implementation MUST use it verbatim as the `cc.link` output and MUST record it on the recipe's export so transitive linkers (§{cat.cc.transitive}) see the actual artifact location. The override is taken verbatim, including the filename portion — the implementation MUST NOT inject a `lib` prefix or replace the suffix. Authors targeting Windows MUST author `.dll`; authors targeting macOS MUST author `.dylib` or `.so` as their runtime loader expects. The field exists for plugin shapes where the host runtime loads the library by a fixed filename and directory (e.g., engine plugins that `dlopen("base.so")` from the binary's directory); when absent, the default `build/lib/lib<name>.<ext>` of §28.3.3 applies.

The PRIVATE/PUBLIC distinction introduced by CS-0080 (§{cat.cc.transitive.priv-pub}) applies to `SharedOpts` as it does to `BinOpts`.

### 28.3.4. cc.headers [#cat.cc.headers]

```
cc.headers(name: string, opts: HeadersOpts) -> string
```

Registers a header-only library: a logical target that records exports for transitive propagation but emits no compile or link units. `HeadersOpts` typically carries `export_includes` and `export_defines`. `cc.headers` MUST register the target in known-targets but MUST NOT call `cook.add_unit(...)`.

`HeadersOpts.needs` (list[string]): system-library names. The implementation MUST register each as a `cc:find:<name>` probe but MUST NOT add compile- or link-unit edges (headers targets export propagation only). The find probe's `libs` and `cflags` MUST flow through `cook.export` (§{cat.cc.transitive}) to downstream consumers, which assemble the appropriate sigils at their own compile and link sites. See §{cat.cc.needs-semantics}.

The PRIVATE/PUBLIC distinction introduced by CS-0080 (§{cat.cc.transitive.priv-pub}) applies to `HeadersOpts`. Because `cc.headers` emits neither compile nor link units, `HeadersOpts` is primarily a propagation surface and authors typically declare `export_includes`/`export_defines` directly. Bare `defines` on a headers target are PRIVATE-by-default and have no in-target effect (no compile unit consumes them).

### 28.3.5. cc.compile [#cat.cc.compile]

```
cc.compile(source: string, opts: CompileOpts) -> string
```

Compiles a single source file to an object. A conforming implementation MUST emit one work unit per call with `output = "build/obj/<target_name>/<stem>.o"` (where `<stem>` is the source's filename without extension and `<target_name>` is `opts.target_name` or `"default"`). The unit's `discovered_inputs` MUST point at a make-format depfile so header dependencies participate in incremental rebuilds.

Returns the output path.

### 28.3.6. cc.archive [#cat.cc.archive]

```
cc.archive(objects: list[string], output: string) -> string
```

Archives `objects` into a static library at `output` via `ar rcs`. The implementation MUST emit one work unit whose `inputs = objects`, `output = output`, `command = "ar rcs <output> <objects...>"`.

### 28.3.7. cc.link [#cat.cc.link]

```
cc.link(objects: list[string], output: string, opts: LinkOpts) -> string
```

Links objects to `output`. `LinkOpts.system_libs` is appended as `-l<name>`; `LinkOpts.extra_ldflags` is appended verbatim; `LinkOpts.shared = true` adds the platform's shared-library flag (§{cat.cc.shared}).

`LinkOpts.frameworks` (list[string]): each entry MUST emit `-framework <name>` at link time on macOS hosts. On non-macOS hosts the implementation MUST treat the field as a no-op, allowing cross-platform call sites where a curated finder returns `system_libs={"openal"}` on Linux and `frameworks={"OpenAL"}` on macOS to use the same `cc.link` invocation.

### 28.3.8. cc.find [#cat.cc.find]

```
cc.find(name: string, opts: FindOpts?) -> FindResult
```

Discovers a package by name. Returns a *sigil record* per §{cat.cc.needs-semantics}: a Lua table whose fields are `$<cc:find:<name>.<field>>` sigils for each documented `FindResult` field below. The implementation MUST register the underlying `cc:find:<name>` probe (idempotently) and MUST NOT execute discovery strategies at register phase; strategies run inside the probe's `produce` callback at execute phase per §{cat.probes.exec}.

`FindOpts` (all optional):

| Field | Type | Semantics |
|---|---|---|
| `version` | string | A semver-style constraint (e.g., `">=4.0"`, `"=4.0.1"`, `">=2.0,<3.0"`). Honoured by every strategy that can determine the discovered package's version. Pre-release tags are excluded (`"4.0.0-rc1"` does NOT satisfy `">=4.0.0"`). |
| `cmake` | boolean | When `true`, the cmake-compat strategy MUST be consulted immediately after curated finders, before pkg-config, and pkg-config MUST be skipped for this call. The default `false` places cmake-compat after pkg-config in the standard order. |

A conforming v0.3 implementation MUST consult strategies in this order, first-match-wins:

1. **Project-registered finders** (per §{cat.cc.register-finder}). MUST be tried first so projects can override curated behaviour.
2. **Curated finders** — implementation-provided finders for a documented short list. The reference implementation ships finders for `raylib`, `sdl2`, `openal`, `gl`/`opengl`, `threads`, `zlib`, `libcurl`. Implementations MAY ship a different set; each curated finder MUST conform to the FindResult contract.
3. **pkg-config** — as v0.1. MUST be skipped when `opts.cmake = true`.
4. **cmake-compat** — described in §{cat.cc.find-cmake-compat}. When `opts.cmake = true`, MUST be consulted in step 3's slot instead.
5. **Bare-lib probe** — file-existence check for `lib<name>.{so,dylib,a}` on the host's default linker search paths (`/usr/lib`, `/usr/local/lib`, and paths reported by `cc -print-search-dirs`). MUST be skipped when `opts.version` is set; bare probe cannot verify version.

A strategy MAY report `outcome = "skip"` (the strategy structurally could not be consulted) or `outcome = "miss"` (the strategy was consulted and rejected the package). Only `outcome = "hit"` populates `found = true` and the result fields. Implementations MUST NOT stop the chain on `skip` or `miss`; they MUST proceed to the next strategy. The `skip`/`miss` distinction is informational.

`FindResult` (returned by every `cc.find` call):

| Field | Type | Semantics |
|---|---|---|
| `found` | boolean | `true` iff some strategy returned `outcome = "hit"`. |
| `cflags` | string | Compile flags from the winning strategy (empty on miss). |
| `libs` | string | Link flags from the winning strategy (empty on miss). |
| `system_libs` | list[string] | Parsed `-l<name>` libraries. |
| `include_dirs` | list[string] | Parsed `-I<dir>` paths. |
| `lib_dirs` | list[string] | Parsed `-L<dir>` paths. |
| `frameworks` | list[string] | Parsed `-framework <name>` entries (macOS). |
| `version` | string \| nil | Version detected by the winning strategy (nil if the winning strategy reports none). |
| `tried` | list[Attempt] | Ordered list of every strategy consulted, including the winning entry on hits. |

`Attempt`:

| Field | Type | Semantics |
|---|---|---|
| `strategy` | string | `"project:<name>"`, `"curated:<name>"`, `"pkg-config"`, `"cmake-compat"`, `"bare-probe"`, or for third-party rocks `"rocks.<rockname>:<finder>"`. |
| `outcome` | string | `"hit"`, `"miss"`, or `"skip"`. |
| `reason` | string | Human-readable explanation. MAY be empty on `"hit"`. |
| `hint` | string \| nil | Install hint (e.g., `"apt: libraylib-dev / brew: raylib"`). Curated finders SHOULD emit on miss; other strategies SHOULD NOT. |

Caching: results MUST be cached in `cook.probes` keyed by `"cc.find:<name>:<canonical-opts>"`, where `canonical-opts` is the deterministic serialisation of `opts` (keys sorted, values stringified, list-valued opts joined with `,`). Repeat calls within one invocation MUST NOT re-consult the chain.

#### 28.3.8.1. cmake-compat strategy [#cat.cc.find-cmake-compat]

A conforming cmake-compat strategy MUST:

1. If `opts.version` is set, return `{ outcome = "skip", reason = "version detection unsupported by legacy cmake --find-package" }` without invoking any subprocess. cmake-compat does not surface version metadata in v0.3; the chain falls through to bare-probe (itself a no-op on `opts.version`).
2. Locate a `cmake` driver on PATH. If absent, return `{ outcome = "skip", reason = "cmake binary not on PATH", hint = "install cmake: apt: cmake / brew: cmake / dnf: cmake" }`.
3. Probe `EXIST` mode for the queried package using `cmake --find-package -DNAME=<name> -DCOMPILER_ID=GNU -DLANGUAGE=C -DMODE=EXIST -DQUIET=TRUE` (or implementation-equivalent invocation). If `EXIST` reports "not found" (cmake exit ≠ 0), return `{ outcome = "miss", reason = "cmake found no Config or Find module for '<name>'", hint = <package-specific or generic> }`.
4. Probe `COMPILE` mode for cflags; probe `LINK` mode for libs.
5. Parse `COMPILE` output per §{cat.cc.find-cmake-compile}.
6. Parse `LINK` output per §{cat.cc.find-cmake-link}.
7. If any token in the parsed LINK output is a filesystem path whose final component matches `*Config.cmake` or `*Targets.cmake`, return `{ outcome = "miss", reason = "imported-target chain too complex for legacy --find-package", hint = "register a project finder for '<name>' via cc.register_finder" }`.
8. On a clean hit, return `{ outcome = "hit", strategy = "cmake-compat", payload = { … } }` with `payload.version = nil`.

Names passed to cmake-compat are case-sensitive and forwarded to `cmake` unchanged. `cc.find("SDL3")` probes for `SDL3Config.cmake`; `cc.find("sdl3")` probes for `sdl3-config.cmake`. The case convention follows cmake's own resolver behaviour.

#### 28.3.8.2. cmake-compat COMPILE-mode grammar [#cat.cc.find-cmake-compile]

`cmake --find-package … -DMODE=COMPILE` emits a shell-token stream containing `-I<dir>`, `-D<macro>`, and (rarely) `-isystem <dir>` tokens, whitespace-separated, with a trailing newline. The implementation MUST reuse the v0.2 token grammar from the pkg-config strategy: `-I` ⇒ `include_dirs`; `-D` ⇒ collected in `cflags`; other tokens preserved verbatim in `cflags`.

#### 28.3.8.3. cmake-compat LINK-mode grammar [#cat.cc.find-cmake-link]

`cmake --find-package … -DMODE=LINK` emits a shell-token stream whose elements MUST be classified as:

| Token shape | Classification | Field |
|---|---|---|
| `-framework <name>` (two tokens) | macOS framework | `frameworks` |
| `-l<name>` | system library | `system_libs`; also appended to `libs` |
| `-L<dir>` | library search path | `lib_dirs`; also appended to `libs` |
| `*.so`, `*.so.<version>`, `*.dylib`, `*.a`, `*.lib` (any token containing `/` or `\` and ending in one of these suffixes) | absolute path to a library file | `libs` (verbatim) |
| `*Config.cmake`, `*Targets.cmake` | imported-target chain marker | triggers `outcome = "miss"` per §{cat.cc.find-cmake-compat} step 7 |
| any other non-empty token | recorded in `libs` verbatim; not propagated to typed fields | `libs` |

Absolute paths emitted to `libs` MUST be passable to the linker as-is; linkers accept absolute paths and resolve them without `-L` search.

### 28.3.9. cc.defaults [#cat.cc.defaults]

```
cc.defaults(opts: DefaultsOpts)
```

Records module-wide defaults that subsequent `cc.bin`, `cc.lib`, `cc.shared`, and `cc.headers` calls MUST honour by additive merge (lists prepend; scalars replace). `cc.defaults` is the canonical mechanism for setting build-wide overrides from `config` blocks; implementations MUST NOT consult `CPP_*` environment variables.

### 28.3.10. cc.toolchain [#cat.cc.toolchain]

```
cc.toolchain(opts: ToolchainOpts)
```

Overrides the active compiler, standard, and warnings. `ToolchainOpts`:

| Field | Type | Semantics |
|---|---|---|
| `compiler` | string | The `cxx` driver to use (e.g., `"clang++"`); the `cc` driver is derived. |
| `standard` | string | Default `-std=` value for subsequent C++ compiles. |
| `warnings` | string | Default warnings preset or raw flags. |

If never called, the implementation MUST auto-detect a compiler via PATH probing during `init()`.

The toolchain probe's value MUST record both the chosen driver names (`cxx`, `cc`) and their **canonical identities** (`cxx_id`, `cc_id`) — the lowercase-hex content hash of each resolved driver binary, obtainable via `cook.tools.id` (§{lua.cook-tools-id}) — so that units sealing the probe (§28.3.14's seal adoption) key on the toolchain's identity rather than its name: a compiler upgrade MUST re-key every sealing unit. The value MUST NOT contain a resolved path or any machine-specific location (§{mods.authoring.seal}, CS-0158).

### 28.3.11. cc.compile_commands [#cat.cc.compile-commands]

```
cc.compile_commands()
```

`cc.compile_commands()` is a **top-level register-phase call**, invoked at column 0 like `cc.uses`, `cc.toolchain`, and `cc.config_header`. It MUST queue — via `cook.on_register_complete` (§{lua.cook-on-register-complete}) — a run-once finalizer that renders `compile_commands.json` in the current working directory from the complete known-target set, after every recipe body of the registration pass has been evaluated. The call itself mints no recipe.

The rendered database MUST include one entry per source per registered target, for **every** registered target regardless of link closure or dispatch reachability, and MUST NOT require any ordering dependency of the author. The output MUST be valid JSON parseable by clangd and equivalent tools.

Repeat top-level `cc.compile_commands()` calls MUST be idempotent: at most one finalizer is queued per registration pass, producing one write of `compile_commands.json` per pass.

A `cc.compile_commands()` call made from inside a recipe body MUST be rejected with a register-phase error naming the top-level form. (This is a hard break from the pre-0.14 recipe-body invocation.)

The finalizer runs on the register-phase VM after the registration pass has closed the target set (§{lua.cook-on-register-complete}), so its inputs MUST be sourced from the implementation's module-held known-target state — live per §22.9 when the callback fires — and from `cook.import`'s per-target `compile_info`. The completeness guarantee is by construction: the finalizer snapshots the module-held target state only once every recipe body has registered its targets, so no target can be missing from the database.

### 28.3.12. cc.register_finder [#cat.cc.register-finder]

```
cc.register_finder(name: string, finder: function(opts) -> FindResult)
```

Registers a project-scoped finder for `name`. Subsequent `cc.find(name, ...)` calls MUST consult `finder` before the curated/pkg-config/bare-probe stages, recording its Attempt with `strategy = "project:<name>"`. The function MUST return a FindResult; the implementation MAY discard the function's own `tried` field (the resolver synthesises the chain-level `tried` list externally). Re-registration replaces the prior finder without warning; this enables `config` blocks to override defaults per build profile.

Raising in the finder is permitted; the error surfaces at the `cc.find` call site at level 2 (per §{cat.cc.errors}).

### 28.3.13. cc.find_or_error [#cat.cc.find-or-error]

```
cc.find_or_error(name: string, opts: FindOpts?) -> FindResult
```

Registers a `cc:find:<name>` probe whose `produce` callback consults the discovery chain and raises a Lua error when no strategy reports `outcome = "hit"`. Returns a *sigil record* per §{cat.cc.needs-semantics}, structurally identical to `cc.find`. The raised error MUST be prefixed `[cc.find_or_error]`, MUST list every Attempt in the result's `tried` field (strategy / outcome / reason), and MUST include any `hint` fields.

Failure is **demand-driven**: per §{cat.probes.exec}, the probe error fails the consuming unit at execute phase and propagates via DAG edge. Register-phase callers of `cc.find_or_error` MUST NOT observe a register-time error for missing libraries; a Cookfile that references `cc.find_or_error("missing")` solely from an unreached branch MUST register and analyse successfully. This is the only function in §{cat.cc} that raises on a missing package.

### 28.3.14. cc.checks namespace [#cat.cc.checks]

```
cc.checks.has_header(name: string, opts: CheckOpts?) -> sigil
cc.checks.has_function(name: string, opts: CheckOpts?) -> sigil
cc.checks.has_define(name: string, opts: CheckOpts?) -> sigil
cc.checks.sizeof(type: string, opts: CheckOpts?) -> sigil
cc.checks.endian() -> sigil
cc.checks.has_compile_flag(flag: string) -> sigil
cc.checks.has_link_flag(flag: string) -> sigil
```

Each function MUST register (or reuse) a `cc:check:<kind>:<name>:<short-fp>` probe and return the sigil string `"$<cc:check:<kind>:<name>:<short-fp>>"`. `<kind>` is one of `has-header`, `has-function`, `has-define`, `sizeof`, `endian`, `has-compile-flag`, `has-link-flag`. `<short-fp>` is an 8-hex-character hash of the canonicalised opts (standard, extra_cflags, extra_ldflags, defines, includes, system_libs). `<name>` is the probed identifier; for `endian` it MUST be the literal `_`; for `has-compile-flag` / `has-link-flag` it MUST be a sanitised form of the flag (characters outside `[A-Za-z0-9._+-]` replaced with `_`).

`CheckOpts` (all optional):

| Field | Type | Semantics |
|---|---|---|
| `standard` | string | `-std=` value for the probe compile (e.g., `"c11"`). |
| `extra_cflags` | string | Raw flags appended at compile. |
| `extra_ldflags` | string | Raw flags appended at link (link-stage kinds only). |
| `defines` | list[string] | Macro names or `NAME=value`. |
| `includes` | list[string] | Additional headers `#include`d before the probe's own header. |
| `system_libs` | list[string] | Libraries linked when the kind requires a link stage. |

Probe return types per kind:

| Kind | Returns | On compile failure |
|---|---|---|
| `has-header` | boolean | `false` |
| `has-function` | boolean | `false` |
| `has-define` | boolean | `false` |
| `sizeof` | integer | `nil` |
| `endian` | `"little"` \| `"big"` | `nil` |
| `has-compile-flag` | boolean | `false` |
| `has-link-flag` | boolean | `false` |

Each check function MUST be idempotent: a repeat call with the same name and equivalent opts MUST NOT re-register the probe and MUST return the same sigil. Different opts produce different fingerprints and therefore different probe keys; conflict-on-same-key is by construction impossible.

The implementation MUST register each probe with `requires = { "cc:compiler:<override-or-auto>" }` so the toolchain probe is a transitive dependency of every check.

### 28.3.15. cc.config_header [#cat.cc.config-header]

```
cc.config_header(template: string, output: string, vars: table) -> string
```

A conforming implementation MUST register a `cook.add_unit` whose `inputs = { template }`, `output = output`, and `command` invokes a CMake-compatible substitution renderer. For each `(k, v)` pair in `vars`:

- If `v` is a string matching `^%$<(cc:check:.+)>$`, the implementation MUST add `<key>` to the unit's `probes` list and MUST emit `k = v` in the command's vars-literal argument verbatim (so the sigil pipeline expands it at execute time).
- If `v` is a literal scalar (boolean, number, string not matching the sigil shape), the implementation MUST emit `k = <lua-syntax-literal>` (quoted for strings, `true`/`false` for booleans, decimal for numbers).

The renderer MUST process the template's content per CMake's `configure_file` rules:

- `@VAR@` → `tostring(vars[VAR])`; missing key → empty string.
- `${VAR}` → identical to `@VAR@`.
- `#cmakedefine SYMBOL [VALUE...]` → if `vars[SYMBOL]` is truthy: `#define SYMBOL <VALUE-after-@VAR@-substitution>`; if falsy: `/* #undef SYMBOL */`. When VALUE is absent or whitespace-only, emit `#define SYMBOL` with no value.
- `#cmakedefine01 SYMBOL` → `#define SYMBOL 1` if truthy, `#define SYMBOL 0` if falsy.

Truthiness MUST follow Lua semantics: only `false` and `nil` are falsy; `0`, `""`, and `{}` are truthy. This matches CMake's quirks for `#cmakedefine` (where `0` is truthy) and is documented explicitly.

Returns `output` (echo for chaining).

Error: a non-string-non-scalar value type in `vars` MUST raise `[cc.config_header] unsupported var type: <type>` at the caller's line.

### 28.3.16. Find semantics under probes [#cat.cc.needs-semantics]

A conforming implementation MUST treat each name passed via `opts.needs`
as a probe-key suffix: `"cc:find:" .. name`. The implementation MUST
register the probe idempotently: subsequent `cc.find` or `needs`
references to the same name MUST NOT re-register the probe. A probe
registered with conflicting `opts` (e.g., two calls for `"raylib"` with
different `version` constraints) MUST raise a register-phase diagnostic
naming both call sites.

The implementation's `cc.find(name, opts)` and `cc.find_or_error(name, opts)`
public functions MUST register the probe (if not already) and MUST
return a *sigil record*: a Lua table whose fields are strings of the
form `"$<cc:find:" .. name .. "." .. field .. ">"`, one per documented
field of the find result (§{cat.cc.find}). The sigil record's fields
MUST NOT support mutation; assigning to a sigil-record field MUST raise.

`cc.find_or_error` MUST register a probe whose `produce` raises a Lua
error when no strategy reports `outcome == "hit"`. Per §{cat.probes.exec}
the raised error fails the consuming unit and propagates via DAG edge.
Register-phase callers of `cc.find_or_error` MUST NOT observe a
register-time error for missing libraries; failure is demand-driven.

The probe-key namespace is normative for this chapter. Implementations
MUST use the keys:

- `cc:compiler:<override-or-auto>` — toolchain detection
- `cc:linker-search-dirs` — bare-probe linker search paths
- `cc:cmake-driver` — cmake-compat driver discovery
- `cc:find:<name>` — per-library find result
- `cc:check:<kind>:<name>:<short-fp>` — per-check feature-test result (per §{cat.cc.checks})

## 28.4. Transitive propagation [#cat.cc.transitive]

Every cc target maker (`cc.bin`/`cc.lib`/`cc.shared`/`cc.headers`) MUST call `cook.export(name, info)` recording the target's outward-facing fields:

- `info.includes` — directories the consumer adds to `-I<dir>` when compiling
- `info.defines` — macros the consumer adds to `-D<def>` when compiling
- `info.system_libs` — libraries the consumer links with `-l<name>` at link time
- `info.frameworks` — macOS framework entries the consumer adds at link time as `-framework <name>`
- `info.extra_ldflags` — raw flags to append at consumer link time
- `info.links` — further transitive targets (forms the propagation graph)
- `info.compile_info` — compile-time view of the target (post-merge: includes and defines as the compiler sees them, including transitively merged values), consumed by `cc.compile_commands` (§{cat.cc.compile-commands})

When `opts.links = { "a", "b" }` is provided to a target maker, the implementation MUST resolve the closure by walking `cook.import("a").links`, `cook.import("b").links`, recursively, deduplicating both nodes (cycle-safe) and field values. The closure's `includes` / `defines` / `system_libs` / `extra_ldflags` MUST be merged into the new target's compile and link commands. The closure's `frameworks` MUST be merged into the new target's link command analogously to `system_libs` (dedup-preserving first-seen order); a non-macOS host implementation MAY merge but MUST NOT emit (§{cat.cc.link}).

### 28.4.1. PRIVATE/PUBLIC distinction [#cat.cc.transitive.priv-pub]

Surface v0.7 (CS-0080) distinguishes PRIVATE from PUBLIC propagation. The bare fields `defines`, `system_libs`, `frameworks`, `extra_ldflags` on each target maker's options are PRIVATE: a conforming implementation MUST use them only for the target's own compile (defines) or link (system_libs, frameworks, extra_ldflags) and MUST NOT include them in the `cook.export(name, info)` record. The PUBLIC counterparts are explicit:

| Bare field (PRIVATE) | Public counterpart | `info` field carried by `cook.export` |
|---|---|---|
| `defines` | `export_defines` | `info.defines = opts.export_defines or {}` |
| `system_libs` | `export_system_libs` | `info.system_libs = opts.export_system_libs or {}` |
| `frameworks` | `export_frameworks` | `info.frameworks = opts.export_frameworks or {}` |
| `extra_ldflags` | `export_extra_ldflags` | `info.extra_ldflags = opts.export_extra_ldflags or ""` |

`export_includes` is the exception: it retains a backcompat fall-back. A conforming implementation MUST set `info.includes = opts.export_includes or opts.includes` — that is, when `export_includes` is absent, bare `includes` is exported. This carve-out exists because (a) bare `includes` had a coherent meaning at v0.1 (header layout is typically shared between target and consumers), (b) the v0.1 fixture corpus and the existing in-tree examples relied on it, and (c) the cost-benefit of forcing every consumer of v0.7 to add `export_includes` everywhere outweighs the clarity gained.

A conforming implementation of surface v0.7 MUST NOT export bare `defines`/`system_libs`/`frameworks`/`extra_ldflags` even if `opts.export_*` is absent (i.e., absence means "nothing exported" for these four). Implementations targeting surface v0.6 MUST continue to export bare values for backcompat with v0.6 fixtures; the surface version is declared by the conforming implementation per §{cat.cc.synopsis}.

`extra_cflags` has no propagation channel by design: compile flags are target-local.

### 28.4.2. Module-wide defaults stay target-local [#cat.cc.transitive.defaults]

Per §{cat.cc.defaults}, `cc.defaults({ ... })` records module-wide defaults that subsequent target makers MUST additively merge into each target's bare fields. Under PRIVATE-by-default (CS-0080), these merged defaults are target-local: they do NOT leak through `cook.export` to consumers. A project that wants a global default propagated to every target's consumers MUST declare it on each target's `export_*` field (or via a headers-only target that consumers `link` against).

`cc.toolchain({ ... })` (§{cat.cc.toolchain}) sets compiler/standard/warnings only and does not interact with the PRIVATE/PUBLIC matrix.

## 28.5. Error model [#cat.cc.errors]

Every error MUST be prefixed with `[cc.<fn>]` and named at level 2 (`error(msg, 2)`) so the blame surfaces at the Cookfile call site, not inside the module. Negative cases that MUST be diagnosed:

| Function | Condition | Message |
|---|---|---|
| `cc.bin`/`lib`/`shared` | `#sources == 0` after globbing | `no sources found for target '<name>'` |
| `cc.compile` | source file does not exist | `source file not found: <path>` |
| `cc.toolchain` (during `init()`) | no compiler on PATH | `no C/C++ compiler on PATH (tried g++, clang++)` |
| `cc.find_or_error` | `result.found == false` | `could not locate '<name>'<version-suffix>:\n  - <strategy>: <outcome> (<reason>)...\n<hints>` |
| `cc.register_finder` | `finder` is not a function | `register_finder for '<name>' requires a function, got <type>` |
| `cc.find` (cmake-compat) | cmake binary not on PATH | (returned as `Attempt`, not raised) `cmake binary not on PATH` |
| `cc.find` (cmake-compat) | EXIST mode reports not found | (returned as `Attempt`) `cmake found no Config or Find module for '<name>'` |
| `cc.find` (cmake-compat) | LINK output contains `*Config.cmake` / `*Targets.cmake` | (returned as `Attempt`) `imported-target chain too complex for legacy --find-package` |

`cc.find` does NOT raise on a miss; it returns `{ found = false }`. Callers that require a hit MUST test `result.found` and raise their own error.

## 28.6. Vendoring after install [#cat.cc.vendoring]

Per §{cat.bootstrap.vendor}, a project MAY hand-vendor a copy of `cook_cc` by placing `cook_modules/cook_cc.lua` (or `cook_modules/cook_cc/init.lua`) at the top of the project's `cook_modules/` directory. The hand-vendored file becomes the source of truth for that project; subsequent `cook modules install` invocations MUST leave it untouched (the install only writes under `cook_modules/share/lua/5.4/`).

A re-installed rock at `share/lua/5.4/cook_cc/init.lua` coexisting with a hand-vendored `cook_cc.lua` at the top level is unambiguously resolved: §{mods.use} specifies hand-vendored top-level wins.

## 28.7. Per-platform source selection [#cat.cc.platform-sources]

Cookfiles that target multiple host operating systems often need to compile a different set of source files per platform: a POSIX backend on Linux and macOS, a Win32 backend on Windows, an SDL or framework-specific shim split by platform, and so on. The `cc.bin`/`cc.lib`/`cc.shared` target makers take `sources` as a plain Lua `array of string`; Cook deliberately does NOT provide a structured per-platform option (e.g. `sources_per_platform = { linux = ..., darwin = ... }`), nor a `cc.platform_split` helper, nor any other declarative shape that branches on the host. Per-platform selection MUST be expressed in user Lua using `cook.platform` (§{lua.cook-platform}) and `fs.glob` (§{lua.fs-glob}).

### 28.7.1. Canonical pattern [#cat.cc.platform-sources.canonical]

The canonical idiom is a plain Lua block that builds the `sources` array by globbing the common patterns first, then conditionally extending the array with per-platform patterns gated on `cook.platform.os`:

```lua
local sys_sources = fs.glob({ "neo/sys/*.cpp", "neo/sys/sdl/*.cpp" })
if cook.platform.os == "linux" then
    for _, s in ipairs(fs.glob({ "neo/sys/posix/*.cpp",
                                 "neo/sys/linux/*.cpp" })) do
        table.insert(sys_sources, s)
    end
end

cook_cc.bin("dhewm3", {
    sources = sys_sources,
    links   = { "idLib", "framework", "renderer", "sound", "game" },
})
```

A conforming implementation MUST evaluate this pattern at register time on the register-phase Lua VM (§{lua.reg}); `cook.platform.os` is a both-phase API (§{lua.cook-platform}), so the read at register time observes the host that ran `cook`, and the resulting `cook.add_unit` calls record only the platform-appropriate sources. The execute phase sees no platform branching — the work-unit DAG is already specialised.

`fs.glob` (§{lua.fs-glob}) accepts an array of patterns (CS-0079), so the per-platform extension MAY be expressed in a single `fs.glob` call when ordering does not matter:

```lua
local sys_sources = fs.glob({
    "neo/sys/*.cpp",
    "neo/sys/sdl/*.cpp",
})
if cook.platform.os == "linux" then
    local posix = fs.glob({ "neo/sys/posix/*.cpp", "neo/sys/linux/*.cpp" })
    for _, s in ipairs(posix) do sys_sources[#sys_sources + 1] = s end
end
```

Both forms are conforming. The `table.insert` form and the `t[#t+1]` form are interchangeable plain-Lua idioms.

### 28.7.2. Non-pattern: structured per-platform tables [#cat.cc.platform-sources.non-pattern]

Cook does NOT define and a conforming implementation MUST NOT recognise a structured per-platform option on cc target makers. The following Cookfile is non-conforming and a conforming implementation MUST raise an error (the surface is "unknown option key" per §{cat.cc.errors}) if the key reaches the target maker:

```lua
-- NON-CONFORMING. Cook does not specify this surface.
cook_cc.bin("dhewm3", {
    sources_per_platform = {
        linux  = { "neo/sys/posix/*.cpp", "neo/sys/linux/*.cpp" },
        darwin = { "neo/sys/posix/*.cpp", "neo/sys/osx/*.cpp" },
    },
})
```

The reasons this surface is deliberately excluded:

1. **The Lua escape hatch already exists and reads cleanly.** Wrapping `cook.platform.os` checks in a declarative table buys no clarity over the four-line `if` block in §{cat.cc.platform-sources.canonical}; it adds a parallel surface that must be specified, conformance-tested, and version-evolved.
2. **The matrix is open-ended.** Real projects branch on compiler, bitness, libc, and feature probes (`cc.checks` — §{cat.cc.checks}), not just on `os`. Any structured surface would either be incomplete or balloon into a mini-DSL. Plain Lua composes with every existing both-phase API uniformly.
3. **Platform branching is not the only conditional source-selection use case.** Toolchain detection, feature probes, optional dependencies — all branch sources the same way. Privileging the OS axis with structured syntax would distort the design.

`cc.find` results (§{cat.cc.find}) and `cc.checks` results (§{cat.cc.checks}) MAY participate in the same source-selection Lua block when a feature probe gates whole files (e.g. a vendored fallback only compiled when a system library is absent). The same `cook.platform`/`fs.glob`/`table.insert` shape applies.

### 28.7.3. Interaction with PRIVATE/PUBLIC propagation [#cat.cc.platform-sources.propagation]

Per-platform source selection is orthogonal to the PRIVATE/PUBLIC distinction introduced by CS-0080 (§{cat.cc.transitive.priv-pub}). When per-platform sources rely on a per-platform define (e.g. `ID_PC_LINUX` only on Linux), the define goes on `defines` (or `export_defines`) per the propagation rules of §{cat.cc.transitive}; the per-platform set of defines is constructed in the same `if cook.platform.os == ...` block that builds the per-platform sources:

```lua
local defs = { "_GNU_SOURCE" }
if cook.platform.os == "linux"  then defs[#defs + 1] = "ID_PC_LINUX"  end
if cook.platform.os == "darwin" then defs[#defs + 1] = "ID_PC_OSX"    end

cook_cc.bin("dhewm3", {
    sources = sys_sources,
    defines = defs,                              -- PRIVATE: target-local
})
```

A conforming implementation MUST treat the resulting bare `defines` (or `export_defines`) array as defined by §{cat.cc.transitive.priv-pub} — the per-platform construction is invisible to the propagation rules, which see only the final array.
