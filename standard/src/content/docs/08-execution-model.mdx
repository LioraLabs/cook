# 8. Execution model [#exec]
> **Normative.** This chapter defines the observable execution semantics of a Cookfile. Implementation details (scheduler thread counts, hash functions, I/O strategies) are not normative; see `docs/architecture/` for the reference implementation's approach.

## 8.1. Two-phase execution [#exec.two-phase]
A conforming implementation MUST evaluate each selected recipe in two phases: a **register phase** and an **execute phase**. The register phase discovers the work a recipe wishes to perform; the execute phase performs it. The phases MUST NOT be interleaved within a single recipe: every unit of work produced by the register phase of a recipe MUST be known before the execute phase of that recipe begins.

During the register phase, the implementation evaluates the recipe body as Lua code (§{lua}) in **capture mode**. Cook Lua API calls that declare work — `cook.add_unit`, `cook.step_group`, and the step kinds in §{recipes} that desugar to these calls — record the work into a list of **work units** rather than performing it. The register phase MUST run on a single thread with a single Lua VM per recipe registration.

During the execute phase, the implementation arranges the recorded work units into a dependency graph (§{exec.step-groups}, §{exec.cross-recipe-ordering}) and executes the ready units. The execute phase MAY run work units in parallel where the dependency rules below permit it. The execute phase MUST respect every ordering relation established during the register phase.

A conforming implementation MAY register and execute different recipes concurrently only when doing so does not observably reorder work units relative to the rules in §{exec.step-groups}–§{exec.interactive-drain}; in particular, a recipe that depends on another recipe (§{exec.cross-recipe-ordering}) MUST NOT begin execution of any of its work units until every work unit of the prerequisite has completed successfully.

### Example 8.1.1

```cook
recipe build
    ingredients "src/*.c"
    cook "build/{in.stem}.o" using { gcc -c {in} -o {out} }
```

Register phase: the recipe body runs once. The `cook` step expands to a step group containing one work unit per matched source file. No `gcc` process is spawned.

Execute phase: the recorded units run, each in its own subprocess. Members of the step group MAY run in parallel (§{exec.step-groups}).

### Note 8.1.1

The two phases use different Cook Lua API bindings for the same call. In the register phase `cook.exec` records a shell work unit and returns the empty string without spawning a process; in the execute phase a shell work unit is dispatched to a subprocess. See §{lua} for the capture-mode contract of each API.

### 8.1.2. Phase classification of every Lua-bearing surface form [#exec.phase-classification]


Every Lua expression in a Cookfile is evaluated in exactly one of three timings. The following table is normative; rows that link to other sections defer the surface-level definition to that section, and merely state the phase here.

| Surface form | Source section | Phase | Produces a unit? |
|---|---|---|---|
| `config` block body                          | §{grammar.config-blocks}, §{grammar.config-composition} | **load** (before any register phase begins) | no — runs against Cook Lua API state directly |
| Cook module top level (`return mod`) and `init()` | §{modules.overview} | **load** (when the importing Cookfile's `use` resolves) | no |
| `inline_lua_line` (`>>` *body*)              | §{recipes.lua-steps} | **register** | no — inlined into the recipe-body Lua function |
| `inline_lua_block` (`>>{ … }`)               | §{recipes.lua-steps} | **register** | no — inlined |
| `module_call` (`alias.fn(…)`)                 | §{recipes.module-call-steps} | **register** | no — desugared to `inline_lua_line` / `inline_lua_block` |
| `lua_line` (`>` *body*)                      | §{recipes.lua-steps} | **execute** | yes — contributes to the body unit per §{recipes.body-bundling} |
| `lua_block` (`>{ … }`)                       | §{recipes.lua-steps} | **execute** | yes — contributes to the body unit |
| `cook STR using >{ … }` (using-clause Lua)   | §{recipes.cook-single-output}, §{recipes.cook-multi-output} | **execute** | yes — recorded with `lua_code` payload, with §{lua.using-block-globals} bindings |
| `cook.add_unit({lua_code = "…"})` payload    | §{lua.add-unit} | **execute** | yes — runs on the worker VM that picks up the unit |
| `plate { … }` (shell block)                  | §{recipes.plate-step} | **execute** | yes — recorded with shell payload, with §{lua.shell-placeholders-plate-test} substitutions |
| `plate >{ … }` (Lua block)                   | §{recipes.plate-step} | **execute** | yes — recorded with `lua_code` payload, with §{lua.using-block-globals-plate-test} bindings |
| `test { … }` (shell block)                   | §{recipes.test-step}  | **execute** | yes — recorded with shell payload via `cook.add_test`, with §{lua.shell-placeholders-plate-test} substitutions |
| `test >{ … }` (Lua block)                    | §{recipes.test-step}  | **execute** | yes — recorded with `lua_code` payload via `cook.add_test`, with §{lua.using-block-globals-plate-test} bindings |

Three corollaries:

1. **The single-`>` family is execute-phase, the doubled-`>` family is register-phase.** This is the surface-level rule a Cookfile author needs to remember; everything else in the table follows from it plus desugaring rules in §{recipes}.
2. **Module-call orchestration is register-phase.** It desugars to `inline_lua_line` / `inline_lua_block` (§{recipes.module-call-steps}), so the module's recorded `cook.add_unit` calls fire during registration. The work units those calls record run at execute, exactly like the cook step's own units.
3. **Inside an execute-phase Lua chunk, register-only Cook Lua API helpers are unavailable.** `cook.add_unit`, `cook.exec`, `cook.interactive`, `cook.step_group`, and `cook.recipe` MUST raise a Lua runtime error when invoked from execute-phase Lua. `cook.sh` is available in both phases (§{lua.cook-sh}). See §{rationale.execute-phase-api}.

## 8.2. Capture-mode semantics [#exec.capture-mode]
Capture-mode Cook Lua API calls MUST be pure with respect to the Cookfile's observable build state: they MUST NOT spawn subprocesses that affect declared outputs, modify files under the working directory, or block on external I/O. Their only effect on conforming evaluation MUST be to append entries to the register-phase record or to read metadata already established (for example, the terminal outputs of a prerequisite recipe that has already completed its register phase — see §{lua}).

Capture-mode calls MUST return synchronously. Any value a capture-mode call produces — for example, the output path string assigned by a `cook` step's expansion — MUST be a deterministic function of the Cookfile source and the register-phase record accumulated so far. It MUST NOT depend on wall-clock time, process identifiers, or the contents of files that the recipe's own work units produce.

An implementation MAY permit narrowly-scoped capture-mode side effects that participate in register-phase Lua control flow rather than in the build itself. Any such exception MUST be documented as part of the Cook Lua API contract in §{lua} and MUST preserve the determinism requirement above with respect to the register-phase record.

### Example 8.2.1

```cook
recipe build
    >> local flag = cook.env.OPTIMISE or "-O0"
    ingredients "src/*.c"
    cook "build/{in.stem}.o" using { gcc -c {in} -o {out} }
```

The `>>` line runs during the register phase (`>>` is the inline-Lua prefix; see §{recipes.lua-steps}). It reads `cook.env` — a value established before the register phase began — and assigns a local visible to subsequent declarative-region steps. The `cook` step then records its work units. No subprocess is spawned during registration.

Substituting `>` for `>>` would change the semantics: `>` is execute-phase, so the `flag` local would be set on a worker VM during execute, after the cook step's units have already been recorded — too late to influence registration.

### Note 8.2.1

The capture-mode contract makes work-unit discovery finite, deterministic, and independent of the order in which work units are later executed. A register phase that blocked on a running subprocess would serialise the register phase of other recipes against that subprocess, defeating the two-phase design.

## 8.3. Step groups and within-recipe parallelism [#exec.step-groups]
Work units registered inside a single invocation of `cook.step_group` form a **step group**. The units in one step group MAY execute in any order, and MAY execute in parallel, subject only to cross-recipe dependencies (§{exec.cross-recipe-ordering}) and the interactive rule (§{exec.interactive-drain}).

Work units registered outside any `cook.step_group` invocation are **sequential**. A sequential unit MUST execute strictly after every work unit that was registered earlier in the same recipe, and strictly before every work unit that is registered later in the same recipe.

When step groups and sequential units are interleaved, the implementation MUST enforce a **barrier** between them: every unit of one step group MUST complete before any unit of a later step group or later sequential unit in the same recipe begins. Equivalently, the set of units registered in a single step group acts as a single synchronisation point from the perspective of units registered after it.

The Cookfile surface syntax in §{recipes} produces step groups and sequential units as follows:

| Surface construct | Registration behaviour |
|---|---|
| `cook` step (§{recipes.cook-single-output}, §{recipes.cook-multi-output}) | One step group per `cook` step. Recorded by the declarative region. |
| `plate` step (§{recipes.plate-step}) | One step group per `plate` step. Recorded by the declarative region. |
| `test` step (§{recipes.test-step}) | One step group per `test` step. Recorded by the declarative region. |
| `inline_lua_line`, `inline_lua_block` (§{recipes.lua-steps}) | Inlined into the recipe-body Lua function and evaluated during the register phase. Produce units only via explicit Cook Lua API calls. |
| `module_call` (§{recipes.module-call-steps}) | Desugars to `inline_lua_line` / `inline_lua_block`; same registration behaviour. |
| `lua_line`, `lua_block` (§{recipes.lua-steps}) | Coalesced into a body unit per imperative-region run (§{recipes.body-bundling}); the unit is sequential and its `lua_code` runs on the worker VM at execute. |
| `shell_command` (§{recipes.shell-steps}) | Coalesced into the same body unit as the surrounding `lua_line` / `lua_block` steps; consecutive shell lines join into one `cook.sh` call within the unit (§{recipes.body-bundling}). |
| `interactive_command` (§{recipes.shell-steps}) | Sequential unit, additionally draining (§{exec.interactive-drain}). Breaks the body-bundling run; the next imperative-region step starts a fresh body unit. |

### Example 8.3.1

```cook
recipe build
    ingredients "src/*.c"
    cook "build/{in.stem}.o" using { gcc -c {in} -o {out} }
    cook "build/app" using { gcc -o {out} {all} }
```

The first `cook` step expands to one step group containing one compile unit per source file; the compiles MAY run in parallel. The second `cook` step is a distinct step group. The barrier rule forces every compile to complete before the link unit begins.

### Note 8.3.1

Parallelism is permissive, not prescriptive. A conforming implementation MAY run step-group members serially — for example, under a one-job flag. The Standard only requires that step-group members are not ordered with respect to one another; it does not require that they actually execute concurrently.

## 8.4. Cross-recipe ordering [#exec.cross-recipe-ordering]
Cross-recipe ordering is established only by name references (§{xref} of this Standard). For each name reference from recipe `A` to recipe `B` — in a dep-list, bare placeholder, or accessor placeholder — a conforming implementation MUST treat `B` as a prerequisite of `A`:

- An implementation MUST validate that `B` is defined in scope before the execute phase begins, and MUST reject the Cookfile with a diagnostic naming `A` and `B` otherwise.
- An implementation MUST reject the Cookfile with a cycle diagnostic if the transitive closure of all cross-recipe edges contains a cycle, before executing any work unit.
- A placeholder-sourced edge MAY be refined per §{xref.dep-implications}. A dep-list-sourced edge MUST NOT be refined.
- No other source of cross-recipe ordering is defined. Specifically, string equality between an ingredient path literal of `A` and a cook output path literal of `B` does NOT establish an edge. See rationale B.5.N.

### 8.4.1. Output-path uniqueness across recipes [#exec.output-uniqueness]

Two distinct recipes `A` and `B` MUST NOT both declare the same canonical output path on `cook` steps unless the cross-recipe ordering rules of §{exec.cross-recipe-ordering} establish a path between them in either direction. The **canonical output path** of a `cook` step is the path produced by joining the recipe's working directory with the step's declared output literal.

A conforming implementation MUST detect this condition at plan time, before any work unit of either recipe runs, and MUST reject the build with a diagnostic naming both colliding recipes and the shared canonical output path.

The rationale is execution-order safety: under parallel execution (`--jobs > 1`), two recipes with no enforced ordering that write to the same path race silently — the artifact's contents become non-deterministic, and downstream cache keys derived from the artifact become unstable. Requiring an explicit dependency edge collapses the race to a defined ordering. Where two recipes legitimately produce the same artifact, authors merge them into one recipe (single declaration site) or add an explicit `: <recipe>` dep edge to serialize them.

The rule applies to the declared output paths of `cook` steps; it does not apply to incidental writes performed by a step's command body to paths it has not declared, which the Standard does not track.

### Example 8.4.1

```cook
recipe writer_a
    cook "build/shared.bin" using { printf 'a' > $<out> }

recipe writer_b
    cook "build/shared.bin" using { printf 'b' > $<out> }

recipe build: writer_a writer_b
```

The Cookfile MUST be rejected at plan time. `writer_a` and `writer_b` both declare the canonical output `build/shared.bin` and there is no dep-list edge between them, so the implementation cannot serialize their writes. The diagnostic MUST name both `writer_a` and `writer_b` and the path `build/shared.bin`.

### Note 8.4.1

The check is plan-time, not execute-time: an implementation that defers detection until a write-write conflict occurs in the filesystem fails the requirement, because a successful first run can mask a race that fails on a subsequent run. The check looks at canonical paths derived from the source, not at filesystem state.

## 8.5. Interactive step draining [#exec.interactive-drain]
A step classified as `interactive_command` (§{recipes.shell-steps}) — that is, a step whose Cookfile source line begins with `@` — MUST, at execute time, **drain** the implementation's concurrent execute-phase work before its command begins. Draining means: at the instant the interactive unit is about to run, no other work unit of any recipe MUST be in progress. The implementation MUST wait until every then-running work unit has completed before beginning the interactive command.

An interactive unit MUST execute on a thread that inherits the controlling terminal of the Cook process, so that the spawned child inherits stdin, stdout, and stderr attached to a real TTY. The Standard does not prescribe which thread; it requires only the inheritance property and the drain property.

After an interactive unit completes, execution resumes under the normal rules: newly-ready units MAY be dispatched in parallel again. The drain applies only while the interactive unit is pending and running.

### Example 8.5.1

```cook
recipe dev
    cook "build/app" using { gcc main.c -o {out} }
    @./build/app
    echo "exited"
```

When the interactive step is reached at execute time, any other work in progress is drained. `./build/app` runs on a thread with the controlling TTY. Once it completes, `echo "exited"` runs. The `cook` step in the same recipe MUST have completed before `./build/app` begins, by the within-recipe rule of §{exec.step-groups}.

### Note 8.5.1

Draining is stronger than "runs on a single thread": it also excludes the interactive unit from running while any other work unit is still active anywhere in the build. This mirrors the common user expectation that an `@`-prefixed command is a standalone interactive session with no concurrent output from background jobs.

## 8.6. Cache semantics (abstract) [#exec.cache]
A conforming implementation MAY maintain a **build cache** that avoids re-executing a work unit whose observable inputs are unchanged since the last successful completion of that unit. The Standard specifies the cache abstractly; the hash function, on-disk layout, eviction policy, and storage medium are implementation-defined (§{intro.non-scope}).

The **observable inputs** of a work unit are those of its properties that are derived from the Cookfile source or from the register-phase record. They comprise, at a minimum:

1. The command text or Lua chunk that the unit will execute.
2. The list of input paths declared for the unit.
3. The list of output paths declared for the unit.
4. The recipe's resolved environment (§{lua}).

A conforming implementation MAY additionally consider the content of the declared input files, the modification times of the declared output files, the content of the recipe's ingredient set, the identity of the host machine on which the unit will execute (e.g., target triple, libc version, locale-affecting environment variables), and the resolved content of the binaries invoked by the unit's command text. It MUST NOT consider wall-clock time, process identifiers, or any state not derivable from the sources above. An implementation MAY exclude from the resolved-environment observable a set of environment variables whose values are known not to affect work-unit output (e.g., ambient session state, credentials unrelated to the build product); the specific set is implementation-defined.

### Note 8.6.2 — Tool-binary fingerprinting (informative)

When an implementation chooses to consider "the resolved content of the binaries invoked by the unit's command text", it MUST identify those binaries by inspecting the unit's literal command text — there is no separate manifest of invoked tools. Because shell quoting, command chaining (`;`, `&&`, `||`, pipes), command substitution (`$(...)`, backticks), subshells, and indirect invocation (`xargs`, `find -exec`, scripts) make exhaustive recovery undecidable in general, an implementation is permitted to recognise only a tractable subset of executables and may document its tractable subset.

The reference implementation recognises the **first executable token of each newline-separated statement**, after skipping leading `VAR=value` env-prefix tokens (POSIX command-prefix env assignments), and folds the content hash of each unique resolved binary into the unit's context fingerprint in deterministic order. Tools invoked indirectly — through `;`/`&&`/`||`/pipes, command substitution, subshells, or scripts that themselves invoke tools — are NOT fingerprinted by this pass; authors who require those tools to participate in cache invalidation should place each invocation on its own line, declare them via the project-level configuration mechanism described below, or add a regular file the recipe declares as an input. This recognition policy is implementation-defined; other implementations are free to recognise more or less.
An implementation MAY additionally accept a project-level **declared toolchain** — a list of executable names the build environment promises will be relevant to every unit's identity regardless of whether the auto-detection above sees them in the unit's command text. When supported, the declared toolchain MUST be resolved once per build; each resolved binary's content hash MUST contribute to the context fingerprint of every cacheable unit; an empty or absent declaration MUST contribute observationally inert bytes (so projects that do not use the mechanism produce a deterministic, distinguishable identity contribution); and a declared name that fails to resolve at build start MUST cause the build to fail before any unit executes. The reference implementation exposes this mechanism as `cache.tools` in `.cook/cloud.toml`. This mechanism is intended for environments where toolchain identity is governed at the environment layer (dev containers, `nix-shell`, hermetic Docker) and the implementation's auto-detection cannot see the relevant binary in the literal command text.

When the cache indicates that a unit is up-to-date, the implementation MAY skip executing the unit and MUST treat it as successfully completed for the purposes of the dependency graph. When the cache indicates that a unit is stale or absent, the implementation MUST execute the unit and, on success, MAY record the unit's completion in the cache for subsequent runs.

A unit for which the register-phase record disables caching — for example, a bare shell step or an `@` step, whose registration form sets the cache flag to false — MUST be executed on every run of the containing recipe.

### 8.6.1. Cache integrity [#exec.cache.integrity]
An implementation that records output content in a content-addressable store — whether local, shared between machines, or remote — MUST treat that store as untrusted at restore time. When the implementation reads bytes from the store with the intent to install them as the on-disk content of a declared output, it MUST recompute the same content fingerprint that it recorded for that output and MUST compare the recomputed fingerprint to the recorded one. If the two fingerprints differ, the implementation MUST NOT install the bytes; it MUST treat the read as a cache miss and MUST fall through to the rebuild path defined in §{exec.cache} above.

This requirement holds for every backend, including a backend that resides on the same machine and the same filesystem as the workspace. The Standard does not distinguish "local" from "remote" backends in the cache contract; the integrity check is the same in both cases. An implementation MAY use any deterministic fingerprint that meets the observability constraint of §{exec.cache} (the fingerprint MUST be a function of the bytes being installed, with no dependence on wall-clock time, process identifiers, or other extrinsic state); it MUST use the same function it recorded the original artifact under. An implementation that exposes the artifact store to writers other than the local build (e.g., a multi-tenant shared backend) SHOULD use a cryptographically secure fingerprint (collision-resistant under adversarial input). The fingerprint verification required by this section MAY be performed as bytes flow through a streaming reader, with the verification surfacing as an end-of-stream error; an implementation is not required to materialise the full artifact in memory before verifying.

The intent is that an implementation's cache restore path MUST NOT be a vector for installing attacker-controlled or otherwise unauthenticated bytes into the workspace. A backend whose contents have drifted, been corrupted, or been replaced — for any reason — MUST be observationally indistinguishable from a backend that simply does not have the artifact.

An implementation's artifact-store write path MUST treat a `put` of bytes that conflict with bytes already stored at the same key as an error. "Conflict" means the bytes differ; re-writing identical bytes is idempotent and MUST succeed. The error MUST surface to the caller; the implementation MUST NOT silently overwrite the prior bytes. This requirement is the write-side analogue of the read-side fingerprint check: it ensures that a key in the artifact store maps to one and only one byte sequence over its lifetime, which is the invariant the read-side verification relies upon.

### Example 8.6.1

```cook
recipe build
    ingredients "src/main.c"
    cook "build/main.o" using { gcc -c {all} -o {out} }
```

On the first run, `build/main.o`'s unit executes and its completion MAY be recorded. On a subsequent run with an unchanged `src/main.c` and unchanged command text, an implementation MAY skip the unit. If the command text changes or `src/main.c` changes, the unit MUST re-run.

### Note 8.6.1

The cache is optional. A conforming implementation that always re-executes every work unit — a clean-build implementation — is conforming so long as the dependency rules of §{exec.two-phase}–§{exec.interactive-drain} are respected.

### 8.6.3. Discovered inputs [#exec.cache.discovered-inputs]

A unit whose declaration carries `discovered_inputs` (§{lua.add-unit-discovered-inputs}) participates in post-execution input discovery. When such a unit is checked against an existing cache entry, a conforming implementation:

1. **MUST** attempt to read the file at `discovered_inputs.from` from the unit's working directory before composing the input-content set passed to the cache check. If the file is present and well-formed under `discovered_inputs.format`, the implementation MUST union the discovered paths with the unit's declared `inputs[]` for the purposes of the check.
2. **MUST**, when no restore context is available, treat a missing or malformed discovery file as no augmentation — the check proceeds with the thin (declared-only) input set, which causes an `InputSetChanged` rebuild that regenerates the discovery file. **MAY**, when a restore context is available, substitute the stored entry's recorded input set so the outputs walk and restore path can proceed; this avoids an avoidable rebuild when only the discovery file has been wiped from disk while the rest of the local entry is intact.
3. **MUST**, after a successful execution, parse the discovery file at `discovered_inputs.from` and amend the recorded `StepEntry`'s input set to include the discovered paths before persisting the entry to the local cache index.
4. **MUST** treat the discovery file as an implicit cache artifact: uploaded under its own artifact key during commit, restored from the backend on a hit-with-drifted-outputs check (§{exec.cache.restore-on-hit}).

The `"make"` discovery format is the Make rule format produced by GCC's `-MMD` flag and equivalent compiler options. The conforming parser MUST: strip the leading target text up to and including the first `:`; join continuation lines (`\\\n` and `\\\r\n`); ignore entries beginning with `/`; ignore the unit's first declared input; ignore entries whose paths do not exist on disk.

### 8.6.4. Test-unit caching [#exec.cache.test-unit]

A conforming implementation MAY cache the result of a `test_step` unit (§{recipes.test-step}). When caching is implemented, the following rules MUST hold.

1. **Cache key inputs.** A test unit's cache key MUST be a content-addressed digest derived from at least the following inputs:
   - The unit's substituted command text. For shell-block tests this is the command string after sigil-placeholder substitution (§{lua.shell-placeholders-plate-test}); for Lua-block tests this is the Lua source body string with bindings resolved per §{lua.using-block-globals-plate-test}.
   - The content fingerprints of every cook-step output the test consumes via §{recipes.iteration-mode-plate-test} source-list flattening.
   - The content fingerprints of every recipe output the test references via `cook.dep_output(NAME)` or `$<NAME>` cross-recipe substitution (§{xref.string-substitution}).
   - The deterministic env-key set as filtered through the implementation's `EnvDenylist` (the existing fingerprint mechanism for cook units per §{exec.cache}).
   - The tool-binary hash set per §{exec.cache} Note 8.6.2.
   - The `timeout` modifier value.
   - The `should_fail` modifier value.

   The cache key MUST NOT include the test's display name (§{recipes.test-step}), the `suite` field (§{lua.cook-add-test}), the recipe name, or the step's source position. Renaming a test, moving it within a recipe, or moving a recipe between Cookfiles MUST NOT bust its cache.

2. **Recordable outcomes.** A conforming implementation MAY record a test result for replay only when the unit's outcome is **passed**. A test is **passed** when:
   - The body's exit status is success and `should_fail` is false; or
   - The body's exit status is failure and `should_fail` is true.

   A test that is **failed** (the dual of passed), **timed out**, or **blocked** (its build dependency failed) MUST NOT be cached. The implementation MAY discard a previously-cached entry whose corresponding test is observed to fail in a subsequent run.

3. **Replayed payload.** A cache hit MUST produce, for reporting purposes, the originally-recorded `(stdout, stderr, duration_secs, success)` quadruple. The duration value of a cache replay is the recorded original duration, not the wall-clock cost of the cache lookup.

4. **Cross-machine portability.** A test result recorded on machine A MUST be a cache hit for the same fingerprint on machine B, given identical inputs.

## 8.7. Output path materialisation [#exec.output-materialisation]
Before invoking the shell text of a `cook` step, a conforming implementation MUST ensure that the parent directory of every output path declared by the step exists, creating it (and any missing ancestors) if necessary. The creation MUST be idempotent: it is not an error if the directory already exists. The creation MUST NOT proceed if the parent path resolves to a non-directory (regular file, symbolic link to a regular file, etc.); in that case the implementation MUST surface a clear diagnostic naming the output path and the offending parent, and MUST NOT execute the step's shell text.

The output paths covered by this rule are exactly those declared by the `cook` step in its surface form (§{recipes.cook-single-output}, §{recipes.cook-multi-output}) — that is, the paths recorded as the unit's output-path list during the register phase (§{exec.cache}). Each output path is resolved against the unit's working directory using the same resolution that the cache fingerprint applies. An output path with no parent component (a path that resolves to the working directory itself) is a no-op.

This rule applies to `cook` steps only. `plate` steps (§{recipes.plate-step}) and `test` steps (§{recipes.test-step}) declare no cacheable output paths and are therefore not covered; authors of `plate` and `test` step bodies that need to create directories MUST do so explicitly.

### Note 8.7.1

This rule turns the common `mkdir -p $<out.dir>` boilerplate into a no-op; authors MAY omit it from new recipes. Existing recipes that contain `mkdir -p` in their `cook`-step shell text continue to work — the explicit call becomes idempotently harmless after the engine's pre-step `mkdir -p`.

### Note 8.7.2

The pre-step directory creation is the engine acting on a path that the register phase has already declared as an output of the unit. It is not a Cook Lua API call and is therefore independent of the `fs.*` sandbox in §{lua.fs-helpers} (CS-0045). Out-of-project output paths are governed by the cache contract in §{exec.cache} and the path-resolution rules in §{modules.workspace-root}; this rule does not relax them.

## 8.8. Diagnostic ordering [#exec.diagnostic-ordering]
An implementation MUST report syntactic errors (those detectable by the rules of §{lexical} and §{grammar}) before it reports semantic errors (those detectable only in the register or execute phase). On encountering a syntactic error, the implementation MUST halt parsing at the first such error and MUST NOT proceed to register-phase evaluation. Consequently, a Cookfile with both a syntactic error and a semantic error MUST produce only the syntactic diagnostic on a given run.

Within the register and execute phases, the implementation MUST report dependency-graph errors — cycles and references to undefined recipes (§{exec.cross-recipe-ordering}) — before attempting to execute any work unit.

The exact wording and formatting of diagnostics are implementation-defined (§{intro.non-scope}). The diagnostic **class** and its **ordering relative to other diagnostics** are normative.

### Example 8.8.1

```cook
recipe build
    ingredients "src/main.c"
    cook "app using { gcc {all} -o {out} }
```

The unterminated output-path string in the `cook` step (`"app` has no closing `"`) is a syntactic error (§{lexical.strings}). The implementation MUST report this diagnostic and MUST NOT proceed to register-phase evaluation; any semantic issue that would have been detected later — for example, a missing dependency — MUST NOT be reported for this run.

### Note 8.8.1

The first-error halt in the syntactic layer means that a single run will not enumerate every syntactic problem in the file. A Cookfile author that wants batch syntax checking can re-run after each fix. Implementations MAY elect to attempt error recovery for better editor integration, but the Standard does not require it and does not relax the ordering rule: recovered-from errors MUST still be reported before any semantic diagnostic.
