# 7. Cross-Cookfile composition [#modules]
> **Normative.** This chapter defines cross-Cookfile composition: how a Cookfile loads a sibling Cookfile via `import`, how qualified name references (`alias.recipe`) resolve across the boundary, and how a workspace of Cookfiles is composed. The `use` declaration, which brings a Lua module into scope, is defined in §{lua.use-env} — not in this chapter. The grammar of the `import` declaration is given in App. A.2.

## 7.1. Overview [#modules.overview]
This chapter specifies cross-Cookfile composition. The `import` declaration (§{modules.import-declaration}) binds a sibling Cookfile to a local alias. A qualified name reference (`alias.recipe`) resolves under the rules of §{xref.resolution}, with the lookup set replaced by the recipe names of the imported Cookfile instead of the current one. All other name-reference rules of §{xref} — accessors (§{xref.path-accessors}), iteration (§{xref.dep-driven}), string-substitution (§{xref.string-substitution}), dependency implications (§{xref.dep-implications}) — apply unchanged.

## 7.2. `import <name> <path>` declarations [#modules.import-declaration]
An `import` declaration has the form `import <name> <path>`, where `<name>` is a `BARE_IDENTIFIER` and `<path>` is a non-whitespace path token (App. A.5).

A conforming implementation MUST validate the `<path>` token at parse time according to one of two shapes:

- **Tree-relative.** `<path>` begins with `./` or with a bare segment, contains no `..` segments, and is not absolute. The path resolves relative to the directory of the enclosing Cookfile.
- **Sigil-anchored.** `<path>` begins with `//` (the **workspace-root sigil**). The remainder of `<path>` after the sigil is forward-only (no `..`, no leading `/`). The path resolves relative to the workspace root (§{modules.workspace-root}).

A `<path>` matching neither shape MUST be rejected at parse time. Specifically:

- `<path>` containing a `..` segment (in the tree-relative form) MUST be rejected. Diagnostic: `import path '<path>': '..' segments are not permitted; use the workspace-root sigil '//path' for cross-cutting imports`.
- `<path>` that is absolute and does not begin with `//` MUST be rejected. Diagnostic: `import path '<path>': absolute paths are not permitted; tree-relative or '//' sigil`.
- `<path>` beginning with `//` and containing a `..` segment after the sigil MUST be rejected. Diagnostic: `import path '<path>': '..' segments are not permitted after '//'`.
- A sigil-anchored path that resolves to a directory outside the workspace root after symlink canonicalisation MUST be rejected at workspace load (§{modules.workspace-root}). Diagnostic: `import '<name>': sigil path resolves outside workspace root '<root>'`.

A conforming implementation MUST resolve `<path>` per the shape rules above (tree-relative against the enclosing Cookfile's directory; sigil-anchored against the workspace root, §{modules.workspace-root}) and MUST load the Cookfile at the resolved-path's `Cookfile` as a sibling Cookfile bound to the local alias `<name>`.

A conforming implementation MUST reject an `import` declaration for which:

- the resolved directory does not exist;
- the resolved directory's `Cookfile` does not exist;
- the resolved directory's `Cookfile` fails to parse.

A conforming implementation MUST treat tree-relative and sigil-anchored imports uniformly for the purposes of qualified name references (§{modules.qualified-refs}), dependency implications (§{xref.dep-implications}), and duplicate-alias detection (§{modules.duplicates-and-cycles}). The two shapes differ only in their resolution rule.

An `import` declaration does NOT bring a Lua value into scope under `<name>`. The alias `<name>` is reserved for cross-Cookfile recipe references of the form `<name>.<recipe>` that appear in dependency lists (§{recipes.ingredients}), not for Lua expressions. A Cookfile loaded by `import` is parsed independently; its own `use` and `import` declarations are resolved relative to its own directory, not the importer's directory.

A conforming implementation MUST reject a Cookfile in which two `import` declarations share the same `<name>` (App. A.2, duplicate-import clause; see also §{modules.duplicates-and-cycles}).

### Example 7.2.1

```cook
import backend ./services/backend       # tree-relative
import frontend ./apps/frontend         # tree-relative
import core //core/lib                  # sigil-anchored to workspace root

recipe "bundle": "backend.build" "frontend.build" "core.core_lib"
```

Here `backend` and `frontend` are tree-relative aliases for sibling Cookfiles below the importer's directory; `core` is anchored at the workspace root via the `//` sigil and resolves to a directory at or below the workspace root, regardless of where the importing Cookfile sits in the tree. The dependencies `"backend.build"`, `"frontend.build"`, and `"core.core_lib"` refer to recipes in those Cookfiles uniformly; the first segment before the dot selects the import alias, and the remainder is the recipe name in the imported Cookfile.

### Note 7.2.1

Because an `import` alias does not bind a Lua value, there is no interaction between `import backend …` and a `use backend` for a module of the same name: they occupy different namespaces. The duplicate rule of §{modules.duplicates-and-cycles} only prohibits collisions within the `import` alias namespace.

## 7.3. Qualified name references across Cookfiles [#modules.qualified-refs]
A **qualified name reference** has the form `alias.recipe`, where `alias` is a local alias introduced by an `import` declaration (§{modules.import-declaration}) in the referring Cookfile and `recipe` is a recipe name defined in the imported Cookfile. Qualified name references are governed by §{xref} (Cross-recipe references) with the following refinement of resolution:

- In §{xref.resolution} step (2), the lookup set is the union of the recipe names in the current Cookfile and the set of `alias.recipe` names, where `alias` ranges over the importing Cookfile's `import` aliases and `recipe` ranges over the recipe names in the Cookfile bound to `alias`. A whole-token match against `alias.recipe` wins over an accessor split on the same token (§{xref.resolution} step (3)).
- In §{xref.resolution} step (3), the accessor-split rule applies as written: when the `PREFIX` is itself a qualified name `alias.recipe`, the reference is `{alias.recipe.ACCESSOR}` and PREFIX is matched whole against the qualified lookup set before the SUFFIX is checked against the path-accessor set (§{xref.path-accessors}).

All other rules of §{xref} apply unchanged: dependency implications (§{xref.dep-implications}), dep-driven iteration (§{xref.dep-driven}), string-substitution (§{xref.string-substitution}), reserved final segment (§{xref.reserved-segment}). In particular, a qualified name reference establishes a cross-recipe edge with the same granularity rules as an unqualified one.

## 7.4. `use` scope is lexical per Cookfile [#modules.use-scope]
A `use` declaration's alias is scoped to its **declaring Cookfile** and is not visible across `import` boundaries. Each Cookfile in a workspace defines its own `use`-resolution scope. The aliases established by `use foo` in Cookfile A are not in scope for any Lua evaluated from Cookfile B's recipes, and vice versa, regardless of import direction.

For an execute-phase work unit, "the calling work unit's source Cookfile" (§{lua.cook-load-module}) is the Cookfile that contained the recipe whose body unit is being evaluated. A conforming implementation MUST resolve `cook.load_module(name)` and `require(name)` against that Cookfile's `cook_modules/` directory and MUST NOT fall back to a sibling Cookfile's `cook_modules/`.

A conforming implementation MUST resolve `cook.load_module(name)` at register phase, and MUST configure `package.path` and `package.cpath` for each execute-phase work unit, such that the calling Cookfile's `cook_modules/` is searched in the following order: (1) `cook_modules/?.lua` and `cook_modules/?/init.lua` (hand-vendored, top level); (2) `cook_modules/share/lua/<lua-version>/?.lua` and `cook_modules/share/lua/<lua-version>/?/init.lua` (LuaRocks-installed pure Lua); for native modules: (a) `cook_modules/?.<so-ext>` (hand-vendored, top level); (b) `cook_modules/lib/lua/<lua-version>/?.<so-ext>` (LuaRocks-installed). `<lua-version>` is the embedded Lua's MAJOR.MINOR (currently `5.4`). `<so-ext>` is the platform's loadable-module extension (`.so` on Linux/macOS, `.dll` on Windows).

A conforming implementation MUST also configure `package.path` and `package.cpath` (following the same search order) on the register-phase Lua VM before executing the body of any module loaded via `cook.load_module`. This ensures that sub-requires within a multi-file module (e.g., `require("cook_cc.toolchain")` inside `cook_cc/init.lua`) resolve against the project's `cook_modules/` rather than the system Lua paths. A conforming implementation MUST allow C extension modules (`.so`/`.dll`) to be loaded during register-phase `cook.load_module` evaluation; the register-phase VM MUST NOT restrict native module loading. [amended CS-0066]

A conforming implementation MAY expose `cook.load_module(name)` on the execute-phase Lua VM as an extension over the register-phase requirement above; when exposed, the execute-phase `cook.load_module` MUST honour the same four-path resolution order. Recipe-body Lua scripts that call `cook.load_module(name)` directly therefore see the same set of candidate paths as register-phase callers, eliminating "works at register but not at execute" surprises. [amended CS-0069]

Cross-Cookfile composition is at the **recipe** level via qualified name references (§{modules.qualified-refs}), not at the **module** level. Sharing module code between Cookfiles happens via the filesystem (a vendored copy, a symlink, or a shared parent) — the cook runtime treats each Cookfile's `cook_modules/` as an independent search root.

### Example 7.4.1

```
ws/Cookfile:        import b ./b
                    use cpp           # resolves to ws/cook_modules/cpp.lua
                    recipe top
                        cpp.plan("a")               # register-phase
                        > cpp.assemble("a")         # execute-phase, same alias
                        b.build                     # qualified dep on B's recipe
                    end

ws/b/Cookfile:      use proto         # resolves to ws/b/cook_modules/proto.lua
                    recipe build
                        proto.regen()
                        > proto.verify()
                    end
```

`top`'s body unit sees `cpp` (only). `b.build`'s body unit sees `proto` (only). Neither sees the other's modules. Filesystem-level sharing — for example, two Cookfiles vendoring the same `cpp.lua` — is independent: each Cookfile's worker VM caches its load by `(working_dir, name)`, so the same source file evaluated from two Cookfiles is two distinct module instances with independent state.

## 7.5. Duplicate and cycle detection [#modules.duplicates-and-cycles]
A conforming implementation MUST reject a Cookfile in which two `import` declarations share the same local alias. This rule is restated normatively in App. A.2 (duplicate-import clause); this section specifies the timing and the diagnostic surface.

**Duplicate imports.** The check is a **parse-time** check performed on the Cookfile in which the duplicates appear. A conforming implementation MUST detect the duplicate before loading the imported Cookfiles (§{modules.import-declaration}) and MUST report both offending lines in the diagnostic. The rule is per-Cookfile: two different Cookfiles in the same workspace MAY each define an `import foo ...` of their own, because each alias namespace is scoped to its declaring Cookfile.

**Import cycles.** A conforming implementation MUST detect cycles in the `import` graph when loading the workspace and MUST reject a workspace whose `import` edges form a cycle. A cycle is defined as a sequence of canonicalised directory paths `D_1, D_2, ..., D_k, D_1` such that each `D_i/Cookfile` contains an `import` declaration whose path resolves to `D_{i+1}` (and the last edge resolves back to `D_1`). Detection MUST use canonicalised paths so that two syntactically different `<path>` strings that refer to the same directory are recognised as the same node.

A conforming implementation MUST NOT reject the diamond case for **sigil-anchored imports**: if two distinct sigil-anchored imports both resolve to the same canonicalised directory, the implementation MUST load that directory once and reuse it. Tree-relative imports cannot form a diamond under §7.2 — two distinct tree-relative paths from sibling subtrees cannot canonicalise to the same directory without `..` traversal, which the §7.2 grammar forbids. Diamond deduplication for sigil imports is not a cycle.

The duplicate check and the cycle check operate on different graphs and are independent. A Cookfile MAY violate either without violating the other.

### Example 7.4.1

```cook
import backend ./services/backend
import backend ./services/backend_v2
```

Rejected at parse time: both lines declare the alias `backend` (App. A.2). The first line's target is irrelevant to the diagnostic; the collision is on the alias, not the path.

### Example 7.4.2

Given three Cookfiles:

```
/ws/a/Cookfile:       import b ../b
/ws/b/Cookfile:       import a ../a
/ws/Cookfile:         import a ./a
```

The workspace load that begins at `/ws/Cookfile` MUST be rejected: canonicalisation yields the cycle `/ws/a -> /ws/b -> /ws/a`.

### Note 7.4.1

Duplicate-import detection fires at parse time because the governing grammar production (App. A.2) constrains a single Cookfile's token stream, which is the unit the parser consumes. Cycle detection, by contrast, is a property of the `import` graph and necessarily crosses Cookfile boundaries; it fires at workspace-load time as the graph is walked (§{rationale.modules}).

## 7.6. Workspace root determination [#modules.workspace-root]

A **workspace root** is the directory against which `//`-anchored sigil imports (§7.2) resolve. A conforming implementation MUST establish the workspace root at invocation time using the following ordered procedure; the first rule that succeeds is authoritative.

1. **Explicit override.** If the implementation provides an explicit-root invocation flag (e.g., `cook --root <path>`) and one was supplied, the workspace root is `<path>` (canonicalised). The invoked Cookfile MUST be at or below `<path>`.

2. **Marker file.** Walk upward from the invoked Cookfile's directory. If any ancestor directory contains a file named `.cookroot` (zero-content sentinel), the **first** such ancestor is the workspace root.

3. **Tree-import inference.** Walk upward from the invoked Cookfile's directory. At each ancestor that contains a `Cookfile`, parse it. The ancestor is a **candidate** when both:
   1. It transitively imports the invoked Cookfile's directory through tree-relative imports only (sigil-anchored imports are ineligible at this stage because their resolution presupposes a workspace root), AND
   2. With the candidate treated as the workspace root, every sigil-anchored import in any Cookfile reachable from the candidate (across both tree-relative and sigil-anchored edges) resolves to a directory at or below the candidate after canonicalisation.

   Continue walking; the **highest** candidate is the workspace root.

4. **Self-root.** If no ancestor satisfies (1)–(3) AND neither the invoked Cookfile nor any Cookfile transitively reachable from it (through tree-relative imports) declares any sigil-anchored import, the invoked Cookfile's own directory is the workspace root.

5. **Reject.** If no ancestor satisfies (1)–(3) AND any reachable Cookfile declares a sigil-anchored import, workspace load MUST fail with a diagnostic naming the invoked Cookfile, the offending sigil import, and the inability to identify a workspace root that anchors it.

A workspace root's defining property is that it anchors sigil paths. A Cookfile that transitively reaches a sigil import therefore cannot itself be a workspace root in the absence of an enclosing anchor; the rejection in (5) makes this contract explicit.

The procedure terminates because each step walks to a strict parent directory; the filesystem root is the upper bound. Once the workspace root is determined, it is fixed for the remainder of the invocation.

### Definition: transitively imports

Cookfile *X* **directly imports** directory *D* when *X* has an `import <name> <path>` declaration whose canonicalised resolved directory equals *D*. *X* **transitively imports** *D* when there is a chain *X = X₀ → X₁ → … → Xₙ = D* where each *Xᵢ → Xᵢ₊₁* is a direct import. For workspace-root inference (rule 3.1 above), each link in the chain MUST be tree-relative; sigil-anchored imports are ineligible at that stage because their resolution presupposes a workspace root.

## 7.7. Cache portability invariants [#modules.cache-invariants]

A conforming implementation that maintains a cache of recipe execution results MUST hold the following invariants for the cache to be portable across teammates and across workspace-relocation moves on disk.

1. **`cookfile_path` is workspace-root-relative.** For every cached recipe, the `cookfile_path` recorded in the cache metadata MUST be the importee's Cookfile path expressed as a forward-slash relative path from the workspace root, and MUST contain no absolute-path components. This invariant follows from the workspace shape established by §7.2 and §7.6.

2. **`cache_key` uniqueness is per-Cookfile-cache-file scope.** A cache entry's identity within its index file is determined by its `cache_key` (typically derived from the unit's primary output path with optional variant suffix). The `cache_key` does not encode the recipe namespace; cross-Cookfile uniqueness is provided by **per-Cookfile cache locality** (each Cookfile maintains its own cache index file under its own directory).

3. **`cache_meta.input_paths` is importer-relative.** For a unit registered by recipe R in Cookfile C with working directory W, every entry in `cache_meta.input_paths` MUST be a path relative to W. This holds for both same-Cookfile inputs and cross-Cookfile dep inputs; substitution-time path rewriting (§7.3) ensures cross-Cookfile dep paths arrive in importer-relative form. For sigil-anchored imports the path can contain `..` segments; the cache check uses `W.join(input_path)` to locate the file on disk, and `..` segments resolve correctly against W.

A conforming implementation MAY layer additional content-addressed artifact storage atop the index files; that layer is outside the scope of this section.
