# 22.5. Probe units [#cat.probes]

> **Normative.** This chapter specifies the `cook.probe` Lua API and the probe-unit execution model. A conforming implementation MUST honour every clause in this chapter.

## 22.5.1. Synopsis [#cat.probes.synopsis]

A *probe unit* is a DAG unit whose output is a Lua value rather than a file. A conforming implementation MUST expose the register-phase API `cook.probe(key, opts)` (§{cat.probes.api}) and MUST treat probe units as ordinary participants in the execution DAG (§{cat.probes.exec}). Probe outputs MUST be cached using the same machinery as recipe outputs (§{exec.cache}), enabling cross-invocation memoisation and cross-machine sharing via Cook Cloud.

A unit MAY declare a dependency on one or more probe outputs via `probes = { key, ... }` on `cook.add_unit` (§{cat.probes.consumer}). The implementation MUST add a DAG edge from each named probe to the consuming unit and MUST fold each probe's fingerprint into the consumer's fingerprint, ensuring transitive invalidation.

## 22.5.2. Native probe declaration [#cat.probes.decl]

In addition to the register-phase `cook.probe()` API (§{cat.probes.api}), a
conforming implementation MUST accept a **native `probe` declaration**: a
column-0 construct (a sibling of `recipe`, `register`, and `chore`), implicitly
register-phase, that declares one probe. The declaration is surface sugar: it
MUST be equivalent to the `cook.probe(key, opts)` call its header and body
denote (§{cat.probes.api}). The `cook.probe()` API is retained for
dynamic registration; the two register into the same probe table and share the
duplicate-key diagnostic of §{cat.probes.api}.

The `probe_body` (App. A) comprises an optional declarative `ingredients_step`
(§{steps.ingredients}) followed by exactly one **producer**. A conforming
implementation MUST reject a probe body that: contains no producer; contains
two or more producers; contains two or more `ingredients`; or contains an
`ingredients` line after the producer. The header dependency list
(`probe NAME: a b`) names upstream probe keys and is equivalent to
`inputs.requires` (§{cat.probes.inputs}).

A **producer** has no introducer keyword: the producer KIND leads, and the
braces are that kind's body. There are six forms — a bare shell block `{ … }`, a
`json`-typed shell block, a `lines`-typed shell block, a `tools` name list, an
`envs` name list, and a `>{ … }` Lua block (specified below). The tokens `json`,
`lines`, `tools`, and `envs` are **contextual keywords** recognised only in this
probe-producer position; elsewhere they are ordinary identifiers.

A probe's `ingredients` declares its **file-input fingerprint set** — NOT an
iteration driver. A probe yields one value and never fans out; the
`ingredients` globs (with `!` excludes, §{steps.ingredients}) populate
`inputs.files` (§{cat.probes.inputs}), each file's content hash folding.

**Diagnostics.** A conforming implementation MUST reject the following probe
bodies at parse time with a diagnostic that identifies the probe by name and
the offending line:

- No producer — a conforming implementation MUST reject the probe with a
  diagnostic indicating that the producer is absent.
- Two or more producers — a conforming implementation MUST reject the second
  producer with a diagnostic indicating that at most one producer is permitted
  per probe.
- Two or more `ingredients` blocks — a conforming implementation MUST reject the
  second `ingredients` with a diagnostic indicating that at most one `ingredients`
  is permitted per probe.
- An `ingredients` block appearing after the producer — a conforming
  implementation MUST reject with a diagnostic indicating that `ingredients`
  must precede the producer.
- A body step that is neither `ingredients` nor a producer (e.g. a `cook`,
  `test`, or other non-producer step) — a conforming implementation MUST
  reject with a diagnostic indicating that only `ingredients` and a producer are
  permitted in a probe body.
- A probe name containing three or more colon-separated segments (e.g.
  `probe a:b:c`) — a conforming implementation MUST reject at the lexical level
  with a diagnostic indicating that a probe key is at most `IDENT:IDENT`.
- A `json` or `lines` kind applied to a `>{ … }` Lua block (e.g. `json >{ … }`)
  — a conforming implementation MUST reject with a diagnostic indicating that
  `json`/`lines` are valid only on a shell block, since a Lua block already
  returns a structured value.
- A `tools` / `envs` name list that is empty, contains a non-`IDENT` name, or is
  given a `>{ … }` Lua block in place of a name list — a conforming
  implementation MUST reject with a diagnostic identifying the probe and the
  offending construct.

Each segment of a bare probe key admits the recipe-name character set (internal `-` and `.`; §A.3.2 `PROBE_SEG`, CS-0131), so `demo:cc-version` is a single two-segment key. The two-`:`-segment maximum is unchanged: a bare key is at most `PROBE_SEG:PROBE_SEG`. The quoted `STRING` form denotes the same key and is the escape hatch for spellings outside `PROBE_SEG`.

A shell-block producer (`{ … }`, `json { … }`, `lines { … }`) and the Lua-block
producer (`>{ … }`) reuse the `body ::= exec_lua_block | shell_block` grammar of
§{steps.cook-single} (the body-dispatch: `>{` is tested before `{`). The probe
value is determined by the producer form:

- `>{ … }` (Lua block): the probe value is what the block `return`s
  (§{cat.probes.values}).
- `{ … }` (bare shell block): the probe value is the block's standard output as a
  string (one trailing newline trimmed).
- `lines { … }` (shell block): the probe value is the block's standard output
  split on `LF` into an array of strings.
- `json { … }` (shell block): the probe value is the block's standard output
  parsed as one JSON value. Decoding is explicit and validating — a conforming
  implementation MUST fail the probe (NOT silently fall back to a string) when
  the output is not well-formed JSON.
- `tools { NAME, … }` (name list): the brace content is a
  comma/whitespace-separated LIST of bare tool names (`IDENT`), NOT a shell or
  Lua body. A conforming implementation MUST PATH-resolve each name and
  content-hash the resolved binary; the probe value is a JSON object keyed by
  tool name, each entry an object `{ "path": <resolved path>, "hash": <content
  hash> }`. The hash is BOTH the probe value AND its re-run trigger: a change to
  the resolved binary changes the hash, re-fingerprinting the probe and any
  dependent unit — freshness without a `volatile` annotation. A name that does
  not resolve on PATH MUST fail the probe with a diagnostic naming the tool. A
  conforming implementation MUST reject an empty list (`{ }`) and a `>{ … }` Lua
  block (the brace content is a name list, not a body).
- `envs { NAME, … }` (name list): the brace content is a
  comma/whitespace-separated LIST of bare environment-variable names (`IDENT`),
  NOT a shell or Lua body. The probe value is a JSON object keyed by variable
  name; an unset variable is OMITTED from the object. Reading each variable
  records it as a consulted-environment determinant (§{cat.probes.inputs}), so
  the environment values fingerprint the probe — an `envs` probe is a named,
  cached, shareable env-set fingerprint, the env-twin of `tools`. A conforming
  implementation MUST reject an empty list (`{ }`) and a `>{ … }` Lua block.

The `json` and `lines` kinds are valid ONLY on the shell-block form; a conforming
implementation MUST reject either applied to a `>{ … }` Lua-block producer with a
register-phase diagnostic, since a Lua block already returns a structured value.
The `tools` and `envs` kinds take a brace NAME LIST, not a body; a conforming
implementation MUST reject a `>{ … }` Lua block (or any body form) supplied where
a name list is required.

**Coexistence.** The native `probe` declaration and the register-phase
`cook.probe()` API register into one probe table within a register pass; a key
declared by both MUST fire the duplicate-key diagnostic of §{cat.probes.api}.

> **Note (informative).** A `probe` whose producer decodes JSON/YAML
> (`json { … }`, or `cook.json_decode`/`cook.yaml_decode` in a `>{ … }` body)
> MAY be consumed through either path: the codecs are both-phase API
> (§{lua.cook-codecs}, CS-0123), installed on the register pre-pass VM and on
> every execute-phase worker VM alike.

**Lowering (informative).** The reference implementation lowers a `probe`
declaration to the equivalent `cook.probe(key, opts)` call (§{cat.probes.api}):
the header dependency list becomes `inputs.requires`; `ingredients` (globs and
`!` excludes) is resolved at register time to the concrete file set populating
`inputs.files` (§{cat.probes.inputs}); and the producer becomes the
`opts.produce` Lua source string — a `>{ … }` block verbatim, a shell block
wrapped as `cook.sh(...)` typed by its leading kind (bare → stdout string with
one trailing newline trimmed; `lines { … }` → newline-split array; `json { … }`
→ `cook.json_decode`). The `tools`/`envs` name lists become `inputs.tools` /
`inputs.env` determinant declarations. Because the fingerprint folds the
`opts.produce` source string (§{cat.probes.inputs}), editing a shell command or
changing the producer kind changes the lowered string and therefore
re-fingerprints; the fingerprint algorithm itself is unchanged.

## 22.5.3. The `cook.probe` API [#cat.probes.api]

**Signature.** `cook.probe(key: string, opts: ProbeOpts) -> nil`

**Phase.** Register-phase only.

`cook.probe(key, opts)` registers a probe unit named `key`. The function MUST be available on the register-phase VM. Calls from the execute-phase VM MUST raise a Lua runtime error per §{rationale.execute-phase-api}.

`key` MUST be a non-empty string. Implementations SHOULD recommend the convention `"<module-prefix>:<name>"` (e.g. `"cc:zlib"`) but MUST NOT enforce it syntactically.

A second `cook.probe` call with the same `key` within one register pass MUST raise a register-phase diagnostic naming both source locations:

```
probe key 'cc:zlib' declared at <file>:<line>; previously declared at <file>:<line>
```

`opts` is a Lua table with the following keys:

| Key | Type | Required | Semantics |
|---|---|---|---|
| `inputs` | table | yes | Declared inputs (§{cat.probes.inputs}). MAY be an empty table. |
| `produce` | string | yes | Lua source code (the body of a producing function); the implementation wraps it in `function() ... end` and invokes it on a worker VM during the execute phase on a cache miss. MUST return a serialisable value (§{cat.probes.values}). |

A conforming implementation MUST raise a register-phase error if `opts.inputs` is absent or not a table, or if `opts.produce` is absent or not a string.

### Example 22.5.3.1

```lua
cook.probe("cc:zlib", {
  inputs = {
    env   = { "PKG_CONFIG_PATH", "PATH" },
    tools = { "pkg-config" },
  },
  produce = [[
    local r = run_pkg_config("zlib")
    return { found = r.found, cflags = r.cflags, libs = r.libs }
  ]],
})
```

## 22.5.4. Probe inputs [#cat.probes.inputs]

`opts.inputs` is a table with optional sub-keys. Omitted sub-keys default to the empty list.

| Sub-key | Type | Semantics |
|---|---|---|
| `env` | list[string] | Env-var names. The current value of each named variable contributes to the probe's fingerprint. Reuses the env-contribution mechanism of §{exec.cache.abstract}. |
| `tools` | list[string] | Tool names (e.g. `"pkg-config"`). Resolved via PATH at probe execution time; the content hash of the resolved binary contributes to the probe's value, which folds into the cache key of any unit that seals on this probe (§{exec.cache.single-key}). A tool not found on PATH folds as `(name, "<missing>")`; the probe runs and SHOULD surface a diagnostic. |
| `files` | list[string] | File paths relative to the Cookfile root. The content hash of each file contributes to the fingerprint. A missing file folds as `(path, "<missing>")`. |
| `requires` | list[string] | Upstream probe keys. Each named probe's full fingerprint is folded into this probe's fingerprint (chained invalidation). The `requires` graph among probes MUST be acyclic (§{cat.probes.cycles}). |

A probe's fingerprint MUST be a 32-byte SHA-256 digest over the canonicalised byte representation of its inputs. The canonical representation is built by concatenating, in this fixed order, each of the following sections separated by an ASCII `LF` (0x0A):

1. The literal bytes `COOK_PROBE_FP_V2`.
2. The probe `key` (UTF-8 bytes).
3. The `produce` source string (UTF-8 bytes).
4. The literal bytes `ENV`, followed by each declared env-var name and its current value (or the literal bytes `<unset>` if not set), `<name>=<value>`-formatted, sorted lexicographically by name.
5. The literal bytes `TOOLS`, followed by each declared tool name and the 32-byte content hash of the resolved binary (32 zero bytes if missing), `<name>=<hex>`-formatted, sorted lexicographically by name.
6. The literal bytes `FILES`, followed by each declared file path and its 32-byte content hash (32 zero bytes if missing), `<path>=<hex>`-formatted, sorted lexicographically by path.
7. The literal bytes `UPSTREAM`, followed by each upstream probe key and its 32-byte fingerprint, `<key>=<hex>`-formatted, sorted lexicographically by key.

The marker was bumped from `COOK_PROBE_FP_V1` by CS-0102 when the probe-value encoding changed from msgpack to canonical JSON, so artifacts written under the old encoding are never addressed by a conforming implementation.

The implementation MAY include a non-cryptographic hash (e.g. xxh3_64) as an internal optimisation provided the final 32-byte fingerprint matches what a direct SHA-256 over the canonical bytes would produce.

## 22.5.5. Value types [#cat.probes.values]

The function produced by wrapping the `produce` source string (§{cat.probes.api}) MUST return one of:

- `nil`
- `boolean`
- `number` (integer or float)
- `string` (valid UTF-8; see below)
- A table whose keys are all strings, OR all integers in `1..N` with no holes (array shape), and whose values are recursively of these types.

Returning a `function`, `userdata`, `thread`, a mixed-key table, or a table containing a cycle MUST cause the probe to fail with a diagnostic naming the offending path. The diagnostic MUST follow the form:

```
probe 'cc:zlib' returned a non-serialisable value at .cflags[3] (function)
```

Probe values are serialised as **canonical JSON** for storage, materialisation, and cloud transport. The canonical rendering is UTF-8 JSON, pretty-printed with two-space indentation, object keys sorted bytewise ascending, with exactly one trailing LF. There is exactly ONE canonical byte sequence per value; a conforming implementation MUST use those bytes verbatim at every serialising site — the `.cook/probes/<key>.json` file (§{cat.probes.exec}), the `CacheBackend` artifact body (`kind: "probe_value"`), and wire transport — without re-encoding. Any hash over a probe value MUST be over the canonical bytes.

The JSON encoding imposes two constraints beyond the value-type list above: a `string` MUST be valid UTF-8, and a `number` MUST be finite (NaN and ±Inf are rejected). A violation MUST cause the probe to fail with the non-serialisable diagnostic above, naming the offending path.

## 22.5.6. Consumer `probes` [#cat.probes.consumer]

`cook.add_unit({ ..., probes = { "key1", "key2" } })` declares that the unit consumes the named probe values. The `probes` field is an optional list of probe-key strings; omitting it is equivalent to `probes = {}`. (Distinct from `opts.inputs.requires` per §{cat.probes.inputs}, which declares probe-to-probe dependencies inside a probe.)

A conforming implementation MUST, after the register pass completes:

1. Resolve each key against the set of registered probes. An unknown key MUST raise:
   ```
   unit '<name>' lists probe key '<key>' in `probes` but no such probe was declared
   ```
2. Add a DAG edge from each resolved probe to the consumer unit.
3. Fold each resolved probe's fingerprint into the consumer unit's fingerprint.

The relative order of `cook.probe` and `cook.add_unit` calls within one register pass is unconstrained; resolution is deferred to end-of-pass.

## 22.5.7. Command-template desugaring [#cat.probes.templates]

Consumer commands MAY include `$<key>`, `$<key.field>`, and `$<key.field[i]>` placeholders that reference probe outputs declared in `probes`. The implementation MUST detect these placeholders during codegen (or register-time capture for direct `cook.add_unit` calls) and rewrite the command into a Lua step that calls `cook.cache.get(key)` (§{lua.both}) and interpolates the resolved value before invoking the shell.

The placeholder syntax reuses Cook's existing `$<IDENT>` sigil grammar (§{phl.token}). A sigil whose IDENT contains a colon (`:`) **and does not begin with the reserved `file:` prefix (§{xref.file-namespace}, CS-0101)** is unambiguously a probe-value reference; the implementation MUST dispatch it to the cache-read path described here. A sigil whose IDENT contains no colon retains its existing register-time semantics (built-ins, recipe names, env vars).

These rules apply uniformly to probe-value references written in native `cook`-step bodies (§{steps.cook-single} / §{steps.cook-multi}, including `for_each` bodies) and to raw `cook.add_unit` command strings. In both cases the reference MUST reach register-time unit capture as literal `$<...>` text inside the unit's `command` **string**; the capture layer performs the rewrite described above. An implementation MUST NOT lower a probe-value reference to a non-string `command` value (e.g. a deferred Lua function): `cook.add_unit` MUST reject a `command` that is not a string with a register-phase diagnostic naming the received type. (CS-0122; defect COOK-187 — the reference implementation previously emitted a function-valued `command` for probe-referencing native bodies, which was coerced to the empty string and silently succeeded.)

The literal-text rule applies regardless of which step kind's body originated the command, including `cook` and `test` bodies in `for_each` recipes (§8.3, §8.6). The register-time rewrite described above turns the command into a Lua chunk; that chunk MUST run under the originating step kind's sandbox policy (§{lua.fs-sandbox}), not a default one. (CS-0127.)

Substitution rules:

- `$<key>` substitutes the entire probe value as a string. The value MUST be a string (or coercible via `tostring`).
- `$<key.field>` substitutes a named field of the probe's table value.
- `$<key.field[i]>` substitutes a one-based array element of a table-valued field.

Malformed placeholders (referencing a key not in `probes`, or accessing a field on a non-table value) MUST raise a register-phase or execute-phase diagnostic naming the placeholder.

### Example 22.5.7.1

```lua
cook.add_unit({
  name     = "myapp.o",
  inputs   = { "myapp.c" },
  outputs  = { "build/myapp.o" },
  probes   = { "cc:compiler", "cc:zlib" },
  command  = "$<cc:compiler.path> -c $<in> -o $<out> $<cc:zlib.cflags>",
})
```

The implementation rewrites this command at codegen time into a Lua step that reads `cook.cache.get("cc:compiler")` and `cook.cache.get("cc:zlib")` and interpolates the results before executing the shell command.

## 22.5.8. Execution semantics [#cat.probes.exec]

A probe unit MUST run at most once per build invocation.

**Cache hit.** When the probe's fingerprint addresses an existing artifact in the cache backend, the implementation MUST load the cached bytes, validate that they parse as JSON (a parse failure MUST be treated as a cache miss), populate the per-run probe-value store, and skip execution of the `produce` source.

**Cache miss.** When no cached artifact is found, the implementation MUST dispatch the probe to a worker VM, wrap the `produce` source string in `function() ... end`, parse and invoke the resulting function, capture the return value, validate it against §{cat.probes.values}, render it to canonical JSON (§{cat.probes.values}), store the artifact via the cache backend, and populate the per-run probe-value store. If the `produce` source string is malformed Lua, the resulting parse error MUST fail the unit with a diagnostic naming the probe key and the parse error.

The per-run probe-value store is keyed by probe key. Consumer units are dispatched after their `requires`-edge predecessors complete; at dispatch time, the store is fully populated for all required probes.

`cook.cache.get(key)` (§{lua.both}) invoked from execute-phase Lua MUST read from the per-run probe-value store. A `cook.cache.get` call for a key that has not completed as a probe MUST return `nil`.

The wrapped `produce` function MAY raise a Lua error. A conforming implementation MUST treat a raised error as a unit failure and MUST propagate failure to every downstream consumer via the DAG edge, consistent with the general unit-failure semantics of §{exec.two-phase}.

**Demand-driven scheduling.** A probe unit MUST NOT execute on an invocation unless at least one scheduled non-probe unit lists the probe's key in its `probes` field, directly or transitively through probe-on-probe `inputs.requires` chains. A conforming implementation MUST silently omit unreached probe nodes from the execution DAG. The implementation MUST NOT compute, look up, or persist the fingerprint of an unreached probe; MUST NOT emit a user-facing diagnostic; and MUST NOT log a warning at the user-facing diagnostic layer. Debug logging at lower verbosity tiers is permitted.

### Example 22.5.8.1

```
register
    cook.probe("env:slow", {
        inputs = {},
        produce = [[ return { ts = os.time() } ]],
    })

recipe foo
    cook.add_unit({
        name = "u1", inputs = {}, outputs = {"build/foo.txt"},
        probes = {"env:slow"},
        command = "echo $<env:slow.ts> > build/foo.txt",
    })
    test { cat build/foo.txt }

recipe bar
    test { echo bar }
```

The bare `cook.add_unit(…)` line is a register-phase `module_call` step (§{steps.dispatch}, rule 7) — the CS-0134 form of what a pre-CS-0134 Cookfile wrote as `>>{ cook.add_unit(…) }`. Each `test { … }` block body is an execute-phase verification unit.

`cook foo` MUST execute the `env:slow` probe and `u1` exactly once.
`cook bar` MUST execute no probes; `env:slow` is unreached and its
fingerprint MUST NOT be computed.

**On-disk layout.** Probe artifacts reuse the existing recipe-cache layout. No new directories are introduced for cache storage. The meta sidecar MUST carry a `kind: "probe_value"` field to disambiguate probe-artifact sidecars from file-artifact sidecars. All other fields of the meta sidecar (integrity hash, schema version) follow §{exec.cache.integrity}.

In addition, on every probe completion — cache hit or miss, demand-driven or pre-pass (§{cat.probes.for-each}) — the implementation MUST materialise the canonical bytes at `.cook/probes/<file>` under the project root, where `<file>` is the probe key with the characters `_`, `/`, and `\` escaped for injectivity (the reference implementation rewrites `_` → `_5f`, `/` → `_2f`, `\` → `_5c`, so no two distinct keys share a file name), suffixed `.json`. This file is the canonical local copy of the probe value; the per-run probe-value store is a read-through cache of it.

## 22.5.9. Cycle detection [#cat.probes.cycles]

The probe `requires` graph MUST be acyclic. A conforming implementation MUST detect cycles at the end of the register pass and MUST raise a diagnostic before any unit executes:

```
probe cycle detected: cc:a -> cc:b -> cc:a
```

The diagnostic MUST render the full cycle path. The order in which nodes are listed MUST be deterministic (implementation-defined ordering of cycle start); the path MUST close back to the first node.

## 22.5.10. Probe values as `ingredients <probe>` sources [#cat.probes.for-each]

An **array-shaped** probe value (§{cat.probes.values}) MAY be consumed as the source of an `ingredients <probe>` declaration (§{steps.ingredients}): `ingredients <key>` registers one work unit per array element, and `ingredients <key>:<field>` iterates the array at the named field. This is a second probe consumption mode alongside the `$<key:field>` injection of §{cat.probes.templates}; the same `cook.probe` definition serves both.

**Source-ref resolution.** A probe key is itself at most two `:`-separated segments (§{cat.probes.decl}; `ns:name` is the canonical naming), so a multi-segment source ref is lexically ambiguous between a probe key and a `<key>:<field>` selector. A conforming implementation MUST resolve the ref against the set of declared probes once registration-time declarations are known: a declared probe whose key equals the entire ref wins; otherwise the segment after the final `:` is a field selector and the remaining prefix MUST name a declared probe. A ref that names no declared probe under either interpretation MUST be rejected with a register-phase diagnostic naming the full ref. A source ref of four or more segments is malformed and MUST be rejected at parse time.

**Two-category discriminator (normative; CS-0131).** In `ingredients` position the bare-vs-quoted distinction is load-bearing and MUST NOT be flattened: a bare `ingredients cards` names a probe key (a namespace identifier — the fan-out member source of this section), whereas a double-quoted `ingredients "cards"` is a literal filesystem glob (§8). The two forms are NOT interchangeable. This is the general law: a double-quoted operand is a literal filesystem string (paths, globs, output patterns); a bare operand is a namespace identifier (module name, probe key, recipe dependency). It differs from the `probe_name` / `probe_ref` positions, where bare and `STRING` denote the *same* probe key and are interchangeable — there the quotes are only an escape hatch for un-spellable characters, not a category switch.

An `ingredients <probe>` source is the canonical case where a recipe's declared output set *shrinks* between runs: removing a data member drops the unit (and the output) it drove, so the dropped member's artifact is swept by stale-output reconciliation (§{exec.cache.reconcile}) on the next run.

**Pre-pass evaluation.** A probe that feeds an `ingredients <probe>` source MUST be evaluated **before unit registration completes**, because the fan-out cardinality — the number of units the consuming recipe produces — is the length of the probe's array value. A conforming implementation MUST resolve every such probe of a reachable recipe (cache hit or `produce` execution, per §{cat.probes.exec}) before it registers that recipe's per-member units, then register the units statically. The build DAG does not grow after registration; data-member iteration introduces no dynamic registration.

**Demand-driven.** The pre-pass preserves the demand-driven rule of §{cat.probes.exec}: a probe feeding an `ingredients <probe>` source MUST NOT be evaluated unless its consuming recipe is reachable from the build target. Recipe-level reachability is determined from recipe headers and dependency edges before per-member fan-out, so the pre-pass evaluates only the probes of recipes that will register units.

**Static-input rule.** A probe used as an `ingredients <probe>` source MUST be statically evaluable at the pre-pass: its declared `inputs` (§{cat.probes.inputs}) MAY reference `files`, `env`, and `tools`, but MUST NOT depend — directly or transitively through `inputs.requires` — on a build artifact produced by another recipe. A conforming implementation MUST reject such a dependence with a register-phase diagnostic:

```
probe '<key>' feeds an ingredients <probe> source but depends on build artifact '<path>'; sources must be statically evaluable
```

Iterating over artifacts produced earlier in the same build (the codegen-then-process pattern) would require interleaving execution and registration; it is reserved for a future amendment.

**Non-array value.** An `ingredients <probe>` source whose resolved probe value (or selected field) is not an array-shaped table (§{cat.probes.values}) MUST surface a register-phase diagnostic naming the probe key and the resolved value's shape.

**Per-member units.** Each array element becomes one data member, bound as `item` in the consuming recipe's per-member steps (§{steps.ingredients}). Each per-member unit's fingerprint folds the materialised element (§{exec.cache}); the probe-value fingerprint covers the set, so editing one element re-runs only its unit, while editing the probe's declared inputs re-evaluates the set and re-runs only the members that changed.

> **Note (informative).** An implementation must learn a recipe's `ingredients <probe>`-feeding probe *before* running the recipe body (which reads the resolved array). The reference implementation surfaces the source on the recipe's registration metadata — a `__for_each` descriptor naming the probe key (and optional field) — so the pre-pass resolves the probe from the recipe header without first executing the body (COOK-64). Recipe-level reachability (above) then bounds the pre-pass to the probes of recipes the build target actually reaches.

## 22.5.11. Implementation status (CS-0074) [#cat.probes.impl-status]

CS-0102 retired the msgpack encoding referenced below in favour of canonical JSON (§{cat.probes.values}); the structural milestones stand as recorded.

C4–C6 implemented: requires resolution (§22.5.6), cycle detection (§22.5.9), execute-VM guard (§22.5.3).

F1–F3 implemented: rmpv dependency, Lua→msgpack walker (`probe_value.rs`), msgpack round-trip tests.

G1 implemented: `WorkPayload::Probe` dispatched on worker VM; `produce` source wrapped in `function() ... end`, return value converted to msgpack bytes, stored in `ProbeOutput`.

G4 implemented: `execute_dag` accepts `probe_units_by_node: &BTreeMap<usize, ProbeUnit>` (keyed by DAG node id). Before dispatching a `WorkPayload::Probe` node to a worker, the scheduler resolves its fingerprint via `resolve_probe_inputs` + `compute_probe_fingerprint` and attempts `CacheBackend::get`. On a hit, the cached msgpack bytes are inserted directly into the per-run probe-value store and the node is completed without worker dispatch (emits `NodeCacheHit`). On a miss, dispatch proceeds normally. Fingerprints are stored in `probe_fingerprint_by_node` keyed by DAG node id; `upstream_probe_fingerprints` is accumulated in topological order as probes complete, so downstream probe fingerprints chain correctly.

G5 implemented: after a worker returns with `probe_output: Some(...)` and `success: true`, the scheduler persists the msgpack bytes to `CacheBackend` via `put_bytes` using the fingerprint from `probe_fingerprint_by_node` and `ArtifactMeta::as_probe_value()`. The upstream fingerprint map is then updated so downstream probes can include this probe's fingerprint in their own. Three unit tests in `executor.rs` cover the cache-hit path (skips dispatch, `NodeCacheHit` emitted), the cache-miss path (worker executes, artifact persisted with correct key), and the invalidation path (changed env var ⇒ different fingerprint ⇒ cache miss even when prior entry exists).
