# 9. Standard Modules [#stdmods]
> **Normative.** This chapter specifies the public surface of blessed Cook modules — modules curated by the Cook project and distributed through the official LuaRocks-backed registry (`rocks.usecook.com`). A module's presence as a numbered subsection of this chapter is what makes it "blessed": a conformance-checkable contract that any implementation of the module MUST honour.

The Standard governs the contract, not the reference implementation. The published rock is one implementation that meets the contract; a reimplementation in another language layer MUST produce semantically equivalent behaviour for every public surface specified here.

## 9.1. Scope and bootstrap [#stdmods.bootstrap]

### 9.1.1. Bootstrap [#stdmods.bootstrap.install]

A conforming project obtains a blessed module by listing it under `[modules]` in `cook.toml` and running `cook modules install`. The install populates `cook_modules/share/lua/5.4/<name>/init.lua` (and `cook_modules/lib/lua/5.4/<name>.<so-ext>` for C extensions) under the calling Cookfile's `cook_modules/` directory, per Standard §7's resolution order.

### 9.1.2. Vendoring escape hatch [#stdmods.bootstrap.vendor]

A project MAY hand-vendor a copy of a blessed module by placing `cook_modules/<name>.lua` or `cook_modules/<name>/init.lua` at the top of the project's `cook_modules/` directory. Per Standard §7, hand-vendored top-level paths take precedence over LuaRocks-installed `share/lua/5.4/` paths. Hand-vendored copies are the project's responsibility; the implementation MUST treat them as the source of truth without consulting the published rock or the lockfile.

A future `cook modules install` MUST NOT overwrite a hand-vendored top-level file; the install step inserts only into `cook_modules/share/lua/5.4/` (and siblings) per the standard layout.

## 9.2. cc — C-family build module [#stdmods.cc]

### 9.2.1. Synopsis and conformance scope [#stdmods.cc.synopsis]

The `cc` module specifies a portable surface for declarative C and C++ native builds. It dispatches on source-file extension (`.c`, `.C` → `cc`; `.cpp`, `.cc`, `.cxx`, `.C++` → `cxx`) so a single target may mix C and C++ sources without per-target language opt-in.

A conforming implementation MUST expose the eleven public functions listed in §9.2.3 and MUST honour their option tables, return values, and propagation semantics. A conforming implementation SHOULD provide informative error diagnostics for the negative cases described per-function.

The official `cook_cc` rock at `rocks.usecook.com` is the reference implementation. Conformance is independent of the rock's internal file decomposition; only the surface returned from `require("cook_cc")` is normative.

### 9.2.2. Identity, resolution, init hook [#stdmods.cc.identity]

The blessed rock is published as `cook_cc`. A Cookfile imports it by:

```
use cook_cc as cc
```

Per Standard §6 (`use`-env) and §7 (cross-Cookfile resolution), `use cook_cc as cc` MUST bind the loaded module table to the local name `cc` in the Cookfile's scope.

The implementation's module table MUST include an `init` function honoured by the loader contract at Standard §6.3.4: the loader MUST call `init()` exactly once per Lua VM that loads the module. The `init` function MAY rehydrate state from `cook.cache` (e.g., a previously-detected compiler triple) and MUST NOT be exposed as part of the user-callable surface (callers reading §9.2.3 do not invoke `init()` directly).
