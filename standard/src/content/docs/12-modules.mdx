# 12. Modules [#mods]
> **Normative.** This chapter defines the `use` declaration, module resolution, the load-time lifecycle, and the catalogue index that points at Part IV. The Lua API call `cook.load_module` that implements resolution at runtime is defined in §{lua.both}; this chapter specifies what the surface declaration means and when module code runs.

## 12.1. The `use` declaration [#mods.use]

A `use` declaration has the form `use <name>`, where `<name>` is either a `BARE_IDENTIFIER` or a `STRING` (App. A.2; the lexical and syntactic rules are restated in §{decl.use}). A conforming implementation MUST resolve a `use` declaration by applying the following steps in order, with the first matching step winning:

1. If `<name>` designates a built-in module (§{mods.builtin}), the built-in module is selected.
2. Otherwise, the implementation MUST attempt to resolve `<name>` as a local module under `cook_modules/` by the lookup algorithm of §{mods.local}.

A conforming implementation MUST reject a Cookfile for which no step succeeds, with a diagnostic that names `<name>` and indicates which locations were searched. The resolution step MUST occur before any recipe body executes; failure to resolve is a load-time error, not a run-time error.

The result of resolution is a Lua table bound in the Cookfile's Lua environment under the identifier derived from `<name>` by replacing each ASCII hyphen (`-`) with an underscore (`_`). All other characters admitted by `BARE_IDENTIFIER` (App. A.5) pass through unchanged. A conforming implementation MUST make this binding visible to every subsequent `lua_line`, `lua_block`, `cook_step` body, top-level `module_call` (§{toplevel.module-call}), recipe-body `module_call` (§{steps.dispatch}), and `register_block` body (§{decl.register}) in the same Cookfile.

A module MAY expose a function named `init` on its returned table. If present, `init` MUST be called exactly once immediately after the module's Lua source has been executed, before any recipe body in the enclosing Cookfile is registered. The full lifecycle contract for `init()` and for module loading is in §{mods.lifecycle}.

### Example 12.1.1

```cook
use cpp

cpp.bin("colorize", { sources = { "src/colorize.c" } })
```

The declaration `use cpp` binds the identifier `cpp` in the Cookfile's Lua environment to the table returned by `cook_modules/cpp.lua` (or `cook_modules/cpp/init.lua`). The top-level `cpp.bin(...)` call is dispatched as a top-level module_call (§{toplevel.module-call}) — a register-phase expression that runs against the same register-phase Lua VM as a `register_block` body.

### Note 12.1.1

The hyphen-to-underscore rewrite exists because `BARE_IDENTIFIER` permits internal hyphens (App. A.5) whereas Lua identifiers do not. Thus `use my-mod` binds the Lua global `my_mod`, but the on-disk lookup searches for `cook_modules/my-mod.lua` verbatim: the disk name is not rewritten.

## 12.2. `use` scope is lexical per Cookfile [#mods.use-scope]
A `use` declaration's alias is scoped to its **declaring Cookfile** and is not visible across `import` boundaries. Each Cookfile in a workspace defines its own `use`-resolution scope. The aliases established by `use foo` in Cookfile A are not in scope for any Lua evaluated from Cookfile B's recipes, and vice versa, regardless of import direction.

For an execute-phase work unit, "the calling work unit's source Cookfile" (§{lua.cook-load-module}) is the Cookfile that contained the recipe whose body unit is being evaluated. A conforming implementation MUST resolve `cook.load_module(name)` and `require(name)` against that Cookfile's `cook_modules/` directory and MUST NOT fall back to a sibling Cookfile's `cook_modules/`.

A conforming implementation MUST resolve `cook.load_module(name)` at register phase, and MUST configure `package.path` and `package.cpath` for each execute-phase work unit, such that the calling Cookfile's `cook_modules/` is searched in the following order: (1) `cook_modules/?.lua` and `cook_modules/?/init.lua` (hand-vendored, top level); (2) `cook_modules/share/lua/<lua-version>/?.lua` and `cook_modules/share/lua/<lua-version>/?/init.lua` (LuaRocks-installed pure Lua); for native modules: (a) `cook_modules/?.<so-ext>` (hand-vendored, top level); (b) `cook_modules/lib/lua/<lua-version>/?.<so-ext>` (LuaRocks-installed). `<lua-version>` is the embedded Lua's MAJOR.MINOR (currently `5.4`). `<so-ext>` is the platform's loadable-module extension (`.so` on Linux/macOS, `.dll` on Windows).

A conforming implementation MUST also configure `package.path` and `package.cpath` (following the same search order) on the register-phase Lua VM before executing the body of any module loaded via `cook.load_module`. This ensures that sub-requires within a multi-file module (e.g., `require("cook_cc.toolchain")` inside `cook_cc/init.lua`) resolve against the project's `cook_modules/` rather than the system Lua paths. A conforming implementation MUST allow C extension modules (`.so`/`.dll`) to be loaded during register-phase `cook.load_module` evaluation; the register-phase VM MUST NOT restrict native module loading. [amended CS-0066]

A conforming implementation MAY expose `cook.load_module(name)` on the execute-phase Lua VM as an extension over the register-phase requirement above; when exposed, the execute-phase `cook.load_module` MUST honour the same four-path resolution order. Recipe-body Lua scripts that call `cook.load_module(name)` directly therefore see the same set of candidate paths as register-phase callers, eliminating "works at register but not at execute" surprises. [amended CS-0069]

Cross-Cookfile composition is at the **recipe** level via qualified name references (§{comp.qualified-refs}), not at the **module** level. Sharing module code between Cookfiles happens via the filesystem (a vendored copy, a symlink, or a shared parent) — the cook runtime treats each Cookfile's `cook_modules/` as an independent search root.

### Example 12.2.1

```
ws/Cookfile:        import b ./b
                    use cpp           # resolves to ws/cook_modules/cpp.lua
                    recipe top
                        cpp.plan("a")               # register-phase
                        > cpp.assemble("a")         # execute-phase, same alias
                        b.build                     # qualified dep on B's recipe
                    end

ws/b/Cookfile:      use proto         # resolves to ws/b/cook_modules/proto.lua
                    recipe build
                        proto.regen()
                        > proto.verify()
                    end
```

`top`'s body unit sees `cpp` (only). `b.build`'s body unit sees `proto` (only). Neither sees the other's modules. Filesystem-level sharing — for example, two Cookfiles vendoring the same `cpp.lua` — is independent: each Cookfile's worker VM caches its load by `(working_dir, name)`, so the same source file evaluated from two Cookfiles is two distinct module instances with independent state.

## 12.3. Module lifecycle [#mods.lifecycle]

This section specifies when module code executes and what scope it observes.

### 12.3.1. Load order [#mods.lifecycle.load-order]

A conforming implementation MUST execute module code in the following order, for each Cookfile in a workspace, before any recipe body of that Cookfile registers:

1. For each `use <name>` declaration (§{mods.use}) in source order, resolve `<name>` against the search paths of §{mods.local} and §{mods.builtin}. The first hit wins.
2. Execute the resolved module's top-level chunk on the calling Cookfile's register-phase Lua VM. The chunk MUST return a Lua table; a chunk that does not return a table is a load-time error. The returned table is bound under the alias derived from `<name>` per §{mods.use}.
3. If the returned table has an `init` field bound to a function value, call `init()` exactly once on the register-phase VM. The `init()` call MAY read `cook.probes` (§{lua.both}) and MAY call register-phase Cook Lua API (§{lua.reg}).
4. Repeat for the next `use <name>` declaration in source order. A later `use` declaration's module body MAY observe state established by an earlier `use` declaration's `init()`.

Module code MUST NOT register work units during top-level chunk execution. Work-unit registration is the responsibility of recipe bodies; modules typically expose functions (e.g. target makers) that recipe bodies call.

### 12.3.2. Per-VM caching [#mods.lifecycle.caching]

A conforming implementation MUST cache the resolved module table by `(working_dir, name)`. A second `use <name>` within the same Cookfile returns the same table without re-reading or re-evaluating the file; the top-level chunk and `init()` MUST run **at most once** per `(working_dir, name)` per Lua VM as a direct consequence of this caching contract. The cache is per-VM: a register-phase VM and an execute-phase VM that both load the same module each independently load the module top-level chunk once.

### 12.3.3. Cycle detection [#mods.lifecycle.cycles]

A conforming implementation MUST detect `cook.load_module` cycles. A cycle is a re-entrant call to `cook.load_module(name)` that occurs while an earlier load of the same `(working_dir, name)` is still in flight on this VM — that is, the earlier call's module body or `init()` has not yet returned. On detection, the implementation MUST raise a runtime error whose message begins with `module cycle detected:` and includes the cycle path rendered as the in-flight module names joined by ` -> `, with the offending re-entered name appended.

Detection MUST survive recoverable errors: when a module's body or `init()` raises, the implementation MUST remove that module from the in-flight set so a later retry on the same VM can proceed (and a genuine subsequent cycle remains detectable).

### 12.3.4. Cross-VM rehydration [#mods.lifecycle.rehydration]

When a module's `init()` records state into `cook.probes` (§{lua.both}) during register-phase load, an execute-phase VM that later loads the same module MUST observe that cached state. The execute-phase implementation MAY use in-memory-only storage scoped to the worker's lifetime (no cross-invocation persistence). [Pinned by CS-0070.]

When a module records transitive-link info via `cook.export` (§{lua.both}) at register time, an execute-phase recipe body that calls `cook.import` MUST observe the recorded info. [Pinned by CS-0071.]

### 12.3.5. Register-completion callbacks [#mods.lifecycle.finalize]

A module MAY queue work to run once, after every recipe body of the registration pass has been evaluated, by calling `cook.on_register_complete` (§{lua.cook-on-register-complete}) — from its top-level chunk, from `init()`, or from a target maker a recipe body calls. §{lua.cook-on-register-complete} is the authoritative definition of the callback's signature, ordering, and restrictions; this subsection situates it in the lifecycle this chapter defines.

A module's top-level chunk and its `init()` function (§{mods.lifecycle.load-order}) run before any recipe body in the enclosing Cookfile registers — they are this lifecycle's **start**. A callback queued via `cook.on_register_complete` runs after every recipe body of the pass has registered — it is the lifecycle's **end**, the finalize counterpart to `init()`'s start. Where `init()` lets a module establish state before any recipe body could depend on it, a queued callback lets a module act on state recipe bodies accumulated while registering — state that, by construction, is not complete until the last body of the pass has run.

## 12.4. Built-in modules [#mods.builtin]
A **built-in module** is supplied by the implementation and available without any project-level file. No built-in modules are defined by v0.1 of this Standard; the built-in kind is reserved to allow future extension (§{rationale.modules}).

## 12.5. Local modules [#mods.local]
A **local module** is a file in the project's `cook_modules/` directory. The lookup algorithm resolves `<name>` against `cook_modules/<name>.lua`, then `cook_modules/<name>/init.lua`. The first existing file wins.

## 12.6. Standard module catalogue [#mods.catalogue-index]

A conforming implementation MAY ship a curated set of **blessed modules** distributed through the official LuaRocks-backed registry (`rocks.usecook.com`). The Standard governs each blessed module's public surface in Part IV (Standard module catalogue, §{cat}). The current catalogue is:

## 12.7. Module-authoring contract [#mods.authoring]

> **Normative.** This section assembles, for the author of a Cook module (Part IV), the obligations that the phase-classified API chapters (§{lua.reg}, §{lua.exe}, §{lua.both}) and the probe and catalogue chapters (§{cat.probes}, §{cat}) define individually. It introduces no new API surface; it constrains how a module composes the existing surfaces so that a module written against it caches soundly and shares across machines. A module MUST satisfy every MUST in this section in addition to the per-API rules it cross-references.

### 12.7.1. Phase obligations [#mods.authoring.phases]

A module's top-level chunk and its `init()` function (§{mods.lifecycle}) run on the **register-phase** Lua VM. During top-level chunk execution a module MUST NOT register work units (§{mods.lifecycle.load-order}); a module instead exposes functions — target makers — that recipe bodies call to register work.

A target maker invoked from a recipe body runs at register phase and MAY call the register-phase API (§{lua.reg}), including `cook.add_unit` (§{lua.add-unit}), `cook.step_group`, `cook.recipe`, and `cook.probe` (§{cat.probes.api}).

Code a module records for later execution — a `>{ … }` body carried on a unit — runs at **execute phase**. Such code MAY call only the execute-phase API (§{lua.exe}) and the both-phase API (§{lua.both}); calling a register-only function from execute-phase Lua MUST raise a runtime error (§{rationale.execute-phase-api}). The both-phase API (§{lua.both}) is the only surface a module may call in either phase. A module MAY also queue work to run once every recipe body of the registration pass has registered, via `cook.on_register_complete` (§{lua.cook-on-register-complete}).

### 12.7.2. Registering work units [#mods.authoring.units]

A module registers a work unit by calling `cook.add_unit` (§{lua.add-unit}); that section is the authoritative field reference and a module MUST conform to its field-typing discipline (CS-0127). A module:

- MUST declare a unit's real file inputs in `inputs` and its real outputs in `output`/`outputs`, so the cache keys on content rather than on an under-declared surface;
- MUST carry a determinant a probe produces into the key via `seal` or `probes` (§{cat.probes.consumer}), never by interpolating a probe value into the command at register time in a form the cache cannot observe;
- SHOULD leave the sharing and reproducibility disposition (`sharing`, `record`) to the recipe author's surface (§{steps.cook-disposition}) rather than hard-coding it, unless the module owns a unit that is intrinsically local or non-reproducible.

### 12.7.3. Registering probes [#mods.authoring.probes]

The sanctioned surface for a module to register a memoised, cache-folded computation — compiler detection, a package-config query, a resolved toolchain identity — is a **probe**. A module registers one by calling `cook.probe(key, opts)` (§{cat.probes.api}) with `opts.inputs` declaring its determinants (`env`, `tools`, `files`, `requires`; §{cat.probes.inputs}) and `opts.produce` a **Lua source string** — the body of a producing function that the implementation wraps and runs on a worker VM on a cache miss (§{cat.probes.exec}), returning a serialisable value (§{cat.probes.values}).

A determinant a module detects MUST be modelled as a probe input so its fingerprint folds into the probe's value and thence into any consuming unit's key: an environment variable via `inputs.env` (or an `envs` native producer, §{cat.probes.decl}); a tool via `inputs.tools` (or a `tools` native producer), whose resolved-binary content hash is both the value and its re-run trigger; a file via `inputs.files`; an upstream probe via `inputs.requires`. A module SHOULD name its probe keys `"<module-prefix>:<name>"` (e.g. `"cc:zlib"`) to avoid collisions (§{cat.probes.api}).

### 12.7.4. Reading probe values and dependency outputs [#mods.authoring.reads]

The sanctioned read of a probe value is `cook.probes.get(key)` (§{lua.cook-probes-get}); on the execute-phase VM it reads the per-run probe-value store. A `$<key>` or `$<key.field>` placeholder in a command desugars to exactly this read (§{cat.probes.templates}), so a module MAY consume a probe either textually in a command or programmatically in a `>{ … }` body.

A module publishes a target's transitive-link information with `cook.export` and reads a dependency's with `cook.import` (§{lua.cook-exports-imports}); it reads a dependency recipe's outputs with `cook.dep_output` / `cook.dep_output_list` (§{lua.cook-dep-output}) or the equivalent `$<NAME>` surface. All are both-phase (§{lua.both}). The register-phase key/value methods `cook.probes.set` and `cook.probes.scope` are deprecated (§{lua.cook-probes-set}); a new module SHOULD use `cook.probe` for memoised values rather than a hand-managed store.

### 12.7.5. Seal policy [#mods.authoring.seal]

> **Normative.** A module MUST seal a unit only on a **deterministic determinant** — one whose value is a pure function of declared inputs, such as a lockfile's content hash or a resolved toolchain's identity/content hash. A module MUST NOT seal on a value a toolchain can produce nondeterministically — a build timestamp, an embedded absolute temporary path, a randomised build id — because sealing folds that value into the shared cache key (§{exec.cache.single-key}) and a nondeterministic value makes the key unreproducible across machines, defeating the cross-machine reuse that §{exec.cache} exists to provide.

Every determinant a module seals MUST be expressed as a **named probe**: the seal surface admits no raw environment or tool reference (§{steps.cook-disposition}). An unconsumed environment variable is folded by declaring an `envs` probe over it and sealing that probe by name; a toolchain is folded by sealing a `tools` probe by name (§{steps.cook-disposition}, §{cat.probes.decl}). A module MUST NOT attempt to fold a machine-inferred determinant — target triple, libc version, locale — that the author did not declare; the engine infers none of its own (§{exec.cache.single-key}), and a module that reaches around the declared-determinant model reintroduces exactly the unsound-cache failure mode the single-key design removed.

This subsection constrains a module's *choice* of what to seal; the seal surface, its grammar, and its cache-key effect are defined normatively at §{steps.cook-disposition} and §{exec.cache.single-key} and are not restated here.

### 12.7.6. Probe-key naming [#mods.authoring.probe-keys]

A probe-value placeholder splits on `.` to separate the probe key from a field selector (`$<key.field>`, §{cat.probes.templates}). A module that derives a probe key from arbitrary text — a package name, a target label — MUST sanitise that text to the `PROBE_SEG` character set (§{cat.probes.decl}) so a stray `.` cannot be misread as a field selector; this is the sanctioned pattern behind the `cc` module's key derivation. A bare probe key is at most two colon-separated segments (`PROBE_SEG:PROBE_SEG`); the quoted-string key form (§{cat.probes.decl}) is the escape hatch for a spelling outside `PROBE_SEG`.

### 12.7.7. Publishing a blessed module [#mods.authoring.publishing]

A module distributed through the official registry (§{mods.catalogue-index}, §{cat.bootstrap}) is published as a LuaRocks rockspec served from `rocks.usecook.com`. A blessed module:

- MUST version its rockspec revisions monotonically in the LuaRocks `X.Y.Z-R` scheme, bumping the revision `R` for a repackage and the version `X.Y.Z` for a surface change;
- MUST reflect any change to its public surface in the module's Part IV catalogue section (§{cat}) under a Standard change entry (App. E) before or alongside the implementation, per the spec-first rule of §{intro.architecture-relationship};
- SHOULD pin its source with a `git+https` URL and omit `source.dir` so the rendered index resolves the module verbatim.

The distribution mechanics — the `cook_modules/` search paths a published module lands in — are §{mods.use-scope}; this subsection governs the publishing contract, not the Lua API.

### 12.7.8. Minting recipes [#mods.authoring.minting]

A recipe's name, and the fact that it is invocable, belong to the author of the Cookfile. A module is ordinarily a **step contributor**: it exposes target makers that a user-written `recipe` body calls to register work units (§{mods.authoring.units}), and the recipe those units belong to is one the user declared. A module SHOULD NOT call `cook.recipe` (§{lua.cook-recipe}) to mint a recipe the user did not ask for.

This is a categorical MUST, not merely a SHOULD-NOT, for the ordinary case: a **single-target maker** — a target maker whose call contributes the units of exactly one target — MUST be a step contributor. It MUST be invoked from within a user-written `recipe` body, from which it derives that recipe's identity via `cook.recipe_name()` (§{lua.cook-recipe-name}), and it MUST NOT itself call `cook.recipe`. Only the two shapes below, neither of which is a single-target maker, are exempted from this MUST.

Exports are orthogonal to this rule. Nothing in this subsection touches `cook.export` / `cook.import` semantics (§{lua.cook-exports-imports}): a target's exported transitive-link surface is a property of the target's own options, independent of which recipe owns the unit that produced it. A step contributor exports under the enclosing recipe's identity exactly as a minted recipe would — who owns the recipe and what the recipe exports are separate questions, and this subsection answers only the first.

Two shapes legitimately require a module to mint recipes:

- **Data-driven fan-out** — one call mints N recipes derived from data the module parses at register time, where neither N nor the names are knowable to the Cookfile author in advance (for example, one build recipe per member of a workspace manifest).
- **Support recipes** — a top-level configuration call whose work has to live in some recipe, where no user-declared recipe is its natural home (for example, rendering a generated configuration header).

A module that mints a recipe under either shape:

- MUST pass an `origin` in the `cook.recipe` metadata table (§{lua.cook-recipe}), so that a listing surface can attribute the minted name to the module that produced it (§{lua.reg.discovery}). The conventional value is the module-qualified name of the minting function — for example, `"cook_pnpm.workspace"` for the fan-out shape, `"cook_cc.config_header"` for the support-recipe shape;
- MUST document that the call mints recipes, and how the minted names are derived, in the module's Part IV catalogue section (§{cat}) — or, for a module outside the catalogue, in that module's own documentation.

The annotation is attribution, not a restriction. A minted recipe is an ordinary member of the discovered set and remains dispatchable by name like any other (§{lua.reg.discovery}); a module author who regards a support recipe as an internal detail conveys that through the annotation and the documentation this subsection requires, not by expecting an implementation to refuse the invocation. See §{rationale.recipe-origin-annotation}.
