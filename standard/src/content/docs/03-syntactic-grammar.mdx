# 3. Syntactic grammar [#grammar]
> **Normative.** This chapter walks through the productions of the Cookfile grammar in prose. The full formal grammar is in Appendix A, which is authoritative where it disagrees with the prose here (§{notation.grammar-precedence}).

## 3.1. Grammar overview [#grammar.overview]
A Cookfile is parsed as a single top-level production, `cookfile`, consisting of zero or more `toplevel_item`s (App. A.1). Each `toplevel_item` is a `recipe`, a `chore`, a `config_block`, a `use_declaration`, an `import_declaration`, a `register_block`, a top-level `module_call`, a `comment`, or a `NEWLINE` (blank line).

A conforming implementation MUST drive parsing off the token stream produced by §{lexical}: each syntactic form listed above is selected by the classification of its leading token. Comments and blank lines MUST be accepted at any top-level position and MUST NOT affect the grammatical shape of the surrounding items. A `Content` token at the top level (§{lexical.line-prefixes}) is not a valid `toplevel_item` and MUST be rejected unless it matches the top-level `module_call` shape per §{grammar.top-level-module-call}.

A top-level `module_call` (§{grammar.top-level-module-call}) is a column-0 `Content` line whose first token matches the module-call shape per §{grammar-appendix.steps} `module_call`. The classification rule mirrors the recipe-body cascade as it stood BEFORE CS-0072 — at top level, the rule now fires; in recipe bodies, the rule is removed per §{grammar.step-dispatch} amendment.

### Note 3.1.1

The grammar is top-down and keyword-led at the outer layer: the leading token of a non-blank `toplevel_item` is sufficient to commit to one of `recipe`, `chore`, `config_block`, `use_declaration`, `import_declaration`, or `comment`, with no backtracking. The rejection of a top-level `Content` token (§{lexical.line-prefixes}) is normative in the body of §{grammar.overview}.

### Note 3.1.2

Inside a recipe body, a different set of productions applies (App. A.3, App. A.4); see §{grammar.step-dispatch}.

## 3.2. Top-level ordering [#grammar.top-level-ordering]
A `use_declaration`, `import_declaration`, or `config_block` MUST NOT follow the first `recipe` or `chore` in a Cookfile. A conforming implementation MUST reject a Cookfile in which any such item appears after any `recipe` or `chore`. The diagnostic MUST identify the offending item's source line.

Comments and blank lines are not ordered: they MAY appear at any top-level position, both before and after recipes, without affecting the classification of any surrounding item.

Within each ordered prefix, `use_declaration`s, `import_declaration`s, and `config_block`s MAY appear in any order and MAY be interleaved freely.

A `register_block` (§{grammar.register-blocks}) and a top-level `module_call` (§{grammar.top-level-module-call}) MAY appear at any top-level position, including before any `recipe`/`chore` and interleaved between them. Neither is subject to the use/import/config prefix rule of §3.2.

### Example 3.2.1

```cook
use cpp
import backend ./services/backend

config
    env.CC = "gcc"

config release
    env.CXXFLAGS = "-O3"

recipe build
    gcc -o main main.c
```

Each of the three ordered forms (`use_declaration`, `import_declaration`, `config_block`) appears before the single `recipe`. The `use`, `import`, and `config` items are in the prefix; any of them placed after `recipe build` would be rejected.

### Note 3.2.1

The `cook-lang` parser test `test_parse_use_after_recipe_fails` (and companion tests for `import` and `config`) pin this ordering rule.

## 3.4. `use` declarations [#grammar.use-declarations]
A `use_declaration` is the top-level form `"use" use_name NEWLINE` (App. A.2), where `use_name` is a `LUA_IDENT` (App. A.5) optionally written either as a bare identifier or as a double-quoted `STRING` whose contents lex as a `LUA_IDENT`. It requests that the named built-in or user-authored module be made available to the Cookfile's recipes and config blocks. Module resolution is governed by §{lua.use-env}.

The keyword `use` MUST be followed by whitespace, a tab, or a `"` character before the name; otherwise the line is not a `use_declaration`. A bare word such as `useful` MUST be classified as a `Content` token rather than a `use_declaration`.

A conforming implementation MUST accept both the quoted form (`use "cpp"`) and the bare form (`use cpp`) and treat them as equivalent.

A conforming implementation MUST reject a `use_declaration` whose `use_name` does not match the `LUA_IDENT` production (App. A.5) — that is, any name containing a character outside `[A-Za-z0-9_]`, any name not starting with `[A-Za-z_]`, or the empty string. In particular, names containing whitespace, hyphens, or dots are rejected. The diagnostic MUST identify the offending name and source line. The restriction reflects the load-time binding (§{lua.use-env}): the `use_name` is bound as a Lua local under the same spelling, and a non-identifier name would yield malformed Lua at codegen time. Hyphens are rejected to avoid silent collisions when codegen rewriters fold `-` to `_` (e.g. `foo-bar` and `foo_bar` would otherwise resolve to the same Lua local).

### Example 3.4.1

```cook
use cpp
use "proto"
```

Both forms are accepted; the module names are `cpp` and `proto` respectively.

### Example 3.4.2

```cook
use foo-bar
```

Rejected: `foo-bar` is not a `LUA_IDENT` (hyphen is not permitted). The same rule rejects `use cpp.bad`, `use "has spaces"`, and `use "9lives"`.

### Note 3.4.1

The `cook-lang` parser tests `test_use_prefix_is_content` and `test_use_bare_name` pin the distinction between `use` as a keyword and `use…` as the beginning of a `Content` token. The tests `test_use_name_with_dash_rejected`, `test_use_name_with_dots_rejected`, `test_use_name_with_space_rejected`, and `test_use_name_starting_with_digit_rejected` pin the `LUA_IDENT` constraint introduced by CS-0035.

## 3.5. `import` declarations [#grammar.import-declarations]
An `import_declaration` is the top-level form `"import" BARE_IDENTIFIER path NEWLINE` (App. A.2), where `path` is a non-empty run of non-whitespace characters (App. A.5). The `BARE_IDENTIFIER` is the local alias under which the imported Cookfile's recipes become addressable; the `path` is the filesystem location of the imported Cookfile. Import resolution is governed by §{modules}.

Unlike `use_declaration`, the alias in `import_declaration` MUST be a `BARE_IDENTIFIER`; the quoted form is not accepted for the alias.

The `BARE_IDENTIFIER` of each `import_declaration` MUST be unique within a single Cookfile. A conforming implementation MUST reject a Cookfile that contains two `import_declaration`s with the same alias. The diagnostic MUST identify the second occurrence's source line.

The `path` MUST be present. A line of the form `import <name>` with no following path MUST be rejected.

### Example 3.5.1

```cook
import backend ./services/backend
import proto ../../libs/proto
```

Two imports with distinct aliases `backend` and `proto`, and relative paths. Recipes from either imported Cookfile are subsequently addressable as `backend.<name>` and `proto.<name>`; see §{modules}.

### Note 3.5.1

The `cook-lang` parser test `test_parse_duplicate_import_names_fails` pins the uniqueness rule.

## 3.6. Config blocks [#grammar.config-blocks]
A `config_block` is the top-level form `"config" name? NEWLINE config_body` (App. A.2). The optional `name` is either a `BARE_IDENTIFIER` or a `STRING`. The `config_body` is a sequence of raw source lines (`LUA_SOURCE`, App. A.5) running from the line after the `config` header up to the next top-level keyword or end-of-file. The contents of `config_body` are not parsed by the Cookfile grammar; the Lua runtime interprets them at load time (§{xref}, §{lua}).

At most one unnamed `config_block` MUST appear in a Cookfile. Named `config_block` names MUST be unique within a Cookfile. A conforming implementation MUST reject a Cookfile that violates either constraint.

A `config_body` extends from the line following the header up to (but not including) the next terminating line of the surrounding Cookfile, or end-of-file (whichever comes first). See §{recipes.termination} for the full implicit-termination rule (which includes `register` and the top-level module_call shape as of CS-0072), which applies symmetrically to recipe and config bodies.

A `config_body` MAY be empty — that is, the next top-level construct or EOF MAY immediately follow the header — in which case the block contributes the empty Lua source.

Composition of multiple `config_block`s — base + at most one selected overlay — is specified in §{grammar.config-composition}.

### Example 3.6.1

```cook
config
    env.CC = "gcc"

config release
    env.CXXFLAGS = "-O3"

config dev
    env.CXXFLAGS = "-O0 -g"
```

Three config blocks: one unnamed, two named. Each body is raw Lua source that the runtime interprets; the Cookfile grammar treats it as opaque text.

### Note 3.6.1

The `config_body` is collected by source-line range rather than by re-tokenising the enclosed lines. Lines that would otherwise tokenise as Cookfile constructs (for example, a line that looks like `BARE_IDENTIFIER "value"`) are passed through verbatim to the Lua runtime.

### 3.6.1. Config-block composition [#grammar.config-composition]

A Cookfile MAY contain at most one unnamed `config_block` (the *base config*) and zero or more named `config_block`s (the *overlay configs*). At load time, a conforming implementation MUST select at most one overlay config by name. The selection mechanism is implementation-defined.

A conforming implementation MUST report a load-time error when the selected name does not match any named `config_block` declared in the Cookfile. The diagnostic MUST identify the requested name.

When a base config is present, its body MUST execute first against the Cook Lua API tables defined in §{lua} — including `cook.env` and any other mutable state those tables expose. When an overlay is selected, its body MUST execute second against the same tables. The overlay's writes therefore observe values established by the base; an overlay write to a key already set by the base replaces the base's value (last-write-wins). When no overlay is selected, only the base (if present) executes. When neither is present, no `config_block` Lua executes during the load phase.

Both bodies execute during the load phase, after `use` resolution (§{lua.use-env}) and before recipe registration (§{exec}).

#### Example 3.6.1.1

```cook
use cpp

config
    env.CC = "gcc"
    env.CXXFLAGS = "-O0 -g"

config release
    env.CXXFLAGS = "-O3"

config dev
    env.CXXFLAGS = "-O0 -g -DDEBUG"

recipe build
    cook "build/main" using { $<CC> $<CXXFLAGS> -o $<out> main.c }
```

When no overlay is selected, the base alone executes and `$<CXXFLAGS>` resolves to `-O0 -g`. When `release` is selected, the base runs first (`env.CXXFLAGS = "-O0 -g"`), then `release` runs (`env.CXXFLAGS = "-O3"`); `$<CXXFLAGS>` resolves to `-O3`. Selecting a name that does not appear as a named `config_block` in the Cookfile is a load-time error.

#### Note 3.6.1.1

Multi-overlay selection is deliberately not specified in this revision: per the rule above, a conforming implementation permits at most one overlay. The single-overlay restriction avoids ordering questions across multiple selected overlays and is non-breaking to relax in a future revision; the design rationale appears in App. B.

## 3.7. Register blocks [#grammar.register-blocks]
A `register_block` is the top-level form `"register" NEWLINE register_body`. The `register_body` is a sequence of raw source lines (`LUA_SOURCE`, App. A.5) running from the line after the `register` header up to the next terminating line of the surrounding Cookfile (§{recipes.termination}) or end-of-file. The contents of `register_body` are not parsed by the Cookfile grammar; the register-phase Lua VM interprets them at recipe-registration time (§{exec.two-phase}).

A Cookfile MAY contain zero or more `register_block`s. The block takes no name. A conforming implementation MUST reject a `register` line whose content after the keyword is non-empty (e.g. `register foo` or `register "release"`); the diagnostic identifies the offending source line.

A `register_body` MAY be empty: the next top-level construct or EOF MAY immediately follow the `register` header, in which case the block contributes the empty Lua source.

### Splicing semantics [#grammar.register-blocks.splicing]

A conforming implementation MUST execute each `register_block` body on the register-phase Lua VM in **source order**, interleaved with recipe registrations. The effect MUST be as if each `register_block` body were spliced verbatim into the compiled register-phase Lua program at its source position. `local` variables and other Lua state declared in one `register_block` body MUST remain in scope for subsequent `register_block` bodies, for `inline_lua_line` / `inline_lua_block` (§{recipes.lua-steps}) lines in recipes that follow, and for the bodies of `cook.recipe(…)` invocations triggered by either, subject to the visibility rules of the underlying Lua language.

A `register_block` body MAY call any register-phase Cook Lua API (§{lua.register-only}), including `cook.recipe`, `cook.add_unit`, `cook.step_group`, `cook.exec`, `cook.interactive`, and the both-phase APIs of §{lua.both-phase}.

### Example 3.7.1

```cook
use cook_cc

register
    local raylib = cook_cc.find_or_error("raylib")
    cook_cc.bin("game", {
        sources     = { "src/main.c" },
        includes    = raylib.include_dirs,
        system_libs = raylib.system_libs,
    })
```

The `register` block calls `cook_cc.bin`, a §9.2 target-maker that internally invokes `cook.recipe("game", …)` to create the recipe header and `cook.add_unit(…)` to record its work units. The block runs once during recipe registration; the resulting `recipe "game"` is indistinguishable from one declared with `recipe game / >> cook_cc.bin(…)` under the v0.3 idiom.

### Example 3.7.2

```cook
register
    zlib = cook_cc.find_or_error("zlib")
    cook_cc.lib("common", { sources = { "src/common.c" } })

recipe run
    @ ./packer test.dat

register
    cook_cc.bin("packer", { sources = { "src/packer.c" }, deps = { "common" }, libs = zlib.system_libs })
```

The two-block-with-recipe-between form demonstrates the source-order splicing rule plus shared scope: the `zlib` table written in the first block is visible in the second block. Both blocks splice into the same register-phase Lua chunk; the only restriction is Lua's normal lexical-scope rules (`local` in a `do … end` is not visible outside it, but assignments and `local` at the chunk's top level — which is where register-block bodies splice — persist through the rest of the chunk).

### Note 3.7.1

Note: the recipe-body step-dispatch cascade in §{grammar.step-dispatch} no longer includes the module_call rule as of CS-0072. A bare `<id>.<id>(…)` line inside a recipe body classifies as `shell_command` and would try to exec the literal text as a shell command. Authors who want register-phase Lua inside a recipe body MUST use `>>` (`inline_lua_line`) or `>>{` (`inline_lua_block`).

## 3.7.5. Top-level module calls [#grammar.top-level-module-call]

A **top-level module_call** is a column-0 `Content` token whose first token matches the module-call shape `BARE_IDENTIFIER "." BARE_IDENTIFIER (…)` per App. A.4 (`module_call`). The first segment (before the `.`) is restricted to ASCII letters, digits, and underscore — narrower than `BARE_IDENTIFIER` at large; this is the same restriction §3.9 step-dispatch rule 5 applied before CS-0072.

A top-level module_call MUST be a single Lua expression statement. The lexer recognises its first source line; a multi-line call is collected by the brace-balance algorithm of §{lexical.brace-blocks}, treating the call as a Lua source span and using the Lua-spans rule of §{lexical.brace-blocks.lua-spans}. The collected source body is the call as written, preserving interior whitespace and comments.

A top-level module_call is register-phase. A conforming implementation MUST splice the collected call source verbatim into the compiled register-phase Lua program at the source position, identical to a `register_block` body containing only that call. The shared-chunk-scope rule of §{grammar.register-blocks.splicing} applies.

The `:` call shape (`alias:method(…)`) is NOT a top-level module_call. A line whose first token matches `<id>:<id>(` is a `Content` token and is rejected at the syntactic layer (§{grammar.overview}) because no top-level construct admits it. Authors using colon-call syntax MUST wrap such calls in a `register` block.

### Example 3.7.5.1

```cook
use cook_cc

config
    raylib = cook_cc.find_or_error("raylib")

cook_cc.bin("game", {
    sources     = { "src/main.c" },
    includes    = raylib.include_dirs,
    system_libs = raylib.system_libs,
})
```

Line 6's `cook_cc.bin("game", { …` is a top-level module_call. The opening `{` on line 6 is not balanced by the closer until line 10's `})`, so the lexer collects all four lines as a single multi-line module_call source body. At register time the body splices into the register-phase Lua chunk; `cook_cc.bin` (defined by `cook_cc 0.4.0`) calls `cook.recipe("game", …)` internally to register the recipe.

### Note 3.7.5.1

The recipe-body step-dispatch cascade in §{grammar.step-dispatch} no longer includes the module_call rule as of CS-0072.

## 3.8. Recipes [#grammar.recipe-syntax]
A `recipe` is the form `recipe_header NEWLINE recipe_body` (App. A.3). The `recipe_header` is the form `"recipe" name (":" dependency_list)?`, where `name` is a `BARE_IDENTIFIER` or a `STRING`.

The `dependency_list` is one or more `name`s separated by whitespace; each element is a `BARE_IDENTIFIER` or a `STRING`, and the two forms MAY be freely mixed within a single list.

A `recipe_header` MUST NOT be introduced by a line whose trimmed content begins with `recipe` but is immediately followed by a non-separator character — for example, `recipes_cleanup`. Such a line is a `Content` token. The keyword `recipe` is recognised only when separated from its name by whitespace, a tab, or a `"` character.

A `recipe_body` extends from the line following the header up to (but not including) the next terminating line of the surrounding Cookfile, or end-of-file (whichever comes first). See §{recipes.termination} for the full terminator-set, which includes `register` and the top-level module_call shape as of CS-0072.

The final segment of a recipe's name (the text following the last `.`, or the entire name if it contains no `.`) MUST NOT be one of the identifiers `stem`, `name`, `ext`, `dir`, `in`, `out`, `all`. A conforming implementation MUST reject a recipe whose final name segment matches any of these reserved identifiers (App. A.2).

The `chore` top-level form (§{chores}) is a peer of `recipe`. A `chore` body has a more restrictive step-kind alternation than a `recipe` body — see §{chores.body} for the differences.

### Example 3.8.1

```cook
recipe build
    gcc -o main main.c

recipe "bundle": backend.build frontend.build
    echo bundling
```

Two recipes, in two valid header shapes: explicit with a bare name (`build`), and explicit with a quoted name and dependency list mixing bare and quoted forms.

## 3.9. Step dispatch inside a recipe [#grammar.step-dispatch]
Inside a `recipe_body`, each source line is first classified by the lexer (§{lexical.line-classification}) into one of `Comment`, `Blank`, `LuaLine`, `LuaBlockOpen`, `InlineLuaLine`, `InlineLuaBlockOpen`, or `Content`. Comments and blank lines are skipped; `LuaLine` becomes a `lua_line` step; `LuaBlockOpen` begins a `lua_block` step collected by the brace-balance algorithm (§{lexical.brace-blocks}); `InlineLuaLine` becomes an `inline_lua_line` step; `InlineLuaBlockOpen` begins an `inline_lua_block` step (also brace-balanced).

Note: prior to CS-0072 (Cook Standard v0.9), this cascade included a fifth rule classifying `BARE_IDENTIFIER "." BARE_IDENTIFIER(…)` as `module_call` (register-phase Lua). CS-0072 removes that rule from the recipe-body cascade and adds an equivalent top-level dispatch in §{grammar.top-level-module-call}. Authors who want register-phase Lua inside a recipe body MUST use the `>>` (`inline_lua_line`) or `>>{` (`inline_lua_block`) prefix.

A `Content` token inside a `recipe_body` is dispatched to exactly one `declarative_item` or `imperative_item` (App. A.3) by the priority cascade below (App. A.4, "Step-dispatch priority"). The first match wins; a conforming implementation MUST test the alternatives in this order:

1. Prefix `ingredients` + separator (space, tab, or end of token) → `ingredients_step`.
2. Prefix `cook` + separator → `cook_step`.
3. Prefix `plate` + separator → `plate_step`.
4. Prefix `test` + separator → `test_step`.
5. First character is `@` and the remainder of the line is non-empty → `interactive_command`.
6. Otherwise → `shell_command`.

In step 5, an `@` with no following content MUST be rejected rather than silently dispatched as any of the preceding alternatives. The diagnostic MUST identify the offending line.

A `Content` token inside a `recipe_body` whose trimmed text begins with the keyword `register` followed by a separator (space, tab, or end-of-line) MUST be rejected with the diagnostic "`register` blocks are top-level only; move this outside the recipe body". The rejection applies regardless of indentation depth: an indented `register` cannot be a `RegisterHeader` (which is column-0 only by §{lexical.line-classification}) and is also not a permitted shell command in this position. A `Content` token whose first token is the bare identifier `register` *without* a separator (e.g. `register_foo`) remains a `shell_command` per the (post-CS-0072) rule 6.

A `recipe_body` MUST contain at most one `ingredients_step`. A conforming implementation MUST reject a recipe with two or more `ingredients_step` items (App. A.3).

A `recipe_body` MUST satisfy the **region ordering rule** (App. A.3, "Region ordering rule"; §{recipes.step-kinds}, Note 4.4.2): every `declarative_item` (other than `comment` or `NEWLINE`) MUST appear before the first `imperative_item` of the recipe body. A conforming implementation MUST reject a recipe in which a declarative-region step (cook, plate, test, ingredients, `>>`, or `>>{`) appears after the first imperative-region step (`>`, `>{`, shell, or `@`). The diagnostic MUST identify the offending step's line and the line on which the imperative region began.

### Example 3.9.1

```cook
recipe build
    ingredients "src/*.c"
    cook "bin/$<in.stem>" using { cc $<in> -o $<out> }
    test { ./bin/app }
    @vim notes.txt
    echo done
```

The five body lines dispatch, in order, to: `ingredients_step`, `cook_step`, `test_step`, `interactive_command` (rule 5), and `shell_command` (rule 6). A bare `cpp.link()` line in a recipe body — which before CS-0072 dispatched to `module_call` — now falls through to rule 6 and becomes a `shell_command`.

### Example 3.9.2

```cook
recipe build
    CC "gcc"
```

The body line `CC "gcc"` is a `Content` token; the step-dispatch cascade matches no preceding alternative and dispatches it as a `shell_command` whose text is the trimmed source line. The emitted command preserves the original surface form — the identifier and its quoted argument.

### Note 3.9.1

The `cook-lang` parser test `test_indented_quoted_pair_is_shell_command` pins the recipe-body treatment of `NAME "value"`. The test `test_empty_interactive_step_errors` pins rejection of a bare `@`.
