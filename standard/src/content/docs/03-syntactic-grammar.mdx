# 3. Syntactic grammar [#grammar]
> **Normative.** This chapter walks through the productions of the Cookfile grammar in prose. The full formal grammar is in Appendix A, which is authoritative where it disagrees with the prose here (§{notation.grammar-precedence}).

## 3.1. Grammar overview [#grammar.overview]
A Cookfile is parsed as a single top-level production, `cookfile`, consisting of zero or more `toplevel_item`s (App. A.1). Each `toplevel_item` is a `recipe`, a `config_block`, a `use_declaration`, an `import_declaration`, a `comment`, or a `NEWLINE` (blank line).

A conforming implementation MUST drive parsing off the token stream produced by §{lexical}: each syntactic form listed above is selected by the classification of its leading token. Comments and blank lines MUST be accepted at any top-level position and MUST NOT affect the grammatical shape of the surrounding items. A `Content` token at the top level (§{lexical.line-prefixes}) is not a valid `toplevel_item` and MUST be rejected.

### Note 3.1.1

The grammar is top-down and keyword-led at the outer layer: the leading token of a non-blank `toplevel_item` is sufficient to commit to one of `recipe`, `config_block`, `use_declaration`, `import_declaration`, or `comment`, with no backtracking. The rejection of a top-level `Content` token (§{lexical.line-prefixes}) is normative in the body of §{grammar.overview}.

### Note 3.1.2

Inside a recipe body, a different set of productions applies (App. A.3, App. A.4); see §{grammar.step-dispatch}.

## 3.2. Top-level ordering [#grammar.top-level-ordering]
A `use_declaration`, `import_declaration`, or `config_block` MUST NOT follow the first `recipe` in a Cookfile. A conforming implementation MUST reject a Cookfile in which any such item appears after any `recipe`. The diagnostic MUST identify the offending item's source line.

Comments and blank lines are not ordered: they MAY appear at any top-level position, both before and after recipes, without affecting the classification of any surrounding item.

Within each ordered prefix, `use_declaration`s, `import_declaration`s, and `config_block`s MAY appear in any order and MAY be interleaved freely.

### Example 3.2.1

```cook
use cpp
import backend ./services/backend

config
    env.CC = "gcc"
end

config release
    env.CXXFLAGS = "-O3"
end

recipe build
    gcc -o main main.c
end
```

Each of the three ordered forms (`use_declaration`, `import_declaration`, `config_block`) appears before the single `recipe`. The `use`, `import`, and `config` items are in the prefix; any of them placed after `recipe build` would be rejected.

### Note 3.2.1

The `cook-lang` parser test `test_parse_use_after_recipe_fails` (and companion tests for `import` and `config`) pin this ordering rule.

## 3.4. `use` declarations [#grammar.use-declarations]
A `use_declaration` is the top-level form `"use" name NEWLINE` (App. A.2), where `name` is either a `BARE_IDENTIFIER` or a `STRING` (App. A.5). It requests that the named built-in or user-authored module be made available to the Cookfile's recipes and config blocks. Module resolution is governed by §{lua.use-env}.

The keyword `use` MUST be followed by whitespace, a tab, or a `"` character before the name; otherwise the line is not a `use_declaration`. A bare word such as `useful` MUST be classified as a `Content` token rather than a `use_declaration`.

A conforming implementation MUST accept both the quoted form (`use "cpp"`) and the bare form (`use cpp`) and treat them as equivalent.

### Example 3.4.1

```cook
use cpp
use "proto"
```

Both forms are accepted; the module names are `cpp` and `proto` respectively.

### Note 3.4.1

The `cook-lang` parser tests `test_use_prefix_is_content` and `test_use_bare_name` pin the distinction between `use` as a keyword and `use…` as the beginning of a `Content` token.

## 3.5. `import` declarations [#grammar.import-declarations]
An `import_declaration` is the top-level form `"import" BARE_IDENTIFIER path NEWLINE` (App. A.2), where `path` is a non-empty run of non-whitespace characters (App. A.5). The `BARE_IDENTIFIER` is the local alias under which the imported Cookfile's recipes become addressable; the `path` is the filesystem location of the imported Cookfile. Import resolution is governed by §{modules}.

Unlike `use_declaration`, the alias in `import_declaration` MUST be a `BARE_IDENTIFIER`; the quoted form is not accepted for the alias.

The `BARE_IDENTIFIER` of each `import_declaration` MUST be unique within a single Cookfile. A conforming implementation MUST reject a Cookfile that contains two `import_declaration`s with the same alias. The diagnostic MUST identify the second occurrence's source line.

The `path` MUST be present. A line of the form `import <name>` with no following path MUST be rejected.

### Example 3.5.1

```cook
import backend ./services/backend
import proto ../../libs/proto
```

Two imports with distinct aliases `backend` and `proto`, and relative paths. Recipes from either imported Cookfile are subsequently addressable as `backend.<name>` and `proto.<name>`; see §{modules}.

### Note 3.5.1

The `cook-lang` parser test `test_parse_duplicate_import_names_fails` pins the uniqueness rule.

## 3.6. Config blocks [#grammar.config-blocks]
A `config_block` is the top-level form `"config" name? NEWLINE config_body "end" NEWLINE` (App. A.2). The optional `name` is either a `BARE_IDENTIFIER` or a `STRING`. The `config_body` is a sequence of raw source lines (`LUA_SOURCE`, App. A.5) running from the line after the `config` header up to, but not including, the closing `end`. The contents of `config_body` are not parsed by the Cookfile grammar; the Lua runtime interprets them at load time (§{xref}, §{lua}).

At most one unnamed `config_block` MUST appear in a Cookfile. Named `config_block` names MUST be unique within a Cookfile. A conforming implementation MUST reject a Cookfile that violates either constraint.

Every `config_block` MUST be terminated by a line whose sole non-whitespace content is `end`. A `config_block` with no closing `end` MUST be rejected, and the diagnostic MUST identify the header's source line.

A `config_body` MAY be empty — that is, the `end` line MAY immediately follow the header line — in which case the block contributes the empty Lua source.

Composition of multiple `config_block`s — base + at most one selected overlay — is specified in §{grammar.config-composition}.

### Example 3.6.1

```cook
config
    env.CC = "gcc"
end

config release
    env.CXXFLAGS = "-O3"
end

config dev
    env.CXXFLAGS = "-O0 -g"
end
```

Three config blocks: one unnamed, two named. Each body is raw Lua source that the runtime interprets; the Cookfile grammar treats it as opaque text.

### Note 3.6.1

The `config_body` is collected by source-line range rather than by re-tokenising the enclosed lines. Lines that would otherwise tokenise as Cookfile constructs (for example, a line that looks like `BARE_IDENTIFIER "value"`) are passed through verbatim to the Lua runtime.

### 3.6.1. Config-block composition [#grammar.config-composition]

A Cookfile MAY contain at most one unnamed `config_block` (the *base config*) and zero or more named `config_block`s (the *overlay configs*). At load time, a conforming implementation MUST select at most one overlay config by name. The selection mechanism is implementation-defined.

A conforming implementation MUST report a load-time error when the selected name does not match any named `config_block` declared in the Cookfile. The diagnostic MUST identify the requested name.

When a base config is present, its body MUST execute first against the Cook Lua API tables defined in §{lua} — including `cook.env` and any other mutable state those tables expose. When an overlay is selected, its body MUST execute second against the same tables. The overlay's writes therefore observe values established by the base; an overlay write to a key already set by the base replaces the base's value (last-write-wins). When no overlay is selected, only the base (if present) executes. When neither is present, no `config_block` Lua executes during the load phase.

Both bodies execute during the load phase, after `use` resolution (§{lua.use-env}) and before recipe registration (§{exec}).

#### Example 3.6.1.1

```cook
use cpp

config
    env.CC = "gcc"
    env.CXXFLAGS = "-O0 -g"
end

config release
    env.CXXFLAGS = "-O3"
end

config dev
    env.CXXFLAGS = "-O0 -g -DDEBUG"
end

recipe build
    cook "build/main" using "{CC} {CXXFLAGS} -o {out} main.c"
end
```

When no overlay is selected, the base alone executes and `{CXXFLAGS}` resolves to `-O0 -g`. When `release` is selected, the base runs first (`env.CXXFLAGS = "-O0 -g"`), then `release` runs (`env.CXXFLAGS = "-O3"`); `{CXXFLAGS}` resolves to `-O3`. Selecting a name that does not appear as a named `config_block` in the Cookfile is a load-time error.

#### Note 3.6.1.1

Multi-overlay selection is deliberately not specified in this revision: per the rule above, a conforming implementation permits at most one overlay. The single-overlay restriction avoids ordering questions across multiple selected overlays and is non-breaking to relax in a future revision; the design rationale appears in App. B.

## 3.7. Recipes [#grammar.recipe-syntax]
A `recipe` is the form `recipe_header NEWLINE recipe_body "end" NEWLINE?` (App. A.3). A `recipe_header` is one of two alternatives:

- An **explicit header** of the form `"recipe" name (":" dependency_list)?`, where `name` is a `BARE_IDENTIFIER` or `STRING`.
- An **implicit header** of the form `BARE_IDENTIFIER ":" dependency_list?`.

The `dependency_list` is one or more `name`s separated by whitespace; each element is a `BARE_IDENTIFIER` or a `STRING`, and the two forms MAY be freely mixed within a single list.

The `implicit_header` form MUST be recognised only when the header line begins at column 0 — that is, when it has no leading whitespace. An indented line of the shape `BARE_IDENTIFIER:` MUST be classified as a `Content` token (and therefore, within a recipe body, dispatched as a `shell_command` per §{grammar.step-dispatch}); outside a recipe body, such an indented line is rejected under §{grammar.overview}.

A `recipe_header` MUST NOT be introduced by a line whose trimmed content begins with `recipe` but is immediately followed by a non-separator character — for example, `recipes_cleanup`. Such a line is a `Content` token. The keyword `recipe` is recognised only when separated from its name by whitespace, a tab, or a `"` character.

Every `recipe` MUST be terminated by a line whose sole non-whitespace content is `end`. A `recipe` with no closing `end` MUST be rejected, and the diagnostic MUST identify the header's source line.

The final segment of a recipe's name (the text following the last `.`, or the entire name if it contains no `.`) MUST NOT be one of the identifiers `stem`, `name`, `ext`, `dir`, `in`, `out`, `all`. A conforming implementation MUST reject a recipe whose final name segment matches any of these reserved identifiers (App. A.2).

### Example 3.7.1

```cook
recipe build
    gcc -o main main.c
end

recipe "bundle": backend.build frontend.build
    echo bundling
end

clean:
    rm -rf build
end
```

Three recipes, in three valid header shapes: explicit with a bare name (`build`), explicit with a quoted name and dependency list mixing bare and quoted forms, and implicit (`clean:`) at column 0.

### Example 3.7.2

```cook
recipe build
    build: lib setup
end
```

The second line is indented. Its shape `BARE_IDENTIFIER ":" ...` does not introduce a nested recipe; it is a `Content` token and is dispatched inside the enclosing recipe body as a `shell_command` (§{grammar.step-dispatch}).

### Note 3.7.1

The `cook-lang` parser tests `test_implicit_recipe_indented_not_detected` and `test_recipe_prefix_is_shell_command` pin the column-0 rule and the keyword-separator rule respectively.

## 3.8. Step dispatch inside a recipe [#grammar.step-dispatch]
Inside a `recipe_body`, each source line is first classified by the lexer (§{lexical.line-classification}) into one of `Comment`, `Blank`, `LuaLine`, `LuaBlockOpen`, or `Content`. Comments and blank lines are skipped; `LuaLine` becomes a `lua_line` step; `LuaBlockOpen` begins a `lua_block` step collected by the brace-balance algorithm (§{lexical.brace-blocks}).

A `Content` token inside a `recipe_body` is dispatched to exactly one `recipe_item` by the priority cascade below (App. A.4, "Step-dispatch priority"). The first match wins; a conforming implementation MUST test the alternatives in this order:

1. Prefix `ingredients` + separator (space, tab, or end of token) → `ingredients_step`.
2. Prefix `cook` + separator → `cook_step`.
3. Prefix `plate` + separator → `plate_step`.
4. Prefix `test` + separator → `test_step`.
5. First token matches the module-call shape `BARE_IDENTIFIER "." BARE_IDENTIFIER` per App. A.4 → `module_call`.
6. First character is `@` and the remainder of the line is non-empty → `interactive_command`.
7. Otherwise → `shell_command`.

In step 5, the first segment (before the `.`) is restricted to ASCII letters, digits, and underscore; hyphens and dots are not permitted in the first segment. This is narrower than `BARE_IDENTIFIER` at large (§{lexical.identifiers}), and the narrower form is what a conforming implementation MUST test against.

In step 6, an `@` with no following content MUST be rejected rather than silently dispatched as any of the preceding alternatives. The diagnostic MUST identify the offending line.

A `recipe_body` MUST contain at most one `ingredients_step`. A conforming implementation MUST reject a recipe with two or more `ingredients_step` items (App. A.3).

### Example 3.8.1

```cook
recipe build
    ingredients "src/*.c"
    cook "bin/app" using "cc {in} -o {out}"
    test "./bin/app"
    cpp.link()
    @vim notes.txt
    echo done
end
```

The six body lines dispatch, in order, to: `ingredients_step`, `cook_step`, `test_step`, `module_call` (rule 5), `interactive_command` (rule 6), and `shell_command` (rule 7).

### Example 3.8.2

```cook
recipe build
    CC "gcc"
end
```

The body line `CC "gcc"` is a `Content` token; the step-dispatch cascade matches no preceding alternative and dispatches it as a `shell_command` whose text is the trimmed source line. The emitted command preserves the original surface form — the identifier and its quoted argument.

### Note 3.8.1

The `cook-lang` parser test `test_indented_quoted_pair_is_shell_command` pins the recipe-body treatment of `NAME "value"`. The test `test_empty_interactive_step_errors` pins rejection of a bare `@`.
