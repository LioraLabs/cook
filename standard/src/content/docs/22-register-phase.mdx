# 22. Register-phase API [#lua.reg]
> **Normative.** This chapter defines every Cook Lua API surface that is register-phase only. A call to any function in this chapter from execute-phase Lua MUST raise a Lua runtime error per §{rationale.execute-phase-api}.

## 22.1. `cook.add_unit` [#lua.add-unit]
**Signature.** `cook.add_unit(spec: table) -> nil`
**Phase.** Register-phase only.

`cook.add_unit` records a single work unit to be executed later. The table argument `spec` accepts the following fields:

| Field               | Type           | Default | Meaning |
|---------------------|----------------|---------|---------|
| `command`           | string         | `""`    | Shell command string, run via `/bin/sh -c` in the execute phase. |
| `lua_code`          | string         | absent  | Lua source executed in the execute phase. If present, `command` is ignored. |
| `interactive`       | boolean        | `false` | If `true`, the unit runs on the main thread and drains concurrent work (§{exec.interactive-drain}). |
| `line`              | integer        | `0`     | Source line number used in diagnostics. |
| `cache`             | boolean        | `true`  | If `false`, the unit is never cached and is re-run on every invocation. |
| `inputs`            | array of string | `{}`   | Files whose content contributes to the unit's cache key. |
| `output`            | string         | absent  | Single declared output path (§{steps.cook-single}). |
| `outputs`           | array of string | absent | Multiple declared output paths or glob patterns (§{steps.cook-multi}, §{lua.add-unit-output-globs}). Glob patterns are resolved against the unit's working directory after the command completes (§{exec.cache.output-globs}). |
| `ingredient_groups` | array of arrays of string | `{}` | For Lua-code units, the sequence of ingredient groups exposed as `input_N` (§{lua.exe}). |
| `discovered_inputs` | table         | absent  | If present, declares a file the command writes during the execute phase that records additional input paths the command consumed (§{exec.cache.discovered-inputs}). Subfields: `from` (string, required, working-directory-relative path) and `format` (string, required; the only mandatory format is `"make"`). |
| `sharing`           | string (`"local"`, `"pinned"`, `"shared"`) | `"shared"` | Sharing disposition. `"local"` opts the unit out of the shared store (served from and written to the local cache only); `"pinned"` is designated-producer / fetch-only (MUST be served from cache, never reproduced by the consumer; a cold miss in both stores is a hard error); `"shared"` (the default, also the effect of an absent field) participates fully. Surface: the trailing `local`/`pinned` `cook_mods` (§{steps.cook-disposition}); effect: §{exec.cache.sharing}. A value outside the three is a register-phase error. |
| `record`            | boolean        | `false` | Marks the unit's output intrinsically non-reproducible; the cache key is unchanged but byte-equivalence is waived at the cache decision (§{exec.cache.record}). Surface: the trailing `nondet` `cook_mod` (§{steps.cook-disposition}); the keyword was renamed from `record` to `nondet`, but the `cook.add_unit` field name remains `record`. |
| `seal`              | array of string | `{}`   | The unit's per-unit **effective seal set** as bare probe keys (§{steps.cook-disposition}). Each named probe's canonical value folds into this unit's cache key at execute phase (§{exec.cache.single-key}); the key set is register-time data and creates a DAG edge from each sealed probe to this unit. |
| `probes`            | array of string | `{}`   | Probe keys this unit consumes as data (§{cat.probes.consumer}). Each resolved probe's fingerprint folds into the unit's fingerprint and adds a DAG edge; an unknown key is a post-pass error. `$<key>` / `$<key.field>` command placeholders desugar to a `probes` entry plus a `cook.probes.get` read (§{cat.probes.templates}). |
| `consulted_env_keys` | array of string, or the string `"*"` | `{}` | Environment-variable names whose current values fold into the unit's cache key (§{exec.cache.lua-env-reads}). `"*"` folds every key visible in `cook.env`. Codegen populates this from `$<KEY>` sigils and `cook.env.KEY` static reads; a non-string element is a hard error. |
| `file_refs`         | array of string | `{}`   | File-reference patterns (`$<file:PATH>`, §{xref.file-namespace}, CS-0101). Each pattern is resolved at register time and the matched files' content folds into the unit's cache input set (never into `inputs`, which drive execute-phase iteration). |
| `env`               | table of string→string | `{}` | Per-unit environment overrides exported to a shell unit (used by chore shell units to surface bound parameter values, §{chores.params}). Keys and values MUST both be strings. |
| `member`            | string         | absent  | A fan-out unit's data member, canonical-rendered, folded into the command hash so each member gets a distinct fingerprint (§{steps.ingredients}). Codegen-populated for `ingredients <probe>` fan-out; not hand-authored. |
| `step_kind`         | string (`"cook"`, `"chore"`) | `"cook"` | The originating step kind, which drives the execute-phase sandbox policy on the resulting unit (§{lua.fs-sandbox}). Codegen-populated; not hand-authored. `"test"` MUST be rejected with a register-phase diagnostic naming `step_kind` and directing the author to `cook.add_test` (§{lua.cook-add-test}, CS-0153): a test work unit is registrable only through `cook.add_test`, never through `cook.add_unit`. |

**Field typing (CS-0127).** Every field listed in the table above has exactly one accepted Lua type. A conforming implementation MUST reject a value of any other type with a register-phase Lua error that names the field, the received Lua type, and the accepted type. A wrong-typed field MUST NOT be coerced to its default — coercion silently discards author intent (a mistyped `cache = "false"` string, a mistakenly Lua-quoted `line`, an `inputs` list containing a non-string element) and the resulting unit runs with a default the author never asked for. Absence (a `nil`/missing field) is not a type error and MUST resolve to the field's documented default.

**Field provenance.** Several fields — `consulted_env_keys`, `file_refs`, `member`, `step_kind`, `env`, and the disposition trio `sharing` / `seal` / `record` — are normally emitted by the code generator from surface syntax (§{steps.cook-disposition}, §{steps.ingredients}, §{cat.probes.templates}) rather than written by hand; `cook.add_unit` nonetheless accepts each directly and applies the same field-typing discipline (CS-0127). The legacy field name `requires` (a pre-`probes` spelling for probe references) is no longer accepted: a conforming implementation MUST reject a non-nil `requires` with a register-phase diagnostic directing the author to `probes`.

**Output-field exclusivity.** A conforming implementation MUST reject `spec` if both `output` and `outputs` are set, raising a Lua runtime error.

**Path-shape restriction.** Every entry in `inputs`, `output`, and `outputs` is a *file* path. A conforming implementation MUST raise a Lua runtime error at register time when any path in `inputs` resolves (relative to the recipe's working directory) to an existing directory, and when any path in `output`/`outputs` resolves to an existing directory. Non-existent output paths MUST be accepted (declared outputs are typically not yet built at register time); non-existent input paths MAY be accepted at register time and surfaced later by the cache or input-iteration layer. The diagnostic MUST contain the substring `is a directory` and the offending path. Cook's cache hashes file content, so silently accepting a directory would produce an incomplete cache record and the unit would re-run on every invocation. The directory-rejection rule applies only to entries that are literal paths (no glob metacharacters: `*`, `?`, `[`). Entries containing glob metacharacters are not subject to this check at register time; the post-execute resolver MUST filter directory matches from the resolved set (§{exec.cache.output-globs}, CS-0064). An `outputs[]` entry ending in `/` is a directory output (CS-0119), not a literal path: it intentionally denotes a directory and is exempt from this directory-rejection rule (see §22.1.2 and the post-execute resolution rules in §17.6).

**Payload selection.** A conforming implementation MUST classify the recorded unit as follows:

1. If `lua_code` is present, the unit is a **Lua-code unit**; `inputs`, `outputs`, and `ingredient_groups` are carried on the payload and used by §{lua.exe} bindings at execute time.
2. Otherwise, if `interactive` is `true`, the unit is an **interactive shell unit**.
3. Otherwise, the unit is a **non-interactive shell unit**.

**Caching.** When `cache` is `true`, a conforming implementation MUST record cache metadata containing the recipe name, an input-path list, an output-path list, and a command hash. The hash inputs, cache-key derivation, and storage format are implementation-defined (§{exec.cache}).

**Return value.** `cook.add_unit` returns nothing. A conforming implementation MUST NOT cause the function to block on external work — registration is synchronous by definition (§{exec.capture}).

### Example 22.1.1

```lua
cook.add_unit({
    inputs  = { "main.c" },
    output  = "main.o",
    command = "gcc -c main.c -o main.o",
})
```

Records one shell unit with one input and one output. The same registration is produced by the surface form `cook "main.o" { gcc -c $<in> -o $<out> }` in a single-output `cook` step (§{steps.cook-single}, §{phl.cook-step}).

### Example 22.1.2

```lua
cook.add_unit({
    inputs     = { "src/wasm.rs" },
    outputs    = { "out.js", "out.wasm" },
    lua_code   = "os.execute('wasm-pack build')",
    ingredient_groups = { { "src/wasm.rs" } },
})
```

Records one Lua-code unit with multiple outputs. At execute time, the `lua_code` runs with `inputs`, `outputs`, `input`, `output`, and `input_1` bound as described in §{lua.exe}.

**Chore-body restriction (§{chores.no-caching}).** When `cook.add_unit` is called from within the register-phase body of a chore, a conforming implementation MUST raise a Lua runtime error if `cache = true` is passed (or if `cache` is absent, since the default is `true`). The codegen layer enforces this at the source level by always emitting `cache = false` for every unit in a chore body; the runtime check is a defence-in-depth guard against register-phase Lua reached from a chore context — for example a bare `module_call` (§{steps.dispatch}) or a `register` block (§{decl.register}) whose body calls `cook.add_unit` — that bypasses the codegen path.

A conforming implementation MAY use implementation-defined internal markers to track whether a chore body is executing. In the reference implementation, the generated Lua calls `cook._enter_chore()` before the body and `cook._exit_chore()` after; `cook.add_unit` checks an internal flag. These helpers are an implementation detail and MUST NOT be relied upon by Cookfile authors.

### Note 22.1.1

Adjacent `cook.add_unit` calls made inside the same register-phase `cook.step_group(fn)` invocation become siblings in the same scheduler step group (§{exec.step-groups}) and MAY run in parallel. Outside a step group they are serial. The surface `cook` step (§{steps.cook-single}, §{steps.cook-multi}) routes through `cook.step_group` implicitly.

### 22.1.1. `discovered_inputs` field [#lua.add-unit-discovered-inputs]

A unit that consumes input files which are not statically known until execute time MAY declare a `discovered_inputs` table on its `cook.add_unit` call:

```lua
cook.add_unit({
    inputs  = { "src/a.c" },
    output  = "build/a.o",
    command = "gcc -c $<in> -MMD -MF .cook/deps/a.d -o $<out>",
    discovered_inputs = {
        from   = ".cook/deps/a.d",
        format = "make",
    },
})
```

A conforming implementation MUST:

- accept `discovered_inputs` as either absent or a Lua table; reject any non-table value with a register-phase Lua error.
- require both `from` (non-empty string) and `format` (string) when the table is present; reject missing or wrong-typed subfields with a register-phase Lua error.
- require `from` to be a relative path under the unit's working directory; reject absolute paths and paths containing `..` segments.
- support `format = "make"`; MAY support additional formats; MUST raise a register-phase Lua error if `format` names a value the implementation does not recognise. The diagnostic MUST include the offending value and the list of supported values.

The semantic effect of `discovered_inputs` on the cache — including the requirement that `from` MUST NOT also appear in `outputs[]` — is specified in §{exec.cache.discovered-inputs}. This subsection defines only the Lua-level field syntax.

### 22.1.2. Output globs [#lua.add-unit-output-globs]

An entry in `outputs[]` that contains any of the glob metacharacters `*`, `?`, `[` is a **glob pattern entry**; an entry that contains none of these is a **literal path entry**. A conforming implementation MUST treat literal entries as before this section (register-time path-shape check per §22.1; post-execute recording of the literal path's content per §{exec.cache.abstract}). For glob pattern entries a conforming implementation:

- MUST NOT perform the path-shape directory-rejection check on glob entries at register time.
- MUST expand glob entries against the unit's working directory immediately after the unit's command completes successfully, before composing the unit's `StepEntry` (§{exec.cache.output-globs}).
- MUST filter out directory matches from the expanded set (CS-0064).
- An empty resolved set MUST NOT be an error; a conforming implementation MAY persist an empty resolved set as the unit's `StepEntry.outputs`.

**Trailing `**` matches files within all subdirectories.** A glob pattern entry of the form `<prefix>/**` (or the bare pattern `**`) is interpreted as if written `<prefix>/**/*` (or `**/*` respectively), so that the entry matches files at any depth under the prefix. A conforming implementation MAY apply this normalisation either as a textual rewrite before pattern resolution or by treating the trailing-`**` case directly in its matcher. The normalisation MUST NOT affect patterns whose `**` segment is not the final segment (e.g. `**/lib/*.so` is left unchanged).

The expanded set replaces the unit's declared `outputs[]` for the purposes of cache recording, artifact upload, and cache restoration. The unit's persisted `StepEntry.outputs` is always a list of literal file records (`path`, `mtime`, `hash`); patterns appear nowhere in the cache.

**Cross-recipe edges.** A recipe whose `outputs[]` declaration contains any glob pattern entry MUST NOT have any of those patterns syntactically match a literal `inputs[]` entry declared by any other recipe in the same workspace. A conforming implementation MUST perform this detection at register time by lexically testing each downstream recipe's literal `inputs[]` strings against each upstream recipe's glob pattern set; the detection MUST NOT consult the filesystem and MUST NOT depend on the unit's working directory or any post-execute state. On a positive match the implementation MUST raise a register-phase error that names both recipes, the offending input path, and the matching pattern. Downstream recipes MAY depend on a recipe with globbed outputs via `requires = { ... }`; such an edge provides DAG ordering only and MUST NOT propagate cache invalidation through file content. See §{rationale.output-globs-terminal}.

A *directory output* — an `outputs[]` entry ending in `/` — denotes the entire subtree rooted at that path and is, like a glob entry, a non-literal terminal output: for the purposes of this rule it is treated as a recursive-wildcard pattern matching every path within the subtree (the entry with a `**` wildcard appended). No other recipe's literal `inputs[]` may resolve to a path inside the subtree; downstream recipes MAY order against it via `requires = { ... }` (ordering only, no content invalidation).

### Example 22.1.3

```lua
cook.add_unit({
    inputs  = { "src/app/page.tsx", "src/lib/util.ts", "package.json" },
    outputs = { "apps/web/.next/**" },
    command = "pnpm --filter web run build",
})
```

Records one shell unit whose declared output set is resolved by globbing `apps/web/.next/**` against the unit's working directory after `pnpm --filter web run build` completes. The cache record holds a literal file list, hashed in the usual way; a subsequent run with unchanged inputs is a cache hit, and a subsequent run after `rm -rf apps/web/.next` restores the recorded files from the backend without rerunning the command.

## 22.2. `cook.exec` and `cook.interactive` [#lua.cook-exec]
**Signatures.**
  - `cook.exec(cmd: string, line: integer) -> string`
  - `cook.interactive(cmd: string, line: integer) -> string`

**Phase.** Register-phase only.

These helpers record shell units from register-phase Lua: `cook.exec` a non-interactive unit and `cook.interactive` an interactive one (the interactivity a chore-body `shell_command` carries by default, §{chores.default-interactive}). Each call records one unit with caching disabled. The return value is the empty string; it is returned only so that callers composing results do not fail on `nil` concatenation.

A conforming implementation MUST raise a Lua runtime error when `cook.exec` or `cook.interactive` is called from execute-phase Lua (a `lua_line`, a `lua_block`, or a `>{ … }` payload). The diagnostic MUST identify the function name and the calling step kind. The same rule applies to other register-only Cook Lua API helpers: `cook.add_unit`, `cook.step_group`, `cook.recipe`. See §{rationale.execute-phase-api}.

An implementation SHOULD include a brief migration hint in the diagnostic — for example, suggesting a bare `module.call()` line in the recipe body (§{steps.dispatch}) instead of `>` to record the call at register phase, or a top-level `register` block (§{decl.register}) as an alternative register-phase context (informative).

## 22.3. `cook.recipe` [#lua.cook-recipe]
**Signature.** `cook.recipe(name: string, metadata: table, body: function) -> nil`
**Phase.** Register-phase only.

Registers a recipe with the given name, metadata (`ingredients`, `excludes`, `requires`, `origin`), and body function. Surface recipes (§{recipes.header-forms}) are compiled to exactly one `cook.recipe` call. A conforming implementation is not required to expose `cook.recipe` as a user-facing extension point; its contract here is what the code generator (§{grammar-appendix.top-level}) produces.

**The `origin` key.** The `metadata` table accepts an optional `origin` key alongside `ingredients`, `excludes`, and `requires`. When present, `origin` MUST be a non-empty string naming the module surface that minted the recipe; the conventional value is the module-qualified name of the function that called `cook.recipe` (for example, `"cook_pnpm.workspace"`). It is an **annotation**: it attributes a registered name to the module that produced it, for the benefit of a reader who did not write that name. Its sole normative effect is on a listing surface (§{lua.reg.discovery}); it MUST NOT affect registration, name uniqueness, dispatch, scheduling, or any cache key. An absent `origin` is not an error and denotes an unannotated recipe.

An `origin` attributes a name to the module that produced it, so only a module-minted recipe carries one. A recipe declared as a surface `recipe NAME` or `chore NAME` (§{recipes.header-forms}, §{chores.header}) is a name the user wrote, and never carries an `origin`: a conforming implementation's code generator MUST NOT supply one for a surface declaration, and the discovered set MUST report no origin for a recipe registered that way. The obligation on a module that mints a recipe to supply an `origin` is normative at §{mods.authoring.minting}.

A present `origin` whose value is not a string, or which is the empty string, MUST be rejected with a register-phase Lua error naming the field and the accepted form, rather than being silently coerced to absence. This is the field-typing discipline §{lua.add-unit} imposes on `cook.add_unit` (CS-0127), applied to `cook.recipe`'s metadata table.

**Name uniqueness within a Cookfile.** Within a single Cookfile registration pass (§{exec.two-phase}), the set of names registered by `cook.recipe` MUST be unique. This applies regardless of registration site — surface `recipe NAME` and `chore NAME` declarations (§{recipes.header-forms}, §{chores.header}) and dynamic `cook.recipe` calls from `register` blocks (§{decl.register}), top-level module calls (§{toplevel.module-call}), or wrapper Lua functions (such as `cook_cc.bin`, §{cat.cc}) all participate in the same uniqueness namespace.

A conforming implementation MUST diagnose a duplicate name with a single error that names the offending recipe and identifies both registration sites by their source line. The diagnostic MUST distinguish surface declarations from register-phase Lua calls. The mechanism by which an implementation distinguishes the two registration paths is implementation-defined; the reference implementation tags surface registrations via a codegen-private helper while leaving the public `cook.recipe` API unchanged. See §{rationale.recipe-name-uniqueness}.

## 22.4. `cook.add_test` [#lua.cook-add-test]
**Signature.** `cook.add_test(spec: table) -> nil`
**Phase.** Register-phase only.

`cook.add_test` records a single test work unit. The code generator lowers each `test_step` (§{steps.test}) to exactly one `cook.add_test` call. The table argument `spec` accepts the following fields:

| Field | Type | Required | Default | Constraint |
|---|---|---|---|---|
| `command` | string | exactly one of `command` / `lua_code` | — | MUST be non-empty when provided. The test body is executed as a shell command via `/bin/sh`. |
| `lua_code` | string | exactly one of `command` / `lua_code` | — | MUST be non-empty when provided. The test body is Lua source, executed on an execute-phase worker VM under the same sandbox policy as a `cook`-step body. The test passes if the chunk completes without raising a Lua error, and fails if it raises one. |
| `suite` | string | no | enclosing recipe's qualified name | If absent or empty, a conforming implementation MUST substitute the fully-qualified name of the recipe currently executing in the register phase (e.g. `"lib.tests"` for a recipe `tests` imported under prefix `lib`). |
| `line` | integer | no | `0` | Source line number of the originating `test_step` in the Cookfile, used in diagnostic output and test labels (e.g. `recipe@2`). The code generator MUST populate this field; a conforming implementation MUST NOT require it to be non-zero. |
| `iteration_item` | string or nil | no | `nil` | For OneToOne test steps, the current iteration input path (e.g. `_test_in`). When non-nil and non-empty, a conforming implementation MUST include it as a discriminator in the test identity (e.g. `recipe:name[item]`) and in the displayed label (e.g. `recipe@line [item]`). For ManyToOne and OneShot test steps, the code generator MUST pass `nil`. |
| `inputs` | array of string | no | `{}` | Declared ingredient files for this test: paths whose content contributes to the test's cache key (CS-0153). A conforming implementation MUST form the test's cache-input set as the union of `inputs` and the enclosing step group's dependency-output paths, deduplicated and order-preserving — mirroring `cook.add_unit`'s `inputs` (§{lua.add-unit}). The code generator populates this field from the recipe's resolved `ingredients` when the recipe declares any; a module target-maker MAY pass it directly to key a hand-registered test on the file set it consumes. |
| `seal` | array of string | no | `{}` | The test unit's **effective** seal set: bare probe keys whose values fold into the test's cache key (§{exec.cache.test-unit} rule 1, CS-0159). Mirrors `cook.add_unit`'s `seal` (§{lua.add-unit}) exactly — only the key set is register-time data; the values are resolved at execute phase. The code generator populates this field from the recipe-level `seal` baseline folded with the `test_step`'s `test_mods` tail (§{steps.cook-disposition.test-tail}), so the baseline/tail arithmetic is resolved *before* this call, not by it. A conforming implementation MUST union these keys into the unit's probe-dependency set, so every sealed probe is scheduled ahead of the test. A module target-maker MAY pass it directly to seal a hand-registered test. |

**Test unit naming (CS-0160).** `cook.add_test` accepts no `name` field (CS-0135). A conforming implementation MUST derive the recorded test unit's name as `<recipe>_test<N>`, where `<recipe>` is the enclosing recipe's fully-qualified name (the same value the `suite` default reads) and `<N>` is the 1-based ordinal of this test unit among the test units recorded so far in the enclosing recipe body. The derived name is the unit's label in progress and log output, and it composes with `iteration_item` in the test identity (e.g. `checks:checks_test1[src/a.c]`) exactly as the removed `name` field once did.

A conforming implementation MUST apply the field defaults and raise register-phase Lua errors for invalid field values at the time `cook.add_test` is called, not deferred to execute time. Error messages MUST identify the offending field by name. Every documented field has exactly one accepted Lua type (or, for `command`/`lua_code`, the "exactly one of" pair above); a conforming implementation MUST reject a wrong-typed value with a register-phase error naming the field, the received type, and the accepted type, rather than silently coercing it to a default (CS-0127, mirroring §22.1's `cook.add_unit` field-typing rule). Supplying both `command` and `lua_code`, or neither, MUST raise a register-phase Lua error naming both fields. An empty-string `command` or `lua_code` is treated as absent for the purpose of this exactly-one check (so `command = ""` with a `lua_code` present is a valid Lua-body test, and both empty is the "neither supplied" error).

## 22.5. `cook.step_group` [#lua.step-group]
**Signature.** `cook.step_group(fn: function) -> nil`
**Phase.** Register-phase only.

`cook.step_group` invokes `fn` synchronously on the register-phase Lua VM. Every `cook.add_unit` (§{lua.add-unit}) call made within the dynamic extent of `fn` records its unit as a member of a single **step group** (§{exec.step-groups}). Step-group members MAY execute in parallel during the execute phase.

`cook.step_group` MUST return after `fn` returns; nested `cook.step_group` calls within `fn` are unspecified.

A conforming implementation MUST raise a Lua runtime error when `cook.step_group` is called from execute-phase Lua. The diagnostic MUST identify the function name and the calling step kind.

### Example 22.5.1

```lua
cook.step_group(function()
    for _, source in ipairs(inputs) do
        cook.add_unit({
            inputs  = { source },
            output  = source:gsub("%.c$", ".o"),
            command = string.format("gcc -c %s -o %s", source, source:gsub("%.c$", ".o")),
        })
    end
end)
```

The N units registered inside the function body form one step group; the N compiles MAY run in parallel under the execute phase.

### Note 22.5.1

The surface `cook` step (§{steps.cook-single}, §{steps.cook-multi}) routes through `cook.step_group` implicitly. Authors of Cook modules (Part IV) typically call `cook.step_group` explicitly to record a fan-out workload.

## 22.6. Recipe discovery [#lua.reg.discovery]

A conforming implementation's **recipe-discovery procedure** is the operation that enumerates the names a Cookfile registration pass has made available for dispatch. The discovered set MUST include every recipe registered by `cook.recipe` (§{lua.cook-recipe}) during the pass, regardless of registration site (surface `recipe NAME` declarations, surface `chore NAME` declarations, `register` block bodies, top-level module calls, or wrapper Lua functions).

A `cook NAME` dispatch invocation MUST resolve `NAME` against the unified discovered set. A dispatch invocation that names a recipe registered only through register-phase Lua (for example, by a `cook_cc.bin(NAME, …)` call inside a `register` block) MUST execute identically to one naming a surface recipe declaration.

A conforming implementation MAY expose a listing surface (such as a `cook list` CLI subcommand) that enumerates the same unified discovered set without invoking any recipe body. When such a listing surface exists, it MUST surface register-phase-Lua–registered names alongside surface declarations. When such a listing surface exists and a discovered recipe carries an `origin` annotation (§{lua.cook-recipe}), the listing MUST surface that recipe's origin along with its name, so that a name the user did not write is attributable to the module that minted it. A recipe carrying no `origin` MUST be listed with no such attribution. How an origin is presented — its wording, placement, and layout relative to the name — is implementation-defined. A listing surface implemented as a no-body-invocation variant of the register pass MUST still evaluate `config` blocks (§{decl.config}) and top-level register-block code, since those may affect the set of names that `cook.recipe` ultimately registers.

**An origin annotation does not affect dispatch.** A `cook NAME` invocation that names an origin-annotated recipe MUST execute identically to one naming any other registered recipe, and a conforming implementation MUST NOT refuse such an invocation. This much *reaffirms* the requirement stated in paragraph 2 of this section rather than amending it: that paragraph already makes identical execution a MUST for every name in the unified discovered set, and an annotation carried on a recipe's metadata changes nothing about the set `NAME` resolves against. It is stated again here because a module author may describe a minted recipe as "non-invocable by intent" (§{mods.authoring.minting}), and a reader could otherwise take that phrase to license a refusal.

Extending that requirement beyond dispatch: a conforming implementation MUST NOT condition ordering or diagnostics on the presence, absence, or value of an `origin` either. An annotation records who minted a name; it is not a channel for behaviour.

A module author who describes a minted recipe — a support recipe in particular (§{mods.authoring.minting}) — as "non-invocable by intent" is stating **authorial intent**, and the annotation is the mechanism by which that intent reaches the user: it tells a reader who did not write the name who did, and why it exists. It carries no engine-enforced prohibition. The Standard defines no invocability tier below "registered"; a registered name is dispatchable. See §{rationale.recipe-origin-annotation}.

## 22.7. `cook.recipe_name` [#lua.cook-recipe-name]
**Signature.** `cook.recipe_name() -> string`
**Phase.** Register-phase only.

`cook.recipe_name` returns the fully-qualified name of the recipe whose body is currently executing in the register phase, consistent with §{lua.cook-add-test}'s `suite` default (§22.4): under an import prefix, the returned name is qualified the same way (e.g. `"lib.build"` for a recipe `build` imported under prefix `lib`), not the bare declared name.

`cook.recipe_name` therefore differs from the `recipe` global's `recipe.name` field (§{lua.recipe-global}), which is bound in the same scope but holds the *declared* (bare) name: in a recipe `build` imported under prefix `lib`, `recipe.name` is `"build"` while `cook.recipe_name()` is `"lib.build"`. The two coincide in a root Cookfile, where no prefix applies. A module that needs the recipe's session-unique identity — to key an export, or to distinguish two like-named recipes imported from different Cookfiles — MUST use `cook.recipe_name`; a module that needs the name the author wrote, such as for a human-facing artifact filename, SHOULD use `recipe.name`.

A conforming implementation MUST raise a register-phase Lua error when `cook.recipe_name` is called outside a recipe body. The diagnostic MUST name the API (`cook.recipe_name`) and state the inside-a-recipe-body requirement. This obligation applies to any register-phase Lua that is not currently executing inside a recipe body — a top-level call, a `register` block body, and a `for_each`-feeding probe's `produce` body when that `produce` is evaluated on the register-phase VM as part of the pre-pass (§{cat.probes.for-each}) are all outside a recipe body and MUST error. A probe's `produce` body is not a recipe body in general; an ordinary (non-`for_each`-feeding) probe's `produce` runs on an execute-phase worker VM, where `cook.recipe_name` is not registered at all, so this MUST does not — and cannot — extend to that case.

### Note 22.7.1

Because the returned name is fully-qualified (per §22.4), a module deriving artifact identity from `cook.recipe_name()` sees the import prefix folded in wherever one applies: a recipe declared `recipe idLib` and imported under prefix `lib` yields `cook.recipe_name() == "lib.idLib"`, which — if a module strings it directly into an artifact or path template — can produce forms like `liblib.idLib.a` or `build/obj/lib.idLib/`. This is legal but visually ugly, and it creates a tension for a module that resolves link references written against *bare* recipe names (as authors normally write them, e.g. `$<idLib>`): such a module would disagree with a qualified derivation taken from `cook.recipe_name()` inside an imported Cookfile, even though both name the same recipe. Root Cookfiles (no import prefix) are unaffected, since the qualified and bare forms coincide there.

The remedy is already available: `recipe.name` (§{lua.recipe-global}) is bound in the same scope and holds the bare declared name. A module deriving a filename SHOULD read `recipe.name` and reserve `cook.recipe_name()` for identity, rather than stripping a prefix back off the qualified form. The two accessors are complementary, and choosing between them per use is the module author's call.

## 22.8. `cook.require_recipe` [#lua.cook-require-recipe]
**Signature.** `cook.require_recipe(name: string) -> nil`
**Phase.** Register-phase only.

`cook.require_recipe` declares, from inside a recipe body, that the enclosing recipe depends on the recipe named `name`. The call has two effects: it orders the current registration pass, and it establishes a cross-recipe dependency edge.

**The register-order guarantee.** When `cook.require_recipe(N)` returns, the body of the recipe named `N` has been fully evaluated in the current Cookfile registration pass (§{exec.two-phase}). A conforming implementation MUST evaluate `N`'s body to completion before the call returns, if it has not already been evaluated in this pass. A recipe body MUST be evaluated at most once per registration pass: a second `cook.require_recipe(N)` call — from the same recipe body or any other — MUST NOT re-evaluate `N`'s body, and MUST return with the guarantee already satisfied by the first evaluation. Everything `N`'s body registers during that evaluation — its work units, its tests, and any value it publishes via `cook.export` (§{lua.cook-exports-imports}) — is therefore observable to the caller when the call returns.

**The guarantee is transitive.** Forcing `N` MUST also guarantee that every recipe `N` itself depends on — by dep-list entry, by `requires` metadata (§{lua.cook-recipe}), or by `N`'s own `cook.require_recipe` calls — has been evaluated first, to the same standard, except where the dependency names a recipe registered in another Cookfile (§{comp.import}), whose evaluation this section does not govern. This follows from the guarantee above rather than extending it: `N`'s body is entitled to read the exports of the recipes `N` declares it depends on, so a `N` evaluated before its own dependencies has not been "fully evaluated" in any sense the caller can use. An implementation that orders a pass by sorting the declared dependency graph up front MUST NOT rely on that order to discharge this obligation, because a forced evaluation by construction departs from it: forcing `N` runs `N`'s body at the moment of the call, which may precede the point the sort assigned to any of `N`'s dependencies. Getting this wrong is worse than an outright failure — the forcing caller observes `N`'s export and is satisfied, while `N` itself silently read `nil` for its own dependency, so the resulting mislink is partial and no diagnostic fires.

**The at-most-once rule does not discharge the transitive guarantee.** A `cook.require_recipe(N)` that finds `N`'s body already evaluated MUST NOT re-evaluate it, but it owes the transitive guarantee in full, exactly as the call that first evaluated `N` did. An implementation that records "`N`'s body has been evaluated" MUST NOT read that record as also settling `N`'s dependencies: if that evaluation was performed by the implementation's own up-front ordering rather than by a force, then nothing has yet stood down whatever skips apply to `N`'s dependencies (see "The forcing MUST NOT degrade to a silent skip" below), and this call is the first thing to place them in a build closure. Whether an implementation is exposed to this is decided by nothing more than whether it happened to reach `N` before or after the caller — which is to say, by the recipes' *names* — so the already-evaluated case and the not-yet-evaluated case MUST be brought to the same result. Where a dependency of an already-evaluated `N` cannot be evaluated, the error the previous paragraph requires MUST still be raised: an implementation that raises it only when the force is what evaluates `N` has made its own designed diagnostic conditional on a spelling.

**The edge.** The call also establishes a cross-recipe dependency from the calling recipe to `N`, identical in kind and granularity to a dep-list entry (`recipe A : N`, §{xref.dep-implications}). The edge is at whole-recipe granularity — every unit of the caller waits for every unit of `N` — and it MUST NOT be refined to a per-unit edge in isolation: §{exec.cross-recipe-ordering} reserves refinement to placeholder-sourced edges and forbids it for dep-list-sourced ones, and this edge is a dep-list-sourced edge in all but its spelling. The one qualification is fine-covered narrowing (§{xref.dep-implications}, CS-0161): where the calling recipe ALSO carries a placeholder-sourced per-unit edge to `N` — a `$<N>` reference, `cook.dep_output("N")`, or `cook.dep_order("N")` (§{lua.cook-dep-order}) — an implementation MAY narrow this edge to those per-unit edges. `cook.require_recipe` alone never triggers narrowing. `N` MUST join the build closure of any invocation that reaches the caller, narrowed or not.

**This is an instance of the name-reference rule, not an exception to it.** `cook.require_recipe(name)` takes a **recipe name** and creates an edge from that name. It is therefore an instance of §{xref.dep-implications}'s "Name reference ⇒ cross-recipe edge" rule. That section's prohibition — "An implementation MUST NOT infer an edge from path-string equality" — constrains the **evidence** from which an edge may be inferred, not the mechanism by which one is declared. A path string that happens to equal another recipe's output is not evidence of a dependency, because the author never named the referent; the match is a coincidence of spelling that the implementation would be guessing about. `cook.require_recipe` supplies exactly the evidence §{xref.dep-implications} demands: the referent recipe is *named* — by the author directly, or by a module acting on a name the author wrote (e.g. `links = {"idLib"}`). The edge is **authored, not inferred**. See §{rationale.name-reference-evidence}.

Consequently §{exec.cross-recipe-ordering}'s "No other source of cross-recipe ordering is defined" **stands**, and this section admits no non-name source of ordering. That section's own first sentence already scopes ordering to name references — "Cross-recipe ordering is established only by name references" — and its enumeration of where such a reference may appear **is amended** by this section to name a fourth spelling: a name reference from recipe `A` to recipe `B` may appear in a dep-list, a bare placeholder, an accessor placeholder, or a register-phase `cook.require_recipe` call. The amendment is to the enumeration only; the **category** it enumerates — name references, and nothing else — is unchanged, which is why the "No other source" closer survives verbatim. §{exec.cross-recipe-ordering}'s specific disclaimer — that string equality between a path literal of `A` and an output path literal of `B` does not establish an edge — likewise **stands**, unchanged in force. Its *wording* is clarified by this entry, in the opposite direction from an exception: where it once named only the ingredient/cook-output pair, it now states that the prohibition holds for every literal path surface alike — `ingredients` line literals, `cook` step output literals, and the `inputs[]` / `outputs[]` entries of `cook.add_unit` (§{lua.add-unit}). Nothing that established no edge before establishes one now; the clarification only makes explicit how many surfaces the prohibition already bound. `cook.require_recipe` is outside all of them, because it never takes a path.

The enumeration is amended rather than merely read broadly because it sits inside a MUST-bearing sentence, closed by "No other source of cross-recipe ordering is defined." A conformance-test author reading §{exec.cross-recipe-ordering} alone would otherwise find a taxonomy with no slot for a `cook.require_recipe` edge, and correctly conclude that honouring one is non-conforming. Widening the list is what makes this section's edge legible from the section that governs ordering; asserting the category argument here while leaving that list unable to express it would put the two sections in contradiction.

By contrast, §{xref.dep-implications}'s prohibition on inferring an edge from path-string equality is **not** amended, and needs no amendment: `cook.require_recipe` takes a **name**, never a path, so it cannot launder a path match into an edge. The two rules are orthogonal — one governs where a name reference may be written, the other governs what does not count as one.

The implementation reinforces the point rather than establishing it: the edge is carried by the very same `requires` metadata field (§{lua.cook-recipe}) that the surface dep-list lowers to, and that CS-0121's `$<NAME>` body references already merge into. `cook.require_recipe` is not a second dependency channel; it is a third writer to the existing one.

**`name` is bare.** `cook.require_recipe` resolves `name` against the **bare declared names** registered in the current Cookfile registration pass — the same namespace the surface dep-list and the `requires` metadata field (§{lua.cook-recipe}) resolve against. `name` MUST be a bare declared name; a conforming implementation MUST NOT resolve it against qualified names (§{comp.qualified-refs}). Three reasons:

1. **Consistency of the field it writes.** The API contributes to `requires`, so it MUST resolve the way `requires` resolves. Any other rule would make one string in one list mean two different things depending on which mechanism placed it there.
2. **Authors write names bare.** Link references are written bare — `$<idLib>`, `links = {"idLib"}` — and §{xref.dep-implications} name references are bare.
3. **A prefix is not knowable from inside.** An import prefix is assigned by the **importer** (§{comp.import}, §{comp.qualified-refs}), so a Cookfile's own body cannot know which prefix it will be imported under, or whether it will be imported at all. A qualified rule would break `links = {"idLib"}` the moment that Cookfile were imported under a prefix; the bare rule is prefix-invariant.

This section relates to §{lua.cook-recipe-name} exactly as Note 22.7.1 anticipates. `cook.recipe_name()` returns the *qualified* name and is for identity; `recipe.name` (§{lua.recipe-global}) holds the *bare declared* name and is the name the author wrote. `cook.require_recipe` takes a name the author wrote, so its argument is bare. This section is the concrete resolution of the tension Note 22.7.1 flags: a module resolving link references written against bare recipe names now has a dependency API that agrees with them, and needs neither to strip a prefix back off a qualified form nor to guess one.

**Errors.** A conforming implementation MUST raise a register-phase Lua error in each of the following cases, at the time `cook.require_recipe` is called. Each diagnostic MUST name the API (`cook.require_recipe`) and state what the caller must do instead.

- **Called outside a recipe body.** The diagnostic MUST state the inside-a-recipe-body requirement. As with §{lua.cook-recipe-name}, this obligation applies to any register-phase Lua that is not currently executing inside a recipe body: a top-level call, a `register` block body (§{decl.register}), and a `for_each`-feeding probe's `produce` body when that `produce` is evaluated on the register-phase VM as part of the pre-pass (§{cat.probes.for-each}) are all outside a recipe body and MUST error. An ordinary (non-`for_each`-feeding) probe's `produce` runs on an execute-phase worker VM, where `cook.require_recipe` is not registered at all, so this MUST does not — and cannot — extend to that case.
- **`name` is not a string, or is the empty string.** The value MUST be rejected with an error naming the field (`name`) and the accepted form, rather than being silently coerced. This is the field-typing discipline §{lua.add-unit} imposes on `cook.add_unit` (CS-0127), applied to `cook.require_recipe`'s argument.
- **`name` names the enclosing recipe itself.** A recipe MUST NOT require itself. This is the degenerate cycle, and MUST be rejected with its own diagnostic naming the recipe, rather than deferred to cycle detection and reported as a one-element cycle path. The comparison is bare-to-bare — `name` against the enclosing recipe's bare declared name (§{lua.recipe-global}), never against its qualified name (§{lua.cook-recipe-name}), since under an import prefix the qualified form never equals the bare argument and the check would silently fail to fire.
- **`name` is not a recipe registered in the current pass.** The reference MUST be rejected, with a diagnostic naming both the unresolved name and the requiring recipe. This is §{xref.dep-implications}'s standing validation requirement — that a named recipe be defined in scope before the execute phase begins — discharged at register phase, where the forcing makes it reachable earlier.
- **The edge closes a cycle.** If the edges established by `cook.require_recipe`, taken together with all other cross-recipe edges (§{exec.cross-recipe-ordering}), contain a cycle, the implementation MUST reject the Cookfile with a diagnostic rendering the cycle path. Because the forcing is synchronous, a cycle is reachable during the registration pass itself — `A` requiring `B` whose body requires `A` would otherwise recurse without bound — and MUST be diagnosed as a cycle rather than surfacing as a stack overflow.

**The forcing MUST NOT degrade to a silent skip.** An implementation may decline to evaluate certain recipe bodies during a pass — a parametric chore that no invocation reaches, say, or a `for_each` recipe whose data source it has not resolved. Where such a skip is decided from information gathered *before* any body runs, it cannot account for an edge a body declares while running. An implementation MUST NOT let a skip of that kind silently swallow a `cook.require_recipe` call: if it cannot evaluate `N`'s body, it MUST raise a register-phase error naming the API, `N`, and the reason. Registering `N` with no work units while the edge still places `N` in the build closure is expressly non-conforming — the register and execute phases would disagree about what is being built, which is the exact failure this section exists to prevent. Two consequences follow for the cases where a skip decision and a forced evaluation meet:

- **A forced recipe is reachable by definition,** so a skip predicated on reachability MUST NOT apply to it. This extends to a recipe forced *transitively*, via the dependencies of a forced recipe: the edge places `N` in the caller's build closure, and every recipe `N` requires is in that closure with it, so a skip that stands down for `N` but applies to `N`'s own dependency reopens the same register/execute disagreement one level down. In particular a parametric chore (§{chores.params}) that is forced MUST be evaluated with no argv supplied, exactly as §{chores.parametric-deps} requires of a chore reached as a dependency of the invocation's target — a required parameter with no default then surfaces a legitimate register-time configuration error, rather than the recipe vanishing.
- **Where evaluation is genuinely unavailable, erroring is conforming;** it is the silence that is not. An implementation whose `for_each` data sources (§{cat.probes.for-each}) are resolved by a pre-pass keyed on the static dependency graph MAY reject a `cook.require_recipe` naming a `for_each` recipe whose source that pre-pass did not resolve. Such an implementation MUST say so, and SHOULD direct the author to declare the dependency statically as well (`recipe A : N`), which brings `N` into the pre-pass's view and makes the same call succeed. This is a bounded implementation limit, not a property of the API: nothing in this section prevents an implementation from resolving the source on demand at force time instead.

### Note 22.8.1

The motivating case is a module that must resolve a link reference from inside a recipe body it does not own. A C/C++ target maker resolving `links = {"idLib"}` reads `idLib`'s export (§{lua.cook-exports-imports}) to obtain its library path and transitive link info (§{cat.cc.transitive}). That read resolves only if `idLib`'s body has already been evaluated and published its export — `cook.import` returns `nil` for a target that has not yet exported (§{lua.cook-imports}).

Registration order within a pass is dependency-driven, not declaration order: two recipes with no declared dependency between them have no guaranteed relative order, so a module cannot assume that a recipe appearing earlier in the Cookfile has already run. Without a declared dependency, nothing makes `idLib`'s body run first, and the read silently returns `nil` — the maker then either fails obscurely or emits a link line missing the library. The author wrote `links = {"idLib"}` and reasonably expects both the ordering and the edge to follow from it.

`cook.require_recipe` is what lets a module bootstrap that edge from inside a recipe body it does not own. On seeing `links = {"idLib"}`, the maker calls `cook.require_recipe("idLib")`: `idLib`'s body is forced to completion, so the subsequent `cook.import("idLib")` resolves, and the recorded edge makes `idLib` build before the linking recipe. The author declared the dependency in their module's vocabulary; the module translated it into the Standard's.

## 22.9. `cook.on_register_complete` [#lua.cook-on-register-complete]
**Signature.** `cook.on_register_complete(fn: function) -> nil`
**Phase.** Register-phase only.

`cook.on_register_complete` queues `fn` to run once, after the current registration pass has finished evaluating every recipe body and performing its whole-graph validation, and before the implementation commits the pass's module-held state for the register→execute handoff (§{mods.lifecycle.rehydration}). It is the finalizer counterpart to a module's `init()` (§{mods.lifecycle.load-order}): `init()` runs before any recipe body registers; a callback queued through `cook.on_register_complete` runs after every recipe body has.

**Argument type.** `fn` MUST be a Lua function. A conforming implementation MUST reject any other type with a register-phase Lua error naming the API (`cook.on_register_complete`), the accepted type (`function`), and the received Lua type, rather than coercing or silently discarding it — the CS-0127 field-typing discipline (§{lua.add-unit}) applied to this API's sole argument.

**Ordering and timing.** Queued callbacks run on the register-phase Lua VM, exactly once each, in registration order — the order in which each `cook.on_register_complete` call was made — strictly after every recipe body of the registration pass has been evaluated and after the pass's whole-graph validation, including the merged cross-recipe cycle check over the edges §22.8 establishes ("The edge closes a cycle", §{lua.cook-require-recipe}) and the `for_each` static-input rule (§{cat.probes.for-each}), and before the implementation commits the pass's module-held state for the handoff of §{mods.lifecycle.rehydration}. Module-held per-VM state (§{mods.lifecycle.caching}) MUST still be live when a callback runs, and any mutation a callback makes to that state MUST be visible to the handoff that follows.

**Outside any recipe body.** A callback runs outside the dynamic extent of any recipe body. A body-scoped register API called from a callback — `cook.add_unit` (§{lua.add-unit}), `cook.add_test` (§{lua.cook-add-test}), `cook.recipe_name` (§{lua.cook-recipe-name}), `cook.require_recipe` (§{lua.cook-require-recipe}) — MUST raise that API's ordinary outside-a-recipe-body error. The recipe set and the work-unit set of the pass are closed by the time any callback runs: every callback in the queue observes the same fixed universe of recipes and units.

**Recipe and probe registration is rejected.** A callback MUST NOT register a recipe via `cook.recipe` (§{lua.cook-recipe}) or a probe via `cook.probe` (§{cat.probes.api}). A conforming implementation MUST reject such a call with a register-phase Lua error naming `cook.on_register_complete` and the API the callback attempted to call, rather than accepting the registration and silently discarding it. The body-invocation loop has already completed when callbacks run: a recipe minted at this point would never have its body evaluated in the current pass, reproducing the exact register/execute silent-disagreement failure mode that this chapter's forcing and skip rules (§22.8, "The forcing MUST NOT degrade to a silent skip") exist to close.

**Re-queuing.** A callback MAY itself call `cook.on_register_complete`. The newly queued callback is appended to the end of the queue and runs after every callback queued before it, including callbacks queued by other callbacks; a conforming implementation MUST drain the queue until it is empty, preserving registration order and the fires-exactly-once guarantee for every callback, however it was queued.

**Errors propagate.** An error raised by a callback MUST fail the registration pass with that error, exactly as an error raised from a recipe body fails the pass.

**Everything else is unrestricted.** Outside the two restrictions above, a callback MAY call any other register-phase or both-phase Cook Lua API reachable from register-phase Lua: `cook.import` / `cook.export` (§{lua.cook-exports-imports}), `cook.probes.get` (§{lua.cook-probes-get}), module-held Lua state, `fs.*` (§{lua.fs}), `cook.sh` (§{lua.cook-sh}, which executes immediately since it is not itself a recipe body), and `cook.load_module` (§{lua.cook-load-module}).

**Discovery surfaces MAY skip the queue.** A listing surface that enumerates the discovered recipe set without invoking any recipe body (§{lua.reg.discovery}) MAY complete its enumeration without draining queued callbacks: because callback registration is rejected (see above), a callback cannot alter the discovered set, so nothing such a surface reports depends on having run them. A call to `cook.on_register_complete` reached by such a surface MUST still be accepted — it is register-phase Lua like any other — and its callback simply MUST NOT be required to run on that pass.

### Note 22.9.1

The motivating case is a module that accumulates per-target state across many recipe bodies and must act on the complete set exactly once, rather than once per target. A `compile_commands()`-style top-level call in a C/C++ module wants to write one `compile_commands.json` covering every `cc.bin` / `cc.lib` target the Cookfile registers — but the module cannot know, from inside any single target's body, whether more targets remain to register. Before `cook.on_register_complete`, the only way to sequence this was a fake ordering dependency — declaring an aggregation recipe that depends on every target, and keeping that list in sync by hand. Queuing the aggregation as a callback removes the fake dependency: the module calls `cook.on_register_complete(fn)` once, and `fn` runs with every target's contribution already recorded, however many targets the Cookfile ultimately declares.

## 22.10. `cook.dep_order` [#lua.cook-dep-order]
**Signature.** `cook.dep_order(name: string) -> nil`
**Phase.** Register-phase only.

`cook.dep_order` records an **ordering-only** fine-grained name reference from the current recipe's subsequently-added units to the recipe named `name`. It is the ordering half of `cook.dep_output` (§{lua.cook-dep-output}) with the data half removed. Like `cook.dep_output`, the reference attaches to every `cook.add_unit` call made after it in the current recipe body, up to the close of the enclosing `cook.step_group` if any (§{lua.step-group}), producing a per-unit execution edge from each such unit to `name`'s terminal units. Unlike `cook.dep_output`, it returns nothing, resolves no output paths, and contributes nothing to the unit's cache-input set: the referenced recipe's outputs are not read by this reference, and a conforming implementation MUST NOT fold them — or any other artifact of the reference — into the cache key of the units it attaches to.

**Purpose.** `cook.dep_order` exists for units that must run after another recipe's work without consuming its artifacts, and — through fine-covered narrowing (§{xref.dep-implications}, CS-0161) — for recipes that want their `requires` edge narrowed to the units that actually consume. The motivating shape is a C/C++ module's static-library target: its recipe requires the libraries it links against because its compiles need their register-time exports (include paths, defines), but no unit of the recipe reads the upstream archive — the archive unit consumes only the recipe's own objects, and the final consumer of the upstream archive is a downstream binary's link unit. Naming the upstream via `cook.dep_order` immediately before registering the archive unit mints the per-unit edge that fine-covers the `requires` edge, freeing the recipe's compile units to run in one flat wave with the upstream's compiles while preserving a defined ordering point for everything registered after.

**Resolution.** `name` resolves exactly as `cook.dep_output`'s argument does — against the same namespace, with the same self-prefix qualification and alias handling (§{lua.cook-dep-output}). Unlike `cook.dep_output`, a recipe with no terminal outputs is a legal referent: a zero-step meta recipe is a legitimate ordering target, so no terminal-output existence check applies at call time.

**Closure membership is not established.** A `cook.dep_order` reference presumes its referent is already in the build closure — via a dep-list entry, `requires` metadata, `cook.require_recipe`, or transitively through them. The call itself MUST NOT pull `name` into the closure and MUST NOT force `name`'s body (§{lua.cook-require-recipe}); it records the reference and returns. Where the referent is absent from the closure at plan time, a conforming implementation MUST reject the build with a diagnostic naming the referring recipe and the referent, and directing the author to add the missing dependency declaration.

**Errors.** A conforming implementation MUST raise a register-phase Lua error when `cook.dep_order` is called outside a recipe body, naming the API and stating the inside-a-recipe-body requirement, mirroring `cook.dep_output`'s error model.

### Note 22.10.1

The division of labour among the three edge-writing APIs: `cook.require_recipe` forces the referent's body and establishes the whole-recipe edge (register-order plus closure membership); `cook.dep_output` reads the referent's outputs and mints a per-unit consuming edge (paths returned and folded into the unit's cache inputs); `cook.dep_order` mints the per-unit edge alone. A module typically pairs the first and the third: `cook.require_recipe` at the top of the maker for exports and closure, `cook.dep_order` immediately before the unit that must observe the upstream's completion.
