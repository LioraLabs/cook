# 5. Cross-recipe references

> **Normative.** This chapter defines *name references*: the surface by which one recipe refers to another. It covers three syntactic forms — dep-list references (`recipe A: B`), bare placeholders (`{B}`), and accessor placeholders (`{B.stem}`) — and specifies their resolution, expansion, and effect on the dependency graph. The formal grammar of the placeholders is given in App. A; this chapter specifies their meaning.

## 5.1. Name references

A **name reference** is the sole mechanism by which a recipe refers to another recipe. A name reference MAY appear in one of three surfaces:

| Surface | Location | Form |
|---|---|---|
| Dep-list | Recipe header `: …` (§ 4.2) | `NAME` |
| Bare placeholder | Any step template string | `{NAME}` |
| Accessor placeholder | `cook` step output pattern (§ 4.5, § 4.6) | `{NAME.ACCESSOR}` |

All three surfaces share the resolution rules of § 5.2 and the dependency implications of § 5.6. Expansion behaviour differs by surface: dep-list references have no template expansion; bare placeholders expand per § 5.5; accessor placeholders drive iteration per § 5.4.

## 5.2. Resolution

A conforming implementation MUST resolve a `{TOKEN}` appearing in a step template by applying the following steps in order, with the first matching step winning:

1. If `TOKEN` is one of the current-iteration builtins (`in`, `out`, `stem`, `name`, `ext`, `dir`, `all`), substitute per § 6.7.
2. If `TOKEN` is exactly equal to a recipe name in scope, the reference names that recipe.
3. If `TOKEN` contains `.`, split on the LAST `.` into `PREFIX.SUFFIX`. If `PREFIX` equals a recipe name in scope AND `SUFFIX` is a path accessor (§ 5.3), the reference names that recipe with the named accessor.
4. Otherwise, `TOKEN` resolves to `cook.env[TOKEN]` (§ 6.7).

A name reference appearing in a dep-list is resolved against the recipe names in scope with no accessor split: the whole string is the recipe name. Qualified cross-Cookfile dep-list references (`alias.recipe`) follow the resolution rule of § 7.

### 5.2.1. Dotted recipe names

Recipe names MAY contain `.` separators. The whole-token match in step (2) above is tried before the accessor-split in step (3), so a dotted recipe name like `backend.build` is matched as a single unit when it exactly equals a recipe name in scope.

### 5.2.2. Reserved final segment

A recipe's final dot-separated segment MUST NOT equal a path accessor (§ 5.3) or any of `in`, `out`, `all`. A conforming implementation MUST reject a Cookfile that declares such a name, at parse time. The rule is enforced by the `check_reserved_recipe_name` production in App. A.

### 5.2.3. Environment-variable shadowing

When step (2) or (3) succeeds, step (4) is NOT consulted. If `cook.env[TOKEN]` is also defined for the same `TOKEN`, the recipe name wins. A conforming implementation MUST emit a warning naming both.

## 5.3. Path accessors

A **path accessor** is one of the four names `stem`, `name`, `ext`, `dir`. For a path `p`, the accessors are defined by the `path.*` helpers of § 6.6:

| Accessor | Equivalent |
|---|---|
| `stem` | `path.stem(p)` (§ 6.6.1) |
| `name` | `path.name(p)` (§ 6.6.2) |
| `ext`  | `path.ext(p)` (§ 6.6.3) |
| `dir`  | `path.dir(p)` (§ 6.6.4) |

The set of path accessors is closed: an accessor not in this list is not a path accessor for the purposes of § 5.2 or § 6.7. Extending the set is a change to this Standard, not a library change. See rationale B.5.P.

## 5.4. Dep-driven iteration

Rule. When a `cook` step's output pattern list contains a `{NAME.ACCESSOR}` accessor placeholder naming recipe `lib`, the step's iteration driver is `lib`'s output list, not this recipe's own input list.

- **Iteration shape.** The step is a **one-to-one step (§ 4.5)**. For each item `x` in `lib`'s output list (§ 5.4.1), one work unit is registered. Within that unit, the iteration item is `x`, and the bare-accessor placeholders (`{stem}`, `{name}`, `{ext}`, `{dir}`) and `{in}` refer to `x`. A placeholder `{lib.ACCESSOR}` in the same step refers to the same accessor of `x`.
- **One driver per step.** A step's output pattern list MAY contain accessor placeholders referring to AT MOST ONE distinct recipe. Multiple occurrences referring to the same `lib` (possibly with different accessors) are permitted and refer to the same driver. Accessor placeholders naming two or more distinct recipes in the same step MUST be rejected with a diagnostic.
- **Bare `{lib}` in an output pattern.** A conforming implementation MUST reject a `cook` step whose output pattern list contains a bare `{lib}` (no accessor) naming a recipe. `{lib}` has no iteration semantics.
- **Accessor placeholder in a using-string.** A `{lib.ACCESSOR}` appearing in a `using`-string, `plate` command, `test` command, or bare shell when the same step's output pattern list does not declare a matching driver MUST be rejected with a diagnostic.
- **String-substitution references coexist.** A step MAY contain any number of `{libX}` bare references (§ 5.5) to other recipes, orthogonally to whichever recipe (if any) drives iteration. Those references do not participate in the "one driver" rule.

### 5.4.1. A recipe's output

A recipe's **output** is the output list of its last step, under the following rule applied per step kind:

| Step kind | Output list |
|---|---|
| `ingredients` | Passthrough: the resolved ingredient paths (§ 4.3) |
| `cook` (one-to-one) | The per-input outputs recorded at register time |
| `cook` (many-to-one) | The single output path |
| `plate` | Passthrough: the step's input list |
| `test` | Passthrough: the step's input list |
| Bare shell (`@` or `@!`) | Passthrough: the step's input list |

"Passthrough" means the step's output equals its input. A recipe with no steps has the empty list as its output.

### 5.4.2. Example

```cook
recipe protos
    ingredients "proto/*.proto"
end

recipe compile_protos
    cook "build/obj/{protos.stem}.pb.cc" using "protoc --cpp_out=build/obj {in}"
    cook "build/lib/libprotos.a" using "ar rcs {out} {all}"
end
```

The `compile_protos` recipe's first step is dep-driven by `protos`; for each `x` in `protos`'s output list (the resolved globs), one work unit is registered with `{in}` = `x` and `{out}` = `build/obj/STEM(x).pb.cc`. The second step chains on the first (§ 4.5), not on `protos`.

## 5.5. String-substitution references

Rule. A `{NAME}` bare reference in a `cook` `using`-string, `plate` command, `test` command, or bare shell MUST be substituted by the space-joined concatenation of the named recipe's output list (§ 5.4.1).

- **Empty output.** If the named recipe's output list is empty, the substitution is the empty string. A conforming implementation MUST emit a warning at register time, naming the referring recipe and the referent. The reference itself is not an error.
- **Lua lowering.** A conforming implementation MAY lower a `{NAME}` reference to `cook.dep_output("NAME")` and a `{NAME.ACCESSOR}` iteration to a `cook.dep_output_list("NAME")`-driven form. The Standard specifies the observable behaviour; the Lua forms are one conforming lowering.

### 5.5.1. Example

```cook
recipe libmath
    ingredients "src/math/*.c"
    cook "build/obj/math/{stem}.o" using "gcc -c {in} -o {out}"
    cook "build/lib/libmath.a" using "ar rcs {out} {all}"
end

recipe app
    cook "build/bin/app" using "gcc -o {out} main.c {libmath}"
end
```

The `{libmath}` reference in `app`'s using-string expands at register time to `"build/lib/libmath.a"` (the single-element output of `libmath`).

## 5.6. Dependency implications

- **Name reference ⇒ cross-recipe edge.** A name reference to recipe `B` appearing in any step or dep-list of recipe `A` establishes a cross-recipe dependency from `A` to `B`, observable through execution ordering (§ 8).
- **Granularity.** A dep-list reference (`recipe A: B`) MUST be interpreted at whole-recipe granularity: every unit of `A` waits for every unit of `B`. A placeholder reference (`{B}` or `{B.ACCESSOR}`) in a specific step of `A` MAY be refined by the implementation to a per-unit edge from that step's unit(s) to the unit(s) of `B` that produced the referenced output items. Coarse edges are always conformant; fine-grained refinement is an optimisation available to placeholder-sourced edges only.
- **Validation and cycles.** An implementation MUST validate that the named recipe is defined in scope before the execute phase begins, and MUST reject the Cookfile with a cycle diagnostic if the transitive closure of all cross-recipe edges contains a cycle, before executing any work unit.
- **Only name references create edges.** A string equal to another recipe's output path, appearing as a literal in an `ingredients` line, does NOT establish a cross-recipe edge. An implementation MUST NOT infer an edge from path-string equality. Authors who want "run this after that" without a name reference MUST use an explicit `: dep`. See rationale B.5.N.
