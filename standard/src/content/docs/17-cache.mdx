# 17. Cache semantics [#exec.cache]
> **Normative.** This chapter defines the abstract cache contract, the integrity rule for restoring artifacts from a content-addressable store, the discovered-inputs mechanism, the test-unit caching contract, and the portability invariants that allow a cache to travel between machines and workspace relocations.

## 17.1. Cache semantics (abstract) [#exec.cache.abstract]
A conforming implementation MAY maintain a **build cache** that avoids re-executing a work unit whose observable inputs are unchanged since the last successful completion of that unit. The Standard specifies the cache abstractly; the hash function, on-disk layout, eviction policy, and storage medium are implementation-defined (§{intro.non-scope}).

The **observable inputs** of a work unit are those of its properties that are derived from the Cookfile source or from the register-phase record. They comprise, at a minimum:

1. The command text or Lua chunk that the unit will execute.
2. The list of input paths declared for the unit.
3. The list of output paths declared for the unit.
4. The recipe's resolved environment (§{lua}).
5. For a unit produced by an `ingredients <probe>` source (§{steps.ingredients}), the **materialised data member**: the current iteration element rendered in a canonical, key-sorted form. Each per-member unit's fingerprint therefore depends on its own member's content — independent of the whole source value — so editing one member re-runs only that unit. (This composes with the probe-value cache of §{cat.probes.exec}, which fingerprints the member *set*: the two layers are independent.)
6. For a unit whose command text contains one or more file references (`$<file:PATH>`, §{phl.cook-step}, CS-0101), the resolved path list and the content of each resolved file, on the same footing as observable 2 plus the declared-input-content consideration below. Editing a referenced file MUST invalidate exactly the units that reference it; a glob gaining or losing a match MUST invalidate via the path-list change. The reference implementation realises this by appending the resolved paths to the unit's recorded input set.

> **Note (informative).** An implementation MAY realise the materialised-member observable (5) by folding the canonical member rendering into the unit's command-text contribution (observable 1) rather than as a distinct fingerprint field; the two are observably equivalent because both make the member part of the inputs the fingerprint depends on. The reference implementation takes this approach (COOK-64): the codegen attaches the canonical member to each fan-out unit, which is mixed into the command hash at registration.

A conforming implementation MAY additionally consider the content of the declared input files, the modification times of the declared output files, and the content of the recipe's ingredient set. It MUST NOT consider wall-clock time, process identifiers, or any state not derivable from the sources above. An implementation MAY exclude from the resolved-environment observable a set of environment variables whose values are known not to affect work-unit output (e.g., ambient session state, credentials unrelated to the build product); the specific set is implementation-defined.

When the cache indicates that a unit is up-to-date, the implementation MAY skip executing the unit and MUST treat it as successfully completed for the purposes of the dependency graph. When the cache indicates that a unit is stale or absent, the implementation MUST execute the unit and, on success, MAY record the unit's completion in the cache for subsequent runs.

A unit for which the register-phase record disables caching — for example, a bare shell step or an `@` step, whose registration form sets the cache flag to false — MUST be executed on every run of the containing recipe.

### 17.1.1. The single cache key [#exec.cache.single-key]

A cacheable work unit — a step with at least one declared output — has **exactly one** cache key. The key is a content-addressed digest:

> `key = H( declared input contents, command text / Lua chunk, declared output paths, consulted environment values, declared probe values )`

where:

- **declared input contents** are the contents of the unit's `ingredients` (§{steps.ingredients}), its `$<file:PATH>` references (§{phl.cook-step}), and its depfile-discovered inputs (§{exec.cache.discovered-inputs}), content-hashed;
- **command text / Lua chunk** is observable 1 of §17.1;
- **declared output paths** is observable 3 of §17.1;
- **consulted environment values** are the post-denylist values of the environment keys the unit reads — `$<KEY>` sigils and the `cook.env.KEY` static reads of §{exec.cache.lua-env-reads};
- **declared probe values** are the canonical values (§{cat.probes.values}) of every probe in the unit's *effective seal set* — the probes the author sealed the unit on (the disposition surface of §8.4.3, an additive set with no project-default and no narrowing form).

The key MUST NOT include any determinant the author did not declare. In particular, host or machine identity — target triple, libc version, locale — is NOT folded into the key unless the author declared it as a probe and sealed on it (`seal host`). The engine infers no machine-identity, tool, or environment determinant of its own; it caches over exactly what the author declared.

The **local cache and the shared (cross-machine) store are addressed by this same key.** There is no separate "shared" key and no two-tier loose/strict split. A consumer reuses a stored artifact for a key K if and only if it computes the same K, which holds if and only if every real determinant of the unit was declared. The cache is **trustworthy** exactly when serving a stored artifact for K — instead of re-running the unit — would produce a byte-equivalent artifact in the consumer's current context. Soundness therefore rests on author-declared determinants; under-declaration is the cache-correctness failure mode, observed and classified by the divergence oracle (the shadow-divergence tool surfaced by the reference implementation as `cook cache verify`, which classifies a divergence as an under-keyed determinant versus benign non-determinism). A unit may further opt out of sharing (cache locally only) or require fetch-only service through its disposition (§8.4.3).

> **Note (informative).** The reference implementation realises the declared-probe-values determinant as a single `seal_contribution` digest — the xxh3_64 of the unit's *effective seal set* rendered as sorted `key\0<canonical-value-bytes>` records (the canonical value bytes are those of §{cat.probes.values}). It is folded into the one cache key alongside the command and consulted-environment contributions. Because a probe's value is only available once the probe has run, the fold happens at execute-phase — after each sealed probe's value has materialised (§{cat.probes.exec}); the seal *key set* itself is register-time data carried on the unit's cache metadata, and a sealing unit therefore depends on its sealed probes. An empty seal set contributes the zero digest, so an unsealed unit's key is unchanged by this determinant. This realisation also bumps the reference implementation's cache schema version (a one-shot rebuild on the upgrade boundary, per §{exec.cache.portability}). (CS-0107.)

### 17.1.2. Tracking Lua reads of `cook.env` [#exec.cache.lua-env-reads]

> **Normative.** A `cook` step whose `>{ … }` body is a Lua block (§{lua.using-block-globals}) MAY read environment values through the `cook.env.<KEY>` accessor (§{lua.cook-env}). When it does, those reads MUST contribute to the unit's `consulted_env_keys` set (§{xref.resolution}) on the same footing as a shell-body `$<KEY>` sigil contributes. The recording is required for the cache-invalidation contract of §17.1: a value change in any consulted key MUST invalidate the unit's cache fingerprint.
>
> A conforming implementation MUST recover the consulted key set by static analysis of the Lua source text. It MUST recognise:
>
> 1. **Dot access** — `cook.env.IDENT` where `IDENT` matches the conventional environment-key shape `[A-Z_][A-Z0-9_]*`.
> 2. **String-indexed access** — `cook.env["KEY"]` and `cook.env['KEY']` where the key is a literal string. No case constraint applies because the author has explicitly named the key.
>
> The analysis MUST ignore occurrences appearing inside Lua short-string literals (`"…"`, `'…'`), long-string literals (`[[…]]`, `[==[…]==]`), line comments (`-- …`), and block comments (`--[[…]]`, `--[==[…]==]`).
>
> The analysis MUST NOT record a key for **dynamic-key reads** — `cook.env[var]`, `cook.env[string.upper(x)]`, and the like — because the key is not statically resolvable from the source text. Authors who need cache invalidation on a dynamic-key read MUST surface the key statically (e.g., assign to a local first: `local v = cook.env["KEY"]`).
>
> The analysis MUST NOT record a key for **writes** — `cook.env.X = …` and `cook.env["X"] = …`. The pattern is distinguished from a read by the assignment operator following the access (an `=` that is not part of `==`, `~=`, `<=`, or `>=`).
>
> An implementation MAY record additional keys (over-record) — over-recording only widens the set of cache-invalidating values and never silently serves stale output. An implementation MUST NOT under-record by ignoring a syntactically recognisable read of one of the forms above.
>
> The recording mechanism is shared with the shell-side `$<KEY>` sigil capture: the keys flow into the unit's `consulted_env_keys` list and are looked up against `cook.env` at unit-registration time, exactly as shell-body sigils are.

> **Note 17.1.2.1 — `envs`-seal vs. auto-fold (informative).** For an environment variable a unit *consumes*, this section's auto-fold already keys the unit on the variable's value. An author SHOULD NOT *additionally* seal an `envs` probe (§{cat.probes.decl}) that names the same consumed variable: the variable would then fold twice — once as a consulted-env value here, and once inside the sealed probe's value (§{exec.cache.single-key}). The two paths are for two different cases, and the author picks one: rely on auto-fold for a *consumed* variable, or declare an `envs` probe and `seal` it for an *unconsumed* one (§{steps.cook-disposition}). Double-folding is not a correctness bug — each fold is over the same value — but it is redundant surface and splits the variable's attribution across two determinants in `cook why` (§{exec.cache.why}).

### 17.1.3. Sharing and disposition effect [#exec.cache.sharing]

Caching over the local index is always on. **Sharing** — publishing a unit's output artifacts to, and fetching them from, the shared content-addressed store — is also on by default for every cacheable unit, keyed by the same single key (§{exec.cache.single-key}). The local index and the shared store are addressed by that one key; a consumer reuses a shared artifact for a key K if and only if it recomputes K. A unit with no declared output (e.g. a `plate`, §{exec.cache.reconcile}) and any side-effecting step is never cached or shared.

The unit's *disposition* (§{steps.cook-disposition}) selects its sharing policy. The disposition does not change the key (§{exec.cache.single-key}); it changes only whether the shared store participates and how a miss is resolved. A conforming implementation MUST observe:

1. *(unannotated)* / `seal` — the unit MUST publish each declared output to the shared store after a successful run, and MUST attempt to fetch its outputs from the shared store by key before executing — both when a local cache entry exists but its outputs have drifted or are missing (the restore of §{exec.cache.integrity}), and on a *cold miss* where no local cache entry exists. On a final miss (neither store holds the key) the unit is rebuilt.

2. `local` — the unit is cached in the local index only. It MUST NOT publish any output to, and MUST NOT fetch any output from, the shared store — not on a cold miss and not on a drifted-output restore. On any miss the unit is rebuilt.

3. `pinned` — the unit is **fetch-only / designated-producer**: it MUST be served from cache — the local index or a fetch-by-key from the shared store — and MUST NEVER be reproduced by the consumer. When neither store holds an artifact for the unit's key (a cold miss), a conforming implementation MUST fail the build with a diagnostic that names the pinned unit; it MUST NOT execute the unit, and it MUST NOT serve any dependent that required the unit's outputs.

`local` and `pinned` are mutually exclusive (§{steps.cook-disposition}). Every fetch from the shared store — drift restore or cold fetch-by-key — is subject to the integrity verification of §{exec.cache.integrity}: a fetched artifact whose bytes fail verification is treated as a miss.

> **Note (informative).** Because the local index and the shared store share the one key (§{exec.cache.single-key}), a machine-independent unit (one that sealed no host probe) shares fleet-wide: any machine that recomputes the key serves the same artifact. A `local` unit is the deliberate exception — its output stays on the machine that produced it. A `pinned` unit shifts the trust boundary from "reproduce on demand" to "a designated producer populated the shared store"; consuming a pinned unit that was never produced is a hard error rather than a silent local rebuild, which is the point of the disposition. (CS-0108.)

A conforming implementation MAY offer a per-client **publish-off (read-only)
mode**: an invocation-level switch (e.g. `--no-publish` / `COOK_NO_PUBLISH`) or a
project config (`[cloud] publish = false` in `.cook/cloud.toml`) under which the
client MUST NOT publish any artifact to the shared store for ANY unit, while it
MUST still fetch by key, drift-restore, and serve `pinned` units from the shared
store exactly as in the default mode. Publish-off is orthogonal to a unit's
disposition: it suppresses every upload globally (as `local` does per-unit) and
changes no key (§{exec.cache.single-key}). The engine MUST NOT assume a
locally-produced artifact was published; a fresh build under publish-off
completes by running the unit and recording it locally. Publish-off is a
client-side convenience for read-only consumers; *write-authorization* to a
shared store is the backend's concern (e.g. credentialed access), not a property
the client asserts. (CS-0111.)

### 17.1.4. The `nondet` disposition [#exec.cache.record]

A `cook` step MAY carry the `nondet` disposition (§{steps.cook-disposition}) — the renamed v3 `record` disposition (CS-0115). Its artifact is cached and shared and keyed by the single cache key (§{exec.cache.single-key}) EXACTLY as an unannotated step: `nondet` adds NO determinant and changes NO component of the key.

A `nondet` step states a *fact* about its output: that it is intrinsically NON-reproducible — for example an LLM completion or a generative image — so that re-running it in an identical context is NOT guaranteed to produce a byte-equivalent artifact.

Byte-equivalence is therefore WAIVED for a `nondet` step. When the cache holds an artifact for the step's key — the **recording** the `nondet` unit reuses — an implementation MUST treat that stored artifact as authoritative and MUST NOT re-execute the producer to reconcile a content difference; a present (on-disk or restorable) output is served as-is. A cold miss — no stored artifact and none restorable — MUST execute the producer once and MAY record its output.

`nondet` composes with `seal`: a change to any sealed determinant changes the key (§{exec.cache.single-key}) and therefore re-generates the recording, exactly as for any unit. `nondet` occupies the single trailing `share_mod` slot (§{steps.cook-disposition}) and is therefore mutually exclusive with `local`/`pinned` at the surface — a unit is `local`, `pinned`, or `nondet`, not a combination. (The byte-equivalence waiver is conceptually independent of any sharing/fetch policy; the v3 surface that allowed combining them was collapsed to one slot in COOK-171, no change to the key model.)

The divergence oracle (`cook cache verify`) MUST NOT classify a content difference on a `nondet` step as an under-keyed determinant: the author has declared the output non-reproducible and owns determinant completeness through the step's effective seal set.

> **Note (informative).** The reference implementation realises the waiver as a boolean carried on the unit's cache metadata (the internal field name is unchanged from v3); at the cache decision a `nondet` unit does not schedule a present-but-content-drifted output for restore or rebuild. The cache schema version is unchanged. (CS-0109, CS-0115.)

### 17.1.5. Producer determinant manifest [#exec.cache.determinant-manifest]

An implementation that publishes a unit's output artifacts to the shared content-addressed store (§{exec.cache.sharing}) SHOULD persist, retrievable by the same single cache key K (§{exec.cache.single-key}), a **producer determinant manifest**: the resolved determinant values that formed K — the declared input contents (as path → content-hash), the command/Lua-chunk hash, the declared output paths, the consulted environment values, and the values of every probe in the unit's effective seal set. The manifest records the resolved *values* behind K's component digests, not the artifact bytes.

The manifest exists for diagnosis and verification: it lets a `cook why` query attribute a cache miss to the specific differing determinant, and it lets the divergence oracle (`cook cache verify`) confirm that a unit's recorded determinants recompose to K. A conforming manifest MUST serialise deterministically — the same K MUST yield byte-identical manifest content — which requires ordered (sorted-key) serialisation of every map.

The manifest is NOT an attestation of *which* producer made the artifact; cryptographic producer-identity attestation is out of scope at this revision. A `local` unit (§{steps.cook-disposition}) publishes no shared artifact and therefore no manifest. A missing or malformed manifest MUST NOT be treated as a cache error — it is best-effort diagnostic data, and an implementation that cannot read it falls back to its key-only behaviour.

> **Note (informative).** The reference implementation persists the manifest as a JSON sidecar adjacent to the artifact bytes, keyed by the unit's cloud key (one manifest per unit, shared across the unit's N output artifacts since they share K). Maps serialise as sorted-key JSON objects and `u64` digests as zero-padded lowercase hex strings, so the same key yields byte-identical content. It is exposed through the cache backend as `put_manifest` / `get_manifest` and is the data source for the `cook why` key explanation (§{exec.cache.why}) and the divergence verifier. (CS-0110.)

### 17.1.6. Key explanation (`cook why`) [#exec.cache.why]

To make author-declared determinant fidelity (§{exec.cache.abstract}) auditable, the reference implementation surfaces a read-only **key-explanation** query, `cook why <recipe>`. For every cacheable unit in the named recipe's reachable closure, it reports the unit's single cache key K (§{exec.cache.single-key}) and the resolved value of every determinant behind K: the declared input contents (as path → content-hash), the command/Lua-chunk hash, the declared output paths, the consulted environment values, and the value of every probe in the unit's effective seal set (§{exec.cache.single-key}). For each unit it also reports whether the unit would be served from cache — locally or from the shared store — or rebuilt.

The query MUST NOT execute any unit, MUST NOT publish any artifact, and MUST NOT write any declared output to the working tree; it is strictly an observation of the current context. Sealed probe values are read from their already-materialised canonical-JSON values (§{cat.probes.values}, §{cat.probes.exec}); a sealed probe whose value has not been materialised is reported as unmaterialised rather than produced. For a `tools` probe (§{cat.probes.inputs}), whose value is an object keyed by tool name with each entry recording the resolved binary's path and content hash, the explanation surfaces that resolved tool path.

On a **shared miss** — the unit is absent from the local index and the consumer cannot fetch its outputs from the shared store by K — when a producer determinant manifest (§{exec.cache.determinant-manifest}) exists for K, the query MUST diff the consumer's resolved determinants against the manifest and name the differing determinant(s), so that the cause of the miss is answerable from the query's output alone. A missing or malformed manifest MUST NOT be treated as an error (§{exec.cache.determinant-manifest}); the query reports that no producer manifest is published for K and falls back to reporting the consumer-side determinants only.

> **Note (informative).** The reference implementation renders one block per unit — status, key, the determinant values, and (on a shared miss) the producer-manifest diff — and also accepts `--json` for a deterministic machine-readable form: object keys sorted, `u64` digests as zero-padded lowercase hex strings, matching the manifest's serialisation convention (§{exec.cache.determinant-manifest}). The query is read-only: on a shared cache hit it confirms the backend holds every output by key without restoring the bytes to the working tree. (CS-0112.)

## 17.2. Cache integrity [#exec.cache.integrity]
An implementation that records output content in a content-addressable store — whether local, shared between machines, or remote — MUST treat that store as untrusted at restore time. When the implementation reads bytes from the store with the intent to install them as the on-disk content of a declared output, it MUST recompute the same content fingerprint that it recorded for that output and MUST compare the recomputed fingerprint to the recorded one. If the two fingerprints differ, the implementation MUST NOT install the bytes; it MUST treat the read as a cache miss and MUST fall through to the rebuild path defined in §{exec.cache.abstract} above.

This requirement holds for every backend, including a backend that resides on the same machine and the same filesystem as the workspace. The Standard does not distinguish "local" from "remote" backends in the cache contract; the integrity check is the same in both cases. An implementation MAY use any deterministic fingerprint that meets the observability constraint of §{exec.cache.abstract} (the fingerprint MUST be a function of the bytes being installed, with no dependence on wall-clock time, process identifiers, or other extrinsic state); it MUST use the same function it recorded the original artifact under. An implementation that exposes the artifact store to writers other than the local build (e.g., a multi-tenant shared backend) SHOULD use a cryptographically secure fingerprint (collision-resistant under adversarial input). The fingerprint verification required by this section MAY be performed as bytes flow through a streaming reader, with the verification surfacing as an end-of-stream error; an implementation is not required to materialise the full artifact in memory before verifying.

The intent is that an implementation's cache restore path MUST NOT be a vector for installing attacker-controlled or otherwise unauthenticated bytes into the workspace. A backend whose contents have drifted, been corrupted, or been replaced — for any reason — MUST be observationally indistinguishable from a backend that simply does not have the artifact.

An implementation's artifact-store write path MUST treat a `put` of bytes that conflict with bytes already stored at the same key as an error. "Conflict" means the bytes differ; re-writing identical bytes is idempotent and MUST succeed. The error MUST surface to the caller; the implementation MUST NOT silently overwrite the prior bytes. This requirement is the write-side analogue of the read-side fingerprint check: it ensures that a key in the artifact store maps to one and only one byte sequence over its lifetime, which is the invariant the read-side verification relies upon.

### Example 17.2.1

```cook
recipe build
    ingredients "src/main.c"
    cook "build/main.o" { gcc -c $<all> -o $<out> }
```

On the first run, `build/main.o`'s unit executes and its completion MAY be recorded. On a subsequent run with an unchanged `src/main.c` and unchanged command text, an implementation MAY skip the unit. If the command text changes or `src/main.c` changes, the unit MUST re-run.

### Note 17.2.1

The cache is optional. A conforming implementation that always re-executes every work unit — a clean-build implementation — is conforming so long as the dependency rules of §{exec.two-phase}–§{exec.interactive-drain} are respected.

## 17.3. Discovered inputs [#exec.cache.discovered-inputs]

A unit whose declaration carries `discovered_inputs` (§{lua.add-unit-discovered-inputs}) participates in post-execution input discovery. When such a unit is checked against an existing cache entry, a conforming implementation:

1. **MUST** attempt to read the file at `discovered_inputs.from` from the unit's working directory before composing the input-content set passed to the cache check. If the file is present and well-formed under `discovered_inputs.format`, the implementation MUST union the discovered paths with the unit's declared `inputs[]` for the purposes of the check.
2. **MUST**, when no restore context is available, treat a missing or malformed discovery file as no augmentation — the check proceeds with the thin (declared-only) input set, which causes an `InputSetChanged` rebuild that regenerates the discovery file. **MAY**, when a restore context is available, substitute the stored entry's recorded input set so the outputs walk and restore path can proceed; this avoids an avoidable rebuild when only the discovery file has been wiped from disk while the rest of the local entry is intact.
3. **MUST**, after a successful execution, parse the discovery file at `discovered_inputs.from` and amend the recorded `StepEntry`'s input set to include the discovered paths before persisting the entry to the local cache index.
4. **MUST** treat the discovery file as an implicit cache artifact: uploaded under its own artifact key during commit, restored from the backend on a hit-with-drifted-outputs check (§{exec.cache.integrity}).

The `"make"` discovery format is the Make rule format produced by GCC's `-MMD` flag and equivalent compiler options. The conforming parser MUST: strip the leading target text up to and including the first `:`; join continuation lines (`\\\n` and `\\\r\n`); ignore entries beginning with `/`; ignore the unit's first declared input; ignore entries whose paths do not exist on disk.

## 17.4. Test-unit caching [#exec.cache.test-unit]

A conforming implementation MAY cache the result of a `test_step` unit (§{steps.test}). When caching is implemented, the following rules MUST hold.

1. **Cache key inputs.** A test unit's cache key MUST be a content-addressed digest derived from at least the following inputs:
   - The unit's substituted command text. For shell-block tests this is the command string after sigil-placeholder substitution (§{phl.plate-test}); for Lua-block tests this is the Lua source body string with bindings resolved per §{lua.using-block-globals-plate-test}.
   - The content fingerprints of every cook-step output the test consumes via §{steps.iteration-mode-plate-test} source-list flattening.
   - The content fingerprints of every recipe output the test references via `cook.dep_output(NAME)` or `$<NAME>` cross-recipe substitution (§{xref.string-substitution}).
   - The deterministic env-key set as filtered through the implementation's `EnvDenylist` (the existing fingerprint mechanism for cook units per §{exec.cache.abstract}).
   - The values of every probe in the unit's effective seal set, on the same footing as the single cache key's probe-value fold (§{exec.cache.single-key}).
   - The `timeout` modifier value.
   - The `should_fail` modifier value.

   The cache key MUST NOT include the test's display name (§{steps.test}), the `suite` field (§{lua.cook-add-test}), the recipe name, or the step's source position. Renaming a test, moving it within a recipe, or moving a recipe between Cookfiles MUST NOT bust its cache.

2. **Recordable outcomes.** A conforming implementation MAY record a test result for replay only when the unit's outcome is **passed**. A test is **passed** when:
   - The body's exit status is success and `should_fail` is false; or
   - The body's exit status is failure and `should_fail` is true.

   A test that is **failed** (the dual of passed), **timed out**, or **blocked** (its build dependency failed) MUST NOT be cached. The implementation MAY discard a previously-cached entry whose corresponding test is observed to fail in a subsequent run.

3. **Replayed payload.** A cache hit MUST produce, for reporting purposes, the originally-recorded `(stdout, stderr, duration_secs, success)` quadruple. The duration value of a cache replay is the recorded original duration, not the wall-clock cost of the cache lookup.

4. **Cross-machine portability.** A test result recorded on machine A MUST be a cache hit for the same fingerprint on machine B, given identical inputs.

## 17.5. Cache portability invariants [#exec.cache.portability]

A conforming implementation that maintains a cache of recipe execution results MUST hold the following invariants for the cache to be portable across teammates and across workspace-relocation moves on disk.

1. **`cookfile_path` is workspace-root-relative.** For every cached recipe, the `cookfile_path` recorded in the cache metadata MUST be the importee's Cookfile path expressed as a forward-slash relative path from the workspace root, and MUST contain no absolute-path components. This invariant follows from the workspace shape established by §{comp.import} and §{exec.ws.determination}.

2. **`cache_key` uniqueness is per-Cookfile-cache-file scope.** A cache entry's identity within its index file is determined by its `cache_key` (typically derived from the unit's primary output path with optional variant suffix). The `cache_key` does not encode the recipe namespace; cross-Cookfile uniqueness is provided by **per-Cookfile cache locality** (each Cookfile maintains its own cache index file under its own directory).

3. **`cache_meta.input_paths` is importer-relative.** For a unit registered by recipe R in Cookfile C with working directory W, every entry in `cache_meta.input_paths` MUST be a path relative to W. This holds for both same-Cookfile inputs and cross-Cookfile dep inputs; substitution-time path rewriting (§{comp.qualified-refs}) ensures cross-Cookfile dep paths arrive in importer-relative form. For sigil-anchored imports the path can contain `..` segments; the cache check uses `W.join(input_path)` to locate the file on disk, and `..` segments resolve correctly against W.

A conforming implementation MAY layer additional content-addressed artifact storage atop the index files; that layer is outside the scope of this section.

Probe units (§{cat.probes}) participate in the same cache machinery: a probe's fingerprint addresses its canonical-JSON-serialised value (§{cat.probes.values}) in `CacheBackend`, and probe artifacts ride the same `events.jsonl` event stream and Cook Cloud wire protocol as file outputs.

### Note 17.5.1 — Chore parameters and caching (informative) [#exec.cache.chore-params]

A reader landing here looking for how chore parameters (§{chores.params}) interact with the cache should consult §{chores.no-caching} (§7.4): a chore body MUST NOT register a cached work unit, so chore parameters never enter the cache fingerprint. Recipes do not take parameters (§{recipes.header-forms}, §{chores.params}), so the parametric-cache-key question does not arise in the current Standard. This note exists so that searches for "parameter" within the cache chapter point readers at the right section.

## 17.6. Output globs [#exec.cache.output-globs]

A unit whose declared `outputs[]` (§{lua.add-unit-output-globs}) contains one or more glob pattern entries participates in **post-execute output resolution**. After the unit's command completes successfully, a conforming implementation:

1. **MUST** walk the unit's declared `outputs[]` once. For each literal path entry, the entry passes through unchanged. For each glob pattern entry, the implementation MUST resolve the pattern against the unit's working directory and substitute the resolved file set for the entry, dropping any directory matches (CS-0064). The combined resolved set is unioned across all entries; if a single literal path matches more than one entry (literal or glob), it MUST appear exactly once in the resulting set.
2. **MUST** compose the unit's `StepEntry.outputs` from the resulting fully-resolved literal-path set, in the same way it would for a unit whose `outputs[]` were originally all literal (§{exec.cache.abstract}).
3. **MUST** treat an empty resolved set as the unit having produced no outputs. A `StepEntry` MUST still be written so that the input fingerprint can short-circuit on subsequent runs; restoration on a hit MUST be a no-op for such an entry.
4. **MUST** upload each resolved output to the backend under the same artifact-key scheme used for literal outputs.

On a subsequent cache check for the same unit, the implementation MUST use the recorded `StepEntry.outputs` directly for the disk-integrity walk and for `try_restore` (§{exec.cache.integrity}). The implementation MUST NOT re-resolve the patterns against the current filesystem during the check. The recorded file set is the source of truth for "what the unit produced last time it ran"; cache invalidation flows through the unit's input fingerprint, not through pattern re-resolution.

### Note 17.6.1

The resolved file set persisted in `StepEntry.outputs` is indistinguishable from the set that would have been persisted by a unit whose `outputs[]` were literal from the start. Future cache reads, cross-machine restores, and artifact lookups operate on the literal paths only; the original glob pattern is not recorded.

## 17.7. Stale-output reconciliation [#exec.cache.reconcile]

§17.1 establishes that the cache changes only *how much* work happens, never *what it produces*. That guarantee extends from cache *values* to the **output tree itself**: when a recipe's declared output set *shrinks* between runs, the outputs it no longer declares MUST NOT be left behind on disk. A build whose output directory accumulates artifacts that nothing declares misrepresents its own graph — a downstream glob (§17.6), `tar`, or deploy step would pick up the stale file.

The canonical case is `ingredients <probe>` (§{cat.probes.for-each}, §{steps.ingredients}), whose output set is *computed from data*: removing a data member means the next run correctly produces the remaining members but would otherwise orphan the dropped member's artifact.

### The rule

For each recipe **R reached and re-registered** in an invocation, after R's work units complete, a conforming implementation MUST compute

> `orphans(R) = prior_declared_outputs(R) − current_declared_outputs(R)`

where the *prior declared outputs* are the output paths R recorded on its last successful run (the cache snapshot, §17.5, read before it is overwritten) and the *current declared outputs* are the union of the output paths of every work unit R registered this run (after `ingredients <probe>` fan-out). For each path `p` in `orphans(R)`:

- if `p` is **not present** on disk, the implementation MUST take no action;
- if `p` is present **and** its current content hash equals the hash Cook recorded when it wrote `p`, the implementation MUST remove `p` and report it (e.g. `swept orphaned output: <p>`);
- if `p` is present but its content hash **differs** from the recorded hash, the implementation MUST keep `p` and SHOULD warn (e.g. `<p> changed since Cook wrote it — not removing`).

R's recorded declared-output set then advances to the current set.

### Scope firewall

Reconciliation MUST run **only** for recipes actually reached in the current invocation. A targeted `cook <recipe>` MUST NOT sweep any other recipe's outputs; recipes that did not run keep their recorded outputs untouched. An orphan of R that is a **current** declared output of some other recipe reached this run MUST NOT be removed — it is still live.

### Safety properties

A conforming implementation:

- MUST only ever delete files it itself wrote and recorded (a tracked-set diff; it MUST NOT sweep a file it did not create);
- MUST NOT delete a file whose content changed since Cook wrote it (the hash guard above);
- on a first run, or against a cache with no recorded outputs, MUST compute an empty prior set and therefore delete nothing;
- reconciles **files only** in this edition — directories are left in place.

`plate` and `test` steps declare no outputs and are no-ops for reconciliation.

### Opt-out

Reconciliation is default-on. An implementation MUST provide an opt-out (the reference implementation: `--no-prune`, or the `COOK_NO_PRUNE` environment variable) that disables the sweep for an invocation; with it set, orphaned outputs are retained.

> **Note (informative).** Reconciliation reuses the output paths and content hashes the cache already records (§17.1, §17.6) — there is no new on-disk ledger — and is guarded so a user-modified artifact is always kept. It is the output-file analogue of the integrity rule of §17.2: Cook never silently leaves stale bytes behind, whether in a restored artifact or an orphaned output.

## 17.8. Determinant-fidelity verification (`cook cache verify`) [#exec.cache.verify]

The divergence oracle is surfaced by the reference implementation as the opt-in
command `cook cache verify`. It is a **fidelity / diagnostic tool, not a trust
gate**: trust rests on author-declared determinants (§17.1.1), and this command
*observes* whether those declarations are complete. An implementation is NOT
required to run it to be conformant; when run, it MUST behave as follows.

For each cacheable unit (§17.1.1) the cache holds an artifact for under its key
K, the verifier re-executes the unit in the current context and compares the
re-produced output bytes against the bytes the cache would serve for K:

- If the unit is **not** a `nondet` unit (§17.1.4) and a re-produced output's
  bytes differ from the cached bytes under the matching key K — or a recorded
  output is not re-produced at all — the verifier MUST report a **divergence**.
  A divergence is either an under-keyed determinant (a real determinant was not
  declared, so K failed to distinguish two contexts) or a non-reproducible
  producer. The command MUST exit non-zero when any divergence is reported.
- If the unit **is** a `nondet` unit, the verifier MUST NOT perform the byte
  comparison: byte-equivalence is waived for `nondet` (§17.1.4). It MAY apply a
  weaker check — that each recorded output is re-produced — and MUST NOT
  classify a `nondet` content difference as a divergence.
- A faithfully-keyed, reproducible unit re-produces byte-identical outputs and
  MUST be reported as passing.

Where a producer determinant manifest (§17.1.5) is retrievable for K, the
implementation SHOULD attach its recorded determinants (consulted environment,
sealed probe values, declared inputs) to a divergence report to aid
attribution. Manifest absence MUST NOT change the byte verdict.

> **Note (informative).** Running the verifier across machines (e.g. in CI on a
> host distinct from the producer) is the intended deployment: a unit that
> shares fleet-wide yet embeds an undeclared host determinant surfaces as a
> divergence there. Running it twice on one machine against the same workspace
> is the local degenerate case — K matches by construction, so any divergence
> is purely a producer/determinant property. (CS-0113.)
