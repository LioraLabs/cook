Pins CS-0149 / Standard §22.9: `cook.on_register_complete` callbacks run exactly once, after every recipe body of the pass has registered, in the order they were queued.

`cook_modules/m.lua` accumulates state across the two recipe bodies (`alpha`, `beta` — each a bare `module_call` per §{steps.dispatch} rule 6) into a module-local table, then a top-level `register` block calls `m.arm()`, which queues two callbacks via `cook.on_register_complete`. The top-level chunk (and therefore `m.arm()`) runs before any recipe body is invoked (§22.6's discovery/dispatch split), so this fixture cannot pass by accident of call-site ordering — the callbacks can only observe both marks because the finalizer queue drains strictly after the body-invocation loop, not because `m.arm()` happened to run late.

The first callback asserts both `marks["alpha"]` and `marks["beta"]` are present, raising a distinctive error naming whichever is missing otherwise; this is the run-after-all-bodies guarantee. The second callback asserts a flag the first callback sets; this can only be true if the first callback already ran, pinning registration order across separate `cook.on_register_complete` calls. If either guarantee regressed — a callback firing before both bodies register, or firing out of queue order — one of these two `error()` calls would fire and the fixture would report a register failure instead of `register_ok.txt`'s success. There is no assertion visible from the Cookfile surface itself: this fixture's only observable is register succeeding at all.

Consumed by `cli/crates/cook-register/tests/conformance.rs`'s `register_positive_conformance_corpus` (via the `register_ok.txt` marker, which parses, codegens, and runs the case through `register_cookfile` end to end) and by the parser-only harness in `cli/crates/cook-lang/tests/conformance.rs` (AST shape only, via `parse.txt`).
