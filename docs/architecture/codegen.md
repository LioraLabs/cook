# Codegen: AST-to-Lua Transpilation

## Overview

Codegen transforms a `Cookfile` AST into a Lua source string that the runtime loads and executes. It lives in the `cook-luagen` crate under `cli/crates/cook-luagen/`. The public surface is re-exported from `cli/crates/cook-luagen/src/lib.rs:15-18`:

| Entry point | Use |
|---|---|
| `generate(&Cookfile) -> String` | Lower a whole Cookfile; panics on codegen-phase validation errors. |
| `generate_with_names(&Cookfile, &BTreeSet<String>) -> Result<String, CodegenError>` | Lower with an explicit set of in-scope recipe names (needed for cross-recipe resolution). |
| `generate_with_names_checked(...)` | As above, but runs the placeholder validator first (`cli/crates/cook-luagen/src/recipe.rs:81`). |
| `generate_with_names_and_warnings(...)` | As `generate_with_names`, plus an extra pass that warns when a `$<NAME>` reference names a recipe with empty outputs. |
| `compile_chore(&Chore, &[UseStatement]) -> String` | Lower a single chore. Recipes call this internally for the `chores` section; external callers (the CLI's chore-runner path) use it directly. |

The output is a sequence of `cook.recipe(...)` calls (one per recipe and per chore), preceded by `cook.load_module(...)` locals for each `use` statement and, when present, a `__cook_run_config_blocks` dispatcher function. The runtime loads this string into a Lua VM and executes each recipe body in *register* phase, where every step lowers into one or more `cook.add_unit({...})` or `cook.add_test({...})` calls. See `runtime.md` for what happens after.

This document covers transformation rules only â€” what shape of AST node produces what Lua text. Step-kind semantics and surface syntax belong to `parser.md`.

## Top-level Shape

`generate_with_names` (`cli/crates/cook-luagen/src/recipe.rs:598`) emits, in order:

1. A header comment: `-- Generated by Cook\n`.
2. One `local <alias> = cook.load_module("<name>")` per `use` statement (`recipe.rs:605-615`). Hyphens in module names are replaced with underscores in the local alias.
3. A `__cook_run_config_blocks(selected_name)` Lua function if the Cookfile has any config blocks (`recipe.rs:618-662`). Unnamed (base) blocks run unconditionally; named blocks run inside an `if selected_name == "NAME" then ... end` arm. Source lines starting with `#` are skipped â€” they are Cookfile comments, not Lua.
4. One `cook.recipe("name", METADATA, function() ... end)` per recipe (`recipe.rs:664-804`).
5. One `cook.recipe(...)` per chore, wrapped with `cook._enter_chore()` / `cook._exit_chore()` markers (`recipe.rs:850-940`).

### Recipe metadata table

`generate_metadata` (`recipe.rs:1007`) emits a Lua table literal containing only the fields that are non-empty:

| Recipe field | Lua key | Shape |
|---|---|---|
| `ingredients` | `ingredients` | `{"glob1", "glob2"}` |
| `excludes` | `excludes` | `{"pat1", "pat2"}` |
| `deps` | `requires` | `{"recipe1"}` |

An empty metadata table is `{}`. Examples:

```lua
cook.recipe("clean", {}, function() ... end)
cook.recipe("compile", {ingredients = {"src/*.c"}}, function() ... end)
cook.recipe("build", {ingredients = {"src/*.c"}, excludes = {"src/_*"}, requires = {"deps"}}, function() ... end)
```

### Local `ingredients`

If the recipe declares any ingredients, the recipe body opens with (`recipe.rs:683-688`):

```lua
    local ingredients = cook.resolve_ingredients({"<includes>"}, {"<excludes>"})
```

This local is the Standard Â§4.3 *resolved ingredient set* â€” the union of include globs minus the union of excludes, flattened to a single array. It is the iteration source for the first cook step (when no prior cook step exists) and for plate/test steps that fall back through to ingredients.

The unrelated table-of-tables `recipe.ingredients[N]` (per-pattern groups) is still available to Lua bodies via the `recipe` global, but codegen never iterates it directly â€” using `recipe.ingredients[1]` would silently drop every glob past the first.

## Step Driver Loop

Inside a recipe body, `recipe.rs:690-801` walks `recipe.steps` with two pieces of bookkeeping:

- `cook_index` â€” incremented for each `Step::Cook` encountered. Each cook step's outputs are written into a local `_cook_outputs_N` where `N = cook_index`.
- `prev_cook_index` â€” last `cook_index` seen, used by the next cook / plate / test step to find its input source (`_cook_outputs_PREV`).

Step kinds dispatch as follows:

| AST `Step` variant | Emission site | Lua surface emitted |
|---|---|---|
| `InlineLua` / `InlineLuaBlock` | `recipe.rs:696-708` | Verbatim register-phase Lua, four-space indented. |
| `Cook` | `recipe.rs:709-728` | `_cook_outputs_N = {}` + a `cook.step_group(function() ... end)` wrapping the per-mode body. |
| `Plate` | `recipe.rs:729-744` | `cook.step_group(function() generate_plate_step(...) end)`. |
| `Test` | `recipe.rs:745-760` | `cook.step_group(function() generate_test_step(...) end)`. |
| `Shell { interactive: true }` | `recipe.rs:761-775` | A single `cook.add_unit({command = ..., interactive = true, line = N, cache = false})`. |
| `Shell { interactive: false }` / `Lua` / `LuaBlock` | `recipe.rs:776-790` | Coalesced into one *body unit* â€” see [Body Bundling](#body-bundling). |

`cook.step_group` is the runtime hook that defines parallelism boundaries: every `cook.add_unit` call inside the same group can run concurrently with the others (they share no file dependencies), and the next group cannot start until the current one drains.

## Cook Step Emission (`cook_step.rs`)

`generate_cook_step` (`cli/crates/cook-luagen/src/cook_step.rs:106`) inspects the `CookStep`'s output patterns and its `using_clause` to pick a `CookMode`, then emits one of five shapes.

### Mode selection

`cook_step_mode_with_names` (`cook_step.rs:45`) classifies the step by analyzing every output pattern through `output_pattern_kind_with_recipes` (`cli/crates/cook-luagen/src/template.rs:164`):

| Pattern shape | `OutputPatternKind` |
|---|---|
| Fully literal (no `$<in.X>`, no `$<RECIPE.X>`) | `Literal` |
| Contains `$<in>` or `$<in.ACCESSOR>` | `OwnInputAccessor` |
| Contains `$<RECIPE.ACCESSOR>` for an in-scope recipe | `DepDriven { dep_name, lua_expr }` |

| `using_clause` | Output count | Pattern(s) | Mode |
|---|---|---|---|
| `None` | any | any | `DeclarationOnly` |
| `Some(_)` | 1 | `Literal` | `ManyToOne` |
| `Some(_)` | 1 | `OwnInputAccessor` or `DepDriven` | `OneToOne` |
| `Some(_)` | â‰Ą 2 | all `Literal` | `BlockStep` |
| `Some(_)` | â‰Ą 2 | any `OwnInputAccessor` or `DepDriven` | `OneToMany` |

Output-pattern coherence is enforced separately by `check_multi_output_coherence` (`recipe.rs:126`): in a multi-output cook step, every pattern must share the same driver (all literal, all `$<in.X>`, or all `$<RECIPE.X>` for the *same* recipe). Mixing drivers is rejected with a `PlaceholderViolation`.

### Input source

The cook step's input list (`cook_step.rs:128-134`) is determined by:

1. The previous cook step's outputs (`_cook_outputs_<prev>`), if any.
2. Otherwise the resolved `ingredients` local, if the recipe has ingredients.
3. Otherwise an empty Lua table `{}`.

For `OneToOne` / `OneToMany` over a `DepDriven` pattern, the iteration source is replaced by `cook.dep_output_list("<dep>")` (`cook_step.rs:148-149, 248-249`) â€” a runtime helper that returns the named recipe's output list.

### DeclarationOnly

`using_clause: None` â€” just record the literal output path. No work, no `cook.add_unit`.

```text
cook "bin/app"
```

```lua
    _cook_outputs_1[1] = "bin/app"
```

### OneToOne

One output per input. Emits a `for` loop. Inside the loop body, `_cook_in` is bound to the current iteration item and `_cook_out` is the expanded output pattern.

```text
ingredients "src/*.c"
cook "build/$<in.stem>.o" { gcc -c $<in> -o $<out> }
```

```lua
    local _cook_outputs_1 = {}
    cook.step_group(function()
    for _, _cook_in in ipairs(ingredients) do
        local _cook_out = "build/" .. path.stem(_cook_in) .. ".o"
        cook.add_unit({inputs = {_cook_in}, output = _cook_out, command = "set -e\ngcc -c " .. _cook_in .. " -o " .. _cook_out, consulted_env_keys = {}})
        table.insert(_cook_outputs_1, _cook_out)
    end
    end)
```

For a dep-driven output pattern (e.g. `build/$<protos.stem>.o`), the loop iterates `cook.dep_output_list("protos")` instead of `ingredients`, and inside the body `$<protos.stem>` lowers to `path.stem(_cook_in)` (`template.rs:294-307`) â€” dep-driven iteration normalizes the dep output to `_cook_in`.

When the `using_clause` is a `LuaBlock` (`cook_step.rs:182-189`), the emitted unit carries `lua_code = [[<source>]]` and an `ingredient_groups` field populated with `{recipe.ingredients[1], recipe.ingredients[2], ...}` â€” the per-pattern groups, which the worker exposes as `input_1`, `input_2`, etc.:

```lua
        cook.add_unit({inputs = {_cook_in}, output = _cook_out, lua_code = [[<USER LUA>]], ingredient_groups = {recipe.ingredients[1]}, consulted_env_keys = "*"})
```

### ManyToOne

Single literal output, single invocation that consumes the whole input list.

```text
cook "build/lib.a" { ar rcs $<out> $<all> }
```

```lua
    local _cook_outputs_2 = {}
    cook.step_group(function()
    local _cook_all = table.concat(_cook_outputs_1, " ")
    local _cook_out = "build/lib.a"
    cook.add_unit({inputs = _cook_outputs_1, output = _cook_out, command = "set -e\nar rcs " .. _cook_out .. " " .. _cook_all, consulted_env_keys = {}})
    table.insert(_cook_outputs_2, _cook_out)
    end)
```

The full input list is passed as `inputs = <input_source>`. `_cook_all` is precomputed because `$<all>` lowers to it directly.

### OneToMany

One iteration item produces N outputs. Pattern is a multi-output cook step where at least one output uses `$<in.X>` or `$<dep.X>`. All patterns must share the same driver (per coherence check above).

```text
cook "build/$<in.stem>.js" "build/$<in.stem>.wasm" { gen $<in> }
```

```lua
    for _, _cook_in in ipairs(ingredients) do
        local _cook_outs = {
            "build/" .. path.stem(_cook_in) .. ".js",
            "build/" .. path.stem(_cook_in) .. ".wasm",
        };
        cook.add_unit({inputs = {_cook_in}, outputs = _cook_outs, command = "set -e\ngen " .. _cook_in, consulted_env_keys = {}})
        table.insert(_cook_outputs_N, _cook_outs[1])
    end
```

Only the first declared output is forwarded into `_cook_outputs_N` (cook step chaining looks at one stream). The downstream Standard Â§5.5 `$<RECIPE>` reference resolves to that primary stream; multi-output access from another step is via `cook.dep_output_list("<dep>")` plus accessors.

### BlockStep

Multi-output cook step where every pattern is literal. One invocation produces all declared outputs; they share a cache entry and are produced atomically.

```text
cook "out/parser.rs" "out/parser.h" { lalrpop src/grammar.lalrpop --out-dir out }
```

```lua
    local _cook_outs = {"out/parser.rs", "out/parser.h"};
    local _cook_ins = _cook_outputs_0;  -- or `ingredients`
    local _cook_all = table.concat(_cook_outputs_0, " ");
    cook.add_unit({inputs = _cook_ins, outputs = _cook_outs, command = "set -e\nlalrpop src/grammar.lalrpop --out-dir out", consulted_env_keys = {}})
    table.insert(_cook_outputs_N, _cook_outs[1])
    table.insert(_cook_outputs_N, _cook_outs[2])
```

All declared outputs flow into `_cook_outputs_N`, so subsequent cook steps see every produced file.

### Shell-block joining

When `using_clause` is `ShellBlock(lines)`, codegen prepends `set -e` and joins all lines with `\n` before substitution (`cook_step.rs:79-86`). This matches the *per-line halt-on-failure* semantic of POSIX `sh -e` and ensures multi-line build blocks fail at the first failing command rather than running on through.

The joined text is then template-expanded via `expand_template_to_lua_with_deps_tracked` (`template.rs:667`); the result becomes the `command =` field of the emitted `cook.add_unit` call.

### `consulted_env_keys`

Every shell-driven cook unit carries `consulted_env_keys = {...}` â€” a sorted Lua table of every env var name that appears as `$<NAME>` (or `$<env.NAME>`) in the resolved command. Codegen collects these into a `ConsultedEnv` accumulator (`template.rs:17-44`) as it walks the template. The runtime hashes these values into the cache key so that changing `$CFLAGS` invalidates affected units without forcing a global rebuild.

LuaBlock cook units use `consulted_env_keys = "*"` instead â€” a wildcard meaning "the env contract is opaque to codegen; assume any env access". The runtime treats `"*"` as "hash the whole consulted-env snapshot".

## Plate Step Emission (`plate_step.rs`)

A `plate` step runs side-effects with no declared outputs (per Standard Â§4.7.1, its output is a *passthrough* of its input list). Implemented in `cli/crates/cook-luagen/src/plate_step.rs:11`.

### Mode detection

`detect_plate_test_mode` (`template.rs:355`) inspects the body for which iteration form it uses:

| Body content (Shell) | Body content (Lua) | Mode |
|---|---|---|
| `$<in>` or `$<in.ACCESSOR>` | free identifier `input` | `OneToOne` |
| `$<all>` | free identifier `inputs` | `ManyToOne` |
| neither | neither | `OneShot` |
| both | both | error (`Mixed`) |

The Lua free-identifier scan (`template.rs:413`) skips string literals (`"..."`, `'...'`, `[[...]]`), comments, and field accesses (`x.input`, `x:input`) â€” only bona-fide variable references count.

### Input source

`plate_step.rs:40-44`: the previous cook step's outputs (`_cook_outputs_N`), else the resolved `ingredients` local. A `OneToOne` or `ManyToOne` plate with no source at all is rejected with `EmptySource` (`plate_step.rs:27-32`) â€” Standard CS-0024 Â§3.5.

### Six emission shapes

`plate_step.rs:46-118` matches on `(body kind, mode)`:

| Body | Mode | Shape |
|---|---|---|
| Shell | OneToOne | `for _, _plate_in in ipairs(<src>) do cook.add_unit({command = ..., cache = false, ...}) end` |
| Shell | ManyToOne | One `cook.add_unit({command = ..., cache = false, ...})`, no loop. |
| Shell | OneShot | One `cook.add_unit({command = ..., cache = false, ...})`, no source binding. |
| Lua | OneToOne | Loop; each unit's `lua_code` is built at register time by prepending `local input = <value>\n` to the body literal via `string.format("%q", _plate_in)`. |
| Lua | ManyToOne | One unit; `lua_code` is built by an IIFE that serializes the source list into a `local inputs = {...}` header. |
| Lua | OneShot | One unit; `lua_code` is the body literal as-is. |

Plate units always carry `cache = false` (plates have no outputs to cache against) and `step_kind = "plate"` for Lua bodies (`plate_step.rs:95`) â€” the latter tells the worker to skip the project-root sandbox, since plates are the explicit ship-outside-the-project surface.

After the unit emission, plate steps emit `cook.passthrough(<src>)` (`plate_step.rs:127-129`) if there is a source â€” this propagates the input list as the step's output for downstream `$<RECIPE>` references.

### Body expansion (`expand_plate_test_body`)

`template.rs:607` walks `$<...>` spans in the plate body and lowers each:

- `$<in>` â†’ the iter variable (e.g. `_plate_in`).
- `$<in.ACCESSOR>` â†’ `path.<ACCESSOR>(<iter_var>)`.
- `$<all>` â†’ `table.concat(<all_var>, " ")`.
- `$<RECIPE>` (in-scope name) â†’ `cook.dep_output("<RECIPE>")`.
- Anything else (including `$<env.X>`) â†’ `cook.require_env("<key>")`, with the key recorded in `consulted_env_keys`.

Several forms are *rejected* in plate / test bodies by `validate_plate_test_placeholders` (`template.rs:526`):

- `$<out>`, `$<out_N>`, `$<out.X>`, `$<out_N.X>` â€” plate / test have no outputs.
- `$<in>` / `$<in.X>` outside `OneToOne` mode.
- `$<all>` outside `ManyToOne` mode.
- Bare path accessors `$<stem>` / `$<name>` / `$<ext>` / `$<dir>`.
- `$<NAME.ACCESSOR>` for an in-scope recipe â€” the Standard Â§5.4 firewall: plate / test have no output pattern to declare a driver, so dep-accessor placeholders cannot iterate here.

## Test Step Emission (`test_step.rs`)

A `test` step is the same iteration shape as a `plate`, but emits `cook.add_test({...})` instead of `cook.add_unit({...})` and carries test-specific metadata. Implemented in `cli/crates/cook-luagen/src/test_step.rs:12`.

The mode detection, input source, validation, and body expansion are all shared with plate via `detect_plate_test_mode` and `expand_plate_test_body`. Per-unit fields added on top:

| Field | Value | Source |
|---|---|---|
| `timeout` | `test_step.timeout` (default `300` seconds) | `test_step.rs:44` |
| `should_fail` | `true` / `false` from `should_fail` flag | `test_step.rs:45` |
| `line` | 1-indexed source line | `test_step.rs:71,81,â€¦` |
| `iteration_item` | `_test_in` (OneToOne) or `nil` (other modes) | per-arm |
| `name` | the expanded `as_name` expression, if the source had `as "..."` | `test_step.rs:51-62` |

The `as_name` is itself a template â€” it goes through the same sigil expander with mode-appropriate iter and all bindings (`test_step.rs:51-62`).

Like plate, test steps emit `cook.passthrough(<src>)` afterward when there is a meaningful source to forward.

## Sigil Scanner (`sigil.rs`)

Templates use the strict `$<IDENT>` placeholder form. `cli/crates/cook-luagen/src/sigil.rs:30` (`scan`) returns the byte range and ident of every well-formed placeholder. The grammar:

```
IDENT     := ALPHA (ALPHA | DIGIT | "_" | ".")*
ALPHA     := "a"â€¦"z" | "A"â€¦"Z" | "_"
```

Strict bail: a `$<` that does not close cleanly is *literal shell text*, not an error. The scanner never searches forward past a malformed inner â€” `$<foo bar> baz` contains zero placeholders, not one.

Common shell idioms that pass through verbatim because they do not start with `$<`:

- Brace expansion: `{a,b,c}`
- Shell parameter expansion: `${HOME:-default}`
- AWK programs: `awk '{print $1}'`
- Bare `$VAR`: `$HOME`, `$1`

Only `$<...>` is a Cook placeholder.

## Resolver (`resolver.rs`)

`cli/crates/cook-luagen/src/resolver.rs:84` (`resolve`) is the closed-set classifier. Given an `IDENT` and a `ResolveCtx` (current `IterMode`, current `OutputShape`, set of in-scope recipe names), it returns one of:

| `Resolved` variant | Meaning |
|---|---|
| `Builtin(BuiltinKind)` | `in`, `in.ACCESSOR`, `out`, `out.ACCESSOR`, `out_N`, `out_N.ACCESSOR`, `all`. Validated against mode and output count. |
| `Recipe { name, accessor }` | Whole ident, or its prefix-before-last-dot, is an in-scope recipe and any suffix is a valid accessor. |
| `EnvRuntime(key)` | Fallthrough â€” lower to `cook.require_env("key")` at runtime. `env.` prefix is stripped. |
| `Error(ResolveError)` | Builtin shape but invalid for current mode/output count, or malformed `out_0`. |

The validated accessor set (`stem`, `name`, `ext`, `dir`) lives in `cook-contracts` as `ACCESSORS` (`cli/crates/cook-contracts/src/lib.rs:15`). Both the resolver and `dep_ref` check against this single source. Adding an accessor means changing the contract crate; codegen picks it up automatically.

Mode and output-count rules:

| Builtin | Allowed mode | Allowed output shape |
|---|---|---|
| `in`, `in.X` | `OneToOne` | any |
| `out`, `out.X` | any | `Single` |
| `out_N`, `out_N.X` | any | `Multi(M)` with `N â‰¤ M` |
| `all` | `ManyToOne` | any |

Special note on shadowing: a bare ident that matches both an in-scope recipe and an env-var name resolves to the *recipe* (recipes win â€” `resolver.rs:92-94`). To force an env lookup that would otherwise be shadowed, use the explicit `$<env.NAME>` form (`resolver.rs:106-108`).

## Cross-recipe References (`dep_ref.rs`)

Cook used to express cross-recipe dependencies via runtime ingredient-serves matching â€” recipe A would consume recipe B's outputs implicitly through a shared file pattern. That model is gone. Today, dependencies are *explicit* placeholders resolved at codegen time, and the dep edge is recorded into the recipe's `requires` set.

`cli/crates/cook-luagen/src/dep_ref.rs:49` (`extract_dep_refs`) walks every step in a recipe and returns a sorted set of `DepRef { recipe_name, accessor }` for every `$<NAME>` / `$<NAME.ACCESSOR>` it finds. The classification rules in `parse_dep_token` (`dep_ref.rs:126`), evaluated in order:

1. Skip builtins (`in`, `out`, `all`).
2. Skip `env.` prefix â€” always env, never recipe.
3. Skip CS-0022 own-input / output forms: `in.X`, `out.X`, `out_N`, `out_N.X`.
4. Whole token is an in-scope recipe â†’ `DepRef` with no accessor.
5. Split on the *last* `.`: if suffix is a valid accessor and prefix is an in-scope recipe â†’ `DepRef` with that accessor.
6. Otherwise â†’ not a dep (an env var, an unknown ident).

The scan walks cook output patterns, cook shell-block bodies, plate / test shell bodies, and bare interactive shell commands. Lua bodies do not participate in `$<...>` substitution â€” cross-recipe access from Lua is via the `cook.dep_output()` API at runtime â€” so `extract_body_tokens` returns an empty list for `Body::LuaBlock`.

Once resolved, dep refs lower to:

- `$<RECIPE>` â†’ `cook.dep_output("RECIPE")` â€” runtime helper returning the recipe's primary output.
- `$<RECIPE.ACCESSOR>` â†’ `path.ACCESSOR(cook.dep_output("RECIPE"))` â€” same, post-processed.

For `DepDriven` output patterns (e.g. `build/$<protos.stem>.o`), codegen *iterates* over `cook.dep_output_list("protos")` and rewrites `$<protos.stem>` inside the output pattern to `path.stem(_cook_in)` â€” see [Cook Step Emission](#cook-step-emission-cook_steprs).

### Recipe-name scope

`extract_recipe_names` (`dep_ref.rs:22`) returns the local Cookfile's recipe names. `extract_recipe_names_with_imports` (`dep_ref.rs:34`) extends that set with `<alias>.<name>` for every recipe in every aliased import (Standard Â§7.3). The expanded set is what callers pass into `generate_with_names` so that imported recipes resolve correctly.

## Template Expansion (`template.rs`)

Three entry points cover the three template surfaces:

| Function | Surface | Output |
|---|---|---|
| `expand_sigil_template` (`template.rs:57`) | cook-step shell body, bare-shell command | A Lua expression. Errors on bad mode/count placeholders. |
| `expand_output_pattern` / `expand_sigil_output_pattern` (`template.rs:235, 249`) | cook output pattern | A Lua expression. Unknown idents fall through to `cook.require_env(...)` (relaxed mode). |
| `expand_plate_test_body` (`template.rs:607`) | plate / test body | A Lua expression. Reuses `iter_var` / `all_var` bindings (e.g. `_plate_in` / `source_expr`). |

All three follow the same emit pattern: walk `sigil::scan(template)` spans in order, interleaving literal segments (double-quoted, escaped) with resolved expressions (raw Lua), joined with ` .. `. The literal pre-quote uses `escape_lua_string`. A template with no placeholders becomes a single `"..."` literal; a template with exactly one span becomes a bare expression with no concatenation; the empty string becomes `""`.

### Builtin lowerings

```text
$<in>          â†’ _cook_in
$<in.stem>     â†’ path.stem(_cook_in)
$<out>         â†’ _cook_out
$<out.dir>     â†’ path.dir(_cook_out)
$<out_1>       â†’ _cook_outs[1]
$<out_2.ext>   â†’ path.ext(_cook_outs[2])
$<all>         â†’ _cook_all      (cook step)
$<all>         â†’ table.concat(<source>, " ")   (plate/test ManyToOne)
```

### Recipe and env lowerings

```text
$<libmath>      â†’ cook.dep_output("libmath")
$<libmath.stem> â†’ path.stem(cook.dep_output("libmath"))
$<HOME>         â†’ cook.require_env("HOME")    ; HOME recorded in consulted_env_keys
$<env.HOME>     â†’ cook.require_env("HOME")    ; forced env lookup, even if HOME is a recipe
```

### Worked example

```text
ingredients "src/*.c"
cook "build/$<in.stem>.o" { $<CC> $<CFLAGS> -c $<in> -o $<out> }
```

Step body is joined as `set -e\n$<CC> $<CFLAGS> -c $<in> -o $<out>` and lowered to:

```lua
"set -e\n" .. cook.require_env("CC") .. " " .. cook.require_env("CFLAGS") .. " -c " .. _cook_in .. " -o " .. _cook_out
```

Codegen records `consulted_env_keys = {"CC", "CFLAGS"}` (sorted, deduplicated by `BTreeSet`).

## Body Bundling (non-cook steps)

A run of *bundleable* steps â€” non-interactive `Shell` and any execute-phase `Lua` / `LuaBlock` â€” coalesces into one `cook.add_unit({lua_code = ..., cache = false})` so they share a single worker invocation. `is_bundleable` (`recipe.rs:345`) decides what bundles; an interactive shell, a cook / plate / test step, or any non-bundleable kind always breaks the run.

`emit_body_unit_with_names` (`recipe.rs:394`) builds the chunk:

- Adjacent raw shell lines (no `$<...>` in the command) coalesce into one `io.write(cook.sh([[set -e\n<line1>\n<line2>\n...]]))` call.
- A shell line *with* `$<...>` sigils is split out as a *register-time shell command piece*: codegen emits a Lua expression (e.g. `"echo " .. cook.dep_output("greet")`) that resolves on the register VM, then the result is baked back into the worker's chunk via `string.format("%q", <expr>)` wrapped in `io.write(cook.sh(...))`. This is the only way `$<NAME>` recipe refs in a bare shell line can work â€” `cook.dep_output` and `cook.require_env` exist on the register VM, not the worker VM.
- Lua and Lua-block steps emit verbatim into the chunk.
- Every `use` statement is re-emitted as a `local <alias> = cook.load_module("<name>")` at the top of the chunk so the worker can resolve them.

`render_chunk_pieces` (`recipe.rs:515`) picks the cheapest output shape: all-static bundles become one long-string literal; mixed bundles become a concatenation.

Interactive shell steps (`Shell { interactive: true }`) emit their own `cook.add_unit({..., interactive = true, cache = false})` and never bundle. Per Standard Â§exec.interactive-drain, an interactive shell pauses the scheduler until it drains.

## Chores (`compile_chore`)

A *chore* is a recipe that always runs interactively and never caches. `compile_chore` (`recipe.rs:850`) emits the same `cook.recipe(...)` shape as a recipe, with two specializations:

1. Every `cook.add_unit` carries `cache = false` (Standard Â§chores.no-caching).
2. Every shell step is `interactive = true`. The parser already enforces this; codegen passes it through and emits an unreachable defensive arm for safety.

Chore bodies are wrapped in `cook._enter_chore()` / `cook._exit_chore()` calls (`recipe.rs:872, 936`). The runtime uses these brackets to validate that no caching call slips through.

Cook / plate / test step kinds are banned in chores by the parser; the codegen match arm falls through silently for them.

`emit_chore_body_unit` (`recipe.rs:946`) is a simplified `emit_body_unit_with_names` for chore Lua bundles â€” no sigil-aware shell handling is needed because all chore shells are interactive (and therefore non-bundleable), so a bundle is always pure Lua.

## String Escaping (`lua_string.rs`)

Two helpers handle the two contexts:

### `escape_lua_string` (`lua_string.rs:1`)

For double-quoted Lua strings. Replaces, in order:

1. `\` â†’ `\\`
2. `"` â†’ `\"`
3. `\n` â†’ `\n` (escape sequence)

Used for recipe names, env var names in `cook.require_env(...)`, the `consulted_env_keys` table entries, and the literal segments of template expansion.

### `wrap_lua_string` (`lua_string.rs:44`)

For shell command bodies and Lua source chunks where double-quote escaping would interfere with the embedded text. Picks the smallest *safe* long-bracket level â€” `[[ ... ]]` if the string contains no `]â€¦]` pair, otherwise enough equals signs that no inner `]==>{n}]` matches the close.

The level-picking algorithm (`pick_long_bracket_level`, `lua_string.rs:23`) scans for every `]=*]` substring in the input and returns `max(equals_run) + 1`, guaranteeing the close marker cannot appear inside the wrapped content:

| Input | Wrapped |
|---|---|
| `hello` | `[[hello]]` |
| `a ]] b` | `[=[a ]] b]=]` |
| `a ]=] b` | `[==[a ]=] b]==]` |
| `]]]` | `[=[]]]]=]` |

### `lua_chunk_literal` (`lua_string.rs:54`)

Same level-picking, but adds the surrounding newlines that Lua eats after `[â€¦[` so the body's first line is preserved verbatim. Used for plate / test Lua bodies where the original line layout matters.

## Codegen-phase Validation

Beyond per-template builtin checks, `validate_accessor_placement` (`recipe.rs:222`) runs once per recipe per step and rejects:

- **Bare path accessors in an output pattern** â€” `$<stem>`, `$<name>`, `$<ext>`, `$<dir>` were removed by CS-0033 Â§6.7. Use `$<in.stem>` (or `$<dep.stem>` for a dep-driven pattern). (`recipe.rs:182-192`)
- **Bare recipe ref in an output pattern** â€” `$<lib>` with no accessor has no iteration semantics in an output pattern (Standard Â§5.4). Use `$<lib.stem>` (or move it into the step body). (`recipe.rs:200-210`)
- **Multi-output coherence** â€” every output pattern in a multi-output cook step must share the same driver. (`recipe.rs:126`)
- **Placeholder violations inside shell-block bodies** â€” wrong mode, wrong output count, malformed `out_0`, etc. (`recipe.rs:264-273`)
- **`$<lib.X>` in plate / test / bare shell bodies** â€” these have no output pattern, so cannot declare a driver for dep-accessor iteration (the Â§5.4 firewall). (`recipe.rs:289-326`, `check_command` at `recipe.rs:569`)

The companion `warn_empty_output_refs` (`recipe.rs:96`) produces non-fatal warnings when a `$<NAME>` reference targets a recipe with no cook steps and no ingredients â€” the reference still lowers (to the empty string at runtime), but the user almost certainly meant something else.
