# 7. Modules

> **Normative.** This chapter defines the module system: how modules are named, loaded, resolved, and what a module is expected to expose. The grammar of the `use` and `import` declarations is given in App. A.2; this chapter specifies their meaning.

## 7.1. Module concept

A **module** is a named bundle of Lua code that augments the Cook runtime by exposing additional step kinds, helpers, or data through a single Lua table. A module is brought into a Cookfile's Lua environment by one of two declaration forms defined in App. A.2:

| Declaration | Grammar (App. A.2) | Selects |
|---|---|---|
| `use` declaration | `"use" name NEWLINE` | A module resolved by name (§ 7.2). |
| `import` declaration | `"import" BARE_IDENTIFIER path NEWLINE` | A sibling Cookfile aliased under a local identifier (§ 7.3). |

The two declarations are distinct and non-interchangeable. A `use` declaration brings a **module** (a Lua table) into scope and does not introduce a new Cookfile; an `import` declaration introduces a **sibling Cookfile** and does not load Lua module code. A conforming implementation MUST reject a Cookfile that places either declaration after the first `recipe` (App. A.2, ordering clause).

The set of modules available to a `use` declaration is partitioned into two disjoint kinds. A **built-in module** is supplied by the implementation and available without any project-level file; no built-in modules are defined by v0.1 of this Standard, and the built-in kind is reserved to allow future extension (§ B.7). A **local module** is a file in the project's `cook_modules/` directory (§ 7.4). No other kinds are defined.

### Note 7.1.1

The test case at `docs/standard/conformance/positive/010-use-and-module-call` pins the shape of a `use` declaration and the module-call that follows it. See also § 4.11 for the step-dispatch rule that routes `cpp.bin(...)` inside a recipe body to a Lua step rather than a shell step.

## 7.2. `use <name>` declarations

A `use` declaration has the form `use <name>`, where `<name>` is either a `BARE_IDENTIFIER` or a `STRING` (App. A.2). A conforming implementation MUST resolve a `use` declaration by applying the following steps in order, with the first matching step winning:

1. If `<name>` designates a built-in module (§ 7.1), the built-in module is selected.
2. Otherwise, the implementation MUST attempt to resolve `<name>` as a local module under `cook_modules/` by the lookup algorithm of § 7.4.

A conforming implementation MUST reject a Cookfile for which no step succeeds, with a diagnostic that names `<name>` and indicates which locations were searched. The resolution step MUST occur before any recipe body executes; failure to resolve is a load-time error, not a run-time error.

The result of resolution is a Lua table bound in the Cookfile's Lua environment under the identifier derived from `<name>` by replacing each ASCII hyphen (`-`) with an underscore (`_`). All other characters admitted by `BARE_IDENTIFIER` (App. A.5) pass through unchanged. A conforming implementation MUST make this binding visible to every subsequent `lua_line`, `lua_block`, `module_call` (§ 4.11), and `cook_step` `using` clause in the same Cookfile.

A module MAY expose a function named `init` on its returned table. If present, `init` MUST be called exactly once immediately after the module's Lua source has been executed, before any recipe body in the enclosing Cookfile is registered (§ 7.5).

### Example 7.2.1

```cook
use cpp

recipe "colorize"
    cpp.bin("colorize", { sources = { "src/colorize.c" } })
end
```

The declaration `use cpp` binds the identifier `cpp` in the Cookfile's Lua environment to the table returned by `cook_modules/cpp.lua` (or `cook_modules/cpp/init.lua`). The `cpp.bin(...)` line in the recipe body is then dispatched as a `module_call` step (App. A.4, dispatch step 5) rather than a `shell_command`, because its first dot-separated segment `cpp` matches a bound identifier pattern (§ 4.11).

### Note 7.2.1

The hyphen-to-underscore rewrite exists because `BARE_IDENTIFIER` permits internal hyphens (App. A.5) whereas Lua identifiers do not. Thus `use my-mod` binds the Lua global `my_mod`, but the on-disk lookup in § 7.4 searches for `cook_modules/my-mod.lua` verbatim: the disk name is not rewritten.

## 7.3. `import <name> <path>` declarations

An `import` declaration has the form `import <name> <path>`, where `<name>` is a `BARE_IDENTIFIER` and `<path>` is a non-whitespace path token (App. A.5). A conforming implementation MUST treat `<path>` as a directory path resolved relative to the directory of the enclosing Cookfile, and MUST load the Cookfile at `<path>/Cookfile` as a sibling Cookfile bound to the local alias `<name>`.

A conforming implementation MUST reject an `import` declaration for which:

- the directory `<path>` does not exist;
- `<path>/Cookfile` does not exist;
- `<path>/Cookfile` fails to parse.

An `import` declaration does NOT bring a Lua value into scope under `<name>`. The alias `<name>` is reserved for cross-Cookfile recipe references of the form `<name>.<recipe>` that appear in dependency lists (§ 4.3), not for Lua expressions. A Cookfile loaded by `import` is parsed independently; its own `use` and `import` declarations are resolved relative to its own directory, not the importer's directory.

A conforming implementation MUST reject a Cookfile in which two `import` declarations share the same `<name>` (App. A.2, duplicate-import clause; see also § 7.6).

### Example 7.3.1

```cook
import backend ./services/backend
import frontend ./apps/frontend

recipe "bundle": "backend.build" "frontend.build"
end
```

Here `backend` and `frontend` are local aliases for the Cookfiles at `./services/backend/Cookfile` and `./apps/frontend/Cookfile`, respectively. The dependencies `"backend.build"` and `"frontend.build"` refer to recipes in those sibling Cookfiles; the first segment before the dot selects the import alias, and the remainder is the recipe name in the imported Cookfile.

### Note 7.3.1

Because an `import` alias does not bind a Lua value, there is no interaction between `import backend …` and a `use backend` for a module of the same name: they occupy different namespaces. The duplicate rule of § 7.6 only prohibits collisions within the `import` alias namespace.

## 7.4. `cook_modules/` resolution

A conforming implementation MUST resolve a local module name `<name>` (from a `use` declaration, § 7.2) against the directory `cook_modules/` located in the enclosing Cookfile's directory. The lookup algorithm is observable by the candidate paths it consults and the order in which it consults them:

1. Let `D` be the directory of the Cookfile containing the `use` declaration.
2. Construct `F = D/cook_modules/<name>.lua` (the **flat form**).
3. Construct `I = D/cook_modules/<name>/init.lua` (the **directory form**).
4. If `F` exists and is readable, `F` is the module file.
5. Otherwise, if `I` exists and is readable, `I` is the module file.
6. Otherwise, resolution fails; the implementation MUST produce a diagnostic that names `<name>` and MUST mention both `F` and `I` as candidates consulted.

The flat form takes precedence over the directory form when both are present. A conforming implementation MUST NOT consult any path outside `D/cook_modules/` during local-module resolution. In particular, parent directories, ancestor `cook_modules/` directories, and any system-wide module paths are NOT part of the algorithm.

The selected module file is loaded as a Lua chunk in the Cookfile's Lua environment. Its return value MUST be a Lua table (§ 7.5); a chunk that returns any other Lua type — including `nil`, by omitting a `return` statement — is a module-authoring error and its observable failure mode is defined in § 7.5.

### Example 7.4.1

Given a Cookfile at `/project/Cookfile` containing `use cpp`, the implementation consults, in order:

1. `/project/cook_modules/cpp.lua` — if present, selected.
2. `/project/cook_modules/cpp/init.lua` — consulted only if (1) is absent; if present, selected.

If neither path exists, the Cookfile is rejected at load time.

### Note 7.4.1

The `cook_modules/` convention is project-local by design: two Cookfiles in the same repository that sit in different directories each consult their own adjacent `cook_modules/`, and neither inherits modules from the other. See § B.7 for the rationale.

## 7.5. Module authoring contract

A **conforming module** is a Lua chunk whose top-level return value is a Lua table. A conforming implementation MUST load the module by evaluating its chunk once per Cookfile that selects it and MUST use the returned table as the value bound to the `use`-declaration's identifier (§ 7.2).

A conforming module MAY expose the following conventional keys on its returned table. When present, a conforming implementation MUST interpret them as described:

| Key | Type | Interpretation |
|---|---|---|
| `init` | function | Called exactly once, with no arguments, immediately after the chunk has been evaluated. Its return value is ignored. |

All other keys on the returned table are module-defined. A conforming module typically exposes step-building helpers (for example, `cpp.bin`, `cpp.lib`, `cpp.compile_commands`) that call into the Cook Lua API defined in § 6. A module MAY call any Cook Lua API accessible at registration time; in particular it MAY call `cook.add_unit`, `cook.export`, `cook.import`, `cook.step_group`, and `cook.cache.*`. The module's side effects on the Cookfile's Lua environment beyond its returned table are module-defined and not governed by this chapter.

A module MUST NOT assume a particular file location, working directory, or project layout other than what the Cook Lua API of § 6 provides. A module MUST NOT mutate the Cookfile's Lua globals other than to define values on the table it returns; in particular, it MUST NOT redefine the `cook`, `fs`, or `path` tables registered by § 6.

If a module's chunk raises a Lua error during evaluation or during its `init` call, the implementation MUST propagate the error as a load-time failure and MUST NOT proceed to register recipes in the enclosing Cookfile.

### Example 7.5.1

```lua
-- cook_modules/greet.lua
local greet = {}

greet.state = { prefix = "hello" }

function greet.init()
    -- Called once by the implementation after this chunk returns.
    greet.state.prefix = cook.env.GREET_PREFIX or greet.state.prefix
end

function greet.say(who)
    cook.add_unit({
        inputs = {},
        output = ".cook/greet/" .. who .. ".txt",
        command = 'echo "' .. greet.state.prefix .. ", " .. who .. '" > .cook/greet/' .. who .. ".txt",
    })
end

return greet
```

A Cookfile that declares `use greet` binds the table returned by this chunk to the identifier `greet`, calls `greet.init()` once, and thereafter admits `greet.say("world")` as a `module_call` step (App. A.4, dispatch step 5).

### Note 7.5.1

The `init` hook is the intended place for a module to perform one-shot discovery (compiler detection, toolchain probing, cache warm-up). Because `init` is called exactly once per Cookfile load, a module that performs idempotent discovery in `init` can amortise its cost across every recipe in the Cookfile without re-running per recipe.

## 7.6. Duplicate and cycle detection

A conforming implementation MUST reject a Cookfile in which two `import` declarations share the same local alias. This rule is restated normatively in App. A.2 (duplicate-import clause); this section specifies the timing and the diagnostic surface.

**Duplicate imports.** The check is a **parse-time** check performed on the Cookfile in which the duplicates appear. A conforming implementation MUST detect the duplicate before loading the imported Cookfiles (§ 7.3) and MUST report both offending lines in the diagnostic. The rule is per-Cookfile: two different Cookfiles in the same workspace MAY each define an `import foo ...` of their own, because each alias namespace is scoped to its declaring Cookfile.

**Import cycles.** A conforming implementation MUST detect cycles in the `import` graph when loading the workspace and MUST reject a workspace whose `import` edges form a cycle. A cycle is defined as a sequence of canonicalised directory paths `D_1, D_2, ..., D_k, D_1` such that each `D_i/Cookfile` contains an `import` declaration whose path resolves to `D_{i+1}` (and the last edge resolves back to `D_1`). Detection MUST use canonicalised paths so that two syntactically different `<path>` strings that refer to the same directory are recognised as the same node.

A conforming implementation MUST NOT reject the diamond case: if two distinct imports both resolve to the same canonicalised `D`, the implementation MUST load `D` once and reuse it. Diamond deduplication is not a cycle.

The duplicate check and the cycle check operate on different graphs and are independent. A Cookfile MAY violate either without violating the other.

### Example 7.6.1

```cook
import backend ./services/backend
import backend ./services/backend_v2
```

Rejected at parse time: both lines declare the alias `backend` (App. A.2). The first line's target is irrelevant to the diagnostic; the collision is on the alias, not the path.

### Example 7.6.2

Given three Cookfiles:

```
/ws/a/Cookfile:       import b ../b
/ws/b/Cookfile:       import a ../a
/ws/Cookfile:         import a ./a
```

The workspace load that begins at `/ws/Cookfile` MUST be rejected: canonicalisation yields the cycle `/ws/a -> /ws/b -> /ws/a`.

### Note 7.6.1

Duplicate-import detection fires at parse time because the governing grammar production (App. A.2) constrains a single Cookfile's token stream, which is the unit the parser consumes. Cycle detection, by contrast, is a property of the `import` graph and necessarily crosses Cookfile boundaries; it fires at workspace-load time as the graph is walked (§ B.7).
