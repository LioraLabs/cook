# 2. Your first recipe

`cook init` dropped two files in your scratch directory:

```
hello-cook/
├── Cookfile
└── .gitignore
```

Open the `Cookfile`. It looks like this:

```cook
recipe build
    echo "Hello from Cook!"
```

Two lines. We're going to spend this whole chapter on those two lines, because everything in Cook is built out of the same shapes.

## 2.1. The shape of a recipe

A **recipe** is a named bundle of work. It starts with the keyword `recipe`, a name, and a colon-separated list of dependencies if it has any. Below the header comes the recipe **body**, indented one level. The body lists the *steps* the recipe runs.

```cook
recipe build
    echo "Hello from Cook!"
```

The first line is the **header** (no dependencies yet); the indented line is **one step**, a bare shell command.

There is no closing `end` keyword. A recipe ends when the file ends, or when another column-0 keyword starts a new top-level form (`recipe`, `chore`, `config`, `use`, `import`). Indentation is what makes a line *part of the body* rather than the start of something new. *(Standard §4.1.1)*

## 2.2. Running it

```bash
cook build
```

Or just:

```bash
cook
```

— with no recipe name, `cook` runs the recipe called `build` by default. That's a convention, not a requirement; you can call your default recipe whatever you want and run it explicitly.

## 2.3. What `cook` actually did

When you typed `cook build`, four things happened in order:

1. **Parsed** the Cookfile. The lexer broke each line into a token; the parser stitched them into a recipe with one shell step.
2. **Registered** the recipe. Cook ran the recipe body once, on a single Lua VM, in **capture mode** — it didn't *run* `echo`, it *recorded* a "shell unit" that says "later, run `echo "Hello from Cook!"`".
3. **Built a DAG.** With one recipe and one unit, the graph is trivial. With more recipes, this is where dependency ordering happens.
4. **Executed** the units. *Now* `echo` actually runs.

This **two-phase model** — register, then execute — is the single biggest difference between Cook and `make` or `npm scripts`. We'll come back to it in chapter 8 when we start writing Lua. For now: when you read a Cookfile, you're reading a *plan*, not a script.

<div class="book-tip">

**Tip — caching is opt-in.** Run `cook build` a second time. The `echo` runs again, every time. Bare shell steps don't cache — Cook re-runs them on every invocation, by design (Standard §8.6). Caching kicks in once a step declares inputs and outputs (`ingredients` plus `cook ... using { ... }`), which we'll get to in chapter 4.

</div>

## 2.4. A second recipe

Add another recipe. Open the Cookfile and replace its contents with:

```cook
recipe build
    echo "Building..."
    echo "Done."

recipe greet
    echo "Hello, $USER!"
```

Now run:

```bash
cook menu
```

`cook menu` lists every recipe and chore visible in the workspace. Useful when you've forgotten what a project provides. Then run each:

```bash
cook build
cook greet
```

Two recipes, no relationship between them. Each runs on its own.

## 2.5. Two flavours of name

A recipe name can be a **bare identifier** or a **quoted string**. Either form below names the same recipe `build` (write one or the other, not both — duplicate names in the same Cookfile are not legal):

```cook
recipe build
    echo same
```

```cook
recipe "build"
    echo same
```

Quoting is what you reach for when the name needs a character a bare identifier can't carry — most often a space. *(Standard §2.3, §2.5, §4.1.)*

A bare identifier accepts ASCII letters, digits, underscore, and hyphen (`-`). It can *not* contain a dot: at a declaration site, `recipe backend.build` is rejected by the parser. Dots appear at *reference* sites (in dependency lists and `$<...>` placeholders), where `backend.build` resolves through an `import` alias. We'll come back to this in chapter 9.

## 2.6. Comments and blank lines

A line whose first non-whitespace character is `#` is a comment. Blank lines are ignored. Both can appear inside a recipe body without ending it:

```cook
# Top-level comment.

recipe build
    echo "starting"
    # mid-body comment
    echo "done"
```

## 2.7. Try it

Make a file `Cookfile`:

```cook
recipe greet
    echo "hi from $USER"
    echo "the time is $(date +%T)"

recipe build: greet
    echo "Build proper starts here"
```

Run `cook build`. Notice that `greet` runs first — that's the colon doing its job. The next chapter is about exactly that.
