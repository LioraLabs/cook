# 9. Modules & composition

Once a project has more than one Cookfile worth of recipes, you start wanting to reuse things. Cook gives you two complementary tools:

1. **`use`** — pull in a Lua module that *defines new functions* in scope. Good for "I want a `mylib.something(...)` helper everywhere."
2. **`import`** — pull in another *Cookfile* under an alias, so its recipes are addressable as `alias.recipe`. Good for monorepos where each subproject has its own Cookfile.

## 9.1. Modules with `use`

A Cook module is a Lua file that returns a table:

```lua
-- cook_modules/greet.lua
local greet = {}

function greet.shout(name)
    cook.add_unit({
        command = "echo HELLO " .. name:upper(),
        cache   = false,
    })
end

return greet
```

Pull it into a Cookfile:

```cook
use greet

recipe wave
    greet.shout("world")
```

`use greet` binds `greet` as a global table inside the Cookfile's Lua context. Any recipe can call `greet.something(...)`. The bare `alias.fn(args)` form on its own line is the **module-call shorthand** introduced in §9.4 — it desugars to a register-phase (`>>`) call. That matters here because `greet.shout` invokes `cook.add_unit`, which is register-only; calling it via `>` (execute phase) raises `attempt to call a nil value (field 'add_unit')`. *(Standard §6.* and §7.*.)*

Cook resolves `use NAME` by looking for `cook_modules/NAME.lua` in the workspace. If not found, it errors when a recipe that depends on the binding is registered.

## 9.2. Where modules come from

You write your own modules in `cook_modules/` for project-specific helpers. That's the path the Book uses everywhere — `use NAME` resolves to `cook_modules/NAME.lua` and you own the source.

For shared functionality across projects, Cook also has a `cook modules` subcommand that manages a LuaRocks-backed module index. The CLI surface (`cook modules install`, `list`, `search`, …) is documented in chapter 10. There aren't any published modules to demo here yet — when there are, each one will get its own chapter showing what it provides and how to use it.

## 9.3. `import` — composing whole Cookfiles

For monorepos: each subproject gets its own Cookfile, and the root composes them.

```cook
# /Cookfile
import backend  ./backend
import frontend ./frontend
import tools    ./tools

recipe ship: backend.build frontend.build
    echo "deploying"

chore fmt: tools.fmt
```

`import ALIAS PATH` says "the Cookfile at `PATH/Cookfile` is now reachable under `ALIAS`." Its recipes show up as `ALIAS.recipe`, its chores as `ALIAS.chore`.

There is no whole-token-vs-split ambiguity to worry about. A recipe declared in a single Cookfile always has an undotted name (Standard §2.3 / App. A.2):

```cook
# this is rejected at parse time:
recipe backend.build
    echo nope
```

Dotted forms only appear at the *reference* site (in dependency lists, in `$<...>` placeholders) and always route through an `import` alias. Want a `backend.build` recipe? Put `recipe build` in `./backend/Cookfile` and `import backend ./backend` at the top of the parent Cookfile. *(Standard §5.2.1, §7.)*

## 9.4. Module-call shorthand

This:

```cook
recipe wave
    greet.shout("world")
```

is shorthand for:

```cook
recipe wave
    >> greet.shout("world")
```

Cook recognises `alias.fn(args)` as a top-level register-phase call inside a recipe body and desugars to `>>` automatically. The shorthand keeps Cookfiles compact when you're calling module helpers; the `>>{ ... }` form is still there when you need register-phase Lua that isn't a single function call. *(Standard §4.11.)*

## 9.5. Building your own module

Three steps:

1. Drop the file in `cook_modules/yourmod.lua`. Make it return a table.
2. `use yourmod` from any Cookfile in the project.
3. Call `yourmod.fn(...)` from a recipe body.

Inside a module, you have the full `cook.*` API available — `cook.add_unit`, `cook.step_group`, `cook.env`, the path/fs helpers, everything. The Standard's chapter 6 is the full reference for what's in scope.

## 9.6. Try it

Make `cook_modules/copy.lua`:

```lua
local copy = {}

-- Expand a glob and register one unit per matched file. Cook's cache
-- hashes file content, not directories, so each input must be a file.
-- Passing a directory (or a directory glob) here raises a register-time
-- error pointing back at this call site.
function copy.tree(src_glob, dst_dir)
    local sources = fs.glob(src_glob)
    for _, src in ipairs(sources) do
        local name = path.name(src)
        local dst = path.join(dst_dir, name)
        cook.add_unit({
            inputs  = { src },
            output  = dst,
            command = string.format("mkdir -p %s && cp -a %s %s", dst_dir, src, dst),
        })
    end
end

return copy
```

Now in `Cookfile`:

```cook
use copy

recipe vendor
    copy.tree("upstream/lib/*", "build/lib")
```

Run `cook vendor`. The first run copies each file; the second run is cached because every declared input resolves to a file whose content cook can hash. (We're using the module-call shorthand from §9.4 — `copy.tree` registers units via `cook.add_unit`, which is register-phase. Writing `> copy.tree(...)` would error.)

> Why a glob and not a directory? Cook's cache hashes file content, so every entry in `inputs` must point at a file. A directory input has no hashable bytes; cook rejects it at register time rather than silently invalidating the cache on every run.
