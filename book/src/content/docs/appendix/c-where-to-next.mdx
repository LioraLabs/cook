# Appendix C — Where to next

The Book covers what you need to *use* Cook. The Standard covers everything Cook *guarantees*. When you're past the tutorial stage and need authoritative answers, go there.

## C.1. The Cook Standard

[docs.usecook.com/standard/](https://docs.usecook.com/standard/)

The Standard is organised so you can read it the way you'd read a language reference: pick the chapter that covers the thing you care about.

| Chapter | What it covers | Read it when |
|---------|----------------|---------------|
| §0      | Scope, conformance, versioning | You want to know what the Standard does and doesn't cover. |
| §1      | Notation conventions (RFC-2119, EBNF) | First time reading the Standard. |
| §2      | Lexical structure (tokens, line classification) | A tokenizer-shaped question. |
| §3      | Syntactic grammar | "Is X a legal Cookfile?" |
| §4      | Recipes, ingredients, every step kind | Core questions about how a recipe works. |
| §4a     | Chores | What's allowed in a chore body. |
| §5      | Cross-recipe references | Placeholder resolution, drivers, accessors. |
| §6      | The Cook Lua API | Every `cook.*` function, `fs.*`, `path.*`. |
| §7      | Cross-Cookfile composition | `import`, `use`, alias resolution. |
| §8      | Execution model | Two-phase execution, caching, ordering. |
| App. A  | Formal grammar | When prose isn't enough. |
| App. B  | Rationale | Why Cook works the way it does. |
| App. C  | Worked examples | Bigger Cookfiles with annotations. |
| App. D  | Changes (CS-NNNN log) | What changed in each version. |
| App. E  | Pre-1.0 checklist | What still has to land before Cook is 1.0. |

## C.2. Examples in the cook repo

The cook source tree ships ten numbered example projects, in learning order:

```
cook/examples/
├── 01-hello-cook/        # a recipe, ingredients, the free second run
├── 02-pipeline/          # fan-out → many-to-one → multi-output
├── 03-chores-and-config/ # uncached verbs; env vars as cache keys
├── 04-probes/            # computed values the cache can see
├── 05-data-fanout/       # the build's shape from a JSON manifest
├── 06-lua-recipes/       # Lua bodies and computed output paths
├── 07-testing/           # tests as cached steps
├── 08-workspace/         # many Cookfiles, run cook from anywhere
├── 09-deploy/            # plate: the ship-it step
├── 10-cache-trust/       # seal/local/pinned/nondet; cook why
└── modules/              # examples that need installed modules (cook_cc, pnpm)
```

Every example has a runnable `Cookfile` and a README with a "watch the
cache work" moment; `examples/README.md` is the index. Start at 01 and
run each one twice.

## C.3. The DAG viewer

`cook dag` is the single best tool for understanding a build you didn't write. Open it with the recipe you want to investigate:

```bash
cook dag ship
```

Every node tells you:
- Its declared inputs and outputs
- Its cache status (hit, miss, or "would re-run")
- Which units it triggers and which units triggered it

If a build "is rebuilding too much" or "isn't rebuilding when I expect", the DAG viewer will tell you why faster than reading the Cookfile.

## C.4. When the Book is wrong

This is a rough first draft. If something in the Book disagrees with the Standard, the Standard is correct and the Book is buggy. Open an issue (or fix it directly) — patches to the Book don't need a CS-NNNN entry, but they do need to land via the same review path as any other change to the cook repo.

## C.5. Reading list, in order

If you're picking up Cook for a real project right now:

1. Re-read **chapter 4** (`ingredients` and `cook`) — that's 90% of what you'll write.
2. Read the [Standard's §4](https://docs.usecook.com/standard/04-recipes/) for the formal rules.
3. Skim the **monorepo example** to see how multi-Cookfile composition fits together.
4. Bookmark `cook --help`, `cook menu` / `cook list` (recipe discovery), and `cook emit-lua` (compiled-Lua dump).

That's enough to ship.
