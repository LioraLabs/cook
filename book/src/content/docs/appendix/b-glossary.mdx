# Appendix B — Glossary

Terms used throughout the Book and the Standard, in plain English.

**Cookfile** — A file (typically `Cookfile`) that declares recipes, chores, configuration, and module imports. The unit Cook operates on.

**Recipe** — A named, dependency-aware bundle of build steps. Started by the `recipe` keyword.

**Chore** — A named bundle of *non-build* commands. Started by the `chore` keyword. No caching, no `ingredients`, no `cook` step. Commands run interactively by default.

**Step** — A single line (or block) inside a recipe or chore body. Each step has a *kind*: `ingredients`, `cook`, `plate`, `test`, shell command, Lua line, Lua block, module call.

**`cook` step** — The build-step keyword (distinct from the `cook` binary). Declares one or more outputs and a shell or Lua body that produces them; every `cook` step is cacheable. *(Standard §4.5–§4.6.)*

**`plate` step** — A `cook`-style step with no declared outputs. Iterates over the preceding `cook` step's outputs (or the recipe's ingredients, if no `cook` step precedes), typically to ship something outside the project (deploy, install, publish). Not cached. *(Standard §4.7.)*

**`test` step** — Like `plate`, but its units report pass/fail rather than producing artifacts. Supports `as 'name'`, `timeout N`, and `should_fail` modifiers in that canonical order. The Standard leaves cross-recipe aggregation implementation-defined in v0.8; the bundled `cook test` runner currently collects failures into one summary and exposes `--fail-fast` to short-circuit. *(Standard §4.8.)*

**`@` step / interactive command** — A shell step prefixed with `@`. Drains all concurrent work before running, inherits the controlling TTY, and is never cached. Chore-body shell steps behave this way by default; recipe-body shell steps need the explicit `@`. *(Standard §4.10, §4a.3.)*

**Step group** — A set of work units registered together that may run in parallel. The surface `cook` step routes through one automatically.

**Work unit** — The smallest scheduling primitive Cook executes. Recorded at register time, dispatched at execute time.

**Register phase** — The first phase of a recipe's execution. Lua code in the body runs once, in *capture mode*; calls like `cook.add_unit` record work units rather than performing them.

**Execute phase** — The second phase. Cook arranges the recorded work units into a graph and runs them, in parallel where it can.

**Capture mode** — The register-phase semantics of the Cook Lua API: side-effecting calls record work instead of doing it.

**Ingredient** — An input file matched by a glob pattern in an `ingredients` step. The full ingredient set of a recipe is `(union of includes) \ (union of excludes)`.

**Driver** — Whatever populates iteration for a `cook` step: either the recipe's own ingredients (referenced via `$<in.ACCESSOR>`) or another recipe's outputs (`$<lib.ACCESSOR>`). One unit per item.

**Iteration mode** — How many units a step produces relative to its inputs. *One-to-one* (one unit per input, `$<in...>` in the output pattern), *many-to-one* (one unit consuming `$<all>` inputs, no `$<in...>` in the output pattern), or *one-shot* (no `ingredients` at all). *(Standard §4.5–§4.6.)*

**Path accessor** — One of `stem`, `name`, `ext`, `dir`. Splits a path inside a placeholder.

**Placeholder** — A `$<IDENT>` (optionally `$<IDENT.ACCESSOR>`, or the indexed form `$<out_N>` / `$<out_N.ACCESSOR>` for multi-output `cook` steps) that resolves to one of: a builtin (`in`/`out`/`all`/`out_N`), another recipe's outputs, or a declared env var. The `$<...>` sigil was picked specifically so it doesn't clash with shell brace expansion (`{a,b,c}`), shell parameter expansion (`${VAR}`), or anything else that uses curly braces.

**Sigil** — A reserved marker at the Cookfile surface. Cook uses two: `$<...>` for placeholders inside shell text, and `//` for workspace-root-anchored import paths.

**String substitution** — The expansion of a bare `$<lib>` (no accessor) inside a step body to the resolved outputs of recipe `lib`, space-separated.

**Dep-driven iteration** — When a `cook` step's output pattern references another recipe with an accessor (`$<lib.stem>`), that recipe drives iteration.

**Config block** — A `config` (or `config NAME`) top-level section. Runs at *load time*, before any recipe registers, and seeds `cook.env` with declared variables.

**Module** — A Lua file under `cook_modules/` (as `NAME.lua` or `NAME/init.lua`) that returns a table, pulled into scope with `use NAME`. If the module's table exposes an `init()` function, Cook calls it once after load. (The Lua API tables `fs` and `path` are *not* modules — they're pre-bound library tables provided by the host environment per Standard §6.5–§6.6. Standard §6.8.1 reserves a built-in-module slot but defines no built-ins in v0.1.)

**Import** — A statement that pulls another Cookfile under an alias. `import backend ./backend` makes `backend.build` resolve to that Cookfile's `build` recipe.

**Namespace** — Informal term for the `alias` segment of a qualified reference: in `backend.build`, the `backend` `import` alias is the namespace and `build` is the recipe name (Standard §7.3 defines `alias.recipe` as the single-segment normative form). Deeper dotted paths like `apps.web.unit` work in practice because `apps` imports a Cookfile that itself imports under alias `web`, but the Standard does not yet normatively define the chained form.

**Drain / single-drain** — Execution mode in which Cook waits for every in-flight work unit to finish before starting the marked unit, and runs nothing else concurrently with it. Used by `@`-prefixed interactive shell steps and by every chore body (single-drain start, Standard §4a.3 / CS-0051).

**Cache key** — The hash Cook computes for a work unit, used to decide whether to re-run it. Built from declared inputs (file content), declared outputs, the rendered command, and any consulted env keys.

**Discovered inputs** — Inputs a unit declares *during* execute time, typically by reading a `.d` dependency file emitted by a compiler. Folded into the cache key for next time.

**Workspace** — The directory tree Cook treats as one project, rooted at the *workspace root*.

**Workspace root** — The anchor for `//`-prefixed import paths. Located, in order: `--root <dir>` on the CLI, a `.cookroot` marker file, or fallback to the invoked Cookfile's directory. *(Standard §7.6.)*

**`.cook/`** — Per-workspace cache directory. Stores hashes, log archives, and any side files referenced by `discovered_inputs`. Safe to delete.

**Surface form** — The Cookfile-side syntax that desugars to `cook.*` Lua calls. Keeps simple things short. Use `cook emit-lua` to see the desugaring.

**Standard** — [The Cook Standard](https://docs.usecook.com/standard/). Authoritative reference for everything the Book covers casually. When the Book and the Standard disagree, the Standard is correct.
