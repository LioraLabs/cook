# 4. Ingredients & `cook`

The recipes we've written so far run shell commands. That's fine for "echo something" or "deploy the thing", but real builds have a shape:

> Take a set of input files, transform each one, produce a set of output files. Don't redo the transform if neither the inputs nor the command have changed.

`ingredients` and the `cook` step are how you describe that shape. Together they give you everything `make` does â€” file globs, per-file iteration, output declarations, caching â€” without writing a single rule.

## 4.1. `ingredients` â€” declare your inputs

The `ingredients` step declares the input file set for a recipe. Patterns are quoted strings; each pattern is a glob.

```cook
recipe build
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" using { gcc -c $<in> -o $<out> }
```

A recipe MAY have at most one `ingredients` line. The set is the union of all include patterns, minus any exclude patterns:

```cook
recipe lib
    ingredients "src/*.c" "include/*.h" !"src/lua.c" !"src/luac.c"
    cook "build/libmath.a" using { ar rcs $<out> $<all> }
```

Excludes are `!` followed *immediately* by a quoted pattern â€” `!"foo"`, not `! "foo"`. The order of items on the line doesn't matter; the result is `(union of includes) \ (union of excludes)`. *(Standard Â§4.3.)*

## 4.2. The `cook` step â€” one input, one output (per ingredient)

```cook
recipe build
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" using { gcc -c $<in> -o $<out> }
```

That `cook` line is the workhorse. Read it as: *for each ingredient, produce an output named `build/<stem>.o` by running this command.*

Three things to notice:

- The output pattern (`"build/$<in.stem>.o"`) has placeholders. `$<in>` is the current ingredient's full path; `$<in.stem>` is its filename without the extension. So `src/foo.c` becomes `build/foo.o`.
- The `using { ... }` block is the **shell template** that builds each output. It also uses placeholders â€” `$<in>` (input), `$<out>` (the output), `$<all>` (every input on one line, space-separated).
- Cook iterates this step over the ingredient set. Five `.c` files â†’ five units, one per file, runnable in parallel.

This shape is called a **one-to-one** `cook` step. *(Standard Â§4.5.)*

## 4.3. Many-to-one (the merge step)

When you want one output from many inputs â€” a static library, a tarball, a bundle â€” leave the `$<in...>` accessor *out* of the output pattern:

```cook
recipe lib
    ingredients "build/*.o"
    cook "build/libmath.a" using { ar rcs $<out> $<all> }
```

There's no per-input iteration here: one output pattern, no input accessor â†’ one unit, with `$<all>` expanding to every ingredient. *(Standard Â§4.6.)*

## 4.4. Path accessors

Inside placeholders, you can pull pieces out of a path with the four **path accessors**:

| Accessor | Meaning                          | `src/sub/foo.tar.gz` â†’       |
|----------|----------------------------------|------------------------------|
| `stem`   | file name, no last ext           | `foo.tar`                   |
| `name`   | full file name                   | `foo.tar.gz`                |
| `ext`    | last extension, with leading dot | `.gz`                       |
| `dir`    | directory part                   | `src/sub`                   |

So:

```cook
recipe assets
    ingredients "src/img/*.png"
    cook "build/img/$<in.stem>.webp" using { cwebp $<in> -o $<out> }
```

`src/img/cat.png` â†’ `build/img/cat.webp`. *(Standard Â§5.3.)*

## 4.5. Multiple outputs from one command

A single command sometimes produces several files at once â€” a wasm-pack invocation, a code generator, a bundler. Declare them with multiple output patterns:

```cook
recipe wasm
    ingredients "src/wasm.rs"
    cook "out.js" "out.wasm" using {
        wasm-pack build
        cp pkg/out.js out.js
        cp pkg/out.wasm out.wasm
    }
```

Inside the `using` body, `$<out_1>` and `$<out_2>` refer to each declared output by index. *(Standard Â§4.6.)*

## 4.6. Driver iteration: `$<lib.stem>`

Sometimes the *driver* of iteration isn't the current recipe's ingredients â€” it's another recipe's outputs. Example: every executable in `tests/` should run as a test, with one test unit per executable.

```cook
recipe buildtests
    ingredients "tests/*.c"
    cook "build/$<in.stem>" using { gcc $<in> -o $<out> }

recipe runtests
    cook "results/$<buildtests.stem>.txt" using {
        ./$<in> > $<out>
    }
```

When `cook` sees `$<buildtests.stem>` in the output pattern, the driver is `buildtests`'s output list, not `runtests`'s own ingredients. One unit per file in that list. *(Standard Â§5.4.)*

> **Note â€” placeholder identifiers don't allow hyphens.** A recipe name like `build-tests` is legal, but `$<build-tests.stem>` won't parse as a placeholder (the lexer stops at the hyphen â€” Standard Â§2.11). If you want to drive iteration off another recipe's outputs, keep its name hyphen-free or use dots: `$<buildtests.stem>` or `$<build.tests.stem>`.

A `cook` step can have at most one driver recipe in its output patterns. Bare `$<lib>` (no accessor) in an output pattern is rejected â€” accessor or nothing.

## 4.7. Caching, automatically

Every `cook` step is cached. The cache key is built from:

- the resolved input file contents (hashed),
- the declared output paths,
- the rendered command string.

Change a `.c` file â†’ only its output is rebuilt. Add a flag to `gcc` â†’ everything rebuilds. Touch a `.c` file but leave its bytes alone â†’ nothing rebuilds. *(Standard Â§8.6.)*

You can opt a unit out by routing through Lua (chapter 8) and setting `cache = false`, but the surface `cook` step always caches.

## 4.8. The cache lives in `.cook/`

Cook keeps its build state in `./.cook/` per workspace. `cook init` writes a `.gitignore` that excludes most of it. To wipe: `rm -rf .cook` (or write a `chore clean`, see chapter 6).

## 4.9. Try it

Drop a few `.c` files into `src/` and try this Cookfile:

```cook
recipe compile
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" using { gcc -c $<in> -o $<out> }

recipe link
    cook "build/app" using { gcc $<compile> -o $<out> }
```

`$<compile>` in the link step is a **string substitution**: it expands to the list of `compile`'s outputs, space-separated. So `link` becomes a one-unit recipe that links every object file. Notice the `link` recipe has no `: compile` dependency â€” `$<compile>` already creates the dependency edge for you, and writing both gets you a "conflicting scheduling intent" warning. We cover string substitution in detail in chapter 7.

Run `cook link`, edit one `.c` file, run again â€” only the affected `.o` rebuilds, then the link.
