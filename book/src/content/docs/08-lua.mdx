# 8. Lua: registering work

Most Cookfiles can be written entirely in surface syntax — `recipe`, `ingredients`, `cook`, `plate`, `test`, `chore`. But underneath it all, Cook is a Lua program: every Cookfile compiles to Lua, and the surface forms desugar to calls on the `cook.*` API.

This chapter is about reaching for that API directly when the surface syntax can't quite express what you need.

## 8.1. Two phases, two prefix kinds

Cook executes a recipe in **two disjoint phases** — register, then execute. Lua appears in both, with different prefixes:

| Prefix | Phase    | One-liner               | Block form              |
|--------|----------|-------------------------|-------------------------|
| `>>`   | register | `>> register-time stuff` | `>>{ ... }`            |
| `>`    | execute  | `> runtime stuff`       | `>{ ... }`             |

**Register-phase Lua** runs once, when the recipe is registered. Its job is to *describe* work, typically by calling `cook.add_unit(...)` to record a unit Cook should run later.

**Execute-phase Lua** runs as part of an actual unit. It's what actually does the work when Cook gets to it. Inside an execute-phase body, the `inputs`, `outputs`, `input`, and `output` globals are bound to the unit's resolved I/O.

*(Standard §6.1, §8.1.)*

## 8.2. The simplest example — a Lua-bodied unit

```cook
recipe build
    ingredients "src/*.txt"
    cook "build/$<in.stem>.upper" using >{
        local f = io.open(input, "r")
        local txt = f:read("*a")
        f:close()
        local out = io.open(output, "w")
        out:write(txt:upper())
        out:close()
    }
```

`using >{ ... }` declares a Lua-bodied work unit. At execute time, the body runs with `input` bound to the current ingredient and `output` bound to the current output. One unit per ingredient, just like the shell `using { ... }` form.

Use Lua-bodied units when the work isn't naturally a shell pipeline — pure data transformations, anything that needs string manipulation, or anything where shell quoting is going to bite you.

## 8.3. `cook.add_unit` — the lowest level

The `cook` step desugars to `cook.add_unit(...)`. You can call it directly:

```cook
recipe build
    >>{
        cook.add_unit({
            inputs  = { "main.c" },
            output  = "main.o",
            command = "gcc -c main.c -o main.o",
        })
    }
```

The `>>{ ... }` block says "run this at register time." This records one shell unit. At execute time, Cook spawns `gcc -c main.c -o main.o`, hashes its inputs and command for the cache, and writes `main.o`. The `>>` *one-liner* form (`>> stuff`) only accepts a single physical line — for any multi-line call, use the block form.

`cook.add_unit` accepts:

| Field               | Type             | Default | Meaning |
|---------------------|------------------|---------|---------|
| `inputs`            | array of string  | `{}`    | Files whose contents go into the cache key. |
| `output`            | string           | absent  | Single output path. |
| `outputs`           | array of string  | absent  | Multiple output paths. |
| `command`           | string           | `""`    | Shell command (run via `/bin/sh -c`). |
| `lua_code`          | string           | absent  | Lua source for execute time. If set, `command` is ignored. |
| `interactive`       | boolean          | `false` | Run on the main thread with a TTY. |
| `cache`             | boolean          | `true`  | Set to `false` to never cache. |
| `discovered_inputs` | table            | absent  | Auto-add inputs the command discovers. See §8.7. |

You'll rarely write this by hand; the surface forms cover the common cases. The escape hatch is here when you need it.

## 8.4. `cook.step_group` — a parallel batch

A `cook` step that iterates produces multiple units. Inside a `step_group`, sibling units run in parallel; outside one, they're serial.

```cook
recipe build
    >>{
        cook.step_group(function()
            for _, f in ipairs({ "a.c", "b.c", "c.c" }) do
                cook.add_unit({
                    inputs  = { f },
                    output  = "build/" .. f:gsub("%.c$", ".o"),
                    command = "gcc -c " .. f .. " -o build/" .. f:gsub("%.c$", ".o"),
                })
            end
        end)
    }
```

The surface `cook` step routes through `step_group` automatically — all per-input units in one step are siblings.

## 8.5. `cook.exec` and the shell helpers

For one-off shell work without going through `add_unit`, the API exposes shell helpers. These are useful in chores and execute-phase Lua:

```cook
chore deploy
    > cook.exec("rsync -a build/ deploy:/var/www/")
```

`cook.exec` runs synchronously, returns the captured stdout, and raises a Lua error on non-zero exit. There are also `cook.exec_status` (returns exit code instead of raising) and `cook.exec_quiet` (suppresses output capture).

Inside a register-phase context (`>>`), `cook.exec` *records* a shell unit and returns an empty string — it doesn't actually shell out until execute time. Same call, two phases, two behaviours. *(Standard §8.1.1.)*

## 8.6. The `recipe` global (register phase)

During register-phase Lua inside a recipe, `recipe` is a table with two fields:

- `recipe.name` — the recipe's declared name.
- `recipe.ingredients` — the resolved ingredient groups (a table of tables, one entry per `ingredients` glob).

```cook
recipe build
    ingredients "src/*.c"
    >>{
        for _, src in ipairs(recipe.ingredients[1]) do
            print("will compile " .. src)
        end
    }
```

`recipe` is register-only: it's not bound during execute phase. *(Standard §6.1.1.)*

## 8.7. Discovered inputs — for compilers that emit `.d` files

C/C++ compilers produce dependency files that list *every* header consumed during a compile. Cook can read those and fold the discovered headers into the cache key. The surface form looks like a normal `cook` step:

```cook
recipe compile
    ingredients "src/*.c"
    cook "build/$<in.stem>.o" using {
        gcc -MMD -MF .cook/deps/$<in.stem>.d -c $<in> -o $<out>
    }
```

— but to declare the `.d` file as a discovered-inputs source, you drop into Lua and call `cook.add_unit` directly, building the command and path strings yourself:

```cook
recipe compile
    ingredients "src/*.c"
    >>{
        for _, src in ipairs(recipe.ingredients[1]) do
            local stem = path.stem(src)
            local obj  = "build/" .. stem .. ".o"
            local dep  = ".cook/deps/" .. stem .. ".d"
            cook.add_unit({
                inputs  = { src },
                output  = obj,
                command = string.format(
                    "gcc -MMD -MF %s -c %s -o %s", dep, src, obj),
                discovered_inputs = { from = dep, format = "make" },
            })
        end
    }
```

The key distinction: surface `cook` steps see `$<in>`/`$<out>` placeholders and substitute them per-unit. `cook.add_unit` takes the *final* command string — you build it with Lua string ops. The Cook placeholder lexer only runs inside surface forms.

After each unit runs, Cook reads `.cook/deps/<stem>.d`, parses it as Make-style dependencies, and adds those headers to the unit's input set for the next build. Header changes pick up automatically. *(Standard §6.2.1.)*

## 8.8. Where to use Lua and where not to

**Use surface syntax when you can.** `recipe build`, `ingredients`, `cook`, `plate`, `test`, `chore` cover most projects. Surface syntax is shorter, denser, and reads more like a list of things than like a program.

**Reach for Lua when:**
- You need a transformation that's natural in Lua and awkward in shell (string manipulation, JSON, anything with quoting).
- You need to register a *variable* number of units based on something computed at register time.
- You need `discovered_inputs` (compilers) or other `cook.add_unit` fields the surface doesn't expose.
- You're writing a **module** (chapter 9) — modules are pure Lua.

## 8.9. Try it

```cook
recipe count
    ingredients "src/*.c"
    cook "build/lines.txt" using >{
        local total = 0
        for _, f in ipairs(inputs) do
            for line in io.lines(f) do
                total = total + 1
            end
        end
        local out = io.open(output, "w")
        out:write(tostring(total) .. "\n")
        out:close()
    }
```

Run `cook count`, then add a line to one of the sources and run again. The Lua-bodied unit re-runs (its inputs changed) and the line count updates. The cache treats Lua units exactly the same as shell units.
