# 5. `test`

The `cook` step produces files. A `test` step lets you *check* the things `cook` produced. (If what you want to do is run something for its side effects — ship a build, deploy, format the tree — that's a `chore`, covered in the next chapter. `test` is specifically for checks that pass or fail.)

## 5.1. `test` — run a check

A `test` step takes the recipe's current input list and runs a shell or Lua block against it. It doesn't declare outputs; it doesn't transform files. Its job is "verify something about each of these inputs."

```cook
recipe demo
    ingredients "examples/*.sh"
    test { ./$<in> }
```

For each `examples/*.sh` ingredient, `./examples/foo.sh` runs and its exit code is the pass/fail verdict. One unit per input.

A failure in one `test` unit doesn't cancel the others — Cook collects all failures and reports them together, with output captured per-unit so you get a clean per-test summary. *(Standard §4.8 for the grammar; §8.3 for the parallelism rule.)*

## 5.2. `test` after a `cook` step

The common case: `test` inherits the *preceding `cook` step's outputs* as its inputs (or, if no `cook` step has run yet, the recipe's resolved ingredients). Chain them:

```cook
recipe build
    ingredients "src/*.c"
    cook "build/$<in.stem>" { gcc $<in> -o $<out> }
    test { ./$<in> --selftest }
```

`build` compiles each `.c` to an executable, then checks each one. The `cook` step's outputs become the next step's inputs.

## 5.3. `cook test` — the subcommand

```bash
cook test                # run every test step in the workspace
cook test apps.web       # scope to recipes under the apps.web namespace
cook test apps.web.unit  # scope to a specific full recipe name
```

`cook test` is the first specialized subcommand we've seen. It walks the workspace and runs every work unit Cook would normally run, then aggregates only the *test* results into a single human-readable summary — non-test units still execute, but only test units show up in the summary. The optional positional is a *recipe scope* (a namespace prefix or a full recipe name), not a substring match. We'll see the rest of these subcommands in chapter 10.

## 5.4. Naming and inverting checks

Tests have no modifiers. There's no `as` to rename one, no `timeout` to cap one, no `should_fail` to flip one's polarity:

- **Naming**: an unnamed test displays as `recipe@line` in `cook test` output — always derived from where it's written, never overridden.
- **Negative tests**: to write a check that's expected to fail (a rejected input, a lint that must fire), invert the command itself with `!` instead of reaching for a modifier:

```cook
recipe lint
    test { ! grep -q TODO $<src> }
```

If a leftover `TODO` is present, `grep -q TODO` exits `0` (found) and `!` flips that to a failing test. Clean source exits `1` (not found) and `!` flips that to a pass — exactly the old `should_fail` behavior, expressed in the body instead of a modifier.

## 5.5. Try it

```cook
recipe compile
    ingredients "src/*.c"
    cook "build/$<in.stem>" { gcc $<in> -o $<out> }

recipe unit: compile
    ingredients "build/test_*"
    test { ./$<in> }

recipe lint: compile
    test { ! grep -rq TODO src }
```

Run `cook test` — every `test` step in the workspace runs (`unit`'s one-to-one units and `lint`'s naked one-shot unit), and Cook reports a per-check pass/fail summary. Scope it to just one recipe with `cook test unit`, or short-circuit on the first failure with `cook test --fail-fast`.
