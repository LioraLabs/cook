# 1. Install cook

`cook` is a single Rust binary with a bundled Lua runtime and a bundled LuaRocks. There's no package manager to fight, no system Lua to match against, and no separate "cook server" — it's one executable that you run from your project directory.

## 1.1. Quick install (Linux / macOS)

The fastest way is the install script:

```bash
curl -fsSL https://getcook.sh | sh
```

This installs the entire bundled tree under `~/.cook/` and puts `cook` on your `PATH` via `~/.cook/bin`. Re-run the script any time you want to update.

To install somewhere else, set `COOK_INSTALL_DIR`:

```bash
COOK_INSTALL_DIR=/opt/cook curl -fsSL https://getcook.sh | sh
```

If you'd rather grab a tarball by hand, releases live on the [GitHub releases page](https://github.com/LioraLabs/cook/releases). Each release ships a tarball per host target plus matching `.sha256` files; the tarball is the same `~/.cook/`-shaped tree, ready to extract anywhere.

## 1.2. Verify the install

```bash
cook --version
cook --help
```

You should see the version banner and a list of subcommands. If `cook: command not found`, your shell hasn't picked up `~/.cook/bin` yet — open a new terminal or add it to your `PATH`.

## 1.3. What got installed

The install is a self-contained `~/.cook/` tree (the `cook` binary, a bundled Lua 5.4 runtime, and a bundled LuaRocks). You don't usually need to think about its internals — `tree ~/.cook/` if you're curious.

The tree is **relocatable**: at runtime the bundled LuaRocks launcher derives its install root from its own location (exposed as the `COOK_PREFIX` env var) and writes nothing into your home directory beyond this tree. Move `~/.cook/` to `/opt/cook/` and it still works.

The one time the layout matters is when you run `cook modules install` (chapter 9) — that command shells out to the bundled LuaRocks, not whatever `lua` and `luarocks` happen to be on your `PATH`.

<div class="book-tip">

**Tip — no system Lua required.** Cookfiles execute Lua, but you don't need a system Lua install. The bundled runtime is what runs your `>` and `>{ ... }` blocks, your modules, and your config files. This is intentional: the version of Lua a Cookfile sees is fixed across machines, so a recipe that works on your laptop works on CI.

</div>

## 1.4. Updating

```bash
curl -fsSL https://getcook.sh | sh
```

Same command as the install. The script is idempotent — it checks the latest release, downloads only if you're behind, and replaces the tree atomically.

## 1.5. Where state lives

| Path | Owner | What's there |
|---|---|---|
| `~/.cook/` (or `$COOK_INSTALL_DIR`) | the install tree | binary + bundled Lua + LuaRocks. Never touched at build time. |
| `<project>/.cook/cache/` | per project | content-hashed cache for `cook` step outputs. Appears once recipes produce cacheable outputs (a recipe with only bare-shell steps leaves no `cache/`). Safe to delete; rebuild repopulates. |
| `<project>/.cook/logs/` | per project | per-build log archive (`cook logs` reads these). |

`cook init` writes a `.gitignore` that excludes `<project>/.cook/` for you.

## 1.6. Uninstalling

`cook` doesn't write outside its install tree and per-project `.cook/` directories. To remove the install:

```bash
rm -rf ~/.cook
```

(Plus whatever per-project `.cook/` directories you want to wipe — they'll regenerate on the next build.)

## 1.7. Try it

Pick a scratch directory, run:

```bash
mkdir hello-cook && cd hello-cook
cook init
cook build
```

You should see something close to:

```
  build                    queued  (0 nodes)
  build/lua                                     0.00s
  build                    done     (1/1)                    0.00s
cook done in 0.00s (0 nodes, 0 cached recipes, 1 done)
Hello from Cook!
```

The exact spacing varies but the shape doesn't: a queued line, a per-step progress line, a recipe summary, a build summary, then your shell command's output. That's it — you've run your first Cookfile. The next chapter explains exactly what just happened.
